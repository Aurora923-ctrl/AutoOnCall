# AutoOnCall 技术长文提示词索引

本文档用于驱动 12 篇项目技术长文和 1 篇校招面试总复盘。每篇文章都应独立成文，但只围绕自己的主题展开，其他链路只做必要背景，不深入抢占其他文章内容。

## 通用写作约束

所有文章统一使用以下要求：

- 使用中文，面向“准备校招但刚开始学习项目”的读者。
- 必须结合当前 AutoOnCall 仓库真实代码，不要编造不存在的能力。
- 每篇开头用 5-8 行交代项目背景：这是一个 Python FastAPI 应用，用于 RAG 问答和 AIOps 智能诊断。
- 只写本文主题相关内容。遇到其他主题，只说明它在链路中的位置，并指向相邻文章，不展开细节。
- 代码讲解要引用具体路径、核心类、核心函数、API 路由、数据模型。
- 业务链路按“请求入口 -> 数据模型 -> 服务层 -> 状态变化 -> 外部依赖 -> 返回/沉淀结果 -> 测试覆盖”的顺序写。
- 每个关键设计都说明：为什么这样设计、解决什么问题、边界情况是什么。
- 末尾必须包含“面试官可能追问”和“推荐回答”。
- 输出为 Markdown，结构清晰，有标题、二级标题、必要的流程图 Mermaid。
- 不要只堆代码清单，要把代码背后的工程取舍讲清楚。
- 如果发现代码与设计文案不一致，标注“代码当前实现”与“可改进方向”。

# AutoOnCall 校招生项目评估与改进建议

生成说明：本文档整理自一次围绕 AutoOnCall 项目的评估对话，视角是“大模型开发项目负责人如何评价一名校招生候选人的入职前作品”，重点回答两个问题：

1. 这个项目当前处在什么水平。
2. 作为校招生，接下来应该按什么顺序、改到什么程度。

## 1. 总体评价

AutoOnCall 是一个面向 OnCall 故障诊断场景的 RAG + AIOps Agent 项目。它不是普通聊天机器人，也不是简单 RAG Demo，而是把告警接入、Incident、Runbook 检索、Plan-Execute-Replan 诊断、工具取证、证据链、风险审批、安全变更、Trace 和报告串成了一条较完整的业务闭环。

如果这是校招生在入职前完成的项目，综合评价是：

> 86/100，强通过。适合作为大模型应用开发、AI Agent 工程、AIOps 平台方向的主项目。

项目最突出的价值不是“调用了大模型 API”，而是已经有了把大模型放进工程边界里的意识：模型负责生成计划和总结，工具负责真实或模拟取证，规则负责风险控制，评测负责回归验证。

## 3. 当前项目的主要亮点

### 3.1 业务链路完整

项目不是停留在“用户提问、模型回答”，而是有一条明确的 AIOps 主链路：

```text
告警 / Incident 输入
-> Planner 生成诊断计划
-> Executor 调用工具
-> Evidence 形成证据链
-> Replanner 判断是否继续、补查、审批或报告
-> 生成诊断报告
-> 审批和安全变更闭环
```

这说明项目已经具备业务流程原型，而不是孤立的模型能力展示。

### 3.2 RAG 可信边界较好

RAG 不是简单向量检索后拼上下文，而是包含：

- Markdown / 文本切分。
- Milvus 向量检索。
- 本地词法索引。
- hybrid search。
- rerank。
- 距离阈值过滤。
- 无可信来源拒答。
- 引用来源补齐。

这些设计都比较贴近真实大模型应用会遇到的问题。

### 3.3 Agent 不让模型随意执行

项目采用 Planner / Executor / Replanner 结构，并且 Executor 优先通过 Tool Registry 调用工具。工具有统一契约，包含输入输出、风险等级、只读属性、超时和降级策略。

这比让大模型自由决定工具调用更可控，也更容易解释和测试。

### 3.4 安全边界意识强

项目对动作做了分级：

| 动作类型 | 当前策略 |
| --- | --- |
| 查询指标、日志、Runbook、状态 | 自动执行 |
| 重启、扩容、回滚、配置变更 | 进入审批 |
| 删除 Pod、危险 shell、未审核 SQL | 直接禁止 |
| 审批后的变更 | pre-check、dry-run、sandbox 或人工记录 |

对校招生项目来说，这一点很加分：它没有宣传“Agent 自动修生产”，而是强调诊断、建议、审批和安全边界。

### 3.5 测试和评测体系扎实

项目已经有较完整的单元测试、接口测试、工具测试、RAG 评测、AIOps 评测和安全变更评测。我们在对话中验证过：

- pytest：303 passed, 1 skipped。
- Ruff：通过。
- AIOps 离线评测：16/16。
- RAG 离线评测：22/22。
- 安全变更评测：9/9。

虽然这些评测是 deterministic/offline，并不代表真实生产准确率，但作为校招生作品，已经很强。

## 4. 当前主要短板

### 4.1 生产化仍偏原型

当前项目有健康检查、配置、SQLite/MySQL、部署文档和 mock 关闭开关，但还不能按成熟商用系统要求评价。

如果按企业生产系统看，还缺少：

- SSO/OIDC。
- 企业级密钥治理。
- 完整审计系统。
- 正式数据库迁移流程。
- CI/CD 合并门禁。
- 更强的线上可观测性。

不过对校招生来说，这些不应该成为下一阶段主要目标。

### 4.2 大文件较多

部分文件职责偏重，例如：

- `report_generator.py`
- `aiops_service.py`
- `change_execution_service.py`
- `executor.py`
- `evidence_analyzer.py`

这对个人项目可以接受，但如果进入团队维护，会增加理解成本。

### 4.3 真实数据验证不足

当前评测更像离线回归，不是真实线上故障样本评测。项目可以证明“这条工程链路能稳定跑”，但还不能证明“线上根因识别准确率足够高”。

对校招生来说，不必补齐真实生产数据集，但需要在文档和面试中讲清楚这个边界。

### 4.4 mock / sandbox / 真实适配器需要讲得更清楚

项目中同时存在 mock、fallback、sandbox、真实适配器雏形。它们本身不是问题，问题是面试官可能会追问：

- 哪些数据是真的？
- 哪些是 mock？
- mock 是否会影响报告可信度？
- 关闭 mock 后系统如何表现？

因此下一阶段不一定要接更多真实系统，但必须把边界讲清楚。

## 5. 校招生后续改进顺序

对校招生来说，改进顺序不要按“最低分先补”，而应该按以下原则：

1. 面试收益高。
2. 能独立完成。
3. 不显得过度包装。
4. 能证明工程能力。
5. 不把项目做成不像校招生能完成的企业平台。

推荐顺序如下：

| 顺序 | 重点维度 | 目标程度 |
| --- | --- | --- |
| 1 | 文档表达 + 演示链路 | 让项目 10 分钟讲清楚 |
| 2 | 测试与评测 | 从“测试多”变成“评测可信” |
| 3 | 安全边界 | 讲清 mock、审批、禁止动作 |
| 4 | 大模型应用能力 | 强化 RAG 拒答、引用、负例 |
| 5 | Agent 架构 | 强化失败降级和计划可解释性 |
| 6 | 工程质量 | 轻量拆大文件，不做大重构 |
| 7 | 前端工作台 | 够演示即可 |
| 8 | 生产化程度 | 做校招生合理版，不做企业平台 |

## 6. 第一优先：整理稳定面试演示链路

这是最重要的改进。

目标不是新增功能，而是让面试官一眼看懂项目主线：

```text
上传 Runbook
-> 发起 Redis / MySQL / Pod 故障诊断
-> 展示 Plan
-> 展示工具调用
-> 展示证据链
-> 展示风险审批
-> 展示诊断报告
```

建议做到：

- README 增加“面试演示路径”。
- 准备 2-3 个固定 demo case。
- 每个 case 写清楚输入、预期调用工具、预期根因、预期报告。
- 保证本地几步命令能跑通。

推荐 demo case：

1. Redis maxclients 导致接口超时。
2. MySQL 慢 SQL 导致响应变慢。
3. Pod OOMKilled 导致服务不可用。

完成标准：

- 10 分钟内能讲完一条主线，不需要临场翻代码找入口。
- README、面试讲解文档和前端 demo 模板使用同一套 case 名称。
- 每个 demo case 都写清输入、预期工具、预期证据、预期报告结论和风险边界。
- mock / sandbox / 真实适配器边界有固定说法，不把本地演示包装成生产接入。
- RAG 至少有一个正例和一个拒答负例，能展示引用和无可信来源拒答。
- 诊断链路至少能展示 Plan、ToolCall、Evidence、Trace 和 Report。
- 验证命令能跑通，并说明离线评测不等于线上准确率。

这个改进的价值很高，因为面试官最怕项目很大但讲不清。

## 7. 第二优先：补高质量评测 case

现在测试已经不少，下一步不要追求“更多测试文件”，而是补更像真实排障的评测 case。

建议目标：

| 评测类型 | 当前方向 | 建议目标 |
| --- | --- | --- |
| AIOps case | 工具选择、根因命中、审批策略 | 15-25 个高质量 case |
| RAG case | 召回、引用、拒答 | 20 个左右 |
| 安全变更 case | 审批、dry-run、禁止动作 | 8-10 个 |

重点覆盖：

- Redis 连接耗尽。
- MySQL 慢 SQL。
- Pod OOMKilled。
- 服务 503。
- 下游服务超时。
- 磁盘满。
- 工具失败后降级报告。
- 危险 SQL 禁止。
- 无知识库依据拒答。

每个 case 建议包含：

- 输入 Incident。
- 期望工具。
- 期望根因关键词。
- 是否需要审批。
- 是否应该拒答。

面试时可以这样表达：

> 我不是只做了一个演示，而是用离线 case 固定验证工具选择、根因命中、审批策略和拒答策略。

本次落实程度：

- `eval/change_cases.yaml` 从 6 个安全变更 case 增加到 9 个，达到 8-10 个目标区间。
- 新增 rejected approval 不能进入安全变更执行，补齐 pending 之外的审批拒绝边界。
- 新增 staging sandbox validated，证明 sandbox 模式只做沙箱验证和观察记录，不执行生产变更。
- 新增 prod sandbox without flag escalates，证明生产环境未显式开启沙箱时会转人工接管。
- 新增 `tests/test_change_eval_cases.py`，固定验证 case 覆盖面、9/9 通过率和关键状态。

当前验收命令：

```bash
make eval-change
venv/bin/python -m pytest tests/test_change_eval_cases.py tests/test_change_execution_service.py
```

预期结果：`Safe-change eval: 9/9 cases passed`，并且 approval、dry-run、rollback、forbidden 指标都是 100%。

## 8. 第三优先：讲清 mock / sandbox / 真实适配器边界

这是校招生项目最容易被质疑的地方。

建议改到：

- README 明确说明哪些是 mock。
- 报告里保留 `data_source`。
- 关闭 mock fallback 时，工具返回 `not_configured` 或 `failed`。
- 文档里说明本项目验证的是 Agent 工程链路，不声称已经接入真实生产系统。

可以增加一节：

```text
本项目能力边界
- 本地演示使用 mock / sandbox 数据。
- 真实生产接入需要替换外部适配器配置。
- Agent 只做诊断和建议，不自动执行生产写操作。
- 离线评测通过率不等同于线上真实根因准确率。
```

这一点不需要做复杂功能，只要说清楚，就会显得很成熟。

## 9. 第四优先：加强 RAG，但只做应用层增强

当前 RAG 已经不错，下一步适合做小而稳的增强。

建议做到：

- 每个回答必须带 citation。
- 无可信来源稳定拒答。
- RAG eval 增加负例。
- 增加 2-3 个容易混淆的 case，例如“CPU 高但根因是慢 SQL”。
- README 画清楚 RAG 链路：切分、向量、词法、rerank、阈值、拒答。

本次落实程度：

- API 层增加 citation guard：成功回答前必须至少有一个 `source_file + chunk_id`，否则按 `refuse_without_citation` 拒答。
- RAG eval 从 17 个 case 增加到 22 个 case，拒答负例从 3 个增加到 5 个。
- 混淆 case 显式标记为 `case_type: confusion`，覆盖 CPU 高但根因像慢查询、503 但根因是依赖超时、磁盘高但根因是容器日志等场景。
- RAG eval 新增 `citation_coverage_rate` 和 `confusion_case_pass_rate`，能证明“有引用才回答”和“容易混淆的问题也能命中正确 Runbook”。
- README 已补完整 RAG 链路图，明确切分、向量、词法、rerank、trust gate、拒答和引用兜底。

当前验收命令：

```bash
venv/bin/python scripts/eval/eval_rag_cases.py
```

预期结果：`RAG eval: 22/22 cases passed`，并且 `cite=100%`、`confusion=100%`、`reject=100%`。

不建议做：

- reranker 训练。
- 知识图谱。
- 模型微调。
- 复杂多路召回平台。

对校招生来说，能讲清“为什么拒答、为什么引用、为什么不让模型自由发挥”已经很加分。

## 10. 第五优先：让 Agent 更稳定，而不是更复杂

不要加“多 Agent 专家组”这类容易显虚的东西。

应该重点做到：

- Planner 失败时 fallback plan 稳定可用。
- 工具失败时 Evidence 里能看到失败原因。
- Replanner 能决定继续、补查、报告、审批或升级人工。
- 报告里能说明证据不足和不确定性。

一句话：

> 让 Agent 可控、可解释、可失败，而不是看起来很智能。

这比继续堆 Agent 名词更适合校招生项目。

## 11. 第六优先：轻量提升工程质量

当前工程质量已经不错，不需要大重构。

建议只做：

- 拆 1-2 个超大文件。
- 加一个简单 CI。
- 保持 `make test-quick`、`make eval`、`make eval-rag`、`make eval-change` 全绿。
- 清理未提交改动和生成文件。

拆分目标：

- 不要求所有文件都很小。
- 只拆最明显的职责。
- 保证行为不变。
- 用现有测试回归。

可以优先考虑：

- 从 `report_generator.py` 拆出 Markdown 渲染辅助。
- 从 `executor.py` 拆出 Evidence 构造辅助。
- 从 `aiops_service.py` 拆出 SSE 事件格式化辅助。

这能体现维护意识，但不会显得过度工程化。

## 12. 第七优先：前端只做演示体验优化

前端不是这个项目的主战场。

建议改到：

- 能清楚展示 Plan。
- 能清楚展示工具调用。
- 能清楚展示证据链。
- 能清楚展示报告和审批状态。
- 状态不要乱跳，关键按钮文案清楚。

不建议把静态 JS 重写成 React/Vue。对这个项目来说，收益不高，而且会分散重点。

## 13. 第八优先：生产化只做校招生合理版

生产化分数最低，但不代表应届生应该优先补齐企业级生产系统。

对校招生来说，做到以下程度就够：

- 有 `.env.example`。
- 有健康检查。
- 有启动风险提示。
- 有 mock 关闭配置。
- 有 SQLite/MySQL 存储选择。
- 有简单部署说明。
- 有 CI 或本地验收命令。

不建议继续做：

- OIDC / SSO。
- Kubernetes Helm。
- 多租户系统。
- 企业级审计平台。
- 自动生产修复。
- 复杂数据库迁移平台。

这些不是不好，而是不像校招生项目下一阶段最合理的投入方向。

## 16. 面试时推荐表达

可以这样介绍项目：

> AutoOnCall 是一个面向 OnCall 故障诊断场景的 RAG + AIOps Agent 原型。它不是简单聊天机器人，而是把告警接入、Runbook 检索、计划生成、工具取证、证据分析、风险审批和诊断报告串成一条可解释闭环。

可以这样介绍大模型边界：

> 我没有让模型直接自由生成结论，而是让模型参与 Planner 和报告生成。真正的诊断依据来自 Tool Registry 里的工具结果，并统一转成 Evidence。没有可信知识来源时，RAG 会拒答。

可以这样介绍安全边界：

> Agent 只自动执行只读排查。重启、回滚、扩容等动作进入审批；删除 Pod、危险 SQL、危险 shell 会直接禁止。审批后也不会直接改生产，只进入 dry-run 或人工执行记录。

可以这样介绍评测：

> 我用离线 case 验证工具选择、根因命中、RAG 拒答、审批策略和安全变更流程。这个评测不代表线上准确率，但能证明核心工程链路不会轻易漂移。

## 17. 总结

这个项目接下来最好的方向不是继续堆功能，而是：

> 少加功能，多增强可信度；少讲生产级，多讲边界和验证。

对校招生来说，最理想的改进结果是让面试官相信：

1. 你知道大模型应用不只是调 API。
2. 你理解 RAG 的可信来源、引用和拒答。
3. 你知道 Agent 必须有工具边界和风险控制。
4. 你能用测试和评测证明系统行为。
5. 你不会把 mock demo 夸成生产系统。

如果能做到这些，这个项目已经足够成为一份很强的校招生大模型开发作品。

# AutoOnCall 校招竞争力对标与改进清单

> 生成时间：2026-07-05  
> 目的：沉淀本次调研结论，帮助 AutoOnCall 在校招中从“不错的大模型应用项目”升级为“有业务闭环、有工程深度、有评测指标的 AIOps Agent 项目”。

## 1. 核心结论

AutoOnCall 不建议推倒重做。当前项目已经明显强于普通 RAG 知识库、PDF 问答、LangChain 课程作业和多数“套壳聊天机器人”项目。

更优路线是继续把 AutoOnCall 往 **HolmesGPT 风格的 SRE / AIOps Agent** 打磨：

```text
告警 / Incident
  -> Planner 生成排查计划
  -> 工具采集 Prometheus / Loki / Redis / MySQL / K8s / 工单 / Runbook 证据
  -> Evidence Analyzer 判断支持、反驳和不确定证据
  -> Replanner 决定补查、审批、拦截或生成报告
  -> Trace / Approval / Report / Eval 闭环
```

校招里真正有竞争力的不是“我用了大模型”，而是：

- 有具体业务：线上故障诊断。
- 有工程闭环：工具调用、风险审批、Trace、报告、评测。
- 有安全边界：只读诊断自动执行，高风险动作审批，危险动作禁止。
- 有可复现指标：RCA 准确率、工具命中率、审批命中率、拒答率、报告完整率。

## 2. 网上值得对标的项目

### 2.1 最值得对标：SRE / AIOps Agent

| 项目 | 类型 | 值得借鉴的点 | 对 AutoOnCall 的启发 |
| --- | --- | --- | --- |

## 3. AutoOnCall 当前竞争力

### 3.1 当前定位

AutoOnCall 的最佳校招定位：

> 面向 OnCall 故障诊断的 RAG + LangGraph AIOps Agent：具备知识检索、工具调用、证据分析、风险审批、Trace、报告生成和离线评测闭环。

这比普通知识库问答项目更有区分度，因为它不仅回答问题，还能围绕一次故障做结构化诊断。

### 3.2 已有亮点

- FastAPI + SSE，支持实时诊断事件流。
- LangGraph Plan-Execute-Replan 主链路。
- RAG 知识库问答，支持引用和无可信来源拒答。
- Tool Registry 统一指标、日志、Redis、MySQL、K8s、Runbook、历史工单等工具调用。
- Evidence / ToolCallRecord / TraceEvent / ApprovalRequest / DiagnosisReport 等结构化对象。
- Risk Controller：只读动作自动执行，中高风险动作审批，危险动作禁止。
- Trace、Approval、Report 默认 SQLite，也可切换 MySQL。
- 已有离线评测和测试体系，覆盖 AIOps、RAG、审批、报告等核心链路。
- 有本地 full-stack sandbox，覆盖 Redis、MySQL、Prometheus、Loki、Alertmanager、Grafana 等组件。

### 3.3 主要短板

- 真实数据源占比还需要继续提高，减少 mock/fallback 依赖。
- K8s、日志网关、工单系统、链路追踪等接入还可以更完整。
- RCA 评测指标可以进一步体系化，不只看 case pass/fail。
- 工具输出压缩、token 预算、超大日志处理还不够像生产系统。
- LangGraph checkpoint 当前仍有内存态边界，审批恢复和服务重启后的状态恢复还可增强。
- 面试材料需要更可视化：Trace 截图、报告样例、评测 dashboard、演示脚本。

## 4. 改进清单

## P0：最优先，直接影响校招展示

### 4.1 固化 3 个稳定端到端故障场景

借鉴：HolmesGPT、ShopSimulator。

建议场景：

1. Redis 连接耗尽 / maxclients 接近上限。
2. MySQL 慢查询 / 锁等待 / 连接池耗尽。
3. Kubernetes Pod 异常 / CrashLoopBackOff / OOMKilled。

每个场景必须展示：

- 告警输入。
- Planner 排查计划。
- 工具调用记录。
- Prometheus 指标。
- Loki / 日志证据。
- Redis / MySQL / K8s 状态。
- Runbook 命中。
- Evidence 支持和反驳关系。
- Risk Controller 风险判断。
- 最终诊断报告。

验收标准：

- 每个场景 5 分钟内能稳定跑通。
- 报告中明确标注数据来源是真实沙箱、mock 还是 not_configured。
- 每个场景至少包含 4 类数据源。

### 4.2 强化 RCA 评测集

借鉴：ShopSimulator、NyayaAI、LexRAG。

新增指标：

- RCA 根因命中率。
- 工具命中率。
- 无关工具率。
- 风险策略命中率。
- 证据支撑率。
- 反证识别率。
- 报告完整率。
- RAG 引用准确率。
- 无可信资料拒答率。

验收标准：

- `scripts/eval_cases.py` 输出 JSON + Markdown summary。
- summary 中不只显示通过数量，还显示上述分类指标。
- 至少增加 5 个 contradiction case，专门测试“用户假设被工具证据反驳”的情况。

### 4.3 证据驱动根因判断

借鉴：HolmesGPT、RCA Agent 论文。

核心要求：

- 不允许仅凭 incident 文本强判根因。
- 每个根因必须关联 supporting evidence。
- 正常指标要形成 refuting evidence。
- 工具失败要形成 unknown evidence。
- 证据冲突时降低置信度并提示人工确认。

验收标准：

- 报告中明确展示：
  - 支持证据。
  - 反驳证据。
  - 不确定证据。
  - 置信度原因。
- Redis 正常、MySQL 正常、日志缺失等反例不会被强行归因为异常。

### 4.4 生成可展示的诊断报告样例

借鉴：DB-GPT 报告流、HolmesGPT findings。

报告建议包含：

- Incident 基本信息。
- 诊断时间线。
- 工具调用表。
- 证据矩阵。
- 根因假设。
- 影响面分析。
- 风险动作与审批状态。
- 下一步建议。
- 数据来源说明。

验收标准：

- `logs/demo_reports/*.md` 或前端报告页可稳定展示。
- 面试时可以直接打开 1 份 Redis、1 份 MySQL、1 份 K8s 报告。

## P1：显著拉开差距

### 4.5 MCP 化运维工具层

借鉴：HolmesGPT、MCP servers。

优先 MCP server：

- `prometheus_mcp_server`
- `loki_mcp_server`
- `redis_mcp_server`
- `mysql_mcp_server`
- `kubernetes_mcp_server`
- `ticket_mcp_server`

收益：

- 面试时可以讲“工具协议化”，不是写死 Python 函数。
- 后续可以接入 Claude Desktop、OpenHands、Codex、Dify 等 MCP 客户端。
- 工具边界更清楚，适合扩展和测试。

验收标准：

- 至少 2 个真实 MCP server 可以跑通。
- Tool Registry 可以统一调用本地工具和 MCP 工具。
- 文档说明 MCP 工具的输入、输出、权限和安全边界。

### 4.6 增加工具输出压缩和预算控制

借鉴：HolmesGPT 的 large payload / output budgeting 思路。

建议能力：

- 日志查询默认限制时间窗口和行数。
- 大响应先写入文件，再生成摘要给模型。
- 工具输出保留 raw artifact path。
- 每次诊断记录 tool_calls、耗时、输出大小、估算 token。

验收标准：

- 超大日志不会直接塞进 LLM prompt。
- 报告中能看到“原始数据路径 + 摘要 + 裁剪原因”。

### 4.7 真实 K8s / Loki / Tempo 场景

借鉴：HolmesGPT 的 Kubernetes、Loki、Tempo 数据源。

建议分阶段：

1. K8s mock -> kind / minikube 半真实集群。
2. Loki 接收真实容器日志。
3. OpenTelemetry + Tempo 接入链路追踪。
4. 构造高延迟 Trace 场景。

验收标准：

- K8s Pod 异常场景不只靠 mock。
- 高延迟场景可以通过 Trace 看到慢 span 或异常依赖。

### 4.8 增加历史工单和变更关联

借鉴：HolmesGPT 的 Jira / ServiceNow / GitHub integration。

建议能力：

- 查询最近部署记录。
- 查询历史相似 incident。
- 查询服务 owner、依赖拓扑、SLO。
- 报告中展示“最近变更是否可能相关”。

验收标准：

- MySQL / Redis / K8s 场景至少有 1 个能关联最近变更或历史工单。
- Evidence 中区分“实时事实”和“历史背景”。

## P2：锦上添花

### 4.9 LangGraph 持久化 checkpoint

借鉴：LangGraph Academy、DeepLearning.AI LangGraph 课程。

收益：

- 服务重启后审批恢复更可靠。
- 更能解释“长事务 Agent 状态管理”。

验收标准：

- 使用 SQLite 或 MySQL checkpointer。
- 审批等待中的诊断在服务重启后仍可恢复状态。

### 4.10 RBAC / Admin Token

收益：

- 从本地 demo 更像企业内网系统。
- 面试中安全边界更完整。

建议角色：

- viewer：只看报告和 Trace。
- operator：发起诊断。
- approver：审批高风险动作。
- admin：配置工具和数据源。

### 4.11 Incident 工作台体验

借鉴：HolmesGPT、NyayaAI。

前端重点：

- Incident 列表。
- Trace 时间线。
- Evidence 矩阵。
- 工具调用详情。
- 审批中心。
- 评测 summary。
- 报告预览。

不要把重点放在炫酷前端，重点是“让面试官一眼看到诊断链路”。

### 4.12 一键 demo 脚本

建议命令：

```powershell
make sandbox-up
make sandbox-seed
make sandbox-verify
make demo-redis
make demo-mysql
make demo-k8s
make eval
```

验收标准：

- 面试前能一键生成 demo 数据、报告和评测 summary。
- README 中有 5 分钟演示脚本。

## 5. 不建议优先做的事

- 不要为了炫技改成多 Agent 群聊系统。
- 不要急着做自动修复生产故障。
- 不要重写成复杂 React 前端。
- 不要堆一堆随机工具。
- 不要做模型微调，除非已经有稳定数据集和明确收益。
- 不要把 mock 结果包装成真实生产接入。
- 不要在简历里写“线上准确率 100%”。

## 6. 推荐简历表述

稳妥版本：

```text
基于 FastAPI、LangGraph、Milvus 和 DashScope 构建面向 OnCall 故障诊断场景的 RAG + AIOps Agent 系统，支持知识库问答、结构化 Incident 诊断、工具调用、风险审批、Trace 追踪、诊断报告和离线评测闭环。

设计 Plan-Execute-Replan 诊断流程，将故障排查拆分为 Planner、Executor、Evidence Analyzer、Replanner 和 Report Generator；通过 Tool Registry 统一指标、日志、Redis、Kubernetes、MySQL、Runbook、历史工单等工具调用，并将工具结果归一为 Evidence 和 ToolCallRecord。

设计 Human-in-the-loop 风险控制链路，对只读排查自动放行，对重启、扩缩容、回滚、配置变更等中高风险动作生成审批请求，对删除 Pod、未审核 SQL、危险 shell 等动作直接阻断，避免 Agent 自动执行生产变更。

构建离线评测集覆盖 AIOps 和 RAG 场景，验证工具命中、根因命中、审批触发、禁止动作拦截、工具失败降级、RAG 引用和无答案拒答；评测结果以 JSON/Markdown 报告形式可复现。
```

升级后可以加一句：

```text
参考 HolmesGPT 等 SRE Agent 的工程实践，将 Prometheus、Loki、Redis、MySQL、Kubernetes 等观测工具逐步 MCP 化，并在本地 full-stack sandbox 中构造 Redis、MySQL、K8s 等可复现故障场景，用于验证 RCA 准确率、工具命中率和报告完整率。
```

## 7. 推荐执行顺序

1. 固定 Redis / MySQL / K8s 三个端到端 demo。
2. 扩展 RCA 评测指标，不只看 pass/fail。
3. 强化 Evidence supporting / refuting / unknown 逻辑。
4. 输出 3 份高质量诊断报告样例。
5. MCP 化 2 个最核心工具：Prometheus 和 Loki。
6. 补 K8s 半真实环境。
7. 增加 checkpoint、RBAC 和更好的前端工作台。

一句话总结：

> AutoOnCall 当前已经不是普通 RAG 项目。下一阶段最有效的加分方式，是把它从“能跑的 AIOps Demo”打磨成“有真实观测数据、有安全边界、有 RCA 评测、有报告闭环的 SRE Agent 原型”。

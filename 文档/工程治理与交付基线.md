# AutoOnCall 工程治理与交付基线

本文档用于约束 AutoOnCall 后续改进方向：不再扩功能面，优先把已有 RAG、AIOps、审批、报告、安全变更、评测和部署说明收敛成一个可维护、可验证、可交付的完整项目。

## 1. 当前阶段目标

当前阶段不是“继续做大”，而是“做薄、做稳、做干净”。

目标：

- 让 README 成为项目入口，而不是演示稿。
- 让核心服务职责更清楚，避免单文件同时承担编排、规则、渲染、持久化和展示逻辑。
- 让质量命令闭环，任何交付前都能用同一套门禁验证。
- 让 mock、sandbox、fallback、真实适配器、离线评测的边界在文档中清楚可见。
- 让仓库卫生接近交付快照，不把本地缓存、日志、数据库、覆盖率和运行产物混进项目叙事。

## 2. 冻结范围

后续默认不做以下事项：

- 不新增业务接口。
- 不新增 Agent 工具。
- 不新增前端页面或新的工作台能力。
- 不新增外部系统适配器。
- 不把本地 demo 或 mock/fallback 包装成真实生产接入。
- 不引入自动生产写操作。
- 不为“看起来更完整”而做大型平台化包装。

允许的改动：

- 等价重构和职责拆分。
- 类型注解、格式化、lint、安全扫描和测试门禁收敛。
- 文档结构调整、边界说明、运行说明和交付检查清单。
- 测试和评测稳定性修复。
- 配置默认值、安全提示和仓库卫生规则收敛。
- CI / Dockerfile / 发布说明等工程交付能力，前提是不改变业务能力。

## 3. 交付门禁

每次准备提交或交付前，优先运行：

```bash
make check-all
```

如果需要拆开定位问题，按顺序运行：

```bash
make lint
make type-check
make security
make test-quick
make eval
make eval-rag
make eval-change
make hygiene-check
```

门禁解释：

- `make lint`：Ruff 代码规范检查。
- `make type-check`：mypy 类型检查。
- `make security`：Bandit 中高风险安全扫描。
- `make test-quick`：仓库级 pytest 快速验证。
- `make eval`：AIOps + RAG 离线综合评测。
- `make eval-rag`：RAG 检索、引用和拒答评测。
- `make eval-change`：审批、安全变更、dry-run、rollback 评测。
- `make hygiene-check`：检查可能误提交的本地产物。

当前本地基线：

- `make check-all` 通过。
- pytest：315 passed，1 skipped。
- 覆盖率：81.05%。
- AIOps 离线评测：38/38。
- RAG 评测：22/22。
- 安全变更评测：9/9。
- 类型检查：`Success: no issues found`。
- 安全扫描：未发现中高风险问题。
- 仓库卫生：未发现未忽略生成物。
- CI 入口：`.github/workflows/quality.yml`，按非修改型顺序运行 lint、type-check、security、test、eval 和 hygiene-check。

前端浏览器 smoke 测试在部分受限环境下可能因无法绑定 localhost 端口而跳过；交付前应在可开放端口的本机环境补测。

## 4. 模块边界

### API 层

API 层只负责：

- 请求参数校验。
- 权限和角色边界。
- 调用 service。
- 返回响应或 SSE 事件。

API 层不应继续堆积：

- 读模型拼装细节。
- 复杂状态机。
- 诊断规则。
- Markdown 渲染。
- 工具执行逻辑。

### Agent 层

Agent 层只负责诊断流程：

- Planner 生成结构化计划。
- Executor 调用工具并形成证据。
- Replanner 决定继续、补查、审批、报告或升级人工。

Agent 层不应绕过：

- Tool Registry。
- Risk Controller。
- 审批和安全变更边界。
- Evidence / Trace / Report 的结构化记录。

### Service 层

Service 层承担业务编排，但应继续控制单文件职责：

- RAG 回答策略独立于 Agent 生命周期。
- AIOps SSE 事件格式化独立于图执行。
- 报告质量计算独立于报告渲染。
- 安全变更规则独立于持久化和事件记录。
- 诊断规则目录独立于规则判断逻辑。

### Integrations 层

外部适配器只负责外部系统协议和数据归一：

- 不直接生成业务结论。
- 不越过风险审批。
- 不隐藏 mock/fallback 数据源。
- 失败时返回可诊断的错误类型和摘要。

## 5. 安全和可信边界

必须持续保留：

- 无可信 RAG 来源时拒答。
- 成功 RAG 回答必须带 `source_file + chunk_id`。
- 高风险动作进入审批。
- forbidden 动作直接阻断。
- 审批通过后仍不自动执行生产写操作。
- mock/fallback 数据源在 Evidence、ToolCall、Report 中可见。
- 生产或严格验收默认关闭 `AIOPS_MOCK_FALLBACK_ENABLED`。
- 外部原始 payload 默认不完整保存。

不应在文档或演示中声称：

- 已接入真实生产全链路。
- 已具备线上根因识别准确率保证。
- Agent 可以自动修复生产故障。
- 离线评测通过率等于线上准确率。
- 本地 sandbox 等于生产环境。

## 6. 仓库卫生

不应提交：

- `.env`
- API Key 或任何密钥。
- `venv/`
- `logs/`
- `data/*.db*`
- `uploads/`
- `htmlcov/`
- `.coverage`
- `.pytest_cache/`
- `.ruff_cache/`
- `__pycache__/`
- `.DS_Store`
- Milvus / Docker 数据卷。

常规检查：

```bash
make hygiene-check
```

如需审计所有本地产物：

```bash
python scripts/maintenance/hygiene_check.py --include-ignored
```

`--include-ignored` 是审计模式，可能会列出被 `.gitignore` 正确忽略的本地运行产物；默认模式用于交付门禁。

## 7. 后续改进顺序

推荐顺序：

1. 复核当前 diff，确认没有混入新功能。
2. 保持 README 精简，把演示、分析、生产说明分流到独立文档。
3. 维护 CI，使用现有质量命令作为合并门禁。
4. 继续拆 `executor.py`、`app/api/aiops.py` 等剩余偏厚文件。
5. 补标准 Dockerfile 或部署检查清单。
6. 只在门禁保持全绿时做下一轮重构。

暂不建议：

- 继续新增 demo case。
- 接更多外部系统。
- 重写前端工作台。
- 引入复杂插件体系。
- 大规模替换框架。

## 8. 提交前检查清单

提交前确认：

- 改动只属于治理、重构、测试、文档、配置或门禁。
- 没有新增业务能力或对外 API 行为。
- `make check-all` 通过。
- `make hygiene-check` 通过。
- README 和相关文档没有互相矛盾。
- mock、sandbox、fallback、生产边界描述一致。
- 新增模块有清晰职责，旧调用路径没有被无意破坏。
- 大文件拆分后没有让业务流变得更难追踪。

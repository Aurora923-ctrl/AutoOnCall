# 负例边界

这些案例用于面试讲解和回归设计，说明 AutoOnCall 在证据不完整时不会强行输出看似确定的根因分析结论。

## 缺少运行手册：降级为 `needs_human`

场景：

- 已获取主故障域证据。
- 已获取现象侧证据。
- 运行手册检索没有可信命中，历史工单也没有匹配结果。

预期行为：

- 核心事实可信但缺少处置参考时，报告状态为 `needs_human`。
- 报告列出缺失证据：可信运行手册或历史工单参考。
- 报告将已确认事实与推断性根因分开呈现。
- 报告限制置信度，并建议人工确认后再采取处置动作。

面试表述：

```text
系统仍能总结 Redis、MySQL 或 K8s 证据表达的事实，但不会把没有依据的处置建议包装成结论。
由于缺少可操作参考，报告会明确标记为需要人工复核。
```

## K8s RBAC 被拒绝：降级为 `degraded`

场景：

- `query_k8s_status` 因 Kubernetes API 拒绝 Pod 或事件访问而失败。
- 日志或指标仍可能显示重启、疑似 OOM 或可用性下降。
- 无法取得 Pod 状态或事件证据。

预期行为：

- 报告状态为 `degraded`，而不是 `completed`。
- 报告和评测摘要中会记录失败工具。
- 指标和日志只作为现象证据，不能替代 K8s 主故障域证据。
- 报告限制置信度，并建议人工检查 Pod 事件、上一次容器状态、内存限制和最近发布。

面试表述：

```text
指标可以说明影响，但不能证明 Pod 事件。K8s 工具失败时，报告会降级并请求人工核验，
而不是编造实时 K8s 根因分析。
```

## 为什么重要

这些负例是回答“为什么可以信任这个 Agent”的关键：可信不是任何时候都给出答案，而是能在证据不足时明确承认不确定性。

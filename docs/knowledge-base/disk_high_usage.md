# 磁盘与 inode 使用率过高告警处理方案

## 适用范围

- 告警：`HighDiskUsage`、`InodeFull`、`DiskReadOnly`
- 适用对象：主机文件系统、容器节点、日志盘、临时目录和数据库数据盘
- Owner：运行平台；数据库数据盘同时升级 DBA

容量阈值应结合增长速度和预计耗尽时间。地域、节点和目录从 Incident 上下文读取。

## 首轮证据

1. 固定时间窗口，记录文件系统、挂载点、容量、inode 和只读状态。
2. 查询使用率增长斜率、预计耗尽时间、写入错误和受影响服务。
3. 定位大目录、大文件、已删除但仍被进程占用的文件和高 inode 目录。
4. 区分日志、临时文件、镜像、备份、业务数据和数据库文件。
5. 只执行读取命令，不在诊断阶段直接删除或截断。

```bash
df -h
df -i
du -x -h --max-depth=2 <mount-point>
find <mount-point> -xdev -type f -size +1G
lsof +L1
docker system df
```

## 原因判别

### 日志与临时文件

检查日志轮转失败、DEBUG 级别、core dump、上传临时文件和任务中间产物。候选文件需要确认
进程占用、保留策略和审计要求。

### Docker 与容器日志

检查镜像、停止容器、构建缓存、匿名卷和容器日志。`docker system prune` 可能删除仍需回滚的
镜像或数据卷，不能作为自动化诊断动作。

### inode 耗尽

空间未满但无法创建文件时，检查 inode 使用率和小文件数量。常见来源包括会话文件、缓存、
分片日志和未清理的临时目录。

### 数据与备份增长

数据库文件、WAL/binlog、快照和本地备份增长时，升级对应 Owner。删除日志或备份前必须确认
恢复点、复制状态和合规保留要求。

### 已删除文件仍占用

文件被删除但进程仍持有句柄时，`lsof +L1` 可定位占用。释放句柄通常需要应用重载或重启，
属于生产变更。

## 处置计划与审批

先估算剩余时间并生成候选清单，再选择压缩、转移、日志轮转修复、扩容或受控清理。所有删除、
截断、Docker 清理、扩容和数据保留策略变更都需要人工审批、备份确认和回滚方案。

紧急情况下优先转移明确可重建的产物或扩容，避免在证据不足时删除业务数据。

## 回滚条件

- 清理或轮转后应用无法写日志或读取依赖文件；
- 数据库复制、恢复点或备份校验异常；
- Docker 清理影响部署回滚能力；
- 扩容后文件系统、挂载或 IO 延迟异常。

## 恢复验证

- 空间和 inode 回到容量基线，预计耗尽时间恢复；
- 应用、日志和数据库写入正常；
- 无只读文件系统或新的 IO 错误；
- 候选文件、执行动作、审批和备份位置已记录；
- 观察增长斜率，确认问题没有继续累积。

## 长期行动

- 为容量、inode 和预计耗尽时间分别告警；
- 验证 logrotate、容器日志上限和临时文件生命周期；
- 将备份转移到专用存储并定期做恢复演练；
- 对大目录建立 Owner 和容量预算；
- 在发布检查中加入磁盘与 inode headroom。

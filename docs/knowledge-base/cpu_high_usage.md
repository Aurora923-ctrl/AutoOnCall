# CPU 使用率过高告警处理方案

## 适用范围

- 告警：`HighCPUUsage`
- 适用对象：虚拟机、容器、Kubernetes 工作负载和应用进程
- 典型条件：CPU 持续超过容量基线，且伴随延迟、错误率、队列或负载上升
- Owner：运行平台与对应服务团队

本文用于诊断 CPU 压力，不把“CPU 高”直接等同于根因。变更地域、集群和日志主题均从
Incident 上下文读取，不使用固定环境值。

## 首轮证据

1. 用 `get_current_time` 固定 Incident 时间窗口。
2. 查询 CPU 使用率、load average、throttling、请求量、P95/P99 和错误率。
3. 按实例、Pod、进程、线程和发布版本拆分，确认是单点还是全局升高。
4. 查询同一窗口的应用错误、重复堆栈、超时、重试和慢查询。
5. 保存进程 PID、容器 ID、线程栈、火焰图或 profiler 摘要。

查询条件示例：

```text
time_range: incident_start-15m .. now
metrics: cpu_usage, cpu_throttled_seconds, load_average, request_rate, error_rate
logs: level:(ERROR OR WARN) AND service_name:<incident-service>
```

## 原因判别

### 单进程热点或死循环

特征包括单进程接近满核、重复堆栈、热点方法集中，且流量没有同比增长。先采集线程栈和
profiling 证据，再判断死循环、无限递归、锁竞争或忙等待。不要在保留现场前直接重启。

### 流量或重试放大

多个实例 CPU 与请求量同步升高时，检查营销活动、上游重试、爬虫或异常调用方。需要同时
观察吞吐、拒绝率和依赖容量，避免扩容后把压力转移到数据库或 Redis。

### 定时任务重叠

CPU 周期性升高且与作业开始时间一致时，检查任务锁、执行时长、补偿任务和并发配置。调整
调度时间或互斥策略属于生产配置变更。

### 数据库慢查询或依赖等待

CPU 升高但火焰图没有明显业务热点，且慢查询、慢SQL、慢 SQL、数据库连接池等待或重试增加时，应把数据库或下游
依赖作为主方向。CPU 可能只是序列化、轮询或重试带来的伴随现象。

## 处置计划与审批

先生成包含证据、预期收益、风险、canary 范围、回滚条件和观察时长的变更计划。扩容、限流、
回滚、终止进程、调整线程池或重启实例均需人工审批。AutoOnCall 只提供诊断、dry-run 和审批
记录，不自动执行生产写操作。

建议优先采用可逆的最小变更，例如单实例摘流、10% canary、降低重试并发或回滚最近版本。

## 回滚条件

出现下列任一情况应停止或回滚变更：

- 5xx、P95/P99 或请求丢弃率继续上升；
- 下游数据库、缓存或消息队列开始接近容量上限；
- CPU throttling、重启次数或可用实例数恶化；
- canary 与对照组没有可解释的改善。

## 恢复验证

- CPU 回到服务容量基线，而不是使用统一的绝对阈值；
- P95/P99、错误率和队列长度恢复；
- 无新的重复堆栈、重试放大或依赖告警；
- canary 和全量阶段均达到预设观察时长；
- Incident 记录包含执行人、审批人、证据链接和后续行动项。

## 长期行动

- 为热点路径建立 profiling 基线和回归压测；
- 将 CPU、throttling、请求量和依赖等待放在同一看板；
- 对定时任务增加互斥、超时和错峰策略；
- 对重试设置预算、指数退避和抖动；
- 定期复核自动扩缩容上下限和下游容量。

# 线程池耗尽与请求排队 Runbook

## 适用范围

适用于 active threads 达上限、queue depth 增长、task rejected、请求堆积和线程长期阻塞。
Owner 为服务团队与运行平台。

## 首轮证据

- 查询 active、pool size、queue depth、rejected tasks 和任务耗时；
- 获取线程栈，按锁、IO、数据库、外部 API 和 CPU 热点分类；
- 关联请求量、超时、重试、连接池等待和最近发布；
- 区分业务线程池、异步执行器、HTTP worker 和调度线程池。

## 判断路径

线程阻塞在数据库或外部 API 时，优先修复依赖等待。CPU 满且 runnable 线程多时转入 CPU
Runbook。线程数不高但队列增长时，检查单任务耗时和串行锁。不要把扩大线程池当作默认修复，
它可能放大数据库或下游并发。

## 处置与审批

扩大线程池、调整队列、超时、并发、限流或重启均需人工审批。计划必须说明下游容量、canary
范围、拒绝策略和回滚条件。

## 回滚与恢复

若下游连接、CPU、内存、错误率或上下文切换恶化，应回滚。恢复要求队列清空、拒绝率归零、
线程栈不再集中阻塞，P95/P99 回到 SLO。

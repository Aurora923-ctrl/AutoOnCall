# AutoOnCall 秋招后续改进方向

## 一、文档定位

本文记录 AutoOnCall 面向大模型应用开发、AI Agent、RAG 和 AIOps 岗位的后续改进方向。

这些方向来自两部分：

1. AutoOnCall 当前业务闭环中仍可加强的环节；
2. 对行业信息助手的 Deep Research、数据分析、知识管理、任务恢复、前端呈现和基础设施能力进行筛选后，适合迁移到智能运维场景的部分。

本文只讨论规划，不代表相关能力已经实现，也不能作为当前项目能力或生产化程度的声明。

改进必须继续服务于以下主线：

```text
告警进入
  -> 形成 Incident
  -> 理解影响和故障窗口
  -> 制定调查计划
  -> 调用受控工具取证
  -> 建立并验证根因假设
  -> 进行风险判断和人工审批
  -> 输出可审计报告
  -> 验证恢复结果
  -> 复盘并沉淀经验
```

## 二、排序方法

本文不按功能新颖程度排序，而按“对秋招项目含金量的综合贡献”排序。 

| 评价维度 | 说明 |
| --- | --- |
| 业务贴合度 | 是否直接解决 Incident 诊断、处置或复盘问题 |
| 大模型岗位相关性 | 是否体现 Agent、RAG、评测、工具调用或模型工程能力 |
| 技术深度 | 是否涉及数据模型、算法、状态机、可靠性或安全边界 |
| 可演示性 | 面试官能否快速看懂改进带来的价值 |
| 可评测性 | 是否可以构建正例、负例、失败边界和量化指标 |
| 实施投入产出比 | 是否能复用现有 Evidence、Trace、Store、RAG 和工作台 |

优先级含义：

- **S 级**：最能强化项目核心叙事，应优先形成一条完整升级主线。
- **A 级**：对主线有明显增强，适合在 S 级能力稳定后实施。
- **B 级**：工程化或产品化补强，有价值但不是秋招前最核心的差异点。
- **暂缓**：当前投入产出比低、偏离主线或会引入不必要风险。

## 三、改进原则

### 3.1 不为了技术栈而改造

每项新增能力至少应回答一个真实运维问题：

- 能否更快理解事故发生过程？
- 能否提高根因判断的可验证性？
- 能否减少无效或重复的工具调用？
- 能否展示结论的证据来源和不确定性？
- 能否帮助值班人员做出更安全的处置决策？
- 能否让历史事故真正服务于下一次诊断？

不能回答这些问题的功能，不进入主线。

### 3.2 不照搬通用 Deep Research

行业信息助手值得借鉴的是任务拆解、并行研究、数据分析、来源追溯、检查点、知识管理和结果呈现，而不是新闻、股票、招投标、通用 Text2SQL、任意代码执行或多个角色型 Agent。

AutoOnCall 中的研究行为必须围绕 Incident、运维信号、根因假设、处置风险和恢复验证展开。

### 3.3 保持确定性控制边界

大模型适合负责：

- 生成调查计划和候选假设；
- 解释证据关系；
- 选择需要补查的方向；
- 总结调查过程；
- 生成受结构约束的报告文本。

后端确定性逻辑继续负责：

- 数值计算和时序分析；
- 工具参数及返回契约；
- 身份、权限和风险策略；
- 状态迁移和审批约束；
- Evidence、Trace、Report 和 Artifact 持久化；
- 生产写操作边界；
- 评测和 provenance。

### 3.4 每项能力必须可评测

实施前必须明确：

- 输入与输出契约；
- 正例、负例和失败状态；
- `offline_fixture`、`local_live` 或 `controlled_fault` 证据等级；
- 可复跑的评测入口；
- 人工接管路径；
- 不能对外声称的范围。

### 3.5 训练只能优化诊断决策，不能替代知识、事实和安全边界

微调适合用于学习相对稳定、可标注、可离线评测的诊断决策模式，例如工具规划、根因分类与排序、补查或停止决策。以下内容不能因为引入训练而交给模型记忆或自由决定：

- 实时运维知识和 Runbook 继续通过 RAG 更新；
- Prometheus、日志、Redis、MySQL、Kubernetes 等现场事实继续来自工具取证；
- 权限、审批、危险动作和生产写操作继续由确定性策略控制；
- Evidence Analyzer、工具白名单、风险门禁和人工接管继续保留；
- 训练集与回归评测集必须隔离，不能用记住 fixture 的结果冒充泛化能力。

因此，模型训练在本项目中的定位是“反馈驱动的诊断策略优化”，而不是为了简历标签给项目附加强化学习。

## 五、逐项分析

## 5.1 排名 1：根因假设竞争与正反证据

### 前因

AutoOnCall 已有 Planner、Executor、Evidence Analyzer、Replanner 和 Evidence Graph，但如果最终仍主要围绕一个 `root_cause` 展示，面试官不容易看出：

- Agent 如何从现象形成候选根因；
- 为什么选择当前根因；
- 其他可能性如何被排除；
- 哪些结论仍然缺少证据。

行业信息助手的研究状态中已经包含假设、支持证据、反驳证据和后续追查思路，这种机制非常适合迁移到 RCA。

### 改进内容

为每个候选根因建立结构化模型：

| 字段 | 含义 |
| --- | --- |
| `hypothesis_id` | 假设标识 |
| `statement` | 可验证的根因描述 |
| `supporting_evidence_ids` | 支持证据 |
| `contradicting_evidence_ids` | 反对证据 |
| `missing_evidence` | 缺失证据 |
| `status` | proposed、investigating、supported、rejected、inconclusive |
| `confidence_cap` | 由来源质量和证据充分性决定的置信度上限 |
| `next_probe` | 最能区分竞争假设的下一步调查 |

Replanner 从“是否继续”升级为：

- 当前哪个假设最值得验证；
- 哪项工具调用具有最高区分价值；
- 是否已存在足够反证；
- 是否满足停止调查条件；
- 是否应进入 `needs_human`。

### 后果

正向结果：

- RCA 过程变得可解释；
- Replan 有明确业务目标；
- 可以评价假设覆盖率、错误排除率和证据闭环率；
- 前端可以直观展示 Agent 不是在猜答案。

新增复杂度：

- 需要定义假设状态迁移；
- 需要处理多个假设同时成立的场景；
- 不能把模型自报置信度直接当作可信概率。

### 验收

- completed 根因必须绑定支持证据；
- rejected 假设必须保留反证或排除理由；
- 证据不足时稳定输出 `inconclusive`；
- 构建“相似症状、不同根因”的竞争假设评测集。

## 5.2 排名 2：确定性运维时序分析器

### 前因

行业信息助手具备数据提取、趋势分析、异常值识别和图表生成能力。但在运维场景中，趋势、峰值、持续时间和变化率不能交给 LLM 自由计算，否则容易出现数值错误和错误相关性。

AutoOnCall 已经能采集 Prometheus、日志、Redis、MySQL 和变更信息，下一步更有价值的是增加一个确定性分析层。

### 改进内容

构建 `SignalAnalyzer` 或同类服务，负责：

- 计算基线、当前值、变化率和偏离程度；
- 检测异常起点、峰值、恢复点和持续时间；
- 比较发布前后、故障前后和依赖之间的信号；
- 计算变更事件与异常起点的时间差；
- 对齐延迟、错误率、连接数、慢查询和资源饱和度；
- 输出结构化 `SignalFinding`，再交给 LLM 解释。

示例：

```json
{
  "signal": "mysql_slow_queries",
  "baseline": 12,
  "peak": 186,
  "change_ratio": 14.5,
  "anomaly_started_at": "2026-07-21T10:06:00+08:00",
  "related_change_delta_seconds": 240,
  "finding": "slow_queries_spiked_after_deployment",
  "calculation_source": "deterministic"
}
```

### 后果

正向结果：

- 时间线和根因假设获得可信数值基础；
- LLM 从“计算并判断”收敛为“解释结构化发现”；
- 可构建时序分析单元测试和 controlled fault 验证；
- 面试中能体现 AI 与传统可观测性算法的合理分工。

新增复杂度：

- 需要处理采样频率、缺失点、时区和不同数据源窗口；
- 简单阈值不能包装成完整异常检测平台；
- 必须避免把时间相关性自动解释为因果关系。

### 验收

- 固定序列下异常起点、峰值和恢复点结果确定；
- 缺失、乱序和重复采样有明确处理策略；
- 分析结果记录输入窗口、算法版本和计算来源；
- LLM 输出不得修改确定性计算字段。

## 5.3 排名 3：统一事故调查时间线

### 前因

当前告警、指标、日志、Trace、运行态、发布、工单、工具调用、审批和恢复记录分散在不同模型和页面中。值班人员需要自己判断这些信息是否位于同一事故窗口。

第 2 项时序分析器解决“信号如何计算”，本项解决“所有事件如何组织和解释”。

### 改进内容

建立统一 `IncidentTimelineEvent`，至少包含：

- 事件时间和采集时间；
- 来源系统；
- 事件类型；
- 所属调查阶段；
- 关联 Evidence、Trace、ToolCall、Change 或 Approval；
- 当前事实等级；
- 是否来自历史回放、当前实时数据或 mock；
- 时间质量和冲突状态。

时间线统一展示：

```text
10:02  发布 payment-service v2.3
10:05  P95 延迟开始升高
10:06  MySQL 慢查询数量上升
10:08  告警触发
10:09  Agent 查询 Prometheus、Loki 和 MySQL
10:11  候选假设更新为“慢查询导致连接池等待”
10:14  进入人工审批
10:19  延迟恢复并完成观察窗口验证
```

### 后果

正向结果：

- 多源信息形成一条事故叙事；
- 可以从结论反向定位到事件和证据；
- Replay、报告和前端共享同一读模型；
- 面试演示更容易在一分钟内讲清。

新增复杂度：

- 需要统一时间格式和事件语义；
- 同一事件可能被多个来源重复记录；
- 历史 Incident 窗口与当前实时状态必须显式区分。

### 验收

- 多源事件按同一时间窗口排序；
- 保留 source timestamp 和 collected timestamp；
- 时间冲突、缺失和延迟不会被静默修正；
- 每个重要事件可回链到原始 Artifact。

## 5.4 排名 4：动态并行取证与信息增益调度

### 前因

行业信息助手会并行执行多个研究子任务。AutoOnCall 已具备工具并发和取消能力，但仍可进一步把并发从“批量执行步骤”提升为“围绕竞争假设动态选择调查分支”。

串行调查耗时长，固定调用所有工具又会产生噪声、上下文膨胀和外部系统压力。

### 改进内容

按信号域组织可选调查分支：

```text
Incident
  |-- Metrics：Prometheus、SLO、容量和饱和度
  |-- Logs/Trace：错误模式、调用链和异常传播
  |-- Runtime：Kubernetes、Redis、MySQL 和消息队列
  |-- Change：发布、配置、CMDB 和工单
  `-- Knowledge：Runbook、历史 Incident 和官方资料
```

Planner 和 Replanner 根据以下条件选择分支：

- 当前假设缺少什么证据；
- 哪个工具最能区分两个竞争假设；
- 调用成本和预期收益；
- 是否已有等价查询结果；
- 剩余时间、工具和 Token 预算。

### 后果

正向结果：

- 复杂事故延迟可能下降；
- 工具调用更有目的性；
- 可以评价工具选择 F1、无效调用数和信息增益；
- 为 Deep Investigation 提供执行基础。

新增复杂度：

- 并发取消、部分成功和状态提交更复杂；
- 需要限制最大并发和外部系统压力；
- 汇总阶段必须保留失败分支和反证。

### 验收

- 与串行基线比较 P50/P95、调用数和有效 Evidence 数；
- 重复查询能够去重；
- 分支失败或取消留下真实 Trace；
- session、Incident 和 Evidence 不发生串扰。

## 5.5 排名 5：反馈驱动的诊断策略微调

### 前因

AutoOnCall 已经具备微调所需要的数据飞轮雏形：Incident、PlanStep、工具调用、Evidence、Trace、根因结论、`root_cause_correct`、`accepted_suggestion`、`operator_note` 以及坏例进入 eval backlog 的回流机制。

但现有 `eval/cases.yaml` 主要属于 `offline_fixture` 和 `development` 数据，部分根因类别只有少量模板变体。这些样本适合验证训练管线、结构化输出和 LoRA 概念验证，不足以证明模型形成了可泛化的生产诊断能力。如果直接训练最终报告，项目很容易变成“模型更会写”，而不是“诊断决策更可靠”。

因此，只有在前 1 至 4 项把假设、证据、工具选择和停止决策结构化后，微调才会获得清晰标签，并成为大模型开发岗位中很有辨识度的模型工程能力。

### 改进内容

优先对可本地部署的小模型进行 LoRA/SFT，训练对象按业务价值排序：

1. **Planner 工具规划**
   - 输入：Incident、Runbook 摘要、可用工具及工具约束；
   - 输出：结构化 `PlanStep`；
   - 目标：减少无关工具，覆盖必要工具，并安排合理的取证顺序。
2. **根因分类与候选排序**
   - 输入：Incident、结构化 `SignalFinding` 和 Evidence；
   - 输出：根因类别、Top-K 假设、支持及反驳 Evidence ID、缺失证据；
   - 目标：提高根因排序质量和证据回链完整性。
3. **Replanner 有限动作决策**
   - 输出限制为 `add_steps`、`retry_failed_tool`、`generate_report`、`request_approval`、`needs_human` 等有限动作；
   - 目标：学习何时补查、重试、停止或转人工；
   - 确定性安全兜底、Evidence Analyzer、工具白名单和风险门禁不得移除。
4. **报告表达微调**
   - 仅优化事实、推断、不确定性、补查建议和引用的表达一致性；
   - 优先级低于 Planner、RCA 和 Replanner，因为报告流畅不等于诊断正确。

训练数据不只保存最终报告，而应形成阶段级样本：

```text
Incident + Runbook + Tool Catalog -> PlanStep
Incident + SignalFinding + Evidence -> RootCauseCandidate Top-K
Current State + Missing Evidence + Budget -> ReplannerAction
Diagnosis + Evidence + Approval State -> Structured Report
```

正式微调前建议至少准备 500 至 1,000 条人工复核的真实或受控 Incident，并保证：

- 每个主要根因类别具有正例、反例、冲突证据和证据不足案例；
- 按时间、服务或故障族划分训练集与 holdout；
- 当前回归集不同时充当训练集；
- 保存数据版本、Prompt 版本、基础模型、LoRA 参数和训练 provenance；
- 训练模型与现有大模型、规则基线在同一离线集上对照。

### 后果

正向结果：

- 项目从“调用现成大模型”升级为“基于业务反馈优化诊断策略”；
- 能完整展示数据构建、LoRA/SFT、结构化输出、离线评测、模型路由和安全兜底；
- 小模型可以承担高频结构化决策，形成质量、延迟和成本对照；
- 工具选择 F1、根因 Top-K、证据引用和停止决策都可以量化。

新增风险：

- 少量 fixture 容易造成模板记忆和数据泄漏；
- 人工反馈可能存在错标、事后偏差和类别不平衡；
- 微调模型可能学会过早停止、少查工具或复制历史根因；
- 训练后的模型仍可能发生契约漂移，不能绕过确定性验证；
- GPU、训练框架和模型部署会增加工程范围，不能抢在诊断数据结构之前建设。

### 验收

- 与未微调模型、规则基线进行盲测，不只展示训练 loss；
- Planner 评价必要工具召回率、工具选择 F1、无价值调用数和顺序正确率；
- RCA 评价根因 Top-1/Top-K、证据引用准确率、反证覆盖率和无依据结论率；
- Replanner 评价正确停止、正确转人工、无效重试和审批绕过率；
- holdout 按服务或时间隔离，并报告每个主要故障类别的结果；
- 未达到预设提升阈值时不接入主链路，可保留 shadow 或离线模式；
- 安全策略、工具权限和生产写操作不进入模型自治范围。

## 5.6 排名 6：相似事故记忆与差异分析

### 前因

行业信息助手的长期记忆可以跨会话召回摘要和洞察，但用户偏好记忆不适合 AutoOnCall。真正有业务价值的是把已确认事故沉淀为 `Incident Memory`。

普通 RAG 只返回历史工单文本，容易诱导模型复制旧根因。

### 改进内容

事故关闭并通过人工确认后，沉淀：

- 故障指纹；
- 服务、环境和依赖；
- 症状和异常信号；
- 已确认根因与促成因素；
- 有效及无效调查步骤；
- 处置和恢复验证；
- 人工反馈；
- 关联 Runbook、发布和工单。

召回结果必须包含共同点和差异点：

```text
相似事故：INC-2026-018
共同点：发布后连接数持续升高、payment-service 超时
差异点：历史事故出现 rejected_connections，当前尚未出现
可复用经验：优先检查连接释放和客户端来源
不可直接复用：当前证据不足以确认 maxclients 耗尽
```

### 后果

正向结果：

- 历史 Incident 进入下一次诊断闭环；
- RAG 从知识问答升级为经验复用；
- 能形成“反馈 -> 记忆 -> 新事故 -> 再评测”的学习链。

新增风险：

- 错误历史结论可能污染当前诊断；
- 症状相似不代表根因相同；
- 需要定义高可信事故的晋级和撤回规则。

### 验收

- 相似事故 Top-K 召回；
- 区分同症状不同根因；
- 错误复制历史根因率；
- 只有人工确认且引用完整的事故进入高可信记忆。

## 5.7 排名 7：复杂事故 Deep Investigation

### 前因

简单 Redis 或 MySQL 故障适合快速诊断；跨服务、证据冲突或第一轮无结果的事故需要更长调查链。行业信息助手的 Deep Research 循环可以借鉴，但不能成为默认路径。

### 改进内容

保留快速诊断，仅在以下条件触发深度调查：

- P0/P1 严重事故；
- 涉及多个服务或依赖；
- 第一轮关键工具未命中；
- 指标、日志和运行态相互矛盾；
- 多个假设接近；
- 操作员手动升级。

流程：

```text
快速诊断
  -> 证据不足或假设冲突
  -> 拆分调查问题
  -> 动态并行取证
  -> 更新假设和反证
  -> 证据覆盖审查
  -> completed / incomplete / needs_human
```

### 后果

正向结果：

- 展示复杂 Agent 工作流；
- 简单事故不承担额外模型成本；
- 能主动说明预算耗尽和人工补查项。

新增风险：

- 多轮循环放大延迟、Token 和失败概率；
- 如果没有前 1 至 4 项支撑，容易退化为多 Agent 对话表演。

### 验收

- 明确最大轮数、工具数、时长和 Token；
- 达到预算后输出缺失证据及人工补查清单；
- 简单故障不得误触发深度模式；
- 跨服务 case 能体现相较快速模式的新增有效证据。

## 5.8 排名 8：Runbook 知识治理工作台

### 前因

行业信息助手提供知识库、文档处理状态和 Chunk 查看页面。AutoOnCall 的知识入库、质量记录、混合检索、Citation 和评测更强，但这些能力主要隐藏在后端、脚本和 Artifact 中。

### 改进内容

建立面向运维知识管理员的只读或受控工作台：

- Runbook 和复盘文档列表；
- pending、processing、completed、failed、stale 状态；
- Chunk 内容、标题路径和 metadata；
- 来源类型、权威等级和更新时间；
- 索引版本与 Embedding 配置；
- 检索预览及 Trust Gate 结果；
- 被哪些 Incident 和报告引用；
- 哪些反馈暴露知识缺口；
- 重建索引和验证任务状态。

### 后果

正向结果：

- 将 RAG 工程能力可视化；
- 面试官可以检查“为什么召回这段内容”；
- 知识质量、运行时检索和反馈缺口形成闭环。

新增复杂度：

- 需要控制重建索引和删除文档权限；
- 页面状态必须来源于真实 indexing record，不能由前端推断；
- 不能为了页面方便绕过现有上传和索引安全检查。

### 验收

- 文档状态与索引实际状态一致；
- Chunk 可回链到源文件和索引身份；
- stale、失败和跳过 Milvus 状态不能显示为完成；
- 检索预览复用真实 retrieval service。

## 5.9 排名 9：Incident 附件证据管道

### 前因

行业信息助手支持会话附件上传和处理状态。真实值班过程中，人工经常拿到监控截图、脱敏日志、SQL 执行计划、变更单和厂商通知，这些材料目前不一定能自然进入 AutoOnCall 的证据链。

### 改进内容

允许操作员向指定 Incident 上传：

- 脱敏日志；
- 指标导出或监控截图；
- SQL `EXPLAIN`；
- 发布记录和配置 diff；
- 厂商公告；
- 人工排查记录。

附件处理流程：

```text
上传
  -> 文件安全校验
  -> 脱敏与解析
  -> pending / processing / completed / failed
  -> 生成独立 Evidence
  -> 人工确认后参与假设验证
```

### 后果

正向结果：

- 人工现场信息进入统一 Evidence；
- 人机协作不再局限于聊天文本；
- 可以展示文档解析、异步任务和证据治理。

新增风险：

- 敏感信息、恶意文件和超大文件风险；
- 截图需要 OCR 时可能引入额外服务和误识别；
- 附件不能直接整段拼入 Prompt。

### 验收

- 保存上传者、时间、hash、解析状态和证据等级；
- 文件类型、大小、路径和内容经过现有安全策略；
- 解析失败不产生虚假 Evidence；
- 附件结论需要人工确认或明确标记未验证。

## 5.10 排名 10：报告质量审查与事故复盘闭环

### 前因

行业信息助手使用 Critic 检查缺少来源、逻辑错误、过期信息和内容不完整。AutoOnCall 已有报告 Evidence Gate，但还可以把报告前检查与事故后复盘区分得更清楚。

### 改进内容

在报告生成前运行结构化质量审查：

- 关键结论是否有 Evidence；
- 是否遗漏失败工具和不确定性；
- 是否混淆实时数据与回放数据；
- 是否存在过期 Runbook；
- 处置建议是否具备审批和回滚边界；
- 是否存在结论与证据矛盾。

同时区分：

**值班诊断报告**

- 当前影响；
- 已确认事实；
- 领先和竞争假设；
- 未确认事项；
- 建议下一步；
- 风险和审批状态。

**事故复盘报告**

- 完整时间线；
- 根因和促成因素；
- 有效及无效调查步骤；
- 恢复验证；
- 监控、工具和 Runbook 缺口；
- 后续行动项和负责人。

### 后果

正向结果：

- 报告从“模型写得完整”升级为“满足业务交付契约”；
- 低质量报告可以回流成 eval backlog；
- 复盘为 Incident Memory 提供高质量输入。

新增风险：

- 不应使用一个 LLM 自评分覆盖确定性门禁；
- Critic 不能输出或展示模型私有思维；
- 行动项不能被模型自动标记完成。

### 验收

- 审查使用分项 rubric，不使用统一总分决定通过；
- 每个问题绑定字段、证据或规则；
- 修订后重新运行同一门禁；
- 复盘能够生成知识、工具或评测缺口。

## 5.11 排名 11：分任务模型路由与 Token 观测

### 前因

行业信息助手为规划、搜索、分析、审核和写作配置不同模型。AutoOnCall 当前已有模型和延迟观测基础，但 AIOps 各阶段 Token、重试和模型身份仍可以更完整。

### 改进内容

先建立统一观测，再考虑模型路由：

| 任务 | 推荐策略 |
| --- | --- |
| 意图分类、查询改写 | 确定性规则或小模型 |
| Planner | 结构化输出稳定的模型 |
| Evidence 摘要 | 低温度、较小上下文模型 |
| 复杂 Replanner | 必要时使用较强模型 |
| 最终报告 | 较强生成模型 |
| 离线 Judge | 与生成模型独立 |

每次调用记录：

- 模型和 Provider；
- Prompt 版本；
- 输入、输出和总 Token；
- 延迟、超时和重试；
- 所属阶段；
- 是否命中 fallback；
- 运行 provenance。

### 后果

正向结果：

- 可以讨论效果、延迟和成本的工程权衡；
- 小任务不必全部调用最强模型；
- 为 Deep Investigation 设置真实预算。

新增风险：

- 多模型会增加配置和回归矩阵；
- 模型差异可能造成输出契约漂移；
- 没有价格快照时不能估算金额。

### 验收

- 先在同一数据集比较质量和延迟，再切换模型；
- 路由失败可回退且留下记录；
- AIOps Token 观测覆盖 Planner、Replanner 和报告；
- 金额不可观测时保持 `not_observed`。

## 5.12 排名 12：训练数据治理与独立人工评审

### 前因

AutoOnCall 已有反馈、坏例和 eval backlog，但原始运行记录不能直接等同于训练数据，部分历史回答质量材料也属于模拟评审。微调前必须先解决标签可信度、样本独立性、数据泄漏和版本追踪问题，否则训练只会放大已有偏差。

### 改进内容

建立从运行记录到训练资产的受控管道：

```text
Trace / Evidence / Feedback
  -> 自动脱敏与候选样本生成
  -> 专家修正 Planner、RCA、Replanner 标签
  -> 双人盲评与分歧仲裁
  -> 数据质量检查
  -> train / validation / holdout 划分
  -> 版本化 Dataset Artifact
```

标签至少覆盖：

- 工具是否必要、是否遗漏、调用顺序是否合理；
- 根因类别和 Top-K 排序；
- 每个假设的支持证据、反证和缺失证据；
- 是否应继续取证、生成报告、请求审批或转人工；
- 人工最终修正、建议是否采纳以及未采纳原因。

评审协议要求：

- 隐藏模型、Prompt 和实现版本；
- 关键样本至少两名 reviewer；
- 记录分歧类型、仲裁结果和标签置信度；
- 按服务、时间和故障族去重，避免模板变体跨集合泄漏；
- 将高价值分歧样本和模型失败样本持续回流 eval backlog；
- 模拟评审、单人标注和双人复核分别标记 evidence level。

### 后果

正向结果：

- 微调和离线评测建立在可审计数据上；
- 能形成 Dataset Card、标签规范、版本对比和坏例分析；
- 真实反馈不再只是页面功能，而成为模型改进闭环；
- 为简历和面试提供边界清晰的模型工程材料。

新增风险：

- 需要人员和时间；
- 操作员“接受建议”不一定等于根因正确；
- 事故结束后的标签可能带有事后信息泄漏；
- 同一模板生成的变体可能导致虚假的高分；
- 小样本仍不能声称生产准确率；
- reviewer 不能查看会泄漏答案的内部标签。

### 验收

- 保存 reviewer、协议、标签规范、数据版本和来源运行 ID；
- 统计一致率和分歧类型；
- train、validation、holdout 无 Incident、模板或近重复泄漏；
- 抽样回链能够还原到脱敏后的 Evidence 和 Trace；
- 模拟评审与真人评审分开，结果继续标记 evidence level；
- 删除或撤回原始 Incident 时，派生数据具备可追踪的处置策略。

## 5.13 排名 13：专家偏好优化与离线规则奖励

### 前因

SFT 更适合学习“一个合格决策应该长什么样”，但多个计划可能都符合结构契约，质量差异体现在工具数量、证据区分度、停止时机和风险边界上。此时专家对两个候选计划或报告的偏好，比强行编写唯一黄金答案更自然。

不过，AutoOnCall 当前不具备经典在线强化学习条件：生产运维不能允许模型探索试错，MTTR 和是否恢复等回报存在延迟并受到人工干预，当前反馈信号也不足以构造可信的端到端奖励。因此本项只考虑离线偏好优化或基于可验证规则的训练，不做开放式在线 RL。

### 改进内容

数据成熟后，可在 SFT 基线之上逐步尝试：

1. **DPO/偏好优化**
   - 专家对两个 Planner 方案、根因排序或报告进行成对选择；
   - 偏好依据包括必要工具覆盖、无效调用、证据回链、停止决策和风险表达；
   - 优先优化 Planner 与 Replanner，不优先做纯文风偏好。
2. **基于规则奖励的 RFT**
   - 奖励结构合法、必要工具命中、证据引用正确、无越权和正确停止；
   - 惩罚无价值工具调用、无依据结论、审批绕过和把历史先验当现场事实；
   - 只使用可确定性验证或经专家确认的奖励项。
3. **离线 contextual bandit 探索**
   - 仅研究“下一项只读取证工具”的选择；
   - 必须使用历史日志进行离线估计，不在生产环境探索；
   - 只有在覆盖度和反事实评估可信时才考虑接入。

奖励必须拆分，不能只设置一个“诊断成功”：

```text
reward =
  根因排序质量
  + 必要证据覆盖
  + 工具选择精度
  + 正确停止或转人工
  + 引用与事实一致性
  - 无价值工具调用
  - 无依据结论
  - 越权或审批绕过
```

### 后果

正向结果：

- 展示从监督微调到偏好数据、奖励设计和离线策略评测的完整模型优化路径；
- 可优化“多个合法答案中哪个更好”，而不必伪造唯一标准答案；
- 能把诊断质量、成本、延迟和安全拆成可讨论的多目标权衡。

新增风险：

- 奖励设计错误会诱导模型少查工具、过早报告或回避困难案例；
- DPO 可能学习 reviewer 的表达偏好，而非真实诊断质量；
- 离线日志存在行为策略偏差，不能直接证明新策略线上更好；
- 训练复杂度明显高于 SFT，秋招前投入产出比取决于真实偏好数据规模。

### 验收

- 先证明 SFT 基线稳定，再进行 DPO/RFT 对照；
- 偏好样本记录选择理由、reviewer 一致率和数据版本；
- 分项报告工具选择、根因排序、正确停止、转人工和安全指标；
- 防止模型通过减少取证步骤刷低延迟或低成本奖励；
- 只以 shadow、离线 replay 或受控故障验证，不直接执行生产处置；
- 安全策略和审批门禁始终位于训练模型之外。

## 5.14 排名 14：受控外部运维情报检索

### 前因

行业信息助手的 Web Search 可以获取最新资料。AutoOnCall 的本地 Runbook 很可靠，但面对新 CVE、厂商故障公告、版本缺陷和云服务状态事件时可能缺少最新信息。

### 改进内容

新增严格受限的外部资料工具，只处理：

- 官方产品文档；
- CVE 和安全公告；
- 云厂商状态公告；
- Kubernetes、Redis、MySQL 等官方版本说明；
- 已配置供应商的故障通知。

结果记录：

- URL 和域名；
- 抓取时间及发布时间；
- 来源等级；
- 内容 hash；
- 与当前组件版本的适用性；
- `external_context` 证据类型。

### 后果

正向结果：

- RAG 不再完全依赖静态知识；
- 能处理最新版本问题和安全事件；
- 展示来源追溯与新鲜度治理。

新增风险：

- 搜索摘要、SEO 内容和恶意网页不可直接信任；
- 存在 SSRF、Prompt Injection 和内容过大的风险；
- 外部资料不能替代当前运行证据。

### 验收

- 官方域名优先或 allowlist；
- 抓取器限制协议、域名、重定向、大小和超时；
- 网页内容作为不可信数据隔离；
- 没有当前 Incident 证据时不得据此完成 RCA。

## 5.15 排名 15：定时知识新鲜度与适配器巡检

### 前因

行业信息助手使用 APScheduler 定时采集资讯。AutoOnCall 不需要定时抓新闻，但需要防止 Runbook 过期、外部适配器失效、证书到期或评测长期不运行。

### 改进内容

建立维护任务：

- Runbook 新鲜度检查；
- 官方资料 hash 和更新时间检查；
- Prometheus、Loki、Redis、MySQL 等适配器契约巡检；
- 知识索引一致性抽检；
- 即将到期证书或配置提醒；
- 定期生成 candidate，而不是自动晋级 official。

### 后果

正向结果：

- 发现静默失效；
- 知识治理从被动上传变成持续维护；
- 可展示任务状态、失败重试和审计记录。

新增风险：

- 进程内 APScheduler 不适合多副本严格调度；
- 定时任务不能自动修改生产系统；
- 自动评测不能自动覆盖历史 official Artifact。

### 验收

- 任务具有 run ID、状态、错误和下一次运行时间；
- 多副本下避免重复执行；
- 失败可重试但有上限；
- 任务只生成报告或候选结果。

## 5.16 排名 16：富内容事件协议与工作台交互升级

### 前因

行业信息助手通过统一内容块展示文本、图表、表格、工具动作和观察结果。AutoOnCall 当前已有丰富 SSE 和工作台，但不同事件仍可进一步收敛为统一展示协议。

### 改进内容

定义面向前端的受控内容块：

- `status`；
- `plan_step`；
- `tool_call`；
- `evidence`；
- `signal_chart`；
- `hypothesis_update`；
- `approval`；
- `report_section`；
- `warning`。

前端重点增强：

- 默认首屏突出影响、领先假设、证据充分性和下一步；
- 时间线、假设和证据互相回链；
- 固定语义的 ECharts 时序图；
- 分支 pending、running、completed、degraded、failed、cancelled 状态；
- 不展示模型内部思维过程。

### 后果

正向结果：

- SSE 事件和 UI 组件边界更稳定；
- 新能力不再依赖字符串拼接；
- 演示体验明显改善。

新增风险：

- 不能为了前端统一破坏后端业务模型；
- 任意 ECharts option 可能引入安全和展示问题；
- 整体重写 React 的收益仍然偏低。

### 验收

- 事件有版本和 Pydantic 契约；
- 前端刷新或重连后可从 progress cursor 恢复；
- 图表数据来自确定性分析器；
- 错误、mock、degraded 和实时来源视觉上可区分。

## 5.17 排名 17：Artifact 对象存储抽象

### 前因

行业信息助手通过 MinIO 支撑文件和 Milvus。AutoOnCall 当前会把大型工具输出脱敏后写入本地 Artifact，这对单机演示足够，但多实例部署和附件证据会需要统一对象存储。

### 改进内容

先定义存储接口，再支持本地文件和 S3/MinIO：

- `put_artifact`；
- `get_artifact_metadata`；
- `open_artifact`；
- `delete_artifact`；
- `verify_artifact_hash`。

适用对象：

- 大工具输出；
- Incident 附件；
- 评测报告；
- 复盘导出；
- 受控故障证明材料。

### 后果

正向结果：

- Store 只保存引用、hash 和 metadata；
- 多实例可以共享 Artifact；
- 生命周期、完整性和访问控制更清晰。

新增风险：

- 引入凭证、桶策略和清理任务；
- 对象存储不能成为主诊断链的单点依赖；
- 本地秋招演示未必需要真正启动 MinIO。

### 验收

- 本地和对象存储实现遵循同一契约；
- 引用带 hash、size、media type 和保留策略；
- 未授权用户不能读取 Artifact；
- 存储不可用时仍保留结构化摘要和失败状态。

## 5.18 排名 18：持久化任务队列与跨进程调度

### 前因

AutoOnCall 已有 durable session snapshot、断开后继续诊断和审批恢复，但 BackgroundTasks 与进程内任务不适合真正的多副本长任务。行业信息助手的检查点和任务状态说明长任务产品体验的重要性。

### 改进内容

在需要多实例部署时，引入独立 Worker 和持久化任务：

- 任务 claim 和租约；
- 幂等执行；
- 心跳和超时回收；
- 有界重试；
- 取消和恢复；
- 任务与 Incident、session、approval 绑定。

### 后果

正向结果：

- API 与长时诊断解耦；
- 服务重启后任务可恢复；
- 更接近生产系统。

新增风险：

- 需要 Redis、数据库队列或专用消息系统；
- 与现有 snapshot、SSE 和审批状态存在较大整合成本；
- 秋招前实施可能稀释 Agent 主线。

### 验收

- 同一任务不会被重复完成；
- Worker 崩溃后可重新 claim；
- 取消、重试和审批恢复保持状态一致；
- 不把业务 Redis 作为无替代方案的控制面单点。

## 六、依赖关系与推荐实施路线

这些方向不是互相独立的功能清单，而是一条有前后因果关系的升级路线。

```text
确定性时序分析
  -> 统一时间线
  -> 根因假设竞争
  -> 信息增益驱动的动态取证
  -> Deep Investigation
  -> 事故复盘
  -> Incident Memory
  -> 新事故中的相似案例与差异分析
  -> 人工复核的诊断数据集
  -> Planner / RCA / Replanner SFT
  -> 专家偏好优化与离线规则奖励
```

知识和工程支撑线：

```text
Runbook 治理
  -> 受控外部情报
  -> 知识新鲜度巡检

Incident 附件
  -> Artifact 存储抽象
  -> 多实例任务队列

模型调用观测
  -> 分任务模型路由
  -> Deep Investigation 预算控制

Trace、Evidence 与人工反馈
  -> 训练数据治理与独立评审
  -> LoRA/SFT 离线对照
  -> shadow 接入
  -> 数据成熟后再考虑 DPO/RFT
```

### 阶段一：形成最强秋招主线

1. 确定性运维时序分析器；
2. 统一事故调查时间线；
3. 根因假设与正反证据；
4. 对应工作台视图和评测集。

阶段成果：

> 把多源运维信号转换为确定性发现，再围绕候选根因收集正反证据，并展示假设如何被支持、排除或转人工。

### 阶段二：提高调查效率

1. 按信号域动态并行取证；
2. 查询去重和预算；
3. 复杂事故 Deep Investigation；
4. 完整模型 Token 和延迟观测。

阶段成果：

> 简单事故保持快速路径，复杂事故才进入受预算约束的深度调查。

### 阶段三：形成经验学习闭环

1. 事故复盘报告；
2. Incident Memory；
3. 相似事故检索和差异分析；
4. 人工反馈和 eval backlog 联动。

阶段成果：

> 历史事故作为可追溯先验服务新事故，但不能替代当前取证。

### 阶段四：形成反馈驱动的诊断策略优化

1. 定义 Planner、RCA 和 Replanner 阶段级标签；
2. 将 Trace、Evidence、反馈和人工修正转成版本化数据集；
3. 对小模型进行 LoRA/SFT，并与现有模型及规则基线盲测；
4. 达到阈值后仅以 shadow 或可回退路由接入；
5. 偏好数据成熟后再尝试 DPO 或基于规则奖励的 RFT。

阶段成果：

> 不是笼统地“给项目加微调”，而是证明反馈数据如何改善工具规划、根因排序和停止决策，并通过独立 holdout、确定性门禁和回退策略控制风险。

### 阶段五：完善知识和人机协作

1. Runbook 知识治理工作台；
2. Incident 附件证据；
3. 受控外部运维情报；
4. 定时知识新鲜度巡检。

阶段成果：

> 系统同时治理内部 Runbook、人工现场材料和受控官方资料。

### 阶段六：生产化储备

1. Artifact 对象存储抽象；
2. 持久化任务队列；
3. 多副本任务治理；
4. 数据集、模型和评测 Artifact 生命周期治理。

该阶段有工程价值，但不应抢占秋招前核心 Agent 主线的时间。

## 七、暂缓或明确不采用的方向

| 方向 | 结论 | 原因 |
| --- | --- | --- |
| 六七个角色型 Agent | 不采用 | 增加成本和调试复杂度，真实职责可以由业务节点承担 |
| 通用 Text2SQL 平台 | 不采用 | 运维 Schema 不稳定且查询安全范围过大，现有受控只读工具更合适 |
| LLM 任意代码执行 | 不采用 | 引入代码执行和数据泄漏风险，时序计算应使用确定性服务 |
| 任意 ECharts 配置生成 | 不采用 | 图表应由固定语义数据模型构建 |
| 通用新闻、股票和招投标 | 不采用 | 偏离 Incident 诊断主线 |
| Elasticsearch | 暂不采用 | 与现有 lexical + Milvus 混合检索重复 |
| 通用用户偏好记忆 | 不采用 | 应转换为 Incident Memory，而不是记忆操作员聊天偏好 |
| 简单 JWT 用户体系 | 不迁移 | 当前 token RBAC 适合演示，生产方向应是 SSO/OIDC |
| Redis 作为唯一控制面 | 不采用 | 被诊断对象可能就是 Redis，不能形成无替代的控制面依赖 |
| 整体重写 React 前端 | 暂缓 | 投入大，当前工作台可通过事件协议和局部组件增强 |
| 自动执行生产变更 | 不采用 | 继续保持审批、dry-run、sandbox 和人工执行边界 |
| 统一质量总分 | 不采用 | 会掩盖检索、生成、安全、证据和延迟的独立失败 |
| 直接用现有 fixture 正式微调 | 不采用 | 样本少且模板重复，容易记住 Redis/MySQL 黄金案例并污染回归集 |
| 微调实时 Runbook 和现场事实 | 不采用 | 知识更新应走 RAG，实时事实必须来自受控工具 |
| 在线强化学习自动处置 | 不采用 | 生产运维不允许探索试错，奖励延迟且容易被低延迟、少取证等代理目标钻空子 |
| 端到端单一“诊断成功”奖励 | 不采用 | 无法区分根因、证据、工具成本、停止决策和安全边界，容易产生奖励投机 |

## 八、总体验收要求

所有改进均应遵守：

1. 新能力有结构化模型和 API 契约。
2. 关键结论可回链到 Evidence、Trace、Citation 或确定性计算。
3. 实时、历史、mock、fallback 和外部资料明确区分。
4. 失败、超时、取消和证据不足有真实状态。
5. 不展示模型内部思维过程。
6. 不使用离线评测冒充生产准确率。
7. 不使用 LLM 自评分替代确定性门禁。
8. 新增指标记录样本、阈值、环境和 provenance。
9. 前端展示复用后端读模型，不在浏览器重新推断业务结论。
10. 所有生产写操作继续受审批和执行边界控制。
11. 训练数据与回归评测集按 Incident、时间、服务和模板族隔离。
12. 微调模型未通过独立 holdout 前不得替换现有主链路。
13. 训练输出继续经过结构校验、Evidence Gate、工具白名单和风险门禁。
14. 所有训练实验记录基础模型、数据版本、参数、指标和可复现 provenance。

## 九、最终推荐主题

如果秋招前只实施一条综合升级主线，建议选择：

> **基于确定性时序分析、根因假设竞争和信息增益取证的复杂事故调查工作台。**

它包含四个最关键层次：

1. 确定性分析器识别异常起点、峰值、恢复点和信号关系；
2. 统一时间线组织告警、指标、日志、变更和工具事件；
3. Agent 围绕多个候选根因收集支持证据和反证；
4. Replanner 依据证据区分度决定补查、完成或转人工。

在这条主线稳定后，最有含金量的第二层升级是：

> **基于人工反馈和可审计证据的诊断策略微调。**

它不改变业务主线，而是分别优化 Planner 的工具选择、RCA 的候选排序和 Replanner 的停止或转人工决策；实时知识继续由 RAG 提供，现场事实继续由工具获取，权限和审批继续由确定性系统控制。强化学习不作为当前卖点，只有积累足够专家偏好数据后，才考虑离线 DPO 或规则奖励训练。

推荐的面试表述：

> 我没有把行业信息助手的多角色 Agent 和通用工具直接搬进 AIOps，而是提取了任务拆解、并行研究、数据分析、来源追溯和检查点等机制。AutoOnCall 先用确定性时序分析生成可信信号，再由 Agent 建立候选根因并围绕假设动态取证；简单事故走快速路径，复杂事故才进入受 Token、时长和工具预算约束的深度调查。历史事故、人工附件和外部官方资料都只作为带来源等级的证据或先验，不能绕过当前取证、风险审批和人工接管边界。在数据闭环成熟后，我会把人工复核的 Trace、Evidence 和反馈转换为阶段级训练样本，通过 LoRA/SFT 优化 Planner、RCA 和 Replanner，并使用独立 holdout、shadow 路由和确定性安全门禁验证收益，而不是直接做线上强化学习。

这条路线最能强化 AutoOnCall 的核心差异：

> AutoOnCall 不是通过堆叠 Agent 猜测根因，而是把多源运维信号转换为可验证假设，通过受控工具调用收集正反证据，并在证据不足或风险过高时明确停止和移交人工。

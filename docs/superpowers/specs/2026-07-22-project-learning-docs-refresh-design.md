# AutoOnCall 项目学习文档全量更新设计

## 目标

以 2026-07-22 当前完整工作区为事实基线，对 `docs/project/` 下的目录计划、阅读导航和 00—15 专题进行全量审计与按影响深度重写，使读者能够从完全不了解项目，逐步建立业务、架构、工程和评测心智模型，并能在秋招面试中准确、深入、可复核地讲述项目。

事实基线同时包括已提交和未提交实现。文档必须区分代码存在、自动化测试验证、离线或本地运行证据、受控故障证据和生产证据，声明强度不得超过证据强度。

## 核心方法

采用“全量事实审计、按影响深度重写”：

- 逐篇核对当前模型、服务、API、集成、脚本、测试和配置；
- 保留仍然准确且教学效果良好的段落；
- 重写过时的实现事实、调用链、指标、工程边界和前后篇接力；
- 对变化较大的 RAG、评测和运行观测章节进行深度重构；
- 对变化较小的章节也完成逐条事实复核，不以“近期主要改了 RAG”为由跳过；
- 同步更新目录计划和阅读导航，使其准确反映更新后的课程顺序与学习产出。

## 叙事设计

全系列继续使用凌晨 02:07 的 Redis Incident 作为贯穿案例，但不强制每篇使用同一章法。每篇根据主题选择最自然的教学结构，例如问题推导、失败演进、状态机展开、一次完整调用链、指标诊断或安全威胁建模。

全系列只统一以下叙事原则：

1. 从业务问题或失败现场进入，不从文件和类名进入；
2. 先解释朴素方案为何不足，再引出必要概念和工程约束；
3. 把当前实现写成设计选择的结果，而不是代码目录说明；
4. 同时覆盖正常路径、降级路径、停止条件和生产边界；
5. 每篇承接上一篇留下的问题，并把一个明确问题交给下一篇；
6. 代码路径只作为核验入口，不支配正文结构；
7. 面试表达融入技术推导，章末可补充 30 秒、3 分钟、追问和练习，但不机械套用固定模板。

## 课程递进关系

课程主线保持为：

1. 建立事实和生产边界；
2. 从人工 OnCall 推导受控诊断闭环；
3. 把告警组织成稳定 Incident；
4. 建立知识入库、可信检索与可审计回答链路；
5. 建立 Plan—Execute—Replan、工具、Evidence、Hypothesis 和报告链路；
6. 建立身份、审批和安全变更边界；
7. 建立 SSE、Trace、Snapshot、Store 与 Replay 的运行态解释能力；
8. 用分层评测和量化证据证明系统没有退化；
9. 选修 A2A 业务门面；
10. 把全项目转化为可复核的校招表达。

每一阶段都要推动 Redis Incident 的认知状态变化：从告警观察，到 Incident 身份，到知识假设，到现场 Evidence，到根因候选，到报告、审批、变更、回放和评测，而不是在每篇重复介绍同一个完整故事。

## 本轮重点变更

当前未提交 RAG 改进需要进入相关课程，包括但不限于：

- 显式问题规划与检索偏好；
- Evidence Plan 和答案覆盖矩阵；
- 多来源与标题覆盖选择；
- Milvus 作为运行时权威向量后端时的定向来源探测；
- 本地词法索引的降级边界；
- 必要来源引用覆盖检查与修复重试；
- 答案子目标覆盖修复；
- 引用校验失败后的受控重试与抽取式兜底；
- 重试成本、后端身份、覆盖结果等可观测字段；
- 与上述能力对应的评测、脚本、知识资产升级和事实边界。

这些内容重点影响 03、04、05、13、15，并要求目录计划、阅读导航、00 和 01 同步更新全局心智模型。其他专题仍需核对当前实际实现和测试。

## 分工与编辑边界

采用按独立文档并行的方式。每个子 agent 只编辑明确分配的文件，并在提交结果时说明：

- 核对过的主要代码和测试来源；
- 发现并修正的旧事实；
- 保留的优秀教学段落；
- 新增或重写的课程推导；
- 仍不能越界声称的能力。

由于并发槽位有限，文档将分批并行处理。主 agent 负责分批调度、差异审查和最终跨篇整合。

## 验收标准

- docs/project 下的目录计划、专题阅读导航和 00—15 专题全部经过审计并产生与需要程度匹配的更新；
- 所有审计日期和事实范围反映 2026-07-22 当前工作区；
- 已提交与未提交实现均被纳入，但未完成验证的功能不得写成已验证成果；
- 关键术语、状态、字段、接口边界和跨篇案例保持一致；
- 近期新增能力在相关章节、导航、评测和校招量化章节中形成闭环；
- 旧指标、文件路径、样本规模和能力声明经过复核，无法复核的内容被降级表述或删除；
- 文档保持课程技术长文风格，不退化为 API、类或函数说明书；
- 各篇结构允许不同，但阅读顺序层层递进，上一篇提出的问题能在后文得到回答；
- Markdown 链接、相对路径、标题层级和本地文件引用通过自动检查；
- 最终通过代码检索、差异检查和针对性测试或文档校验确认事实一致性。

## 非目标

- 不把文档改造成统一僵化模板；
- 不为追求篇幅而重复同一套概念；
- 不虚构生产运行、真实故障或官方协议认证证据；
- 不修改与文档事实修正无关的业务代码；
- 不覆盖或回退用户当前未提交改动；
- 不擅自提交包含用户现有工作区改动的 Git commit。

# AutoOnCall README 招聘向完整版改版设计

## 目标

将仓库根目录 `README.md` 重构为面向招聘面试官的完整版项目说明。读者应能在首屏快速判断项目解决的问题、核心业务闭环与技术含金量，并在继续阅读后理解系统架构、关键实现、量化证据、运行方法和真实能力边界。

本轮只调整 README 的内容组织和表达，不修改业务代码、运行配置或评测资产。所有能力、指标、命令和边界均以 2026-07-22 当前工作区为事实基线；未验证能力不得写成既成成果，离线、candidate、official、live 和生产证据必须明确区分。

## 目标读者与阅读路径

首要读者是招聘面试官，其次才是准备运行或接手项目的开发者。README 采用渐进式阅读路径：

1. 一句话定位与核心业务问题；
2. 项目价值、技术亮点与可信量化证据；
3. 总体架构和端到端业务闭环；
4. 关键技术难点、设计取舍与技术栈；
5. RAG、告警、诊断和安全变更四条核心链路；
6. 快速开始、演示、API、配置、质量门禁与目录结构；
7. 生产边界、已知限制和深入文档导航。

首屏必须回答三个问题：这是什么、为什么值得做、与普通 RAG 聊天机器人有什么不同。后续章节则为这些判断提供可复核的实现和证据。

## 内容组织

采用“招聘叙事优先的完整版”，目标体量与当前 README 相近，但减少重复并提高每一节的信息密度。

### 第一层：项目价值

- 项目定位：受控、可解释、可审计、可评测的 OnCall 智能诊断系统；
- 业务问题：信息源割裂、经验依赖、诊断不可追溯、自动操作风险；
- 核心价值：把告警、知识、工具取证、证据分析、审批、报告和回放连接成闭环；
- 核心亮点：Plan—Execute—Replan、可信 RAG、Evidence 驱动 RCA、风险治理、安全变更和分层评测。

### 第二层：系统设计

- 保留并优化架构 Mermaid 图；
- 展开从 Alert/Incident 到报告或人工升级的主业务流；
- 用“问题—设计—收益—边界”的方式解释关键技术难点；
- 技术栈按职责分层，不做缺少上下文的名词堆砌。

### 第三层：实现与证据

- 分别描述告警接入、RAG 问答、AIOps 诊断、安全变更四条链路；
- 保留当前 Scorecard 与演示结果，但后移到架构和设计之后；
- 清楚标注 candidate、official、offline、live adapter 和生产证据的差异；
- 保留关键 API、配置、命令、目录和测试门禁，供技术面试追问与开发者复现。

## 编辑原则

1. 业务价值先于目录和类名，设计理由先于技术名词；
2. 每个亮点都能回到代码、测试、评测或文档证据；
3. 不用“全自动根因定位”“生产级”等超出证据的表述；
4. 不把 mock、fallback、离线 eval 或本地沙箱包装成真实生产能力；
5. 精简重复的面试清单、边界说明和评测解释，但不删掉关键事实；
6. 保留完整的快速开始、接口、配置、验证、目录结构和文档导航；
7. 统一中英文术语、标点、标题层级和命令格式；
8. README 只承担全局入口职责，专题推导通过链接进入 `docs/project/topics/`。

## 验收标准

- README 为面向招聘面试官的完整项目叙事，首屏突出业务价值和技术差异；
- 核心业务、架构、技术栈、关键链路、难点、设计取舍和安全边界均有清晰说明；
- 当前版本号、依赖、命令、接口、文件路径和评测数据与仓库一致；
- 重复表头、重复段落和散落的同义说明得到合并；
- 所有本地 Markdown 链接可解析，代码块和 Mermaid 语法边界完整；
- `make verify`、`make live-eval` 等门禁的适用范围表述准确；
- README 改动不覆盖工作区内其他未提交修改。

## 非目标

- 不修改项目功能、依赖或部署方式；
- 不新增未经验证的性能或准确率声明；
- 不把 README 改成纯简历话术或仅供面试背诵的材料；
- 不复制所有专题文档内容到 README；
- 不提交或回退用户当前其他工作区改动。

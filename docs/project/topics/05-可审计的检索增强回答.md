# 检索结果可信，为什么答案仍可能胡说：可审计 RAG 的最后一道边界

## Runbook 只说“建议检查”，模型为什么敢说“已经确认”

继续 Redis 主线：`order-service` 出现 503、P95 上升，日志有 `Redis connection timeout`，`connected_clients` 接近 `maxclients`。现场现象是事实，官方机制与 Runbook 是知识，“连接耗尽已经导致 503”仍是推断，事故窗口和连接池状态仍未知。检索器返回两块可信知识：官方说明称达到 `maxclients` 后新客户端连接可能被拒绝；内部 Runbook 要求扩容前检查 `connected_clients` 和连接池，并保留审批、验证与回滚边界。

用户问：“`order-service` 的 timeout 是否已经确认由 maxclients 导致？现在可以直接扩容吗？”一个追求有帮助的模型很容易回答：“已确认连接耗尽，建议立即扩容并重启应用。”这句话把三种信息混在了一起：官方机制是知识，本次指标才是事实，“已经确认根因”是推断；扩容与重启还跨过了审批边界。

因此 trusted chunk 只表示“这段资料有资格进入上下文”，不表示模型会正确理解、完整引用或不越权。检索控制的是输入集合，生成模型仍可能省略限定词、把历史事故写成当前事实、拼接两个片段形成新因果，或给出文档从未授权的动作。

本章始终区分三类材料：503、P95、timeout 与实时连接数属于本次 Incident 的现场事实；官方限制和 Runbook 是知识；“maxclients 已经导致 503”是 **Hypothesis**，即需要被事故窗口 **Evidence** 支持或反驳的根因假设。Evidence 是带来源、时间窗口和查询上下文的观测。RAG 能提供知识层材料，却不能替代实时工具完成现场取证。

## 三种朴素回答方案为什么都会失控

### 检索后直接调用模型

`llm(context + question)` 没有回答资格门禁。检索无结果、向量库失败或 citation metadata 缺失时，模型仍可能依靠参数知识组织一段流畅答案。用户看不出它究竟使用了哪块知识，也无法区分业务拒答和系统异常。

### 在 Prompt 里要求“请引用来源”

Prompt 是软约束。模型可能漏写引用、伪造看似合理的 ID、在末尾罗列全部 Top-K 却不说明哪条主张由谁支持。即使 ID 合法，“建议检查”仍可能被曲解成“已确认”。引用存在与引用正确是两个不同指标。

### 让通用 Agent 自己决定何时检索

开放工具的 Agent 可以绕过本轮 trusted chunks，调用时间、MCP 或其他知识工具，导致答案来源集合无法冻结。文档中的恶意指令还可能诱导模型尝试工具。知识问答需要固定证据集合；实时调查应交给下一章受风险治理的 AIOps Agent。

由此推导出可审计回答的最小架构：生成前决定是否有资格回答，生成时只提供批准片段且不提供工具，生成后再核对模型实际使用的引用；任何一层失败都返回结构化 no-answer，而不是让模型补全。

## 先定义“可审计回答”负责什么

**Citation（引用）**通俗地说是答案通往原文的路标；严格定义是本轮 trusted result 派生的 `source_file + chunk_id`，以及标题路径、文档类型、页码、sheet/行/table/主键、分数与检索理由等可选定位信息。它解决用户和测试能否回到批准知识单元；不负责自动证明一句话在语义上被原文蕴含。

**Grounded generation（有依据生成）**通俗地说是“只用给定材料组织语言”；严格定义是把固定 evidence blocks 放入系统/用户消息，并禁止此次模型访问外部工具的生成过程。它解决模型知识边界；不负责保证 Prompt 必然被遵守。

**Claim-to-Chunk 契约**要求每个可验证主张指向一个允许 chunk。当前硬检查能确认答案出现了允许的 `chunk_id`，不能确认自然语言 claim 与 chunk 之间完成语义 entailment。反例是答案写对 ID 却把“建议检查”曲解成“现场已确认”。在 AutoOnCall 中，它是引用 ID 契约与生成前后 guard，不是逐句事实核验。

可审计回答最终分为三类：有 trusted chunk 且使用有效 ID 的 `answer_with_citations`；服务正常但知识或引用不合格的业务 no-answer；依赖或接口未处理异常的系统失败。拒答不是 500，而是知识边界内的正常结果。

**Trust Gate（可信门禁）**通俗地说是“候选资料够不够格进入回答”；严格地说，它依据检索状态、距离阈值、融合结果和来源边界，把结果分成 trusted、rejected、no-answer 或 failed。它解决低质量候选是否允许进入生成，不负责检查模型生成后的自然语言。

**No-answer（业务拒答）**是服务正常完成但知识资格不足、citation 不完整或模型主动认为上下文不足。它应返回成功的业务结构与明确 policy，不是 HTTP 500。**System failure（系统失败）**则是未被业务降级吸收的依赖或代码异常：非流式 API 返回 500 与公开安全错误，流式连接通常已是 HTTP 200，只能发送 `error` SSE。

把这些概念连起来，可以得到四条不等式：

```text
trusted chunk ≠ 正确答案
出现 citation ≠ claim 得到支持
业务 no-answer ≠ 系统失败
工具隔离 ≠ 完整 Prompt Injection 防御
```

## 模型是否有资格生成，必须在调用前决定

`/api/chat` 先显式调用上一章的结构化 retrieval。同步检索函数通过 `asyncio.to_thread` 执行，避免其阻塞 FastAPI event loop。若状态是 `no_answer`，说明没有 chunk 通过 Trust Gate，直接返回 `refuse_without_trusted_source`；若 retrieval `failed`，保留 `retrieval_failed`，提示检查向量库或配置。两者都不调用 LLM，避免空上下文强答，也节约一次模型调用。

retrieval 成功后，系统把结果压缩成安全 citation：文件名级 source、`chunk_id`、`heading_path`、`doc_type`、页/表/行/主键、分数与理由。生成前至少要有一个同时包含有效 `source_file` 和 `chunk_id` 的 citation，否则改写为 no-answer：`refuse_without_citation`。

这里的约束是“至少一个 citation 完整”，不是逐项保证所有候选 metadata 都完整。`source_file=未知来源` 不算有效；正文相关但缺定位时，修复方向是重新索引或补 metadata，不是放宽审计要求。

请求本身也有入口边界：session ID 最长 `128`，问题最长 `8000`，两者先 strip，空白会被 Pydantic 拒绝；可选 `metadataFilter/MetadataFilter` 会透传给结构化检索，影响本轮允许证据集合。chat 与 clear 需要 chat-write scope，history 查询需要 read scope。当前是静态 token/scope，不是面向最终用户的细粒度知识 ACL。

`/api/chat` 没有声明 response_model，`ChatResponse` 也没有承载 citations、retrieval、noAnswer 与 answerPolicy；当前 camelCase 响应靠手工 dict 与测试维持。这是已实现 API 事实，也意味着字段漂移无法完全依赖 Pydantic 输出校验，生产化可补专用响应模型与 OpenAPI 契约。

### API 成功、业务成功和回答成功是三层状态

非流式正常请求返回外层 `code=200/message=success`，data 中还有 `success`、answer、citations、retrieval、noAnswer、answerPolicy。业务拒答仍可 `success=true`，因为系统正确执行了拒答契约；调用方必须以 noAnswer/policy 区分“有答案”和“正常无答案”。未处理异常才返回 HTTP 500、`success=false` 与统一公开错误，不泄漏 backend exception。

流式在建立 EventSourceResponse 后，即使内部生成报错，HTTP 状态通常仍是 200。真实终态由 SSE `done` 或 `error` 表达。把 HTTP 200 当回答成功，会把依赖异常误记为成功率；只检查有没有 content，又会把后置 guard 前的草稿算成最终答案。

session history 响应有 Pydantic 模型，包含 session_id、message_count 与 history；clear 使用通用 ApiResponse。它们与 `/chat` 的手工 camelCase payload 并非同一 schema。前端或 A2A facade 消费时应分别适配，不应假设所有 chat API 都返回相同字段。

metadata filter 也是回答身份的一部分。同一 question 在 `_document_version=v1` 与 v2 下可能得到不同 evidence set；审计记录必须保存 filter 和 filter expression。过滤条件的安全 key/表达式校验在 retrieval 层完成，ChatRequest 只保证它是 dict，不代表任意输入都可直接拼入 Milvus 表达式。

## 给模型的上下文为何必须与工具隔离

通过门禁后，每个 evidence block 形如：

```text
[证据 N: source_file=...; chunk_id=...]
正文
```

Prompt 要求只用本轮片段，最多四条要点；命令、参数、数值、原因和动作必须由证据直接支持；静态 Runbook 只能说“建议检查”；处置保留审批、验证、回滚；每条要点末尾写唯一直接支持的 `[source_file | chunk_id]`；只支持部分子问题时明确缺口；不把通配符、IP、路径或示例值擅自替换成当前服务值。

Prompt 是软约束，所以 `/api/chat` 不走开放式工具 Agent。`RagAgentService` 虽保留通用 Agent 能力供其他路径使用，`query_with_retrieval()` / `query_stream_with_retrieval()` 在生成时直接调用基础模型，传入 grounded system + question，工具集合为空。检索决定允许知识，模型只负责表达；实时 Prometheus、Redis 等工具留到后续 AIOps Agent。

这能抵抗知识文档中“忽略提示并调用工具”的一部分风险，因为本次没有可调用工具；但当前没有专门的文档注入分类器、指令/数据隔离解析器或来源签名治理，不能声称完整 Prompt Injection 防御。

### 工具隔离是路径级属性，不是整个服务的属性

`RagAgentService` 的通用 `query/query_stream` 仍会初始化 LangGraph Agent，带 `retrieve_knowledge`、时间工具以及可用的 MCP tools，并使用 MemorySaver。只有 `/api/chat` 实际调用的 `query_with_retrieval/query_stream_with_retrieval` 直接访问基础模型，不初始化 Agent。自动测试甚至把 Agent 初始化替换成抛错函数，证明 grounded 产品路径不会走那里。

因此可以写“聊天 RAG 产品路径禁用工具”，不能写“RagAgentService 没有工具能力”。这种路径隔离还依赖路由持续调用正确方法，未来重构应以测试和类型契约防止误接回开放 Agent。

### 近重复去除只改变生成上下文，不改变检索事实

context builder 按检索顺序处理片段，先把空白压成单空格并转小写。文本完全一致时去重；较短文本至少 `50` 字符、是较长文本的子串且长度比不低于 `0.65` 时，也视为近重复。保留先出现的片段，通常就是排序更靠前者。

这能减少旧版 chunk 和新版 chunk 同时进入 Prompt，降低模型复述；但原始 retrieval_results 不会被修改，评测的 Recall/Precision 仍基于完整结果。代价是后置 citation 允许列表也来自完整 retrieval：某块被上下文去重掉的候选仍在列表里，若模型凭空写出其 ID，当前字符串 guard 理论上仍会接受。更严格实现应让 generation evidence set 与 citation allowlist 使用同一个对象。

## 生成以后，为什么还要再检查一次

模型输出后先识别主动拒答表达，如“当前知识库无法回答”。retrieval 虽成功，但模型认为上下文不足时，系统规范为 no-answer，不保留一组看似已使用的 citations。它与 Trust Gate no-answer 的发生阶段不同。

主动拒答识别目前是中文 marker 子串启发式，例如“当前知识库无法回答”“知识库中没有”“无相关信息”。它不是模型返回的结构化枚举，可能误判带否定或引用这些短语的正常回答，也可能漏掉新的拒答措辞。最终 `no_answer=true` 时，retrieval.status 仍可能是 success，因为检索成功、生成阶段拒答；调用方必须同时看 answer policy，不能只看 retrieval status。

随后，服务只从候选 citations 中保留 `chunk_id` 实际出现在答案里的项。若一项也没有，最终改为 `refuse_without_citation`；若出现 chunk ID 但漏写 source file，则追加标准引用区块，补齐 source、chunk 与页/行定位。

假设允许列表只有：

```text
official_redis_clients.md#0001
redis_capacity_wiki.html#0003
```

模型写 `redis.md#9999` 时，后置选择为空，答案被缺失引用消息替换，retrieval 标为 citation-guard no-answer。

这道检查能证明答案使用的 ID 来自本轮允许列表，最终 citation 对象来自 trusted results；不能证明每个 claim 都被语义蕴含、模型没有曲解、所有子问题都完整覆盖。更强方案需要固定生成评测、人工复核或 claim-level verifier，并为 verifier 成本、偏差和故障设计降级。

### 当前后置 guard 的严格实现边界

`select_supporting_citations` 对每个候选做的是：chunk_id 非空且其字符串出现在 answer 中。它没有解析 `[source_file | chunk_id]` 语法，也没有检查 ID 边界、引用所在句子或 source/chunk 配对。因此存在三类误通过：短 ID 恰好是更长 ID 的前缀，正文偶然提到允许 ID，或一条错误 claim 后面放了合法 ID。

若选中的 citation 缺 source_file 文本，`ensure_citation_block` 会在答案末尾追加“引用来源”区块，列出 source、chunk、页/sheet/row/primary key 与格式化到 `4` 位小数的 score。它提高回查性，却不能证明模型原本完成 claim-level 标注。引用区块是审计补全，不是事实 verifier。

更强的实现可以要求模型输出结构化数组：每项含 claim、citation IDs、answer span 与 unsupported reason；服务端精确解析 ID，再用确定性规则验证允许集合，必要时调用 claim entailment Judge。Judge 也会带来模型偏差、成本和不可用问题，因此应超时降级为部分保留或人工复核，而不是默认为通过。

### 公开读模型只暴露安全定位信息

compact retrieval 把服务器绝对 source_path 压成文件名，隐藏 `vector_error_detail` 之类内部地址；仍可返回安全的 vector error type/message、降级标记、metadata filter、融合策略、候选计数与 observability。Citation 可以包含页、表、行和主键，用户能定位原文，却拿不到服务器目录。

安全压缩不等于来源真实性。`source_file` 是索引 metadata，当前没有文档签名、发布者证明或不可篡改知识治理。恶意文档即使通过 metadata 契约，仍可能把指令作为内容送入模型；工具隔离限制其副作用，但不能消除回答被内容诱导的风险。

## Redis 回答的正常路径与失败路径

用户问：“Redis maxclients 告警时怎样取证？是否可直接扩容？”结构化 retrieval 返回官方说明与容量 Runbook；生成前 citation 完整；近重复 evidence 被去除；无工具基础模型回答；后置检查只保留实际使用的 citations；API 返回 answer、compact retrieval、`no_answer=false`、`answer_policy=answer_with_citations`。

合格措辞应接近：

```text
- 建议先检查 connected_clients 与 maxclients 的关系；静态文档只能说明取证方法，
  不能单独确认本次事故根因。[official_redis_clients.md | ...#0001]
- 扩容前需按 Runbook 保留审批、验证和回滚边界，当前片段不代表动作已经获批。
  [redis_capacity_wiki.html | ...#0003]
```

当前已知的是知识库机制与流程；本次 503、P95、timeout、实时连接数仍要由 AIOps 工具取证。缺现场事实时不把知识改写成根因；无匹配 Runbook 时不调用模型；合法 ID 但主张被曲解时，ID guard 仍可能通过，应在评测或人工复核发现；只支持一个子问题时允许部分回答，但必须写出缺口。

MySQL 对照案例同样要把历史 rc3 相关性写成历史证据，不能授权本次回滚。Kubernetes 知识若只解释 Pod 排查，不应补造生产重启资格。

## 同一个问题可能在四个阶段停止

### Trust Gate 拒答：根本没有批准知识

若所有候选距离超阈值或主题不匹配，retrieval.status 为 no_answer，`answer_policy=refuse_without_trusted_source`，trusted results 为空、rejected results 可供前端解释。系统不调用 LLM，回答提示补充知识文档。这里能说“没有通过当前门槛的知识”，不能说“世界上没有答案”。

### Retrieval failed：依赖不可用

若向量库或检索配置失败且未被词法降级吸收，policy 为 `retrieval_failed`，消息提示检查依赖。它与知识不足不同：重试或恢复服务可能得到答案。API 仍把已处理的 retrieval failure 作为业务 no-answer 返回；只有服务方法抛出未处理异常，非流式才返回 HTTP 500。

### Citation pre-guard：有正文但无法回查

retrieval.status 虽 success，但没有任何一项同时具备有效 source_file 与 chunk_id。系统把全部 trusted results 移入 guarded payload 的 rejected_results，status 改为 no_answer，policy 为 `refuse_without_citation`，不调用模型。正确修复是重新索引 metadata，而不是允许匿名知识进入回答。

### Generation/post-guard：模型自己拒答或没有使用允许 ID

模型主动拒答时统一成 `refuse_without_trusted_source`；生成了内容却未出现允许 chunk ID 时改为 `refuse_without_citation`。两者 citations 都为空。前者表示候选过了检索门禁但模型认为仍不足，后者表示回答未满足审计契约。

| 停止点 | 是否调用 LLM | final policy | 主要责任方 |
|---|---:|---|---|
| Trust Gate no-answer | 否 | `refuse_without_trusted_source` | 知识覆盖/阈值 |
| retrieval failed | 否 | `retrieval_failed` | 检索依赖 |
| citation pre-guard | 否 | `refuse_without_citation` | 索引 metadata |
| model explicit refusal | 是 | `refuse_without_trusted_source` | 上下文与生成 |
| used-citation 为空 | 是 | `refuse_without_citation` | 生成契约 |
| 未处理异常 | 不定 | HTTP 500 或 SSE error | 系统可靠性 |

这张表帮助值班人员决定下一步：补文档、修索引、恢复依赖、调整 Prompt/模型，还是调查系统异常。把所有失败都显示成“知识库无法回答”，会掩盖可运维性。

## 流式最终契约相同，过程安全性却不同

流式路径发送 `search_results → content... → done`。检索或前置 citation guard 失败时不会调用模型，直接发送拒答与 done。问题在后置检查：token 已作为 content 发出后，系统才知道是否引用了允许 chunk；失败时只能再发拒答 content，并在 done 中把最终 answer/policy 改成拒答，SSE 无法撤回先前 token。

所以非流式只暴露通过检查的最终结果；流式的权威状态是 `done.data.answer`，此前 token 只是草稿。`search_results` 也只是本轮检索材料，不等于最终实际使用 citations。严格客户端应缓存草稿，收到 done 后以 final answer/citations/policy 覆盖；服务端更强方案是完整生成检查后再回放，代价是首字延迟和“真流式”体验。

系统异常还有另一条路径：SSE HTTP 响应通常已经是 200，API 只能发送公开安全的 `error` event，未必再有 done。客户端若只等待 done 会永久 loading；正确状态机必须把 done 和 error 都当终态，并在断流无终态时查询或提示重试。原始异常不会直接发给客户端。

当前 grounded stream 不构造完整 generation observability，也没有 token usage、TTFT 和 end-to-end total 的 final payload。不能用非流式观测回填流式路径，更不能从客户端看到 token 间隔就冒充服务端 TTFT Artifact。

## 多轮“能查询历史”为什么不等于“模型拥有可信记忆”

grounded 问答把用户原问题与最终 answer/no-answer 追加到进程内 `_grounded_history`，每轮两条记录共用时间戳。下一次生成仍只接收本轮 grounded system 与本轮 question，不把历史 turns 重新喂给模型，也不自动继承旧 citations。用户若问“那现在可以扩容吗”，必须在新问题补齐对象和上下文，否则本轮检索可能无法理解“那”。

history API 会先读取通用 Agent 的 MemorySaver checkpoint，排除 SystemMessage，再合并 `_grounded_history` 并按 timestamp 排序。checkpoint 消息若没有 timestamp，读取时会补当前时间，这可能让同一历史在不同读取时排序变化。当前 grounded 产品路径通常只依赖独立 dict，但不能把合并结果描述成严格时序日志。

clear 会删除 MemorySaver thread 与 grounded dict。两者都在进程内，没有数据库持久化、跨进程共享、TTL、容量上限或并发治理；进程重启后消失。`get_session_history` 内部异常会捕获并返回空列表，所以 API 未必以 500 暴露读取失败；clear 返回 false 时也可能是 HTTP 200 加业务 error。空历史不一定证明从未对话。

服务中虽然定义了保留系统消息和最近若干消息的 trim middleware，但当前 `create_agent` 没有接入它；更不适用于 direct grounded path。不能把注释中的“最近 3 轮”写成已生效的上下文治理。

## 观测字段必须区分 observed 与 not_observed

非流式 grounded generation 使用 perf counter 记录 `llm_generation_ms`，并尝试从 LangChain `usage_metadata` 或 OpenAI-compatible `response_metadata` 提取 input/output/total tokens；只有 provider 返回字段时才标 `observed`，缺失时保持 `not_observed`，不会自行估算。若只有输入和输出，可求和得到 total。

与 retrieval observability 合并后，stages 可包含 retrieval_total、vector search 等检索阶段、LLM generation 和 total；但 `context_build_ms` 与 `llm_ttft_ms` 当前明确为 `not_observed`。没有带日期的价格快照时，estimated cost 也保持 amount/price snapshot 为空，不能拿今天价格回填历史。

另一个容易忽略的边界是：service 的非流式返回包含顶层 observability，但 `/api/chat` 当前手工响应只转发 answer、citations、retrieval、noAnswer、answerPolicy，没有转发该顶层字段。compact retrieval 内仍带 retrieval observability，却不等于完整生成总览。性能脚本可以直接调用 service 或读取 Artifact；普通 API 用户当前拿不到完整 end-to-end observation。

## 用六条不变量检查可审计回答

### 没有 trusted source 就不能调用生成模型

这条不变量同时约束 no-answer 和 retrieval failed。模型参数知识再丰富，也不能补充当前知识库之外的结论。否则同一个 API 在依赖故障时会悄悄变成普通聊天，用户无法知道证据边界已经改变。

### 生成证据集合必须被冻结

进入 Prompt 的 evidence blocks、允许 citation 列表和最终返回 citations 应来自同一轮 retrieval。当前近重复去重与全量 allowlist 尚未完全同源，是需要收紧的边界。metadata filter 也应进入审计信息，防止两个相同问题因过滤条件不同却被当成同一实验。

### 每个最终 citation 必须来自允许集合

模型不能通过伪造 ID 扩大来源。当前字符串选择实现了一个低成本近似，但没有证明 citation 紧邻 claim 或 source/chunk 配对正确。硬门禁的结论必须限制在 ID 级。

### no-answer 必须是正常、可分类结果

缺知识、缺定位、模型拒答与依赖失败需要不同 policy。这样知识运营、索引维护和平台运维才能分别处理。系统异常则走 500 或 SSE error，不能伪装成知识不足。

### 历史不能自动成为本轮证据

会话历史适合用户回看，不代表下轮模型可继续依赖。若未来引入多轮 memory，必须把历史 claim 重新关联到仍有效的 chunk/version，并重新经过 Trust Gate；否则旧 citation 在知识更新后可能失效。

### 未观测指标必须保持未知

TTFT、Token、成本、P50/P95/P99 只有在对应路径和样本真实记录时才能报告。`not_observed` 比猜测值更有工程意义，因为它指出下一步应该补哪一层 instrumentation。

这些不变量比具体 LangChain API 更稳定。未来替换模型、向量库或 SSE 库时，只要它们仍成立，可审计回答的业务语义就没有被框架变化带走。

## 量化质量不能只看“读起来不错”

2026-07-12 的回答质量冻结记录包含 12 个核心 case，每个运行 3 次，共 36 个 answer-run：

| 指标 | 结果 | 能证明什么 / 不能证明什么 |
|---|---:|---|
| 确定性通过率 | `12/12 = 1.00` | 核心 case 产品契约通过 |
| 三次运行稳定率 / 标准差 | `1.00 / 0` | 仅该固定样本与配置 |
| Incident ID Recall/Precision | `1.00 / 0.70` | 参考上下文全覆盖，但仍有额外上下文/引用空间 |
| OnCall Actionability | `1.00` | 当前 rubric 包含可执行检查与受控处置建议 |
| 引用存在/支持/grounding | `1.00 / 1.00 / 1.00` | 评测规则命中，不等于逐 claim 语义证明 |
| 引用正确率 | `0.70` | 不能被总通过率掩盖，仍需改进 |
| 拒答边界 | `1.00` | 固定 OOD/证据不足 case 按契约拒答 |
| Incident 边界/混淆消解 | `1.00 / 1.00` | 只覆盖设计好的混淆案例 |
| 事实错误率/严重幻觉率 | `0 / 0` | 不是生产“零幻觉” |

`36/36` 模拟评审与自动结果一致，只证明 review 模板、导入和统计链可工作，不能写“真人盲评一致率 100%”。

### 这些指标为何不能互相替代

确定性通过率检查产品契约是否满足，包括引用存在、拒答和边界 rubric；它不是自然语言真实性的统一概率。Incident ID Recall `1.00` 表示参考上下文全部被覆盖，Precision `0.70` 表示返回上下文/引用仍包含额外项。Recall 高而 Precision 较低，可能让模型面对更多噪声，因此不能只报告 Recall。

引用存在率回答“有没有路标”，引用支持/grounding rubric 回答固定规则是否认为路标与答案有关，引用正确率 `0.70` 则直接揭示仍有不精确引用。三者分母和判定方法不同。`12/12 = 1.00` 不能把 `0.70` 覆盖成“引用完全正确”。

事实错误率和严重幻觉率为 `0` 只表示这组固定 case 的评测没有命中相应错误。样本为 `12` 个核心 case、每个 `3` 次，共 `36` answer-run；小而受控的数据不能推导生产用户分布。稳定率 `1.00`、标准差 `0` 也只说明固定配置下三次一致，不表示换模型、Prompt、知识版本或网络后仍稳定。

RAGAS Faithfulness 关注 claim 是否由上下文支持，Response Relevancy 关注是否围绕问题，OnCall Actionability 检查项目 rubric 的取证/受控动作，Answer Completeness 检查子问题覆盖。阈值 `0.85/0.80/0.80/1.00` 是项目门槛，不是行业统一标准；ID Context Precision/Recall `0.75/0.75` 又是确定性 ID 指标。报告分数时必须同时写 profile、answer source、Judge、样本与 Artifact。

这组 `12 case × 3` 的机器 Artifact 为 `logs/ragas_stage3_core_deterministic_v3.json`，模拟评审为 `logs/ragas_stage3_core_simulated_review.json`。两者记录 `evidence_level=offline_fixture`、commit `00aa78a5e2ef2798030368d4aa559edf46f818f4` 和 dirty worktree，所以是历史 candidate，不是 2026-07-12 official benchmark。前一 Artifact 记录 `eval/ragas_stage3_core_cases.yaml` 及 worktree/知识资产/config 指纹，但未独立保存 case-set SHA-256 与原始命令；当前文件 SHA-256 `365f2c6fec8a8dd4f876962a9731f2cdd8af6f7d48d4038d7e254ea608bf3817` 只帮助当前复跑，不能反向补成当时 provenance。

RAGAS 分两种 profile：`id-smoke` 用 ID-based Context Precision/Recall 和确定性业务门禁提供 CI 信号；`full` 才调用 Faithfulness、Response Relevancy、OnCall Actionability、Answer Completeness Judge。当前默认阈值依次为 `0.85`、`0.80`、`0.80`、`1.00`，ID Context Precision/Recall 阈值均为 `0.75`。完整定义与 answer-source 隔离见[第 15 篇](15-项目量化指标与校招验收.md)。

## 设计取舍与生产边界

### 为什么选择显式 retrieval，而不是 Agent 自主检索

显式路径更固定，牺牲了模型根据对话动态选择工具的灵活性，却能在生成前得到结构化状态、rejected candidates、metadata filter 和 allowlist。知识问答最重要的是冻结证据集合；动态 Prometheus、Redis、日志等工具交给下一章有 Evidence 与 Risk Controller 的 AIOps Agent。

### 为什么 ID guard 与语义 Judge 应分层

ID guard 快、确定、离线可测，适合作为不能绕过的硬门禁；它只能回答“引用是否来自允许集合”。Judge 可以检查 claim entailment、Faithfulness 和 Relevancy，却增加模型成本、延迟、随机性和供应商故障，还可能对专业术语误判。合理方案是 ID guard 永远执行，语义 Judge 用于离线评测、高风险回答或抽样复核，并保留人工申诉。

### 为什么允许部分回答

用户同时问“如何取证”和“是否可扩容”时，官方文档可能只支持取证。整体拒答会丢掉有价值内容，因此 Prompt 允许回答有证据的部分并明确具体缺口。代价是当前没有结构化问题分解器，模型可能误判自己已覆盖所有子问题。生产化可先提取 sub-questions，再要求每项输出 answered/unsupported 与 citations，服务端检查覆盖率。

### 为什么流式不能只靠最终 guard

后置检查能让 done 诚实，却无法撤回已显示 token。服务端全缓冲后回放最安全，但失去低 TTFT；按 claim 单元缓冲、验证、发送可折中，却要求稳定分句和 citation parser。对于生产操作建议，宁可增加延迟，也不应把未验证草稿直接渲染为权威答案。

### 为什么会话历史暂时不进入生成

不使用历史降低代词理解能力，却让每轮 evidence set 清晰，不会隐式继承过时 citation。若要支持可信多轮，应持久化 turn、question rewrite、retrieval fingerprint、chunk version 与 final policy；新一轮对引用重新校验，而不是把整段历史直接拼进 Prompt。

### 走向生产还缺哪些治理

| 能力 | 当前状态 | 生产演进 |
|---|---|---|
| 来源身份 | source/chunk metadata | 文档签名、发布者、版本与撤回 |
| 引用检查 | chunk ID 子串 | 结构化 citation parser、精确 allowlist |
| 事实检查 | Prompt + 离线 Judge | claim-level verifier、抽样人工复核 |
| 注入防护 | 工具隔离 | 指令/数据分区、文档扫描、来源信任 |
| 流式安全 | done 最终改写 | server buffer 或 claim 单元验证 |
| 历史 | 进程内 dict/MemorySaver | durable turn store、版本化 citation |
| 观测 | 部分阶段与 provider Token | API 透传、TTFT、价格快照、SLO |
| API schema | 手工 dict | 专用 Pydantic response、版本兼容 |

LangChain 提供消息对象、`create_agent`、MemorySaver 和 `ChatQwen`；检索、Trust Gate、生成路径选择、citation allowlist、no-answer policy 与后置检查都是项目自己的业务约束。不能把框架能力当成项目已经完成的治理。

当前尚未实现逐 claim entailment、完整知识库 Prompt Injection 防御、数据库 grounded memory、token 级可撤回安全流、结构化引用解析和完整 API generation observability。它们是边界，不是可以被“回答看起来合理”覆盖的细节。

## 面试转化：从“有引用”讲到“引用能证明什么”

### 一句话定位

> AutoOnCall 用显式检索、无工具 grounded generation 和生成前后 citation guard，把知识问答约束成可拒答、可回查的引用 ID 契约。

### 30 秒回答

> `/api/chat` 先做 Hybrid Retrieval，再由禁用工具的模型生成。生成前检查 trusted source 和 citation metadata，生成后只保留答案实际出现的允许 chunk ID；无来源、模型拒答或引用失败统一为结构化 no-answer。它是 ID 级审计，不是逐 claim 语义核验，流式最终 `done` 才权威。

### 3 分钟展开

> 我先用“建议检查”被改成“已确认”说明 retrieval trusted 不等于 generation grounded。产品路由先显式检索，no-answer/failed 不调用模型；success 后至少一条 citation 必须有 source_file+chunk_id。上下文按 evidence block 构造并去近重复，基础模型直接调用，不初始化带 MCP 的通用 Agent。生成后只保留答案中出现的允许 chunk ID，无 ID 则拒答，缺 source 则追加定位区块。这个 guard 本质是字符串 ID allowlist，不是 claim entailment。主动模型拒答、citation failure 和系统异常分别有 policy。流式 content 在 post-guard 前已经发出，所以 done 才权威，error 也是终态。量化上要同时讲 `12/12` 契约通过、引用正确率 `0.70`、冻结演示 `3/3` 与全量 runtime `9/12`，不能只挑最好数字。

### 不可夸大的表述

- 不写“实现逐句事实核验”，当前只做 ID 字符串门禁。
- 不写“完整防御 Prompt Injection”，当前主要依靠工具隔离和 Prompt。
- 不写“支持持久化多轮可信记忆”，grounded history 仅进程内且不参与下轮生成。
- 不写“流式输出每个 token 都已通过引用检查”，done 前是草稿。
- 不写“API 已观测完整 Token、TTFT 与成本”，多项仍 not_observed 或未透传。
- 不把 `12/12`、`3/3` 或事实错误率 `0` 写成生产零幻觉。

### 高频追问

1. **答案出现 chunk ID 为什么不算事实核验？** 只能证明 ID 属于允许列表，不能证明 claim 被语义蕴含。
2. **检索成功但模型拒答怎么办？** 规范为 no-answer 且不保留 citations，表示候选过门禁但不足以组织回答。
3. **为什么不让 Agent 自己检索？** 产品路径需要固定证据集合和可测资格门禁，开放工具会模糊来源。
4. **前置与后置 guard 分别防什么？** 前置防无可信/无定位上下文调用模型；后置防模型漏引或伪造 ID。
5. **流式如何更安全？** 服务端完整生成检查后回放，或按 claim/citation 单元验证后发送；代价是延迟与复杂度。
6. **为什么 `12/12` 仍有引用正确率 `0.70`？** 总契约与引用精度是不同指标，前者通过不应掩盖额外/不精确引用。
7. **部分回答怎样避免误导？** 明确列出未覆盖子问题；更强方案需结构化问题分解和覆盖校验。
8. **为什么允许列表与 generation context 应完全同源？** 否则被去重掉的候选仍可能因 ID 在全量列表而被错误接受。
9. **主动拒答为什么不能只靠字符串？** marker 有误报/漏报，应让模型返回结构化状态并保留兼容 fallback。
10. **非流式和流式观测有何差异？** 非流式 service 可采 generation time/provider Token；流式当前没有完整 total/Token，API 也未透传顶层观测。
11. **会话历史为什么不自动进入下一轮？** 防止旧 claim 与 citation 被隐式继承；可信多轮需重新检索和版本校验。
12. **retrieval.status=success 与 no_answer=true 能否同时出现？** 可以，说明检索成功但生成模型主动拒答，阶段不同。

## 检验理解的练习

1. 从 `/api/chat` 分别追踪 retrieval no-answer、citation failure 与系统失败字段。
2. 让流式模型输出不存在的 ID，记录先发 content 与最终 done 改写。
3. 设计 claim-level verifier 的输入输出、超时、部分保留与降级策略。
4. 将恶意指令写入知识文档，列出工具隔离仍有效和未覆盖的风险。
5. 用同一 Redis chunk 写一个 ID 合法但语义曲解的答案，并设计能抓住它的评测。
6. 构造 chunk ID 前缀碰撞，让子串 guard 误选 citation，再设计精确 parser 的测试。
7. 让 generation context 去掉近重复 chunk，却让模型输出被去掉的 ID，说明 allowlist 怎样收紧。
8. 设计可信多轮问答数据模型，至少包含 turn、rewritten question、retrieval fingerprint、chunk version、final policy 与过期规则。
9. 为 SSE 客户端写状态机：search_results、content、done、error、无终态 EOF 分别怎样展示和恢复。
10. 设计端到端 observability，使非流式/流式同时记录 retrieval、context build、TTFT、generation、total、Token 和带日期价格快照。

## 什么时候应该停止、降级或转人工

- Trust Gate 无 trusted result：停止生成，业务拒答并保留 rejected candidates。
- retrieval failed：停止生成，提示依赖；若有明确词法降级则显示 degraded，不能伪装正常向量检索。
- 所有 citation 缺 source/chunk：停止生成，要求重新索引。
- 模型主动拒答：返回 no-answer，不携带貌似已用的 citations。
- 生成未使用允许 ID：后置拒答；非流式不泄露草稿，流式 done 覆盖草稿。
- 只有部分子问题有证据：回答支持部分并写具体缺口，不补通用建议。
- claim 与合法 ID 明显不一致：当前硬 guard 未必发现，高风险场景应转人工或语义 verifier。
- SSE error 或无终态断流：客户端不得把已收 content 当最终成功，应提示重试/查询。
- 实时状态、执行资格或生产动作问题：知识问答只解释 Runbook，转交 AIOps 取证与审批链。

这套停止规则的目标不是提高拒答数量，而是让失败可诊断：知识缺失交给知识运营，metadata 缺失交给索引，依赖失败交给平台，生成不守约交给 Prompt/模型评测，现场事实交给工具调查。

至此知识链路闭合：多格式入库 → Hybrid + Trust Gate → 工具隔离生成 → 引用/拒答。下一篇将从“Runbook 怎么说”转向“本次 Incident 实际发生了什么”，并引入受治理的多步工具调查。

本章最重要的结论不是“答案末尾有来源”，而是回答资格、证据集合、生成权限、最终引用和失败策略都能被外部检查。Redis Runbook 可以告诉值班人员应该检查什么、动作需要哪些审批边界；它不能替代本次 Incident 的实时 Evidence。只要系统仍能准确说出自己不知道什么、为什么拒答以及下一步应修知识、索引、依赖还是诊断链，可审计 RAG 就比一段看似自信的自然语言更接近生产需要。

对调用方而言，正确消费契约同样重要。非流式应同时展示 final answer、实际 citations、noAnswer 和 policy，并允许用户展开 compact retrieval 了解 trusted/rejected 与降级原因；流式应把 content 标为生成中草稿，以 done 的 answer/citations 覆盖，收到 error 或无终态 EOF 时不能保留为成功回答。会话历史只用于回看，不应被前端自动拼成下一轮隐藏 Prompt。

对知识运营而言，拒答也应形成可行动反馈：`refuse_without_trusted_source` 指向知识覆盖或阈值，`refuse_without_citation` 指向索引定位或模型引用契约，`retrieval_failed` 指向依赖，ID 合法但 claim 错误则进入 RAGAS/人工复核。只有把失败分流，团队才能降低真实错误，而不是通过放宽门禁让拒答率看起来更漂亮。

对面试表达而言，最有说服力的不是罗列 LangChain、Milvus 和 RAGAS，而是能画出资格门禁、无工具生成、后置引用和最终 API 契约，随后主动指出字符串 guard、流式撤回、进程内历史和未观测成本的边界。工程设计的可信度，来自知道每项机制防住了什么，也知道它明确没有防住什么。

如果未来指标变好，也必须沿用同一原则：先确认 case、知识版本、Prompt、模型、阈值和 answer source 是否可比，再讨论改动贡献；若 Artifact stale、工作区 dirty 或样本选择发生变化，应生成新的 candidate，而不是覆盖旧结果。可复现性本身就是回答可信链的一部分。

最终，系统应把“我为什么能回答”和“我为什么不能回答”都变成结构化、可复查、可测试的产品行为，而不是依赖模型临场自觉。

真正上线以后，还应定期抽样拒答、成功回答与流式异常，分别检查来源是否仍有效、引用是否支持具体结论、客户端是否正确处理最终覆盖，并把发现的问题回灌到知识治理、阈值评估和契约测试中。这个闭环不会让模型永远正确，却能让错误更早暴露、责任边界更清楚，也让每次改进都有可以复核的依据。

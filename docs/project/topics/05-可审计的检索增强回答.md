# 检索命中了，为什么仍要审计答案：从可信 chunk 到受约束回答

> 专题编号：05
> 审计日期：2026-07-22
> 上一篇：[可信混合检索](04-可信混合检索.md)把候选收敛为可信检索结果；本篇负责把它们变成可检查的知识回答。
> 下一篇：[规划—执行—再规划诊断循环](06-规划执行再规划诊断循环.md)将处理本次 Incident 的实时取证，而非继续让知识库猜测现场。
> 事实范围：2026-07-22 当前完整工作区的实现与自动化测试，包含未提交的 RAG 改进；不等于 `local_live`、`controlled_fault` 或生产事实性、安全率与 SLO。

凌晨 02:07，`order-service` 的 503 和 P95 同时上升，日志出现 `Redis connection timeout`，`connected_clients` 又接近 `maxclients`。检索命中了两段很有价值的材料：Redis 官方快照说明连接上限的机制；历史复盘说明一次 retry amplification 如何放大连接需求。它们足以回答“应该检查什么、历史上发生过什么”，却不足以回答“本次 503 已经由 maxclients 确认”“现在可以直接扩容”。

这正是 RAG 最容易被误解的地方。上一篇的 Trust Gate 给出的结论是：这些 chunk 有资格成为知识证据；它从未给模型颁发“可以确认当前根因、可以执行生产动作”的许可证。知识、当前 Incident Evidence 和根因 Hypothesis 仍是三种对象：

- 官方文档与 Runbook 是静态知识，可解释机制、排查方法和动作边界。
- 监控、日志、变更记录与事故窗口查询才是本次 Incident 的 Evidence。
- “连接耗尽导致 503”是等待 Evidence 支持或反驳的 Hypothesis。

如果把三者混成一句顺滑的自然语言，引用再漂亮也仍会误导值班人员。本章的目标不是让答案“看起来像有来源”，而是把“能不能答、可用哪些片段、最终用了哪些片段、答不出来怎么办”做成可复查的行为。

## 朴素方案为什么不够

最短路径是 `llm(top_k + question)`。它在向量库失败、结果没有定位 metadata、候选与问题跑题时，仍可能利用参数知识写出流畅建议。第二种做法是在 Prompt 中写“请标注来源”；这只是软约束，模型可以漏引、虚构 ID，或把多个事实和多个来源堆在同一句中。第三种是让通用 Agent 自由选择是否检索或调用工具；这样本轮允许的来源集合无法冻结，文档里的指令也有机会诱导它绕过知识回答边界。

AutoOnCall 的产品路径因此采用显式检索、无工具生成和前后门禁。这里的“无工具”很具体：`RagAgentService.query_grounded*` 直接向基础模型发送 system/human messages；它不是让带 MCP 工具的通用 Agent 在回答时继续访问 Redis、Prometheus 或日志。实时调查要进入下一篇的受治理 AIOps 链路。

```text
用户问题
  → structured retrieval / Trust Gate
  → prepare_grounded_generation
      → 问题子目标 + Frozen Generation Evidence
      → 生成上下文、citation 编号、allowlist 同源
      → metadata / 必要来源不足时拒答
  → tool-free model
  → claim/citation guard 与主题聚焦检查
  → 仅公布实际支持答案的 citation，或结构化 no-answer
```

这条图是“实现接线图”，不是逐 claim 事实证明图。后置检查会要求 claim 与所引 chunk 有具体词项或内置概念重叠，也会限制引用身份；它不是通用的语义蕴含模型，更不能把静态资料升级为当前现场事实。

## 先冻结证据，而不是冻结一份 Top-K 的名字

检索返回的是候选结果；真正放给模型的输入由 `prepare_grounded_generation` 构造。它先根据问题建立 Question Plan：识别 Redis、MySQL 等领域、显式实体、取证/判断/告警/动作/历史边界等子目标，以及“官方文档”“事故复盘”等来源角色。随后 `build_frozen_generation_evidence` 做四件事：

1. 对候选选取与问题相关的 excerpt，并保留稳定 `source_file + chunk_id`；
2. 按需要的来源角色、`required_sources`、子目标和显式实体挑选 chunk；
3. 在字符预算内固定顺序、分配 citation index；必要来源装不下或缺失时 fail closed；
4. 记录每个子目标绑定了哪个 chunk、缺了哪些子目标/实体。

于是模型看到的不是“检索器曾经返回过什么”，而是不可变的 Frozen Generation Evidence。`build_generation_context` 发现私有字段 `_frozen_generation_evidence` 后，直接按冻结后的内容和编号渲染；generation allowlist 与预发的 citations 也都从同一批 `items` 派生。一个被去重、截断或预算排除的候选，不应在答案结束时突然获得引用资格。

以 Redis 问题为例，官方片段可以绑定 `connected_clients`、`maxclients`、`effective_capacity`、`blocked_clients`，历史 PDF 绑定“历史/复盘”子目标。正确答案会把前者说成检查方法，把后者说成历史风险，并明确“历史复盘不能替代当前 incident-window 证据”。如果某一来源是 retrieval 明确要求的，冻结阶段会优先给该来源预留空间；没有任何一个必要来源能进入上下文，就拒答，而非偷偷丢掉它后继续生成。

### 一个容易说错的边界：Evidence Plan 与 Answer Contract 并非同一种已上线约束

当前工作区确实在 `prepare_grounded_generation` 中强制调用 Frozen Evidence Plan，且非流式与流式都经由该 preparation 生成 Prompt、allowlist 和 citation 列表。这是已接线的运行时事实。

同一处还会构造 `AnswerContract`：它把计划中的 answer slot、允许 citation index、来源角色和最大 claim 数绑定到 frozen evidence；`rag_answer_contract.py` 也提供 `validate_answer_contract` 与 `contract_repair_instructions`，并有独立单元测试验证例如“缺 MySQL 实体”“缺 postmortem 来源角色”“一条 claim 绑两个引用”等违规。

但截至本次审计，公开的 `query_with_retrieval` 和 `query_stream_with_retrieval` 没有调用 `validate_answer_contract` 或 `contract_repair_instructions`。因此不能把 Answer Contract 写成每次 API 回答都会执行的强制后门禁，也不能把它的修复 helper 写成已发生的线上重试。它是已实现、已测试的契约构件，并被 preparation 作为私有 `_answer_contract` 保存；运行时真正强制的仍是冻结证据、citation/claim guard、主题聚焦以及下文的服务级修复。

目标设计希望把必要来源、子目标、实体、claim 数和 citation 违规统一收敛成一次有界的 contract repair：先对完整答案得到稳定 violations，再用同一 frozen evidence 修复一次，最后重新验证。当前工作区尚未做到这一点；运行路径仍是多个条件分支依次判断和修复，不能用这个目标态掩盖现有调用次数与成本上界。

公开读模型也要分清。`compact_retrieval_payload` 有能力在传入相应 payload 时投影 `generation_allowlist`、`answer_coverage` 等字段，并会隐藏服务器绝对路径；然而聊天服务在 preparation 之前就把原始 retrieval payload 压缩为 `retrieval_context`，非流式最终返回它，流式 `search_results` 也发送它。当前 `/api/chat` 和 `/api/chat_stream` 并不公开 Frozen Evidence Plan、private Answer Contract 或其 bindings；不要据此承诺调用方可以审计完整计划。非流式的 `include_evaluation_context` 仅为内部评测调用额外返回冻结证据内容，不是公开 API 契约。

## citation pair：引用不是尾注，而是允许集合中的身份

生成证据被编号成 `[证据 N]`。每个编号在服务端映射回一个 citation pair：`(source_file, chunk_id)`。答案结束后，`finalize_grounded_answer` 会做如下处理：

1. 压缩旧式栏目、去掉泛化的不确定性套话；“当前证据不足：具体缺口”可以保留，但会移除其 citation，避免把未知事项伪装成有证据的事实。
2. 解析答案中引用到的编号/允许格式，映射回预发的 pair；任何未知编号、伪造 pair、没有实质 claim 的答案或缺 citation 的 claim 都不能通过。
3. 对每个 cited claim 检查它与对应 frozen chunk 是否具有具体术语或受控概念重叠，再检查答案是否仍聚焦用户问题的故障对象。
4. 只把实际被答案使用且通过检查的 citation 返回给调用方；不是把全部 Top-K 作为“参考资料”罗列出来。

这比“答案末尾有链接”强得多：引用的身份来自本轮冻结集，claim 不能无引，且无关 topic 会被拒绝。它仍然不是逐句 entailment proof：词项/概念重叠可能漏过细微曲解，也不检查事实在引用之外是否已过期。高风险答复仍需要人工复核或后续的 claim-level verifier。

## 失败不是一个 if：非流式回答的分层修复

在 `query_with_retrieval` 中，初次生成会先经过 `finalize_grounded_answer`。失败后的处理不是盲目无限重试，而是针对不同缺口做条件分支；每一次额外模型调用依旧使用同一份 grounded prompt 加上修复指令，不能扩大证据集。按实际可达控制流，一条最坏路径是“首次生成 → 必要来源修复 → coverage 修复”，最多三次模型语义生成；另一条路径是首次结果直接成为 citation no-answer 后再做一次 citation retry，共两次。citation retry 要求当前 decision 已是 `refuse_without_citation`，而前两类修复只在已有可用 answer decision 时运行，所以两条路径不会串成四次。若 citation retry 仍不能通过 guard，之后还可能进入一次不调用模型的抽取式 fallback。

| 发现的缺口 | 当前非流式行为 | 不能推出的结论 |
|---|---|---|
| 需要来源没有贡献 citation | 以缺失来源与对应 `[证据 N]` 再调一次模型；只有 retry 合法且所有必要来源都实际被引用才替换原答案 | 不表示官方与复盘已证明当前根因 |
| 答案遗漏 coverage matrix 子目标 | 以 `uncovered_answer_subgoals` 的启发式结果请求一次完整重答；仅当缺口数量变少才采用 retry | 不表示已执行 Answer Contract 的强校验；coverage 也不是语义完整性证明 |
| claim 缺引用 | `finalize_grounded_answer` 先尝试 `repair_missing_claim_citations`：只有与 allowlisted evidence 有词项/概念重叠的无引 claim 才补一个编号；仍失败再请求模型重答一次 | 自动补号不创造新证据，也不会修复无关或语义错误 claim |
| citation 仍失败 | 从 frozen evidence 抽取可引用的原文句，最多三条，优先覆盖必要来源；该抽取式答案仍要再过同一 guard | 它不是自由摘要，也不能保证把每个用户子问题讲清 |

这也解释了“必要来源覆盖”和“子目标覆盖”为什么分开。前者关注多来源问题中每个明确要求的文档是否至少贡献一条直接 claim；后者关注问题是否提到取证、判断、动作边界或历史/当前区别。一个答案可以引用官方和复盘，却忘记交代审批/回滚边界；也可以提到边界，却没使用被指定的复盘。二者都不应被“已经有 citation”掩盖。

coverage matrix 的识别目前是基于问题计划、chunk 文本/标题标记和回答中的关键词，`uncovered_answer_subgoals` 同样是启发式。它的价值是把“部分答到”变成可触发修复的信号，不是判定真伪的裁判。没有足够 Evidence 时，正确的输出是清楚的“当前证据不足：缺少当前事件的审批和回滚记录”，而不是编造一般建议或强行给这句话加 citation。

### 重试成本必须合并，而不是被第一轮掩盖

服务每发生一次语义修复，都会再次调用 `_merge_generation_observability`：它把此前已经累计的 `llm_generation_ms` 与本次修复耗时逐次相加；当各轮 provider usage 可用时，也逐次汇总 input/output/total tokens。因此最坏三次语义生成的耗时和 token 会沿“必要来源 → coverage”修复链累计；citation no-answer 分支则累计首次与 citation retry 两轮，不能只拿最后一轮代表整次请求。后续抽取式 fallback 不调用模型，所以不会再增加模型 token，但它本身仍处在请求总耗时内。`build_rag_observability` 当前对外会给出阶段耗时、provider token usage（可能仍是 `not_observed`）、估算成本的 `not_observed`，以及 `generation_repair`。

这里还存在可观测性边界：对外的 `generation_repair.reason/sources` 主要标识“必要来源覆盖”修复；coverage retry 与 extractive fallback 的内部标记并未形成同等完备的公开字段。不能把现有对象宣传为所有修复分支都可逐一审计的成本账本。当前也没有一个统一的“语义修复总预算”把这些业务分支收敛为一次 contract repair；配置中的重试次数只约束每次 provider invoke 自身的韧性重试，与最多三次业务语义生成是两个维度。

## 拒答是一种结果，不是异常文案

如果 retrieval status 不是 success，或没有可信来源，服务不会调用模型，直接返回 no-answer。pre-generation 阶段若 citation metadata 不完整、必要来源无法装入 frozen context，也会拒答。生成后遇到显式知识库拒答、不可修复的 citation/主题聚焦失败，最终同样返回 no-answer 和对应 policy，而非泄漏草稿。

这使失败可以分流：

- `refuse_without_trusted_source` 指向知识覆盖、Trust Gate 或必要来源/预算问题；
- `refuse_without_citation` 指向定位 metadata、模型引用格式、claim 对齐或 topic drift；
- retrieval 的 failed/degraded 语义仍应由调用方展示，不能伪装成正常向量检索；
- “当前 Incident 是否已确认”“是否执行扩容、重启或回滚”超出静态知识的部分，转交带实时 Evidence、风险控制和审批的 AIOps 流程。

拒答并不等于“没有任何资料”。它说明系统不能在当前冻结证据与守卫下交付一条通过资格检查的答案；这比把不确定性藏在自信措辞里更利于知识运营和故障处理。

## 流式与非流式：相同证据前门，不同的最终内容契约

两条路径都先检索、调用 `prepare_grounded_generation`、使用同一 frozen evidence 生成 Prompt，并在末尾调用 `finalize_grounded_answer`。但它们的用户可见时序不同，不能沿用“流式只是非流式逐 token 版”的说法。

| 维度 | `/api/chat` 非流式 | `/api/chat_stream` SSE |
|---|---|---|
| 业务级修复 | 已接入必要来源 retry、子目标 retry、citation retry 与 extractive fallback | 当前未接入这组服务级修复/抽取式兜底 |
| 传输前检查 | 返回前完成最终 decision | `search_results` 先发；模型输出累计时，只有带允许 citation 的完整前缀才可能作为 `content` 提前发送 |
| 终态不合格 | 只返回拒答或合格最终答案 | 发出 `replace_content` 覆盖已显示内容为拒答或规范化最终答案，再发 `done` |
| provider 流异常重试 | 普通调用按韧性策略重试 | 仅在本次 provider stream 尚未观察到任何内容时重试；已经发出 provider 内容后出错不重试 |
| 调用方完成条件 | HTTP 响应中的 `answer/citations/noAnswer/answerPolicy` | 以 `done` 为最终权威；收到 `error` 或 EOF 而无终态不得当作成功 |
| generation observability | 有 `build_rag_observability` 返回 | 当前 `done` 不携带同等 generation observability |

这意味着旧的“服务端完整 spool、重试、验证后才向客户端发送任何答案”表述已经不准确。`rag_stream_spool_max_memory_bytes` 在当前实现中是单次 provider 输出的字节上限，并不是把整份答案落盘、最终一次性回放的完整 spool 协议。流式路径可以在模型结束前释放一个引用身份已合法的前缀；最终 guard 仍可能压缩、补 citation 或替换内容。前缀检查也只验证引用 allowlist 形状，最终的 claim/evidence/topic 检查发生在完整答案到达之后。

前端因此必须支持 `replace_content`，并以 `done.data.answer`、citations、no_answer 和 policy 覆盖中间显示。若浏览器把第一段 `content` 截屏、持久化或当作已经成功的操作建议，就破坏了服务契约。当前自动化测试已覆盖未引用草稿不提前输出、合法引用前缀可提前释放、终态替换、公开错误和缺少终态的 EOF 处理；这仍不是浏览器真实网络或生产用户流量的证据。

## 从 Redis Runbook 到正确的回答

面对“`connected_clients` 接近 `maxclients` 是否已经确认导致 timeout，现在能直接扩容吗？”一个合格的知识回答应大致完成以下推理，而不是给出未经许可的行动：

1. 指出应检查哪些 Redis 指标和有效容量，引用官方 chunk；
2. 若使用复盘，明确它是历史中 retry amplification 的风险线索，而非本次证明；
3. 说明还缺本次 incident-window 的连接拒绝、应用连接池、变更/审批与回滚 Evidence；
4. 对扩容只解释 Runbook 中已有的审批、验证、回滚或只读边界，不声称自己执行了动作。

因此，“引用正确”不能替代“根因确认”；“有动作建议”也不能替代“具有执行资格”。RAG 的职责是把知识层说清楚，并诚实地把现场问题交给诊断与审批系统。

面试中可以这样概括：

> 我们把“检索到的可信结果”进一步冻结为预算内、带来源角色和 chunk 身份的生成证据；Prompt、citation allowlist 和最终 citation 同源。非流式再对必要来源、问题子目标和 citation 失败做有限修复，必要时从原文做抽取式兜底；无法通过门禁就明确拒答。流式复用冻结与终态 guard，但允许已验证引用前缀提前发送，并以 `replace_content`/`done` 修正最终答案，所以它没有宣称与非流式同样的修复或整段缓冲语义。

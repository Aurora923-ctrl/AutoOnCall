## 02:07：Runbook 已经知道“通常怎样查”，但它还不知道这次出了什么事

凌晨 02:07，`order-service` 出现 503，P95 升高，日志里有 `Redis connection timeout`，告警摘要还说 Redis 的连接数接近 `maxclients`。上一章的 RAG 能给出一份像样的 Runbook：检查连接池、检查 Redis 状态、核对发布、谨慎准备回滚。

但值班现场的问题比“Redis 连接耗尽通常怎么排查”严格得多：这些信号是否属于同一时间窗？Redis 是根因、放大因素还是巧合？日志查询失败时该重试、补旁证还是交给人？如果计划最后写了“重启”，谁有权让它发生？

把它们压成一句话会产生危险的跳跃：

```text
503 + P95 升高 + Redis timeout + 连接数接近上限
→ Redis 是唯一根因
→ 重启或扩容
```

这里混入了三种完全不同的东西：观测、待验证解释和可能改变生产环境的动作。一次自由工具调用无法替它们建立顺序、成本与权限边界。因此 AutoOnCall 把诊断写成一个有限的状态协议：先提出可执行的调查路线，取得观察，再依据观察决定是继续、补查、停止、等待审批还是转人工。

这篇的重点不是“让 Agent 多想几轮”，而是让每一轮都能回答：**这一步为什么值得做、它改变了什么状态、下一步还能否安全发生、何时必须停下。**

## 先把事故、知识、证据、假设和动作放回各自的位置

`Incident` 固定的是“正在调查哪一次异常”：服务、环境、时间窗、症状、严重级别和稳定 `incident_id`。它不是根因结论。`order-service` 在 02:07 出现 503 可以是 Incident 的事实；“Redis maxclients 已确认”不能预先写进 Incident。

下面的分层避免把 Runbook、工具回执和变更请求互相冒充：

| 层 | Redis 案例 | 在本循环中的作用 | 不能据此做什么 |
|---|---|---|---|
| 告警与 Incident | 503、P95、timeout 摘要、调查时间窗 | 界定对象与初始调查方向 | 直接确认根因 |
| 知识 | Runbook 阈值、历史工单 | 约束优先级、补充解释和处置前提 | 替代本次现场观测 |
| 工具回执 | 指标、日志、Redis 状态或调用失败 | 成为可追溯的执行记录与观察候选 | 因为调用成功就证明结论 |
| Evidence / Hypothesis | 支持、反驳、缺口与候选解释 | 驱动是否续查及报告的充分性 | 宣称通用因果发现 |
| 动作 | 重启、扩容、改配置、执行 SQL | 进入风险策略和人工审批 | 因为在计划里就自动获得授权 |

本章负责前四层如何在循环里流动；[第 07 篇](07-智能体工具工程.md)负责一次工具调用的 Contract、适配器与副作用边界；[第 08 篇](08-从工具结果到根因假设.md)负责把回执解释为 Evidence、再组织成可反驳 Hypothesis。这样，Plan-Execute-Replan 不把“工具调用成功”偷换成“根因已证实”。

## 它不是自由循环：状态协议先于模型能力

一个能反复聊天的 Agent 未必是可靠的工作流。可靠性来自共享状态中哪些字段可替换、哪些字段只能累积，以及图只允许哪些转移。

### 主队列、兼容队列与累积历史

`PlanStep` 是机器可消费的一步调查，而不是“看一下 Redis”这样的自然语言待办。它带稳定 `step_id`、`tool_name`、`input_args`、`purpose`、`expected_evidence`、`risk_level`、`status` 和 `retry_count`。`retry_count` 的模型取值可到 3，但这只是数据校验范围；当前工作流级策略只会产生一次 `-retry` 步骤，并不意味着会自动重试三次。

```json
{
  "step_id": "s2",
  "tool_name": "query_redis_status",
  "purpose": "核对事故窗口相关的 Redis 连接压力",
  "input_args": {"service_name": "order-service", "time_range": "10m"},
  "expected_evidence": "connected_clients、maxclients 与阻塞连接状态",
  "risk_level": "low",
  "status": "pending",
  "retry_count": 0
}
```

状态并非一团不断拼接的 Prompt。以下约定尤其关键：

| 字段组 | 更新语义 | 为什么 |
|---|---|---|
| `current_plan` | 规范的结构化待执行队列；每次更新同时重新派生 `plan` | Executor 依赖前者，旧 SSE / 兼容路径仍可读后者；两者不能漂移 |
| `plan` | 人可读的兼容队列 | 不是重新解释计划的第二个真相来源 |
| `past_steps`、`executed_steps` | 追加 | 保留每一个实际消费过的步骤，fan-out 也逐项计数 |
| `tool_call_records`、`gathered_evidence` | 追加 | 失败回执同样要留下，不能让下一轮只看成功样本 |
| `errors`、`warnings`、`progress_events` | 追加 | 让降级与过程可审计 |
| `response`、`report`、`pending_approval`、风险/审批字段 | 以最新业务状态覆盖或清除 | 它们决定终态、暂停与恢复材料 |

`incident` 在图状态中保持 JSON-safe 字典，而不是让 Pydantic 对象穿越 checkpoint 与持久化边界。持久 Snapshot 合并节点增量时还会对追加字段去重，避免 checkpoint 与服务层同时带回同一条 Evidence 后制造“双份事实”。

### 图只有一个语义起点

当前图的拓扑是：

```text
Planner → Executor → Replanner ──→ END
              ↑          │
              └──────────┘
```

Planner 只在入口运行一次。它可以用结构化 LLM 给 Incident 建立调查路线，却不在每轮重新写一份全局计划；Replanner 只修改剩余队列并决定下一状态。这样做牺牲了开放式探索，却避免计划在每轮因模型措辞而抖动，也保留了“为什么新增这一步”的决策记录。

条件边只允许三种继续前提同时成立：没有 `response`、没有 `pending_approval`，并且结构化 `current_plan` 或兼容 `plan` 仍有项。否则图结束。这里的 END 不等于“根因确认”：它也可能表示等待审批、被禁止、降级报告、人工接管或运行失败。

## 三个角色各自只做一件事

### Planner：一次把问题翻译成可检验的路线

Planner 读取 Incident、用户描述、检索到的 Runbook、标准工具 Contract 以及可发现的安全 MCP 工具，再要求 LLM 输出 `list[PlanStep]`。Runbook 命中会留下知识类 Evidence，明确说明它用于约束计划，仍需实时工具验证现场。`risk_actions` 只能形成建议或审批候选，不能变成自动修复。

它的降级顺序也属于契约：

- Runbook 检索失败，不伪造知识命中；仍可继续规划。
- MCP 工具发现失败，记录 warning，保留本地与标准工具 Contract。
- 模型创建、调用或结构化解析失败，使用确定性的 `build_fallback_plan`。
- fallback 依据 Redis、MySQL、Kubernetes 等有限故障域组织只读检查，保留 Incident 显式请求的高风险动作供风险控制处理，而不是把它悄悄删掉。

fallback 的价值是“模型或外部发现不可用时仍有有界路径”，不是证明规则计划和模型计划同等聪明。测试覆盖了 Redis timeout、MySQL 慢查询、显式 Redis 实例和 MCP 发现失败仍能得到同步的结构化/兼容计划；它们是自动化回归证据，不是实时依赖成功率。

### Executor：只消费队首（或相邻安全批次），不临场改题

Executor 通过 Registry 找到规范工具，先做风险判断，再执行；每个结果都会写为 `ToolCallRecord`，并转换为 `gathered_evidence`。成功、failed、无数据、兼容 fallback 的身份必须保留。Executor 不负责宣布根因，也不能因为模型会调用工具就自由发明新任务。

为避免相互独立的只读查询被串行延迟放大，当前实现可以对相邻步骤做有界 fan-out：仅允许白名单中的低风险只读取证工具，逐项再过策略，最多 4 个；结果仍按原计划顺序归档。一个工具失败会产生 failed Evidence，而不是丢弃整个批次。重启、扩容、写 SQL 和配置修改不在这个批次里。

因此“并发”不是吞吐承诺，更不是任意工具并行。fan-out 中每个实际执行的步骤都会进入 `executed_steps`，与串行步骤一样消耗总预算。

### Replanner：不是第二个 Planner，而是有限的状态裁决器

Replanner 读取已收集 Evidence、失败记录、剩余队列、风险和 Evidence Analyzer 的结构化分析，选择下面六个枚举之一：

| 决策 | 含义 | 对队列和流程的影响 |
|---|---|---|
| `continue_investigation` | 剩余计划仍值得执行 | 保持队列，回到 Executor |
| `add_steps` | 有新的、安全的只读证据缺口 | 新步骤置于剩余队列前；保留既有步骤 |
| `retry_failed_tool` | 某失败调用尚未做工作流级重试 | 前置一次 `-retry`，保留原队列 |
| `generate_report` | 调查可收敛到报告 | 仍先检查剩余风险动作 |
| `request_approval` | 后续处置可能改变线上状态 | 创建风险/变更/审批材料并暂停 |
| `escalate_to_human` | 无足够证据且没有安全续查路径 | 形成明确的人工接管结果 |

普通补查会按工具名与规范化参数过滤已经执行过的语义重复项；若新增项全被过滤且没有剩余队列，流程转人工，而不是查询同一指标直到撞上上限。显式重试是唯一例外，因为它的语义就是允许再次执行同一调用；但它也只允许一次工作流级 `-retry`，并且仍须通过风险策略。

默认裁决来自确定性 Evidence Analyzer，配置 `AIOPS_REPLANNER_LLM_ENABLED` 的默认值为 `false`。若启用可选 LLM critic，输出仍须是 `ReplanDecision`，并经过后置校验：证据不足不能生成报告；没有待执行队列不能继续；没有失败记录不能凭空重试；新增步骤必须在 Registry 中且策略允许。`retry_failed_tool` 与 `request_approval` 直接跳过 critic，模型失败或解析失败退回 Analyzer。这是“LLM 提供受约束建议”，不是双 LLM 自主反思。

## 一次 Redis 调查怎样前进，而不是怎样讲故事

### 第 0 轮：方向不是结论

初始状态只有 Incident、`session_id`、`trace_id` 与空队列。告警里的 Redis timeout 只是需要验证的线索。Planner 可能安排服务上下文、Redis 状态、指标、日志、发布历史、Runbook 与历史工单；最后的处置建议也只是建议，尚未执行。

### 第 1 轮：用现场观察缩小未知

相邻的低风险步骤可以并发：指标显示同窗 P95/5xx 上升，日志命中 timeout，Redis 返回连接压力。它们能支持“连接耗尽或连接池问题值得继续验证”的假设，但还不能证明唯一因果关系。必须继续保留时间窗、数据来源和反证：今天 Redis 已正常，不会自动反驳事故窗口里曾出现的压力；三个 mock 回执“成功”也不会变成生产现场。

### 第 2 轮：知识解释，现场取证

Runbook 与历史工单帮助回答“这种组合通常如何排查、处置前要满足什么条件”，却不替代 Redis、日志和指标的现场证据。若日志工具失败，失败本身进入记录，工具内部有限重试耗尽后，Replanner 才可能额外安排一次工作流级重试或选择另一项安全旁证。若证据冲突，正确结果是暴露冲突、降低报告状态或人工接管，而不是删除不合叙事的回执。

这就是与第 08 篇的分界：本篇保证每轮留下可靠的“做过什么、得到什么、还剩什么”；下一篇才解释哪些观察支持/反驳候选，以及为什么排序不是确认。

## 停止条件与预算：防止循环把不确定变成成本和风险

没有停止条件的 Agent 只是不可预测的 `while` 循环。当前实现至少有四类停止/暂停语义：

1. 已有 `response`：保留已有终态或风险拦截结果，结束图。
2. 有 `pending_approval`：暂停自动路径，保留 waiting-approval 材料。
3. 两个计划队列都空：不再虚构下一步，结束图。
4. `MAX_STEPS = 8`：优先按 `executed_steps` 统计实际执行数；旧状态缺失该字段时才兼容 `tool_call_records`、`past_steps`。到达上限后仍先过剩余风险门，而不是直接输出普通成功报告。

八步是成本和失控护栏，不是“八步内必然找到根因”的保证。弱证据、冲突和失败应体现为 `degraded`、`incomplete`、`needs_human` 等受限结果，不能用更长的循环伪造确定性。

还应区分三层重试与三种预算：

| 层/预算 | 当前语义 | 不能误说成 |
|---|---|---|
| MCP interceptor | 可重试的远端连接/超时类失败可退避；权限、参数、not-found 等立即停止 | 每一次工具调用都会自动重试 |
| 标准工具 | 只读工具按显式策略在同一总 deadline 内重试；非只读最多一次；attempt 元数据随回执记录 | 每个 attempt 都获得完整新的超时预算 |
| 工作流 | Analyzer 在结构化失败后至多加一次 `-retry` | `retry_count ≤ 3` 就会执行三次 |
| 执行数量 | 实际 `executed_steps` 最多 8，fan-out 逐项计数 | 图循环次数最多 8 |
| 上下文体积 | ContextBudgeter 对片段做字符截断，原始告警与普通片段有不同上限 | 已精确控制 tokenizer token、费用或端到端 SLA |

取消会向上传播，不应被伪装为依赖失败和继续回退；同时，当前没有可据此声称的持久化 AIOps 任务取消 API。精确 token/费用预算、跨恢复累计的 wall-clock 预算和生产 SLA 仍是后续工程缺口。

## 审批暂停与恢复：确认人工决定，不替人执行变更

Risk Controller 对每个结构化步骤判断 `allow`、`approval_required` 或 `forbidden`。这与 RAG 的 Trust Gate 完全不同：前者约束动作权限，后者约束知识候选是否能进入回答。Runbook 被检索到，绝不表示其中的重启命令已获授权。

当剩余步骤包含需要审批的动作时，Executor 或 Replanner 会写入 `risk_assessment`、`change_plan`、`pending_approval` 与等待响应，并终止本次图运行；无法安全解析的 legacy 文本计划会 fail-closed，而不是跳过风险判断继续执行。即使 Evidence 已足够、或八步预算耗尽，剩余风险动作也要先经过这道门。

审批后的诊断恢复也有严格边界：

- 仅接受属于同一 Incident / session、状态为 approved 且未过期的审批；并对并发 resume 进行单次 claim，防止同一审批同时恢复两次。
- 恢复材料优先来自持久 `AIOpsSessionSnapshot`；没有可用 Snapshot 时才校验并使用持久报告 fallback。
- 服务记录 `diagnosis_resumed` Trace、生成带审批决定的 post-approval 报告并更新 Snapshot；它不会从 `MemorySaver` 恢复任意图节点，也不会重新调用已发生的工具。
- 更重要的是：该 diagnosis-resume 接口**不执行生产变更**。真正的安全变更执行、幂等和回执属于第 11 篇的独立工作流。

所以“支持审批恢复”准确指的是有限诊断生命周期闭环；它不等于跨实例 durable workflow engine，也不等于审批通过后 Agent 自动修复生产。

## 运行态材料各自证明什么

SSE、Trace、Snapshot 和 Replay 常被统称为“可观测”，但它们回答的是不同问题。

| 材料 | 回答的问题 | 不提供的保证 |
|---|---|---|
| SSE progress / terminal event | 当前连接看见进行到哪个节点、是否产生报告或审批 | 历史唯一真相、逐字节断线续传或 exactly-once |
| Trace | 做过什么、为何补查/暂停/报告 | 某个根因结论一定为真 |
| 持久 Snapshot | 此 session 最新的业务状态与有限恢复材料 | 每一版计划的完整 diff、任意节点精确续跑 |
| Replay 读模型 | 从已有 Trace、Snapshot、报告、审批与变更记录复盘 | 重新查询今天的 Redis 并还原 02:07 现场 |

进程内 `MemorySaver` 只是运行时图 checkpoint；持久 `AIOpsSessionSnapshot` 才承载恢复读取的业务状态。两者不能互换。Snapshot 也偏向“最新状态”，若生产审计需要每次计划变更的完整谱系，还应增加计划版本、父版本、来源和变更理由，不能从最后一个 `current_plan` 倒推历史。

## 选择受约束循环，是可回归性与开放性的取舍

| 方案 | 优点 | 缺点 | 适用范围 |
|---|---|---|---|
| 固定脚本 | 便宜、稳定、容易回归 | 不会根据证据提前停止或转向 | 封闭 Runbook、规则 fallback |
| 每轮自由 ReAct | 开放探索能力强 | 工具、成本、权限与重复调用难控 | 低风险沙箱、强人工监督 |
| 每轮重做 Planner | 可全面改写路线 | 模型成本高，计划易抖动 | 治理成熟且故障域快速变化的系统 |
| 一次 Planner + 有限 Replanner | 有语义起点、可验证续查与停止 | 规则目录需维护，未知域覆盖有限 | AutoOnCall 当前主线 |

LangGraph 提供节点、共享状态、条件边和进程内 checkpoint；LangChain/模型负责结构化生成。真正约束系统的是 Incident、PlanStep、Registry、Risk Controller、Evidence 分析、报告状态与持久业务记录。使用框架不等于已经获得自主 RCA、生产级持久工作流或生产准确率。

## 面试中怎样准确描述

**一句话：** AutoOnCall 用一次结构化规划和一个由状态协议、Evidence 缺口、风险门与硬停止条件约束的执行/再规划循环，把 Runbook 问答变成可审计的多步调查。

**30 秒展开：** Planner 只在入口把 Incident、Runbook 与工具 Contract 变成 `PlanStep`，模型不可用时降级为规则计划；Executor 只消费经过风险判断的步骤，把成功和失败都沉淀为执行记录与观察；Replanner 默认依赖确定性 Analyzer，在继续、补查、一次重试、报告、审批和人工接管之间选择。循环最多按实际执行步骤计 8 项，相邻只读查询最多 4 路 fan-out；高风险动作暂停到审批，审批 resume 只更新诊断报告闭环，不执行生产变更。

**不要夸大：**

- 不说“每轮 Planner 自主反思”，Planner 只运行一次；
- 不说“Replanner 做通用因果推理”，它是受限的结构化分析与规则裁决；
- 不说“模型字段允许 3 次重试，所以会重试 3 次”；
- 不说“MemorySaver 支持跨进程任意节点恢复”；
- 不说“审批通过后 Agent 自动修复生产”；
- 不把自动化测试或 offline fixture 写成 local-live / production 证据。

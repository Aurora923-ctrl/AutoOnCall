# 为什么一次工具调用不能完成故障诊断：规划、执行与再规划

> 专题编号：06  
> 审计日期：2026-07-14  
> 事实范围：当前仓库实现、自动化测试与已冻结评测 Artifact，不代表生产运行证据  
> 上一章已经解决：知识如何被检索、引用，以及没有可信来源时如何拒答。  
> 下一章将解决：计划中的工具名怎样经过统一契约、超时、重试和风险策略变成受治理调用。  
> 目录计划：[AutoOnCall 专题文档目录计划](../AutoOnCall专题文档目录计划.md)

## 从“Runbook 怎么说”走向“这次事故究竟发生了什么”

凌晨 02:07，`order-service` 开始返回 503，P95 延迟升到 3 秒以上。应用日志连续出现 `Redis connection timeout`，Redis 侧的 `connected_clients` 又接近 `maxclients`。上一章的 RAG 已经可以回答“Redis 连接耗尽通常怎样排查”，但值班员此刻需要的不是一篇通用说明，而是一项针对本次事故的判断：这些现象是否发生在同一时间窗？Redis 是根因、放大因素，还是巧合？证据不够时接下来应该查什么？什么时候必须停止自动调查？

朴素的单次问答很容易写出一条看似完整的因果链：

```text
503 + P95 上升 + Redis timeout + connected_clients 接近 maxclients
→ Redis 连接耗尽
→ 扩容或重启
```

问题在于，箭头两侧混合了不同性质的信息。503、延迟、日志和连接数是观测；“连接耗尽导致 503”是待验证解释；“扩容或重启”是可能改变线上系统的动作。只要把这三层压进一段自然语言，系统就会同时失去证据边界、停止边界和权限边界。

### 先把事实、知识、推断和动作分开

本章贯穿案例始终使用四个篮子：

| 篮子 | Redis 案例中的内容 | 可以做什么 | 不能做什么 |
|---|---|---|---|
| 事实 | 事故窗口的 503、P95、日志、Redis 状态 | 被复核、支持或反驳假设 | 单条事实直接宣布唯一根因 |
| 知识 | Runbook 的阈值说明、历史工单的处置经验 | 约束调查顺序与解释 | 替代本次现场观测 |
| 推断 | “Redis 连接耗尽可能导致 timeout 与 503” | 形成可检验假设 | 冒充已确认事实 |
| 动作 | 补查指标、查询日志、扩容、重启 | 进入计划与风险判断 | 因为写进计划就自动获得授权 |

这四类信息不会一次性齐全。诊断真正的工作，是在每一轮执行后重新回答五个问题：当前知道什么？哪些是事实、知识和推断？还缺什么？为什么下一步值得执行？什么情况下停止、降级或交给人？

### Incident 固定“调查的是哪一次异常”

**Incident（故障事件）**通俗地说是一张正在调查的事故单；严格地说，它是包含服务、环境、时间窗、症状、严重级别、原始告警与稳定标识的诊断对象。它解决多轮工具调用是否仍围绕同一次异常的问题，不负责证明根因，也不等于一条原始告警。

例如，“`order-service` 在 02:07 出现 503 与 P95 上升”可以进入 Incident；“根因已经是 Redis maxclients”不能作为 Incident 的既定事实。反例是把十条重复 firing 告警当成十次彼此无关的调查，或把报告结论反写成原始事件。AutoOnCall 把 Incident 以 JSON-safe 字典放入共享状态，Planner、Executor、Replanner、Trace 和报告都引用同一 `incident_id`。

Incident 给循环一个稳定问题，却没有给出答案。下一步需要把“调查”建模为可改变的状态，而不是一段不断增长的聊天文本。

## 为什么固定脚本和自由 Agent 都会在具体地方失败

在引入框架前，先看两个最自然的方案。它们并非一无是处；理解其适用边界，才能知道为什么需要中间形态。

### 固定脚本的问题不是不智能，而是不会根据证据提前结束

最容易实现的脚本是：查询指标，查询日志，查询 Redis，检索 Runbook，最后输出报告。对一个封闭 Redis Runbook，这种方式便宜、确定、容易测试。但它把“步骤顺序”当成永远正确的事实。

如果事故实际是 MySQL 慢查询，先查 Redis 就是无效调用；如果是 Kubernetes `CrashLoopBackOff`，关键证据可能是 Pod 状态、重启原因和容器日志。更典型的失败发生在中途：第一批指标、日志和 Redis 状态已经互相支持时，脚本仍会执行所有剩余步骤；Redis 工具超时时，它不知道应当重试、改查旁证还是直接转人工；计划末尾若写了“重启服务”，它还可能把只读调查和生产变更串成同一条无条件流水线。

固定脚本适合作为有限场景的 fallback，不适合独立承担“何时转向、何时停止”的判断。

### 自由 ReAct 的问题不是能力不足，而是约束难以复现

另一个极端是每轮把完整历史交给模型，让它自由决定下一工具。开放探索中，这种方式很有吸引力，但 OnCall 会遇到几类具体故障：

- 模型把 `query_redis_status` 写成不存在的工具名，或改变参数字段；
- 同一个日志窗口被反复查询，成本上升却没有新增信息；
- 历史持续膨胀，失败堆栈和大工具输出挤掉关键证据；
- 已有足够证据时继续追求“更完美”的答案；
- 把“检查重启效果”偷换成“先执行重启”；
- 相同输入在回归测试里走出不同路径，难以定位是工具、Prompt 还是策略变化。

自由 ReAct 适合低风险探索，但故障诊断还需要机器可验证的工具契约、有限决策集合、硬停止条件和可审计记录。由此可以推导出一个折中：首次用语义能力建立方向，执行只消费结构化步骤，每轮用受约束规则判断继续、补查、停止或转人工。这就是本章的 Plan-Execute-Replan。

## 先建立一份能被机器更新的诊断状态

多步诊断不是“多调用几次 LLM”，而是一个状态机。状态里哪些字段代表队列，哪些字段代表累积事实，哪些字段代表终止信号，决定了循环能否正确工作。

### Plan 是可执行假设，不是自然语言待办

**Plan（计划）**通俗地说是一张排查清单；严格地说，它是带有顺序、稳定标识、工具契约、输入参数、预期证据、风险等级、执行状态和重试计数的步骤集合。它解决 Executor 不必二次猜测“该调用什么”的问题，不保证步骤一定正确，也不构成生产动作授权。

“看看 Redis，必要时处理一下”是反例：工具、参数、预期观测和风险都不明确。AutoOnCall 用 `PlanStep` 表达一步，核心形态可以简化为：

```json
{
  "step_id": "s1",
  "tool_name": "query_redis_status",
  "purpose": "检查连接数是否逼近 maxclients",
  "input_args": {"service_name": "order-service", "time_range": "10m"},
  "expected_evidence": "connected_clients、maxclients、blocked_clients",
  "risk_level": "low",
  "status": "pending",
  "retry_count": 0
}
```

`step_id` 让工具记录、Evidence、Trace 与报告可以回链；`expected_evidence` 告诉再规划器“执行后应该补上哪块空白”；`risk_level` 只是风险控制输入，不是授权。模型层约束 `input_args` 序列化后最多 20,000 字符，`retry_count` 取值为 0–3。这里必须区分“数据模型允许到 3”和“当前工作流策略会重试 3 次”：后者并不成立，当前 Evidence Analyzer 只创建一次工作流级 `-retry` 步骤。

### Evidence 记录观察，不替系统宣布结论

**Evidence（证据）**通俗地说是能拿给另一位值班员复核的观察；严格地说，它是带来源工具、步骤、类型、数据源、立场、事实、推断、不确定性、建议下一步和置信信息的结构化记录。它解决“结论依据从哪里来”的问题，不负责单独确认唯一根因。

`connected_clients=9940, maxclients=10000` 是事实型 Evidence；“所以 Redis 必然是唯一根因”是推断。工具失败同样会形成 Evidence，只是状态为 failed、立场通常是 unknown、置信度更低。反例是丢弃失败记录，让报告看起来所有查询都成功。AutoOnCall 在 Executor 中把统一工具结果转换成 `gathered_evidence`，同时生成 `tool_call_records`；二者分别服务证据分析和执行审计。

### Hypothesis 把支持、反驳和缺口组织起来

**Hypothesis（根因假设）**通俗地说是当前最值得验证的解释；严格地说，它是包含候选根因、支持 Evidence、反驳 Evidence、缺失 Evidence、类别和启发式置信信息的可检验对象。它解决多种信号如何围绕一个候选解释组织的问题，不负责把统计相关性自动升级为因果真相。

“Redis maxclients 或连接池耗尽导致 timeout 和 503”是 Hypothesis；只有一条 timeout 日志就写“根因确认”是反例。AutoOnCall 的 Evidence Analyzer 依据有限信号目录建立并排序候选。这个排序是工程启发式，不是通用因果发现模型。

### 哪些字段覆盖，哪些字段必须追加

共享状态中，`current_plan` 是规范的结构化待执行队列，`plan` 是面向旧路径和 SSE 的兼容文本队列；两者必须同步更新。`response`、`pending_approval` 和空计划可成为图的终止信号。

另一组字段不能被每个节点随意覆盖：`past_steps`、`executed_steps`、`tool_call_records`、`gathered_evidence`、`progress_events`、`errors` 和 `warnings` 使用追加语义。否则 Executor 第二轮返回的一条新 Evidence 会抹掉第一轮观测，Replanner 就会基于残缺历史作决定。`incident` 保持字典形态，避免 Pydantic 对象在 checkpointer 和持久化序列化之间出现兼容问题。

可以把状态更新抽象成：

```text
新状态 = 覆盖型字段的最新值 + 追加型字段的历史与本轮增量
```

这条规则也解释了为什么持久 Snapshot 合并节点 delta 时要对追加字段去重：LangGraph checkpoint 可能已经包含本轮增量，若服务层无条件再拼一次，就会制造重复 Evidence 和步骤。

## 三个角色如何形成一个有界循环

Plan-Execute-Replan 的核心不是三个英文名称，而是职责隔离。

### Planner 只负责第一次把问题变成调查路线

Planner 像医生第一次开检查单：读取 Incident 和用户任务，检索相关 Runbook，读取本地、MCP 与标准工具契约，再让结构化 LLM 输出 `list[PlanStep]`。它负责提出初始路线，不执行工具，不确认根因，也不决定高风险动作已获授权。

AutoOnCall 的主路径使用 `with_structured_output(Plan)`，而不是让模型返回任意散文。检索到的 Runbook 还会形成知识类 Evidence，置信说明明确写出“用于约束计划，但仍需实时工具验证现场”。这避免了一个常见错误：把知识库命中当成本次事故的运行事实。

外部能力一定会失败，因此规划必须有退化路径：

- Runbook 检索失败：继续规划，但不伪造知识命中；
- MCP 工具发现失败：保留本地工具与标准工具契约，并写入 warning；
- 模型创建、调用或结构化解析失败：进入确定性 `build_fallback_plan`；
- Runbook 的 `diagnosis_steps` 可转成只读候选，`risk_actions` 只能成为建议或审批候选。

Redis fallback 会覆盖服务上下文、Redis 状态、指标、日志、发布、Runbook 和历史工单等有限路径；MySQL 则换成 MySQL 状态与发布关联。fallback 的目标是外部模型不可用时仍可有限收敛，不是证明规则与模型能力等价。

### Executor 只执行下一步，并把结果变成可审计记录

Executor 像检验科：读取队首 `PlanStep`，通过 Tool Registry 找到结构化工具，先执行风险判断，再调用工具，把结果转换成 Evidence、ToolCallRecord 和已执行步骤。它不应该重新解释 Incident 并自由发明一个新调查目标。

正常情况下，一轮消费一个步骤。为了避免独立只读查询被串行延迟放大，当前实现允许把相邻的低风险、只读取证步骤做有界 fan-out；候选仅限指标、日志、Redis/MySQL/Kubernetes 状态、Runbook、历史工单、服务上下文和发布历史等白名单工具，最多并发 4 个。并发并不改变计划顺序，结果按原队列顺序归档；其中一个失败会变成 failed Evidence，不使整个批次丢失。

这里的边界很重要。最多 4 个是实现常量，不是系统吞吐承诺；只读 fan-out 不是任意工具并行，更不能把重启、扩容或 SQL 混进去。fan-out 一次执行多个步骤时，`past_steps` 会记录每个实际步骤，因此八步上限按实际完成项累计，而不是简单按图循环轮数累计。

未注册工具或 `manual_analysis` 会进入兼容 fallback。人工分析结果的证据置信度受限；未注册结构化工具即使 LLM fallback 返回一段文本，也会保留失败状态和 warning，不能把“生成了文字”包装成“真实工具成功”。

### Replanner 不是重新调用 Planner，而是决定下一种状态

Replanner 像拿到检验结果后的会诊：读取 Evidence、工具失败、剩余计划、假设和风险，返回有限枚举决策：

| 决策 | 何时使用 | 结果 |
|---|---|---|
| `continue_investigation` | 证据不足且剩余计划仍有必要 | 保持队列，继续 Executor |
| `add_steps` | 证据不足、计划已空，但还有安全只读缺口 | 用补证步骤替换待执行队列 |
| `retry_failed_tool` | 存在尚未做工作流级重试的失败工具 | 只创建一个 `-retry` 步骤 |
| `generate_report` | 证据充分，或失败/冲突后只能降级收敛 | 生成 completed、degraded、incomplete 等报告 |
| `request_approval` | 后续动作可能改变线上状态 | 写审批与变更计划，暂停本次图运行 |
| `escalate_to_human` | 无足够证据且无安全补查路径 | 形成 needs-human 结果 |

真实图拓扑是：

```text
Planner → Executor → Replanner ──→ 终态
              ↑          │
              └──────────┘
```

它不是 `Planner → Executor → Replanner → Planner`。Planner 只在入口运行一次；Replanner 直接保留、替换或补充 `current_plan`。这降低了每轮全局重写带来的计划抖动，也使回归测试能够明确断言“为什么补了这一步”。代价是信号目录和补证规则需要持续维护，对未知故障的开放探索能力有限。

## 为什么“再规划”默认不交给第二个自由 LLM

AutoOnCall 默认由确定性 Evidence Analyzer 产生再规划决策，配置 `AIOPS_REPLANNER_LLM_ENABLED` 默认是 `false`。这不是因为 LLM 无法分析，而是补查、停止、重试和审批都需要可重复的安全边界。

### 证据充分性不是“成功工具数量大于某个数”

Analyzer 会先汇总成功工具、失败工具、候选假设、缺失工具、冲突和数据来源质量。普通 Redis 场景中，Redis 主域证据加指标或日志旁证，可以形成较强收敛条件；项目的 Redis golden incident 还要求 Runbook 与历史工单，用于验证知识和相似经验覆盖。MySQL golden incident 则要求 MySQL 状态、指标、日志、发布历史、Runbook 和历史工单。若仅有“至少三类成功证据并形成假设”，也可能得到较低的启发式充分性判断。

这不是一套普遍适用的科学定律，而是当前 fixture 与演示域中的规则。它负责让判断可回归，不负责保证世界上所有 Redis 故障都遵守相同组合。

### 来源质量会把看似充分的证据重新降级

工具调用成功不等于来源可信。Analyzer 会建立 Evidence 质量画像：如果关键证据全部来自 Mock、合成、未配置或未知来源，`source_quality=fallback_only`，证据充分性被撤销，置信度封顶为 `0.50`；真实与回退来源混合时，`source_quality=mixed_with_fallback`，置信度封顶为 `0.72`。这两个上限来自当前实现，不能解释为概率校准后的真实正确率。

反例是三个 Mock 工具都返回“Redis maxclients”，系统便输出高置信 confirmed。AutoOnCall 允许这类结果形成低置信报告，却不把它标为充分生产证据。日志指向 Redis timeout、Redis 状态却明确正常时，会记录证据冲突并降低置信；冲突报告的价值是把分歧暴露给人，不是自动挑一个喜欢的信号。

### 可选 LLM 只是有护栏的 critic

启用可选 LLM Replanner 后，模型看到的是经过字符预算裁剪的 Incident、计划、历史、Evidence 和工具记录，而不是无限完整上下文。它返回结构化 `ReplanDecision`，随后还要经过确定性检查：证据不足不能改成 `generate_report`；没有剩余计划不能声称继续；没有失败记录不能凭空重试；新增步骤必须是已注册、风险策略允许的步骤。`retry_failed_tool` 和 `request_approval` 属于安全优先决策，直接跳过 critic。

因此准确描述是“结构化 LLM Planner + 默认确定性 Evidence Replanner + 可选受约束 LLM critic”，不是“双 LLM 自主反思”。

## 一个可靠循环必须先写清楚怎样停下来

没有停止条件的 Agent 只是成本和风险不确定的 while 循环。停止也不只意味着“成功”：完成报告、等待审批、降级、需要人工、禁止动作和运行错误都是不同终态。

### 四类直接停止条件

图的条件边检查三件事：状态已有 `response`，立即结束；存在 `pending_approval`，暂停自动执行；`current_plan` 与兼容 `plan` 都为空，结束。Replanner 内还有 `MAX_STEPS = 8`：当 `past_steps` 数量达到 8 时强制生成响应。

八步的含义是成本和失控护栏，不是“八步内一定能找到根因”。如果第八步仍只有弱证据，报告必须保留 degraded、incomplete 或 needs-human 边界。把上限调成 100 并不会自然提高正确率，反而可能制造重复调用、上下文膨胀和更长恢复链。

### 字符预算能限制输入体积，却不是 token 或费用预算

通用 `ContextBudgeter` 默认每段 `3000` 字符，原始告警 `4000` 字符；超限时统一截断，而且截断标记也计入总预算，最终输出不会超过声明上限。它还支持在一份共享预算内只保留完整有序 sections，供需要避免证据块被拦腰切断的路径复用；Replanner 仍使用 `REPLANNER_CONTEXT_CHAR_LIMIT = 3000`。负数和非整数 limit 会显式失败，避免错误配置被悄悄收敛成空上下文。这些规则让离线测试和非 LLM 路径保持确定性。

字符不等于模型 token，因此当前实现不能声称精确限制 token、费用或端到端 SLA。生产预算还应考虑模型 tokenizer、每轮总预算、工具 wall-clock、并发 fan-out 和恢复后的累计消耗。

### 两层重试解决的是不同失败

第一层在工具内部。只读工具可以声明 `max_attempts`、backoff 和可重试错误类型；所有 attempt 共享一个 `timeout_seconds` 总 deadline，而不是每次重试重新获得完整超时。非只读工具强制最多一次。每次 attempt、失败类型、耗时、是否耗尽都会进入 retry metadata。

第二层在工作流。若最终工具结果仍失败，Analyzer 最多创建一个 `step_id` 以 `-retry` 结尾、`retry_count=1` 的步骤；同一工具的 `-retry` 也失败后，不再循环创建 `-retry-retry`。若已有假设，则基于现存证据生成不完整诊断；没有足够证据和安全补查时转人工。

内部重试适合吸收短暂网络抖动，工作流重试适合让 Replanner 看到“该工具整体失败”后再决定是否值得试一次。把两层混成一个“重试三次”会隐藏总超时、重复副作用和降级语义。

### Trust Gate 与风险门禁不是同一个东西

**Trust Gate（可信门禁）**通俗地说是“检索结果即使排第一，也要足够可信才准进入回答”；严格地说，它依据原始向量/词法信号和阈值接受或拒绝 RAG 候选。它解决知识候选的准入，不负责验证运行态 Evidence，也不是生产动作授权。在 AutoOnCall 中它属于上一章的混合检索链；本章 Planner 检索 Runbook 时继承其 no-answer 边界。

本章还有另一类门禁：Risk Controller 根据步骤、工具契约、Incident 和策略判断 allow、approval-required 或 forbidden。它解决动作能否自动执行，不负责判断知识相关性。反例是“Runbook 通过 Trust Gate，所以其中的重启命令可以自动执行”。知识可信与动作获权是两次独立判断。

## Redis 主线：看系统怎样一轮一轮缩小未知

下面不按源码类名讲，而按真实调查过程推进。每一轮都标出已知、推断、缺口和下一步理由。

### 第零轮：告警只能形成 Incident，不能形成根因

初始状态包含用户任务、Incident、`session_id`、`trace_id`，以及空计划、空 Evidence、空工具记录。此时事实只有输入携带的 503、P95、日志提示和连接数提示；如果原始告警只是摘要，这些值仍需要工具复核。知识可能来自上一章检索到的 Runbook。推断是 Redis 连接耗尽。缺口包括时间窗对齐、Redis 主域状态、发布变更和其他故障域。

Planner 可能建立如下路线：服务上下文、Redis 状态、指标、日志、发布历史、Runbook、历史工单，最后给出处置建议。这里“最后给建议”仍不是执行修复。

为什么不先让模型直接写报告？因为目前没有工具 Evidence；报告只能复述告警，不能区分告警事实与现场复核。

### 第一轮：并发收集独立只读事实

相邻低风险步骤可以最多四个 fan-out。假设指标返回 P95 3250ms、503 或 5xx 同窗上升；日志返回 `/api/order/create` 的 Redis timeout；Redis 状态返回 `connected_clients=9940/10000`。此时新增的事实来自三个不同工具域，Hypothesis“Redis 连接逼近 maxclients 导致连接获取失败”得到支持。

仍不能立刻说唯一根因。需要检查数据源是 live adapter、事故窗口存档、fixture 还是 Mock；需要看 Redis 正常状态是否与日志时间一致；还要确认是否存在同时发生的发布或连接池重试风暴。若 Redis 工具返回当前连接数正常，而日志只来自历史窗口，就不能用“现在正常”反驳“事故时异常”，必须保留时间语义。

为什么下一步查知识和历史？不是为了再证明连接数数值，而是为了验证该信号组合在本服务上的诊断依据、处置前提和相似事故模式。知识补的是解释，工具补的是现场。

### 第二轮：Runbook 和历史工单只提高可解释性

Runbook 可能说明：连接使用率接近阈值时检查连接池复用、超时配置和连接泄漏，并在变更前保留回滚路径。历史工单可能记录同一服务曾因连接池配置不当出现相似 timeout。两者都不能替代 `connected_clients` 的现场 Evidence。

对普通 Redis 场景，主域证据加指标或日志旁证可能已经满足规则；对当前冻结的 Redis golden incident，Analyzer 还会要求 `search_runbook` 与 `search_history_ticket` 成功覆盖，否则置信度最多收敛到有限水平并继续补查。这个额外条件是测试域设计，不应包装成所有生产 Redis 事故的必要定律。

### 第三轮：证据充分后也要先检查剩余风险动作

如果 Evidence 一致、来源质量合格、缺口已覆盖，Replanner 可以选择 `generate_report`。但若剩余计划里还有“重启服务”“扩容 Redis”或修改配置，Replanner 会先提取风险决策。只读步骤尚未完成时，审批动作可以被移到剩余只读调查之后；真正到达风险步骤时，系统写入 `risk_assessment`、`pending_approval` 和 change plan，然后结束本次图运行。

“审批通过以后系统完成了什么”也必须准确：诊断 resume 会优先从内存 checkpoint，其次持久 Snapshot，最后持久报告 fallback 恢复材料，把人工决策写入新的诊断报告和 Trace，关闭审批后的诊断生命周期；它明确不会由该接口自动执行生产变更。真正的安全变更执行是后续专题的独立链路。

### 三条失败分支比正常路径更能说明系统边界

第一条，日志工具失败。工具内部先在总 deadline 内按策略重试；仍失败后形成 failed Evidence，Replanner 可安排一次工作流级 `-retry`。再次失败且 Redis 与指标证据仍可形成假设时，系统输出不完整或降级报告，并写明日志缺失。

第二条，证据冲突。日志明确指向 Redis timeout，但同窗 Redis Evidence 反驳连接饱和，Analyzer 记录冲突、降低置信，生成待确认报告，而不是偷偷删除不符合主叙事的证据。

第三条，只有 Mock 或未知来源。即便三个工具都“成功”，来源门禁仍把充分性撤销，置信度封顶。若没有安全的补查步骤则转人工。系统能在演示数据上走通流程，不等于已经获得生产事实。

MySQL 对照案例只替换证据域：慢 SQL digest、连接池等待、指标、日志与发布关联成为主线；Kubernetes 对照案例则优先 Pod 状态、重启原因和容器日志。二者都复用“事实—假设—缺口—下一步—停止”的骨架，而不是再造一个完全不同的 Agent。

## 流式进度、Trace、Snapshot 和 Replay 分别保存什么

长流程需要可观察和有限恢复，但“能看到”“能审计”“能恢复”“能复盘”不是同一能力。

### SSE 让当前连接看到过程，不承担历史真相

`POST /api/aiops` 返回 `EventSourceResponse`。服务使用 LangGraph `astream(..., stream_mode="updates")`，每个 Planner、Executor 或 Replanner 节点输出后，先生成 progress 事件，再生成业务节点事件，两者都附加 Trace 标识；最终产生 complete，异常被规范成 error/failed。路由辅助在收到终态后停止迭代。

SSE 解决前端当前连接如何看到阶段和结果，不负责持久历史，也没有因为流式输出就获得 exactly-once 语义。浏览器断线时应查询 run status 或 Snapshot；当前章节没有证据证明支持基于 `Last-Event-ID` 的逐字节重放。

### Trace 记录“做过什么”，不证明“结论为真”

**Trace（轨迹）**通俗地说是诊断流水账；严格地说，它是按时间记录工作流开始、节点输出、工具调用、Replan 决策、风险、报告和结束事件的审计序列。它解决“系统做过什么、为何转向”的问题，不负责替代 Evidence 证明根因，也不等于可恢复 checkpoint。

“Replanner 因缺 Redis 状态而追加查询”是 Trace；把这条决策事件当作“Redis 已异常”的证据是反例。AutoOnCall 将节点和决策写入 Trace，并把 `trace_id`、`trace_event_id` 附到 SSE。

### Snapshot 保存最新业务状态，不保存完整时间线

**Snapshot（快照）**通俗地说是某个 session 最近一次状态副本；严格地说，它把计划、已执行步骤、Evidence、工具记录、假设、风险、审批、报告、进度、错误和 warning 等最新业务状态持久化。它解决进程内 checkpoint 丢失后仍有材料完成有限恢复的问题，不负责保存每次状态变化的完整历史，也不保证从任意 LangGraph 节点精确续跑。

AutoOnCall 同时存在两层：`MemorySaver` 是进程内 LangGraph checkpoint；业务 Store 中的 `AIOpsSessionSnapshot` 是可持久化 latest snapshot。反例是看到 SQLite/MySQL Store 接口，就声称已有数据库级 LangGraph checkpointer、跨实例租约和任意节点恢复。当前审批恢复顺序是 checkpoint → Snapshot → persisted Report，恢复目标是更新报告闭环，不重新执行已经发生过的工具副作用。

### Replay 是只读复盘视图，不会重新制造事故现场

**Replay（回放读模型）**通俗地说是把事故过程整理成复盘页；严格地说，它从 Trace、Snapshot/IncidentState、报告、审批、变更记录和可用评测材料派生阶段与时间线。它解决分散 Artifact 如何被阅读的问题，不会重新调用 Planner、Executor 或 Redis 工具，也不是 time travel。

展示事故时保存的 `connected_clients=9940/10000` 是 Replay；今天重新查询 Redis，再把当前值冒充 02:07 的现场，是反例。AutoOnCall 的 Replay 在后续专题由读模型构造，本章循环只负责留下它所需的结构化材料。

## 设计取舍：当前架构选择了可回归，而不是无限开放

| 方案 | 得到什么 | 失去什么 | 更合适的场景 |
|---|---|---|---|
| 固定脚本 | 确定、便宜、容易测试 | 不会因证据变化调整或提前停止 | 封闭 Runbook、规则 fallback |
| 自由 ReAct | 开放任务适应性强 | 工具、成本、权限和回归难控制 | 低风险探索或人工监督沙箱 |
| 每轮重新 Planner | 能重写全局路线 | 模型成本高、计划易抖动 | 故障域快速变化且治理成熟 |
| 初始 Planner + 确定性 Replanner | 语义起点、机器契约、可审计续查 | 规则目录要维护，未知域较弱 | AutoOnCall 当前诊断主线 |

LangGraph 在这里提供节点、共享状态、边、条件边和进程内 checkpoint；LangChain Core 提供 Prompt、消息、`ChatQwen` 与结构化输出。Incident、Evidence、Plan、Analyzer、来源质量、风险策略和报告状态才是项目自己的业务约束。采用 LangGraph 不自动等于自主 RCA，也不等于持久化工作流引擎。

还有一个容易被忽略的取舍：当前循环以“最新待执行队列”为中心，而不是保存每一版完整计划。这样做使执行路径直接、状态体积可控，但若要研究“模型为什么从 A 路线改成 B 路线”，必须依赖 Replan Trace 中的决策原因与新增步骤，而不能只看最终 `current_plan`。生产审计若要求逐版计划 diff，应增加计划版本号、父版本、生成来源和变更理由，不能从 latest Snapshot 反推已经被替换的全部历史。

生产化仍有明确缺口：数据库级 checkpointer、跨实例幂等任务与租约、严格 token/费用预算、真实依赖故障注入、长期阈值校准、未知故障域覆盖、重复副作用防护、Artifact 分布式存储和多实例恢复演练。当前代码与测试可以说明机制存在，不能替这些能力提供生产证据。

## 把这套设计转成面试表达

### 一句话定位

AutoOnCall 用一次结构化规划和一个证据驱动、带硬停止与风险边界的执行/再规划循环，把“Runbook 问答”升级为可审计的多步调查。

### 30 秒回答

> Planner 读取 Incident、Runbook 和工具契约，用结构化 LLM 生成 `PlanStep`，失败时退化为确定性规则计划；Executor 通过 Registry 执行下一步，把成功和失败都沉淀为 ToolCall 与 Evidence；Replanner 默认由确定性 Evidence Analyzer 决定继续、补查、单次工作流重试、生成报告、等待审批或转人工。真实拓扑不会回到 Planner，并用最多 4 路只读 fan-out、`MAX_STEPS = 8`、来源质量上限和审批暂停限制成本与风险。

### 3 分钟展开思路

先从两个失败方案讲起：固定脚本不会根据证据提前停止，自由 ReAct 又难稳定工具、成本和权限。然后说明 Incident 固定调查对象，PlanStep 把工具、参数、预期证据和风险变成机器契约；Planner 只负责初始路线，Executor 只负责取证，Replanner 用有限决策更新队列。

接着用 Redis 主线说明第一批指标、日志和 Redis 状态怎样支持 Hypothesis，Runbook 和历史工单为什么只是解释依据，Evidence 仍要区分来源和时间窗。再讲失败路径：只读工具内部共享总 timeout 重试，工作流只补一个 `-retry`；来源全是 Mock 时置信度封顶 `0.50`，混合来源封顶 `0.72`；证据冲突、八步耗尽或无安全补查时生成降级报告或转人工。

最后说明图因 response、审批或空计划停止，审批 resume 只更新报告闭环，不执行生产变更；SSE、Trace、Snapshot、Replay 分别解决当前进度、审计、最新状态和复盘。量化时引用 48 条 `offline_fixture`、Replan TP `12`、needs-human TP `3`、工具选择 F1 `0.9583` 及 official Run provenance，并主动声明不能外推生产。

### 可写入简历的准确表述

> 基于 LangGraph 实现有界 Plan-Execute-Replan AIOps 诊断循环，采用结构化 LLM Planner、规则 fallback、最多 4 路低风险只读取证与确定性 Evidence Replanner，并通过 `MAX_STEPS = 8`、两层有限重试、来源质量降级和审批暂停约束多步调查；在 48 条冻结 offline-fixture case 上 Replan F1 与 needs-human F1 均为 1.00，工具选择 F1 为 0.9583。

### 不可夸大的表述

- 不说“每轮 Planner 都自主反思”，因为 Planner 只在入口运行一次；
- 不说“Replanner 完成通用因果推理”，因为当前主要是确定性信号规则；
- 不说“`retry_count` 最大 3，所以工具会重试三次”，因为当前工作流只创建一次 `-retry`；
- 不说“MemorySaver 支持跨进程任意节点恢复”，因为它是进程内 checkpoint；
- 不说“审批通过后 Agent 自动修复生产”，因为诊断 resume 只更新报告闭环；
- 不说“48/48 代表生产准确率”，因为证据等级是 `offline_fixture`；
- 不说“official baseline 永远有效”，因为还要核对 commit、dirty、dataset 指纹和 stale 状态。

### 高频追问与回答要点

1. **它和 ReAct 的根本区别是什么？**  
   都会依据观察决定下一步，但本实现只有一次初始 Planner，执行消费结构化步骤，续查由有限枚举和确定性 Analyzer 主导；它更强调可回归和可停止。

2. **为什么 Replanner 默认不用 LLM？**  
   补查、重试、报告和审批是安全敏感决策，需要稳定测试。可选 LLM 只能作 critic，最终仍受证据充分性、工具存在性和风险策略约束。

3. **八步上限会不会导致漏诊？**  
   会，所以它是成本护栏，不是完备性证明。到上限时必须降低状态和置信表述，必要时 needs-human，不能冒充确认。

4. **为什么既有工具内部重试，又有 Replanner 重试？**  
   内部重试吸收同一次调用的瞬时错误，共享总 deadline；工作流重试是在看到结构化失败后再决定是否值得补一次。二者的审计与停止语义不同。

5. **fan-out 会不会破坏证据顺序或风险控制？**  
   只选择相邻、低风险、read-only、白名单且策略 allow 的步骤，最多 4 个；并发执行后按计划顺序归档，风险步骤不进入批次。

6. **Evidence、Hypothesis 和 Trace 有什么区别？**  
   Evidence 是可复核观察，Hypothesis 是由支持/反驳/缺口组织的候选解释，Trace 是过程事件。调用发生过不等于结果可信，结果存在也不等于根因确认。

7. **Mock 工具都成功，为什么还不能高置信报告？**  
   成功描述调用状态，不描述来源真实性。fallback-only 会撤销充分性并把置信度封顶 `0.50`；混合回退来源封顶 `0.72`。

8. **Snapshot 与 checkpoint 有什么区别？**  
   MemorySaver checkpoint 服务当前进程内图状态；Snapshot 是持久化的最新业务状态。后者能支持有限报告恢复，但不是任意节点精确续跑。

9. **Replay 为什么不重新执行工具？**  
   重新查询得到的是现在的状态，可能篡改事故历史并触发副作用。Replay 只聚合已有 Trace、报告、审批和状态材料。

10. **工具选择 Recall 已经 1.00，为什么 F1 只有 0.9583？**  
    因为有 TP `276`、FP `24`、FN `0`，Recall `1.00` 但 Precision `0.92`。系统覆盖了标注必要工具，同时仍有额外选择。

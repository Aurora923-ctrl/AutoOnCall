## 为什么 Redis 演示成功，仍然不能回答“系统有没有退化”

沿用全项目的主案例：`order-service` 开始返回 503，P95 延迟上升，日志出现 `Redis connection timeout`，同时 `connected_clients` 已接近 `maxclients`。开发者调整了 Markdown splitter，使 Redis Runbook 中“连接上限”和“处置步骤”落入更完整的 chunk。随后手工提问一次，系统找到了 Runbook，Agent 也把 Redis 连接耗尽排在根因第一位。

这只能证明“这一次演示在当时的输入和环境下成功”。它没有回答至少六个问题：MySQL 与 Kubernetes case 是否退化；拒答边界是否被放宽；命中的到底是真实 Milvus 还是确定性替代索引；答案来自真实模型还是 fixture；Planner 是否选对工具；危险动作是否仍需审批。若 splitter、Prompt、知识资产、模型、Embedding、阈值和数据集同时变化，即使总分上升，也无法知道是谁带来的变化。

朴素方案常把一次端到端成功当作唯一指标：输入问题，最后比较答案是否“看起来正确”。这种做法的具体失败是，分数下降时无法定位是文档解析、chunk、召回、排序、生成扩写、工具选择、Replanner 停止、风险策略还是外部依赖；分数上升时也可能只是删除了困难样本，或换了一套更宽松阈值。

因此，评测的起点不是选择 Recall、F1 还是 RAGAS，而是写出一条有限结论：

```text
“在固定 80 条 relevance case、确定性离线索引、Top-K=3 的条件下，
标注相关 chunk 的召回和排序没有低于冻结基线。”
```

这句话没有声称真实 Milvus 可用，没有声称答案忠于上下文，也没有声称 Agent 根因正确。边界越清楚，后续实验才越容易设计。

## 评测对象不分清，指标就会测错东西

在进入分层架构前，先把项目中经常连在一起、职责却不同的对象讲清楚。

### Incident、Evidence 与 Hypothesis 分别是什么

**Incident（事故上下文）**通俗地说，是同一次故障的“档案袋”；严格地说，它用统一标识关联告警、时间窗口、诊断、审批、变更和报告。它解决材料属于哪次故障，不负责判断根因。Redis 503、P95 上升和后续审批可以属于同一个 Incident，但 Incident ID 本身不是根因标签。在 AutoOnCall 中，Agent case、Trace、Report 与读模型都通过它关联。

**Evidence（证据）**通俗地说，是可以回到来源核验的观察；严格地说，它带来源、时间窗口、查询上下文和状态，可支持或反驳某个根因假设。它解决“依据是什么”，不自动证明因果，也不把 Runbook 变成现场事实。事故窗口内的 `connected_clients` 接近 `maxclients` 是 Evidence；Redis 官方文档中“达到上限会拒绝新连接”是知识。AutoOnCall 的 Agent 评测会检查所需工具、Evidence 字段和结论回链是否存在。

**Hypothesis（根因假设）**通俗地说，是等待证据验证的解释；严格地说，它是带类别、排序、支持与反驳 Evidence ID 的候选结论。它解决多种解释如何竞争，不保证排第一的就是真相。例如“连接耗尽导致 503”可同时被 Redis 指标和 timeout 日志支持，也可能被“事故窗口连接数其实正常”的证据反驳。AutoOnCall 用 Top-1、Top-3、Macro-F1 和 conclusion-evidence support 检查限定 case 上的排序与回链。

### Trace、Snapshot 与 Replay 为什么不是三个同义词

**Trace（运行轨迹）**是节点、工具调用、风险判断、审批和报告等追加式事件。它解决“系统走过什么路径”，不保证每一步正确，也不一定保存所有原始响应。评测可用 Trace 检查工具选择、阶段延迟和链路完整性，但“有 Trace”不是“根因正确”。

**Snapshot（快照）**是一个 session 的最新业务状态 upsert，用于快速查询和有限恢复。它不保存完整历史，也不是任意 LangGraph 节点 checkpoint。测试证明 SQLite 路径能按契约读写 Snapshot，不等于生产崩溃恢复已经验证。

**Replay（复盘读模型）**聚合 Trace、Report、Approval、ChangeExecution 与 IncidentState，派生阶段和时间线。它解决分散材料怎样被阅读，不重新执行工具，也不能用当前 Redis 健康值覆盖事故窗口。一个离线 fixture 即使出现在 Replay 页面，证据等级仍是离线 fixture。

### Trust Gate 管拒答，不替答案背书

**Trust Gate（可信门禁）**通俗地说是“资料不足时不要硬答”；严格地说，它依据检索结果、引用和产品边界，决定正常回答、有限回答或拒答。它解决无可信来源时的扩写，不负责逐句事实核验，不是权限系统，也不是通用 Prompt Injection 防御。Redis Runbook 没有回答一个人事制度问题时，正确行为是拒答；找到 Redis 文档后，仍需另测引用是否真的支持答案。

这组定义带来一个重要结论：评测不能把“有引用”“有 Trace”“通过 Trust Gate”或“生成 Report”直接计作根因正确。每个指标只能评价它对应的对象。

## 一条总分为什么必须拆成逐级增加真实度的证据梯子

pytest 擅长验证模型字段、API 契约和状态转换，速度快、可定位、可重复；它不会自动证明知识库能召回正确 Runbook，也不会证明真实模型不扩写。Demo 覆盖组合链路，却通常样本少、受网络和缓存影响，还容易被人工重试美化。全部使用真实服务又会昂贵、随机，并在失败时难以定位。

AutoOnCall 因而采用从便宜确定到昂贵真实的梯子：

```text
契约与测试
  → 知识资产与切分
    → 确定性检索 surrogate
      → 固定上下文生成
        → runtime RAG
          → Agent RCA / Replanner / Safe Change
            → 本地真实请求
              → 受控故障
                → 生产证据
```

上一层通过，只允许继续问下一层，不能推出下一层也通过。例如离线 Recall@3 为 `1.00`，只说明 surrogate 在冻结 relevance 标注上没有漏召；真实 Embedding、Milvus 网络、模型 Faithfulness 和生产容量仍然未知。

### API 契约测试不是“只看状态码”

FastAPI 路由能返回 JSON，不代表 OpenAPI 描述与真实行为一致。若聊天接口实际返回 `{code, message, data}`，却仍声明旧的 `{answer, session_id}`；或 readiness 会返回 503，schema 却只描述成功结构，SDK、前端和网关都会基于错误契约生成代码。当前非流式 Chat 与 Health liveness/readiness 使用显式 Pydantic response model，503 也复用同一健康响应结构；测试直接生成 `app.openapi()`，检查目标 method 的 schema 引用。

这个测试仍有边界。response model 证明字段形状，不自动证明每个分支都返回相同语义；项目中仍有 `dict[str, Any]` 的开放字段，流式 SSE 也不由普通 OpenAPI 完整表达。更完整的契约层还需要对真实响应做 schema validation、保存 OpenAPI diff，并给 SSE 维护独立事件 schema。面试中应把它说成“阻止文档与实现静默漂移”，而不是“API 永远兼容”。

请求模型也属于契约。`session_id` 与问题文本在长度校验前先去除首尾空白，并拒绝 ASCII 控制字符；这既避免全空字符串绕过 `min_length`，也阻止业务 ID 把换行带入日志和关联键。中文等正常 Unicode 仍保留。模型校验与日志净化同时存在，体现的是边界防御，不是重复劳动。

### 验证形态和 Evidence level 是两个正交维度

验证形态回答“做了什么”：Implementation 表示代码存在；Automated Test 表示自动输入验证了行为；Evaluation Artifact 表示批量 case、输出、指标和 provenance 被持久化。Evidence level 回答“在哪种环境做”：

| Evidence level | 实际输入和环境 | 可以证明 | 不能自动证明 |
|---|---|---|---|
| `offline_fixture` | 固定数据、替身或确定性索引 | 离线回归可重复 | 真实依赖和生产效果 |
| `local_live` | 本地真实服务或模型 | 本地组合路径可运行 | 生产稳定性、容量和 SLO |
| `controlled_fault` | schema 明确的人为故障 | 已设计故障的行为 | 未知故障和生产 MTTR |
| `production` | 真实生产流量与 Incident | Artifact 覆盖窗口内的生产结论 | 超出样本和时间窗口的泛化 |

真实 Python 代码运行 fixture，仍是 `offline_fixture`。同理，`official` 描述基线治理状态，`production` 描述证据环境；二者不是同一坐标轴。

## 每次运行前，先写“结论检查卡”

一份可比较的评测必须先固定以下内容：

```text
Claim：想证明哪条有限结论？
Path：调用产品路径、固定上下文路径，还是 surrogate？
Data：case、标签、知识资产和样本数是什么？
Environment：offline_fixture / local_live / controlled_fault / production？
Metric：公式、阈值、分子、分母和聚合方法是什么？
Provenance：日期、命令、commit、dirty、hash、依赖、配置、Prompt、Artifact？
Boundary：这次结果明确不能证明什么？
```

以 Redis 为例。若 Claim 是“Redis Runbook 的相关 chunk 仍进入前三”，应固定 relevance case、Top-K、chunk 身份和 Recall@3；若 Claim 是“回答没有把 Runbook 写成现场事实”，应固定上下文并检查 Faithfulness、incident boundary 与引用；若 Claim 是“Agent 会查 Redis 指标和日志后再形成假设”，应使用 Agent RCA case 与 Trace。三条 Claim 不能用同一个“准确率”代替。

### 指标之前先治理数据集，否则分数没有稳定含义

一条评测 case 至少要说明输入、期望、允许的替代答案和禁止行为。检索 case 需要 query、相关 chunk、相关等级、必需来源、可接受来源、禁止来源和是否应拒答；Agent case 还需要预期工具、允许的工具顺序、根因类别、Top-3 可接受集合、所需 Evidence、报告状态和风险边界。只写一个 expected string，会把同义表达判错，也无法检测“答案文字相似但引用错误”。

数据集还应包含不同失败切片。正例检查该找的资料能否出现；拒答负例检查 Trust Gate；confusion case 故意放入词面相近但语义不同的文档，例如 Loki query 与 ingestion；multi-source case 要求同时找 Runbook 和历史工单，防止只命中一份资料就算通过。Redis 主线既要有 `maxclients` 正例，也要有“当前连接数正常、timeout 来自网络”的对照，才能避免规则只学会看到 Redis 就输出连接耗尽。

冻结集、调参集和真正 holdout 不能混用。若开发者根据 80 条 case 的失败逐条添加关键词，这 80 条仍可作为回归集，却不能继续被称为未见测试集。项目测试会检查 stage-two 数据集没有把 tuned case 误标成 holdout。准确的工程表述是“冻结回归集防止已知能力退化”，不是“对未知问题有无偏泛化估计”。

数据泄漏也可能来自评测器，而不只来自人。旧版离线检索会把 case 自带的 `fixture` 直接插入待评索引，还包含针对具体查询、具体 source/chunk 的重排与豁免规则；这等于先读标签，再把标准答案或专用捷径放进被测系统。当前评测只从 `docs/knowledge-base/` 构建共享索引，删除 case 注入和已知 query 特判，并保留真实失败。修复后的 `eval/rag_cases.yaml` 基线是 `29/30`，Recall@3 与检索引用元数据率为 `0.96`、confusion pass 为 `0.80`；失败 case `cpu_slow_sql_relation` 继续进入 failed cases，而不是再用 fixture 把它抬成满分。

Dataset hash 解决的是身份，不解决标注质量。相同 SHA256 说明输入字节相同，不能证明 relevance grade 正确；不同 hash 则说明运行对象已变化，两个分数不能无条件作同比。新增用户差评 case 后，应生成新 candidate、复核标签与切片，再决定是否建立新 official，而不是把新结果覆盖到旧 Run ID 目录。

## 先测原材料：文档能解析，不等于语义能召回

知识质量层调用生产 loader 和 splitter 解析仓库内真实资产，检查文件格式、chunk、引用元数据、空块、重复、freshness 和 index-ready 状态。2026-07-12 official Artifact 记录 20 个资产、209 个 chunk：解析、切分和 index-ready 都是 `20/20 = 1.00`；chunk 长度平均 `902.34`，P50/P95/Max 为 `762/1596/1858` 字符；空 chunk `0/209`，元数据缺失 `0/209`，过长 chunk `4/209 = 0.0191`，近重复 chunk `8/209 = 0.0383`。

这些数字说明 loader/splitter 在该资产集合上工作，并暴露长块和近重复风险。它们不能证明一个 Redis query 会命中正确 chunk。可选 Milvus 校验用临时 collection 和确定性向量做 CRUD；连接真实本地 Milvus时最多是 `local_live` 存储链证据，CRUD 成功仍不等于真实 Embedding 语义召回正确。

## 检索层要同时回答“找没找到”和“排得好不好”

`eval_rag_cases.py` 使用本地文档构造确定性离线索引，不调用运行时 Milvus 和外部 Embedding。它适合快速比较 splitter、融合策略和拒答规则，所以是一种 surrogate：保留产品问题的关键结构，但不是产品依赖本身。

### Recall、Precision、MRR、MAP 和 nDCG 各管一类失败

对一个正例 query，设标注相关 chunk 集合为 `R`，Top-K 结果为 `S_K`：

```text
Recall@K    = |R ∩ S_K| / |R|
Precision@K = |R ∩ S_K| / K
MRR         = 所有 query 的 1 / 首个相关结果排名 的平均
```

MAP@K 继续评价多个相关项在前 K 位的整体排序；nDCG@K 给高相关等级更大收益，再按排名折损并除以理想 DCG。AutoOnCall 以 `(source_file, chunk_id)` 作为稳定身份，重复 chunk 不会重复计相关命中；nDCG 使用 relevance grade，而不是只看文件名。

Recall 高只说明少漏召。把 K 从 1 扩到 5，Recall 可能上升，Precision 却会下降，模型看到的噪声也更多。MRR 高说明第一条有用结果靠前，却可能漏掉第二份必需资料；所以多来源 case 还检查 `strict_multisource@K`。拒答则把“应该拒答”视作正类，分别计算 Precision、Recall 和 F1，避免只靠全部回答或全部拒答获得虚假高分。

### 80 条 official 检索集究竟证明了什么

冻结集共 80 条：64 条非拒答、16 条拒答。以下是 weighted 策略的正例宏平均和独立拒答统计，不是把 80 条所有结果混成一个分数：

| 指标 | 结果 | 它发现什么 |
|---|---:|---|
| 总体 case | `80/80 = 1.00` | 当前 case 门禁全部通过 |
| Recall@1 / @3 / @5 | `0.6641 / 1.0000 / 1.0000` | Top-1 有漏召，Top-3 覆盖标注相关项 |
| Precision@1 / @3 / @5 | `0.7812 / 0.4166 / 0.2500` | K 增大时噪声明显增加 |
| MRR | `0.8776` | 首个相关结果通常靠前 |
| MAP@1 / @3 / @5 | `0.7812 / 0.8763 / 0.8763` | 多相关项排序在 K=3 后不再改善 |
| nDCG@1 / @3 / @5 | `0.7188 / 0.8889 / 0.8889` | 高等级相关项仍非总在第一位 |
| 严格多来源@1 / @3 / @5 | `0 / 1.00 / 1.00` | 多来源问题无法由 Top-1 完整覆盖 |
| 拒答 Precision / Recall / F1 | `1.00 / 1.00 / 1.00` | 16 条拒答 case 无误拒和漏拒 |
| 离线延迟 P50 / P95 / P99 | `12.2704 / 18.3825 / 20.1033ms` | 该机器上的 surrogate 计算时间 |

Dataset 是 `eval/rag_relevance_cases.yaml`，80 case，SHA256 为 `7bd84558b5c76143c2ff49f564def5ccf62f13994e31168b91a2c6c2aff38de3`。这些结果的 evidence level 是 `offline_fixture`。它能证明该 clean commit 的离线排序回归，不能证明 DashScope Embedding + Milvus runtime 的延迟和召回。

### 手推一个小例子，理解为什么不能只看一个数

假设 Redis query 标注两个相关 chunk：A 解释 `maxclients`，B 给出处置与观察项。系统 Top-3 返回 A、无关的 C、B，那么 Recall@3 是 `2/2=1.00`，Precision@3 是 `2/3≈0.667`，首个相关结果排名为 1，所以该 query 的 reciprocal rank 是 `1.00`。若顺序变成 C、A、B，Recall 与 Precision 不变，MRR 贡献却降为 `1/2=0.50`，nDCG 也会下降，因为高相关信息被推后。

再假设第二个 query 只有一个相关 chunk，却排在第 4 位。它的 Recall@3 为 `0`，MRR 贡献为 `1/4=0.25`。把两个 query 的 Recall 与 reciprocal rank 分别取平均，得到宏平均；不能把第一个 query 的两个相关 chunk 和第二个 query 的一个相关 chunk直接堆在一起做全局微平均，因为那会让相关项多的 query 权重更大。官方表中的 Recall、Precision、MRR、MAP 和 nDCG 都应结合 case 聚合口径阅读。

拒答又是另一张混淆矩阵。若 16 条应拒答全部拒答，但 64 条正例中有 4 条也被误拒，则拒答 Recall 是 `16/16=1.00`，Precision 却是 `16/20=0.80`；只报“拒答命中率 100%”会掩盖产品过度保守。当前冻结集 P/R/F1 都为 `1.00`，正是因为没有误拒和漏拒，而不是因为系统简单地全部拒答。

Redis case 在这里已经知道“标注 Runbook chunk 进入 Top-3”；仍缺“回答是否忠于这些 chunk”。因此下一步必须固定检索输入，再单独测生成。

## 检索正确以后，为什么答案仍可能不可信

模型可能把一般 Runbook 写成“事故现场已经发生”，可能引用了正确文件却引用错 chunk，也可能为了更像专家而补出未经检索支持的数值。把 retrieval 和 generation 混测，低分时不知道该调索引还是 Prompt。

### 四种 answer source 隔离不同变量

| Answer source | 检索 | 答案 | 最适合回答 |
|---|---|---|---|
| `product-offline` | 固定离线检索 | reference fixture 经产品契约 | citation、结构和 Trust Gate 是否回归 |
| `reference-fixture` | 固定离线检索 | 直接使用固定答案 | 指标与数据管道是否稳定 |
| `context-fixture` | 固定上下文 | 真实 grounded LLM | 相同证据下生成是否改善 |
| `runtime` | 运行时 Milvus | 真实 grounded LLM | 本地组合路径是否可用 |

`runtime` 是路径名称，不自动提升 evidence level。历史 2026-07-13 runtime RAGAS Artifact 的环境字段仍写 `offline_fixture`；引用时必须按 Artifact 自报等级和 dirty 状态描述，不能仅凭文件名写成 production 或 official。

`product-offline` 也必须遵守当前产品的 generation evidence 边界。fixture answer 不再默认引用 retrieval payload 的全部 Top-K，而是从 grounded prompt 提取实际允许的 chunk IDs，再生成内联 citation；因此被近重复去除或字符预算排除的候选不会被评测替身重新塞回答案。这个修复保证评测验证的是产品契约，而不是一个比产品更宽松的 fixture。

### RAGAS 不是一个单值，profile 还决定哪些指标真正拦截

Faithfulness 检查回答主张是否受上下文支持，不证明上下文本身正确；Response Relevancy 检查是否聚焦问题，不保证可执行；ID Context Precision/Recall 直接比较 retrieved/reference context ID，不依赖语义 Judge；OnCall Actionability 与 Answer Completeness 是用 RAGAS `AspectCritic` 承载的项目 rubric，不是通用根因指标。

默认 `id-smoke` 不需要外部 Judge，运行 ID metrics 与确定性 citation、边界、行动性门禁。`full` 才调用 Judge 模型、Embedding、网络和 API Key。当前配置阈值为 Faithfulness `0.85`、Response Relevancy `0.80`、ID Context Precision/Recall `0.75/0.75`、OnCall Actionability `0.80`、Answer Completeness `1.00`、Refusal Boundary `1.00`、Core Case Pass Rate `1.00`。

阈值不全是 hard gate。`id-smoke` 中 ID Context Precision 只是 watch metric；Faithfulness 和 Response Relevancy 是 `not_run`。只有 `full` 且 Judge ready 时，ID Precision 和 Judge 指标才加入 hard gates。供应商指标逐项记录 `duration_ms` 与 `error`；`Arrearage` 等错误导致 `case_count=0` 时，产物中的 `0.0` 必须解释为 `not_observed`，不能包装成质量分。

RAGAS 输入与版本兼容也属于评测结论的一部分。当前脚本按 `SingleTurnSample` 显式传入 `user_input`、`retrieved_contexts`、`retrieved_context_ids`、`reference_context_ids`、`response`、`reference` 和 `rubrics`，只解析已安装版本公开暴露的 `Faithfulness`、`ResponseRelevancy`、`IDBasedContextPrecision/Recall` 与 `AspectCritic` 兼容类，不依赖私有导出。ID smoke 若因版本不兼容退回确定性实现，Artifact 必须写出实际 engine、状态和原因；每个聚合指标同时公布 available/expected 与缺失 case。任一应有分数缺失时，平均值保持不可用且 case 失败，不能把其余成功行缩小分母后得到更高均值。

还要区分“确定性契约跑了”和“回答质量真的跑了”。`reference-fixture` 与 `product-offline` 没有调用真实生成模型，因此 `answer_generation_executed=false`，顶层 summary 与 quality contract 为 `not_run`；它们可以验证检索、citation plumbing、拒答和指标管道，却不能再写成 answer-quality passed。只有 `context-fixture` 或 `runtime` 实际执行生成后，才有资格进入回答质量门禁。

### official answer-quality 模块并没有跑完整 Judge

2026-07-12 official 的 [ragas.json](../../../logs/benchmarks/20260712T074629Z-81f1f98c-0d69abdc/ragas.json) 使用 `product-offline + id-smoke`，30 case × 1，历史 Artifact 记录 `30/30`。ID Context Recall 为 `1.00`，OnCall Actionability 为 `1.00`，Refusal Boundary 与 Core Case Pass Rate 均为 `1.00`；ID Context Precision 为 `0.6867`。但按修复后的口径，`product-offline` 没有执行真实生成，因此这份旧结果只能证明产品离线检索、引用与拒答契约，不能作为回答质量通过。Faithfulness、Response Relevancy 和人工评审也都未运行。新的基线运行会把该模块标为 `not_run/incomplete`，而不是继续让 fixture answer 帮助 official 全绿。

这也解释了为什么“昨天跑过 `90%`”不是可复核的评测结论：没有说明指标名称、分母、代码状态、数据集和 Artifact，百分比本身无法进入基线治理。

### 回答质量其实有三种“对齐”，不能用一个 citation 字段代替

第一种是检索对齐：retrieval result 的 ID 是否覆盖标注 reference ID，由 ID Context Precision/Recall 衡量。第二种是生成资格对齐：去重和预算后的 frozen evidence 是否同时决定 Prompt 与 allowlist。第三种是引用对齐：答案实际声明的 source/chunk pair 是否能在 generation evidence 中精确找到，且答案中不能混入任何伪造 pair；裸 chunk ID 不再属于产品引用契约。第四种是主张对齐：引用 chunk 的文本是否真的支持答案中的那句话，这才接近 Faithfulness 或人工 claim review。

反例很常见：系统检索到 Redis Runbook 的三个 chunk，其中第三块因预算没有进入 generation evidence；模型不能再靠引用第三块 ID 过门。即使答案引用了实际允许的真实 pair，“citation exists”也只证明身份正确；若答案写“事故窗口连接数已经达到 10000”，而引用 chunk 只说明一般 `maxclients` 原理，主张仍不受支持。此时 ID Recall 可能为 `1.00`，引用身份检查也通过，Faithfulness 仍可能失败。

评测实现还要避免 Judge 自己看到错材料。若答案只引用 Top-3 中第二个 chunk，而 Judge context 却拼接了整个 Top-K 或按另一顺序截断，Faithfulness 分数同时混入未引用内容和错位证据。AutoOnCall 后续修复按 citation 的 `chunk_id` 回取文本，保留引用顺序，并在引用找不到时记录 fallback；测试覆盖正常对齐与 fallback。这个机制提高的是 Judge 输入可信度，不是自动证明每个 claim 正确。

因此，线上产品至少要保留 retrieval payload、answer、citation、case ID、metric error 与 duration。只保存最终平均分，会让“模型变差”和“Judge context 组装错误”看起来完全一样，也无法在一次供应商故障后重算指标。

## Agent 评测为什么不能复用 RAG 的 Recall

RAG 检索到正确 Runbook，只解决知识参考。Agent 还要选择工具、执行查询、形成 Evidence、排序 Hypothesis、处理失败、经过风险门禁并生成可审计报告。因此 Agent RCA 的单位是一次结构化诊断 case，不是一个 chunk。

### 48 条 RCA case 检查完整诊断闭环

`eval_cases.py` 使用限定的 Redis、MySQL、Kubernetes 和负例 fixture。Top-1 衡量第一根因类别是否命中；Top-3 衡量前三候选是否覆盖；Macro Precision/Recall/F1 先按类别计算再等权平均，避免大类掩盖小类；Tool Selection F1 比较计划工具集合与标注集合。

2026-07-12 official 的 48 条结构化 case 结果为：总体 `48/48`；Top-1 Accuracy `1.00`，Top-3 Recall `1.00`，Macro Precision/Recall/F1 `1.00/1.00/1.00`；Tool Selection Precision/Recall/F1 为 `0.9200/1.0000/0.9583`，对应 `tp=276、fp=24、fn=0`；工具执行 `267` 次、成功 `258` 次，execution success rate `0.9663`。根因、所需 Evidence、结论回链的 48-case 比例均为 `1.00`，Wilson 95% CI 为 `[0.9259, 1.0000]`，不能因为点估计满分就声称任意故障都正确。

Redis case 在这一层回答的是：是否选择 Redis 指标、日志和参考知识，是否将 `redis_maxclients` 排到正确位置，是否保留 Evidence ID，失败工具是否触发降级。它仍使用 fixture 和静态信号规则，不能把当前健康状态当事故窗口证据，也不能当作生产根因准确率。

### Replanner 要测“为什么继续或停止”，不只测最终答案

Agent 有循环后，最终根因命中仍可能掩盖低效或危险路径。例如第一次已经获得 Redis 指标、timeout 日志和 Runbook，Replanner 却重复查询五次；或者 Redis 工具明确失败，系统仅凭 P95 过早停止并宣称根因。最终文本偶然正确，决策过程仍然不合格。

Replanner 专项 case 因而检查证据充足时停止、证据不足时补查、失败时转 human-needed、禁止工具是否被避开、决策来源是否记录到 Trace，以及 LLM 是否只在策略允许时调用。2026-07-12 official 模块有 5 条 case，结果 `5/5`；decision、decision source、structured positive path、plan tool、forbidden avoidance、LLM call policy、blocked guardrail、first step 与 trace decision 都是 `5/5 = 1.00`，evidence level 为 `offline_fixture`。

样本只有 5 条，所以这组结果主要证明控制分支可回归，不支持“Agent 在任意长任务上都能最优规划”。更完整的生产评测还应统计平均循环轮数、重复工具率、无信息增益调用、转人工时机与 token/成本，并按故障类别切片。

当前实现已把一个旧评测盲区补进 Replanner case 构造：case 可以携带既有 `current_plan`，因此能验证 LLM 新增只读步骤时，后面的生产重启等审批动作仍留在队列中。2026-07-17 当前 dirty worktree 的评测文件扩展为 6 条，并通过 `6/6` Automated Test；新增 case 使用 `prod-cn` 和既有 `restart_service`，断言结果队列为“新指标查询 + 原生产动作”。这不是回写 2026-07-12 的 5-case official Artifact，而是提醒评测设计必须构造真实状态前置条件，否则“输出工具正确”也可能掩盖队列覆盖。

单元级边界还覆盖：补证与剩余队列合并、已执行语义重复步骤过滤、显式失败重试例外、达到八步前后的风险门、非法剩余步骤 fail closed，以及 Raw Alert 安全字段和最新失败 Evidence 在字符预算内保留。真实模型是否更稳定地利用首尾上下文仍属于待真实模型验证；确定性队列、解析和安全契约则已由 Automated Test 覆盖。

## 性能数字为什么必须同时保留样本、分位数和接受门槛

P50 是一半样本不超过的延迟，P95/P99 关注长尾。它们不能脱离请求数、并发、环境和失败率：`2/2` 与 `2000/2000` 都可写成 100%，不确定性却完全不同；20 个请求的 P95 也不能推出 100 并发容量。

### 点估计相同，证据强度为什么仍然不同

比例指标必须保存分子、分母和区间，而不只是小数。`48/48=1.00` 的 Wilson 95% 区间下界为 `0.9259`；它表达“固定样本全中，但总体真实比例仍有不确定性”。若只有 `2/2=1.00`，下界会更低，显然不能与 48 条或 2000 条样本视为同等证据。

置信区间也不能修复有偏数据。48 条 case 若都来自人工规则可轻易识别的模式，即使区间很窄，也只估计这套抽样分布；它不会自动覆盖未知生产故障。相反，一个关键安全负例失败，即使总体通过率仍很高，也可能足以阻塞发布。因此区间、切片和 hard gate 要一起使用：区间描述随机不确定性，切片暴露结构性偏差，hard gate 表达业务不可接受条件。

项目的共享 `proportion_metric` 会保存 value、numerator、denominator、sample_count、Wilson interval 和 source。对于 Recall、MRR 等正例宏平均，分子可以是平均贡献乘样本数的审计值，不应误读成整数命中次数；拒答 P/R 则可直接回到 TP、FP、FN。读者在比较两个 `1.00` 前，应先确认它们是否同一公式、同一聚合方式和同一数据集。

official baseline 的 performance 模块读取 50 条持久化事件，但没有识别出真实完成请求：`request_sample_count=0`，RAG 覆盖 `0/30`、AIOps 覆盖 `0/20`，状态为 `observed_not_accepted`，real request gate 为 `not_run`。其中 P50/P95/P99 都是 `0.0/0.0/0.0ms`，含义是没有请求样本，不是零延迟。

旧版 Baseline runner 会把 `observed_not_accepted` 视为“performance 模块成功产出诚实观察”，所以 8 个 required modules 仍可 `8/8` 并形成 official。修复后，`observed_not_accepted`、`passed_without_milvus`、`retrieval_only_passed`、`not_run` 和 `incomplete` 都统一映射为 incomplete module，不能满足 official 的 required-module 门。模块成功生成报告与业务验收通过仍是两件事，但前者不再替后者放行基线。

该 Artifact 没有完整 command、commit、dirty、配置与资产指纹，也没有记录 P99；Token 为 `not_observed`，金额为 `not_run`。历史 runner 还只检查 HTTP/SSE 表面成功，没有把固定 case、required source、approved chunk、answer policy、runtime retrieval mode 与模型身份绑定起来。当前 runner 已从冻结 RAG case 读取 query 和引用契约，要求每个 required source 都由 approved chunk 支持；AIOps 只接受 `completed` 终态，`degraded`、`needs_human` 或 error 不再计成功。若 case 还声明自然语言 required/forbidden claims，而没有绑定 Judge 或人工复核，整轮状态为 `incomplete` 并标记待验证，citation 不能冒充语义蕴含。

性能读模型也只用成功请求满足覆盖：失败请求仍进入样本与失败分布，却不会填满 `30/20` 门槛；任一已完成请求失败会使状态为 `failed`。RAG runtime benchmark 默认跑完整数据集，`--generate-limit` 的子集由 case ID 稳定选择，不再依赖 YAML 顺序；检索与生成分别报告，未生成时状态是 `retrieval_only_passed`，不能写成端到端 passed。它只能证明曾在本地观察到 30/30 历史请求和上述延迟，不能晋级 official，更不能推导生产吞吐。

### 分位数怎样算，也应进入评测契约

不同库对分位数可能使用最近秩、线性插值或其他约定，小样本时结果差异明显。项目的性能脚本先升序排列请求延迟，再取 `ceil(q × n)-1` 的最近秩位置；同时保存 count、min、max、mean、median、P50、P95、P99 和总体标准差。若没有样本，函数稳定返回零值结构，调用方必须结合 `count=0` 与 `status=not_run/observed_not_accepted` 展示，而不能只渲染 `P95=0`。

Trace 事件也不能一条算一个请求。脚本先按 trace/request 聚合完成请求，内部 planner、tool、approval 等 phase 事件只用于阶段延迟，不增加 request count。否则一个步骤多的 AIOps 请求会在分位数中被重复加权，看起来样本量比真实请求多。evidence level 不一致的事件会形成 conflict，并使结果 `invalid_evidence`，避免把 fixture 与 local-live 延迟混算。

成本同样受证据约束。只有 provider 返回 token usage，且应用了带日期和来源的 price snapshot，才能计算已定价样本；未知模型进入 unpriced，而不是使用猜测价格。当前真实模型历史的 token 未暴露，所以 `not_observed` 比伪造“单次成本约为零”更有价值。

### 真正的生产验收还要增加哪些量

生产阶段不应只复制离线 Recall。检索需要按业务、语言、文档版本和流量切片观察零结果率、点击或人工有效率；生成需要抽样做 claim/citation 审核、严重幻觉率和拒答误伤；Agent 需要看工具失败、转人工、重复循环、审批绕过和建议采纳；运行效率需要请求成功率、P50/P95/P99、超时、重试恢复、token、成本和并发曲线。

更关键的是建立结果回写。建议执行成功不等于事故恢复，必须在明确观察窗口内验证 503、P95、Redis 连接数和错误日志，再把恢复结果与 Incident、ChangeExecution 和 Report 关联。只有这些生产样本带上时间窗口、流量与治理记录，才可以讨论真实 MTTD/MTTR；当前仓库没有这类持续 production Artifact。

### candidate、official 与 stale 分别回答什么

`candidate` 是尚未满足晋级条件的候选运行；`official` 表示这次 baseline 的 required modules 全部产出可接受状态、模块不 stale、commit 可识别且 worktree 干净；`stale` 表示当前 worktree、资产、配置、依赖、Prompt 或整体 evaluation fingerprint 与 Artifact 生成时不同。

stale 不表示历史结果必然错误，只表示它不能直接代表当前代码。dirty worktree 可以生成 candidate，却不能 official，因为 commit 无法还原未提交内容。`official + offline_fixture` 完全合理，`candidate + local_live` 也可能存在；official 从来不是 production 的别名。

Baseline runner 为每次运行创建独立目录，并更新轻量 latest pointer，不覆盖历史。评测与 RAGAS 读模型仍可返回 stale Artifact 的原始内容供审计，但会设置 `available=false`、`stale=true` 并给出 reason；Interview Scorecard 还会校验所有 module row 的 Run ID 一致，stale 或混入其他 Run ID 时同样不可用。因此“API 能读到旧分数”不等于“旧分数仍可用于当前发布”。

独立 RAG Scorecard 也不再把多个日期的候选 JSON 拼成全绿页面。它逐个检查 stale、summary status 和 commit/worktree/asset/config/dependency/prompt provenance；来源签名不一致时标记 `mixed_artifact_provenance`，任一层未实际通过时标记 `layer_not_passed`，fixed-context 没有真实生成则显示 `not_run`。只有单一 provenance 且 deterministic retrieval、fixed-context generation、runtime end-to-end 三层都通过时，Scorecard 才返回 passed；否则状态是 `invalid`，命令退出非零。

### official 的判定顺序为什么必须是机器可执行规则

如果由开发者看完图表后手工宣布“这轮算 official”，失败模块很容易被忽略，dirty 状态也可能在汇报中消失。当前 runner 的核心规则可以压缩成：

```text
逐个运行 8 个 required module，并写入同一 run directory
若模块产物缺失、返回失败或自身 stale → 模块失败
若任一模块失败或 stale → candidate_incomplete / candidate_environment_changed
若 worktree dirty → candidate_dirty_worktree
若 commit 缺失 → candidate
只有全部满足，才 official
```

每个 module 还保留自己的 evidence level，所以顶层 run 的环境字段是 `local_live`，不代表其中 RAG、RAGAS、Agent 和 Safety 也变成 local-live。Scorecard 从同一 manifest 构建，而不是从 `latest rag + latest aiops + latest safety` 拼盘；每一行必须带相同 Run ID。这个约束阻止团队拿不同日期的最好成绩组成一张从未真实运行过的“全绿基线”。

`latest.json` 只是导航指针，不是历史真相。新 candidate 可以更新 latest，却不应覆盖旧 official 目录；读取方若只找文件名 `latest` 而忽略 baseline status，就可能把 dirty 失败实验展示成发布基线。正确界面应同时呈现 Run ID、baseline status、stale、evidence level、样本数和 Artifact 链接。

这种治理还要求阈值变更可追踪。若把 Faithfulness 门槛从 `0.85` 降到 `0.70`，即使模型输出不变，pass/fail 也会改变；因此阈值属于评测配置和 provenance，而不是藏在汇报表格里。新阈值需要新的 candidate 和理由，不能回写旧 Artifact 让历史“自动通过”。

同样，修复评测脚本本身也应生成新运行。旧 Artifact 记录的是旧代码如何解释旧数据，不能在文件上直接重算后仍沿用原 Run ID。需要保留原结果、新结果、脚本差异和迁移说明，才能区分“产品能力提升”与“计分器修正”。这对 RAGAS Judge context 错位尤其重要：分数改善可能首先来自测量变准确，而不是模型突然变好。

## 用户差评为什么先进入 backlog，而不是直接改 Prompt

一次差评可能是真的召回失败，也可能是恶意输入、含糊评价或缺少事故上下文。若自动把它追加到 YAML，稳定回归集会被噪声和数据投毒污染；若自动改 Prompt，更无法归因下一次分数变化。

AutoOnCall 的反馈链是：

```text
反馈
  → query/answer/citation/retrieval/Trace/tool context 脱敏与截断
  → 分类和 high-value 判断
  → reviewable backlog
  → 人工标记 reviewed / rejected
  → 显式 promotion
  → 对应回归集
```

分类包括 retrieval failure、missing citation、tool failure、hallucination risk、permission denied 和 poor report quality。只有 high-value 且 backlog 状态为 `reviewed` 或 `promoted` 的记录才可进入 promotion；重复 case ID 会去重。RAG、AIOps 和 Change 分别写入对应 YAML；RAGAS answer-quality feedback 被明确跳过普通 retrieval YAML，只保留为已审阅的专用 fixture 草稿。

这是一条人工治理的 bad-case 飞轮，不是在线学习。系统不会因一次 thumbs-down 自动训练模型、重写 Runbook、修改 Prompt 或改变生产风险策略。

## 分数下降以后，怎样定位而不是立刻调 Prompt

一套分层评测真正有用的时刻，是发布门失败时能给出最小排查面。

| 观察 | 优先检查 | 不应立即得出的结论 |
|---|---|---|
| parse/index-ready 下降 | loader、格式解析、文件权限、splitter | 模型能力下降 |
| Recall 下降、Faithfulness 稳定 | chunk、索引、融合、过滤、Embedding | Prompt 一定有问题 |
| Recall 稳定、Faithfulness 下降 | answer Prompt、citation/context 对齐、模型版本 | 知识资产一定错误 |
| ID Recall 高、ID Precision 低 | Top-K、融合噪声、标注粒度 | 召回失败 |
| Agent Top-1 下降、tool recall 稳定 | Evidence 解释、Hypothesis 排序、时序边界 | 工具没调用 |
| tool recall 下降 | Planner、工具注册、权限、失败降级 | 根因分类器错误 |
| Safe Change F1 下降 | 策略分类、参数解析、审批状态 | RAG 文档召回问题 |
| local-live 延迟升高、fixture 稳定 | 外部服务、网络、并发、模型供应商 | 代码算法必然退化 |

还要检查变化是不是评测系统自身造成的。dataset hash、阈值、Judge 模型或 Prompt 变化时，应先把结果标成不可直接同比；`stale` 时应重跑，不应截图旧分数；某个 Judge error 时应看逐指标 error/duration，不能把缺失项补零。若只修平均分而不看 failed case，Redis 平均提升可能掩盖 Kubernetes RBAC 或 Loki 单例持续失败。

合理的修复循环是：定位失败层和切片，构造最小复现实验，只改变一个主要变量，保存 candidate 与失败 Artifact，再跑更高层验证副作用。Round 2 Faithfulness 从 `0.3277` 降到 `0.2412` 正是有价值的负向证据：它说明“更短、无引用就全拒答”并非可靠改进，应撤回而不是删除记录。

## 用 Redis 主线走完一次分层判定

现在把各层重新串起来，但不混合它们的结论。

第一步，知识质量告诉我们 Redis 文档可解析、chunk 有来源元数据；仍不知道这个 query 能否召回。第二步，80-case surrogate 中 Redis 相关 chunk 进入 Top-3；这是检索事实，仍不知道模型会如何表述。第三步，固定上下文生成检查回答是否只说“Runbook 描述了连接上限”，而没有冒充“事故已确认”；若 citation 对不上 chunk，就修 Judge context 或生成约束。

第四步，Agent case 检查 Redis、日志、指标与知识工具是否按预期被选择，`connected_clients` 是否作为现场 Evidence，Runbook 是否只作参考，`redis_maxclients` Hypothesis 是否有支持链接。第五步，Safe Change 确保调整容量或重启不会越过审批；第六步才用 local-live 或 controlled fault 观察真实服务与恢复。

每一步都持续回答五个问题：当前知道什么；哪些是现场事实、知识和推断；还缺什么；为何执行下一层；什么情况下停止、降级或转人工。若 Redis 查询失败，Agent 应降级而不是用 P95 替代 Redis 主域证据；若 Runbook no-answer，应有限回答或 needs-human；若高风险变更待批，报告可完成诊断交接，但不能声称已执行和恢复。

### 一次真实的发布决策应怎样写

假设本次只修改 splitter。知识质量 `20/20` 仍通过，80-case Recall@3 仍为 `1.00`，但 Precision@3 从冻结值 `0.4166` 明显下降；固定上下文生成不变，runtime 尚未跑。此时不能写“全部通过”，也不必立刻否定 splitter。合理结论是：召回覆盖未退化，但新增噪声可能增加生成风险；该轮只能保留 candidate，进一步查看下降集中在哪些文档类型和 confusion case，再决定调整 chunk 或 Top-K。

若 Precision 恢复，full Judge 的 Faithfulness 却下降，则说明 splitter 可能改变了 chunk 边界，使被引用片段不足以支持完整 claim。应检查 citation-context 对齐和具体 failed case，而不是因为 Recall 满分就发布。若所有离线层通过、local-live 真实请求因 Milvus 超时失败，则准确状态是“算法回归通过、运行时集成阻塞”，发布仍应停止。

反过来，local-live Redis Demo 成功，但 dataset hash 已变化、worktree dirty、43-case safety 未运行，也不能用 Demo 越过门禁。发布证据的基本原则是短板约束：昂贵高层成功不能替代便宜底层缺失，单个成功样本不能覆盖系统性回归，平均值不能覆盖关键失败 case。

## 这套体系的取舍与仍未闭合的生产边界

确定性 surrogate 快、稳定、易定位，却不能代表真实 Embedding；runtime 更接近产品，却受网络、模型和依赖随机性影响。保留两者增加维护成本，但能区分 splitter/retrieval 失败与 generation/provider 失败。固定上下文路径同理：它不是产品流量，却是隔离生成变量的有效实验。

独立 run directory 占用更多空间，却让失败历史和 provenance 不被 latest 覆盖。人工 review 比自动入集慢，却降低反馈投毒。规则阈值可解释，但可能与真实用户价值不一致；LLM Judge 能评价语义，却有模型偏差、成本和供应商错误。当前方案是项目规模下的折中，不是唯一正确答案。

证据边界必须逐项陈述：

| 层级 | 本章已有材料 | 准确结论 |
|---|---|---|
| 已实现 | 分层脚本、指标、provenance、baseline、读模型、反馈 promotion | 工程链存在 |
| 自动化测试已验证 | OpenAPI response model、请求边界、无 case 注入检索、缺失指标分母、RAGAS 公开 API 兼容、stale 不可用、baseline incomplete、Scorecard provenance、性能失败样本等 | 指定输入的行为可回归 |
| 离线 fixture 已验证 | 80 RAG 历史基线、30-case 历史检索/引用契约、48 RCA、43 safety、12×3 | 冻结样本上的明确指标；未实际生成的 answer source 不再算回答质量 |
| 本地真实依赖已验证 | 独立 20+10 历史请求与部分 runtime 材料 | 本地路径曾运行；旧 provenance/语义验收不完整，修复后真实模型结果待验证 |
| 受控故障已验证 | `7/7` cleanup、`2/2` RCA 恢复 | 预设计故障行为 |
| 生产环境已验证 | 当前没有 production Artifact | 不能声称生产准确率、容量、MTTR |
| 尚未实现/未闭合 | 长期阈值校准、真人盲评、稳定并发、未知故障覆盖、持续生产指标 | 明确列为后续工作 |

## 面试中怎样把“评测体系”说成工程能力

### 一句话定位

AutoOnCall 用分层实验和完整 provenance，把“一个高分”约束成“某条路径、某组数据、某种环境下可复核的有限证据”。

### 30 秒回答

> 我把评测拆成契约、知识质量、确定性检索、固定上下文生成、runtime RAG、Agent RCA、Replanner、Safe Change、性能和 controlled fault。评测器不能把 case fixture 或查询特判注入被测索引，未真实生成、未跑 Judge、跳过 Milvus、只跑检索、存在失败请求或混合 provenance 都不能算完整通过。每个 Artifact 记录命令、commit、dirty、数据集、资产、配置、依赖和 Prompt 指纹，并区分 offline fixture、local live、controlled fault 与 production；dirty 或 stale 只能做 candidate，反馈要经过脱敏、人工 review 和显式 promotion。

### 3 分钟展开思路

先从失败定位讲分层：pytest 验证请求模型、OpenAPI response model、SSE helper 和状态转换等结构契约；knowledge quality 验证生产 loader/splitter 与可选 Milvus CRUD；80 条离线 relevance case 用 Recall、Precision、MRR、MAP、nDCG 和拒答 F1 评价 surrogate；RAGAS 再按 answer source 和 profile 隔离生成，`id-smoke` 不冒充完整 Judge。Agent 用 48 条结构化 case 测工具、Evidence、Hypothesis、RCA、报告和 Trace，安全用 43 条 case 测 forbidden、approval 和 allow。

再讲治理：2026-07-12 run 是历史 official 证据，但旧规则曾把 `observed_not_accepted` 和未生成的 product-offline answer quality 接受为完整模块；修复后这些状态会变成 incomplete/not_run，因此必须生成新 candidate，不能回写旧 Run ID。RAGAS 缺失指标不缩分母，fallback engine、metric coverage 和 review binding 都进入 Artifact；真实模型验收要求 approved citation 与运行时证据，语义 claim 未审则待验证。最后主动说明 local-live 小样本、controlled fault 和 official fixture 都不是 production evidence。

### 简历可以准确写

> 建立 RAG + AIOps Agent 分层评测体系，覆盖确定性检索 surrogate、固定上下文/运行时回答质量、48-case RCA、43-case Safe Change、性能与受控故障；通过 commit、worktree、数据集、资产、配置、依赖和 Prompt 指纹治理 candidate/official/stale Artifact，并构建人工审核的 bad-case promotion 链。

> 为 FastAPI 核心接口补齐请求/响应契约回归：通过 Pydantic 模型限制空白与控制字符输入，显式声明 Chat 和 Health 成功/503 响应，并从生成的 OpenAPI 校验 schema 引用，防止实现、文档和客户端静默漂移。

> 在 2026-07-12 clean-commit 的 80 条离线检索集上，Recall@3 `1.00`、MRR `0.8776`、nDCG@3 `0.8889`；同时保留 Precision@3 `0.4166`、64/16 正负样本构成、dataset hash 和 `offline_fixture` 边界，避免用单一 Recall 掩盖噪声。

### 不可夸大的表述

- 不写“80/80 等于真实 Milvus 检索准确率”。
- 不写“official baseline 代表生产准确率”。
- 不写“48/48 证明任意故障根因都正确”。
- 不写“43 条安全 case 证明免疫 Prompt Injection”。
- 不写“36/36 是真人盲评一致率”。
- 不把 `case_count=0` 的 `0.0` 当真实质量分。
- 不把 `product-offline/reference-fixture` 的 deterministic pass 写成回答质量通过。
- 不把 `passed_without_milvus`、`retrieval_only_passed` 或 `observed_not_accepted` 写成完整基线通过。
- 不把缺 command、commit、dirty 和 hash 的 local-live 数字写成独立简历成果。
- 不写“用户反馈会自动训练模型或修改 Prompt”。

### 高频追问与回答要点

1. **为什么 retrieval 和 generation 必须分开？** 低分时才能判断是相关 chunk 没进入 Top-K，还是模型在正确 context 外扩写。
2. **Recall@3 已是 1.00，为什么还要 Precision、MRR 和 nDCG？** Recall 不管噪声和顺序；本基线 Precision@3 只有 `0.4166`，nDCG@3 也只有 `0.8889`。
3. **official 为什么仍可以是 offline fixture？** official 管运行完整、可复现和干净 worktree；evidence level 才管环境真实度。
4. **`id-smoke` 的 ID Precision 低于 0.75，为什么模块仍通过？** 当前 profile 将 Precision 作为 watch metric；ID Recall、引用、边界、行动性与全 case 通过才是 hard gates，`full` 才把 ID Precision 纳入硬门。
5. **stale 怎样判定？** 比较 worktree、资产、配置、依赖、Prompt 和整体 evaluation fingerprint；它表示旧 Artifact 不代表当前状态，不表示历史必然错误。
6. **为什么 `2/2` 不能只写 100%？** 分母小导致不确定性大，必须保留样本、环境和置信区间；它也不能推出未知故障覆盖。
7. **旧 performance 模块为什么 official 但真实请求为零？** 旧 runner 把 `observed_not_accepted` 当模块 passed；当前已改为 incomplete，未来 required baseline 不会再被零真实请求放行。
8. **为什么 product-offline `30/30` 现在是 not_run？** 它没有调用真实生成，只验证检索、fixture answer、引用和拒答管道；回答质量必须由 context-fixture/runtime 的实际生成承担。
9. **RAGAS 返回部分 NaN 时怎样平均？** 不缩小分母；平均值为 unavailable，列出缺失 case，并让依赖该指标的 case/门禁失败。
10. **12×3 能证明稳定吗？** 只能证明这 12 个固定 case 的三次链路和统计；simulated review 还必须与 case/answer/context hash 绑定，且不是独立真人评审。
11. **反馈为什么不能自动入集？** 需要防恶意、误标和上下文不足；只有 reviewed high-value case 才显式 promotion。
12. **controlled fault 能证明什么？** 证明预设计故障在指定环境下的诊断/清理/恢复行为，不证明生产 MTTR 和未知多故障。
13. **有 response model 为什么还要测 OpenAPI？** 路由可能忘记挂模型、method 可能指向旧 schema、错误响应也可能未声明；直接检查生成结果才能覆盖框架注册后的最终契约。

# 让另一个 Agent 调用 AutoOnCall，为什么不能直接开放工具

上一章解决的是评测问题：怎样用分层样本、指标与 Artifact 证明检索、诊断、风险控制等链路没有退化。本章讨论一个更靠近系统边界的问题：当 Incident Commander Agent、客服 Agent 或企业协作平台想使用 AutoOnCall 时，应该开放多少能力，哪些内部控制权必须继续留在 AutoOnCall 内部。

下一章会统一汇总项目量化指标和校招验收口径。本章先给出结论：**Agent 协作的核心不是让调用者获得更多工具，而是把一个稳定业务意图委托给仍然受原有证据、风险与审批链约束的服务。**

本文事实审计日期为 `2026-07-13`。当前实现是默认关闭的 A2A-compatible HTTP+JSON/SSE 业务门面，不代表通过官方 conformance，不代表跨厂商互操作，也不新增审批或生产变更能力。完整指标见[第 15 篇：项目量化指标与校招验收](15-项目量化指标与校招验收.md)。

## Redis 故障需要协作，外部 Agent 应该拿到什么

假设一个 Incident Commander Agent 收到告警：`order-service` 返回 503，P95 延迟上升，日志出现 `Redis connection timeout`，`connected_clients` 接近 `maxclients`。它希望委托 AutoOnCall 做四件事：启动诊断、查看进度和最终状态、获取事故复盘、引用 Redis Runbook 生成交接说明。

最直接的方案是把 `query_metrics`、`query_logs`、`query_redis_status`、审批接口、SQL 工具和变更接口全部注册给它。调用方会很自由，却也获得了本不需要的控制权：它可以跳过 Planner 的调查顺序，自己拼接不同时窗的数据；可以把知识当 Evidence；可以绕过 Risk Controller 直接尝试工具；还可能把“审批通过”误解成“Redis 已经修改”。

此刻应该区分三类信息：

- **事实**：503、P95、超时日志和连接数来自事故窗口的观测。
- **知识**：Redis 达到连接上限可能拒绝连接，提高上限前要检查文件描述符、内存和连接泄漏。这来自 Runbook。
- **推断**：连接压力可能是根因，仍需多源 Evidence 支持或反驳。

外部 Agent 的业务目标是“得到一次受控会诊”，不是“亲自操作每个取证工具”。因此 AutoOnCall 只公开四项固定业务能力：诊断、查询状态、解释 Replay、回答 Runbook 问题。底层 Tool、Approval 和 Change 继续留在原有治理面。

这不是削弱协作，而是应用最小权限原则：调用者得到完成业务所需的最小能力，同时失去破坏内部不变量的自由。

## 三种朴素开放方案为什么都会失败

### 直接公开 OpenAPI：把内部状态机外包给每个调用方

AutoOnCall 已有聊天、诊断、Incident、审批、变更和评测 API。让上游 Agent 自行阅读 OpenAPI 并组合调用，看似没有适配成本，实际要求每个调用方都理解 Incident、session、Trace、Snapshot、Report、Approval 和 ChangeExecution 的关系。

不同调用方会写出不同的终态判断：有人把 SSE 断流当完成，有人把 `waiting_approval` 当失败，有人看到 `approved` 就开始找生产变更接口。内部路由或状态名变化时，所有上游同时失效。更严重的是，低层 API 的“可调用”会被误当成“业务上允许调用”。

### 每个 Tool 都包装成 Skill：只是把越权标准化

第二种方案给每个工具加上漂亮描述，注册成 Agent Skill。它没有消除风险，只是把 `registry.execute(tool_name)` 变成一个标准化远程入口。

`query_logs` 单独看是只读工具，但合理参数依赖 Incident 时间窗；`execute_sql` 更不能因为包装成 Skill 就绕过禁止规则；重启和调整配置必须先形成 Evidence、Hypothesis、ChangePlan 与 Approval。业务流程的安全性来自它们之间的顺序和约束，不来自工具名字。

### 复制一套专用诊断引擎：形成两个真相源

第三种方案为 A2A 单独实现 Planner、Executor、任务库和报告生成器。短期看协议更“原生”，长期会出现两套风险规则、两套状态和两套修复逻辑。内部 API 已修复的问题可能仍留在 A2A 分支，A2A Task 与本地诊断 Snapshot 还会互相打架。

更稳妥的结构是：

```text
外部业务意图
  → 固定 Skill 解析与最小 Scope
  → 复用现有领域服务或读模型
  → 映射为 Task、Status、Artifact 或 SSE
```

Facade 负责协议翻译，不复制诊断引擎，也不赋予调用方内部编排权。

## 先把协作层术语翻译成人话

### Agent Card：能力说明书，不是通行证

通俗地说，Agent Card 是外部调用者看到的服务目录。严格定义上，它描述服务 URL、协议版本、传输方式、能力标志、Security Scheme 和 Skill 列表，用于发现与协商。它解决“这个服务声称能做什么”；它不负责验证调用者身份，也不证明服务通过标准认证。

例如 Card 可以声明支持诊断和 Runbook 问答，但不应列出 `execute_sql` 或 Redis 写工具。AutoOnCall 提供公开 well-known Card 和受 Scope 保护的 extended Card，协议字段标为 `1.0`，响应媒体类型为 `application/a2a+json`。这是项目自定义兼容面，不能据此宣称完整 A2A conformance。

### Skill：稳定业务委托，不是内部函数别名

通俗地说，Skill 是“你可以委托我办的一件事”。严格定义上，它有稳定 ID、业务输入输出、权限要求和明确副作用边界。它解决外部意图到内部领域能力的映射；它不负责暴露内部每个函数。

`diagnose_incident` 是合理 Skill，因为它保留 Plan-Execute-Replan、Evidence 和风险控制；`query_redis_status` 是内部工具，不应成为对外业务 Skill。AutoOnCall 的 Skill 集合是代码中固定的四项，不是动态 Tool 市场。

### Task：一次委托的对外状态视图

通俗地说，Task 是一张会诊单。严格定义上，它包含稳定 ID、Context、Status、Artifact、Metadata 和可选 History，用于表达一次委托的生命周期。它解决状态查询和结果关联；它不自动成为第二套工作流数据库，也不能凭一个状态词停止后台任务。

诊断 Task 使用服务端生成的 `a2a-diagnosis-*` ID，并把同一 ID 作为 AutoOnCall session ID，因此可通过持久化 Snapshot 查询。反例是模型虽然允许 `TASK_STATE_CANCELED`，系统却没有 cancel endpoint 或取消信号，不能声称支持取消。

### Artifact：对外交付物，不等于安全过滤器

通俗地说，Artifact 是 Task 交付的报告或结构化结果。严格定义上，它由 ID、名称、描述和若干 Text/Data Part 组成，用于承载 Run Status、Diagnosis Report、Evidence、Replay 或 Runbook Answer。它解决结果交换；它不天然保证每个字段都适合跨边界公开。

AutoOnCall 的 Evidence Artifact 是归一化观测，Report Artifact 同时提供 Markdown 与结构数据。但 Run Status Artifact 还包含计划、工具调用记录、过去步骤、错误、待审批信息等完整读模型。底层持久化会脱敏和压缩工具记录，Facade 自身却没有第二层字段 allowlist，所以不能笼统说“绝不会输出工具参数”。

### Incident：一次故障的归属边界

Incident 是“这次事故”的稳定业务上下文。严格定义上，它关联服务、环境、症状、严重度和 Incident ID，解决告警、诊断、审批、变更与报告归属；它不证明根因或恢复。

Redis 503 属于一个 Incident，另一次 MySQL 慢查询不能混入。diagnose Skill 可以接受结构化 Incident，也可以把纯文本症状构造成 Incident。当前调用方可以在结构对象中提供 Incident ID，因此生产化还需要租户和资源归属校验，不能只保护 Task ID。

### Evidence 与 Hypothesis：观测和解释必须分开

Evidence 是带来源、步骤和时窗的可追溯观测，解决“系统看到了什么”；它不等于知识，也不自动证明因果。Hypothesis 是对多条 Evidence 的可证伪解释，解决“这些现象为什么出现”；它不负责授权动作。

`connected_clients` 接近 `maxclients` 是 Evidence，“连接耗尽导致 503”是 Hypothesis。A2A Artifact 可以交付两者，但协议包装不会提高它们的证据等级。

### Trace：过程事件，不是模型私有思维

Trace 是按 Trace ID 和 Incident 关联的结构化事件账本，记录节点、工具、风险、审批和报告等过程。它解决审计、进度和复盘；它不等于外部系统当前状态，也不应暴露模型私有推理链。

AutoOnCall 的 A2A Status 和 Replay 复用 Trace 摘要与事件，但不会把自然语言“思维过程”当协议内容。Trace 若缺失，也不能由 Facade 编造一条成功时间线。

### Snapshot：最新运行书签，不是完整 Task 历史

Snapshot 保存一次 session 当前可恢复的业务状态，如 Incident、计划、Evidence、报告和进度。它解决服务重启后的有限查询与恢复；它不是任意节点的 durable checkpoint，也不是完整事件日志。

AutoOnCall 的 diagnosis Task 主要由 `AIOpsSessionSnapshot` 构造。Agent Card 虽声明 `stateTransitionHistory=true`，当前 A2ATask 的 `history` 实际始终为空，只有最新状态、Trace 摘要和 Artifact。准确说法应是“有可查询的最新状态与过程材料”，不能说“完整填充 A2A Task history”。

### Replay：从历史事实源重建解释，不重新执行诊断

Replay 是将 Report、Trace、Approval、IncidentState 与 ChangeExecution 组合成时间线和解释视图。它解决复盘；它不调用 Planner、Executor 或当前 Redis，也不能重现事故当时全部外部依赖。

若这五类事实源全部为空，AutoOnCall 返回 not found，而不是生成空白成功 Artifact。Replay Skill 每次返回一个即时 completed Task，但这个 Task 不单独持久化。

### Trust Gate：没有可靠依据时允许拒答

Trust Gate 是进入下一阶段前的可信门。严格定义上，它根据检索分数、来源、引用和策略决定回答、有限回答或拒答。它解决 Runbook 问答无证据硬答；它不保证知识库内容绝对正确，也不是通用 Prompt Injection 防火墙。

`answer_runbook_question` 复用现有 RAG 的 citation 与 no-answer policy。Facade 只包装结果，不重新实现一套检索策略。

## 为什么四个 Skill 已经覆盖协作主线

| Skill | 外部业务目的 | 最小 Scope | 内部复用 | 明确不做 |
|---|---|---|---|---|
| `diagnose_incident` | 启动一次受控 Incident 诊断 | `diagnose` | AIOpsService | 不开放底层 Tool，不改生产 |
| `get_incident_status` | 查询已有诊断运行 | `read` | Snapshot + Run Status | 不启动新诊断 |
| `explain_incident_replay` | 获取时间线、证据、审批与变更视图 | `read` | Replay Builder | 不重跑工具，不查询当前现场 |
| `answer_runbook_question` | 回答带引用的运维知识问题 | `read` | RAG Service | 不绕过 citation/no-answer |

这四项覆盖了 Redis 协作的“启动—查询—复盘—知识”闭环。审批和变更没有做成 Skill，不是漏实现，而是控制面隔离：外部 Agent 得知 Task 正在等待输入后，应交给已有 Approver 和 Safe Change 流程处理。

如果未来确实需要跨 Agent 审批，也不应简单增加 `approve` Skill。至少要定义审批主体身份、双人复核、Plan 版本、有效期、资源范围和不可委托规则，并确保它不能签发生产凭据。

## 业务门面为什么要分成五层

### 发现层只声明能力，不决定权限

调用方首先需要知道服务地址、输入输出格式和支持的 Skill。Agent Card 解决发现问题，因此只应出现稳定业务名称和必要示例。如果 Card 直接列出 Tool Registry、内部数据库或审批路由，攻击面在真正调用前就已经扩大。

发现层不做授权决策。公开 Card 能被访问，不等于访问者可以诊断；Card 里声明 Bearer，也不等于服务器真的验证了某个 Agent 的组织身份。AutoOnCall 只有在 API 鉴权开启时才把 Bearer Scheme 写入 Card，这种声明与运行配置的一致性值得保留，但还缺少启动时对“A2A 已开启而鉴权关闭”的生产级阻断。

### 归一化层把多种外形收敛成一个 Envelope

外部实现可能把内容放在顶层、`params`、`message.parts` 或 data 中。归一化层将它们收敛为 message ID、task ID、context ID、text、data 和 metadata，后续业务无需到处判断字段位置。

归一化不是验证。它应当在合并前定义字段优先级、冲突处理和大小上限，否则“支持多种输入”会变成“调用方可以选择最有利的解释”。当前实现更偏向宽容解析，适合 demo；生产实现应先用版本化 schema 验证，再归一化为内部不可变对象。

### Skill 路由层决定业务意图与最小 Scope

Skill 路由层不调用工具，只回答“这是什么业务”。完成解析后才能选择 read 或 diagnose Scope，这比所有消息统一要求 operator 权限更符合最小权限，也比先执行再检查更安全。

这里必须避免两个极端：一是自然语言模糊匹配到任意 Tool，二是任何字段缺失都自动默认高权限动作。AutoOnCall 的隐式默认最终落到只读 Runbook，而不是诊断，这是相对保守的；但隐式 status 与 runbook 仍可能掩盖客户端错误，所以正式协议最好要求显式 Skill。

### 领域委托层复用原有不变量

业务 Skill 被确认并授权后，Facade 调用现有 AIOpsService、RAG Service 或读模型。它不重新排序工具、不自己批准、不解释 ChangePlan 为执行命令。这样 Redis 诊断仍由 Planner 决定取证顺序，Executor 仍生成 Evidence，Risk Controller 仍可等待审批。

这一层是 Facade 与普通 API 聚合器的根本区别。普通聚合器可能只是顺序调用若干端点；业务委托层必须保证无论入口来自 Web UI、REST 还是另一个 Agent，领域不变量只有一份实现。

### 输出映射层压缩状态，但不能改写事实

最后，Facade 把内部对象映射为 Task、Status、Artifact 和 SSE。映射的目标是降低外部认知成本，而不是把复杂状态美化为成功。`waiting_approval` 可以压缩为 input-required，因为调用方需要行动；`rollback_recommended` 却不能因为没有专门分支就落到 completed 后被误读为恢复。

输出层还应承担最小披露责任：明确哪些字段是外部契约、哪些只是内部调试。当前复用 Run Status 让信息完整，却也让契约面随内部模型增长。生产化时应让 A2A DTO 与内部读模型分离，并通过契约测试维护映射。

## 一条消息进入 Facade 后，系统实际上怎样理解它

### 入参兼容很宽松，不能当成严格协议验证

API 入口接收的是普通 `dict`，没有把请求体绑定到 A2AMessage Pydantic 模型。解析器同时接受类 A2A JSON 和简单 JSON-RPC 风格：从 `params`、`message`、`parts`、`data` 与 `metadata` 中合并字段。消息中的 `role` 不参与业务校验，额外字段也不会因模型规则被拒绝。

这种宽松解析便于本地兼容，代价是歧义。多个 Data Part 中后出现的字段可以覆盖先前字段，data 内 metadata 又能覆盖外层 metadata。生产接口应定义唯一 schema、请求大小上限、冲突字段规则和签名覆盖范围。

### 显式 Skill 与隐式推断不是一回事

若请求显式提供 `skill`、`skill_id` 或 `intent`，系统直接使用该值；显式未知 Skill 会返回 400，公共错误不会泄漏内部工具名。

若没有显式 Skill，解析器会按内容推断：有 Incident 对象则诊断；请求 Replay 则解释 Replay；有 Incident ID 或 Task ID 则查询状态；都没有则默认 Runbook 问答。因此“未知 Skill 一律拒绝”只适用于显式未知值。对安全敏感的生产协作，推荐要求显式 Skill，避免一个字段缺失就落到另一个业务能力。

### 诊断 Task ID 由服务端生成

客户端即使发送一个已有 taskId，diagnose Skill 也会生成新的 `a2a-diagnosis-<uuid>`，防止覆盖或读取已有敏感 session。该 ID同时成为 diagnosis session ID，之后可从 Snapshot 查回。

服务端拥有 Task ID 是必要边界，却不是全部租户隔离。Incident ID 和 contextId 仍可能来自请求；读接口也能按已有 session ID 查询。因此生产化要把 Agent 身份、租户、Incident 和 Task 资源权限绑定起来，而不是只检查字符串格式。

## 开关与鉴权必须同时成立

### 默认关闭只防止意外发布新路由面

`a2a_enabled` 默认是 `false`。公开 discovery、消息、流和任务接口都在业务逻辑中检查它；默认本地访问 well-known Card 会得到 404。路由虽然始终挂载到 FastAPI，Feature Flag 决定是否对外工作。

但“所有入口第一步都检查开关”并不完全准确。extended Card、Task 查询和列表把 `require_scope` 写成 FastAPI dependency，依赖可能在 endpoint 内的 `ensure_a2a_enabled` 之前执行；当 API 鉴权开启且请求无 Token 时，调用方可能先看到 401/403/503，而不是 404。消息接口则在函数中先检查 Feature Flag，再做 Skill 级鉴权。

### 开启 A2A 不会自动开启 API 鉴权

`api_auth_enabled` 也默认是 `false`。如果只设置 `a2a_enabled=true`，读、诊断和任务路由仍按本地开放模式放行，Agent Card 也不会声明 Bearer Security Scheme。只有同时开启 API 鉴权后，静态 Token 和 Scope 才成为真实门禁。

鉴权开启时，Runbook、Status 与 Replay 要求 `read`；diagnose 要求 `diagnose`。viewer/read token 能只读，operator token 能诊断。系统支持 Bearer 和 `X-AutoOnCall-Token`，Card 只声明 Bearer。当前没有 OIDC agent identity、mTLS 绑定、短期凭证或资源级授权，静态 Token 不能写成企业级 Agent 联邦身份。

### 公开 Card 仍然需要部署层治理

well-known Card 启用后无需 Token，便于发现；extended Card 才要求 read。公开 Card 只列业务 Skill，不含凭据和低层工具地址，但仍暴露服务存在、版本和能力。生产部署需要评估是否应由网关限制 discovery，并补 TLS、网络策略、限流、审计和跨租户资源校验。

## Redis diagnosis 通过同步接口时发生了什么

外部 Agent 发送 `message:send`，显式指定 `diagnose_incident` 并附带 `order-service` 的结构化 Incident。

1. 接口确认 A2A 开启，解析 Skill，并要求 diagnose Scope。
2. Facade 忽略客户端 taskId，生成服务端 Task/session ID。
3. Incident 进入原有 Plan-Execute-Replan，工具调用仍受 Evidence、风险与审批链约束。
4. Facade 会把整个 diagnosis async generator 消费到结束。
5. 若存在 Snapshot，就从 Snapshot、Trace、Approval 和 Report 构建 Task；测试替身未保存 Snapshot 时，才用最后一个 terminal event 兜底。

关键事实是：`message:send` 不是“立即提交后台任务然后返回 submitted”。它会等待诊断流结束后返回当前或终态 Task。若诊断因为调整 Redis `maxclients` 等动作等待审批，返回状态可以是 input-required；这仍不表示 Facade 有 approve 能力。

已经知道的内容包括当前 Task 状态和已收集 Evidence；Runbook 是知识，根因排序仍是 Hypothesis。若只看到 input-required，外部 Agent 应说明“等待内部治理动作”，不能推断变更已经执行。

## SSE 把内部事件压缩成什么，又丢掉了什么

### 正常映射

`message:stream` 对 diagnosis 先发送 submitted。普通 planning、tool、progress 等内部事件统一映射为 working；`approval_required` 映射为 input-required；report 事件映射成 Artifact；error 映射 failed；complete 根据 AutoOnCall status 映射 A2A Task State。

SSE 的事件名固定为 `message`，data 中携带 JSON。Facade 传播结构化状态、报告和业务信息，不传播模型私有思维过程。

### 终态可能出现两种 final 表示

内部 complete 事件会被转换为 `final=true` 的 Status Update。诊断 generator 结束后，Facade 又会从 Snapshot 或末事件构造完整 Task，并再发一个 `{"task": ..., "final": true}`。因此当前流中可能出现两个 final 表示：一个终态状态更新，一个最终 Task。

这不一定影响本项目客户端，却说明它还没有经过完整协议兼容验证。调用方不能简单“看到第一个 final 就断开”，也不能假设只有一个终结帧。生产化应选定一种规范语义并加入断流、重连和重复 final 测试。

### 非诊断 Skill 并不真正流式

Runbook、Status 与 Replay 走 `message:stream` 时，Facade 内部仍调用同步业务函数，只发送一个包含 Task 的 final 事件。它们没有分阶段状态。

stream generator 会把 LookupError 和 ValueError 转成安全的 not_found/bad_request 数据事件；其他底层异常没有在这一层统一转成结构化 A2A error，可能直接结束流。网络结束仍不能被业务客户端自动解释成 completed。

## Task 状态映射为什么是有损压缩

AutoOnCall 的生命周期比 A2A Task State 更细。当前映射如下：

| AutoOnCall 状态 | A2A 状态 | 业务解释 |
|---|---|---|
| running/planning/executing/diagnosing 等 | `WORKING` | 仍在运行 |
| waiting_approval/approval_approved/waiting_manual_execution | `INPUT_REQUIRED` | 需要治理动作或人工结果 |
| approval_rejected/approval_cancelled/blocked/escalated | `REJECTED` | 自动路径不继续 |
| failed/precheck_failed/dry_run_failed | `FAILED` | 流程失败 |
| 其他任何非空状态 | `COMPLETED` | 当前实现的默认兜底 |
| 空状态 | `UNKNOWN` | 无法判断 |

最后两行最值得警惕。`rollback_recommended`、`needs_human`、`degraded`、`incomplete`、`created`、`change_validated` 等没有单独分支，会落到 completed。A2A completed 只说明 Facade 映射认为任务终止，不能自动翻译成 Incident 恢复或报告完整。

模型列出了 `TASK_STATE_CANCELED`，但当前没有任何状态映射会返回它，也没有 cancel API 或向运行中 AIOpsService 传播停止信号。状态枚举存在不等于能力存在。

生产化应建立显式、穷尽的映射表：未知新状态默认 UNKNOWN 或失败关闭，而不是 completed；还要分别表达“诊断完成但证据不足”“等待审批”“建议回滚”和“生产已验证恢复”。

## 三类 Task 的持久化能力并不相同

### Diagnosis Task 可以查回

诊断用服务端 Task ID 作为 session ID，并保存 AIOpsSessionSnapshot。`GET /tasks/{task_id}` 从 Snapshot 加载 Trace、Approval 与 Report，生成最新 Task。若 Snapshot 不存在就返回脱敏后的 404。

Task 列表直接枚举最近的 AIOps session snapshots，按更新时间倒序。它并不只筛选 A2A 发起、`a2a-diagnosis-` 前缀的 session，因此本地 API 或其他入口创建的诊断也可能出现在 A2A Task 列表中。这个设计复用了统一运行视图，但不是独立 A2A Task 命名空间。

### Status Skill 是已有诊断的读视图

Status 优先按 task/session ID 查 Snapshot；找不到且有 Incident ID 时，读取该 Incident 最近一条 Snapshot。没有匹配就 not found。它不启动诊断，也不写新 Task。

按 Incident 查询“最近一次”方便使用，却可能掩盖同一 Incident 下多个 run。严谨客户端应保存服务端 Task ID；生产接口还应返回选择规则和 run 列表，避免把另一次诊断当成当前请求结果。

### Replay 与 Runbook Task 是即时结果

Replay 和 Runbook 每次生成新的 `a2a-replay-*` 或 `a2a-runbook-*` ID，直接返回 completed Task。它们没有写入 AIOps Snapshot 或独立 Task Store，随后用该 ID调用 `GET /tasks/{id}` 会找不到。

所以“所有 A2A Task 都可持久化查询”是不准确的。当前只有诊断 Task 借助 session Snapshot 具备该能力；只读 Skill 的 Task 更像一次带 Task 外形的同步响应。

## Artifact 边界要按字段而不是按名字审查

诊断 Task 至少包含一个 Run Status Data Artifact。若有报告，再附 Markdown+Data Report Artifact；若有 Evidence，再附 Evidence Artifact。Replay 返回组合历史视图，Runbook 返回回答、citation、retrieval 与 no-answer metadata。

Run Status 读模型包含 Incident、计划、执行步骤、past steps、tool call records、Evidence、Hypothesis、风险、待审批、变更计划、错误、警告、进度和链接。工具记录在进入持久化前会做敏感信息脱敏、大载荷压缩，但 Facade 没有为外部 Agent 定义独立投影。若内部字段未来增加，可能自动进入 Artifact。

这体现了复用读模型的取舍：开发快、内外状态一致，但外部最小披露边界不稳定。生产化应建立版本化 A2A DTO 与字段 allowlist，对工具参数、错误文本、审批 metadata、内部 URL 和大型载荷做二次审查；敏感字段缺失测试应作为发布门禁。

## Replay 为什么是安全的只读协作，但仍有边界

viewer 请求 `explain_incident_replay` 并提供 Incident ID。Facade 加载 Report、Trace、Approval、IncidentState 与 ChangeExecution，全部为空时返回 not found；有任一事实源时交给 Replay Builder 生成 timeline、Evidence 质量、审批/变更流和报告摘要。

这条路径不调用 Planner、Executor 或 Tool Registry，也不查询当前 Redis。它回答的是“历史记录显示当时发生了什么”，不会拿今天的 `connected_clients` 冒充事故窗口事实。

但只读不等于无风险。Replay 可能包含审批人、动作说明、内部链接和工具摘要；read Scope 是否允许看到整个 Incident 还取决于资源级权限，而当前静态 Scope 没有租户/服务归属。Facade 也不会修复底层事实源之间的不一致，只会组合现有材料。

MySQL 与 Kubernetes 可复用同一 Replay 抽象，但统一 Artifact 不代表底层 Evidence、dry-run 或变更语义相同。

## 用 Redis 事故走完四种委托，认知怎样一步步变化

### 第一次委托：diagnose 只增加受控 Evidence

Incident Commander 先发送结构化 Redis Incident。AutoOnCall 查询指标、日志、Redis 状态和 Runbook，外部 Agent 获得 working 更新，随后看到 Report 或 input-required。

这一阶段新增加的是 Evidence：哪些指标来自哪个工具、哪个时间窗，日志和连接数是否对齐。Runbook 仍是知识；“连接耗尽导致 503”仍是 Hypothesis。若关键数据源失败，Report 应保留缺失证据，Task 不应因为诊断 generator 正常结束就掩盖降级。

下一步为什么可能等待审批？因为“读取 Redis 状态”和“提高 maxclients”不是同一能力。前者可由只读诊断完成，后者必须进入内部风险与审批链。Facade 把 `waiting_approval` 映为 input-required，提醒外部协调者有人类动作缺口，却不给它 approve Skill。

什么情况下停止？Incident 无法构造、Token 没有 diagnose Scope、显式 Skill 未支持时，在诊断开始前停止；工具失败或 Evidence 不足时，领域服务降级或转人工；命中 forbidden 时，不能把 rejected 再包装成 completed。

### 第二次委托：status 回答“现在到哪”，不重新诊断

外部 Agent 保存服务端 Task ID，稍后调用 status。Facade 读取同一个 Snapshot，并结合 Trace、Approval 和 Report 生成 Run Status Artifact。它不会再次查询 Redis，也不会产生第二次诊断成本。

当前已知的是持久化状态最后更新到哪一步；仍未知的是 Snapshot 之后外部 Redis 是否变化。如果 Task 显示 input-required，调用方可以通知值班人员；如果显示 completed，还必须看 `autooncall_status` 和 Report 证据边界，不能只看 A2A 的粗粒度状态。

若客户端丢失 Task ID，只给 Incident ID，系统会选最近 Snapshot。这是方便的 fallback，不是精确关联。多个诊断 run 并存时，应把“选中了哪一个 session、更新时间是什么”展示给调用者。

### 第三次委托：Replay 回答“当时为何这样判断”

事故交班时，另一个只读 Agent 请求 Replay。它得到由 Trace、Approval、ChangeExecution、Report 和 IncidentState 组合的时间线：何时看到 Redis 超时，何时形成 Hypothesis，何时请求审批，安全变更走到哪一步。

Replay 中“人工回执成功”是一条历史事实，“Redis 已独立验证恢复”只有在存在可信观察证据时才能成立。Facade 不应该因为 ChangeExecution `closed` 或 Task `completed` 自动增加因果结论。事实源冲突时，应保留冲突，而不是选最乐观的一项。

缺少全部事实源时返回 not found；只有部分事实源时可以有限 Replay，并清楚标明缺失。当前 Builder 能组合已有对象，但 Task 包装本身不提供完整性评分，调用方仍需读取 Artifact 内容。

### 第四次委托：Runbook 回答“通常怎样处理”

新的协作 Agent 可能只问“Redis maxclients 告警如何排查”。这个问题不需要启动 Incident 诊断，应走 read-only Runbook Skill。RAG 返回回答、citation、retrieval 信息和 no-answer policy，Task 立即 completed。

这里的 completed 表示这次知识问答完成，不表示任何 Incident 完成。引用内容是知识，不是当前现场 Evidence。若 Trust Gate 拒答，协议仍可能把函数执行结束包装为 completed，但 Artifact 中 `no_answer=true` 才表达业务结果；这再次说明 Task State 与答案质量必须分开看。

四次委托合起来形成清晰分工：diagnose 产生本次事故的受控观测和报告，status 读取最新状态，Replay 解释历史，Runbook 提供通用知识。没有一项授予生产变更权。

## Facade 的威胁模型不只是“防止调用危险工具”

### 混淆代理：调用者借服务的身份做自己无权做的事

Facade 自身可以访问内部诊断服务和读模型。如果它只检查“Token 有效”而不检查 Skill 与资源，低权限调用者就可能借 Facade 读取其他团队 Incident，或让系统替它发起昂贵诊断。这类问题叫混淆代理。

Scope 能区分 read 与 diagnose，但当前没有 service、tenant、environment 级授权。生产方案要把 Principal、Incident、Task 和数据域放在同一授权判断中，并把决定写入审计。

### 标识符注入：Task ID 安全不代表 Incident ID 安全

服务端生成 diagnosis Task ID，防止客户端覆盖已有 session，这是重要措施。但客户端仍可提供 Incident ID 和 contextId；Status/Task 接口又能按标识符读取 Snapshot。若 ID 可猜且没有资源 ACL，就可能发生越权读取。

错误消息已避免回显不存在的 Task ID，降低枚举信息泄漏，却不能替代资源授权。还需使用高熵 ID、租户绑定、访问审计和异常枚举告警。

### 数据外泄：内部读模型可能比外部合同更宽

Run Status Artifact 能包含工具参数、错误、计划、内部链接和审批摘要。虽然持久化链会脱敏常见密钥，未知敏感字段或新加字段仍可能穿透。Replay 也可能包含操作者和变更原因。

因此应实施两次最小化：进入 Store 前做通用脱敏，跨 A2A 边界时再按外部 DTO 过滤。测试要使用嵌套 Token、URL 查询参数、日志正文和未知字段，而不只测固定 `password` 键。

### 成本与可用性攻击：只读也可能很贵

Runbook 检索、完整 Replay 和诊断都可能消耗模型、向量库、数据库与日志查询资源。只读 Scope 不代表调用廉价。message:send 又会保持连接直到诊断结束，容易被大量长请求占满 worker。

生产面需要按 Principal 和 Skill 做速率限制、并发配额、超时、最大 Incident 大小和成本预算。诊断还要具备服务器端异步队列，而不是让 HTTP 请求承担后台任务生命周期。

### 语义注入：外部 Incident 是数据，不是系统指令

Incident 症状、日志片段和 Runbook 问题都可能包含“忽略规则、直接执行工具”等文本。Facade 不应据此动态创建 Skill；领域服务也必须继续依赖确定性 Tool Policy。固定 Skill 表缩小了协议入口，但不能代替下游 Prompt Injection 防护。

### 状态误导：completed 是最危险的乐观默认

调用方往往只看 Task State 做自动化决策。如果 needs_human、rollback_recommended 或未知新状态被压成 completed，上游可能自动关闭工单或通知恢复。状态误导虽然没有直接执行命令，也能造成严重业务后果。

安全默认应是未知即 UNKNOWN、需要人工即 INPUT_REQUIRED，只有明确的业务终态才能 COMPLETED。上游还应同时检查 Artifact 中的领域状态与证据等级。

## 失败路径最能暴露真实边界

- A2A 默认关闭：公开 Card 测试得到 404；对于带依赖鉴权的路由，实际错误顺序还受 auth 配置影响。
- 显式未知 Skill：返回 400 和公共错误文本，不泄漏 `Unsupported A2A skill` 内部细节。
- read token 发起 diagnosis：返回 403；Skill 推断发生在鉴权前，以便选择最小 Scope。
- 请求 approve、change 或底层 Tool：不在固定 Skill 表中，显式请求被拒绝。
- 客户端伪造已有 diagnosis taskId：服务端生成新 ID，不覆盖旧 Snapshot。
- Task 查询不存在：返回通用 404，不回显可能含敏感信息的请求 ID。
- Replay 五类事实源全空：not found，不构造空成功。
- 请求 cancel：没有 endpoint、状态迁移和运行信号，必须明确不支持。
- SSE 中发生未分类底层异常：当前没有完整 A2A error envelope 兜底，客户端只能按断流处理为未知。
- Snapshot stale 或缺失：沿用底层读模型与 fallback，Facade 不会提高恢复能力。

## “可以互相发送 JSON”离互操作还有四级距离

### 第一级：载荷兼容

双方能解析 Agent Card、camelCase 字段、Message Part、Task、Artifact 和媒体类型，只能证明语法兼容。当前单元测试主要覆盖这一层的一小部分，而且还特意断言 Part 中没有 `kind` 字段，说明它依赖本项目的轻量对象形状，不能直接外推到其他实现。

### 第二级：时序兼容

双方必须对 submitted、working、Artifact、input-required、final、错误和断流有相同理解。两个 final 是否允许、哪个事件后可以关闭连接、重连是否会重放、事件如何去重，都属于时序契约。只测同步 `message:send` 无法证明流式兼容。

### 第三级：业务语义正确

即使字段和时序一致，completed 是否表示诊断完成、知识回答完成还是 Incident 恢复，仍可能不同。input-required 是等待审批还是缺少用户输入，也需要 metadata 或扩展契约解释。业务语义测试应使用 Redis 主线检查 Evidence、Report、Approval 与 Task State 是否一致。

### 第四级：安全与故障互操作

最后还要验证身份、Scope、资源归属、超时、取消、重复提交、服务重启、下游失败和敏感信息。第三方 Agent 能成功调用一次 happy path，不能证明互联网暴露安全。只有四级都有可复核 Artifact，才适合谈跨实现互操作；conformance 仍需以正式规范和套件为准。

## 当前设计的取舍与生产化方向

### 固定业务 Facade 为什么适合当前项目

它保留了原诊断、Evidence、风险、审批和变更链，外部 Agent 不需要理解低层工具；Task 复用 Snapshot，Replay 复用既有事实源，减少双写。代价是协议语义较轻、粒度较粗，部分 Task 只是同步结果，状态映射也有损。

与“每个 Tool 一个 Skill”相比，固定 Facade 的扩展速度慢一些，却更容易审查副作用。与独立 Task Store 相比，它减少一致性问题，却无法为 Runbook/Replay 提供持久 Task，也会把非 A2A session 暴露到列表。

### 生产化至少要补八类能力

第一，使用明确版本的官方规范/SDK，加入 schema 验证、版本协商、媒体类型和 conformance suite。

第二，接入 Agent identity：OIDC/mTLS、短期凭证、租户和资源级授权，而不只是一组静态 Scope。

第三，明确异步语义：message send 是提交还是等待完成，Task 怎样持久化，何时可查询，重试如何幂等。

第四，穷尽状态映射。未知状态默认 UNKNOWN，不让 `rollback_recommended` 或 `needs_human` 落成 completed。

第五，实现真实取消与超时传播：持久化 cancel request，向运行图和在途工具发送协作式停止信号，并记录无法取消的副作用。

第六，统一 SSE 终态、事件 ID、断流重连和重复事件规则，避免两个 final 造成客户端歧义。

第七，建立外部 Artifact DTO 与字段 allowlist，做敏感信息、内部链接和 payload 大小测试。

第八，补 push notification、限流配额、审计、第三方互操作和故障注入，再逐级形成 local_live、controlled_fault 与 production Artifact。

## 面试时怎样把“Agent 协作”讲准确

### 一句话定位

AutoOnCall 用默认关闭的业务级 A2A-compatible Facade，让外部 Agent 委托诊断和读取结果，却拿不到底层工具、审批与生产变更控制面。

### 30 秒回答

我没有把每个工具直接开放，而是固定四个 Skill：diagnosis、status、Replay 和 Runbook。Facade 按 Skill 选择 read 或 diagnose Scope，复用现有 AIOpsService、Snapshot、读模型和 RAG，再映射为 Task、Artifact 与 SSE。诊断 Task 可查回，Replay/Runbook Task 目前只是即时结果；A2A 与鉴权都默认关闭。当前是轻量兼容面，没有 cancel、push、OIDC 或 conformance，状态和 Artifact 还有需要收紧的边界。

### 3 分钟展开

先讲为什么：直接公开 OpenAPI 或 Tool 会让上游绕过 Planner、Evidence、Risk 和 Approval，也把内部状态机耦合给每个调用方。

再讲输入和权限：Facade 解析消息中的固定 Skill，显式未知值拒绝；只读 Skill 要 read，诊断要 diagnose。diagnosis taskId 由服务端生成，防止覆盖已有 session。但入参目前是宽松 dict，缺少严格 schema 和资源级授权。

然后讲执行和结果：message:send 会消费完整诊断流后返回；message:stream 映射 working/input-required/report/complete，当前可能有两个 final。Diagnosis Task 复用 Snapshot 可查询；Status 读最近运行；Replay 组合历史事实源但不重跑；Runbook 复用 citation/no-answer。

最后讲边界：任意未识别的底层状态当前可能默认 completed，Run Status Artifact 也复用较宽的内部读模型；Card 声明历史能力但 Task history 为空。13/13 是 dirty worktree 自动化测试，其中只有 8 个 A2A 专项 case，不是协议认证或生产证据。

### 可写入简历的准确表述

> 设计默认关闭的 A2A-compatible 北向业务 Facade，将 Incident 诊断、运行状态、Replay 和带引用 Runbook 问答映射为固定 Skill、Task、Artifact 与 SSE，复用既有 AIOps Snapshot/读模型并按 read/diagnose Scope 隔离权限，避免暴露底层工具、审批和生产变更入口；专项自动化测试 8/8 通过（offline_fixture）。

### 不可夸大的表述

- 不写“完整实现并通过 A2A conformance”。
- 不写“基于官方 SDK 完成跨厂商 Agent 互操作认证”。
- 不写“支持任务取消、push notification 和断流恢复”。
- 不写“所有 A2A Task 都持久化”。当前主要是 diagnosis Task。
- 不写“Artifact 只包含 Evidence，绝无内部字段”。Run Status 是较宽的复用读模型。
- 不写“外部 Agent 可以审批或执行生产变更”。
- 不把静态 Token Scope 写成 OIDC Agent identity。
- 不把 dirty `13/13` 写成 official 或 production evidence。

## 高频追问与回答要点

### 1. 为什么不把每个 Tool 做成 Skill？

工具是内部能力，安全依赖计划、证据、风险和审批顺序。业务 Skill 委托完整闭环，能避免调用方任意重排或绕过门禁。

### 2. read 与 diagnose Scope 怎样选择？

先解析 Skill，再选择最小 Scope：Status、Replay、Runbook 用 read，diagnosis 用 diagnose。显式未知 Skill 在鉴权执行前被识别并拒绝。生产化还需资源级授权。

### 3. Task 状态来自哪里？

Diagnosis Task 从 AIOpsSessionSnapshot 加 Trace、Approval、Report 的 Run Status 构建，没有第二套任务数据库。Replay 和 Runbook Task 是即时响应，不可按 Task ID 查回。

### 4. 为什么 message:send 不算标准异步提交？

它会消费 diagnosis generator 到结束后才返回 Task，没有先持久化 submitted 再让后台独立运行。长诊断会占用请求生命周期。

### 5. input-required 代表什么？

它表示 AutoOnCall 状态需要外部治理动作，例如等待审批或人工变更结果；它不代表 Facade 能完成审批，也不代表 Task 失败。

### 6. 为什么 canceled 枚举存在仍不能说支持取消？

没有 cancel endpoint、状态映射、持久请求和运行信号。只把响应字段写成 canceled 不会停止后台诊断或在途工具。

### 7. SSE 的主要兼容风险是什么？

当前 internal complete 会产生一个 final Status，循环结束后又发 final Task；没有覆盖重连、重复事件和通用异常 envelope。需要规范化单一终态与 event ID。

### 8. Facade 如何避免扩大生产能力？

Skill 表不包含 Tool、Approval 和 Change；diagnosis 调用原有 AIOpsService，仍受内部 Risk Controller；只读 Skill 调读模型或 RAG。Facade 没有生产写 adapter。

### 9. `13/13` 为什么不能叫协议通过率？

其中 5 个是通用鉴权测试，8 个 A2A case 也未覆盖 SSE、Replay、Status、完整状态映射和第三方互操作。它只能描述这些断言在指定 dirty worktree 通过。

## 检验真正理解的练习

### 练习一：追踪 Redis diagnosis Task

从一个显式 `diagnose_incident` Message 开始，依次写出 Skill 解析、Scope、服务端 Task ID、Incident、AIOpsService、Snapshot、Run Status 和三个 Artifact。标出哪些是事实、知识和推断，以及等待审批时为什么不能调用 change。

### 练习二：修正状态映射

为 completed、incomplete、needs_human、waiting_approval、approval_rejected、change_validated、rollback_recommended、resolved 设计穷尽映射。说明未知新状态为什么应 UNKNOWN，而不是默认 COMPLETED。

### 练习三：设计真正的取消能力

覆盖权限、Task 归属、请求幂等、cancel requested/canceled/cancel failed 状态、运行图信号、在途只读工具、不可撤销外部副作用、Snapshot 和 Trace。解释“收到取消请求”和“已经停止”为什么是两件事。

### 练习四：给 Artifact 做最小披露审计

列出 Run Status 当前字段，设计外部 allowlist。分别处理 tool input/output、error、approval metadata、内部 URL、Evidence、Report 和大型 payload，并为敏感值泄漏写负向测试。

### 练习五：设计第三方互操作测试矩阵

至少覆盖 Agent Card、显式/隐式 Skill、camelCase、media type、同步与流式、两个 final、断流重连、Task 查询、状态映射、错误脱敏、鉴权和取消。区分 schema 兼容、状态语义、业务正确性和生产安全四类结论。

## 实现证据与复现入口

理解原理后再查代码：路由与 Skill 级鉴权位于 `app/api/a2a.py`；协议对象位于 `app/models/a2a.py`；业务编排在 `app/services/a2a_facade_core.py`；固定 Skill 在 `app/services/a2a_skills.py`；消息归一化在 `app/services/a2a_messages.py`；Task、Status、Artifact 与 SSE 映射在 `app/services/a2a_payloads.py`；Feature Flag 与路径配置在 `app/config.py`；Run Status 读模型在 `app/services/aiops_read_models/run.py`。

```powershell
.\.venv\Scripts\python.exe -m pytest tests/test_a2a_facade.py tests/test_auth_rbac.py -q -p no:cacheprovider --no-cov
```

该命令验证现有 13 个断言，不会自动产生 official Artifact。若要升级证据，需要补齐未覆盖的 SSE、Replay、Status、Task 列表和状态映射测试，再与至少一个独立实现做带版本、环境、原始消息和结果 Artifact 的互操作实验。

验收时还应保留失败请求、断流位置和服务端 Trace；只有成功响应而没有失败样本，无法证明协作边界在异常情况下仍然成立。

# 工具调用成功，为什么仍不能直接当作事故证据

> 专题编号：07  
> 审计日期：2026-07-22
> 事实范围：当前仓库实现、自动化测试和已保存的本地评测材料；除明确标注外，不代表生产运行证据。  
> 上一章已经解决：Agent 怎样规划、执行、补查并有界停止。  
> 下一章将解决：异构工具回执怎样进一步变成可复核证据，并支撑或反驳根因假设。  
> 完整量化口径见[第 15 篇：项目量化指标与校招验收](15-项目量化指标与校招验收.md)。

## Redis 已经返回数据，为什么结论仍可能不可信

继续使用贯穿全系列的故障：`order-service` 出现 503，P95 上升，应用日志出现 `Redis connection timeout`，`connected_clients` 接近 `maxclients`。Planner 生成一步 `query_redis_status(order-service, 10m)`，Executor 也拿到一个 JSON。很多入门实现到这里就会宣布“工具调用成功，根因是 Redis 连接耗尽”。问题恰好从这里开始。

当前我们可能同时看见三类材料。503、P95、timeout 和某个时间窗口的连接数是现场观察；Redis 达到 `maxclients` 后可能拒绝新连接是通用知识；“连接耗尽导致本次 503”是仍待交叉验证的推断。若连接数来自当前空闲容器，而 503 发生在昨晚，字段再精确也没有对齐事故窗口；若来自本地生成数据或离线 fixture，它能验证链路，却不是生产事实；若工具只返回 `status=success`，没有真实 `source`，我们甚至不知道数据从哪里来。

新手最自然的做法，是在 Executor 里写一串 `if/elif`：看到 `query_logs` 就调日志 SDK，看到 `query_redis_status` 就调 Redis 客户端，异常时捕获并返回一句“查询失败”。这个方案在两个工具的演示里很快，扩展到 Prometheus、Loki、Redis、MySQL、Kubernetes、CMDB、工单和 MCP 后会出现具体失败：

- Planner 拼错一个名字，流程直接抛异常，还是静默转成模型自由回答？
- 同一个“查指标”既可能来自 MCP 的 CPU/内存，也可能来自 Prometheus 的 QPS/P95/error-rate；字段不齐时能否补一个看似合理的值？
- 日志权限失败、连接失败和超时是否都重试？第一次写操作已成功但响应丢失时怎么办？
- 一次返回十万行日志时，是把它塞入 LangGraph state、SQLite 和 Prompt，还是丢弃原文只留摘要？
- Planner 在只读日志检索参数中写了危险命令文本，策略引擎是在“执行命令”还是“搜索这个字符串”？
- 本地 MCP server 返回了生成样例，能否在报告里写成生产监控事实？

因此，工具工程不是“给模型绑定几个函数”，而是把一个不可信的字符串意图，逐层转换为有稳定语义、预算、来源、风险、失败状态和审计指针的观察。工具执行只负责获取材料；它不自动完成时间对齐、因果判断和根因证明。

本章沿用两个事故分析对象。**Incident（事故上下文）**通俗地说是“这次出了什么事”的档案；严格定义是绑定服务、环境、严重级别、症状、窗口和标识的诊断上下文。它解决不同工具围绕同一事件取证，不负责证明根因。例如 `order-service` 的 503 和时间窗口可以构成 Incident；把 `connected_clients` 写进 Incident 就宣称 Redis 是根因则是反例。在 AutoOnCall 中，它对应工作流 state 和持久化事故模型。**Hypothesis（根因假设）**通俗地说是一个可被推翻的解释；严格定义是需要证据支持或反驳的候选因果命题。它解决多种解释如何比较，不负责自动授权处置。例如“Redis 连接容量耗尽导致 503”是 Hypothesis；“Redis 返回过一次数据，所以假设成立”是反例。在 AutoOnCall 中，它由后续证据分析和排序链管理。

## 先把六个容易混淆的对象分开

工具系统之所以难讲清，往往不是代码复杂，而是“计划、调用、结果、证据、制品和轨迹”被口头统称为工具输出。它们承担不同责任。

### PlanStep：想做什么，不代表能做或已经做过

**PlanStep（计划步骤）**通俗上是一张待执行工单；严格定义是带 `step_id`、稳定 `tool_name`、目的、输入参数、预期证据、风险、状态和重试计数的机器对象。它解决 Planner 与 Executor 之间如何稳定交接，不负责证明工具已注册、参数已授权或结果可信。

“查一下 Redis”不是足够稳定的 PlanStep；“对 `order-service` 调用 `query_redis_status`，查看 10 分钟窗口的连接容量，预期获得 `connected_clients/maxclients`”才是。AutoOnCall 的 `PlanStep.input_args` 会限制序列化后的总长度不超过 `20,000` 个字符，`retry_count` 范围为 `0..3`，但这仍不是完整 JSON Schema 校验。

### Tool Contract：说明能力，不负责证明实现诚实

**Tool Contract（工具契约）**通俗地说是说明书；严格定义是稳定名称、描述、输入/输出 schema、风险等级、只读标记、超时、重试策略、数据源和降级策略组成的机器可读约定。它解决 Planner、Registry、UI 和审计者对“这个工具声称做什么”达成一致，不负责验证底层系统真的返回了正确数据，也不是一次调用的最终授权。

例如 `query_redis_status` 声明来源包括 Redis INFO、事故证据键和服务拓扑，风险低且只读。契约能约束名称和预期语义；它不能证明某次结果来自生产 Redis。Registry 会先复制入参、应用 `input_schema.properties` 的顶层默认值，再用 Draft 2020-12 JSON Schema 校验；缺少必填字段、类型不符或 schema 自身非法时，返回 `source=tool_contract`、`error_type=invalid_input` 的结构化失败，底层工具不会被调用。默认对象使用深拷贝，避免一次执行修改类级默认值。

成功回执离开 Registry 前也会经过 `output_schema` 校验，并要求能安全序列化为 JSON、大小不超过 `1,048,576` 字节；不合格时同样转为 `source=tool_contract`、`error_type=invalid_output`，而不是把坏 payload 当 success 送去 Evidence。测试覆盖缺字段、不可序列化值和超限输出。这仍不是“所有参数和输出都已强校验”：通用默认值只处理顶层 properties；当前十个标准工具的输出 schema 大多仅声明为 object，未统一表达字段级必填、跨字段语义、深层默认值或 schema version；`ToolExecutionResult.output` 仍保留 `Any` 以承载来源方言。准确说法是“非空输入和成功输出契约均在真实执行入口 fail closed”，而不是“所有工具已具备完整双向类型系统”。

### ToolExecutionResult：标准回执，不是根因结论

**ToolExecutionResult（工具执行结果）**是每个标准工具统一返回的 envelope，包含工具名、`success/failed`、实际输入、输出、耗时、风险、只读属性、公开错误和 metadata。它解决 Executor 不必理解每个 SDK 的异常方言；不负责把 output 自动升级为事故事实。

工具内部会规范化 `time_range`、`interval`、`limit` 等边界，并把规范值写回 `input_args`。因此审计回执记录的是实际执行参数，而不一定是 Planner 最初给出的原始值。例如指标窗口 `999h` 会被夹到 `1h`，间隔 `999m` 会被夹到 `5m`，日志条数 `99999` 会被夹到 `200`。这类局部约束很有用，但仍不能替代全局 schema 与业务语义验证。

### ToolCallRecord：证明调用发生，不解释调用意味着什么

**ToolCallRecord（工具调用记录）**通俗上是执行小票；严格定义是绑定诊断链、事故和步骤标识、工具、脱敏输入摘要、输出或外置结果指针、数据源、耗时、状态、风险、错误和执行 metadata 的审计记录。它解决“谁在什么诊断链上调用了什么、结果状态如何”可回放，不负责判断 output 支持哪个根因。

一次 `query_redis_status` 的 `status=success` 只能说明标准调用路径认为输出没有命中失败协议。它不能证明时间窗口正确、采样完整，更不能证明连接耗尽是 503 的唯一原因。

### Evidence：可复核观察，不是原始 JSON 的别名

**Evidence（证据）**通俗上是能被别人复查的观察；严格定义是在工具回执基础上补充 source、类型、支持/反驳/中立/未知立场、fact、inference、uncertainty、next step、confidence 和原始数据引用的对象。它解决分析器怎样消费异构回执，不负责永久保存所有原文，也不保证规则生成的 inference 必然正确。

AutoOnCall 会给真实配置来源、MCP、mock、规则、失败和未配置路径不同的启发式 confidence。例如真实适配器成功通常为 `0.82`，MCP monitor/RAG 为 `0.72`，未配置失败为 `0.05`，一般失败为 `0.1`，人工分析上限为 `0.35`。这些值是治理规则，不是经过统计校准的“根因正确概率”。下一章会进一步讨论 Evidence 如何支撑 Hypothesis。

### Artifact：外置后的持久化载荷，不天然等于原始数据

**Artifact（制品）**通俗上是放不进状态的大结果文件；严格定义是由 `artifact_id`、本地引用、SHA-256、字节数和截断标记描述的持久化输出。它解决状态、轨迹、报告和 Prompt 被大载荷拖垮的问题，不负责证明上游数据真实、文件永久存在或所有原始字段都被保存。

尤其要注意 AutoOnCall 的处理顺序：先递归脱敏 `input_args`、`output`、`error_message` 和 `metadata`；当原始外部载荷保存开关关闭时，再压缩已知外部 source 的 `raw`；最后才按字节阈值决定是否外置。因此 Evidence、ToolCallRecord 与 Artifact 使用的是同一份允许持久化的安全结果，不会出现 output 已脱敏、错误或 retry metadata 仍泄漏 secret 的分叉。Artifact 保存的是“允许持久化的脱敏后、可能已经压缩的输出”，不是无条件保存完整外部原包。

### Trace：记录发生顺序，不替代 Evidence

**Trace（轨迹）**是节点、工具、风险、审批和变更事件的时间序列。它解决回放“先计划、再调用、何时失败、为何暂停”，不负责把 ToolCall 自动解释为根因。TraceService 还会再次脱敏参数、结果、错误和 metadata，并写入 SQLite；这个二次防线不代表调用前、MCP 服务自身日志或第三方系统已经完成同等脱敏。

## 架构不是从目录长出来的，而是从四类变化推出来的

工具系统面对四种独立变化：Planner 的表达会变，风险规则会变，重试和持久化语义会变，外部 API 也会变。若它们都写进 Executor 的条件分支，任何一处变化都可能破坏整条诊断链。更稳定的推导是四层边界。

### 第一层：稳定动作语言

Planner 只使用有限的标准名称和结构化 PlanStep。名称不是 SDK 方法名，而是业务语义端口。`query_metrics` 表示“取服务指标”，不承诺底层一定是 Prometheus；`search_runbook` 表示“检索受可信资格检查约束的知识”，不承诺一定返回答案。

这一层降低模型输出空间，却也会牺牲任意插件即插即用。对于运维取证，稳定、可评测的十个动作通常比动态暴露几十个远端函数更容易治理。若未来接入新 MCP 工具，应先映射到标准语义或经过独立审核，而不是把远端 description 当作授权依据。

### 第二层：Registry 与执行前治理

Registry 按稳定名称保存工具对象、列出契约并提供统一入口。它像派单中心：负责找到谁能做、在执行入口再次运行 Policy Guard；不负责连接 Redis 或解释结果。直接调用未注册名称时，Registry 返回 `Tool is not registered` 的结构化失败。

修复后的注册边界还增加了三个 fail-closed 不变量。第一，同一 Registry 重复注册同名工具会抛错，不再由后注册对象静默覆盖前者；MCP/LangChain 发现列表构造成名称映射时也拒绝重名。第二，传入的 `PlanStep.tool_name` 必须与实际请求名称一致，否则按 high-risk `tool:step-name-mismatch` 拦截，不能拿“只读查询”的步骤上下文替另一个工具过策略。第三，测试或评测替换必须显式调用 `replace_for_test`，且只能替换已经存在的名称，避免普通注册路径在 fixture 中被无声改写。

但 Executor 为兼容旧计划还有一条不同路径：结构化工具名未注册时，不直接使用该 Registry 失败，而是进入 LLM + ToolNode fallback。fallback 只暴露 `get_current_time` 和 `retrieve_knowledge` 两个安全本地辅助工具，任何 MCP 危险工具都被过滤；所得文本仍被规范为 `failed`、`structured_tool_registered=false`、低置信 Evidence 和显式 warning，不能充当标准工具成功证据。`manual_analysis` 是另一个特例：可以返回 success，但 confidence 最多 `0.35`，必须与真实工具交叉验证。

### 懒加载与实例生命周期也是注册边界

导入 `app.tools` 只加载轻量基类和本地辅助导出，不再立即导入 Registry 及其 RAG/Adapter 依赖；只有访问 `ToolRegistry` 或 `create_default_tool_registry` 时才通过模块级 `__getattr__` 懒加载。这个边界减少“只想导入工具模型，却顺带初始化重依赖”的启动耦合，不改变公开导入方式。

每个 `AIOpsTool` 实例会深拷贝 class-level schema、数据源和 retry policy，避免测试或一次调用修改另一个实例的契约。默认 Registry 每次构建自己的十个标准工具实例。MCP client 则是另一种生命周期：全局实例初始化由异步锁串行化，并把 server 配置和 interceptor 实例写入 cache key；单例已经建立后再请求不兼容配置会明确失败，需要隔离配置时必须 `force_new=True`。这防止并发初始化产生多个“全局单例”，也防止先到请求把后续调用悄悄绑定到错误 server 或 interceptor。

### 第三层：AIOpsTool 统一预算与失败语义

标准工具继承同一基类。基类负责一个总 deadline、只读重试、结构化失败识别、公开错误和 attempts metadata；它不负责外部系统的参数转义。成功输出、返回 `status=failed` 的字典和抛出的异常，最终都进入同一种 ToolExecutionResult。

统一 envelope 的价值不是让所有输出长得完全一样，而是让 Executor 能稳定处理：成功继续、失败形成 Evidence、危险动作暂停、超大输出外置。底层 source 特有字段仍保留在 output 中，避免为了统一而抹掉关键语义。

### 第四层：Adapter 说外部系统的方言

Adapter 负责鉴权、连接、查询表达式、标签转义、时间窗口和错误分类。例如 Prometheus Adapter 构造查询，Loki Adapter 计算纳秒窗口，Redis Adapter 走 wire protocol 并区分当前 INFO 与事故证据键，Kubernetes Adapter 校验 label value 防止空选择器扩大查询范围。它不负责决定下一步还要查什么，也不负责批准变更。

四层形成的核心不变量是：Planner 不直接拿连接对象；Registry 不伪造外部数据；基类不解释业务根因；Adapter 不决定诊断流程。更换某个供应商时，只要标准语义、失败协议和 source 没有漂移，上层评测仍可复用。

## 十个工具不是十个生产系统

默认 Registry 注册十个标准工具。这是当前代码的确定事实，也是 `/api/aiops/tools/contracts` 返回的只读契约目录；该接口只列契约，不调用任何外部系统。

表中还会出现 **Trust Gate（可信门禁）**。通俗地说，它是知识进入回答或 Runbook 工具前的资格检查；严格定义是依据检索状态、来源、阈值和引用定位，将候选分成可信、拒绝或无答案的确定性边界。它解决低质量知识不能被静默当作依据，不负责验证本次事故的实时状态，也不证明每个自然语言主张都正确。例如没有可信 chunk 时返回 no-answer 是正确门禁；用一篇 Redis 手册断言本次 503 已确诊则是反例。在 AutoOnCall 中，它对应结构化 RAG 检索及 `search_runbook` 的拒答契约。

| 标准工具 | 当前语义与可能来源 | 明确不负责什么 |
|---|---|---|
| `query_metrics` | MCP monitor 的 CPU/内存，或 Prometheus 的服务指标 | 不用生成值补齐 QPS、P95、error-rate |
| `query_logs` | Loki、HTTP log gateway、CLS MCP | 空列表不自动等于没有异常；失败必须另有状态 |
| `query_service_context` | CMDB HTTP 或 MySQL 服务目录 | owner、依赖与业务背景不能证明实时根因 |
| `query_deploy_history` | 发布历史 HTTP 或 MySQL 表 | 时间相关性不自动等于发布导致事故 |
| `query_redis_status` | Redis INFO、事故窗口证据键、服务拓扑 | 当前 runtime 不替代历史事故窗口 |
| `query_k8s_status` | Kubernetes API | 默认面试栈没有生产 K8s 观测 |
| `query_mysql_status` | MySQL status SQL、事故证据表、服务拓扑 | 慢查询历史记录不自动授权改 SQL/索引 |
| `search_runbook` | Milvus/词法 RAG，经 Trust Gate | 知识告诉如何排查，不证明本次现场事实 |
| `search_history_ticket` | 工单 HTTP 或 MySQL 历史表 | 相似事故只是先验，不是当前根因 |
| `suggest_remediation` | 当前 Evidence 加规则模板 | 只生成建议，绝不执行变更 |

`suggest_remediation` 的 contract 是 `risk_level=medium`、`read_only=true`。Policy Guard 对它有专门规则：生成建议本身可以自动执行，建议里的 SQL、配置、扩容、重启仍需后续审批和安全变更工作流。只读不等于低风险，风险等级也不能单独推导是否有副作用。

### 来源选择顺序会改变结果语义

`query_metrics` 先尝试 MCP monitor，再尝试 Prometheus。仓库 MCP 只包装 CPU 和内存；即使两者都成功，`qps`、`p95_latency_ms`、`error_rate` 仍显式标为 unavailable。若 MCP 缺工具、只成功一项或返回结构化 failed，标准工具不会用 synthetic 值补齐，而是继续查询 Prometheus；Prometheus 成功时整体来源保持 `prometheus`，并把前一来源摘要写入 `fallback_errors`，不把两套字段拼成一个混合“完整观测”。若 Prometheus 也失败，返回最终失败和已尝试来源，而不是停在第一个已配置 Adapter。

本地 monitor MCP 已显式返回 `status`、`source=mock` 和 `synthetic=true`。若 CPU、内存都来自该 server，标准工具整体 `source=mock`；若只有部分字段是 synthetic，则记录 `source=mcp_monitor_mixed`、`synthetic_fields` 和 uncertainty。这样 MCP 协议调用成功不会再自动获得“实时监控”身份。

`query_logs` 按 Loki、HTTP gateway、CLS MCP 的顺序降级。某一来源已配置但调用失败时，失败不会被吞掉，也不会阻止后续来源：后备来源成功时成为整体结果，前序失败进入 `fallback_errors`；全部失败时保留最后可用的结构化失败。CLS 路径现在按传入的 `time_range` 计算 `start_time/end_time`，不再固定最近 10 分钟；topic 查询或日志查询返回 failed 时，整体工具也返回 failed，而不是把失败字典当成空结果或“未配置”。本地 CLS 同样显式标记 `source=mock`、`synthetic=true`。

`MultiServerMCPClient` 负责发现 LangChain Tool，包装层按其 `args_schema` 过滤参数并兼容 `ainvoke`、`invoke` 或 callable。返回值还要经过统一解包：支持 `CallToolResult.isError`、`structuredContent`、artifact 中的 structured content、`content_and_artifact` tuple、TextContent 对象、文本块列表和 JSON 文本。远端 `isError=true` 会规范为 `status=failed/error_type=mcp_error`，结构化失败不再因为外层对象“调用成功”而被误判。远端发现仍只是互操作，不是治理完成；仓库 monitor/CLS server 使用生成数据，显式 mock 标记只能防止冒充，不能把它升级成生产观测。

## Redis 主线：同一个字段可能来自两个时间世界

Redis Adapter 是理解“调用成功不等于证据成立”的最好例子。它连接配置的 Redis，读取当前 INFO、可选 CONFIG/SLOWLOG，并尝试读取为演示种入的 incident evidence、evidence hash、timeline 和 hotkeys。返回中可以同时出现：

- `live_info`：当前容器 runtime 的 `connected_clients/maxclients`；
- `incident_evidence`：Redis 键保存的回放事故窗口数据；
- `evidence_timeline`：种入的历史事件、hotkey 上下文和当前 INFO；
- `uncertainty`：明确声明当前 runtime 不能替代事故窗口。

事故回放数据现在会按请求窗口过滤：Hash 中的 `observed_at`、`timestamp`、`created_at` 或 `event_time` 早于窗口起点时不会覆盖 INFO；Stream 则通过完整 RESP 数组解析 `XREVRANGE` 的 event ID 和 field/value，再按 ID 或事件时间过滤。窗口内没有历史记录时，诊断字段退回当前 INFO。因此 `1s` 查询不会继续拿数天前的 `9940/10000` 冒充当前状态，而覆盖事故时刻的窗口仍可保留历史证据。

在贯穿案例中，若窗口内 incident evidence 报告 `connected_clients` 接近 `maxclients`，而当前 INFO 已恢复正常，正确表达是“事故窗口记录支持容量耗尽假设，当前实例可达且此刻不饱和”。错误表达是“现在 Redis 仍然连接耗尽”或“因为当前正常，所以昨晚没有故障”。工具工程要保留两个时间世界；Evidence Analyzer 才能在下一层决定它们如何支持 Hypothesis。

若 Redis 未配置，`query_redis_status` 返回 `source=redis_info`、`error_type=not_configured` 的 failed，不会因为 `aiops_mock_fallback_enabled=true` 就伪造 `9950/10000`。这时已经知道的是“取证链没有连接 Redis”；缺少的是现场连接状态；下一步应配置真实 Adapter、选择其他独立 Evidence 或转人工，而不是把工具失败解释成 Redis 正常。

实例路由同样 fail closed。只要调用方显式提供 `redis_instance` 或 `mysql_instance`，配置映射中找不到就返回 `not_found`，不会静默退回默认 URL/DSN 查询另一个实例。默认端点只服务于调用方没有指定实例的情况；这避免拓扑拼写错误把错误集群的数据包装成目标服务证据。

## Prometheus 与 MySQL 字段为什么必须把窗口和单位写进名字

Prometheus Adapter 已从 instant query 改为 `/api/v1/query_range`。`time_range` 决定 `start/end`，`interval` 决定 `step`；QPS、错误率和 P95 模板中的固定 `[5m]` 会替换为夹取后的请求窗口。响应必须是成功对象并包含可解析的 `data.result`；每条序列从后向前寻找最后一个有限样本，`NaN`、`+Inf`、`-Inf` 和空序列都按 missing 处理。若五项查询全部无数据，整体 failed；部分缺失则 `data_quality=partial`，不能把非有限值判成 normal。

这仍不是完整时间序列分析。当前聚合取每项 range query 的最后一个有限样本，不计算整个窗口的 max、avg 或趋势；把 instant query 改成 range query解决的是“参数真正参与查询”和“非有限值不冒充正常”，不等于已经实现异常检测算法。

MySQL Adapter 则把数据库服务端连接和应用连接池拆开。`Threads_connected`、`Max_used_connections` 与 `max_connections` 来自当前 MySQL `SHOW GLOBAL STATUS/SHOW VARIABLES`；`pool_active`、`pool_max`、`pool_waiting` 来自应用事故证据表。输出不再用 `active_connections/max_connections` 混写两种单位，Evidence chain 也明确说明 pool waiting 不是 MySQL 原生 counter。这样 `188/200` 可以表示应用池占用，却不能再被读成 MySQL server 当前已有 188 条连接。

## “最多重试两次”为什么仍可能包含三套重试

重试的第一个问题不是次数，而是“谁在重试哪一层失败”。AutoOnCall 当前实际存在三层机制，若只说两层，会漏掉 MCP interceptor。

### MCP interceptor：远端协议调用内的最多三次

MCP 客户端默认加入重试拦截器。它对 handler 异常最多尝试 `3` 次，退避为 `1s`、`2s`，最终用 `CallToolResult(isError=true)` 返回失败。这是 MCP 调用内部的恢复逻辑，不等于一个新的 PlanStep。

拦截器会拒绝 `max_retries<1` 和负 delay；收到 `asyncio.CancelledError` 时立即向上传播，不进入退避，也不伪装成普通失败。其他异常只在日志和最终错误中保留异常类型与短 `error_id` 指纹，不回显可能含 Token、Authorization 或连接串的原始异常文本。

这层重试目前主要写日志，AIOpsTool 的 `metadata.retry.attempts` 不会完整展开每次 MCP handler attempt。因此“ToolCallRecord 显示 outer attempt_count=1”不一定表示远端只收过一次请求。对于只读查询问题较小；若未来让 MCP 承载写动作，必须统一幂等、预算和审计，不能把隐藏重试留在远端包装层。

### AIOpsTool：一个工具步骤共享一个总 deadline

第二层是标准工具基类。contract 声明 `max_attempts`、`backoff_seconds` 和 `retry_on`。只有 `read_only=true`、failure kind 在 allowlist 且输出没有 `retryable=false` 时，才启动下一次 outer attempt；非只读工具强制 `max_attempts=1`。

所有 outer attempts 共用一个 `timeout_seconds` deadline，不是每次都拿到完整预算。可用伪代码表达：

```text
deadline = now + timeout_seconds
for attempt in 1..max_attempts:
    remaining = deadline - now
    output = wait_for(call(), remaining)
    if success: return
    if not read_only or kind not allowed or retryable=false: stop
    if backoff cannot fit remaining budget: stop
```

有一个容易误解的细节：若 `_call` 真正阻塞到 `asyncio.wait_for` 超时，它已经耗尽本次总 deadline，基类立即标记 `total_timeout_exhausted`，不会启动“幽灵重试”。测试中的三次 timeout 后恢复，是工具快速返回结构化 `error_type=timeout`，每次没有耗尽 deadline。权限失败不在标准 retry_on 中并且常带 `retryable=false`，因此一次停止；`not_configured` payload 虽标 `retryable=true`，但 retry_on 不包含 `not_configured`，同样通常一次停止。

每次 outer attempt 记录序号、状态、failure kind 和 latency；最终 metadata 还记录 attempt_count、max_attempts、retried、retry_exhausted、stop_reason 和 total_timeout_seconds。`retry_exhausted=false` 不能简单解释为“没有失败”，它也可能表示总预算在第一次超时已耗尽。

外部取消也不会进入普通失败 envelope。`AIOpsTool` 对 `asyncio.CancelledError` 直接重新抛出，因此上层任务取消不会被错误记录成一次可重试 dependency failure。这里实现的是协作式取消传播，不等于整个 AIOps Task 已具备持久 cancel API；后者仍需运行图、Snapshot 和外部副作用共同设计。

### Replanner：失败已经成为 Evidence 后的新步骤

第三层发生在工作流。AIOpsTool 已结束，失败回执先形成 ToolCallRecord 和 Evidence；Evidence Analyzer 判断这项 Evidence 是否关键，才可能生成 `<step_id>-retry`，Replanner 把它作为新的 PlanStep。再次失败会被识别为 retry exhausted，避免无限追加。

三层的决策上下文不同：MCP interceptor 只看 handler 异常；AIOpsTool 看只读性、failure kind 和 deadline；Replanner 看整个 Incident 还缺什么证据。生产化应把三层 attempt identity、统一 deadline 和幂等策略串起来，并考虑 circuit breaker、限流和供应商侧 retry-after，否则嵌套 `3 × 2` 理论上可能放大为最多六次底层请求，虽然外层总预算会限制实际次数。

## 四路并发为什么不是把整个计划扔进 gather

串行查询指标、日志、Redis 和 Runbook 会把外部等待时间相加。朴素优化是 `asyncio.gather(*all_steps)`，但它会把后续依赖步骤、审批动作和未知工具一起启动，破坏计划语义。AutoOnCall 只从 `current_plan` 开头挑选相邻步骤，并同时满足五个条件：

1. PlanStep 自报 `risk_level=low`；
2. 工具名位于显式只读取证 allowlist；
3. Registry 能找到工具，且 contract 的 `read_only=true`；
4. Policy Guard 再判定 `allow`、只读且最终风险低；
5. 当前批次不超过 `4` 步。

allowlist 当前有九个查询/检索工具，不包含 `suggest_remediation`。因此指标、日志、Redis、MySQL、Kubernetes、Runbook、历史工单、服务上下文和发布历史可以参与；建议生成即使 read-only，也因中风险和语义不同而留在后续单步。

`asyncio.gather` 返回后按任务创建顺序组装 ToolCallRecord、Evidence 和 plan state。并发降低等待，却保持回放确定性。一个工具返回 failed 不取消同批其他项；若某个 coroutine 意外抛到 fan-out 包装层，也会转成公开失败 Evidence。批次 metadata 保存 batch_id、size、index 和 `bounded_read_only_fanout`，审计者能区分串行与并行。

遇到不合格步骤时选择立即停止扩批，而不是跳过它继续抓后面的安全步骤。假设第二步日志 query 包含 `kubectl delete pod`，第一步指标可以单独执行，第二、三步留给下一轮策略处理。这种前缀语义简单、可预测，代价是一个被阻断步骤会暂时挡住后续安全查询。

## 双重 Policy Guard 能防旁路，也会产生文本误报

Executor 在调用前做一次风险检查，用于更新工作流状态、Trace、Evidence 和审批；Registry 又在统一执行入口检查一次，防止其他调用者绕开 Executor。双门禁解决的是入口旁路，不代表两套独立授权系统。

Policy Guard 结合 contract 与 PlanStep 推导 read-only 和最高风险，并扫描工具名、purpose、expected evidence、input args 以及 Incident 文本。命中 `rm -rf`、`kubectl delete`、`DROP TABLE` 等规则时 forbidden；重启、扩缩容、回滚、调整 maxclients 等通常 approval_required；普通低风险只读查询 allow。生产环境中的非只读动作会提升审批风险。

规则扫描有一个真实取舍：若用户要在日志中搜索字符串 `kubectl delete pod`，系统无法仅凭正则区分“执行危险命令”和“查询危险命令痕迹”，当前会按 forbidden 处理。这是安全优先的误报，不应宣传为完整语义授权。改进方向可以把参数分成 command、query text、resource selector 等受类型约束的字段，对危险词只在可执行槽位生效，再结合主体、资源、环境和变更窗口做授权。

若计划首步是 approval_required，Executor 还会检查后面是否存在低风险只读诊断；有的话先把审批动作移到这些取证步骤之后。这样能避免证据不足时过早申请变更。它不是批准动作，也不保证重排后的每步都最终成功；真正到达风险步骤时仍创建 ApprovalRequest 或 forbidden Evidence。

## 大结果进入状态前，实际发生了什么

工具输出治理的顺序决定了系统能否既可审计又不扩散秘密。

### 先脱敏，但不要把启发式当成数据防泄漏证明

持久化前会递归处理 dict、list、tuple 和字符串。敏感 key 通过 `password/token/secret/key/dsn/authorization/cookie/credential/bearer` 等子串识别；文本通过 Bearer token 和赋值模式正则替换。测试覆盖了嵌套 password、API token、Authorization、cookie，以及输出和 Trace 二次脱敏。

这种策略会误伤含 `key` 的普通字段，也可能漏掉非标准命名、自然语言 PII、编码后的秘密和二进制载荷。它只证明已覆盖模式在持久化/Trace 路径被替换，不证明第三方 SDK、网络、第三方 MCP server 日志或异常堆栈从未接触秘密。仓库自带 MCP server 已改为只写长度与哈希，但不能用这个本地实现替所有远端服务背书。

还要区分两个脱敏位置：`result_for_persistence` 现在会统一递归处理整个持久化回执的 `input_args`、`output`、`error_message` 和 `metadata`；TraceService 落库时仍会二次脱敏。这样标准 Evidence/state、ToolCallRecord 和 Artifact 不会因为错误字段或 retry metadata 漏掉第一道防线。它仍不是完整 DLP：未知凭据格式、自然语言 PII、二进制载荷和第三方系统自己的日志需要独立治理。

仓库 monitor/CLS MCP server 的调用装饰器也不再记录原始 kwargs 或异常文本，只写长度与 SHA-256 短指纹。指纹可用于关联重复输入和同类错误，却不能恢复原文，也不能证明第三方 MCP server 遵守相同日志策略。

### 再压缩外部 raw，而不是默认保存全量原包

当 `AIOPS_STORE_RAW_EXTERNAL_PAYLOAD` 为 false 时，Prometheus、Loki、log gateway、CMDB、deploy history、Redis、Kubernetes、MySQL、ticket API 等已知 source 的 `raw` 会先压缩：列表常只留前五项，嵌套字典只保留部分诊断字段并标 `raw_truncated=true`。这是数据最小化，不是无损归档。

### 最后按字节阈值外置

默认 `AIOPS_TOOL_OUTPUT_INLINE_BYTES=20,000`。对脱敏和压缩后的 output 做稳定 JSON 序列化；未超过阈值继续内联，超过则写到默认本地目录 `data/aiops_tool_artifacts`。Artifact ID 由工具名和完整 SHA-256 的前 16 位构成，metadata 另保存完整 SHA-256、size 和 `truncated=true`；内联只保留摘要、指针和少量 source/status/error_type。

相同工具和相同持久化输出会得到相同 ID，文件已存在时不会重复写。SHA-256 能帮助判断这段序列化内容是否变化，不能证明数据由真实 Redis 产生、没有在上游被篡改，也不能保证文件永久可读。

当前仓库没有 Artifact 下载/解引用 API、对象存储、租户隔离、统一加密、跨节点共享和 retention。Read model 会把 artifact pointer 带到 ToolCallRecord、Evidence、报告或 Replay，但不会替调用方验证文件仍存在。因此生产版本至少需要受权 Artifact Store、生命周期、内容类型、加密、访问审计和清理策略；本地绝对或相对路径不应直接暴露给跨租户客户端。

## 一次 Redis 取证怎样变成可回放链路

现在把正常路径拆开，并持续回答“知道什么、还缺什么、为什么继续”。

### 第一步：计划只表达待验证问题

PlanStep 请求 `query_redis_status(order-service, 10m)`，预期证据是连接数是否接近上限。此时只知道 503、P95 和 timeout 告警背景；还不知道工具存在、数据源配置、事故窗口和当前窗口是否一致。

### 第二步：Executor 和 Registry 两次检查风险

工具已注册、contract 只读低风险，输入没有命中危险规则，因此 allow。若工具名未注册，Executor 会进入低可信兼容 fallback；若输入出现危险动作，先生成 risk Evidence 或审批，而不是调用 Adapter。

### 第三步：Adapter 选择实例并读取来源

服务拓扑可能给出 Redis instance；否则使用显式参数或默认实例名。Adapter 连接真实配置，读取 INFO 和可选事故证据。此时能够确认的是“该配置端点返回了哪些字段”；仍缺少与指标、日志的独立时窗交叉验证。

### 第四步：基类在总预算内执行

成功就记录一次 attempt。连接错误若快速失败且命中 retry_on，可以在剩余 deadline 内退避重试；权限失败不重试；实际 wait_for 超时耗尽预算后停止。若底层走 MCP，还要考虑 interceptor 的内部 attempts。

### 第五步：输出先治理再持久化

输入和输出脱敏，外部 raw 根据配置压缩，超过 `20,000` 字节才写本地 Artifact。ToolCallRecord 和 Evidence 得到同一 output_artifact pointer。此时可回查的是持久化版本，不一定是上游完整原始包。

### 第六步：ToolCallRecord 证明过程

记录 trace/incident/step、实际参数、data_source、latency、status、risk、error、retry 和 batch metadata。标准 Registry 路径写 `invocation_kind=tool`、`actual_tool_invoked=true`；`manual_analysis` 与未注册工具的 LLM 兼容路径写 `invocation_kind=analysis_fallback`、`actual_tool_invoked=false`。因此“生成了一条 ToolCallRecord”不再等于“真实外部工具已调用”，Read Model 与 Evidence Graph 可以区分取证调用和分析兜底。它能回答“调用发生了吗、走了哪条路径”，还不能回答“根因是什么”。

### 第七步：Evidence 分离事实、推断和不确定性

成功时 fact 概括 Redis 返回及 source；inference 可能写“支持连接耗尽假设”；uncertainty 明确当前 runtime 与 replay window 边界。失败时 fact 是未返回可用数据，stance 为 unknown，next_step 指向配置、网络、权限或替代取证。

### 第八步：Trace 与状态供 Replanner 消费

并行批次仍按计划顺序进入 state，TraceService 再脱敏后写 SQLite。若上层取消 fan-out，Executor 会取消尚未完成的 sibling tasks，并用 `gather(return_exceptions=true)` 等待它们退出后继续抛出 `CancelledError`；取消不会被包装成 failed Evidence。Replanner 检查是否已有指标、日志、Redis 和知识的足够交叉支持；关键日志缺失时可以重试或补另一来源，而不是把失败从报告里删掉。

### 第九步：满足停止条件，或明确降级

若 Redis 事故窗口、P95/error-rate 和 timeout 日志时间对齐，且没有更强反证，可以提高该 Hypothesis 排名，但变更仍需审批。若 Redis 未配置、日志权限失败或时间窗口冲突，报告应降级为证据不足、列出缺口并转人工。停止条件不是“某个工具 success”，而是证据闭环达到阈值、预算耗尽、风险要求人工或继续查询的边际收益过低。

## 正常、失败和降级必须是三种可观察状态

工具工程最危险的错误，是把“没有数据”和“没有异常”做成同一个空数组。AutoOnCall 用 status、source、error_type、error_message 和 uncertainty 区分它们。

| 情况 | 正确语义 | 不应得出的结论 |
|---|---|---|
| Redis 返回当前正常 INFO | 当前端点此刻可达且未饱和 | 事故窗口从未饱和 |
| success 但 output 为空 | 工具没有返回可用观测，规范化为 failed/no_data | 目标系统正常或没有匹配异常 |
| success 但 payload 标记 stale/expired | 旧数据保留为审计缺口，stance=unknown、不可用于 RCA | 当前事故窗口仍满足旧观测 |
| Redis `not_configured` | 真实 Adapter 没有配置 | Redis 正常或没有 Redis |
| 日志权限失败 | 无法取得日志 Evidence | 日志中没有 timeout |
| MCP CPU 成功、内存失败 | 部分指标可用，整体取证不完整 | 自动补齐缺失 QPS/P95/error-rate |
| Runbook no-answer | 知识库没有可信支持 | 现场工具也没有异常 |
| 未注册工具 fallback | 兼容文本已保存，但标准取证失败 | ToolNode 文本等于标准工具成功 |
| Policy forbidden | 动作未执行，风险规则命中 | 现场已经完成处置 |

`adapter_failure` 会把异常归为 timeout、connection_error、permission_denied、not_found、server_error、http_error、not_configured 或 adapter_error，并返回公开安全消息。HTTP 200 也不会自动算成功：共享响应校验会拒绝 `status=failed/error`、`success=false`、`ok=false`，各 Adapter 还校验 `data.result`、`items`、`logs`、`tickets` 等业务字段类型或工单 ID。这样“传输成功、业务失败”的响应不会进入 success Evidence。

原始 exception 不进入用户 Evidence。Kubernetes/MySQL 的 `partial_errors` 会转换为分类和公共消息，公开 Run Status 还递归压缩 `processlist_sample`，只保留 Command、Time、State 和是否存在 statement，不返回用户、主机、库名或 SQL 文本。公开错误有利于避免泄密，但会损失排障细节；服务端受控日志和 Trace 需要保留 correlation ID 或错误指纹，而不能把敏感堆栈直接回传。

## Read Model 能展示审计链，但不会替你验证真相

**Snapshot（运行快照）**通俗上是某一诊断任务当前状态的存档；严格定义是持久化的 plan、已执行步骤、工具记录、证据、错误、审批和报告引用集合。它解决断线后查询和恢复展示，不负责证明每个事件按真实时间完整发生，也不能替代逐事件 Trace。例如快照中有一个 failed tool record 可以证明状态已保存；仅凭快照最终状态反推全部中间调用则是反例。在 AutoOnCall 中，它对应 AIOps session snapshot 和 run status 数据来源。

**Replay（故障回放）**通俗上是把已保存材料按阶段重新展示；严格定义是从 Trace、Snapshot、报告、审批和变更记录构造的只读时间线与汇总。它解决复盘时看见计划、工具、重规划和报告之间的关系，不负责重新执行工具、验证 Artifact 文件仍存在或重算根因。例如 Replay 显示两次 Redis 调用是可审计线索；把它当作实时 Redis 查询则是反例。在 AutoOnCall 中，它对应 incident replay 的 timeline、tooling、metrics 和 evaluation read models。

工具契约 API 返回十项 contract 和 count，受 read scope 保护，不触发连接。诊断 run detail 会返回 plan、executed_steps、past_steps、tool_call_records、gathered_evidence、warnings 和 errors；其中嵌套 `partial_errors` 和 `processlist_sample` 在公开投影层再次收敛。Incident overview 汇总 data source，并标记是否出现 mock 或 not_configured；Replay 将 Trace 规范成阶段时间线。

Replay tooling 统计 `by_tool`、`by_status`、`by_source`、failure count，并把调用次数大于 1 的工具列为 `duplicate_tool_candidates`。它只是候选提示：同一工具可能因不同时间窗口、Replanner retry 或故障恢复而合理重复，不能看到 duplicate 就自动判定浪费。

Replay 的 latency 汇总来自 Trace 中大于零的事件延迟，P95 使用当前实现的近似排序索引；它不是外部 Adapter 专属 P95，也不自动区分并发重叠时间，不能当成端到端生产 SLO。若 report 中没有完整 tool_calls，read model 会从 timeline 构造较简的工具项，字段可能少于原始 ToolCallRecord。

Read model 同样不会打开 Artifact 校验 SHA、重放远端查询、检查权限是否仍有效，也不会验证 Evidence inference。它负责把已存事实组织给前端和面试回放；事实真实性仍依赖 source、窗口、Artifact 和独立交叉验证。

## 指标为什么既要看 F1，也要公开 24 次多查

2026-07-12 的 official Agent RCA 基线包含 `48` 条结构化 `offline_fixture` case。工具工程相关结果不能只报一个“全通过”，因为工具选择既可能漏查，也可能多查。

工具选择的原始计数为 TP=`276`、FP=`24`、FN=`0`。因此 Precision=`276/(276+24)=0.92`，Recall=`276/(276+0)=1.00`，F1=`0.9583`。Recall 为满分表示固定标注所需工具没有漏掉；Precision 只有 `0.92` 表示仍出现 24 个额外工具选择。F1 不是“工具调用成功率”，也不是根因准确率。

| 工具工程指标 | 结果与分母 | 能证明什么 | 不能证明什么 |
|---|---:|---|---|
| 工具选择 Precision | `276/300 = 0.92` | 固定标注下多数选择是期望工具 | 多查一定毫无价值 |
| 工具选择 Recall | `276/276 = 1.00` | 没漏固定集标注的必要工具 | 生产未知故障也不会漏 |
| 工具选择 F1 | `0.9583` | 综合多查与漏查 | 外部数据正确或根因正确 |
| 必要证据召回率 | `48/48 = 1.00` | 48 case 的 required evidence tools 均覆盖 | 证据内容真实、窗口对齐 |
| 无效工具率 | `0/300 = 0`，300 个 planned tool 选择中没有无效名称 | 标准工具边界在该数据集可控 | 任意模型输入都不会造名 |
| 执行成功率 | `258/267 = 0.9663` | 记录的执行中 258 次成功 | 失败应被隐藏或生产可用率 |
| 工具失败降级 | `9/9 = 1.00` | 9 条失败 case 都保留失败 Evidence 并按预期降级 | 任意依赖故障都能恢复 |

这里的失败 case 数与 `267-258=9` 次未成功执行在本次汇总中数值恰好同为 `9`，但两者单位不同：一个按 case，一个按执行。若不逐条核对 case 行，不能仅凭减法断言它们一一对应，更不能在其他 run 中默认视为同一口径。`unnecessary_tool_rate` 等 summary 字段中的 `1.0` 可能表示对应验收规则的通过率，不能误读为“100% 的工具都多余”；工具选择的真实多查由 FP=`24` 和 Precision=`0.92` 表达。

这些结果验收的是 Registry、Executor、Evidence、Replanner 和报告在固定离线数据上的串联。它们不证明 Adapter 已连接生产 Prometheus、Loki、Redis、MySQL 或 Kubernetes，不证明生产吞吐、权限治理和超时率，也不证明 48 条以外的 Planner 不会选择危险或无效工具。

## 设计取舍：当前方案不是唯一答案

### 静态 Registry 与动态插件市场

静态十工具让 Planner 输出空间、回归集、风险规则和 UI 契约稳定，代价是新增系统需要开发包装层。动态 MCP marketplace 扩展快，却必须解决工具签名漂移、供应链信任、远端风险声明造假、权限和租户隔离。对于只读诊断，可以逐步引入审核后动态注册；对于写操作，更适合显式 change capability 和独立授权，而不是复用通用 ToolNode。

### 描述性 schema 与强校验

当前 Registry 已能对非空 `input_schema` 应用顶层默认值并执行 Draft 2020-12 校验，拒绝必填缺失、类型错误和非法 schema；对成功 output 也执行 schema、JSON 可序列化性和 `1 MiB` 上限校验。这解决了“契约只展示、不执行”的主要缺口。它仍不会自动拒绝 schema 未声明但允许的未知字段，也没有标准工具输出的字段级语义、深层默认值扩展、跨字段业务约束和 schema version。加强方案可以为每个工具定义 Pydantic input/output model，在 Registry 调用前后校验，保留原始请求和规范化请求的差异，并为 schema version 做兼容。代价是 Adapter 字段演进更慢、部分响应要明确版本化。

### 正则 Policy 与类型化能力授权

正则能低成本挡住常见危险文本，并通过双入口减少旁路；它会误伤日志搜索，也无法理解编码、拆词和间接提示注入。生产授权应以主体、环境、资源、动作、参数槽位、审批令牌和有效期为中心，正则只作为补充防线。对任何变更，还需要幂等键、pre-check、执行回执、verify 和 rollback，而不是把 read_only 改成 false 就算完成。

### 本地 Artifact 与对象存储

本地文件实现简单、测试可控，适合单机演示；跨进程、容器滚动和多租户会使 pointer 失效或越权。生产版本需要对象存储或内容寻址 Store、服务端下载授权、retention、legal hold、加密、恶意内容扫描和删除审计。若只保存摘要，审计能力下降；若保存全量，隐私和成本上升，应按 source 与字段敏感度制定策略。

### 失败 Evidence 与自动恢复

把失败也存成 Evidence，能让 Replanner 和报告诚实说明缺口；但失败过多会膨胀状态并诱发重试风暴。可按工具/实例设置 retry budget、circuit breaker、最大补查次数和全局 Incident deadline，把重复失败聚合，同时保留首尾样本和计数。任何自动降级都应区分“换来源”“缩短窗口”“使用缓存”和“转人工”，不能用旧缓存伪装 live data。

## 什么时候继续、停止、降级或转人工

Redis 主线可以用以下判断收束：

- 指标、日志和 Redis 事故窗口已经对齐，来源不是 mock，关键反证已检查：停止取证，生成带不确定性的报告；处置仍按风险走审批。
- Redis success 但只有当前 INFO，没有事故窗口：继续查历史指标、日志、工单或部署记录，不把当前值反向覆盖历史。
- Redis `not_configured`，其他来源可用：记录缺口，换独立只读来源；结论置信度下降。
- 日志 permission_denied：不要自动重试；转平台权限处理或人工补证。
- 多个工具 connection/server error：全局依赖可能异常，停止局部无限重试，启用 circuit breaker 或转人工。
- Policy forbidden：不执行，不把 blocked 写成 remediation success；允许追加不触碰危险资源的只读步骤。
- 只剩 manual/LLM fallback：最多形成低置信背景，不足以确认根因。
- Incident deadline、step budget 或 retry budget 用尽：输出 needs_human/degraded，列明缺失 source、已尝试次数和下一步。

这组规则的核心是：失败必须可观察，成功必须带 source，推断必须带 uncertainty，变更必须与取证分离。

## 面试中怎样把工具工程讲成一个完整决策

### 一句话定位

> AutoOnCall 的工具工程把模型生成的动作转换为有稳定契约、双重策略门禁、总预算、来源、结构化失败和审计指针的受控取证。

### 30 秒回答

> 我把 Planner 与外部系统之间拆成 PlanStep、静态 Tool Registry、AIOpsTool 和 Adapter。Registry 对重名、步骤错配、未知实例和非法输入 fail closed；十个标准工具统一 success/failed 回执，HTTP 2xx 业务失败、空输出、stale 标记和 MCP structured content/error 都先规范化；指标与日志主源失败后继续有序降级，Prometheus 真正使用 range/step，Redis 历史证据按窗口过滤，MySQL 区分服务端连接与应用池。只读瞬时错误在一个总 deadline 内重试，fan-out 取消会清理 sibling tasks 后继续传播；ToolCallRecord 还区分真实调用与 analysis fallback。失败也生成 Evidence，但不会进入根因支持链，因此 Replanner 能降级而不是伪造结果。

### 3 分钟展开思路

先用 Redis 反例开场：`status=success` 可能只查到当前容器，不能证明事故窗口。然后讲四层推导：PlanStep 稳定动作语义，Registry 管目录和入口策略，AIOpsTool 管总预算与失败 envelope，Adapter 管外部方言。接着解释三层重试，特别指出 MCP 最多三次、AIOps outer attempt 和 Replanner 新步骤不是同一件事；真实 wait_for 超时耗尽 deadline 后不会幽灵重试。

再讲并发和安全：只有显式 allowlist 中相邻、低风险、Registry 声明只读且 Policy allow 的步骤最多四路 fan-out，结果保序、失败隔离；Executor 与 Registry 双门禁防旁路，但正则会误伤危险文本检索。最后讲持久化：脱敏、raw 压缩、20,000 字节阈值、本地内容哈希 Artifact，主动承认没有对象存储和下载 API。

量化时给出完整口径：48 条 offline fixture，工具选择 TP/FP/FN=`276/24/0`，Precision/Recall/F1=`0.92/1.00/0.9583`；必要证据 `48/48`；9 条工具失败 case 降级 `9/9`；执行 `258/267=0.9663`。随后给 Run ID、clean commit 和 evidence level，并明确不等于生产准确率。

### 可写入简历的准确表述

> 设计并实现静态 Tool Registry 与统一执行契约，覆盖十个 AIOps 标准取证工具；在真实 Registry 入口执行重名、PlanStep/调用名错配、未知实例和非法输入 fail closed，统一识别 HTTP 业务失败与 MCP structured content/error；为指标/日志实现保留失败轨迹的有序降级，修正 Prometheus 时间窗口与非有限值、Redis 历史窗口/Stream 解析、MySQL 服务端连接与应用池语义，并对公开 partial error/PROCESSLIST 做最小化输出；在 48 条离线 RCA case 上工具选择 F1 为 0.9583，9/9 工具失败 case 按预期降级。

若篇幅有限，可以保留“48 条离线”“0.9583”“9/9”三个限定，不能删掉离线和分母。

### 不可夸大的表述

- 不要说“任意 MCP 工具即插即用”；当前标准 Registry 静态注册，fallback 只放行两个本地辅助工具。
- 不要说“所有 schema 和输出均强校验”；当前已校验非空输入和成功输出的通用 JSON Schema/序列化/大小边界，但标准输出大多仍只是 object，未形成字段级语义类型系统。
- 不要说“超时一定重试两次”；真正耗尽总 deadline 的 wait_for timeout 一次即停。
- 不要说“Artifact 保存所有完整原始日志”；默认可能先压缩 external raw，且 Store 只是本地文件。
- 不要说“SHA-256 证明来源真实”；它只校验所引用持久化内容的一致性。
- 不要说“48/48 证明生产 RCA 准确”；它是 official baseline 内的 `offline_fixture`。
- 不要说“7/7 controlled fault 都诊断成功”；该组只完成注入和 cleanup，diagnosis 为 `not_run=7`。

### 高频追问与回答要点

1. **为什么不让 Planner 直接调用 SDK？** Planner 输出不可信且外部 API 会变；稳定语义、授权、预算和审计需要独立边界。
2. **为什么 ToolCallRecord 不能直接当 Evidence？** Record 证明调用过程，Evidence 还要区分 fact、inference、uncertainty 和 source；两者都不自动证明因果。
3. **为什么有三层重试？** MCP 恢复远端 handler，AIOpsTool 恢复一个只读步骤的瞬时错误，Replanner 根据全局证据缺口新建步骤；上下文和审计身份不同。
4. **写工具为什么不自动重试？** 第一次可能已成功但响应丢失；没有幂等键和结果查询会重复副作用。
5. **为什么 fan-out 上限是四？** 它是当前有界并发策略，用来控制依赖压力和状态复杂度，不是性能最优常数；生产应按 source 限流和容量测量调整。
6. **为什么并发结果还要保序？** 降低等待不应改变 plan/state 的确定性，便于回放和回归测试；取消时还要显式清理 sibling tasks，避免调用在请求结束后继续运行。
7. **双门禁会不会重复？** Executor 负责工作流暂停、审批和 Evidence；Registry 保护所有执行入口。规则复用，审计职责不同。
8. **Policy 为什么会误伤日志查询？** 当前按文本正则扫描多个字段，不能理解字符串所在参数槽位；生产应改为类型化 capability 授权。
9. **not_configured 和空结果有什么区别？** 前者说明取证链未连接，不能推断系统正常；空结果只有在查询成功、窗口和过滤条件明确时才是观察。
10. **SHA-256 能证明什么？** 能发现持久化 JSON 内容变化，不能证明来源、时间窗口、完整性或永久可用。
11. **MCP fixture 为什么不是 local-live 生产观测？** 仓库 server 使用生成时序和 mock topic/log，只验证协议路径；必须有真实外部配置、source 与运行 provenance 才能升级证据等级。
12. **0.9583 是怎样来的？** TP=276、FP=24、FN=0，所以 Precision=0.92、Recall=1.00，再计算调和平均；它揭示没有漏查但仍有多查。
13. **为什么重复工具名必须直接失败？** 静默覆盖会让实际执行对象取决于发现顺序，风险契约、schema 和测试替身都可能被换掉；fail closed 才能让名称成为稳定能力身份。
14. **为什么 MCP `isError` 还要再规范化？** LangChain 调用本身完成不代表远端业务成功；必须读取协议错误位和 structured content，才能阻止失败 payload 进入 success Evidence。
15. **取消为什么不能转换成 failed？** 取消是上层调度意图，不是依赖故障；吞掉 `CancelledError` 会触发重试、错误 Evidence 和错误终态。
16. **为什么主 Adapter 失败后还要继续降级？** “已配置”不等于“已返回可用数据”；只要后备源语义兼容，就应继续尝试，并在成功结果中保留前序 `fallback_errors`，避免可用性和故障可见性二选一。
17. **Prometheus 为什么使用 query_range 后仍不能说做了完整窗口分析？** 请求窗口和 step 已真正生效，非有限值也会缺失化；但当前每项只取最后一个有限样本，没有计算趋势、峰值或异常持续时长。
18. **MySQL 为什么不能只输出 active/max？** MySQL `Threads_connected/max_connections` 与应用池 `pool_active/pool_max` 来自不同系统、含义和采样时刻；混写会让事故回放值冒充数据库服务端实时连接。
19. **为什么 ToolCallRecord 还要记录 `actual_tool_invoked`？** manual/LLM fallback 也会形成审计记录，但没有真实调用目标工具；不区分会让 Trace 和 Evidence Graph 把分析文本画成外部取证。

最终要记住的不是十个类名，而是一条判断链：工具名稳定不等于数据源稳定，调用成功不等于窗口正确，ToolCall 不等于 Evidence，Evidence 不等于根因，Artifact 哈希不等于来源真实。只有把这些“不等于”写进 contract、执行、持久化、read model、评测和面试表述，Agent 的工具调用才从演示函数变成可治理的取证基础设施。

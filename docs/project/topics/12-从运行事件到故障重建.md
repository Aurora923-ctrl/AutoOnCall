## 浏览器最后显示“等待审批”，然后服务重启了

继续使用贯穿全系列的 Redis 故障。`order-service` 开始返回 503，接口 P95 明显上升，日志出现 `Redis connection timeout`，Redis 指标里的 `connected_clients` 已接近 `maxclients`。诊断 Agent 先制定计划，再查询指标、日志和 Runbook；它形成“连接数逼近上限”的根因假设，但重启服务属于高风险动作，所以系统停在人工审批前。

这时浏览器断开，FastAPI 进程也重启。值班同学重新打开页面，真正想知道的不是一句“任务失败”，而是下面这些问题：

1. 上一次诊断属于哪起故障，走到了哪一步？
2. 哪些工具确实调用过，输入、结果和失败原因是什么？
3. 当时已经收集了哪些证据，根因仍只是推断还是已有充分支持？
4. 人工审批通过后，系统能从什么材料恢复，又会完成什么？
5. 复盘页展示的是事故现场留下的事实，还是服务重启后重新查询出的“现在”？

如果系统只把文字打印到控制台，以上问题几乎都无法可靠回答。即使日志文件仍然存在，也可能缺少稳定的故障标识、结构化状态和审批记录；更糟的是，原始工具参数可能包含凭据，完整日志结果可能有数万行。于是，“多打印一些日志”并不能自然升级为“可恢复、可审计的 Agent”。

本章要解决的核心不是日志格式，而是一个更一般的工程问题：**实时连接会断、进程会死、材料会不完整时，怎样保存足够的运行事实，并在不编造、不重做副作用的前提下重建故障过程。**

先划清证据：重启后能否找回内容，取决于 Trace、审批、Session Snapshot、Report 和 IncidentState 是否已经成功持久化；Runbook 是知识，不是本次事故事实；“有快照”也不等于“能从任意 Agent 节点精确续跑”。AutoOnCall 当前实现的是有限恢复和只读重建，不是数据库级工作流 time travel。

## 三个朴素方案为什么会在故障时失效

### 只记终端日志：人能看，系统却无法可靠查询

最初的实现往往只有三行输出：

```text
planner started
query_redis_status success
waiting approval
```

它对本地调试有用，却没有回答“这是哪个 Incident”“工具成功对应哪份输出”“waiting approval 对应哪个审批单”。文本顺序还会受异步并发影响，跨进程、多副本后更难拼接。如果直接打印完整参数和结果，又会把密码、Token、连接串和大段日志写入长期保存介质。

所以日志解决的是开发者观察问题，不天然承担业务恢复契约。生产日志当然仍然重要，但不能是故障状态的唯一数据库。

### 所有信息都存成事件：历史清楚，当前状态却很昂贵

第二个直觉是“每一步都追加事件，需要当前状态时从头回放”。这接近事件溯源，但要成立至少需要：事件不可变、全序或可解释的局部顺序、事件版本兼容、幂等投影、缺失检测和副作用隔离。Agent 状态包含计划、剩余步骤、证据、假设、审批、报告等复杂结构；如果某次部署改变了事件 schema，从第一条事件重算可能得到与当时不同的状态。

更关键的是，当前 AutoOnCall 的事件并非被设计成完整状态转移日志。Trace 会脱敏、截断和摘要，它有意不是无损原始载荷。拿审计事件冒充完整 checkpoint，恢复时一定会遇到信息不足。

### 只保存最新快照：能看“现在”，却解释不了“为什么”

第三个方案是每次覆盖一个最新 JSON。它能快速回答“当前是否等待审批”，却无法证明工具何时调用、失败后是否重试、Replanner 为什么追加步骤、审批前后发生了什么。如果快照先写成“完成”，而报告或 Trace 写入失败，还会出现多份材料不一致。

因此，一个可靠的运行态通常不是在日志、事件和快照之间三选一，而是让它们分别承担不同的失败模式：实时传输面向当前连接，审计事件解释过去，最新快照回答当前状态，审批/报告等业务记录保存有明确语义的事实，读模型最后负责组合展示。

## 先建立词汇边界，避免把所有材料都叫“状态”

### Incident：一次故障的稳定业务边界

通俗地说，**Incident** 就是“一起需要处理的故障”。严格定义是：它用稳定的 `incident_id` 把告警、诊断运行、Trace、Evidence、审批、变更和报告归入同一业务上下文。它解决的是“这些材料属于哪起事故”，不负责证明根因正确，也不表示一次 Incident 只能有一次诊断 session。

例如，`order-service` 503、Redis timeout 日志和本次重启审批应归入同一 Incident；把另一时段的 MySQL 慢查询 Trace 仅因“也导致延迟”而混入，就是反例。在 AutoOnCall 中，Incident 的耐久生命周期由 `IncidentState` 表达，其他记录都通过 `incident_id` 与它关联。这里没有一个包办所有字段的巨大 Incident 对象，而是多个业务记录共同描述它。

### 运行事件与 Trace：发生过什么，不是完整现场复制

**运行事件**通俗理解是 Agent 在执行过程中发出的消息，如“开始规划”“工具完成”“等待审批”。严格地说，它是带类型、时间、关联标识和结构化载荷的状态通知。它可以被实时发送，也可以被选择性持久化；它不天然等于审计记录。

**Trace** 通俗地说是经过治理的诊断流水。严格定义是：按 Incident 和 trace 关联的追加式过程记录，描述节点、工具、风险判断、审批、报告与变更等已经发生的事件，并保存状态、耗时、错误、摘要及 Artifact 指针。Trace 解决“过去发生过什么、怎样解释执行链”，不负责保存 Python 调用栈、完整模型上下文或全部原始工具输出。

例如，“`query_redis_status` 调用成功，耗时 380ms，结果摘要显示连接数逼近上限”适合写入 Trace；把一万行 Redis 日志逐行塞进 Trace 是反例。AutoOnCall 由 `TraceService` 创建并持久化 `TraceEvent`，写入前执行脱敏和压缩。

### Snapshot：最近一次业务存档，不是历史录像

**Snapshot** 通俗地说是“这个诊断最近保存成什么样”。严格定义是：以 `session_id` 为键保存的最新 `AIOpsSessionSnapshot`，包含 Incident/Trace 身份、状态、当前节点、计划、已执行步骤、工具调用、Evidence、Hypothesis、风险判断、待审批信息、报告引用、最终 response、恢复审批 ID、恢复状态、恢复次数、错误和最近进度。

它解决快速查询当前业务状态与重启后的有限 fallback，不保留每次更新的历史版本，也不负责证明每个中间状态都落盘成功。例如等待审批前保存 current plan、风险结论和审批单引用是正确用法；声称能凭它回到某次 HTTP 请求已经读取了 37% 字节的位置，是反例。AutoOnCall 的 Session Snapshot 是应用层最新状态，不是 LangGraph 数据库 checkpointer。

### Checkpoint：工作流引擎可继续计算的执行状态

**Checkpoint** 通俗地说是“工作流引擎的断点”。严格定义是：由图执行框架按线程或 session 保存、足以恢复图状态和下一节点调度的执行数据。它解决工作流继续计算问题，不自动解决外部工具的副作用幂等、业务审计或报告查询。

AutoOnCall 当前 LangGraph 使用 `MemorySaver` 作为运行期缓存，但恢复和读模型只认 SQLite/MySQL 中的 Session Snapshot。服务重启后内存消失，不影响持久状态的权威性；同时这仍不会让 LangGraph 自动获得数据库 checkpoint。因此本章必须始终区分“运行期图缓存”和“持久化应用 Snapshot”。

### Evidence 与 Hypothesis：事实不能偷换成结论

**Evidence** 通俗地说是可追溯的观测证据。严格定义是：带来源、时间窗口、查询或工具结果摘要及质量边界的事实，用于支持或反驳根因假设。它解决“结论依据是什么”，不等于 Runbook 知识，也不自动证明因果关系。

`connected_clients` 接近 `maxclients`、事故窗口出现 `Redis connection timeout` 都是 Evidence；“连接上限是唯一根因”仍是 **Hypothesis（根因假设）**。Hypothesis 是对多个事实的可检验解释，不是事实本身。AutoOnCall 会在 Snapshot、Report 和 Replay 中保存或归一化这些材料，但读模型不能把低质量 Evidence 自动升级为确定根因。

### Artifact：大结果的外置制品，不是永远可用的承诺

**Artifact** 通俗地说是“没有直接塞进事件的大文件或机器结果”。严格定义是：由某次运行生成、可通过路径或标识定位，并应附带完整性和来源信息的持久制品。它解决大载荷与结构化 Trace 的分离，不负责保证多副本可见、永久保留或合规不可篡改。

例如工具输出的大段日志可以落到本地文件，Trace 只保存摘要、哈希或路径；如果服务迁到另一台机器后路径不可读，就不能假装原始结果仍可访问。AutoOnCall 当前存在本地 Artifact 用法，但尚未实现统一的分布式 Artifact Store、全局 retention 和 WORM 治理。

### Replay 与 Read Model：重建视图，不重新制造事实

**Replay** 通俗地说是事故复盘。严格定义是：从已经持久化的 Trace、Approval、ChangeExecution、Report、IncidentState 和可选评测材料，派生阶段、时间线、工具统计、Evidence 质量及审批/变更流程。它解决“人怎样理解分散材料”，不重新执行 Planner、Executor 或外部工具，也不会回到事故时间点。

**Read Model（读模型）**是为特定查询准备的派生视图。它把多个写模型组合成 Incident Overview、Run Status 或 Replay；它解决页面查询复杂度，不是新的事实源。例如读模型可以把审批单和报告组合成“等待审批”，但不能凭缺失材料创造一次工具成功。在 AutoOnCall 中，这些组合逻辑位于 `aiops_read_models/`。

## 从失败模式推导四层运行架构

现在可以把需求推导为四层，而不是先背项目目录：

```text
Agent 节点 / 工具 / 风险 / 审批 / 报告
        │
        ├── 实时传输：SSE progress，服务当前连接
        ├── 过程审计：TraceEvent，解释已经发生的动作
        ├── 耐久事实：latest Snapshot + Approval/Change/Report/IncidentState
        └── 查询投影：Overview / Run Status / Replay
```

四层看似重复，实际回答不同问题。SSE 追求及时，允许连接丢失；Trace 追求可审计，因此要脱敏和可查询；Snapshot 追求快速读当前状态，因此覆盖旧版本；审批、变更、报告则是有业务语义的耐久记录；读模型只在查询时组合事实。任何一层缺失都应降级展示，而不是用另一层编造等价信息。

贯穿标识也各司其职：`incident_id` 连接同一故障的多次活动；`session_id` 标识一次诊断运行并定位 latest Snapshot；公开 Run Status 中的 `diagnosis_run_id` 当前就是 `session_id` 的兼容别名，不是第三个运行身份；`trace_id` 连接这次运行的过程事件；`report_id`、`approval_id` 和 `change_plan_id` 定位独立业务单据。只用一个 ID 虽看似简单，却会让“同一 Incident 多次重诊断”和“同一 session 多个业务记录”变得含糊。

## SSE 只负责把“现在”送到浏览器

### EOF 为什么不能等于成功

Server-Sent Events 适合服务端单向持续推送。AutoOnCall 将状态、progress 和业务事件编码成 SSE `message`；若 payload 有 `progress_cursor` 或 `cursor`，helper 会把它作为 SSE `id`。通用 helper 认定的终态类型是 `complete`、`done` 和 `error`：AIOps 内部使用 complete/error，聊天 API 将 service complete 映射为客户端 done。`approval_required` 不是流结束成功，它表示工作流等待外部决定。

协议 helper 还要处理两个很小但真实的边界。第一，cursor 的合法值可能是整数 `0`；若使用 `if cursor` 判断，首个事件 ID 会被当成假值丢失，客户端无法稳定去重。当前实现以 `is not None` 判断，并优先使用 `progress_cursor`。第二，终态类型属于协议枚举，输入可能出现大小写或首尾空格；helper 会先规范化再判断，使 `" ERROR "` 仍能关闭流。规范化不能无限扩张协议，未知类型仍不是终态。

网络层的 EOF 只表示“这条连接读完了”，可能来自正常关闭、代理超时、浏览器网络切换或进程崩溃。若前端在没有收到业务终态时遇到 EOF，它不会直接显示成功，而是调用 Run Status 查询持久化最新状态。服务端路由也不再假设“async for 正常结束就一定发过终态”：若诊断或恢复生成器提前耗尽，会主动补发 `stream_ended_without_terminal` 或 `resume_ended_without_terminal` error。这个规则非常重要：传输事实不能替代业务事实。

聊天前端只有收到 done、error 或兼容旧 `[DONE]` 才结束；AIOps 前端按 complete/error 与业务状态消费。普通诊断、审批恢复和 Safe Change 生成器如果在终态前耗尽，路由分别补发稳定的 error 事件；A2A 流遇到未知服务异常时也发送 `internal_error` 与 `final=true`，而不是把截断连接伪装成成功。parser 还要处理 CRLF、SSE `id/event` 行以及没有尾随换行的最后一帧。裸 EOF 无论在哪条链路都不等于成功；聊天显示重试，AIOps 再查询 Run Status。这里的重点不是强行统一所有业务事件名，而是让生产者、helper 与消费者对各自映射后的有限终态集合保持一致。

### cursor 有什么用，又缺什么

progress cursor 采用类似 `session_id:000003` 的单调序号，帮助页面识别当前进度；Snapshot 会保存最新 cursor 与最近一段 progress events。但当前项目没有根据浏览器 `Last-Event-ID` 从持久事件队列补发缺失消息的实现。断线恢复是“查询最新读模型”，不是“从上一个字节或事件继续推送”。

这意味着用户可以找回当前状态，却不保证重新看到断线期间每一条动画消息。要支持严格事件补发，需要持久消息队列、每 session 的事件序列、消费确认、过期策略和重复消费幂等，这与当前 latest Snapshot fallback 是不同级别的能力。

## Trace 必须在“可审计”和“不过度保存”之间取舍

### 写入前先治理，不能把隐私问题留给页面

AutoOnCall 的 Trace 创建流程会对 input/output summary 做敏感信息脱敏并截断，摘要上限为 `1000` 字符；对 tool args、tool result 和 metadata 做结构化脱敏。列表型结构最多保留前 `20` 项，防止一次工具结果无限膨胀。工具调用还可保存状态、耗时、错误和 Artifact 元数据。

这里需要准确理解“限量”：当前实现对列表有 `20` 项限制，对摘要有 `1000` 字符限制，但普通字典在脱敏后没有统一的递归总字节预算。因此不能笼统宣称“所有 Trace payload 都已严格限长”。新工具仍应在结果生成处限制字段和规模，大输出外置为 Artifact，不能指望 Trace 层自动消除所有体积风险。

脱敏也不等于匿名化。当前键名归一化同时识别 snake_case、kebab-case、点号分隔和 camelCase/PascalCase，因此 `api_key`、`apiKey`、`APIKey`、`accessToken`、`clientSecret` 等常见字段会进入同一敏感词判断；规则仍只能覆盖已知键名和文本模式，未知凭据格式、业务隐私和模型复述都可能漏出。生产系统还需要字段白名单、密钥扫描、访问控制、静态加密和审计抽样。

公开读模型还需要比持久化层更窄的字段白名单。当前 Run Status 会递归把 Adapter `partial_errors` 变成分类后的公共错误；MySQL `processlist_sample` 只保留 Command、Time、State 与 `has_statement`，不会返回 User、Host、db 或 SQL `Info`。这道边界解决“内部取证可以暂存，公共运行详情不应原样下发”，但不代表 Snapshot 和本地 Artifact 已完成租户级访问控制。

日志安全还包括完整性，而不只是保密性。`session_id` 等用户可控字段若包含 `\r\n`，直接插入文本日志会伪造新行、级别或 actor。请求模型先拒绝 ASCII 控制字符，日志 helper 再把换行和不可打印字符替换为单行占位，同时保留中文等可打印 Unicode。这是防御纵深：模型验证保护业务标识，日志净化保护其他未经过同一模型的调用路径。它仍不能替代结构化日志字段、集中式访问控制和敏感值扫描。

### Trace 是追加历史，Snapshot 是覆盖状态

Redis 案例里，Planner 开始、指标工具成功、日志工具成功、风险判断、审批请求都适合各自成为 Trace Event。之后 Snapshot 可以从 `running` 覆盖成 `waiting_approval`。前者保留过程，后者优化当前查询。Trace 的 `event_id` 采用不可变幂等语义：同 ID 重试只视为已存在，不允许后到 payload 改写原 trace、incident 或摘要；这避免网络重试把历史变成覆盖写，但也要求生产者为不同事实生成不同事件 ID。

如果只看 Snapshot，我们知道“等待审批”，却不知道两个工具的先后与错误；如果只看 Trace，就要重新解释所有事件才能得到当前计划和待审批对象。将两者分开不是形式主义，而是读写成本、审计完整性和恢复能力之间的明确取舍。

工具执行记录还区分“真实调用”和“分析兜底”。标准工具写 `invocation_kind=tool/actual_tool_invoked=true`；manual 或 LLM fallback 写 `analysis_fallback/false`。Evidence Graph 的 ToolCall 节点会保留这两个字段，因此 Replay 可以展示“系统生成过兼容分析文本”，却不能把它统计成真实 Redis、Prometheus 或 MCP 调用。这个身份只覆盖修复后的新记录；旧数据缺字段时按默认真实调用读取，迁移语义待验证。

## Snapshot 能找回业务上下文，但不能替代跨对象事务

新诊断开始时，服务先在进程内 claim `session_id`，再以持久层原子 create 建立第一条 running Snapshot；同一 ID 正在运行、已完成或被并发请求抢先创建时都会冲突，不能用 upsert 覆盖旧运行。之后每个图节点产出后，服务会尽力合并节点结果、保存 progress，并更新 Session Snapshot；报告附近也会再次保存。保存逻辑会沿用已有 `incident_id`、`trace_id` 等身份，避免一次不完整更新把关联键清空；progress events 会按 cursor 合并，最终只保留最近 `20` 条。

“尽力”只适用于中间节点。中间 Snapshot 写入异常会记录 warning，诊断可以继续，因此浏览器可能看到刚刚发出的 SSE，而数据库仍停在更早 Snapshot；Run Status 仍是“最近成功持久化状态”，不是实时流的逐消息镜像。但最终 complete/error 前的状态收尾更严格：complete 必须先成功写入终态 Snapshot，失败则不发送成功终态；流被取消或生成器关闭且尚未持久化终态时，服务会把最新可得状态写成 failed。Trace 启动、节点或终态事件写入失败只减少审计字段，不应把 durable complete/error 吞掉。

Snapshot 目前也没有与 Trace 相同的统一全局脱敏器和严格总大小预算。因为它会保存计划、工具调用、Evidence、Hypothesis 和报告信息，源头对象若塞入大载荷或敏感字段，Snapshot 会继承风险。合理做法是在工具执行记录和状态建模阶段就只保留恢复所需字段，同时为 Snapshot 增加 schema 版本、字段白名单、大小门禁和写入失败监控。

还有一个常见误解：Snapshot 表按 `session_id` upsert，一次 session 只保留最新行，不是版本化快照链。要做“回到第 8 步时的状态”，应引入版本化 checkpoint 或完整事件投影，而不能从当前表结构推断已经支持。

Store 层对最容易发生竞争的身份采用了比普通 upsert 更窄的写契约。首次 Session Snapshot 使用原子 create，只负责占用 `session_id`，已有记录不会被新请求覆盖；审批恢复使用 `WHERE status IN (...)` 语义的条件更新，只有允许恢复的旧状态才能迁移到 `resume_running`。SQLite 用 `BEGIN IMMEDIATE` 串行化同库写竞争，MySQL 用事务和 `SELECT ... FOR UPDATE` 锁住目标行。二者都保留原始 `created_at`，并通过 Pydantic JSON mode 序列化时间、空值、嵌套对象和枚举值。

这些条件写解决的是单记录覆盖写和丢失更新的一部分，不是所有对象的全局事务。Report、Approval、ChangeExecution、IncidentState 和 Trace 仍各自持久化；审批或变更主记录成功后，IncidentState、Report、Trace 投影失败只会告警，不会回滚主记录，也不会把已提交事实伪装成 API 失败。七张表仍没有用数据库外键把 Incident、Session、Trace、Report、Approval 和 ChangeExecution 强制绑在一起，关联完整性主要依赖业务 ID、创建/恢复校验、retention 的关联保护和读模型过滤。面试中应区分“关键身份和状态迁移使用原子/CAS 风格写入”与“整个 Incident 聚合具备 ACID 事务或外键级完整性”。

## 七类耐久记录如何拼出一条故障链

`AIOpsStateStore` 为七类业务对象定义统一契约：

| 记录 | 回答的问题 | Redis 例子 |
|---|---|---|
| Alert | 故障怎样进入系统 | `order-service` 503 告警 |
| Trace | 过程发生了什么 | 查询 Redis 指标、请求审批 |
| Approval | 谁对什么风险动作做了决定 | 重启服务审批通过或拒绝 |
| ChangeExecution | 受限变更推进到了哪里 | dry-run、人工回执或失败 |
| SessionSnapshot | 这次诊断最近状态是什么 | `waiting_approval` 与 current plan |
| IncidentState | 故障生命周期当前状态是什么 | diagnosis、approval、change 阶段 |
| Report | 对根因与建议的结构化解释是什么 | Redis maxclients 报告 |

SQLite 和 MySQL backend 都实现这套读写接口。显式传入本地 `storage_path` 时工厂选择 SQLite；默认部署配置可切换为 MySQL。统一协议降低读模型对后端的依赖，但“代码实现 MySQL”与“真实 MySQL 已验证”是两件事。Store 查询只在“确实没有该主键”时返回 `None` 或空列表；连接、SQL、JSON 解码和 Pydantic 校验错误继续抛出，不能把存储故障解释成“没有审批、没有报告、没有恢复材料”。

当前自动化测试主要以 SQLite、替身服务和固定数据验证协议行为；MySQL 替身测试覆盖事务 begin/commit/rollback、非覆盖导入和维护操作部分失败回滚。2026-07-17 对本地 MySQL 容器只做了只读核验：容器 healthy，七张运行表均存在且为 InnoDB，运行账户对目标 schema 具备写权限。它证明初始化 Schema 与当前查询基本对齐，不证明真实 MySQL 写入后的进程重启恢复、断线重连、多实例并发和 HA。

## 审批之后所谓“恢复”，严格说完成了什么

### 实际恢复顺序比一句“先快照后报告”更细

人工批准后，恢复逻辑先核对审批确实属于当前 Incident，且状态为 approved，再从请求或审批 metadata 解析创建该审批的原 `session_id`。若显式 ID 与审批不一致，或该 session 的 pending approval 不是当前 `approval_id`，直接拒绝。随后按证据强度寻找状态：

```text
1. 同 session 的持久化 Session Snapshot
2. 数据库中同 session 的 AIOpsSessionSnapshot
3. 与该审批 trace/approval 归属一致，且仍处于 waiting_approval/approval_approved 的 DiagnosisReport
4. 全部缺失：抛出恢复失败，转人工
```

第 2 步如果找到 Snapshot 但其 Incident、pending approval 不匹配，会拒绝恢复。系统不会再把“同 Incident 最新 Snapshot”当作隐式候选，因为同一 Incident 可能有多轮诊断；也不会在身份缺失时临时生成 `resume-{incident_id}`。找不到原运行身份时宁可失败，也不把批准应用到另一轮调查。

恢复执行前还要 claim：进程内按 `approval_id` 防止同一批准被并发消费，持久层把 Snapshot 从 `waiting_approval/approval_approved/failed` 条件迁移为 `resume_running`。若有 checkpoint 或 Snapshot，系统基于保存下来的业务状态生成审批后的报告；若只有 Report，则构造最小 post-approval 报告。之后记录 `diagnosis_resumed`、`report_resumed`、`resume_completed` 等 Trace，清除 Snapshot 中的 pending approval，并写入新的报告状态。完成后的重复恢复因状态已不在允许集合而被拒绝；若中途失败，`resume_running` 转为 failed，保存 `resume_approval_id`、`resume_status` 和递增 `resume_attempt`，后续重试使用不同 cursor。

### 它没有做三件容易被夸大的事

第一，它没有从崩溃前的 LangGraph 下一节点重新执行完整图。持久 Snapshot 被转换为报告所需状态，恢复目标是闭合审批后的报告生命周期。

第二，它不会重新调用事故窗口的 Redis、Prometheus 或日志工具。因此它没有“重新抓到当时现场”，而是使用崩溃前已保存的材料。

第三，审批通过不等于 Agent 自动执行生产重启。恢复响应明确保留 `agent_does_not_execute_production_change` 的边界；真正变更仍遵守上一章的受限执行与人工回执约束。

所以面试中更准确的说法是“checkpoint→snapshot→report 的审批后有限恢复”，而不是“任意节点断点续跑”。

## Replay 怎样重建故事，又可能在哪里误导

### Replay 是多源只读投影

Replay Builder 会对 Trace、审批和变更按时间排序，再结合 Report 与 IncidentState，形成阶段卡片、时间线、Replanner 决策、工具统计、Evidence 质量、审批流、变更流和报告摘要。Trace 在时间戳相同时再按不可变 `event_id` 排序，审批按 `approval_id` 打破平局，使相同持久化材料重复构建时得到稳定顺序；这只是确定性展示，不表示系统获得了跨进程全序。阶段归类依赖事件类型和节点名映射；某些 Replanner 决策优先读取结构化 metadata，缺失时还会从 summary 中解析 `decision=`。

这说明 Replay 是解释层，不是原始事实的逐字复制。它能帮助人快速复盘，但阶段分类和 fallback 解析都可能受旧数据或命名变化影响。严谨页面应允许下钻到原始、已脱敏的 Trace Event，并标明推导字段。

Redis 案例中，Replay 可以展示：告警进入；Planner 计划；指标和日志工具返回；Evidence 支持 maxclients 假设；Replanner 请求审批；审批已通过；报告已恢复。它不应该在复盘时再次查询当前 `connected_clients`，否则“事故时接近上限”和“现在已恢复”会被混成一条伪历史。

### 评测关联是便利功能，不是天然可靠的外键

Replay 可关联离线评测 case，但先经过 Artifact 可用性门禁：`available=false` 或 `stale=true` 时不允许进入 linked evaluation，即使 case ID 完全相同也只显示启发式质量检查。可用 summary 先尝试精确 candidate ID；找不到时再做文本 token overlap，分数至少为 `3` 才匹配。若仍无法关联，则把根因命中、工具命中保留为未知，不能伪造通过。

关联结果现在同时返回 `match.method` 与 `match.score`，并透传 Artifact 名、Run ID、Git commit、dirty 状态、evaluation fingerprint、dataset 信息、stale reason 及生成/当前 fingerprint。精确 ID 匹配标记为 `identifier`；文本兜底标记为 `token_overlap`。这些字段解决“为什么关联、关联的是哪次运行、是否仍代表当前代码”的审计问题，但不会把启发式文本匹配升级为外键，也不会证明该 case 的期望根因适用于生产事故。

质量分数也要按指标方向解释。`unnecessary_tool_rate` 是越低越好的非负比例，旧式布尔转换会把任意非零数当成 `unknown`；当前实现把 `0` 判为 passed，正数判为 warning，缺失才是 unknown。Replay 的启发式“证据充分”除了要求至少两条可用 Evidence，还要求至少一条非 mock、非 unknown、非 fallback 的可信来源；两条高置信 mock 不再因为数量和 confidence 达标而显示充分。这里的规则仍是可解释门禁，不是统计校准后的根因概率。

## 把 Redis 故障按时间走一遍

### 第一阶段：事实仍在增长，不能过早停止

初始只知道 `order-service` 503 和 P95 上升。它们是告警事实，不足以定位 Redis。日志出现 `Redis connection timeout` 后，Redis 成为强候选，但仍可能是网络、连接池泄漏或下游放大。下一步查询 `connected_clients/maxclients`，是为了验证资源上限假设，而不是为了给既定结论找证据。

此时材料应被分层保存：SSE 告诉浏览器正在查 Redis；Trace 记录实际工具调用；Evidence 保存事故窗口的指标与日志摘要；Runbook 只提供“若接近 maxclients，应检查连接泄漏和连接池配置”的知识；Hypothesis 负责把它们组织成可检验解释。

如果指标工具失败，只能保留日志支持并降低置信度；如果时间窗口错位，则应补查正确窗口；如果 Redis 指标完全正常，则不能因为日志里出现 Redis 就停止。停止条件来自证据充分、风险门禁或预算耗尽，不来自“页面已有一句看似合理的话”。

### 第二阶段：高风险动作让系统停在审批

当 Evidence 表明 `connected_clients` 接近 `maxclients`，诊断可以把连接上限列为 Top-1 假设，但“重启 order-service”仍是动作，不是根因。风险策略生成 ApprovalRequest，IncidentState 和 Snapshot 转为 `waiting_approval`，Trace 记录为何请求审批，SSE 通知当前浏览器。

如果这时进程崩溃，内存 checkpoint 丢失。已经成功写入数据库的 Trace、Approval、Snapshot、IncidentState 与 Report 仍在；尚未来得及持久化的最后几条 SSE 不保证存在；本地 Artifact 能否读取还取决于文件系统是否保留。这就是为什么恢复页必须允许“不完整但诚实”，而不是承诺每次都恢复到最后一个像素。

### 第三阶段：重启后先查询，再决定能恢复到哪一级

浏览器发现 EOF 且没有终态，于是查询 Run Status。读模型以 Snapshot 为主，只合并 metadata 中 `session_id` 与本次运行一致的 Approval，以及 `report_id` 或 `trace_id` 与本次运行一致的 Report，避免同 Incident 的另一轮审批或报告污染当前 run。有效状态会回写到 progress 的 status/phase；因此旧 progress 仍写 running、但 Snapshot 已 failed 时，查询结果保持 failed。若旧 Snapshot 没有 progress，读模型会根据现有字段派生兼容 progress，并给出类似 `session:snapshot` 的 cursor；这种派生值是展示兼容，不是事故时真实发送过的进度事件。

审批通过后，如果同 session Snapshot 存在，就从它生成审批恢复报告；若只剩与审批身份一致的 Report，则标记 `report_fallback`；如果两类持久材料和内存 checkpoint 都没有，恢复失败并转人工。任何 fallback 都不应宣称找回了不存在的中间分析。

### 第四阶段：复盘只使用事故时留下的材料

Replay 聚合 Trace、审批、变更和报告。当前已知事实是：事故窗口 503、P95 上升、Redis timeout、连接数逼近上限、审批决定以及后续报告状态；知识是 Runbook；推断是“maxclients 压力导致连接超时并传导到 503”。若缺少变更后验证，就不能进一步推断“重启已经修复”。

MySQL 慢查询和 Kubernetes Pod 异常只用于说明同一套标识、Store 和读模型能够承载不同故障类型。它们不改变主线，也不能把本地 MySQL adapter 或本地 K8s 事件外推成生产集群恢复证据。

## 保留策略和迁移脚本为什么也是恢复设计的一部分

运行数据不可能无限保存。Store 的 `cleanup_older_than` 要求 `keep_days >= 1`，支持 dry-run 和实际删除。清理脚本默认 dry-run，只有显式 `--execute` 才删除；MySQL 将整轮 count/delete 放在一个事务中，任一表失败就回滚，SQLite 使用 `BEGIN IMMEDIATE`。重置脚本则要求显式 `--confirm-reset`，避免维护命令因参数遗漏直接清空运行态。

Retention 也不再只按时间做无条件删除。旧 firing Alert、pending Approval、running/planning/executing/resume-running/waiting-approval Session、非终态 IncidentState 和非终态 ChangeExecution 都会保留；Trace 与 Report 还会检查同 Incident 是否仍有活跃 State、Session、Approval、Alert 或 ChangeExecution。特别是 `waiting_manual_execution` 这类状态，即使诊断 Session 已 completed，仍必须保留它依赖的 Approval、Snapshot、IncidentState、Trace 和 Report，否则重启后只能看到一条孤立的变更记录。

这套 retention 只清理七张运行表，不覆盖本地工具 Artifact、上传文件、评测日志、Milvus 或其他知识资产。它没有数据库外键级联，而是用业务状态和 `NOT EXISTS` 保护恢复链；新增状态若没有同步更新终态集合，会偏向保留而不是误删。若数据库 Trace 已保留 90 天、对应 Artifact 却 7 天删除，复盘就只能看到摘要；这不一定错误，但必须形成明确契约。反过来，Artifact 保留更久也可能产生隐私与成本风险。

SQLite→MySQL 迁移脚本以 SQLite `mode=ro` 打开源库，并在一个只读事务快照中读取七表 Pydantic payload；缺少旧表时按空表兼容，其他 SQL 或 payload 错误继续抛出，不把读取失败伪装成零条数据。命令默认 dry-run，只有显式 `--execute` 才连接目标 MySQL。执行时七类记录进入同一个 MySQL 事务，任一 insert 失败会全部回滚；目标已有业务主键时不覆盖，汇总同时给出 `imported` 与 `skipped_existing`，因此重复执行不会把目标库较新的状态改回 SQLite 旧值。

这种幂等策略是 fail-safe 的“只插入缺失记录”，不是自动合并冲突。若源目标同一 `session_id` 或 `report_id` 内容不同，脚本会保留目标并报告 skipped，而不会判断哪份更新；它也没有双写、断点续传、增量追平、payload hash 对账和在线切流协议，所以仍不是零停机迁移方案。旧 JSONL 兼容读取也只是历史迁移辅助，不是与数据库并行的主存储。

生产迁移至少要加入冻结点或双写窗口、行数与业务键校验、冲突清单、payload hash、灰度读、回滚条件和迁移 Artifact。当前脚本已解决“部分失败留下半套表”和“重跑覆盖目标新数据”，但“脚本运行完”仍不能证明所有 Incident 都能在新库重建。

### 初始化 Schema 为什么必须和代码一起审查

SQLite Store 会自行创建七张表和索引；MySQL Store 的 `_initialize` 也定义相同运行表，部署目录另提供 `002_aiops_runtime_store.sql`。两处都包含 Alert、Trace、Approval、ChangeExecution、Session、IncidentState 和 Report，ChangeExecution 还以 `(incident_id, change_plan_id, approval_id)` 建业务唯一键。旧 MySQL 表若缺该索引，代码会先检查历史重复组；存在重复时只记录 migration warning，不擅自删除或合并历史数据。

部署初始化脚本授予运行账户对 `autooncall.*` 的 SELECT/INSERT/UPDATE/DELETE 以及建表、改表和索引权限，使首次启动和兼容索引检查不再因旧版只读授权失败。当前本地容器的只读核验只说明这些表和权限已经存在。生产上更合理的权限拆分仍是由迁移账户管理 DDL，应用账户只保留必要 DML。

## 用指标验证机制，但不要让数字越过证据边界

### 延迟为什么看 P50、P95，还必须说明 P99 缺失

P50 表示一半请求延迟不超过该值，适合描述典型体验；P95 观察慢尾，能暴露少数工具或模型调用拖长链路；P99 更接近极端尾延迟，但需要足够样本才稳定。只报平均值，会掩盖 Agent 多步骤链中的长尾。

2026-07-12 的 `local_live` 模型性能 run 为 `stage6-20260712T072656Z`，并发配置记录为 `10`，材料是 [performance_real_model.json](../../../logs/performance_real_model.json)。它是独立观察产物，不是 official baseline `20260712T074629Z-81f1f98c-0d69abdc` 的同一次运行：

| 场景 | 请求数 | 成功率 | P50 | P95 | P99 |
|---|---:|---:|---:|---:|---:|
| RAG | `20` | `20/20` | `3414.33ms` | `5816.70ms` | `未记录` |
| AIOps | `10` | `10/10` | `49123.07ms` | `63262.73ms` | `未记录` |

Artifact 实际只保存 count、P50、P95、min 和 max，没有 P99；不能用 max 冒充 P99，也不能事后凭小样本自行补算并覆盖历史结果。AIOps P95 高于 RAG，符合规划、跨源工具和状态推进的长链路结构，但 `10` 个 AIOps 请求与 `20` 个 RAG 请求不能证明容量、并发稳定性或生产 SLO。

该 run 的 Token 状态是 `not_observed`，原因是 HTTP 响应与持久 Trace 契约未暴露 provider token usage；成本状态为 `not_run`，因为没有应用带日期和来源的价格快照。2026-07-13 另一份 RAG candidate 独立观测到 provider input/output/total Token `4190/503/4693`，它不是同一 run，不能回填到 2026-07-12 的性能 Artifact。

### `7/7` 和 `2/2` 分别证明什么

受控故障材料保留了以下分子和分母：

- Redis `2/2`、MySQL `2/2`、下游依赖 `3/3`，合计 `7/7` 完成故障注入与 cleanup verification；
- Prometheus、Loki、Redis、MySQL、CMDB、部署历史、工单共 `7` 类 source 通过本地真实适配器验证；
- Redis 与 MySQL 主链各 `3/3`；
- Redis maxclients 与 MySQL 慢查询两条同窗口实验均输出正确 Top-1 并完成恢复；
- 两条端到端 controlled-fault 的“告警至诊断”为 `3.474s`、`0.678s`，恢复为 `0.325s`、`2.603s`。

`7/7` 不是诊断准确率。[七例 summary](../../../logs/controlled_fault/controlled-fault-stage7-original-20260712/summary.json) 的 diagnosis 状态实际是 `not_run`，所以它只证明七个预设计实验完成注入和清理。只有[两例端到端 summary](../../../logs/controlled_fault/controlled-fault-e2e-stage7-20260712/summary.json)记录 Redis/MySQL diagnosis `2/2`。即使 `2/2=100%`，它与 `2000/2000` 的不确定性也完全不同。

这些时间产生于本地、预设计的 `controlled_fault`，不能称为生产 MTTD 或 MTTR。历史文件有样本、时间和 evidence level，但缺少完整 command、Git commit、dirty 状态、dataset hash 与环境指纹，因此只能作为历史支持材料；未知 provenance 不得自行补写。

### official、candidate、stale 与 evidence level 不在同一条轴上

`candidate` 表示一次可供分析、但可能来自 dirty worktree 或尚未满足晋级条件的候选结果；`official` 表示满足当次冻结治理规则的正式基线；`stale` 表示当前代码、数据集、知识、配置或 Prompt 已与 Artifact 生成时不一致。它们描述结果治理状态。

`offline_fixture`、`local_live`、`controlled_fault`、`production` 描述证据产生环境。official 仍可能只是 offline fixture，local-live 小样本也可能只是 candidate，二者不能互相替代。

文中用于对照的历史 official Run ID 是 `20260712T074629Z-81f1f98c-0d69abdc`，命令为 `scripts/eval/run_benchmark_baseline.py --output-root logs/benchmarks`，commit 为 `81f1f98ccbc4588f8470ad82d3fb82ad849c431f`，`git_dirty=false`；Agent 数据集 `eval/cases.yaml` 的 Git blob / dataset hash 为 `94ddd598e018e632b6464dd17b99e793ab864b27`，其模块 evidence level 为 `offline_fixture`。运行清单见 [baseline_manifest.json](../../../logs/benchmarks/20260712T074629Z-81f1f98c-0d69abdc/baseline_manifest.json)。它不是本章性能 run，也不是生产恢复证据；当前 dirty worktree 不能继承其 official 身份。完整指标、RAGAS、Recall、Precision、MRR、nDCG、F1、覆盖率与阈值统一见[第 15 篇](15-项目量化指标与校招验收.md)，本章不重新计算历史分数。

## 哪些故障必须降级或转人工

正常路径讲完后，更重要的是知道何时停止自动推断：

- SSE 无终态 EOF：查询 Run Status，不标成功；如果 Snapshot 也不存在，展示连接中断并转人工排查。
- 同 session Snapshot 不存在但同 Incident 只有其他运行的 Snapshot：拒绝恢复，避免跨运行误用审批。
- 只有 Report：只能 `report_fallback`，不能声称恢复完整 current plan、Evidence、Hypothesis 和 progress。
- Snapshot、Report 与可用 checkpoint 全部缺失：恢复失败，不凭 Trace 摘要拼造可继续执行的状态。
- Snapshot 与 Incident 不匹配：拒绝恢复，优先调查关联键和数据污染。
- Artifact 指针不可读：保留摘要并标明原始制品缺失，不能查询当前依赖冒充历史结果。
- Replay 关联到缺 provenance 或可能 stale 的评测：只能作为线索，不能显示为当前质量结论。
- Snapshot 写入失败或材料时间互相矛盾：显示“最近成功持久化时间”，不要用页面最后一条 SSE 覆盖数据库事实。
- 审批通过但没有人工变更回执：只能说明许可成立，不能推断生产重启或恢复已经发生。

降级不是系统“不够智能”，而是证据系统的核心能力。能明确说“我只找到报告，无法还原完整过程”，比生成一条顺滑但虚假的时间线更可信。

## 当前方案的取舍与生产化路线

Trace + latest Snapshot + 业务记录 + Read Model 的优点是实现直接、查询清晰，并且不要求把每个事件都设计成可重算状态。代价是多份事实可能短暂不一致，fallback 逻辑复杂，Snapshot 又只有最新版本。

全事件溯源是另一种可行方案，能提供更完整历史与版本化投影，但必须承担事件 schema 演进、重算成本、顺序、一致性和副作用隔离。数据库 LangGraph checkpointer 可以增强节点级耐久恢复，却仍无法自动保证外部工具幂等：如果崩溃发生在“Redis 查询已发出、checkpoint 未写入”之间，恢复后可能重复调用。对于只读查询通常可接受，对变更工具则必须有幂等业务键和执行回执。

Replay 中重执行工具也并非绝对不可用，它可以作为“重新诊断”功能，但必须创建新的 session/trace，标明新时间窗口，并与历史 Replay 分开。AutoOnCall 当前选择只读 Replay，是为了避免把现在的数据写进过去。

走向多实例生产至少还缺：

1. 数据库级 graph checkpointer 或耐久工作队列，并验证崩溃点恢复与幂等；
2. Snapshot/Trace/业务单据的一致性策略、schema 版本和写入失败告警；
3. 分布式 Artifact Store、内容哈希、加密、访问控制和与数据库联动的 retention；
4. SSE 持久事件序列、Last-Event-ID 补发或明确的“只恢复最新状态”产品契约；
5. 真实 MySQL 中断恢复、连接池故障、多实例并发和备份恢复实验；
6. 以稳定业务键替代 token overlap 的评测关联，并让页面下钻到匹配 case 与原始 Artifact；
7. production 环境的持续成功率、丢事件率、Snapshot 写入失败率、恢复成功率和 P95/P99，而不是把本地 controlled fault 时间改名为生产 MTTR。

这些缺项不否定当前实现。它们说明当前系统适合演示和验证“运行材料怎样被组织”，尚未证明可以承担生产级耐久工作流。

## 面试中怎样把“恢复”说准确

### 一句话定位

AutoOnCall 用 SSE 实时进度、脱敏 Trace、latest Session Snapshot 和业务读模型，支持故障过程可观察、可查询，以及审批后的有限报告恢复。

### 30 秒回答

> SSE 只负责当前连接，`0` cursor 也必须保留，通用终态规范化为 `complete/done/error`；AIOps 使用 complete/error，聊天消费映射后的 done/error。服务生成器在终态前耗尽会补 error，客户端裸 EOF 则查询 Run Status。Trace 保存脱敏后的过程事件，ToolCall 区分真实调用与 analysis fallback，latest Snapshot 保存一次 session 的最近业务状态，Replay 聚合已有事实生成时间线。诊断以 `session_id` 原子占位，complete 先落 Snapshot；审批恢复绑定原 session 和 approval，按同 session Snapshot、匹配 Report 逐级 fallback，并用 `resume_running` 条件迁移抑制重复恢复。LangGraph MemorySaver 只作运行期缓存，所以恢复的是审批后的报告闭环，不是任意节点续跑，也不会重执行工具或自动执行生产变更。

### 3 分钟展开思路

> 我先从失败模式拆分需求：只写日志无法结构化查询，只存事件难以快速得到当前状态，只存快照又解释不了过程。因此系统保留四层材料：SSE 传输实时 progress；Trace 记录节点、工具、风险与审批历史；Session Snapshot upsert 当前计划、证据、假设和最近 20 条 progress；Approval、Change、Report、IncidentState 保存业务事实；Overview、Run Status 和 Replay 作为只读投影。
>
> Trace 写入前会脱敏，摘要最多 1000 字符，列表最多 20 项；但字典和 Snapshot 还没有统一总大小预算，所以不能夸大为全局数据治理。Trace 写失败不阻塞已经持久化的业务终态。LangGraph MemorySaver 不耐进程重启，也不参与恢复决策。审批恢复优先同 session Snapshot，最后使用身份匹配的 Report fallback；不会挑同 Incident 的其他运行。恢复先 claim approval 和 `resume_running` 状态，失败会落 failed 并允许带新 attempt 的显式重试；它补报告和 Trace，不重跑图，也不执行生产动作。
>
> Store Protocol 有 SQLite/MySQL 两个实现和七类记录；关键 Session/Incident 写入使用原子 create、条件更新或行锁，Trace ID 不可变幂等，审批终态拒绝 stale payload 回退，读取异常不伪装成空数据。retention 会保护活跃审批与变更依赖，SQLite→MySQL 导入采用只读源快照、单目标事务和非覆盖式幂等写。当前自动化仍以 SQLite、stub、offline fixture 为主；本地 MySQL 只验证了 health、Schema 和权限，没有真实重启恢复与 HA 证据。Replay 是历史材料的派生展示；评测关联会拒绝 stale Artifact，并展示 provenance 与 match method/score，但文本 overlap 仍不是可靠外键。现有 local-live 性能样本是 RAG 20/20、AIOps 10/10，只记录 P50/P95；controlled fault 的 7/7 只证明注入与清理，诊断 E2E 是另一个 2/2 小样本，均不能外推为生产 SLO 或 MTTR。

### 简历可以写什么

> 为长时 AIOps Agent 构建 SSE 进度、脱敏 Trace、SQLite/MySQL Store Protocol、latest Session Snapshot 与 Incident Replay 读模型，以 `session_id` 原子占位诊断运行、终态先持久化后发送，并实现 approval-bound checkpoint→same-session snapshot→matched report 的审批恢复 fallback、重复恢复拒绝与失败重试状态；同时明确 Replay 不重执行工具、Snapshot 不等于数据库级任意节点恢复。

> 对 SSE 和日志输入补充协议边界：保留 `0` cursor、统一识别 complete/done/error、兼容 CRLF 与无尾随换行末帧，拒绝业务 ID 控制字符并将日志字段单行化，避免首事件丢 ID、误判 EOF 与日志注入。

> 补强 AIOps 数据运维边界：retention 仅删除过期终态并保护活跃 Approval/Change 关联链；SQLite→MySQL 使用只读源快照、七表单事务导入、目标记录不覆盖和部分失败回滚，维护脚本默认 dry-run/显式确认，降低误删、半迁移和重跑回退风险。

### 不可夸大的表述：哪些说法会越界

- 不写“实现 LangGraph 数据库级任意节点恢复”。
- 不写“基于完整事件溯源实现 time travel”。
- 不写“Replay 重放工具并复现事故现场”。
- 不写“审批通过后 Agent 自动完成生产重启”。
- 不写“Artifact 已实现分布式存储、合规 retention 和 WORM”。
- 不写“MySQL 持久化与 HA 已在真实集群验证”。
- 不把 `7/7` 注入清理写成 `7/7` 根因诊断，也不把两例 controlled-fault 时间写成生产 MTTD/MTTR。
- 不给未记录的性能 P99、dataset hash 或环境指纹补造数字。

### 高频追问与回答要点

1. **SSE、Trace、Snapshot 为什么不能合并？**  
   SSE 优化当前传输且允许断线；Trace 优化历史审计；Snapshot 优化当前状态查询。可靠性、保留周期和载荷治理不同，强行合并会让任一用途都变差。

2. **MemorySaver 和 Session Snapshot 的本质区别是什么？**  
   前者只在当前图执行期间缓存引擎状态；后者是应用数据库中的 latest business state，也是查询和恢复唯一真相，但不会自动让 LangGraph 从下一节点继续。

3. **为什么 EOF 不能直接视为 complete？**  
   EOF 只说明连接结束，进程崩溃和代理超时也会产生 EOF；业务成功必须由明确终态或持久状态证明。

4. **为什么 cursor 为 `0` 也必须写入 SSE `id`？**  
   `0` 是合法序号，不是“没有 cursor”。用 truthy 判断会丢掉首事件身份，破坏去重和后续补发协议的基础。

5. **只有 Report 时到底恢复了什么？**  
   能把审批决定写进 post-approval 报告并闭合生命周期；不能还原完整 current plan、所有 Evidence、工具上下文和真实 progress。

6. **Replay 为什么不是重放？**  
   它读取历史记录并构造视图，不调用外部系统。若重新查询 Redis，得到的是当前数据，必须建立新的诊断 session，不能混进历史时间线。

7. **Trace 已脱敏，为什么仍有数据风险？**  
   规则只能覆盖已知敏感模式；摘要和列表有限额，但字典没有统一总预算，Snapshot 也无同等级全局治理。仍需要白名单、加密、访问控制和泄漏检测。

8. **多副本部署最先暴露什么问题？**  
   MemorySaver 不共享，本地 Artifact 路径可能不可见，latest Snapshot 可能竞争覆盖，还缺耐久任务调度与严格消息补发。

9. **如何证明恢复不是只在单元测试里成立？**  
   应在真实 MySQL 和多实例环境注入进程崩溃，覆盖不同断点，记录 command、Run ID、commit、dirty、环境指纹和 Artifact；再核对工具是否重复、Snapshot/Report 是否一致、恢复耗时 P50/P95/P99 与失败分母。

10. **为什么 stale evaluation 会误导 Replay？**  
   旧 case 可能匹配当前 Incident，但代码、数据或配置已变化。当前实现会拒绝 stale/不可用 summary 的 linked evaluation，并保留 Artifact、Run、commit、dirty、dataset、fingerprint、stale reason 与 match method/score；因此历史材料仍可解释，但不会显示为当前 passed。剩余边界是 token overlap 仍是启发式关联，不是业务外键。

11. **latest Snapshot 与事件溯源相比如何选？**  
    如果目标是快速展示当前状态与有限恢复，latest Snapshot 成本较低；如果需要任意版本回溯、审计投影重算，就要付出事件版本、一致性和副作用隔离成本。两者也可以并存。

12. **为什么同 Incident 的最新 Snapshot 不能兜底审批恢复？**
    Incident 是多轮调查的业务边界，ApprovalRequest 只授权其中一轮的具体动作。选择另一 session 会把批准、Evidence 和报告串错；因此恢复必须匹配原 session/pending approval，缺失时失败关闭。

13. **如何避免 API 已发 complete、重启后 Run Status 仍是 running？**
    complete 前把最终 Snapshot 作为 required write；写入失败就不发 complete，并把已 claim 的恢复状态收敛为 failed。中间 Snapshot 仍是 best-effort，所以查询只能保证最近成功持久化状态。

14. **迁移脚本为什么既幂等又不能自动解决冲突？**
    目标主键已存在时不更新，所以重跑不会覆盖目标较新的状态；但源目标内容不同时只会记为 skipped，不会判断谁更可信。上线切换前仍需冻结点、hash 对账和冲突清单。

15. **Retention 为什么要沿关联链判断，而不是每张表独立按时间删？**
    completed Session 旁边仍可能存在 `waiting_manual_execution` ChangeExecution。若只看 Session 时间，会删掉 Approval、Report 和 Trace，导致变更流程无法恢复。清理必须先判断 Incident 是否还有任何活跃业务对象。

16. **新增回滚状态为什么也必须更新 retention？**
    状态模型加入 `rollback_failed` 后，如果只修改 Pydantic、读模型和页面标签，却遗漏 SQLite/MySQL 清理 SQL，它会被永久当成活跃记录，连带 Approval、Session、Trace 和 Report 无法过期。当前 `closed`、`rolled_back`、`rollback_failed` 均按终态处理，`partial_success` 与 `recovery_pending` 仍保护关联链。

## 检验真正理解的练习

1. 为 Redis 等待审批案例画出 `incident_id`、`session_id/diagnosis_run_id`、`trace_id`、`approval_id` 和 `report_id` 的关联。先证明删除 MemorySaver 不改变恢复结果，再分别删除同 session Snapshot 和匹配 Report，写出每一步的恢复结果；再加入同 Incident 的另一 session Snapshot，解释为什么必须拒绝。
2. 给出一个“工具已成功、Trace 已写、Snapshot 未写时进程崩溃”的时间线。说明重启后 Run Status、Replay 和审批恢复各能看到什么，哪里会出现不一致。
3. 设计数据库 LangGraph checkpointer 方案。至少回答 checkpoint schema、节点幂等键、只读工具重复调用、变更工具副作用、版本升级和 controlled-fault 验证方法。
4. 为 SSE 增加 Last-Event-ID 补发设计：事件保存在哪里，cursor 怎样单调，重复事件怎样去重，等待审批状态是否终止连接，retention 到期后怎样降级。
5. 设计分布式 Artifact Store 契约，包含内容哈希、加密、租户权限、实例无关 URI、数据库记录和 Artifact 不同保留期的处理。

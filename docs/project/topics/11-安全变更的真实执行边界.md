# 审批通过以后，系统究竟完成了什么

上一章解决的是行动权问题：谁能发起诊断，工具动作如何被分成自动允许、需要审批和绝对禁止，人的决定又如何持久化。审批通过只说明一个高风险方案获得了继续评估的资格。本章要继续追问：计划是否仍适用于当前现场，系统所谓的 dry-run 到底验证了什么，人工回执能否代表恢复，以及 AutoOnCall 为什么刻意不把“批准”直接翻译成生产写操作。

下一章将转向 SSE、Trace、Snapshot、Store 与 Replay，解释这些状态怎样被观察、保存和有限恢复。本章先建立最重要的边界：**授权、可行、执行、观察、恢复是五种不同事实；前一项成立，不能自动推出后一项。**

本文事实审计日期为 `2026-07-13`。讨论范围是当前仓库实现、自动化测试和离线 fixture，不代表生产 Redis、Kubernetes 或 MySQL 写能力。完整跨专题指标见[第 15 篇：项目量化指标与校招验收](15-项目量化指标与校招验收.md)。

## Redis 根因很清楚，为什么仍不能直接 CONFIG SET

主线事故仍然是 `order-service`：接口出现 503，P95 延迟上升，日志反复出现 `Redis connection timeout`，同一事故窗口中的 Redis 观测显示 `connected_clients` 已接近 `maxclients`。诊断报告据此提出假设：连接压力可能导致新连接等待或失败，并建议先限流、排查连接泄漏，再在变更窗口评估提高 `maxclients`。审批人同意了这份方案。

一个自动修复 Demo 很容易接着执行：

```text
CONFIG SET maxclients <new-value>
```

这条路径短，却至少跳过了七个问题：批准是否属于本次 Incident；目标是不是同一 Redis 实例；当前值和批准时是否一致；文件描述符与内存能否承受；连接泄漏是否仍在增长；失败时怎样回退；系统是否真的持有生产写凭据。

AutoOnCall 对最后一个问题的答案很明确：当前没有 Redis、Kubernetes 或 MySQL 的生产写适配器。它不会因为审批状态变成 `approved` 就获得一份隐藏的生产凭据。因此，本章真正实现的是一条**安全变更记录工作流**，而不是生产自动修复器。

此刻应先盘点认知边界：

- 已知事实是 503、P95 上升、Redis 超时日志和连接数逼近上限。
- Runbook 关于连接上限、文件描述符和连接池属于知识。
- “连接上限是主要根因”是由多源证据支持的推断，不是新的观测事实。
- 审批事实只表示人允许这个方案进入受控检查，不表示参数已修改。
- 仍缺少当前现场复采、资源余量、目标版本、真实 dry-run 和变更后独立观察。

所以正确的下一步不是“既然批准就执行”，而是把批准后的过程拆成可停止、可审计的阶段。一旦对象错绑、计划过期、回滚缺失或演练边界不满足，流程应停止或转人工。

## 一个 approved 布尔值为什么装不下真实变更

### 失败一：把审批通过直接写成 completed

最朴素的实现只保存 `approved=true`，随后把 Incident 标成完成。交班者却无法回答：计划有没有过期，谁执行了什么，是否出现部分成功，观察窗口多长，回滚是否发生。授权记录被误当成结果记录。

### 失败二：先执行，再用日志补审计

另一个常见做法是让 Agent 直接执行命令，事后把 stdout 保存下来。它把最重要的约束放在风险已经发生之后。目标漂移、重复请求、命令超时和部分写入不会因为留下日志而消失；一条“command succeeded”也不等于业务指标恢复。

### 失败三：把 dry-run 当成“小规模真实执行”

不同基础设施的 dry-run 语义并不统一。Kubernetes server-side dry-run 可以让 API Server 做部分校验，却不证明 Pod 能调度并健康；MySQL `EXPLAIN` 不等于 DDL 能安全上线；Redis 配置预览更不等于真实 `CONFIG SET`。如果只是检查 JSON 字段，却把名称写成“基础设施演练成功”，读者会被状态名误导。

### 失败四：人工说成功，系统就自称已恢复

人工回填有价值，因为它能把外部平台操作纳入统一时间线。但回填仍是操作者的主张。若系统没有从监控平台重新拉取 5xx、P95、超时日志和 Redis 指标，就不能独立证明恢复。

因此，批准后的认知链应写成：

```text
人允许继续
    ≠ 计划仍适用
    ≠ 结构校验通过
    ≠ 外部动作已执行
    ≠ 观察指标达标
    ≠ 生产故障已恢复
```

安全变更状态机的价值，就在于让这些不等式在数据模型和 API 上可见，而不是只写在提示词里。

## 先建立不会互相冒充的概念

### Incident：本次故障的稳定上下文

通俗地说，Incident 是“这次事故”的档案袋。严格定义上，它有稳定 `incident_id`，关联服务、环境、症状、时间和生命周期。它解决告警、诊断、审批、变更记录归属问题；它不负责证明根因或恢复。

例如 `order-service` 的 Redis 事故与 `catalog-service` 的另一次 Redis 告警必须使用不同 Incident。AutoOnCall 会校验 URL 中的 Incident 与 Approval、ChangePlan 的归属，防止跨事故复用授权。

### Evidence：可追溯观测，不是方案

Evidence 通俗上是“当时实际看到了什么”。严格定义上，它带来源、时间、步骤和内容，用于支持或反驳假设。它解决事实追溯；它不提供生产写权限，也不会自动决定回滚。

`connected_clients=9800`、`maxclients=10000` 可以是 Evidence；“应该提高到 20000”是计划参数，不是 Evidence。AutoOnCall 的 pre-check 虽然保存名为 `evidence_snapshot` 的字段，但当前内容主要是审批与计划摘要，并没有重新采集 Redis 现场证据，这一点不能被字段名掩盖。

### Hypothesis：对事实的因果解释

Hypothesis 即根因假设。严格地说，它是可被新证据推翻的因果解释，通常包含支持、反证和置信度。它解决“为什么会发生”；它不负责决定生产动作。

“连接逼近上限导致 Redis 超时”是 Hypothesis。若当前连接数已经回落、503 仍持续，假设就需要调整。AutoOnCall 的安全变更入口不会自动重跑整套根因分析，所以计划过期或现场变化时应停止并重新诊断。

### ChangePlan：批准前后都不执行的结构化施工单

通俗地说，ChangePlan 是“准备怎么改”。严格定义上，它与 Incident 绑定，包含动作、风险、有效期、前置清单、执行步骤、回滚步骤、验证步骤、影响面、观察指标和人工执行边界。它解决方案可审查、可绑定；它不负责调用生产接口。

Redis 计划可以写“人工评估并调整 `maxclients`，观察 `redis_connected_clients`、`redis_rejected_connections`、5xx、P95 和 timeout 日志，异常时恢复原值”。反例是一段已经执行的 `CONFIG SET` 输出，那是执行结果，不是计划。AutoOnCall 生成的计划默认 `manual_execution_required=true`，备注也明确 Agent 不自动执行生产动作。

### ChangeExecution：一次批准计划的办理记录

通俗地说，ChangeExecution 是“这张施工单办到哪一步”。严格定义上，它用稳定 ID 关联 Approval、ChangePlan、Incident 和 Trace，保存模式、状态、pre-check、dry-run、人工回执、Observation 与回滚建议。它解决重复请求、状态恢复和审计；它本身不是生产执行器。

例如 `waiting_manual_execution` 只表示系统等待外部操作者回填，并不表示 Redis 正被 Agent 修改。AutoOnCall 会用同一 Approval 与 ChangePlan 复用同一条 ChangeExecution。

### Pre-check：治理条件复核，不是现场健康检查

通俗地说，Pre-check 是开工前核对“是不是同一张单、还能不能继续”。严格地说，当前实现检查归属、批准状态、风险一致、计划有效期和高风险回滚覆盖。它解决错绑、过期和明显治理缺口；它不重新查询 Redis，不验证目标当前版本，也不证明资源余量。

高风险 Redis 计划没有任何 rollback step 时会被阻断。反例是看到 `checked_items` 里写着“目标服务、环境、动作已校验”，就认为代码逐字段比较过这些值；当前实现会无条件追加这段说明，并没有对应的逐字段条件判断。

### Dry-run：当前只是确定性结构门，不是基础设施演练

通俗地说，Dry-run 是“在不写生产的前提下先检查计划”。严格定义上，当前实现读取结构化步骤，阻断 `can_dry_run=false` 的步骤，并生成固定差异预览；测试 metadata 还能强制模拟失败。它解决明显不可演练步骤继续前进的问题；它不向 Redis、Kubernetes 或 MySQL 发请求，也不验证参数真的可被接受。

Redis 预览会写明“不调用生产 Redis `CONFIG SET`”。反例是把 `dry_run_completed` 翻译成“Redis 配置已在测试实例验证”，仓库没有这项能力。

### Observation：状态中的观察结果，不一定来自观察器

Observation 通俗上是“变更后看到了什么”。严格定义上，它保存窗口秒数、指标、日志、成功/失败条件和建议。它解决结果呈现；它不保证数据来自可信监控源。

当前 manual 模式的指标由请求体提供，sandbox 模式则把计划中的指标统一填为 `sandbox_ok`。没有定时任务等待窗口，也没有阈值计算器。`window_seconds=300` 是记录字段，不表示系统真的观察了 300 秒。

### Sandbox：合成状态流，不是预生产环境

通俗地说，Sandbox 是“用固定道具走一遍流程”。严格定义上，当前实现是进程内的确定性 fixture：记录固定日志，把观察指标写成 `sandbox_ok`，不调用生产写接口。它解决模型、服务、Store、Trace 和报告之间的联动测试；它不验证真实基础设施。

生产环境标签默认会进入 `escalated`。如果计划 metadata 显式包含 `sandbox_enabled`，合成 fixture 也可以继续，但它仍然只是本地结果，不能因计划标了 prod 就升级为生产证据。

### Trace：事件账本，不是外部系统事实源

Trace 记录 pre-check、dry-run、等待人工、sandbox 观察和回滚建议等事件。严格定义上，它是带 Trace ID、Incident、执行 ID、类型、状态和脱敏元数据的运行时间线。它解决审计与调试；它不保证 Redis 当前状态，也不与所有业务表构成单一事务。

AutoOnCall 每个关键阶段都会写 Trace，但 ChangeExecution 先持久化、IncidentState 再投影、报告 best-effort 同步、Trace 另行写入。因此少一条 Trace 不能推翻已经持久化的执行记录，反之也不能只凭 Trace 声称外部变更发生。

### Snapshot 与 Replay：恢复的是系统认知，不是基础设施

Snapshot 是某个诊断会话的持久化状态书签，解决进程重启后的有限恢复；它不是 Redis RDB，也不能回滚配置。Replay 是用历史 Incident、Trace、报告、ChangeExecution 或 fixture 重建过程，解决复盘和离线评测；它不能重现当时全部外部依赖。

下一章会详细展开二者。本章只需记住：从 Snapshot 恢复出一条 `closed` 记录，不等于重新验证生产仍然健康；用 fixture Replay 得到 `sandbox_ok`，也不等于本地真实依赖验证。

### Trust Gate：允许进入下一阶段的条件集合

Trust Gate 可译为可信门。严格定义上，它根据显式规则决定继续、停止或升级，并保存理由。它解决阶段之间的最小信任条件；它不负责保证规则完备。

本章的 Pre-check 和 Dry-run 都是 Trust Gate。Pre-check 通过并不替代 Dry-run，Dry-run 通过也不替代真实执行和观察。AutoOnCall 当前门禁规则较窄，生产化必须补目标版本、参数哈希和实时证据。

## 从风险一步步推导四段式工作流

### 第一段：重新绑定授权与计划

服务收到 Incident、ChangePlan 和 Approval 三个 ID 后，先从持久化审批记录加载计划。入口会确认：Approval 属于 URL 中的 Incident，状态为 approved，包含 ChangePlan，且 ChangePlan ID 与 URL 一致。随后 Pre-check 再检查：Approval 与 Plan 的 Incident 一致、两者状态均为 approved、风险等级相同、计划未过期；若风险为 high，还必须有 `rollback_steps` 或 `rollback_plan`。

计划过期按 `created_at + expires_in_seconds` 判断，默认有效期 3600 秒。若时间对象没有时区，按 UTC 补齐。这个规则简单、确定，但它只表示“文档年龄未超阈值”，并没有检查 Redis 目标的资源版本。

当前 evidence snapshot 保存 Approval ID/状态/决定时间、Plan ID/状态、风险、blast radius、observe metrics 和最新报告状态。它没有保存完整 Evidence、计划内容 hash、目标配置版本或重新采样时间。因此更准确的名字是“治理摘要快照”。

### 第二段：检查计划结构，不触碰生产

Pre-check 通过后进入 Dry-run。若 Plan 有结构化 `steps`，系统遍历它们；`can_dry_run=false` 的 step ID 进入 `blocked_steps`，整体失败。若只有旧式文本 `execution_steps`，系统会把它们转换成默认可 dry-run 的人工步骤。

这里存在一个必须主动说明的边界：若 `steps` 和 `execution_steps` 都为空，当前逻辑没有阻断项，仍可能给出 passed。Dry-run 也没有逐项验证 target 非空、input args 完整、observe metrics 有阈值或 rollback step 可达。它主要是一道“不可 dry-run 标志”门和静态预览生成器。

因此 Redis 案例中 `diff_preview` 的意义是提醒审阅者将检查 maxclients 动作、目标、回滚和观察指标，并明确不调用 `CONFIG SET`；它不是这些内容已经被真实 Redis 接受的证明。

### 第三段：按模式停在不同边界

Dry-run 成功后，调用方必须明确选择模式：

| 模式 | 可能状态 | 系统实际完成的事 | 没有完成的事 |
|---|---|---|---|
| `dry_run_only` | `dry_run_completed` | 保存 Pre-check、结构门结果和预览 | 没有执行生产变更 |
| `manual_record` | `waiting_manual_execution` | 等待外部操作者提交回执 | 没有替人登录变更平台 |
| `sandbox` | `sandbox_validated` 或 `escalated` | 运行本地合成 fixture，或因生产边界转人工 | 没有连接真实 Redis/K8s/MySQL |

若同一执行已经是 `dry_run_completed`，之后可以请求进入 manual_record 或 sandbox。系统会重新跑 Pre-check，避免永久复用旧校验。若重验失败，原 execution 进入 `precheck_failed`，已有 DryRunResult 仍保留用于审计。

其他终态通常不会自动重跑，而是返回既有记录。也就是说，`dry_run_failed` 后用同一 Approval/Plan 重试不会自然产生一次新执行；应修正计划、重新审批或设计显式重试语义。

### 第四段：记录人的主张或合成观察

manual_record 只接受处于 `waiting_manual_execution` 的记录。请求可提交 `succeeded` 或 `failed`、操作者、说明、外部证据、observed metrics 和观察窗口；evidence 与 metrics 各自序列化后不能超过 20000 个字符。

若人工填 `succeeded`，当前实现不计算任何指标阈值，直接生成 passed Observation，success criteria 只是 `manual_result_recorded`，最终 execution 状态为 `closed`。若填 `failed`，Observation 为 failed，状态转为 `rollback_recommended`，并保存一条“建议按 ChangePlan 回滚”的结果。

这套设计能把外部动作纳入审计，却不能独立证明结果。更值得注意的是，读模型会把 `closed` 映射为 Incident/Report 的 `resolved`，界面标签是“已恢复”。所以这里的 resolved 是**由人工成功回执驱动的乐观投影**，不是监控系统验证的客观恢复。生产实现应至少区分 `reported_succeeded`、`observing` 和 `verified_resolved`。

## 一份 Redis ChangePlan 应该怎样被人真正读懂

### 计划完整不等于计划正确

结构完整性是安全变更的必要条件。没有目标、参数、回滚和观察指标的自然语言建议，无法可靠审批。但字段齐全仍可能内容错误：目标写成 `order-service`，实际需要修改的是共享 Redis 实例；回滚写“恢复原值”，却没有保存原值；观察指标列了 P95，却没有基线和阈值。

AutoOnCall 的计划构建器会根据 action、tool name、reason 和 metadata 拼成的文本推断领域。文本包含 `redis` 或 `maxclients` 时，执行步骤类型变为 `redis_config_change`，并生成 Redis 观察指标与安全说明；包含 MySQL、慢查询、连接池、索引等词时生成数据库 playbook。这让计划更贴近事故类型，却仍是关键字启发式。metadata 的键和值也会参与判断，因此错误标签可能把通用处置误归到 Redis 或 MySQL。

审阅者应把计划分成三层：第一层是对象身份，例如 Incident、环境、服务、Redis 实例和配置版本；第二层是动作语义，例如从什么值改到什么值、谁执行、是否可逆；第三层是成功判据，例如 5xx、P95、连接拒绝和 timeout 日志在多长窗口内达到什么阈值。当前模型覆盖了字段容器，尚未强制第三层的阈值表达。

### 前置清单必须回答“为什么现在仍适合改”

计划构建器默认要求重新查看最近 10—15 分钟指标、日志和依赖状态，确认变更窗口、审批人、旁路监控与回滚负责人。对于 Redis，还应回答：

- `connected_clients` 是持续增长、稳定逼近还是已经回落？
- `rejected_connections` 是否增加，应用超时与它是否时间对齐？
- 文件描述符、内存和实例容量能否承受目标上限？
- 客户端是否存在连接泄漏，调大上限会不会只是延后故障？
- 是否为共享实例，blast radius 中还有哪些服务？

这些问题说明为什么要在执行前重查。但当前服务并不会自动执行默认 pre_checklist，也不会把每一条变成可验证断言；它只检查较窄的治理条件。课程阅读时要区分“计划建议检查什么”和“代码已经检查了什么”。

### 回滚步骤必须能落到确定对象

“如果失败就回滚”不是可执行回滚。Redis 计划至少应保存变更前值、目标实例、回滚操作者、触发阈值和回滚后的再观察。若变更同时涉及连接池、限流和 Redis 参数，还要说明回滚顺序，避免只恢复一个参数却留下组合状态。

AutoOnCall 会为构建出的执行 step 关联一个 rollback step ID，高风险计划只要 `rollback_steps` 或 `rollback_plan` 非空便能通过当前门禁。这证明有回滚容器，不证明回滚内容与每个执行动作一一对应，也不检查 rollback target 是否与 execute target 相同。生产门禁应建立执行—回滚可达图，并拒绝不可逆步骤混入普通审批。

### 观察指标必须有来源、阈值和窗口

Redis playbook 默认列出 `redis_connected_clients`、`redis_rejected_connections`、`service_5xx_rate`、`service_p95_latency_ms` 和 `redis_timeout_log_count`。这是合理的多层观察集：Redis 自身、应用结果和日志都被覆盖。

但指标名不能独立构成成功条件。需要进一步绑定数据源、查询语句、事故前基线、通过阈值、连续窗口、缺失数据处理和冲突处理。例如“P95 下降”可能只是流量跌落；连接数下降却 rejected connections 增加，也不能判为恢复。当前 Observation 模型可以容纳 metrics 与 criteria，但 manual 路径没有计算这些关系，sandbox 更只是填入字符串。

### 停止条件比成功步骤更值得审查

正常步骤通常容易写，真正决定安全性的是停止条件：dry-run 输出与审批动作不一致、核心指标继续恶化、出现新 P1/P2 告警、只读证据显示根因变化、当前 runtime 已低于阈值而历史 Incident 证据不再支持变更。

如果停止条件触发，系统不应尝试“把计划做完”。它应保存当前阶段、阻断后续动作、说明还缺什么，并根据是否已产生外部写入决定重新诊断、转人工或建议回滚。当前 AutoOnCall 能表达部分停止状态，但没有自动数据采集器去触发这些业务条件。

## 状态机不仅要画正常箭头，还要解释失败以后怎么办

### 第一次启动的正常路径

一次新的 dry_run_only 请求会依次产生：

```text
created
  → precheck_running
  → dry_run_running
  → dry_run_completed
```

每次转换都会先保存 ChangeExecution，并投影 IncidentState、尝试同步 Report。Pre-check 和 Dry-run 还各自产生 running 与 completed Trace。SSE 客户端看到的是阶段事件，不应把收到最后一个 complete 事件误认为生产变更完成；complete 只表示这次状态机调用结束。

manual_record 的前半段相同，Dry-run 成功后转为 `waiting_manual_execution`。提交回执时内部短暂写 `manual_execution_recorded`，随后在同一同步函数中写 `closed` 或 `rollback_recommended`。代码没有真正停在 `observing` 等待窗口，`manual_execution_recorded` 通常也不会成为正常 API 响应中的长期状态。

sandbox 路径在 Dry-run 后先得到 `sandbox_executing` 或 `escalated`。若允许执行合成 fixture，函数立即构造 Observation 并保存 `sandbox_validated`，并没有后台作业持续运行。

### 校验失败后的记录为什么不能删除

Pre-check 或 Dry-run 失败时，execution 被保留为 `precheck_failed` 或 `dry_run_failed`。这样交班者能看到失败原因，也能证明系统没有悄悄跳过门禁。如果直接删除失败记录，重复请求可能像“第一次尝试”一样重新开始，审计者也无法统计哪些计划长期不完整。

同一 Approval/Plan 的后续请求会返回既有失败记录，不会自动清空错误再试。这是一种保守取舍，但也意味着修正一个小字段后不能复用原计划悄悄重跑；更准确的业务流程是生成新版本 ChangePlan，必要时重新审批，并显式关联它替代了哪个失败版本。

### 已完成 Dry-run 为什么可以继续，却必须重验

`dry_run_completed` 是当前唯一设计了“换模式继续”的关键状态。调用方可选择进入 manual_record 或 sandbox。继续前重跑 Pre-check 的原因是 Dry-run 与外部执行之间可能已经过去很久，计划也可能过期。

重验通过时，系统会写一条成功 Trace，但不会把新的 PreCheckResult 覆盖回 execution；重验失败时才保存失败结果并把状态改为 `precheck_failed`。因此审计读取 execution 中的 pre_check 时间时，不能假设它一定是最近一次成功重验结果。生产化应保存每次检查历史，而不是只有最后一份槽位。

### 人工回执失败后，建议回滚仍然不是回滚完成

人工提交 failed 后，`rollback_result.status` 是 `recommended`，原因是“人工执行失败或观察指标未达标，建议按 ChangePlan 回滚步骤处理”。它没有 rollback execution ID、操作者、开始/结束时间或外部结果。

这一区分非常重要：`rollback_recommended` 证明系统生成了建议，不能证明回滚已开始、已成功或指标已恢复。下一步应由人工确认是否已经产生外部写入；若根本没有执行，可能只需关闭失败记录；若部分执行，则需要单独受控的回滚流程。

## 原始状态、生命周期状态和页面文案不是同一个层次

ChangeExecution 保存细粒度原始状态，Incident 与 Report 使用更适合业务列表的生命周期状态，页面再根据 metadata 显示标签。三层映射能让 UI 更易读，也可能隐藏证据差异。

| 原始 ChangeExecution 状态 | Incident/Report 状态 | 正确解释 |
|---|---|---|
| `precheck_running` | `change_prechecking` | 正在做治理条件检查 |
| `dry_run_running` | `change_dry_run` | 正在做确定性结构检查 |
| `dry_run_completed` | `change_validated` | 计划通过当前规则，未执行生产 |
| `sandbox_validated` | `change_validated` | 合成 fixture 通过，未验证真实依赖 |
| `waiting_manual_execution` | 同名状态 | 等待外部执行回执 |
| `manual_execution_recorded` | `observing` | 代码中的短暂过渡，并无异步观察器 |
| `closed` | `resolved` | 人工成功回执驱动的乐观投影 |
| `rollback_recommended` | 同名状态 | 仅建议回滚，尚未证明已回滚 |
| `escalated` | 同名状态 | 自动路径停止并转人工 |

读模型还把流程展示成 Pre-check、Dry-run、Execute、Observe 四个阶段。dry_run_only 的 Execute 会显示 skipped，sandbox 的 Execute 可显示 passed，manual 的 Execute 显示回执说明。这个时间线反映系统记录，不应被当作外部控制面的完整日志。

状态命名的设计取舍值得面试讨论。将 `closed` 映射为 `resolved` 有利于列表收敛，却把“记录闭环”和“故障恢复”合并了。更严格的设计可以保留 `change_record_closed`，只有可信观察器满足阈值后才进入 `resolved`；若只获得人工证言，则显示 `reported_recovered` 并保留证据等级。

## Redis 主线的三条结局

### 结局一：计划校验通过，但现场仍未知

已批准的 Redis 计划包含回滚步骤，尚在有效期，选择 `dry_run_only`。Pre-check 通过，Dry-run 没发现 `can_dry_run=false`，产生“不调用生产 Redis CONFIG SET”的预览，状态为 `dry_run_completed`。

此时已知的是：Approval/Plan 的基本绑定、状态、风险、有效期和高风险回滚覆盖满足当前规则；结构化步骤没有声明为不可 dry-run。事实来源是持久化对象和确定性规则。

未知的是：Redis 是否接受目标值，文件描述符和内存是否足够，连接泄漏是否仍在发生，503 是否会恢复。下一步应转真实隔离环境或人工变更评审。若目标无法唯一定位、Plan 已过期或当前根因变化，则停止并重新诊断。

### 结局二：人工报告成功，记录闭环但证据仍分层

同一 execution 从 `dry_run_completed` 转入 `manual_record`。系统重新 Pre-check 后进入 `waiting_manual_execution`。工程师在外部平台操作，再提交：status=succeeded、notes、外部 evidence、观测指标和窗口。

AutoOnCall 记录这份回执，状态变为 `closed`，Incident/Report 投影为 `resolved`。此时的事实是“某主体提交了成功回执及这些字段”；知识是“若 5xx、P95、rejected connections 和 timeout 日志回落，通常支持恢复判断”；推断是“本次变更可能促成恢复”。在没有独立监控采集和阈值判断时，不能把第三项升级成系统验证事实。

若人工填 failed，流程进入 `rollback_recommended`。系统不会执行回滚，也没有定时观察器稍后自动触发回滚。值班人员仍需核对影响面、在外部平台实施回退并再次记录。

### 结局三：sandbox 全绿，只证明代码路径跑通

在 staging 计划上选择 sandbox，系统立即构造 passed Observation，把每个 observe metric 的值填成 `sandbox_ok`，日志写“sandbox adapter completed; no production mutation executed”，状态为 `sandbox_validated`。Incident/Report 映射为 `change_validated`，不会映射成 resolved。

这能证明 ChangePlan、ChangeExecution、Store、Trace、IncidentState 和 Report 的联动代码可以走通。它不能证明 staging Redis 真实可连，更不能证明生产故障恢复。若生产标签没有显式 `sandbox_enabled`，状态进入 `escalated`，rollback_result 标为 `not_started`；系统不会新建另一张审批，也不会自动回滚。

MySQL 与 Kubernetes 只作对照。MySQL 计划会强调不执行生产 DDL/DML，Kubernetes 即使未来接入 server-side dry-run，也只能证明 API 对象校验，不能证明调度和健康。统一的状态机可以复用，基础设施的真实验证契约不能强行统一。

## 幂等解决了哪一半重复问题

### 创建执行记录有稳定业务键

ChangeExecution ID 由 `sha256(approval_id:change_plan_id)` 的前 24 个十六进制字符生成，并加 `chgexec-` 前缀。SQLite 与 MySQL 表还对 `(incident_id, change_plan_id, approval_id)` 建立唯一约束。`create_change_execution_once` 先查业务范围，再使用 conflict-safe insert；并发冲突后读取已经存在的记录。

这比“先查后无条件插入”更强：HTTP 重试换了临时 ID，也会复用同一业务记录。同一 execution 可以从 dry_run_only 继续到 manual_record 或 sandbox，mode 不进入幂等键正是为了表达“同一批准计划的一次办理过程”。

旧数据库若已经存在重复业务记录，迁移不会擅自删除历史数据。系统记录 migration warning，无法创建唯一索引时退回应用层预查，并优先返回最早记录。这保留历史但不能消除既有歧义，生产上需要人工对账和迁移方案。

### 状态迁移和人工回执没有同等强度

稳定创建不等于整个状态机 exactly-once。后续 `save_change_execution` 是按 execution ID 的 upsert，没有版本号、compare-and-set 或状态条件更新。两个并发人工回执都可能先读到 waiting，再分别写入，形成 last-write-wins 风险。

人工结果还分两次保存：先写 `manual_execution_recorded`，再构造 Observation 并写 `closed` 或 `rollback_recommended`。若进程在中间崩溃，记录可能停在过渡态，而再次提交会因不再是 waiting 返回 409。成功请求的网络响应丢失后，客户端重试也不会返回原结果，而会遇到状态冲突。

所以准确说法是“执行记录创建具备数据库业务唯一约束”，不能扩张成“安全变更全链路 exactly-once”。生产化需要状态版本、条件更新、请求幂等键、事务或 outbox，以及过渡态恢复任务。

## 四份状态为什么可能暂时不同

每次保存 ChangeExecution 后，服务依次保存 IncidentState，并 best-effort 更新最新 DiagnosisReport；Trace 由另一次调用写入。它们不是一个跨对象事务。

这意味着可能出现：execution 已是 `dry_run_completed`，IncidentState 尚未更新；Incident 已变 `resolved`，报告同步失败仍停在 waiting；业务状态成功但 Trace 事件缺失。报告异常只记录 warning，不回滚 ChangeExecution。

事实优先级应按用途区分：ChangeExecution 是安全变更办理状态的主记录；Approval 是授权事实源；IncidentState 和 Report 是面向列表、交班和阅读的投影；Trace 是过程审计。读模型能防止旧报告把活跃变更生命周期拉回早期状态，但不能替代对账。

还有一个容易误读的字段：`dry_run_completed` 和 `sandbox_validated` 会让 `manual_action_required=false`。它表示这条 execution 当前没有正在等待的人工回执，不表示生产恢复不需要人。对应 next step 仍明确提示，如需恢复生产，应走人工执行或变更平台。

## 身份边界在变更 API 中仍然存在

启动安全变更和提交 manual result 都要求 `change` Scope；列表和详情要求 `read`。鉴权开启时，审计 operator 来自已认证 Principal 的 token name，请求体不能伪造；Approver 角色拥有 `change`，Operator 角色默认没有。

但当前 `api_auth_enabled` 默认是 `False`。本地默认模式会放行这些路由，operator 回退使用请求体字段。因此自动化 API 测试成功不等于身份真实，生产部署必须强制开启鉴权并接企业身份源。

即使鉴权开启，Change Scope 也不等于 Redis 管理凭据。当前服务根本没有生产写适配器；未来执行器还应使用范围更窄、短期、与 Plan 和资源版本绑定的一次性执行授权。

## 停止、降级和转人工的判断表

- Approval 不存在、非 approved、跨 Incident 或 Plan ID 不匹配：在创建 execution 前拒绝。
- Approval/Plan 状态或风险不一致、Plan 过期：`precheck_failed`，停止。
- high 风险计划缺少 rollback：`precheck_failed`，不进入 Dry-run。
- 任一步骤 `can_dry_run=false` 或 metadata 强制失败：`dry_run_failed`，不能静默跳过后进入 manual。
- 生产环境请求 sandbox 且未显式启用：`escalated`，记录未开始并转人工。
- 人工上报 failed：`rollback_recommended`，只建议，不自动回滚。
- 真实目标、当前配置、Evidence 新鲜度无法确认：即使现有 Pre-check 可能通过，也应在组织流程中停止并重新诊断。
- Report/Trace 投影失败：保留 ChangeExecution 主记录，告警并对账，不伪造全链成功。

这些分支体现的不是“当前方案唯一正确”，而是保守边界。某些组织可以接入成熟变更平台自动执行，但必须用真实 adapter、凭据隔离、资源锁、原生预检、观察器和回滚机制替换当前 fixture，而不是仅修改状态名称。

## 设计取舍和生产化路线

### 为什么当前宁可少做，也不伪装闭环

结构化 Dry-run 确定、快速、不会写生产，适合回归状态机；代价是它只能证明规则。合成 Sandbox 能让前后端和存储看到完整状态；代价是基础设施真实性为零。人工回执兼容企业已有变更平台；代价是系统依赖操作者主张。

这些取舍对课程项目是合理的，因为它把边界写进事件和报告，而不是假装有生产权限。问题不在“没有自动执行”，而在是否准确描述没有执行。

### 接入真实变更平台至少还缺六层

第一层是不可漂移的授权绑定：对规范化 Plan、参数、目标资源版本、Evidence 时间和策略版本做 hash，批准过期或目标变化即失效。

第二层是隔离执行身份：执行器使用短期、最小权限、单次有效的凭证，API 调用者的 Token 不能直接变成 Redis 管理权限。

第三层是基础设施特定的验证器：Redis 校验配置范围与资源余量，Kubernetes 分开 API 校验、调度可行和运行健康，MySQL 区分 Explain、锁风险和真实事务语义。

第四层是并发安全状态机：所有迁移使用版本号或 `WHERE status=...` 的 compare-and-set；manual result 带请求幂等键，并能从中间态恢复。

第五层是真实观察器：从受信数据源按窗口采集指标，计算阈值和趋势，区分“操作者上报成功”和“系统验证恢复”。

第六层是回滚执行和兜底：回滚也要审批边界、幂等写、目标锁和失败升级，不能因“是回滚”就降低安全要求。

此外还要通过 transactional outbox 或等价机制同步 Incident、Report 和 Trace，并建立对账告警。只有逐层补齐并提供 sandbox live、controlled fault 和 production Artifact，证据等级才可以升级。

## 面试表达：亮点是边界清楚，不是声称自动修复

### 一句话定位

AutoOnCall 把审批后的变更建模为可审计、创建幂等、默认不写生产的受限状态机，并把结构校验、人工回执与真实恢复明确分开。

### 30 秒回答

审批只代表允许继续。我用 ChangeExecution 把绑定与过期检查、确定性结构 Dry-run、人工回执或合成 Sandbox、Observation 和回滚建议分开；相同 Incident/Approval/Plan 通过数据库唯一键复用一条执行记录。生产 Sandbox 默认升级人工，系统没有 Redis/K8s/MySQL 写 adapter。当前 manual succeeded 会投影为 resolved，但它只基于人工回执，没有独立指标验证，这是我会优先修正的生产边界。

### 3 分钟展开

先讲问题：根因正确和审批通过都不能证明计划仍适用，更不能证明动作成功。直接执行会遇到目标漂移、重复请求、无回滚和观察缺失。

再讲状态机：入口绑定 Incident、Approval 和 ChangePlan；Pre-check 检查批准状态、风险、有效期及高风险回滚；Dry-run 当前只阻断不可演练步骤并生成静态预览，不调用生产。成功后可停在 dry_run_only、等待 manual record，或走合成 sandbox；生产默认 escalated。

然后讲幂等和投影：稳定 execution ID 加 SQLite/MySQL 业务唯一键避免重复创建，但后续状态迁移没有 CAS；Execution、Incident、Report 和 Trace 也不是一个事务。人工成功回执直接生成 passed Observation 并映射 resolved，所以那是操作者声明，不是监控验证。

最后讲证据：2026-07-13 dirty worktree 的 34/34 是自动化测试，没有 Artifact；2026-07-12 official Run 的 safe_change 是 8/8 offline fixture。两者都不能写成生产自动修复。

### 可写入简历的准确表述

> 设计审批后的 Safe Change 状态流，覆盖 Incident/Approval/ChangePlan 绑定、计划过期与高风险回滚门禁、无生产写的结构化 Dry-run、人工执行结果记录、合成 Sandbox、稳定执行 ID 与数据库业务唯一约束，并同步 Trace、IncidentState 和诊断报告；冻结 safe-change 离线 fixture 8/8 通过。

### 不可夸大的表述

- 不写“实现 Redis、Kubernetes、MySQL 真实 dry-run”。

- 不写“审批后自动执行、自动观察和自动回滚”。

## 高频追问与回答要点

### 1. Pre-check 和 Dry-run 分别防什么？

Pre-check 防对象错绑、未批准、风险不一致、计划过期和 high 风险无回滚；Dry-run 当前主要防步骤显式不可演练，并生成静态预览。二者都不重新查询 Redis，也不执行原生基础设施验证。

### 2. 为什么审批后还要查有效期？

审批与执行有时间差，根因、目标配置、流量和资源余量都可能变化。当前只按 Plan 创建时间判断，生产化还需目标资源版本和 Evidence 新鲜度。

### 3. 稳定 ID 如何防重复？

ID 由 Approval 与 Plan 确定性派生，数据库再以 Incident/Plan/Approval 建唯一键，冲突后返回已有记录。它防重复创建，不保证每次状态迁移 exactly-once。

### 4. 为什么 closed 不等于客观恢复？

manual succeeded 不评估 observed metrics 的阈值，只把操作者提交的状态转为 passed Observation。当前读模型虽映射成 resolved，但证据仍是人工主张；应增加独立观察器和 verified 状态。

### 5. 为什么 Sandbox 不能算预生产验证？

它没有连接任何真实依赖，只把指标填成 `sandbox_ok` 并返回固定日志。它验证代码状态流，不验证 Redis 命令、Kubernetes 调度或 MySQL 锁风险。

### 6. 人工结果接口有什么并发风险？

状态检查和保存不是 compare-and-set。并发请求可能同时读取 waiting；两阶段保存还可能在 manual_execution_recorded 中断。需要请求幂等键、状态版本和原子迁移。

### 7. 为什么失败只建议回滚？

回滚也是高风险写操作。当前没有可信观察器、生产凭据和回滚 adapter，自动回滚只会制造第二次未经充分验证的写入。

### 8. Report 与 Execution 不一致怎么办？

以 ChangeExecution 为变更状态主记录，以 Approval 为授权事实源；Report 和 Incident 是投影。记录同步失败并告警，通过 outbox、幂等消费者或对账任务重建，不能回滚已经发生的人类决定。

### 9. 34/34 和 8/8 有什么区别？

34/34 是特定 dirty worktree 的 pytest 结果，没有独立 Artifact；8/8 来自 clean commit 的 official baseline Artifact。二者 evidence level 都是 offline_fixture，区别在可复现 provenance，不在生产真实性。

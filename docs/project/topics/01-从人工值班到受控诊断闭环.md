# 从人工值班到受控诊断闭环：为什么“会回答”远远不够

> 专题编号：01  
> 审计日期：2026-07-13  
> 前置认知：[全系列事实口径与生产边界](00-全系列事实口径与生产边界.md)已经区分实现、测试与运行证据  
> 本篇事实范围：当前仓库实现、自动化测试与明确标注来源的历史运行材料；除非单独写明，不代表生产运行证据  
> 下一篇将解决：重复、恢复和乱序的外部告警怎样稳定投影为同一个故障事件

## 凌晨 02:07，告警只完成了“叫醒人”

凌晨 02:07，`order-service` 大量返回 503，P95 延迟超过 3 秒，日志持续出现 `Redis connection timeout`。监控同时发现 Redis `connected_clients` 接近 `maxclients`，部分客户端处于阻塞状态，于是发出 P1 告警。

如果系统只把告警转发到群里，它确实完成了通知，却没有完成诊断。值班工程师仍要确认告警来自哪个环境，合并重复 firing，检查错误率、延迟和流量是否同窗变化，核对 Redis 当前值与事故窗口值，定位 timeout 到底指向 Redis、网络还是其他依赖，查看最近发布和连接池配置，搜索 Runbook 与历史工单，提出候选根因，再主动寻找反证。若建议涉及调整 `maxclients`、限流或重启，还要判断谁有权批准、如何观察、失败如何回滚。最后，所有结论必须留下可交接、可复盘的记录。

这串工作有三个关键特征。第一，它有状态：后一项检查必须知道前面查过什么。第二，它有反馈：新证据会改变下一步。第三，它有停止边界：证据充分、风险待批、工具持续失败或预算耗尽时，都不能继续无界调用。

因此，本章讨论的不是“如何给告警加一个聊天框”，而是如何从人工调查的真实约束，逐步推导出一个受控诊断闭环。读完后，你应能解释为什么 RAG、工具调用、工作流、证据、审批、流式进度和读模型缺一不可，也能准确说出它们各自不负责什么。

## 从五个朴素版本，看清架构为什么不断变复杂

### 只有告警通知：知道出事，不知道是不是同一件事

第一版系统接收 Alertmanager 消息并发送通知。它解决了“没人知道”，却没有稳定调查对象。相同 Redis 问题连续触发十次时，系统可能创建十个聊天上下文；恢复后又 firing 时，也无法表达“同一事件重新打开”。调查进度、工具结果和责任人无法聚合，所谓自动化只是更快地制造消息。

### 直接询问大模型：建议听起来专业，却没有现场

第二版把告警文本交给大模型。模型会建议检查网络、连接池、`maxclients` 和慢查询，甚至写出“必要时重启”。这些回答来源于语言模式，不能告诉我们本次窗口的 `connected_clients` 是多少、503 是否与连接数同步、最近是否发布。它可以生成调查方向，不能凭空获得实时事实。

### 加入 RAG：有知识依据，仍不是本次事故事实

**RAG（检索增强生成）**通俗地说，是回答前先搜索知识库；严格地说，是从已索引文档检索相关片段，把片段和来源交给模型生成带引用回答。它负责提供内部 Runbook、官方资料和历史经验，不负责观测当前系统，也不保证检索到的历史案例就是本次根因。例如 Runbook 写“连接接近上限时检查连接池”是知识；02:07 的 Redis 是否接近上限仍要现场工具确认。AutoOnCall 使用 RAG 支持规划和建议，并保留拒答与 citation 边界。

### 接入工具：看见了数据，仍不知道下一步

第四版接入 Prometheus、日志网关、Redis、MySQL、Kubernetes、服务目录、发布记录与工单。系统终于能看现场，但“拥有工具”不等于“会诊断”：先查哪个，能否并发，超时是否重试，权限错误能否重试，结果来源是否可信，冲突时相信谁，证据何时足够，危险动作如何拦截，都没有由工具数量自动解决。

### 让模型自由循环：会继续行动，却可能永远不停

第五版让模型每轮自由思考、选工具、观察再思考。这比固定脚本灵活，却扩大了工具名漂移、重复查询、上下文膨胀、费用不可控和越权风险。同样输入的调查顺序难以回归；模型也可能在证据不足时过早结束，或把写操作包装成“检查”。

朴素方案的失败链可以压缩为：

```text
告警只通知人
→ 普通 LLM 没有现场
→ RAG 有静态知识但没有实时事实
→ 单次工具调用不会持续调查
→ 自由循环缺少契约、停止、安全和审计
```

下一步不是继续堆框架，而是先建立诊断领域的共同语言。

## 六个核心对象，分别守住六条边界

### Incident：把零散消息变成一次可协作调查

**Incident（故障事件）**通俗地说，是“一次需要被调查和处置的故障”；严格定义是由稳定身份、服务、环境、严重级别、症状、原始告警和生命周期组成的结构化诊断对象。它负责回答“我们在调查哪一次异常”，不负责证明根因，也不等于一条原始 Alertmanager 消息。

Redis 例子中，“`order-service` 在 prod 出现 P1 级 503、P95 超过 3 秒与 Redis timeout”可以成为 Incident。反例是把每次重复 firing 都当作全新事件，或把“Redis 一定是根因”写进 Incident 事实。在 AutoOnCall 中，`Incident` 是 Planner、Executor、风险控制和报告的共同输入；手工请求、demo 与告警 webhook 最终都要落到这个语义上。

### ToolCallRecord：证明调用发生过，不解释根因

**ToolCallRecord（工具调用记录）**通俗地说是一张执行回执；严格定义是记录工具名、输入摘要、状态、数据来源、耗时、错误、重试元数据和大结果引用的审计对象。它负责回答“调用了什么、是否成功、来自哪里”，不负责解释结果对根因意味着什么。

“`query_redis_status` 成功，source 为 `redis_info`，耗时若干毫秒”是 ToolCallRecord；“因此 Redis 必然是根因”不是。AutoOnCall 在 Executor 中把标准工具结果转换为调用记录，随后再提炼诊断意义。

### Evidence：事实、解释和未知必须分栏

**Evidence（证据）**通俗地说是“我们实际看到了什么”；严格定义是带 `evidence_id`、来源工具、步骤、类型、数据来源、立场、事实、推断、不确定性、下一步和置信理由的结构化观察。它负责支持、反驳或保留未知，不负责单独宣布唯一根因。

“事故窗口 `connected_clients=9940`、`maxclients=10000`、`blocked_clients=37`”是事实；“连接余量很小，支持连接耗尽”是有限推断；“当前值未必等于峰值”是不确定性。把三者混成一句“Redis 已确认耗尽”会破坏审计边界。工具未配置时，Evidence 应表达 `not_configured` 或 failed，而不是制造一份漂亮的正常数据。

### Hypothesis：排名第一仍只是待验证解释

**Hypothesis（根因假设）**通俗地说是“目前最值得验证的解释”；严格定义是由类别、支持 Evidence ID、反驳 Evidence ID、缺失证据和启发式置信信息组成的候选根因。它负责组织调查方向和排序，不负责把相关性升级为唯一因果。

“连接接近上限导致新连接 timeout 和 503”是 Redis 主假设。如果 Redis 当前和事故窗口都正常，只有某个实例出现网络抖动，它就应降权或被替代。AutoOnCall 的 Evidence Analyzer 使用规则、关键词、阈值和静态信号目录形成候选；这里的分数用于排序，不是校准后的真实概率。

### Trace 与 Snapshot：一个解释历史，一个表达最新状态

**Trace（轨迹）**通俗地说是运行流水账；严格定义是按时间保存工作流、节点、工具、风险、审批和报告事件的审计序列。它负责回答“系统做过什么、为何转向”，不负责保存可直接恢复的完整执行内存，也不能替代 Evidence 证明根因。

**Snapshot（快照）**通俗地说是某个会话在某一时刻的状态副本；严格定义是包含阶段、节点、计划、已执行步骤、Evidence、报告、审批与进度游标等最新投影。它负责回答“现在走到哪里”，不是完整事件历史，也不天然等于 durable checkpoint。

Redis 调查中的 `tool_started`、`tool_completed`、`risk_decision` 属于 Trace；等待审批前保存的当前状态属于 Snapshot。AutoOnCall 的 LangGraph checkpointer 当前是 `MemorySaver`，只承诺进程内 checkpoint；另有业务 Store 保存 latest Snapshot，但不能据此声称崩溃后从任意节点精确续跑。

### Replay：为复盘组织材料，不重演历史副作用

**Replay（回放读模型）**通俗地说是事故复盘页面的数据包；严格定义是把 IncidentState、Trace、DiagnosisReport、审批、变更记录与可用评测摘要聚合成阶段、时间线、再规划决策、证据质量、工具、审批流、变更流和报告摘要的只读视图。它负责查询和展示，不会重新调用历史工具，不是 time travel，也不是事件溯源系统的完整重放。

这六个对象建立了第一条重要原则：调查对象、调用事实、诊断证据、候选解释、运行历史和查询视图必须分开。只要混在一段聊天文本中，就无法分别测试、降级与治理。

## 从人工推理步骤推导 Plan-Execute-Replan

人工值班并不是先写一份永不变化的脚本。工程师会先形成初步检查单，拿到一批观察后再决定补查、收敛或升级。AutoOnCall 将这种过程约束为 Plan-Execute-Replan。

### Plan 负责给出初始方向，不负责授权和确认

Planner 根据用户输入、结构化 Incident、Runbook 检索结果和可用工具契约生成 `PlanStep`。一步计划必须包含稳定 `tool_name`、参数、目的、预期证据、风险、状态和重试计数。自然语言“先看看 Redis”不够；机器需要知道查哪个服务、哪个时间范围、希望看到哪些字段。

主路径使用结构化 LLM 输出；MCP 发现、知识检索或模型调用失败时，进入确定性 fallback plan。fallback 让常见 Redis/MySQL 场景仍可有限调查，不代表规则与模型能力等价。Runbook 中出现“重启”只能形成建议或审批候选，不是授权。

### Execute 负责受控取证，不重新发明目标

Executor 消费规范 `current_plan` 队列，同时维护兼容旧 SSE 的文本 `plan`。它从静态 Registry 找到工具，先做风险判断，再调用统一执行 envelope，把结果转换为 ToolCallRecord 和 Evidence，并追加到共享状态。`past_steps`、`executed_steps`、`tool_call_records`、`gathered_evidence`、`errors`、`warnings` 与 `progress_events` 使用追加语义，避免节点更新意外覆盖历史。

相邻、低风险、只读且策略允许的取证步骤可以有界 fan-out，单批最多 4 步；返回仍按计划顺序组装，一个失败只形成失败 Evidence，不取消整批。写操作、未知工具或策略拒绝会在并发批次之前截断。

### Replan 负责判断缺口和终止，不每轮回到 Planner

Replanner 默认调用确定性 Evidence Analyzer，识别成功工具、失败工具、来源质量、缺失域证据、候选假设与有限冲突，再在继续、追加步骤、工作流级重试、生成报告、请求审批、转人工之间选择。可选 LLM 只是受约束 critic，不能在证据不足时强行报告，不能凭空重试，也不能绕过风险决定。

真实图是：

```text
Planner → Executor → Replanner ──→ END
              ↑          │
              └──────────┘
```

它不是 `Planner → Executor → Replanner → Planner`。Planner 只在入口生成一次初始计划，之后由 Replanner 直接补充或替换队列。这也是它与宽泛 ReAct 描述的关键差异。

### 停止不等于成功

图在已有 `response`、出现 `pending_approval`、没有剩余计划时结束；Replanner 还有 `MAX_STEPS = 8` 硬上限。八步是成本与失控护栏，不是“八步内一定找到根因”的质量保证。工具内部瞬时重试耗尽后，工作流最多再安排受控补查；仍失败则生成 `incomplete`、`degraded`、`needs_human` 或失败结果，而不是无限循环。

用伪代码表达：

```python
state = create_state(incident)
state.plan = planner(state)
while not terminal(state):
    state += executor(state)
    decision = analyze_evidence(state)
    state = replanner(state, decision)
return build_report_with_boundaries(state)
```

这里的 `+=` 强调追加事实，而不是暗示 Python 实际实现完全相同。

## 共享状态不是一个大字典，而是节点之间的协议

初学 LangGraph 时，最容易把 State 理解成“把所有变量都塞进去”。真正重要的不是字段数量，而是谁可以写、如何合并、哪些字段代表事实、哪些字段只是兼容投影。没有这些规则，两个节点都能随意覆盖同一字段，回放时便无法判断是历史真的消失，还是某次 update 把它抹掉了。

AutoOnCall 的 `PlanExecuteState` 可以按用途分成五组。第一组是身份和输入：`session_id`、`trace_id`、`input`、`incident`，它们让一次运行与一次事故既能关联又不混为同一个 ID。一次 Incident 可以有多个诊断 session；session 标识某次运行，incident 标识业务事故，trace 标识审计链。把三者都叫 request_id，会让重试、恢复和多次诊断无法区分。

第二组是计划与执行：`current_plan` 是规范结构化队列，`plan` 是兼容旧客户端的文本队列；`executed_steps` 和 `past_steps` 分别服务新模型与旧逻辑。消费一步时应先更新 canonical 队列，再由它派生兼容队列。反过来只修改文本 `plan`，可能使 Executor 与 SSE 看见不同剩余步骤。

第三组是观察与推理：`tool_call_records` 保存调用审计，`gathered_evidence` 保存诊断意义，`hypotheses` 与 `evidence_analysis` 保存当前候选和分析。前两者必须追加，因为后一步判断需要完整历史；`evidence_analysis` 可以替换，因为它表达“基于目前全部证据的最新分析”。这种“事实追加、投影替换”的区别，是设计事件/状态系统时非常通用的原则。

第四组是治理与终态：`risk_assessment`、`pending_approval`、`change_plan`、`report`、`response`。`pending_approval` 一旦存在，图在当前轮停止；恢复时必须确认审批属于同一个 Incident，状态确实 approved，才能清除暂停标记。不能只根据客户端传来的 approval_id 继续，否则跨 Incident 的审批可能被误用。

第五组是可观察性：`progress` 表示最新进度，`progress_events` 保存追加历史，`progress_cursor` 支持客户端判断顺序；`errors` 与 `warnings` 同样追加。error 表示执行失败，warning 表示仍可产生有限结果的边界，二者不能为了让 UI 显示绿色而合并或删除。

`incident` 在图状态中保持 JSON-safe dict，而不是直接保存任意 Pydantic 对象，是为了降低 checkpointer 与序列化兼容风险。这个细节体现了工作流状态与领域模型之间的边界：API 入口可以使用强类型模型验证，进入共享状态后要保证跨节点、快照和持久化能够稳定编码。

共享状态也不能无限增长。大工具结果被移到 Artifact；可选 LLM Replanner 会裁剪 Incident、计划、历史、Evidence 和工具记录。当前上下文控制主要使用确定性字符预算，不是模型 tokenizer 的精确 token 预算，因此只能说“限制上下文膨胀”，不能声称精确控制每次模型费用。AIOps Token 历史上仍缺完整观测，也不应从字符数反算金额。

从面试角度看，这一节比“用了 StateGraph”更有分量。框架只提供合并机制，真正的工程判断是：哪些信息不可丢、哪些可以重新计算、哪个 ID 代表哪一层身份，以及暂停恢复时怎样防止跨事故串线。

## 工具工程为什么是“契约系统”而不是函数列表

默认 Registry 明确注册 10 个稳定工具：指标、日志、服务上下文、发布历史、Redis、Kubernetes、MySQL、Runbook、历史工单和处置建议。稳定名字隔离 Planner 与底层来源；Prometheus、Loki、Redis、MySQL、HTTP API 或 MCP 可以变化，但 Agent 看到的诊断语义不能随意漂移。

工具契约声明名称、描述、输入/输出 schema、风险、只读属性、总超时、重试、数据源和降级策略。当前 schema 主要用于描述、展示和局部边界，并非所有输入输出都经过统一 JSON Schema 强校验，`output` 仍允许异构载荷。

每个工具通过统一基类执行。所有 attempt 共享一个 `timeout_seconds` 总 deadline，不是每次重试都重新获得完整超时；只有 `read_only=True` 才可能自动重试，权限拒绝、不可重试返回和写操作不会盲目重试。每次 attempt 的失败类别和耗时进入 metadata。

为什么工具内部重试之外还需要工作流级重试？前者回答“同一次网络调用是否遇到瞬时错误”，依据 failure kind 和 deadline；后者回答“从整个 Incident 看，这项证据是否仍值得再查”，依据当前 Evidence 缺口。两者的决策上下文和审计单位不同。

Registry 还会在真正执行前再次运行 Policy Guard。即便未来某段代码绕过 Executor，非 `allow` 动作仍返回结构化失败。这不是重复劳动：Executor 保护工作流语义，Registry 保护最终调用边界。

日志或指标原始载荷可能很大。持久化前会脱敏、压缩；超过 `AIOPS_TOOL_OUTPUT_INLINE_BYTES` 的结果写入本地 JSON **Artifact（制品）**，状态只保留摘要、引用、ID、SHA-256、大小和截断标记。Artifact 负责防止状态和 SSE 膨胀，不代表已经实现分布式对象存储、租户隔离、统一 retention 或跨节点共享。

最重要的失败原则是：Adapter 未配置必须返回 `not_configured`，不能合成一份“看起来成功”的 Redis 数据。函数返回了 JSON，不等于拿到了生产观测；仓库内 MCP fixture 证明互操作，不证明数据来自生产。

## 从工具回执到根因，中间还要过两类质量门

工具调用成功后，结果先被规范化为 Evidence：data source 表明 provenance，stance 表明 supporting/refuting/neutral/unknown，fact 与 inference 分开，uncertainty 和 next step 保留缺口。失败工具是 unknown，不是“系统正常”的反证。

Evidence Analyzer 使用静态信号目录覆盖 Redis maxclients、MySQL 慢查询、Kubernetes CrashLoop/OOM、CPU、内存、磁盘、依赖 timeout 和 unknown-needs-human 等有限类别。它能检测几类常见冲突，例如指标异常而日志明确无异常、日志指向 Redis timeout 而 Redis 当前状态正常、Kubernetes OOM 与 MySQL 慢查询同时存在。当前只是规则、关键词与模式基线，不会自动解决采样延迟、时窗对齐、拓扑关系和通用因果方向。

形成 Hypothesis 后，报告仍不能直接宣布 completed。第一类是 **Evidence Sufficiency（证据充分性）门**：完整报告通常要同时覆盖主故障域、症状和 Runbook/历史参考。只有指标和 timeout 日志时，可以说“存在依赖异常症状”，不能确认 Redis；Runbook `no_answer_rejected` 也不能算有效参考。

第二类是 **Conclusion Alignment（结论回链）门**：根因、每条关键发现与处置建议必须连接到 Evidence ID 或 RAG citation。它能发现完全没有依据的字段并促使降级，却不能证明被连接证据在语义上必然推出全文，也不能核验扩容到具体数值一定安全。当前回链依赖 Hypothesis 支持 ID、有限术语、兼容 fallback 和已有 citation，不是逐 claim 的自然语言蕴含验证。

**DiagnosisReport（诊断报告）**因此是结构化单一事实源，包含事实、推断、候选排名、所选根因、Evidence、工具调用、Trace 摘要、风险、审批、变更计划、不确定性和状态。Markdown 从这个对象派生，而不是让模型另写一份事实。报告内还派生 Evidence Graph，把 Incident、Hypothesis、Evidence、ToolCall 与 Citation 连接成 portable dict；它是读模型，不是 Neo4j，也不执行图算法 RCA。

## 风险治理为什么必须发生在工具调用之前

如果规划结果出现 `query_redis_status`、重启服务和未审核 `DELETE FROM orders`，二元“允许/拒绝”无法表达差异。AutoOnCall 的 pre-tool policy 使用三态：`allow` 表示当前边界内可自动执行；`approval_required` 表示动作可能合理，但要创建审批并暂停；`forbidden` 表示当前 Agent 边界内禁止自动执行，不能靠审批洗白。

策略综合工具 `read_only`、工具风险、PlanStep 风险、用途、参数、Incident 环境和有限规则。Executor 第一层门禁负责写风险 Evidence、Trace、审批和用户可见状态；Registry 第二层防止调用旁路。`suggest_remediation` 只生成建议，即使文本包含“重启”也没有副作用；真正重启动作必须作为另一个步骤重新过门。

身份链与动作链也要分开。静态 token/scope 回答谁能访问诊断、读取、审批和 change API；工具策略回答某动作能否自动执行。有 approve scope 的人不能翻转 forbidden，有 allow 动作也不能绕过路由 scope。当前是轻量静态 RBAC，不是 SSO、OIDC、LDAP、动态 ABAC 或严格四眼制度。

审批请求从 `pending` 只能条件更新一次为 approved/rejected；并发决定中，后提交者不能覆盖第一终态。相同 pending 请求可通过业务幂等键复用，但这是应用层“查询再创建”，不能夸大为所有并发创建下数据库 exactly-once。

批准依然不等于生产动作已执行。当前 safe-change 链强调结构化 ChangePlan、pre-check、dry-run/sandbox 或人工结果记录；文档不能把 approved 写成“已修改 Redis”。生产自动变更还需要真实写 Adapter、资源锁、变更窗口、观察指标、阈值、幂等、自动回滚和独立安全证据。

## 长任务为什么需要 SSE、Trace、Snapshot 三种可见性

同步 HTTP 只有“请求返回”一个时刻，无法让值班员看见规划、工具、证据、风险和报告进度。AutoOnCall 的 `POST /api/aiops` 返回 SSE（Server-Sent Events）流。服务通过 LangGraph `astream(..., stream_mode="updates")` 获取 Planner、Executor、Replanner 更新；每个节点既发送 progress 事件，也发送节点业务事件，并附 Trace。进度游标使用 `session_id:000001` 这类单调格式，Snapshot 随阶段更新。

SSE 的优势是服务端单向推送简单，适合诊断进度；代价是客户端必须理解三层状态：HTTP 200 只表示流建立成功，连接正常结束只表示没有更多事件，真正业务结果要看结构化 terminal event、报告状态、`pending_approval`、errors 与 warnings。事件流捕获异常时会发送 `type=error`、`status=failed`，而不是让前端把断流误判为完成。

这也是为什么“流式输出”不能只理解为逐 token 打字效果。这里流的是工作流状态：计划形成、工具开始/完成、风险决定、报告生成和终态。模型 token 是否可见是另一个问题。

## 持久化和读模型怎样支撑交接，而不夸大恢复能力

运行时有三类持久信息。Trace 保存事件序列；业务 Store 保存 IncidentState、审批、报告、变更执行与 latest session Snapshot；LangGraph `MemorySaver` 保存进程内 checkpoint。三者职责不同，不能把“数据库里有报告”说成“图可以从任意节点恢复”。

审批后恢复会优先尝试内存 checkpoint，再使用 latest Snapshot 或 persisted Report 构造有限状态，清除 `pending_approval` 并闭合诊断/报告语义。这个 fallback 目标是让用户看到一致终态，不保证精确回到崩溃前指令指针，也不重新执行历史工具。

Incident overview 从 report、trace、approval 和 IncidentState 聚合有效状态、根因、调用链、数据来源与链接；Incident replay 进一步组合 change execution 与评测摘要，构造阶段卡片、timeline、Replanner decisions、evidence quality、tooling、approval/change flow 和 evaluation。这里采用 CQRS 风格的读模型思想：写侧对象保持各自事实，读侧按页面需求组合。它不是额外的事实源，也不应覆盖底层冲突。

一个容易忽略的细节是，Incident 列表会从 report、trace、approval、state 的 ID 并集构造。即使某次运行只留下失败 Trace 或等待审批，没有完整报告，它仍应可见，而不是从工作台消失。

## Redis 主线：每一步都回答“已知什么、为什么继续”

仓库 demo 使用 `INC-REDIS-001`：服务 `order-service`、严重级别 P1、环境 prod；症状为大量 503、P95 超过 3 秒、日志出现 Redis timeout；受控输入是 `connected_clients=9940`、`maxclients=10000`、`blocked_clients=37`。请求动作包含把 `maxclients` 调到 `12000`，并写明若连接未恢复，再由人工评估重启。它是 demo fixture，不是本文实时查询生产所得。

### 第一步：建立 Incident，而不是直接问模型

当前事实是受控告警字段；知识尚未检索；推断是“Redis 容量或客户端连接池可能耗尽”；未知包括指标同窗、日志对象、发布关联、当前值与峰值差异。系统创建 session/trace ID 和空计划、空 Evidence、空调用记录，让所有后续对象共享稳定身份。

### 第二步：用知识指导检查，但不让知识代替现场

Planner 检索 Runbook，发现应核对连接数、阻塞、应用池和回滚前提。它再安排 Redis 状态、指标、日志、服务上下文、发布和历史工单。若 LLM 或 MCP 不可用，fallback 仍组织有限只读步骤。此刻 Runbook 只是 knowledge Evidence，不能把连接耗尽写成 confirmed fact。

### 第三步：先补症状和主故障域

Redis、指标、日志等低风险相邻步骤可以最多 4 路 fan-out。Redis 工具若连接真实依赖，应保留 `redis_info` 来源；如果未配置，应返回 `not_configured`。指标要确认 P95、503、流量时间窗，日志要确认 timeout 的依赖和实例。现在已知现场影响与主域异常是否相关，仍未知是否由最近发布、连接泄漏或瞬时峰值引起。

### 第四步：让冲突改变计划

如果事故窗口连接接近上限且 blocked clients 上升，日志与服务影响同窗，Redis Hypothesis 得到支持；如果 02:15 当前值恢复为 `1200/10000`，它只反驳“当前仍异常”，不能自动推翻 02:07 峰值。Replanner 应保留时间不确定性并查询事故窗口、发布和连接池，而不是选择一条更顺眼的数据。

### 第五步：证据充分才生成报告

主域 Redis、症状指标/日志、Runbook/历史参考覆盖后，报告可以保持 completed；缺主域则 incomplete，关键工具失败则 degraded，冲突无法仲裁或缺回链则 needs_human。confirmed facts、inferred conclusions、uncertainties 分开，根因和建议连接到 Evidence ID/citation。

### 第六步：风险动作到门口就暂停

调整 `maxclients` 不是只读查询。策略创建 ApprovalRequest 与 ChangePlan，状态进入 waiting_approval，SSE 发出结构化终态。operator 能诊断不代表能审批；approver 同意也只表示计划获准进入下一阶段。当前系统不会偷偷执行 Redis `CONFIG SET`，重启也仍由人工评估。

### 第七步：用 overview 与 Replay 交接

接班人无需重读所有日志文本，可以从 overview 看有效状态、根因、审批和链接，从 Replay 看计划、工具、Evidence、Replan、报告与变更阶段。Replay 只组织已有 Artifact；如果底层没有真实观测，它不会把 fixture 变成生产事实。

MySQL 慢查询可复用“症状—主域—变更—知识”的骨架，但主域换成 slow query、连接池等待和锁；Kubernetes CrashLoop 需要 Pod/OOMKilled/restart 与日志、容量证据。当前 Kubernetes golden case 更多是离线 fixture。统一结构不代表所有故障域成熟度相同。

## 失败路径比正常演示更能说明系统是否受控

### 告警默认只入库，不自动消耗诊断资源

Alertmanager webhook 的 `auto_diagnose` 默认是 `false`。显式开启后，也只有新建或 reopened 且非 resolved 的事件才进入后台诊断。后台并发 Semaphore 为 2，同一 incident 的 in-flight 集合避免本进程重复启动。这两者都是进程内机制；多副本没有共享锁，因此不能称为分布式 exactly-once。

### 模型不可用时必须标明 fallback

Planner 模型创建、调用或结构化解析失败时使用规则计划。报告可以说“使用降级路径完成有限调查”，不能说模型成功规划。Replanner 默认确定性，因此也不能包装成“双 LLM 自主反思”。

### 工具失败要成为证据缺口

超时、权限、not_configured、未知工具都返回结构化失败，进入 ToolCallRecord、Evidence、Trace 和报告 warning。权限错误不盲重试；关键工具工作流补查仍失败时要降级。没有查询到数据和查询根本没有运行，是完全不同的语义。

### 预算耗尽必须诚实结束

达到 8 步上限或无安全步骤可补时，系统结束循环。结束只证明过程有界，不证明根因确认。报告状态、置信理由和 unknown/failed 来源必须暴露这种差异。

### 等待审批时进程重启不能冒充精确续跑

审批记录和 latest Snapshot 可以保留，`MemorySaver` 中的图指针可能丢失。恢复 fallback 可以构造一致报告，但不能保证所有节点中间状态、外部副作用和 token 上下文精确重现。

### 对外暴露还要处理部署默认值

应用启动时会检查外部绑定地址。如果绑定 `0.0.0.0`、`::` 或 `[::]`，同时关闭 API 鉴权或允许 CORS `*`，系统会告警；严格配置可 fail closed。这证明代码意识到 demo 默认值的暴露风险，但不等于完成 WAF、TLS、密钥轮换、租户隔离和企业身份治理。

## 最终架构是失败约束的集合，不是框架名录

| 现实问题 | 朴素失败 | 被迫加入的约束 | AutoOnCall 当前落点 |
|---|---|---|---|
| 重复消息无法协作 | 每条告警一段聊天 | 稳定身份与生命周期 | Incident / IncidentState |
| 模型不了解内部知识 | 泛化建议 | 可引用知识和拒答 | RAG / citation |
| 知识不是现场 | Runbook 冒充事实 | 跨源只读观测 | 10 个标准工具与 Adapter |
| 单次调用不会推进 | 固定脚本过时 | 观察后决策 | Plan-Execute-Replan |
| 工具 JSON 不能作结论 | 成功即根因 | 来源、立场、事实/推断 | ToolCallRecord / Evidence |
| 多个解释相互竞争 | 关键词投票 | 支持、反驳、缺口、冲突 | Hypothesis / Analyzer |
| 图结束不等于可信 | 有文本就 completed | 覆盖和回链双门 | Sufficiency / Alignment |
| Agent 可能越权 | Prompt 说谨慎 | 调用前硬门禁 | allow/approval/forbidden |
| 长任务不可见 | HTTP 长等待 | 状态流与终态契约 | SSE / progress / terminal |
| 崩溃与交接困难 | 只保留聊天 | 事件、最新状态、读模型分层 | Trace / Snapshot / Report / Replay |
| 改动后质量不明 | 只演示成功案例 | 指标、Artifact、provenance | candidate / official / stale 与 evidence level |

这里的“闭环”是信息与治理闭环：Incident 输入，计划，跨源取证，证据评估，风险门禁，报告，查询与反馈。它不是自动修改生产并自动回滚。

## 关键设计取舍：为什么没有追求最大自治

固定脚本最确定、成本最低，适合极封闭 Runbook；自由 Agent 适应性更强，却扩大不确定性。AutoOnCall 选择一次结构化 Planner 加确定性 Replanner，是因为当前故障域有限且需要稳定回归，代价是信号目录和规则需要人工维护。

静态 Registry 牺牲热插拔，换来稳定名称、风险和降级语义。MCP 工具可被本地包装复用，但远端发现对象不会自动进入信任边界。统一 schema 当前偏描述性，工程轻量，但缺少所有工具的通用运行时强校验。

只读自动重试与最多 4 路 fan-out 降低瞬时失败和等待时间，代价是并发只覆盖相邻安全取证，不是任意 DAG 或分布式任务队列。非只读动作不自动重试，因为“第一次成功但响应丢失”可能造成重复副作用。

SSE 比 WebSocket 更适合当前服务端单向进度，客户端实现简单；它不适合复杂双向协商，断线续传也依赖游标和读 API，而不是协议自动完成。

结构化报告比 LLM 直接 Markdown 更容易查询、测试和降级，代价是模板固定、规则维护较多。Evidence Graph 选择 portable dict 而不是图数据库，足以支持页面审计，但不提供图查询和图算法推理。

不默认自动修复是最重要的克制。诊断置信度不等于变更授权；审批也不等于具备生产写入、观察与回滚能力。当前设计把高风险动作停在人前，是能力边界而不是尚未删除的“临时限制”。

## 走向生产前，闭环还缺哪些外部条件

一个课程项目形成了信息闭环，不代表已经具备生产值班资格。生产系统首先需要可靠入口：多副本告警消费要有共享去重、幂等键、租约或队列语义，不能依赖单进程 set；还要处理乱序 resolved/firing、告警风暴、租户隔离和背压。专题 02 会继续展开入口身份与生命周期。

其次是依赖可靠性。Prometheus、Loki、Redis、MySQL、CMDB、发布和工单 Adapter 需要真实网络超时、权限、限流、凭证轮换、连接池、熔断和容量数据。一次 local-live 成功只证明当时连通过，不能证明长期 SLA；仓库 MCP mock 更不能参与生产根因确认。

再次是执行安全。若未来允许生产写入，必须把变更对象版本、目标环境、工单、审批人、有效期、幂等键、资源锁、pre-check、dry-run、观察窗口、成功阈值和 rollback 全部绑定在同一个可审计执行对象上。只增加一个 `execute=True` 开关，会把当前清晰的安全边界重新打穿。

恢复与高可用也尚未闭合。`MemorySaver` 需要被 durable checkpointer 或任务队列取代，外部副作用要有幂等和对账；Snapshot、Trace、报告、审批和变更记录需要一致的 retention、备份、恢复演练与数据迁移。即使采用数据库，也不能只凭“写入成功”声称 HA/DR。

最后是持续质量。知识库、Prompt、模型、工具契约和数据集会漂移，旧 Artifact 应被标为 stale；生产指标必须收集真实的误诊、转人工、审批绕过、恢复时间、值班反馈与未知故障，而不是把离线 1.00 长期展示为当前成绩。只有这些外部条件逐层补齐，受控诊断闭环才可能从“已实现的工程基线”走向“经过生产证明的运行能力”。

### candidate、official、stale 与 evidence level 是两条轴

`candidate` 表示可用于分析但可能来自 dirty worktree 或模块不完整；`official` 表示满足干净 commit、模块完整与无 stale 条件的正式基线；`stale` 表示当前代码、数据集、知识、配置或 Prompt 已与 Artifact 不一致。它们描述基线治理状态。

`offline_fixture`、`local_live`、`controlled_fault`、`production` 描述数据在哪种环境产生。official Artifact 仍可以主要是 offline fixture；local live 小样本也未必能晋级 official。旧结果可以解释历史，不得覆盖当前失败。完整公式、分子分母、dataset hash、RAGAS、P50/P95/P99、性能、受控故障与生产不足状态统一见[第 15 篇](15-项目量化指标与校招验收.md)，本篇不重新计算或改写其历史数字。

## 面试时把“复杂系统”讲成一条清楚因果链

### 一句话定位

> AutoOnCall 是面向限定工具和故障场景、用结构化证据、硬停止边界和人工治理约束多步调查的 Incident 诊断 Agent。

### 30 秒回答

> 它不是运维聊天机器人，而是把 Incident、RAG 知识、实时工具取证、一次 Planner 与 Executor/Replanner 循环、Evidence/Hypothesis、报告质量门、风险审批、SSE、Trace 和 Replay 串成受控闭环；证据不足会降级，高风险动作会暂停，当前不会自动执行生产修复。

### 3 分钟展开

先从凌晨 Redis 告警讲人工值班为何是有状态调查：普通 LLM 没现场，RAG 只有知识，单次工具不会续查，自由 Agent 又难停止。然后说明结构化 Incident 固定调查对象；Planner 只形成初始步骤，Executor 经 Registry 和双重风险门取证，Replanner 默认用确定性 Analyzer 补查或停止，最大 8 步。接着区分 ToolCall、Evidence 和 Hypothesis，说明报告还要过主域/症状/参考覆盖门与 conclusion alignment。最后解释 SSE 业务终态、Trace/Snapshot/Replay 的不同职责，以及审批只记录授权、不代表生产写入。量化时同时给出 13 passed 的 dirty candidate 边界和 official Run provenance，主动声明 offline fixture 不是生产准确率。

### 可写入简历的准确表述

> 基于 FastAPI 和 LangGraph 实现受控 Incident 诊断闭环，将结构化规划、10 类标准工具的跨源取证、证据驱动续查、报告充分性/回链门禁、风险审批、SSE 进度、Trace 与 Replay 读模型纳入统一工作流，并对证据缺失和高风险动作提供降级与人工接管路径。

### 不可夸大的表述

- “适用于任意故障的通用自主 RCA 平台。”
- “Agent 已自动修改生产并完成自动回滚。”
- “实现分布式 exactly-once 告警消费。”
- “任意节点崩溃后都能精确恢复。”
- “Evidence Graph 使用图算法发现根因。”
- “13 passed 或离线 1.00 就是生产准确率、MTTR 和容量证明。”

### 高频追问与回答要点

1. **为什么不用一次 LLM 调用？** 一次调用无法稳定约束实时工具、失败、来源、补查和停止；分节点后状态、契约和降级可以分别测试。
2. **RAG 为什么不能直接做 RCA？** 它提供静态知识与引用，不证明本次时间窗发生了什么；仍需症状和主故障域现场 Evidence。
3. **它与 ReAct 有什么区别？** 当前先规划一次，随后主要在 Executor/Replanner 间循环，默认续查由确定性 Analyzer 决定，不每轮回 Planner。
4. **为什么要分 ToolCall、Evidence 和 Hypothesis？** 调用事实、诊断意义和候选解释的责任不同；混在一起会把成功调用误当根因。
5. **MAX_STEPS=8 是否会漏诊？** 会，所以它是成本/安全护栏；达到上限要降级，不是宣布成功。
6. **为什么 Executor 检查后 Registry 还要检查？** 前者维护可见工作流语义，后者守最终调用入口，防未来旁路。
7. **审批通过为何不直接执行？** 授权、计划验证、真实执行、观察和回滚是不同阶段；当前没有生产写 Adapter 的完整证据链。
8. **HTTP 200 与诊断完成有何区别？** 200 只表示 SSE 建链；必须读取 terminal event 和 report status，连接结束也可能是 failed 或 waiting_approval。
9. **为什么 Trace 不能替代 Snapshot？** Trace 是历史事件，Snapshot 是最新状态；二者都不等于 LangGraph 的 durable checkpoint。
10. **Replay 会不会重新执行工具？** 不会，它是聚合已有状态、事件、报告、审批和变更的读模型。
11. **怎样证明不是演示脚本？** 主链 API、真实图节点、工具契约、Evidence、报告、审批和读模型都有自动化契约；但这仍不是生产证据。
12. **什么时候必须转人工？** 主域证据缺失或冲突、工具持续失败、预算耗尽、风险动作待批、来源不可信或系统没有安全补查步骤时。

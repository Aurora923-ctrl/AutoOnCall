# 排名第一的根因，怎样变成一份可审计报告

> 专题编号：09  
> 审计日期：2026-07-14  
> 事实范围：当前仓库实现、自动化测试与冻结评测 Artifact，不代表生产运行证据  
> 上一章已经解决：工具结果怎样成为支持、反驳或未知的 Evidence，并形成候选根因。  
> 下一章将解决：报告中的高风险建议由谁授权、怎样暂停和审批。  
> 目录计划：[AutoOnCall 专题文档目录计划](../AutoOnCall专题文档目录计划.md)

## 为什么“根因排第一”仍然不能直接写“已经确认”

继续使用贯穿全系列的 Redis 事故：`order-service` 出现 503，P95 升到 3 秒以上，日志反复出现 `Redis connection timeout`，事故窗口的 `connected_clients` 接近 `maxclients`。上一章已经可以把“Redis maxclients 或连接池耗尽”排到候选根因第一名。很多系统到这里会让 LLM 写一段漂亮 Markdown，并把标题定为“根因分析完成”。

问题是，排序只回答“在当前候选和当前规则下谁更像”，没有回答以下问题：

- Redis 主故障域事实、用户侧症状和处置参考是否都存在？
- 连接数是事故窗口值，还是当前容器刚查到的值？
- 工具失败、反驳证据和未知来源有没有被如实保留？
- 根因、关键发现和处置建议分别能回到哪些 Evidence？
- “调整到 20000 并重启”来自容量计算、Runbook 还是模型想象？
- 报告状态是 completed、degraded、incomplete、needs-human，还是 waiting-approval？

如果这些问题没有结构化答案，“有一篇报告”只是生成成功，不等于诊断可信。

### 先区分事故、观察、解释和结论单

**Incident（故障事件）**通俗地说是正在调查的事故单；严格地说，它是带 `incident_id`、服务、环境、时间窗、症状、严重级别和原始告警的稳定对象。它解决报告究竟描述哪一次异常的问题，不负责证明根因。`order-service` 在 02:07 出现 503 可以属于 Incident；把“Redis 已确认”提前写进原始 Incident 是反例。在 AutoOnCall 中，报告从共享 state 的 `incident` 字段取得标题、服务、严重级别和环境。

**Evidence（证据）**通俗地说是可以交给另一位值班员复核的观察；严格地说，它是带稳定 ID、来源工具、数据源、时间语义、立场、事实、推断、不确定性与置信说明的结构化记录。它解决结论依据从哪里来的问题，不负责单条确认唯一根因，也不授权生产动作。`connected_clients=9940/10000` 是 Evidence；“因此立即重启”不是现场事实。AutoOnCall 把它保存进 `DiagnosisReport.evidence`。

**Hypothesis（根因假设）**通俗地说是当前最值得验证的解释；严格地说，它包含候选类别、支持 Evidence ID、反驳 Evidence ID、缺失证据和启发式置信信息。它解决多条证据围绕哪个解释组织的问题，不负责宣布客观真相。排名第一的 Redis 假设仍可能缺少 Runbook 或历史依据，也可能被同窗的正常 Redis 状态反驳。在 AutoOnCall 中，它进入 `hypothesis_ranking`，第一项产生 `selected_root_cause_id` 与 category。

真正的报告必须把这三层分开：Incident 固定对象，Evidence 固定观察，Hypothesis 组织推断，报告再说明当前结论能肯定到什么程度。

### 直接让 LLM 写 Markdown 会在哪些地方失真

最朴素方案是把全部工具输出塞进 Prompt，请模型返回报告。它的优点是语言流畅，缺点是业务语义完全埋在自然语言里：某次写“待确认”，另一次写“基本确认”；失败工具可能被省略；API 很难稳定查询 `status`、`approval_status` 和 `selected_root_cause_id`；测试只能搜索字符串；前端、复盘页和审批页还会各自解析同一段 Markdown。

另一个方案是先生成 Markdown，再用正则抽取字段。这会产生两个真相来源：结构化字段说 degraded，原文标题仍写“故障已解决”；修改审批状态时只更新数据库字段，旧 Markdown 不变；模型换一种标题，抽取器就失效。

由此得到本章的第一条设计原则：报告首先是受约束的数据契约，Markdown 只是它的展示派生物。生成顺序必须是“先结构化、过质量门、再渲染和保存”，而不是“先写文章、再猜业务字段”。

## 一份报告需要三道不同的质量检查

“证据够不够”“结论有没有链接”“关系能否被查询”看似接近，实际是三个不同问题。把它们混成一个 confidence 分数，会让系统无法解释为什么降级。

### 第一扇门：Evidence Sufficiency 检查角色覆盖

**Evidence Sufficiency（证据充分性）**通俗地说是“交卷材料是否齐全”；严格地说，它是对预定义证据角色做确定性覆盖检查：是否有主故障域、现象侧和处置参考。它解决报告是否有资格保持 `completed` 的问题，不负责数学证明、因果唯一性或生产人工确认。

Redis 报告的三类角色是：

```text
主故障域：Redis 状态或等价依赖域证据
现象侧：metrics 或 logs
处置参考：Runbook 或历史工单

complete = primary_domain and symptom and reference
```

主域回答“依赖本身发生了什么”；症状回答“用户和服务看到了什么”；参考回答“为什么这样解释、处置前提是什么”。三类全有，sufficiency 才是 `complete`。主域与症状存在但缺参考，profile 状态是 `degraded`，置信上限 `0.74`；缺主域或症状，profile 状态是 `incomplete`，置信上限 `0.55`。

这套门是当前项目的业务规则，不是所有 SRE 团队唯一正确的充分性定义。对纯 CPU、内存、磁盘事件，带资源关键词的 metric 可以成为主域；对 Redis、MySQL、Kubernetes 事故，则要求相应依赖域证据。若 `query_k8s_status` 已失败，普通 metric 会从主域候选中移除，不能把“内存高”无条件冒充 Pod/OOMKilled 事实。

Runbook 检索执行成功也不一定算参考。若结果明确 `no_answer_rejected=true`，说明没有可信知识通过检索门，Sufficiency 会同时记录“处置参考”与“可信 Runbook/历史工单”缺口。拒答是边界证据，不是正向参考。

### 第二扇门：Conclusion Alignment 检查关键字段能否回链

**Conclusion Alignment（结论回链）**通俗地说是给关键结论加可核对脚注；严格地说，它检查 `root_cause`、每条 `key_findings` 和 `remediation_suggestion` 是否至少连接到 Evidence ID 或 RAG citation。它解决报告关键字段的可追踪性，不负责证明所链接证据在语义、数值和时间上必然支持全文。

根因优先使用选中 Hypothesis 的 `supporting_evidence_ids`；关键发现依据 Redis、MySQL、P95、timeout、K8s 等有限词项匹配 Evidence；处置建议优先连接 Runbook、历史工单、发布或风险 Evidence。RAG Evidence 还能提取：

```json
{
  "source_file": "redis_postmortem.pdf",
  "chunk_id": "redis-postmortem-001"
}
```

若报告请求状态仍是 `completed`，而任一必需字段无法链接，Alignment 状态变为 `needs_human_confirmation`，最终报告降为 `needs_human`；未对齐的根因、发现和建议会加“待人工确认”。如果报告在上一道门已经是 incomplete、degraded 或其他非 completed 状态，Alignment 仍被计算和展示，但当前 gate 不再二次改写状态。

必须主动说明 Alignment 的弱点。实现存在兼容 fallback：选中 Hypothesis 没有 ID 时会回退到支持性 Evidence；文本没有匹配词时也可能选择任意带 ID 的 Evidence；处置没有参考类证据时可回退到一般支持证据。因此 `aligned=true` 只能说明“存在链接”，不能说明“链接正确地蕴含这句话”。反例是把一条 P95 Evidence 链给具体扩容数值，字段形式上 aligned，容量结论仍未经证明。

升级为逐 claim 校验还需要：把段落拆成原子 claim，校验实体、数值、单位、时间窗、否定词和比较关系，再配合 NLI、规则、人工标注与错误类型评测。当前尚未实现。

### 第三层：Evidence Graph 组织关系，但不产生新事实

**Evidence Graph（证据图）**通俗地说是报告对象关系索引；严格地说，它是从 DiagnosisReport 已有 Incident、Hypothesis、Evidence、ToolCall 和 Citation 派生的 portable nodes/edges 字典。它解决前端与审计如何查询“哪条证据由哪个工具产生、支持哪个假设、引用哪段知识”的问题，不负责图算法发现根因，也不是 Neo4j。

主要节点和关系如下：

| 节点 | 关键关系 | 含义 |
|---|---|---|
| Incident | `has_hypothesis`、`investigated_by` | 事故有哪些候选解释、由哪些工具调查 |
| Hypothesis | `supported_by`、`refuted_by` | 哪些 Evidence 支持或反驳候选 |
| Evidence | `observed_for`、`produced_by`、`grounded_in` | 观察属于哪个事故、由哪个 ToolCall 产生、引用哪段知识 |
| ToolCall | 被 Incident 的 `investigated_by` 指向 | 执行工具、步骤、状态和数据源 |
| Citation | 被 Evidence 的 `grounded_in` 指向 | RAG 来源文件与 chunk |

证据图还有 selected-root-cause closure：被选中根因连接的 Evidence 中，至少要同时有 live 层和 knowledge/history 层，才标为 `closed`。要区分两个容易混淆的 closure：Evidence Quality Profile 的 closure 看全体可用 Evidence；Graph closure 只看选中根因真正链接到的支持 Evidence。全局上既有 Redis 现场也有 Runbook，不代表首选根因已经连到两者。

图是派生索引，不是新的事实源。图中存在 `supported_by` 边，只说明报告保存了这条关系；它不能反过来证明假设为真。

## 从状态到报告：顺序决定质量门是否有效

AutoOnCall 的生成器是确定性装配器，而不是让第二个 LLM 自由重写。理解顺序比记类名更重要。

### 先规范输入，再决定初始状态

生成器从 LangGraph state 读取 Incident、Evidence、ToolCall、Hypothesis、Evidence Analysis、风险、待审批、错误和 warning。ToolCall 会压缩成查询友好的字段，避免把全部外部载荷直接复制进报告。

然后建立 Evidence Quality Profile，统计类型、立场、数据源、live/knowledge/history 层、失败工具、Artifact 数、来源质量、Sufficiency 与全局 root-cause closure。若调用方请求 `status=completed`，才应用 Sufficiency 降级算法：

```text
if complete:
    completed
elif errors or failed_tools:
    degraded
elif missing_primary or missing_symptom:
    incomplete
else:
    needs_human
```

这个优先级很重要：只要存在 error 或 profile 中的 failed tool，当前实现优先降为 `degraded`，即使同时缺主域；因此不能把代码描述成“缺主域永远 incomplete”。若调用方一开始传入 waiting-approval、blocked、escalated 等状态，Sufficiency 只记录质量信息，不覆盖已有业务终态。

### 再选择根因，分开事实和推断

生成器优先采用 Evidence Analyzer 的 `hypothesis_ranking`；如果只有旧版 `hypotheses` 文本，则生成 `hyp-fallback-*` 兼容对象，并最多补入 5 条支持性 Evidence ID。第一项给出 `selected_root_cause_id`、category 和根因文本。

`confirmed_facts` 优先读取 Evidence 的 `fact`；缺少时使用 summary、来源和状态。失败 ToolCall 也会作为“调用失败事实”出现，防止报告只展示成功路径。`inferred_conclusions` 则来自 hypotheses 与 Evidence 的 inference。两者分开，值班员才能辨认“Redis 连接数接近上限”与“连接耗尽导致 503”不是同一层级。

这里也有兼容边界：旧 Evidence 只有 summary 时会进入 confirmed facts，生成器不会执行逐句事实验证。因此“confirmed”准确含义是被系统归入事实栏并保留来源，不等于生产人员签署。

### 处置建议必须与风险状态一起生成

若 state 已有明确 `remediation_suggestion`，报告沿用；有待审批时，建议写成“等待人工审批后再执行”；策略 forbidden 时写明禁止自动执行；否则根据 Redis/MySQL 等有限信号生成只读优先的确定性建议。

具体扩容数值、连接池参数或重启顺序若没有容量计算、Runbook、风险和审批依据，不应写成已授权动作。`approval_status`、`manual_action_required`、change plan 与审批 decision 都是报告字段，但报告本身不执行建议。下一章会继续处理授权边界。

### 建议、审批和执行结果为什么必须分成三层

新手常把“建议扩容 Redis”写成“已扩容 Redis”，因为两句话只差一个时态；在审计系统里，这却是三种完全不同的事实。建议是诊断系统基于当前证据提出的候选动作，审批是有权限的人或策略对风险作出的授权决定，执行结果才是外部系统实际返回的状态。前一层不能替后一层背书：报告给出正确建议，不代表审批已经通过；审批通过，也不代表命令执行成功；命令返回成功，还不代表用户侧 503 已恢复。

**Approval（审批记录）**通俗地说是“谁在什么条件下同意做什么”；严格地说，它是与 change plan 和 incident 关联、包含 decision、决策人、时间和上下文的治理对象。它解决高风险动作的责任与授权问题，不证明根因正确，也不证明动作产生预期效果。例如值班员同意临时提高 `maxclients`，只能证明该动作获准，不能把“连接泄漏”自动升级为事实。AutoOnCall 会把审批状态和决定放进报告附录与 Incident 读模型，等待审批时优先显示 `waiting-approval`。

**ChangeExecution（变更执行记录）**通俗地说是“动作到底跑了没有、返回了什么”；严格地说，它记录计划步骤、执行状态、结果与回滚相关信息。它解决授权之后的可追踪执行，不负责替监控验证业务恢复。反例是 Kubernetes API 返回滚动重启成功，但 order-service 的 P95 仍在升高：ChangeExecution 可以是成功，Incident 仍不能标记 resolved。AutoOnCall 的 Replay 可聚合这些对象，当前报告生成器则主要负责把建议、风险、审批和已知执行上下文分栏呈现。

因此，Redis 主线中的安全表达应是：“当前证据支持连接逼近上限与超时共同出现；建议先核对连接池与泄漏来源，若需调整容量则进入审批；执行后持续观察 503、P95、连接数和错误日志。”不安全的表达是：“Redis maxclients 导致故障，已自动扩容并恢复。”后一句同时越过了根因确认、授权、执行和恢复验证四道边界。即使 Markdown 更简洁，也不能用一句自然语言抹掉结构化对象之间的状态差异。

这种分层还有一个工程收益：失败能够被精确定位。若审批拒绝，系统不应把它记录成工具失败；若执行失败，系统不应把它解释为建议错误；若执行成功但指标未恢复，则需要回到 Hypothesis 和 Evidence 重新诊断，而不是重试同一动作。报告因而不是流程终点，而是诊断阶段向治理与执行阶段交接的一份带边界契约。

交接契约还意味着下游只能消费明确字段，不能从一段措辞中猜状态。通知系统可以展示摘要，工作台可以渲染时间线，审批页可以读取计划和风险，但它们都应保留同一个 Report ID、Incident ID、Evidence ID 与 Trace ID。若复制 Markdown 后重新解释“建议”“等待”和“完成”，同一事故会在不同页面产生互相矛盾的真相。结构化报告的价值，正是在多人、多页面和多次回放之间维持可核对的引用关系。

### 最后过 Alignment，派生 Graph 与 Markdown

根因、关键发现和建议形成后，生成器建立 Alignment；若 completed 路径缺回链，改成 needs-human 并降低关键文本的确定性。随后才构造 Evidence Graph 和 `DiagnosisReport`，计算 confidence、uncertainties、timeline 与 Trace summary，再从该对象渲染 Markdown。

Markdown 面向值班员的前九章依次是：故障摘要、用户影响、初步根因、关键证据、排查过程、风险动作判断、建议处置、回滚与观察指标、未确认事项。附录再展开 Evidence/Graph/Citation、ToolCall/Trace，以及风险/审批/变更。这样正文可阅读，审计信息又没有丢失。

Replanner 即便先让 LLM 生成 narrative，最终返回的 `response` 仍由 `report.markdown` 覆盖；模型创建失败时，确定性 Report Generator 仍能产出结构化报告。这一行为有自动化测试，不代表所有输入都能生成正确根因。

## DiagnosisReport 不是字段堆砌，而是一组可检查的不变量

结构化对象很容易走向另一个极端：把所有可能信息都放进几十个字段，却没有说明它们之间必须满足什么关系。真正有价值的契约应当让 API、页面和测试能检查不变量。

### 结论字段、证据字段和治理字段各自回答什么

DiagnosisReport 可以分成三组。第一组面向结论：`summary`、`root_cause`、`key_findings`、`confirmed_facts`、`inferred_conclusions`、`impact`、`remediation_suggestion`、`prevention`、`next_steps`、`status` 与 `confidence`。它回答“目前怎样解释事故，接下来做什么”。

第二组面向证据边界：`hypotheses`、`hypothesis_ranking`、`selected_root_cause_id/category`、`evidence`、`evidence_profile`、`evidence_sufficiency`、`conclusion_alignment`、`evidence_graph`、`uncertainties` 与 `warnings`。它回答“这个结论为什么能这样写、哪里仍不能肯定”。

第三组面向运行治理：`incident_id`、`trace_id`、`tool_calls`、`timeline`、`trace_summary`、`risk_summary`、`approval_status/decision`、`manual_action_required`、`change_plan` 和 `change_executions`。它回答“系统怎样走到这里、建议是否涉及人工与变更”。

分组之后可以提出至少六个不变量：

1. `selected_root_cause_id` 应当能在 `hypothesis_ranking` 中找到；
2. 根因的 supporting IDs 应当指向真实存在的 Evidence 节点；
3. `status=completed` 时，Sufficiency 应为 complete，Alignment 不应有 missing fields；
4. failed ToolCall 应当进入事实、warning、uncertainty 或充分性边界，不能静默消失；
5. Markdown 中的状态、根因、置信和审批必须来自同一个 Report 对象；
6. 报告不能因为保存成功就把 Incident 标成 resolved。

当前实现覆盖了这些不变量中的主要路径，但并非全部由 Pydantic validator 强制。例如模型允许手工构造 `status=completed` 且空 Evidence 的对象；质量门是在 Report Generator 中执行，而不是 `DiagnosisReport` 构造器本身。生产代码应尽量统一从 Generator 创建报告，直接实例化模型只适合迁移、测试或受控导入。

### 为什么 status 和 confidence 都必须保留

假设两份报告 confidence 都是 `0.55`。第一份可能是 incomplete：缺 Redis 主域；第二份可能是 degraded：K8s 工具失败但旁证足够。值班员下一步不同，前者要补关键域，后者要修工具或换可信来源。只保存数值会丢掉行为语义；只保存 status 又无法比较来源质量和候选强弱。

同理，`warnings` 与 `uncertainties` 也不能合并成一个字符串。warning 可记录执行 fallback、结构兼容或报告降级；uncertainty 面向结论边界，包括证据冲突、缺失工具、时间窗和审批。当前 Builder 会把 warning 和 error 也带入 uncertainty，方便值班员看到，但原字段仍保留，审计时可以区分来源。

### confirmed facts 为什么仍需谨慎命名

生成器优先将 Evidence.fact 放入 `confirmed_facts`，否则把 summary、数据源与状态组成事实句；failed ToolCall 也作为“调用失败”事实。这个字段解决“观测与推断分栏”，但其 confirmed 是系统分类，不是人类签字。

例如 Evidence.summary 写着“Redis capacity exhausted”，即使原始输出只有连接数，兼容路径仍可能把整句 summary 放进事实栏。要升级为更严格契约，应要求 fact 只描述原始可验证值，inference 单独承载“耗尽”，并增加 schema 或生成时检查：fact 不允许出现未经支持的因果词，数值必须在 raw output 或 Artifact 中找到。

## 手推一次双门算法，才能发现字段之间的真实依赖

只背“主域、症状、参考”不够。下面用几组最小输入手工执行当前算法。

### 输入 A：Redis、指标和工单全部成功

假设存在三条 Evidence：

```text
evd-redis: type=redis, source=redis_info, status=success
evd-metric: type=metric, source=prometheus, status=success
evd-ticket: type=ticket, source=ticket_api, status=success
```

`evd-redis` 命中 primary；`evd-metric` 命中 symptom；`evd-ticket` 命中 reference。Sufficiency 得到 complete，无 confidence cap。全局 closure 看见 live 加 history，得到 satisfied。

若选中 Hypothesis 的 supporting IDs 只有 `evd-redis` 与 `evd-metric`，Alignment 的 root cause 有回链，字段可以 aligned；但 Evidence Graph 的 selected closure 只看到 live，没有 knowledge/history，仍是 incomplete。这正是为什么全局材料齐全不等于选中根因闭环。修复方式不是在图里自动加边，而是让 Hypothesis 或 Alignment 明确把 `evd-ticket` 作为解释依据；否则报告应承认该工单只是背景材料。

### 输入 B：Redis 与指标成功，Runbook 明确拒答

Runbook Evidence 即使 `status=success`，只要 `no_answer_rejected=true`，就不会进入 reference，且会新增“可信 Runbook/历史工单处置参考”缺口。primary 与 symptom 存在，所以 profile 为 degraded、cap `0.74`。若生成器收到 requested completed，且没有 error/failed tool，它不会沿用 profile 名字直接输出 degraded，而是进入最后的“其他覆盖缺口”分支，最终 status 为 needs-human。

这是一个容易说错的细节：Evidence Sufficiency 自身的 `status=degraded` 与最终 DiagnosisReport.status 不一定同名。前者描述覆盖形态；后者还经过 Generator 的业务映射。文档和面试应分别指出两层，而不是把它们当一个枚举。

### 输入 C：Redis 工具失败，指标、日志和工单成功

failed Redis Evidence 不进入 primary，并被记录到 failed tools。指标和日志进入 symptom，工单进入 reference。Sufficiency profile 因缺 primary 得到 incomplete/cap `0.55`。但 Generator 对 requested completed 先看到 failed tools，最终 Report.status 选择 degraded。

confidence 计算会综合 Evidence 均值、Analyzer confidence 和 top Hypothesis confidence，再应用 source quality、Sufficiency cap 与 Report status cap。因此最终仍不能超过更严格的 `0.55`。这说明 status 可以是 degraded，confidence cap 却来自 incomplete Sufficiency；二者并不冲突，分别反映失败类型与证据覆盖。

### 输入 D：人为伪造 complete profile，但没有任何 Evidence

自动化测试专门构造 `evidence_profile.sufficiency.complete=true`，同时清空 Evidence 和 ToolCall。第一道门因此被绕过，随后 root cause、key findings 和 remediation 都无法获得 ID/citation。Alignment 把三项列入 missing，completed 降为 needs-human，文本加“待人工确认”。

这证明第二道门不是重复劳动：它能兜住旧数据、兼容迁移或错误 profile。反过来也要看到限制，如果存在任意 Evidence ID，宽泛 fallback 可能让 Alignment 通过，所以它不是不可绕过的语义验证。

## 持久化不是一次 INSERT，而是多个读写对象的一致性问题

报告被生成后，会同时服务报告 API、Incident overview、run history、Replay 和审批/变更更新。只把 JSON 写进一个表并不能自然保证这些视图一致。

### 为什么同时保存 Report 与 IncidentState

Report 是完整、较重的诊断事实源；IncidentState 是面向列表和生命周期的 latest projection，只保留状态、摘要、根因、审批、报告 ID、session 和 trace 等字段。保存 Report 时同步构建 IncidentState，可以让列表页不必每次解析巨大报告，也能让告警、审批和变更共享同一 incident 生命周期。

SQLite 的 `diagnosis_reports` 以 `report_id` 为主键 upsert，按 `incident_id` 与 `created_at/rowid` 获取最新报告。审批决定和 ChangeExecution 更新会在同一个 Report 上更新状态、审批字段或变更快照，再重新渲染 Markdown并保存。它解决“结构化字段与展示文本同时更新”，但不是完整版本控制：同一个 `report_id` 的旧 payload 会被覆盖。

如果生产审计要求查看每次审批前后的报告版本，应增加不可变 revision、父 revision、更新时间、操作者和 diff；不能把当前 latest Report 当作完整历史。Trace 可以说明某次更新发生过，也不保存每版全文。

### 读模型怎样避免状态回退

一次事故可能同时有 Snapshot.status=completed、Report.status=waiting-approval、Approval.status=approved、ChangeExecution.status=observing。若页面随便选一个，就会显示错误阶段。

run-status 当前优先 pending approval；若审批已有决定，再根据批准/拒绝映射；随后采用 Report.status；没有 Report 才使用 Snapshot.status。IncidentState merge 还识别 change lifecycle：已经进入 observing、resolved 或 rollback-recommended 后，旧的 report-only 更新不能把状态拉回 completed。

这种优先级本质是业务冲突解决规则，不是数据库事务自动提供的。新增状态时必须同时修改 lifecycle 集合、metadata、过滤列表、read model 与测试，否则同一个字符串在不同页面会有不同含义。

### SQLite/MySQL 接口相同，不代表生产恢复证据相同

Store Protocol 同时定义 SQLite 和 MySQL 实现，配置可选择 `AIOPS_STORAGE_BACKEND=mysql`；传入显式测试路径时固定使用 SQLite。接口一致只证明调用方可替换，不证明 MySQL 连接、迁移、并发、故障恢复、隔离级别和备份已经在生产验证。

本章自动化持久化测试主要使用临时 SQLite：能证明 Report 可保存、重新实例化后可读，以及 API 返回 latest Report；不能证明跨实例高可用或 MySQL 灾备。

## 为什么当前方案不是唯一答案

### 规则门与 LLM Judge 的取舍

确定性 Sufficiency 和 Alignment 便宜、稳定、能精确回归，适合明确的 Redis/MySQL/K8s 证据角色；代价是规则目录要维护，文本匹配容易宽泛。LLM Judge 能理解更多自然语言关系，却引入成本、漂移和自身幻觉，而且仍需要输入来源与输出约束。

更现实的组合是：规则负责硬缺口、ID 完整性、数值可定位和风险边界；NLI/LLM 只评估“这条 Evidence 是否支持这个原子 claim”；低分、冲突或高风险结论进入人工抽检。评测也要区分 link coverage、entailment precision、numeric consistency 和 reviewer agreement，不能只保留一个总分。

### Report 内嵌 Graph 与独立图存储的取舍

内嵌 portable dict 易保存、易导出、无额外服务，适合当前规模；但跨 Incident 查询、版本差异和图遍历能力有限。独立图存储适合大规模跨事故相似性和路径分析，却需要节点版本、双写一致性、权限、数据删除与重建策略。没有这些需求时，为了“架构看起来高级”引入图数据库只会扩大故障面。

### latest Report 与不可变事件流的取舍

latest Report 读取简单，适合工作台；覆盖更新会丢失旧正文。全事件溯源能重建每版状态，却要求事件 schema 版本、投影重算、幂等与副作用隔离。AutoOnCall 当前同时保留 Trace 与 latest Report/IncidentState，是成本适中的实现，但不能宣传为完整 event sourcing。

### 生产报告还缺哪些验证

除 claim-level 语义外，还缺生产值班员盲审、跨团队术语一致性、长期阈值校准、报告阅读耗时、误导性建议率、行动采纳率、修复后结果回写、版本签署、敏感数据审计和通知模板兼容。报告字段多不代表值班员能快速找到重点；真实评测应同时测准确性、边界诚实度和可操作性。

## 报告状态不是一个布尔值

“成功/失败”无法表达证据、人工和变更边界。报告至少需要区分 completed、degraded、incomplete、needs-human、waiting-approval、blocked 和后续 approval/change 状态。

### 状态与置信度回答不同问题

状态描述业务阶段，confidence 是有限启发式评分。Sufficiency 自带 `0.55`/`0.74` 上限；最终 Report Quality 还对 `incomplete=0.55`、`degraded=0.74`、`needs_human=0.62` 设上限。来源全为 Mock、not-configured、failed、manual 或未知等低质量证据时，`fallback_only` 封顶 `0.50`；真实与回退来源混合时，`mixed_with_fallback` 封顶 `0.72`。多个上限同时存在时取更严格值。

这些数值是当前工程规则，不是经过生产校准的后验概率。`confidence=0.74` 不表示“根因有 74% 概率正确”，只能表示当前规则综合 Evidence、Analyzer、来源与状态后给出的有界分数。

### “流程完成”“报告 completed”和“事故恢复”必须分开

Replanner 选择 generate-report，表示调查准备收敛；HTTP/SSE complete 表示本次流结束；`DiagnosisReport.status=completed` 表示预定义覆盖与回链门通过；Incident `resolved` 则应来自后续恢复或变更生命周期。四者不能互相替代。

报告保存后，Store 同时 upsert `DiagnosisReport` 与由它派生的 latest `IncidentState`。Store Protocol 可选择 SQLite 或 MySQL；显式测试路径固定使用 SQLite。Incident overview 和 run-status 读模型会组合 Snapshot、Report、Trace 与 Approval，而不是只相信单一表。

状态解析存在优先级：有 pending approval 先显示 waiting-approval；再看最新审批 decision；有 Report 则优先 report.status；最后才回退 Snapshot status。这防止旧运行快照把新报告状态覆盖掉。若安全变更已经进入更晚生命周期，IncidentState merge 还会阻止较旧的 report-only 状态把它拉回诊断阶段。

### 当前状态模型仍有两个必须主动暴露的缺口

第一，Report Generator 的 `manual_action_required` 当前只直接考虑 pending approval、risk need-approval、forbidden，以及 waiting-approval、blocked、escalated 等状态；Alignment 导致的 `needs_human` 并不会自动把该布尔字段设为 true。于是可能出现 `status=needs_human`、`manual_action_required=false`。调用方不能只看一个字段；更合理的后续改进是统一状态到人工动作语义并补迁移测试。

第二，`REPORT_ONLY_STATUSES` 已包含 incomplete、degraded、needs-human，但 `STATUS_METADATA` 与运行筛选列表当前没有为这三项完整配置标签、阶段、tone 和 terminal。读模型会回退到 neutral/unknown/terminal=false。代码能保存和返回这些状态，不等于前端生命周期展示已经完全治理。

## Redis 主线：五类材料怎样成为一份诚实报告

下面用完整路径和三个失败路径说明三道质量检查如何协同。

### 正常路径：现场、症状和参考同时闭合

假设报告输入包含：Redis 状态 `connected_clients=9940/10000`；Prometheus 的 P95=3250ms、5xx=8.2%；日志中的 Redis timeout；Runbook 的 maxclients 排查步骤；历史工单中的相似连接池问题。首选 Hypothesis 是“Redis maxclients 或连接池耗尽放大 timeout 与 503”。

当前已知事实是事故窗口的连接数、P95、错误率与日志；Runbook 和工单是知识/历史材料；根因仍是由这些事实支持的推断。还缺的可能是具体连接泄漏位置、当前是否恢复以及变更容量计算。报告不能把这些未知抹掉。

Sufficiency 看到 Redis 主域、指标/日志症状和 Runbook/工单参考，允许 completed。选中 Hypothesis 保存稳定 ID 和 supporting IDs。confirmed facts 只陈述观测与来源；inferred conclusions 明确写候选解释；Alignment 将根因连到 Redis、指标和日志 Evidence，将建议连到 Runbook/工单 citation。Graph 的选中根因 closure 同时包含 live 与 knowledge/history，状态为 closed。最后 Markdown 才把相同对象渲染给值班员。

“closed”仍不是生产确认。它只说明报告保存的首选根因链接同时覆盖两层。容量数值和执行动作仍需下一章的风险与审批。

### 失败路径一：只有 Redis 状态，没有现象和参考

若只剩 `query_redis_status` 一条 Evidence，主域有了，症状侧与处置参考缺失。请求 completed 会降为 incomplete，置信度不高于 `0.55`；Graph closure 因没有 knowledge/history 为 incomplete。报告应写“Redis 状态异常可能相关”，而不能写“已确认导致 503”。

为什么不直接 needs-human？当前算法在没有 tool failure/error 时，缺主域或缺症状优先 incomplete；needs-human 用于主域和症状已有、但其他完整性条件未满足的 completed 请求，或 Alignment 缺关键回链的情况。

### 失败路径二：Kubernetes 工具失败，指标和 Runbook 仍成功

假设 K8s 状态因 RBAC denied 失败，metric、log、Runbook 可用。failed Evidence 和 ToolCall 必须进入报告；主域不能由普通 metric 顶替。由于 errors/failed-tools 在报告降级算法中优先，最终可能是 degraded，而不是简单依据主域缺失写 incomplete。置信度还会受来源和状态上限控制。

这种状态允许输出“现象与可能方向”，不允许把 OOMKilled 写成已确认。生产值班员需要先修复权限或通过其他可信主域来源复核。

### 失败路径三：有链接，但链接并不真正支持句子

假设模型建议“把 maxclients 精确调到 20000”。Alignment 可能通过一般 Redis Evidence 或 Runbook citation 获得形式链接，但这些材料未必包含容量计算。报告虽然 `aligned=true`，仍应在 uncertainty 与审批边界中写明数值待确认。

这说明系统已经实现“无链接时降级”，尚未实现“有链接就验证蕴含”。面试时主动给出这个反例，比把 Alignment 宣传成防幻觉系统更可信。

MySQL 对照案例把主域换成慢 SQL、连接池等待和锁状态，并特别区分当前 `Slow_queries` 与事故窗口证据；Kubernetes 对照案例把主域换成 Pod 状态、重启原因和 OOMKilled。两者不改变“现场—症状—参考—回链—状态”的报告骨架。

## Trace、Snapshot 与 Replay 为什么不能被报告替代

报告是结论单，但长流程还需要过程、最新状态和复盘视图。

### Trace 保存过程，不替 Evidence 证明根因

**Trace（轨迹）**通俗地说是诊断流水账；严格地说，它按时间记录节点、工具、Replan、风险、审批和报告事件。它解决“系统做过什么、为何转向”的问题，不负责证明根因，也不是恢复 checkpoint。AutoOnCall 的报告 timeline 优先使用 Trace；存在既有事件且 `trace_id` 有效时，保存报告还会追加 `report_generated` 事件。没有既有事件时不会凭空创建完整历史。

### Snapshot 保存最新运行状态，不是完整报告历史

**Snapshot（快照）**通俗地说是某次 session 最新状态副本；严格地说，它保存计划、步骤、Evidence、风险、进度和可选报告等最新业务状态。它解决 run-status 恢复显示的问题，不负责保存每次报告版本，也不是跨进程任意节点续跑。读模型在没有独立最新 Report 时可以回退 snapshot.report，但独立 Report 存在时以后者为准。

### Replay 聚合已有材料，不重新生成根因

**Replay（回放读模型）**通俗地说是事故复盘页；严格地说，它聚合 Trace、Report、Approval、ChangeExecution、IncidentState 和评测摘要，派生阶段和时间线。它解决分散对象如何被阅读，不重新执行工具，不用当前 Redis 值覆盖事故窗口，也不把 stale 评测包装成现场证据。AutoOnCall 的报告和 Evidence Graph 为后续 Replay 提供材料，但 Replay 不应修改报告事实。

## Trust Gate 通过的知识，也只是报告参考

**Trust Gate（可信门禁）**通俗地说是“检索第一名也要达到绝对门槛才准进入回答”；严格地说，它依据原始向量距离或词法信号接受/拒绝 RAG 候选。它解决 Runbook 候选准入，不负责验证本次运行态 Evidence、报告 Alignment 或动作授权。

因此链路上至少有三类不同的门：Trust Gate 管知识能否进入；Sufficiency 管报告角色是否齐；Alignment 管关键字段有没有回链。Runbook 通过 Trust Gate，不等于它描述的阈值适合本次环境；Alignment 连到该 Runbook，也不等于建议获得审批。把三类门分开，才能解释失败发生在哪一层。

## 用数字检验报告闭环，而不是只统计“生成成功”

2026-07-12 的 Agent RCA 冻结集有 48 条结构化 `offline_fixture` case。与本章直接相关的结果如下，完整指标体系见[项目量化指标与校招验收](15-项目量化指标与校招验收.md)。

### 为什么要同时看五个数字

| 指标 | 定义与分母 | 冻结结果 | 能证明什么 | 不能证明什么 |
|---|---|---:|---|---|
| 报告生成 | 产生结构化 Report 的 case / 48 | `48/48 = 1.00` | 固定 case 没有漏产出 | 根因或文字正确 |
| 证据充分性命中率 | Report sufficiency 与标注边界一致的 case / 48 | `48/48 = 1.00` | 固定主域/症状/参考规则可回归 | 通用数学充分性 |
| 结论证据支持率 | 根因结论有 Evidence 回链的 case / 48 | `48/48 = 1.00`；95% CI `[0.9259, 1.0000]` | 固定 case 结论链路完整 | 链接必然语义蕴含 |
| 报告结论一致率 | Report 结论与诊断终态一致的 case / 48 | `48/48 = 1.00` | 结构化终态与报告未脱节 | 终态本身真实 |
| Trace 完整率 | 关键节点、工具、决策存在的 case / 48 | `48/48 = 1.00` | 固定审计字段完整 | Trace 能证明根因 |

## 把设计转成面试表达

### 一句话定位

AutoOnCall 先用证据角色覆盖控制报告状态，再用关键结论回链和选中根因证据图保证可追踪，最后从结构化 DiagnosisReport 派生 Markdown 与读模型。

### 30 秒回答

> 排名第一的 Hypothesis 不能直接叫 confirmed。Report Generator 先构造 Evidence Profile，要求主故障域、metrics/logs 症状和 Runbook/历史参考都覆盖；缺失时把 completed 降为 degraded、incomplete 或 needs-human。然后把 root cause、key findings 和 remediation 回链到 Evidence ID/RAG citation，再派生 portable Evidence Graph 与 Markdown。Alignment 只证明有链接，Graph 只是读模型，都不等于语义蕴含或图算法 RCA。

### 3 分钟展开思路

先说明自由 Markdown 的问题：字段不稳定、失败被省略、状态无法查询、多个展示各自解释。接着定义 DiagnosisReport 作为单一事实源，并讲三层：Sufficiency 看主域/症状/参考；Alignment 看根因/发现/建议是否回链；Graph 把 Incident、Hypothesis、Evidence、ToolCall、Citation 组织成关系。

用 Redis 案例说明事故窗口连接数、P95、日志属于现场，Runbook/工单属于知识与历史，根因是推断。三类角色齐全才允许 completed；缺症状或主域且无工具失败时 incomplete，有失败优先 degraded，无回链时 needs-human。置信还受 incomplete `0.55`、degraded `0.74`、needs-human `0.62`、fallback-only `0.50` 和 mixed `0.72` 上限约束。

最后说明 Report 先结构化再渲染，保存时同步 IncidentState，读模型按审批、报告、Snapshot 优先级解析状态。量化引用 48 条 offline fixture、报告 `48/48`、四项 `1.00`、CI 与 official provenance，并主动说明没有 claim-level 蕴含、人工盲审或生产证据。

### 可写入简历的准确表述

> 构建结构化可审计 AIOps DiagnosisReport，以主故障域/症状/处置参考三类 Evidence coverage gate 和结论到 Evidence/RAG Citation 回链约束报告状态，派生 Evidence Graph、Markdown 与 Incident 读模型；在 48 条冻结 offline-fixture case 上报告生成 `48/48`，证据充分性、结论支持、报告一致和 Trace 完整率均为 `1.00`。

### 不可夸大的表述

- 不说“排名第一就是根因确认”；
- 不说“Evidence Sufficiency 完成数学充分性证明”；
- 不说“Alignment 已逐句防幻觉”，因为存在宽泛 fallback；
- 不说“Evidence Graph 会用图算法发现根因”；
- 不说“Markdown 是独立事实源”；
- 不说“needs-human 一定同步 manual-action 字段”，当前仍有语义缺口；
- 不说“completed 等于事故恢复或人工签署”；
- 不说“48/48 是生产报告准确率”。

### 高频追问与回答要点

1. **Replanner 已说证据够了，Report 为什么还要检查？**  
   Replanner 决定流程是否收敛；Report Gate 检查最终字段是否覆盖、可回链。兼容路径、LLM fallback 和状态拼装都可能在两者之间产生缺口。

2. **Sufficiency 为什么分主域、症状和参考？**  
   主域证明依赖事实，症状证明用户影响，参考约束解释和处置。只看成功 Evidence 数量会把三条 metric 误当完整 Redis 诊断。

3. **有 failed tool 时为什么可能是 degraded，不是 incomplete？**  
   当前 gate 对 completed 请求先判断 errors/failed-tools，再判断主域/症状缺失；这是实现优先级，报告必须如实描述，不能凭直觉改写。

4. **Alignment 能防幻觉吗？**  
   能发现完全无链接的关键字段并降级，不能证明链接语义正确。还需 claim 拆分、数字/时间/否定校验、NLI 和人工标注。

5. **Evidence Graph 为什么不单独放 Neo4j？**  
   当前需求是 portable 审计索引，随 Report 保存即可；引入图库会增加一致性和运维成本，且不能自动产生因果真相。

6. **全局 closure 与 selected RCA closure 有何区别？**  
   前者看整个 Evidence 集是否有 live 与 knowledge/history；后者只看首选根因真正链接的 Evidence。报告完整更应关注后者。

7. **为什么 Markdown 必须最后生成？**  
   状态、置信和回链经过门禁后才能稳定；从结构化对象派生可保证 API、存储和页面使用同一语义。

8. **报告保存后状态如何传播？**  
   Store 保存 Report 并投影 IncidentState；overview/run read model 再合并 Approval、Report、Snapshot 和 Trace，优先展示更新的审批或报告状态。

9. **needs-human 与 manual-action-required 是一回事吗？**  
   业务上高度相关，但当前字段并未完全耦合。调用方应优先看有效 status 与审批/风险，后续应统一状态契约。

10. **Runbook 通过 Trust Gate 为什么仍不能直接授权建议？**  
    Trust Gate 只管知识候选准入；Sufficiency 管报告覆盖；Alignment 管回链；授权由风险和审批链处理。

# 观察到了异常，为什么还不能宣布根因

> 专题编号：08
>
> 审计日期：2026-07-22
>
> 事实范围：2026-07-22 当前工作区中的实现与自动化测试；它们不是生产事故或生产准确率证据。
>
> 上一章已经解决：计划中的工具怎样在契约、超时、失败、Artifact 与风险策略内执行。
>
> 下一章将解决：候选怎样经过报告充分性与结论回链，形成可交接而不过度声称的报告。
> 目录计划：[AutoOnCall 专题文档目录计划](../AutoOnCall专题文档目录计划.md)

## 四次调用成功，不等于已经找到根因

凌晨 02:07，`order-service` 出现 503，P95 升高；日志里有 `Redis connection timeout`；Redis 查询显示 `connected_clients=9950`、`maxclients=10000`。再检索到一份 Runbook，说连接接近上限时应检查连接池。把这些文本拼在一起，很容易得到一句顺滑但越界的话：**“Redis maxclients 就是根因，立即扩容。”**

这句话同时混淆了三件事：

| 层次 | 本例中的内容 | 能说明什么 | 不能说明什么 |
|---|---|---|---|
| 事实 | 某窗口内的 503、P95、日志命中、Redis 数值 | 当时观察到了什么 | 单独证明因果或唯一根因 |
| 解释 | “连接耗尽可能造成 timeout 并放大 5xx” | 一个值得验证的候选 | 已被确认的事故真相 |
| 未知 | 指标、日志、Redis 是否同一窗口；是否有发布、K8s 或下游因素 | 下一步为何还要查 | 可以被沉默地忽略 |

Runbook 和历史工单也很重要，但它们是知识或经验层：告诉我们通常怎么查、怎么处置，不能替代这一次的现场观测。反过来，02:15 的 Redis 数值已经回落为 `1200/10000`，也不能自动反驳 02:07 的峰值；它首先暴露的是**时间窗尚未对齐**。若这个“成功”结果来自 Mock、fixture、未知来源，或工具虽返回成功但没有任何有效输出，结论还应再降一层。

所以本章不把诊断描述成“工具调用后由模型宣布答案”，而是把它拆成一条可复核链：

```text
ToolExecutionResult / ToolCallRecord
        │  规范化状态、脱敏、来源、质量、Artifact
        ▼
Evidence（事实 / 推断 / 不确定性 / 立场）
        │  支持、反驳、缺口、冲突与来源质量
        ▼
RootCauseHypothesis（候选排序，而非因果确认）
        │
        ├── 追加只读取证 / 一次受控重试 / 继续调查
        └── 降级报告或转人工；第 09 篇再审查报告门禁
```

这是一套针对当前故障域的确定性、可回归基线，不是通用因果发现器。

## 先区分“系统做过什么”和“这条观察意味着什么”

Incident 规定调查对象：服务、环境、时间上下文、症状和稳定 `incident_id`。它可以说“`order-service` 在 02:07 出现 503”，不能预装“Redis 已经是根因”。在这个对象上，工具记录、Evidence 与假设各有不同职责。

| 对象 | 保存的核心内容 | 回答的问题 | 不负责的事 |
|---|---|---|---|
| `ToolCallRecord` | `trace_id`、`incident_id`、步骤、参数摘要、状态、来源、耗时、风险、真实调用身份、输出或 Artifact | 系统是否真的执行、怎么执行、在哪里失败 | 把结果解释成根因 |
| `Evidence` | 来源工具/步骤、类型、数据源、`fact`、`inference`、`uncertainty`、`stance`、置信说明及 Artifact 引用 | 这次观察支持、反驳还是尚不能判断什么 | 宣布唯一因果 |
| `RootCauseHypothesis` | 类别、支持/反驳 Evidence ID、缺失证据、排序分和原因 | 哪个解释最值得继续验证 | 把排序分当概率或确认书 |

以 Redis 查询为例，`connected_clients=9950` 与 `maxclients=10000` 是 `fact`；“它支持连接耗尽候选”是 `inference`；“仍需同窗日志、指标和真实来源交叉验证”是 `uncertainty`。这些字段被分开保存，避免报告把模型的解释重新伪装成工具原文。

大输出不会因此丢失。持久化前会做脱敏与外部原始 payload 压缩；超过内联大小的输出写成带 hash、大小和引用的 Artifact，Evidence 与调用记录保留引用。这提高了回放可审计性，却不替代原系统中的数据保留、访问控制和时间窗语义。

### 外层成功也要先过状态和质量门

工具输出不是只看 HTTP/MCP 是否返回。持久化链会先统一结果状态：结构化失败 payload 会转为 `failed`；`None`、空字符串、空列表或空字典会变成 `failed/no_data`，而不是被当成“系统正常”。显式 `stale`、`expired` 或过期质量标记会保留下来，但质量为不可用，且 Evidence 置信度最多为 `0.15`；其他不可用结果最多为 `0.10`。

因此，失败、空结果和过期结果仍值得保存：它们说明了证据缺口、重试原因或交接风险。但它们不会进入根因候选的支持/反驳计数。一个工具失败只能说明“没有拿到观测”，不能反证目标系统正常。

### 立场来自有限规则，而不是语言万能理解

`stance` 只有 `supporting`、`refuting`、`neutral` 与 `unknown`：

- 指标阈值异常可支持，明确正常可反驳，部分缺失保持中性；
- 日志的结构化空结果或“未发现”优先反驳，避免查询条件中的 `ERROR OR timeout` 自己变成异常；观测日志出现 error、timeout、5xx 等才支持；
- Redis、MySQL、K8s 有各自的异常和正常规则；Runbook、工单、风险类信息通常只是中性背景；
- failed 或不可用数据为 unknown，不把缺测偷换成反证。

这能处理仓库已编码的结构化阈值、空日志、部分指标和常见关键词，却不是通用的自然语言否定范围识别，也不能自动校准不同时钟和采样窗口。"当前正常"与"事故窗口异常"并列时，正确输出是冲突和不确定性，而不是任选一个覆盖另一个。

## 来源与时间先决定证据能不能参与推理

同一段数值的可信度取决于它从哪里来、是否新鲜、是不是实际观测。数据源会被归一成如 `prometheus`、`loki`、`redis_info`、`mysql`、`kubernetes`、`ticket_api`、`rag`、MCP 来源、Mock、规则/人工/LLM fallback 或 unknown 等标签。来源标签记录 provenance，不是对某个外部系统的生产可用性承诺。

Evidence quality 会汇总类型、立场、来源、失败工具、Artifact 数量和 live/knowledge/history 层。用于再规划的 RCA 候选会排除：失败、空、过期或标记为不可用的结果，以及 `mock`、`eval_fixture`、未配置、手工分析、LLM tool-node fallback、规则 fallback 和 unknown stance。这样既避免模拟现场冒充真实现场，也避免重复失败文本累加成“证据”。

质量画像另有保守门：如果诊断性 Evidence 全部来自 fallback 或降级来源，`source_quality=fallback_only`，Analyzer 撤销“充分”并将总体分数封顶在 `0.50`；可信与低质量来源混合时为 `mixed_with_fallback`，封顶 `0.72`。这里的 `0.50` 和 `0.72` 是代码中的控制阈值，**不是根因有 50% 或 72% 概率为真的统计校准**。

时间语义的真实边界同样必须保留：当前实现能识别明确标记为 stale/expired 的 payload 并要求重新取证；它没有实现跨数据源的自动时钟校正、采样对齐或因果时序仲裁。来源不同的多条日志也尚未做完整的相关性建模。

## 从 Evidence 矩阵生成候选，而不是从关键词直接判案

Analyzer 使用共享的静态信号目录。当前目录覆盖 Redis maxclients、MySQL 慢查询/连接池/锁等待、K8s CrashLoop、K8s OOMKilled、CPU hot loop、内存泄漏、磁盘容量、下游 timeout，以及 `unknown_needs_human` 九类候选。它不是开放领域分类器；没有被目录和规则表达的消息队列、复杂多跳链路或新型故障，不能因为文本相似就被声称已可靠识别。

对每个场景，规则先要求场景上下文或相应 Evidence，再按类别匹配支持和反驳的 Evidence ID。支持与反驳各自保留，重复持久化或重试得到的同一工具、同一参数、同一输出、同一立场会用观察指纹去重，不能靠复制一条日志把候选刷高。候选的可解释结构可读作：

```text
假设 H（Redis maxclients）
  supporting_evidence_ids = [事故窗口 Redis 饱和、timeout 日志、异常指标]
  refuting_evidence_ids  = [另一窗口 Redis 状态正常]
  missing_evidence       = [尚未成功的查询或当前规则要求的旁证]
```

当前排序函数是固定启发式：上下文命中从 `0.22`（未命中 `0.12`）起步；每条去重支持最多加到 `0.54`，每条反驳最多扣 `0.30`，每个缺失工具最多扣 `0.18`，最后裁剪到 0–1。它适合让测试稳定、理由可展示；不代表贝叶斯后验、校准置信概率，更不表示“0.76 的候选已经确认”。

排名第一也不自动变成 `selected_root_cause`。报告构建侧只会选择至少有一条直接、可用 `supporting_evidence_id` 的候选；没有支持 ID 的高分假设保持为候选，不能靠 Incident 关键词、无关的 supporting fact 或旧兼容字段继承证据。反驳 ID 也必须确实对应可用、方向正确的 Evidence，不能在图或报告中倒接。

## 冲突不是要消灭的噪声，而是继续调查或降级的理由

当前 Analyzer 显式检测三类固定冲突：指标异常而日志未发现对应 error/timeout；日志指向 Redis timeout 而 Redis 连接数状态正常；K8s OOM 与 MySQL 慢查询同时出现。检测到冲突且已有假设时，默认决策是生成待确认、`evidence_sufficient=false` 的报告，而不是继续堆叠相同查询或挑选更顺眼的证据。

这是一种安全收敛，不是自动裁决主次根因。例如“事故窗口的日志支持 Redis、15 分钟后的 Redis 反驳”可能是恢复、延迟或真冲突；现有规则不会替人做精确时序解释。报告必须展示冲突、缺失和下一步，而不是把反证静默删除。

Evidence Graph 将 Incident、候选、Evidence、工具调用及可提取 citation 连接为读模型。它只链接同一 Incident/Trace 范围内、状态成功且质量可用、立场方向一致的 Evidence；stale 或无效证据不会被接成候选支持。图中的 root-cause closure 还要求所选支持中同时有 live 层和 knowledge/history 层；这个“闭合”只是回链层覆盖检查，仍不是因果证明。

## 候选决定下一步，而不是强迫系统作答

Analyzer 输出的是有限的 `EvidenceAnalysis`：候选、排序、冲突、缺口、质量画像、总体启发式分数和建议。它按风险优先级作出再规划建议：

1. 若有可重试的失败诊断工具，最多创建一个显式 `-retry` 只读步骤；同一工具已出现 retry 且仍失败时，不无限重试。
2. 已重试/不可恢复但仍有可用候选时，可生成不完整诊断；有冲突时生成待确认报告，二者都不标记充分。
3. 规则满足当前 Redis/MySQL 或“三类成功证据且有直接支持假设”等条件时，才建议 `generate_report`；golden incident 还会额外要求 Runbook、历史或发布覆盖。它们是当前 fixture/演示域规则，不是所有事故的充分性定律。
4. 证据不足且没有剩余计划时，根据缺失工具追加注册的低风险只读步骤，例如指标、日志、Redis/MySQL/K8s、Runbook、历史或发布查询；知识与历史只提供依据，不替代现场事实。
5. 低质量来源可以收敛为低置信报告；没有安全补查、没有可用计划时转人工。

Replanner 将上述分析规范化为 `continue_investigation`、`add_steps`、`retry_failed_tool`、`request_approval`、`generate_report` 或 `escalate_to_human`。默认决定者是确定性 Analyzer。可选结构化 LLM critic 即使启用，也不能在证据不足时改成生成报告，不能在没有失败记录时凭空重试，补充步骤必须已注册并通过风险策略；重试和审批是安全优先分支，跳过 critic。模型失败或输出不合法会回退到 Analyzer。

本章的“充分”只是再规划层是否有理由停止继续取证；它不等于报告已经合格。第 09 篇会继续检查主故障域、症状、处置参考、结论—Evidence 对齐、报告状态和人工动作边界。

## 本篇不使用旧离线分数冒充当前或生产能力

此前文稿中出现过冻结 `offline_fixture` benchmark 的 Top-1、Top-3、F1 和工具执行数字。那些数字即使有历史 Artifact 和运行 provenance，也只描述当时的固定样本、固定代码与 fixture；不能被升级为 2026-07-22 当前 dirty workspace 的结果，更不能称为生产 RCA 准确率。本篇因此不把它们当作能力结论；分母、运行命令、Artifact 新鲜度和跨版本比较应由第 15 篇的量化验收统一核验。

当前可以准确说的是：实现和自动化测试覆盖了 Evidence 的 provenance/立场/质量、候选支持与反驳、重复观察去重、三类固定冲突、来源封顶、缺失工具建议、一次受控重试、无支持候选不得选为根因，以及图中 stale/反向链接过滤。没有本篇可引用的 `local_live`、`controlled_fault` 或 `production` RCA 准确率、概率校准和长期漂移证据。

## 面试时怎样讲得既具体又不夸大

一句话可以这样说：

> 我把工具回执转换成带 provenance、fact/inference/uncertainty 和 stance 的 Evidence，再用支持、反驳、缺口与质量门组织候选根因；排名只驱动补查或降级，不能替代根因确认。

常见追问的关键边界是：

- **为什么成功调用还不够？** 成功只表示调用层完成；结构化失败、空输出、过期数据、Mock 与未知来源都可能不能参与推理。
- **分数是不是概率？** 不是。它是固定加减权重的排序分，没有训练校准、独立性建模或生产标签验证。
- **为什么保留反驳 Evidence？** 它防止确认偏误，并告诉系统是补同窗数据、换一个域，还是交给人工。
- **Runbook 能不能确认根因？** 不能。它提供排查和处置参考，现场根因仍需 live Evidence。
- **为何不是所有反证都推翻候选？** 当前实现没有自动时钟对齐；反证可能来自另一窗口、采样不同或真实的竞争根因。
- **什么时候会选定候选？** 至少要有方向正确、质量可用的直接支持 ID；即使如此，它仍是报告中的受证据约束结论，而不是自动修复授权。

不可使用的说法包括：“实现了任意领域因果发现”“0.76 就是 76% 真根因概率”“离线满分证明生产准确率”“图闭合就证明因果”“报告生成就能自动扩容或重启”。

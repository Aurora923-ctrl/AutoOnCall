# 观察到了异常，为什么还不能宣布根因

> 专题编号：08  
> 审计日期：2026-07-17
> 事实范围：当前仓库实现与自动化测试，不代表生产运行证据  
> 上一章已经解决：工具怎样稳定执行、保留来源并把失败显式化。  
> 下一章将解决：候选根因怎样通过充分性与回链门禁变成可审计报告。  
> 目录计划：[AutoOnCall 专题文档目录计划](../AutoOnCall专题文档目录计划.md)

## 四个工具都调用成功，根因仍可能是错的

凌晨 02:07，`order-service` 大量返回 503，P95 延迟升高，日志连续出现 `Redis connection timeout`。Redis 状态查询又给出 `connected_clients=9950`、`maxclients=10000`，连接使用率已达 99.5%；Runbook 建议检查连接池，并在审批后考虑限流或扩容。

把四段结果直接拼进提示词，大模型很容易回答：“Redis 达到 maxclients 是根因，立即扩容。”这句话听起来完整，却一次跨过了三条不能省略的边界：

- 503、P95 与 timeout 是本次事故窗口的现象；Redis 数值是主故障域的观测；二者都属于事实层。
- Runbook 描述一般性的排查与处置方法，属于知识层，不能证明本次事故就是同一原因。
- “连接耗尽造成 503”是因果解释，“立即扩容”是带风险的处置建议，二者都不是工具原样返回的事实。

还可能存在反证。02:15 再查 Redis，当前值已回落为 `1200/10000`；或者这条成功结果来自 fixture、mock、unknown source，而不是生产适配器。若系统只看 `status=success`，就会把“调用成功”误写成“结论可信”。

更朴素的关键词投票同样会失败。“未发现 timeout”包含 timeout，却表达否定；十条来自同一日志查询的相似行，不等于十个独立来源；Redis timeout、K8s OOM 和 MySQL 慢查询同时出现时，出现次数也无法决定谁是主因、谁是伴随现象。

因此，工具结果到根因之间至少还隔着四步：把调用事实与诊断含义拆开；保留来源、事实、推断和未知；形成可被支持也可被反驳的候选；根据缺口与冲突决定继续查、降级报告还是交给人工。AutoOnCall 的 Evidence Analyzer 正是一个确定性、可回归的基础实现，而不是通用因果发现器。

## 先把四个对象分开，否则审计链一定混乱

### Incident 限定“我们正在解释哪一次异常”

**Incident（故障事件）**通俗地说，是当前正在调查的一张事故单。严格定义是：绑定服务、环境、严重级别、症状、时间上下文和稳定身份的结构化调查对象。它解决告警字段分散、重复消息和跨步骤上下文漂移；不负责证明根因。

“`order-service` 在 production 于 02:07 出现 503 与 P95 上升”属于 Incident 上下文；“一定是 Redis 连接耗尽”不属于 Incident 事实。AutoOnCall 把 Incident 和用户输入一起交给信号匹配与证据分析，golden case 还会读取 `raw_alert` 中的告警名和依赖字段来触发特定覆盖要求。

### ToolCallRecord 证明系统做过什么

**ToolCallRecord（工具调用记录）**通俗地说，是一次执行尝试的回执存根。严格定义是：记录 `trace_id`、`incident_id`、步骤、工具名、输入摘要、输出摘要或 Artifact、数据源、耗时、状态、风险、只读属性、错误，以及 `invocation_kind/actual_tool_invoked` 的审计对象。它解决“系统走了哪条执行路径、是否真的调用工具、是否失败、花了多久、结果存在哪里”；不负责解释结果支持哪个根因。

例如，“`query_redis_status` 调用成功，耗时 18 ms，来源为 `redis_info`”是 ToolCallRecord。`manual_analysis` 或未注册工具兼容路径也会形成记录，但会明确 `actual_tool_invoked=false`，不能写成真实 Redis、Prometheus 或 MCP 调用。由此直接写“Redis 是根因”仍是越界。失败记录同样有价值：它证明系统尝试过但没有拿到数据，不等于目标系统正常。

### Evidence 把一次观察变成可复核陈述

**Evidence（证据）**通俗地说，是可以拿给另一位工程师复核的一条观察。严格定义是：带稳定 `evidence_id`、来源工具、步骤、证据类型、数据源、立场、事实、有限推断、不确定性、建议下一步、原始载荷或 Artifact 引用和置信理由的结构化对象。它解决裸 JSON 缺少诊断语义和审计边界的问题；不负责单独确认唯一根因。

`connected_clients=9950, maxclients=10000` 可以写入 `fact`；“连接余量很小，支持连接耗尽假设”写入 `inference`；“当前采样可能晚于故障峰值”写入 `uncertainty`。三者必须并存，不能把推断伪装成工具原始输出。AutoOnCall 的 `related_hypothesis` 当前保存的是计划步骤的 `expected_evidence` 文本，不是稳定的 `hypothesis_id`；真正的假设回链依赖 supporting/refuting Evidence ID。

### Hypothesis 是可反驳解释，不是润色后的结论

**Hypothesis（根因假设）**通俗地说，是“目前最值得优先验证的解释”。严格定义是：带类别、支持 Evidence ID、反驳 Evidence ID、缺失工具、启发式分数和置信理由的候选根因。它解决多种解释如何组织、排序与补查；不负责把第一名自动升级为客观真相。

“Redis 达到 maxclients 或连接池耗尽，导致应用侧 timeout 与 5xx”是可验证假设。若事故窗口 Redis 连接率确实超过 90%，它得到支持；若同窗口 Redis 稳定在 12%，它应获得反驳。只写“Redis 异常”而没有可观察条件，就很难被真正证伪。

```text
Incident：我们正在解释哪一次异常？
ToolCallRecord：系统实际调用了什么？
Evidence：这次观察是什么，来源和边界在哪里？
Hypothesis：哪些观察共同支持或反驳哪一种解释？
```

四者之间最关键的连接是稳定 ID。报告若只写“根据前述监控”，审计者无法确认依据；假设保存 `supporting_evidence_ids` 与 `refuting_evidence_ids`，才能回到具体工具、步骤、原始载荷和时间语境。

## 一条工具结果怎样变成 Evidence

### 第一步：先持久化安全结果，再做诊断解释

Executor 不直接把适配器原始返回塞进状态。整个 `ToolExecutionResult` 的 `input_args`、`output`、`error_message` 和 `metadata` 先递归脱敏；外部载荷按配置压缩；超过内联大小的输出写成 JSON Artifact，状态里只保留摘要、引用、SHA-256、大小和 `truncated=true`。Evidence 与 ToolCallRecord 都引用同一份持久化结果，从而避免报告使用已脱敏 output、审计错误或 retry metadata 却泄露原始 token 的分叉。

大结果落 Artifact 只解决上下文和持久化大小，不等于长期证据仓库。当前 Artifact 是本地文件引用；生产化仍需要访问控制、保留期限、对象存储、不可篡改策略和清理流程。

### 第二步：来源必须先归一化

工具名告诉我们“调用了谁”，`data_source` 才告诉我们“数据来自哪里”。AutoOnCall 会把 Prometheus、Loki、Redis INFO、MySQL、Kubernetes、工单、RAG、MCP、mock、规则回退、未配置与失败等来源归一化。一个名为 `query_metrics` 的工具若输出来源只是通用 `metrics`，当前规则会把它视为 `not_configured`，而不是默认为真实 Prometheus。

修复后的本地 monitor/CLS MCP 会显式返回 `source=mock`、`synthetic=true`；标准指标工具在 CPU/内存全为 synthetic 时整体标为 mock，只有部分 synthetic 时标为 `mcp_monitor_mixed` 并列出 `synthetic_fields`。LangChain/MCP 的 `CallToolResult.isError`、TextContent、content block 列表、structured content 和 artifact 也会先还原成普通结构化 payload，再进入 ToolExecutionResult。于是“协议调用完成”“返回了文本块”和“远端业务成功”不再被混成同一个成功条件。

Adapter 侧也采用同一原则：HTTP 2xx 仍要检查业务 envelope 和关键字段类型；Prometheus 的 NaN/Infinity 或空 range series 变成 missing/no_data；Redis 事故 hash/Stream 只有落在请求窗口才可成为当前 Evidence；MySQL 服务端连接与应用池连接分成不同字段。Evidence 的来源归一不仅看 `source` 字符串，也依赖这些上游语义没有把失败、旧窗口和不同单位先混成 success。

来源影响 Evidence 的基础置信度：真实外部适配器为 0.82；MCP 或 RAG 为 0.72；MCP 混合合成字段为 0.60；mock 为 0.50；rule-based 为 0.65；一般失败为 0.10，`not_configured` 为 0.05。`manual_analysis` 或 `llm_toolnode_fallback` 即使成功也封顶 0.35，LLM fallback 失败则为 0.10。

这些数字是工程启发式，不是经校准的真实性概率。尤其需要注意：质量分组中 `eval_fixture` 被列入内部 trusted source 集合，只表示离线评测流程可以按既定契约使用它，不表示 fixture 等于生产真实来源。

### 第三步：事实、推断和未知分别写

如果工具返回了显式 `fact`、`inference`、`uncertainty`，Evidence 优先保留；否则系统根据摘要、立场和来源生成保守文本。Registry 输入校验失败、未知实例、Policy Guard 拦截、MCP `isError`、嵌套 `status=failed`、HTTP 2xx 业务失败、Adapter 异常和超时最终都保持 failed 语义；外层 success 但 output 为 `None`、空字符串、空集合或空字典，也会先规范化为 `failed/failure_kind=no_data`。失败或空结果的事实是“工具未返回可用数据”，推断是“不能支持根因判断”，下一步是检查参数契约、实例映射、策略、配置、网络、权限、窗口或超时后再决定是否重试，而不是伪造正常状态。

这里的关键面试点是：失败 Evidence 仍然是审计材料，但不是正向现场 Evidence。它可以证明“系统尝试过 query_logs，CLS topic lookup 失败”，不能证明“日志中没有异常”；也不能因为 raw_data 中还保留某些 error 字段，就被关键词规则提升为 supporting。失败状态先于内容语义。

success 也还要过质量分类。若 output 或 metadata 显式标记 `stale/expired/outdated`，结果保留 success 回执但被写入 `evidence_quality={status: stale, usable: false}`，Evidence 立场强制为 unknown，置信度封顶 `0.20`，下一步要求重新获取事故窗口内数据。这样“调用成功”与“数据仍可用于当前 RCA”被拆成两个判断。

Redis Evidence 的最小形态可以写成：

```json
{
  "evidence_id": "evd-redis-window",
  "source_tool": "query_redis_status",
  "evidence_type": "redis",
  "data_source": "redis_info",
  "stance": "supporting",
  "fact": "connected_clients=9950, maxclients=10000",
  "inference": "连接余量很小，支持连接耗尽假设",
  "uncertainty": "需确认采样是否覆盖 02:07 事故窗口",
  "confidence": 0.82
}
```

Runbook 与历史工单也会成为 Evidence，但它们属于知识层和历史层。Runbook 说明怎样排查与处置；工单说明过去发生过什么；二者都不能替代本次现场观测。

## supporting、refuting、neutral、unknown 到底怎样判

**Evidence stance（证据立场）**通俗地说，是这条观察对当前异常解释的方向。严格定义是四值枚举：`supporting` 表示与异常假设一致，`refuting` 表示与之不一致，`neutral` 表示只提供背景，`unknown` 表示没有足够可用数据判断。它解决“成功结果不一定支持、失败结果不等于反驳”；不负责统计显著性、因果方向和时间窗校准。

当前规则先看结构化输出，再使用受限文本模式：

- 指标中 P95、5xx、CPU/内存告警达到规则阈值时 supporting；显式 normal/ok、below threshold 或未超过阈值时 refuting。扁平 `error_rate` 使用 ratio 语义，`0.01` 表示 1%；带 `%` 的文本则按百分数语义比较。单独出现 `p95`、`5xx` 或 `high` 不再足以判异常，必须有结构化状态、数值阈值或“延迟/错误率 + 升高/超过阈值”的组合。
- 日志先识别结构化空结果：`total=0`、空日志列表或 `log_count=0` 直接 refuting；随后只在实际 summary/日志条目中查 timeout、error、5xx 等异常，不把查询条件 `ERROR OR timeout` 当成现场观察。
- Redis 的 `client_usage_ratio >= 0.9`，或 `connected_clients/maxclients >= 0.9`，以及连接耗尽、blocked、慢日志异常文本会 supporting，否则 refuting。
- K8s 的 CrashLoop、OOMKilled、NotReady、restart 等文本 supporting；Running、正常、未发现倾向 refuting；其他为 neutral。
- MySQL 有慢查询、锁等待或 `pool_waiting` 时 supporting；否则通常 neutral，而不是直接反驳。
- 服务上下文、发布记录、告警和 Trace 只在命中特定模式时 supporting，否则多为 neutral。
- Runbook、历史工单和风险记录有文本时通常 supporting，但这里的“支持”更多表示提供依据，不表示已证明现场根因。
- 工具或嵌套输出状态为 failed、结果为空，或质量标记 `usable=false` 时统一 unknown。

### 否定语义从关键词顺序修到了结构化优先，但仍不是通用语言理解

旧规则先检查 `error/timeout/5xx`，导致“0 条日志，查询条件为 ERROR OR timeout”被误判 supporting。修复后先读取 `logs.total`、entries 和 `signals.log_count`，空结果优先 refuting；异常判断只读取 summary 与真实日志条目，不再扫描 query metadata。指标也先识别 normal/ok 和 below threshold，避免“P95 正常、5xx 未超过阈值”因关键词存在而被判 supporting。

这仍不是通用否定范围解析。任意自然语言双重否定、比较对象、跨句指代和“曾异常、现已恢复”的时序语义都可能超出规则。生产适配器仍应优先输出 `match_count`、`window`、结构化 status/verdict 与单位，文本规则只做可回归的兼容层。

立场还缺少时间窗语义。02:15 的 `1200/10000` 可以反驳“当前 Redis 仍饱和”，却不能自动反驳“02:07 曾饱和”。若 Evidence 没有明确 observation window，stance 只能是粗略方向，不能代替事故时序分析。

## 假设不是模型自由发挥，而是有限信号目录的组合

AutoOnCall 默认 Evidence Analyzer 是确定性函数。它读取 Incident、输入、Evidence 和 ToolCallRecord，找成功工具、失败与重试耗尽、缺失工具、粗粒度假设、候选排名、来源质量、覆盖度和冲突，再向 Replanner 给出结构化决定。

静态信号目录当前有 9 个候选：`redis_maxclients`、`mysql_slow_query`、`k8s_crashloop`、`k8s_oomkilled`、`cpu_hot_loop`、`memory_leak`、`disk_capacity`、`dependency_timeout`、`unknown_needs_human`。后文 official benchmark 的标注集合是 8 类，不包含独立的 `memory_leak` 类；“实现目录 9 个候选”和“评测标签 8 类”不能混成同一个覆盖数字。

每个候选声明关键词、必要工具和主证据类型。上下文命中关键词，或 Evidence 的 `raw_data.output.rca_category` 显式等于该类别时，候选进入排名。若有显式 `rca_category`，该 Evidence 只链接到对应类别；否则必须同时满足“允许的 Evidence 类型 + 该类别的领域信号”。

旧实现只要 Evidence 类型等于候选主类型就直接匹配，健康 Redis Evidence 可能支持 `redis_maxclients`，任意 K8s Evidence 也可能同时关联 CrashLoop 与 OOM。修复后 `redis_maxclients` 必须命中 maxclients/connected_clients/连接耗尽，MySQL 必须命中慢查询或锁/池等待，CrashLoop 与 OOM 分开识别，普通 `memory usage normal` 不再当作 OOM；dependency timeout 还要求依赖实体与 timeout/5xx 同时出现。它仍是透明可测的领域规则，不是精确因果分类器。

## 排名分是怎样计算的，为什么不能叫概率

每个进入排名的候选按同一公式计算：

```text
base = 0.22                 # 上下文命中关键词
base = 0.12                 # 仅有证据进入、上下文未命中

score = base
      + min(0.54, supporting_count × 0.18)
      - min(0.30, refuting_count × 0.15)
      - min(0.18, missing_count × 0.06)

confidence = clamp(round(score, 2), 0.00, 1.00)
```

假设上下文提到 Redis，3 条支持证据、1 条反驳、缺 1 个工具，则分数是 `0.22 + 0.54 - 0.15 - 0.06 = 0.55`。这个 0.55 只适合候选排序和降级决策，不表示“根因有 55% 概率是真的”。分数没有使用独立性、先验概率、采样偏差和历史校准；三条同源 Evidence 也可能被当作三条支持计数。

排名按分数降序，但报告不会再机械选择第一项。Analyzer 先从排名输入中剔除 failed、empty、stale、manual、LLM fallback、rule-based 和 unknown-stance Evidence；Report Generator 再过滤不存在、立场方向不符或 `usable=false` 的 Evidence ID，只选择“至少有一条直接可用 supporting Evidence”的最高排名候选。若所有候选都只有上下文命中、反证或旧版文本，`selected_root_cause_id` 为空，根因写为证据不足。分数相同仍保留目录的稳定顺序，因此目录顺序可能影响候选展示，但不会让无支持的 Top-1 自动成为最终根因。

## “证据覆盖”不等于“证据越多越好”

**Evidence coverage（证据覆盖度）**通俗地说，是关键问题是否都有对应来源。严格定义是：从 Evidence 类型、来源、状态和层次检查主故障域、症状侧、知识/历史依据以及必要工具是否存在。它解决“只靠一种来源就收敛”；不负责判断每条内容绝对正确，也不奖励重复堆积。

当前质量画像会统计类型、立场、数据源、live/knowledge/history/other 层、失败工具、Artifact 数、平均 Evidence 置信度和来源分组。角色覆盖与根因闭环只接受 `status=success` 且 `evidence_quality.usable!=false` 的 Evidence；旧版“参考类 Evidence 即使失败，只要 stance=supporting 也算参考”的兼容口已经关闭。根因闭环至少需要一条 live Evidence，以及 Runbook 或历史依据；没有现场只有知识，不构成根因闭环；只有现场没有知识/历史，处置与经验依据仍不完整。

### Analyzer 充分性与报告充分性是两道不同门

第一道门服务于调查收敛。普通 Redis 场景若形成 Redis maxclients 假设，必须有 supporting 的 Redis 主域，并有 supporting 的 metrics 或 logs 旁证，Analyzer 才可按 0.86 基准判断能够生成报告；单纯工具 success 但结果正常、refuting、empty 或 stale 不再算覆盖。通用场景有至少 3 类可用成功工具、且排名中至少一个候选拥有直接 supporting ID 时，基准为 0.72；全是 refuting success 不会再触发充分。只有初步假设时为 0.55；尚无假设为 0.20。冲突扣 0.18，重试耗尽按数量扣分，最多扣 0.20。

第二道门服务于报告发布。它更严格地要求：主故障域 Evidence、metrics/logs 症状 Evidence、Runbook 或历史工单参考三者齐备。只有主域和症状、缺参考时，充分性状态为 degraded，置信上限 0.74；缺主域或症状时为 incomplete，上限 0.55。主域工具明确失败时，通用指标不能冒充 Redis/MySQL/K8s 主域证据；Runbook 明确 `no_answer_rejected` 也不能算参考。

所以“Analyzer 决定 generate_report”不等于报告一定是 completed。报告生成器还会把缺证据的 completed 请求降级为 degraded、incomplete 或 needs_human。这种两阶段设计允许流程诚实结束，同时阻止一个可生成的文本被包装成证据完整的结论。

### 来源质量还会再次封顶

若诊断成功 Evidence 全来自 mock、未配置、失败、规则或兜底，`source_quality=fallback_only`，Analyzer 与报告置信度封顶 0.50，并且 `evidence_sufficient=false`。若可信来源与 fallback/unknown 混合，封顶 0.72。它们是来源门禁，不覆盖报告层 0.55/0.74 的充分性上限；最终应采用更保守的约束。

内部名字 `trusted_or_unlabeled` 只表示没有检测到 fallback 组合，不是生产认证。来源标签本身仍可能写错，时间窗也可能不匹配。

## Evidence Matrix 比“证据列表”多解决了什么

把所有 Evidence 平铺成列表，只能看到系统拿到了多少条数据；矩阵还要回答每条数据处于哪个诊断角色、支持哪个候选、是否来自独立来源、时间窗是否可比较。Redis 主线可以组织成下面的工作表：

| 角色 | 观察 | 立场 | 主要回答的问题 | 不能回答的问题 |
|---|---|---|---|---|
| 症状 | P95、5xx | supporting | 用户链路是否受影响 | Redis 是否是唯一根因 |
| 症状 | Redis timeout 日志 | supporting | 调用失败是否指向缓存依赖 | timeout 是容量、网络还是客户端池造成 |
| 主故障域 | `9950/10000` | supporting | Redis 连接容量是否逼近上限 | 连接为何上升、是否覆盖峰值 |
| 知识 | Runbook | neutral/supporting | 下一步怎样确认、如何安全处置 | 本次现场是否真的满足条件 |
| 历史 | 相似工单 | neutral/supporting | 过去有哪些已知放大因素 | 本次是否与历史完全相同 |
| 上下文 | 最近发布、服务拓扑 | neutral/supporting | 为什么可能在此时、此路径发生 | 变更是否已构成因果 |

矩阵的价值不是为了表格好看，而是暴露“证据角色空洞”。假如有 20 条 timeout 日志，却没有 Redis 主域数据，症状列很满，主故障域仍为空；假如只有 Runbook 和历史工单，知识很多，live 层仍然为空。按总条数判断会把重复观察误当充分覆盖，按角色判断才会继续补主域或现场证据。

### 同源重复为什么不能简单累加

当前启发式分数按 supporting Evidence 条数增加，最多三条达到 `+0.54`。如果一次 `query_logs` 把相同错误按 Pod 拆成三条 Evidence，或者相同 Prometheus 序列被不同摘要重复保存，它们会占满支持加分，但并没有带来三个独立观察者。当前代码会去重 Evidence ID 字符串，却不会按来源、查询窗口、原始 Artifact 或事件指纹计算相关性。

生产升级时应给 Evidence 增加 `observation_start/end`、查询或事件指纹、集群与实例、采样方式和父调用 ID。打分时先在同源同窗内聚合，再对跨源一致性加分。日志量本身可以作为强度指标，但不应同时被当作独立来源数量。

### 立场属于“某条假设”，不应永久属于 Evidence

当前 `Evidence.stance` 是单一字段，便于演示与报告，却隐含一个简化：同一观察对不同候选可能方向不同。Redis 当前正常会反驳“Redis 仍饱和”，却可能支持“故障已经恢复”；一次发布记录可能支持“变更引入慢 SQL”，对“历史流量自然增长”则是反证。

更通用的数据模型应把立场放在 Evidence—Hypothesis 边上，例如 `relation(evidence_id, hypothesis_id, stance, reason)`，而 Evidence 本体只保存观察。AutoOnCall 当前通过每个 Hypothesis 分别保存 supporting/refuting ID 做了部分表达，但 Evidence 自身的单 stance 仍是粗化。理解这个边界，才能避免把 supporting 解读为“对所有候选都支持”。

### Evidence ID 稳定还不等于内容不可篡改

`evidence_id` 让对象可回链，Artifact 的 SHA-256 让大输出可以校验内容，但当前普通内联 Evidence 不一定都有独立内容哈希，ID 也不是数字签名。审计链若要求不可抵赖，还需要来源系统的时间戳、查询参数、响应摘要、签名或写入后不可变存储。课程项目实现的是可追踪引用，不是法证级证据保全。

### 后来的新数据应新增 Evidence，而不是覆盖历史

事故调查是时间序列。02:07 的 `9950/10000` 与 02:15 的 `1200/10000` 都可能真实；若后一次查询直接覆盖前一次，报告会失去“故障发生—恢复”的变化过程，也无法解释为什么两个工程师得出不同判断。更合理的做法是保留两个 Evidence ID、各自的观察时间与来源，在 hypothesis relation 中分别表达支持和反驳。

如果发现早期 Evidence 的来源标签错误，也不应静默原地修改后假装从未发生。生产审计通常要追加更正记录，指出被更正的 Evidence、原因、操作者和新对象；旧报告是否需要重新生成也应留下版本。AutoOnCall 当前状态以追加 `gathered_evidence` 为主，Trace 与报告能保存调查结果，但还没有完整实现 Evidence 修订工作流、签名或报告版本依赖追踪。这一边界决定了它适合可观察诊断与课程验证，不能直接宣传为法证审计系统。

换句话说，证据链不仅要回答“现在相信什么”，还要回答“此前为什么相信、后来因为什么改变”。没有这段变化历史，即使最终结论正确，也很难复盘调查质量、修正规则并向下一班值守人员交接。

## 冲突的价值不是选边站，而是阻止过度收敛

**Evidence conflict（证据冲突）**通俗地说，是两组观察对同一解释给出不一致方向。严格定义是：Analyzer 能识别的有限跨工具模式列表。它解决常见矛盾被报告忽略的问题；不负责通用仲裁、自动校准时钟或确定因果主次。

当前只识别三种模式：

1. 指标异常，但日志 Evidence 是 refuting。
2. 日志指向 Redis timeout，但 Redis Evidence 是 refuting。
3. 明确的 K8s OOM 信号与 MySQL 慢查询同时出现，主次不明。

第三条不再把普通 `memory` 字样视为 OOM。只有 `OOM/OOMKilled/out of memory/memory limit exceeded/内存溢出` 等明确模式才进入冲突，避免“memory usage normal; mysql slow query”制造假冲突。

命中冲突且已有假设时，Analyzer 会直接选择 `generate_report`，但 `evidence_sufficient=false`，理由写成“生成待确认报告”，并降低置信度。它不是继续无限补查，也不会凭目录顺序强行挑一个赢家。

Redis 例子中，02:07 日志支持 timeout，02:15 当前 Redis 为 `1200/10000`。最合理的表述是：“事故窗口日志指向 Redis，当前状态未复现连接饱和；可能已恢复，也可能存在网络或客户端连接池因素。”若没有统一时间窗，不能写“Redis 正常，所以日志错误”，也不能写“日志有 timeout，所以当前 Redis 仍打满”。

这三条模式之外的冲突不会被自动发现。多集群拓扑、采样延迟、指标聚合、日志丢失、级联故障、共同上游和观察者偏差都需要更丰富的时序和依赖模型。

## 缺什么 Evidence，应直接变成下一步

Analyzer 不只输出排名，还把缺失工具转成低风险只读 PlanStep：缺指标补 `query_metrics`，缺日志补 `query_logs`；上下文指向 Redis、MySQL、K8s 时分别补主域状态；Redis golden case 还补 Runbook 和历史工单；MySQL golden case还要求发布历史、Runbook 和历史工单。

这里的 golden 规则是课程与面试演示基线的特殊约束，不是任意 Incident 的通用充分性定理。它刻意要求 Redis 主线展示“现场 + 知识 + 历史”，要求 MySQL 主线展示发布关联，以稳定离线回归和教学叙事。

决策优先级如下：

```text
存在首次失败工具
→ 只生成一个 retry step，step_id 追加 -retry

同一工具 retry 已失败且已有可用现场 Evidence 支持的假设
→ 不再重试，生成降级不完整报告

有冲突且已有假设
→ 生成待确认报告

证据充分
→ 生成报告

证据不足、计划为空、还有安全缺失步骤
→ add_steps

只有 fallback 背景、计划为空且形成候选
→ 可终止为低置信报告，但不选择无直接支持的根因

仍有原计划
→ continue_investigation

无足够证据、无安全步骤
→ escalate_to_human
```

“只重试一次”是通过失败 step ID 的 `-retry` 后缀和工具去重实现的工作流约束，不是跨所有历史运行的持久化重试预算。Replanner 总步骤还受 8 步上限保护。

可选 LLM critic 可以对部分 ReplanDecision 提建议，但 `retry_failed_tool` 与审批类安全决策跳过 LLM；模型不能在证据不足时强行 `generate_report`，也不能添加未注册、非只读或风险策略不允许的步骤。默认 Analyzer 仍是确定性基线。

## Redis 主线：五条 Evidence 怎样改变我们知道的事

### 第一阶段：只有告警与症状

`evd-metric` 记录 P95 与 5xx 上升，`evd-log` 记录 Redis timeout。此时已知的是请求链受影响、日志提到 Redis；连接耗尽、网络抖动、客户端池堵塞都仍可能成立。若直接宣布 maxclients，属于从相关性跳到因果。

下一步查 Redis 主域，因为它能区分“日志指向 Redis”与“Redis 容量确实异常”。若 Redis 工具未配置，Evidence stance 必须是 unknown，置信只有 0.05，系统应记录证据缺口而不是用 Runbook 填空。

### 第二阶段：主故障域加入

`evd-redis` 显示事故窗口 `connected_clients=9950`、`maxclients=10000`，可能还包括 `blocked_clients`。这条 Evidence 与指标、日志共同支持 `redis_maxclients`，候选排名上升。当前已经有现场事实链，但仍缺少处置依据和历史相似性。

若 Redis 返回的是当前 `1200/10000`，则它对“当前饱和”是 refuting，并触发日志—Redis 冲突。正确动作是保留两个时间点，而不是删除其中一个。

### 第三阶段：知识与历史补齐解释边界

`evd-runbook` 给出容量确认、审批和回滚条件；`evd-ticket` 表示历史事故曾被连接池配置放大。它们帮助回答“还应查什么”和“怎样安全处置”，但历史相似不证明本次相同，Runbook 命中也不证明当前 Redis 打满。

五类覆盖满足 Redis golden case 后，Analyzer 可以收敛；报告仍要保留以下未知：采样是否完全覆盖事故峰值，连接数上升是泄漏、配置还是流量导致，当前恢复是否由外部动作触发，扩容是否具备审批与资源前置条件。

### 第四阶段：根因与处置必须继续分开

“Redis 连接逼近 maxclients 是首选根因”可以由 hypothesis ranking 与 Evidence ID 支持；“将 maxclients 调到 12000”仍是 ChangePlan 建议，需要风险策略、审批、pre-check、观察和回滚。Evidence Analyzer 只负责解释与补查，不负责把诊断置信度转换成生产写权限。

MySQL 对照中，主域是慢查询、连接池等待或锁等待，发布记录只提高或降低关联假设，不能单独证明根因。Kubernetes 对照中，CrashLoop/OOMKilled 需与内存指标和日志交叉验证；没有真实 Adapter 结果时，fixture 不得写成生产现场 Evidence。

## 真正能检验假设的不是更多支持，而是反事实问题

如果只问“还有什么支持 Redis”，系统会不断收集同方向信息。更好的调查问题是：“假如 Redis 连接耗尽不是根因，我们应该看到什么？”这就是反事实思维：为候选设计能使它降权或被排除的观察。

### Redis 假设怎样被证伪

对“Redis maxclients 导致 503”至少可以设计四类反事实：事故窗口连接使用率始终远低于 90%；timeout 只集中在一个应用实例而 Redis 端没有拒绝；同一时刻非 Redis 路径也同样 503；连接回落后 P95 与错误率仍未恢复。任何一条都不必单独判死刑，但应进入 refuting IDs，推动网络、线程池、下游依赖或应用发布候选上升。

反过来，只有当前时刻 Redis 正常并不足以证伪事故窗口假设。时间窗必须与 Incident 对齐：`observed_at` 晚于恢复动作时，这条 Evidence 更适合描述“已恢复”而不是“从未发生”。当前实现主要靠摘要和有限 conflict 记录这一区别，没有通用窗口模型。

### MySQL 与 K8s 对照怎样防止相关性冒充因果

MySQL 慢查询与发布时间接近，只能建立时间关联。更强反事实包括：回滚 Feature Flag 后 SQL 延迟是否恢复；相同 SQL 在无发布实例上是否正常；连接池等待是否先于接口 P95 上升。K8s OOMKilled 则要区分根因与结果：如果应用因下游 timeout 堆积请求导致内存上涨，OOM 可能是次生故障；如果 RSS 先持续增长、OOM 后实例容量下降，内存泄漏候选更强。

当前静态目录不会自动完成这种先后关系推断。它能把 OOM、慢 SQL、timeout 放入候选并提示有限冲突，但事件顺序、共同原因和中介变量仍需人工或更强时序模型。

### 负例为什么比继续加正例更能暴露规则缺陷

一个候选的正例只证明规则能在理想输入上命中；难负例才检验它会不会乱报。Redis 至少应有这些负例：文本出现“Redis”但只是依赖清单；日志返回 0 条且 query 含 `ERROR OR timeout`；当前连接率 12% 且事故窗指标也正常；Runbook 命中 maxclients 但没有任何 live Evidence；工具失败消息包含 Redis 和 timeout。

每个负例都应断言 stance、ranking、missing Evidence、report status 和 needs-human，而不只断言候选字符串是否存在。现有测试已覆盖失败为 unknown、未知来源封顶、正常指标 refuting、空日志不读取 query metadata、Evidence 类型不能直接匹配无关类别、普通 memory 不产生 OOM 冲突、Runbook no-answer 不计参考等边界。任意自然语言否定范围和跨句时序仍应继续补难例。

### 什么时候应停止自动补查

反事实并不意味着无限穷举。系统应在三种情况下停止：关键工具已经按策略重试一次仍失败；现有候选冲突无法通过注册的低风险只读工具解决；继续补查的成本或时间超过 Incident 允许预算。此时正确输出是待确认或 incomplete 报告，列出最能区分候选的人工检查，而不是为了得到 completed 状态继续调用无关工具。

## 从假设到报告，还要通过回链门禁

报告生成器优先使用 Analyzer 的结构化 `hypothesis_ranking`，选择第一名的 `hypothesis_id`、类别与文本作为首选根因。若旧状态只有字符串 hypotheses，兼容路径会生成 0.45 的 fallback ranking；若某个排名项没有 supporting IDs，还可能回填最多 5 个当前 supporting Evidence ID。

这个兼容行为保证旧数据可显示，却不是新的因果证明。回填可能把宽泛的 supporting Evidence 关联到假设，所以真正严格的链路应由 Analyzer 在形成候选时就提供准确 supporting/refuting IDs。

随后进行 **Conclusion Alignment（结论回链）**：通俗地说，是检查报告中的根因、关键发现和处置建议能否回到 Evidence ID 或 RAG citation；严格定义是三个必需字段的结构化映射门禁。它解决报告文字与证据矩阵脱节；不负责验证 Evidence 本身是否真实。

根因从 selected hypothesis 的 supporting IDs 回链，关键发现按文本与 Evidence 匹配，处置建议优先链接 Runbook citation 或相关 Evidence。任何必需字段缺少链接时，requested `completed` 会降为 `needs_human`，未对齐文本加“待人工确认”。报告还生成 Evidence Graph，把 Incident、假设、Evidence、工具调用和 citation 连接起来，供查询与复盘使用。

因此，排名第一只是候选选择；Evidence 充分性决定报告能否完整；Conclusion Alignment 决定报告的每个关键结论能否被定位。这三层任何一层失败，都不应输出无条件的“根因已确认”。

## 量化指标：满分必须带上分母和样本结构

2026-07-12 的 official benchmark 中，Agent RCA 使用 48 条 `offline_fixture` case，标注 8 类根因。类别 support 分别为：`cpu_hot_loop=3`、`dependency_timeout=15`、`disk_capacity=3`、`k8s_crashloop=3`、`k8s_oomkilled=6`、`mysql_slow_query=3`、`redis_maxclients=6`、`unknown_needs_human=9`，合计 48。类别明显不均衡，所以必须同时看 Macro 指标。

| 指标 | 结果与分母 | 为什么需要 / 能证明什么 |
|---|---:|---|
| Top-1 Accuracy | `48/48 = 1.00`，95% CI `[0.9259, 1.0000]` | 首选类别在冻结 case 中全部命中；不能证明未知生产故障仍命中 |
| Top-3 Recall | `48/48 = 1.00`，95% CI `[0.9259, 1.0000]` | 正确类别都进入前三；不能证明前三候选之间已完成因果仲裁 |
| Macro Precision / Recall / F1 | `1.00 / 1.00 / 1.00`，8 类等权 | 防止 15 条 dependency 类掩盖只有 3 条的少数类；每类样本仍很少 |
| 必要证据召回率 | `48/48 = 1.00`，95% CI `[0.9259, 1.0000]` | 标注要求的 Evidence 工具均被覆盖；不证明工具返回内容一定真实 |
| 结论证据支持率 | `48/48 = 1.00`，95% CI `[0.9259, 1.0000]` | 报告假设有 Evidence ID 回链；不证明回链内容构成真实因果 |
| `needs_human` Precision / Recall / F1 | `1.00 / 1.00 / 1.00` | `tp=3, fp=0, fn=0, tn=45`，即 3 个正例、45 个反例；正例分母很小 |
| Replan Precision / Recall / F1 | `1.00 / 1.00 / 1.00` | `tp=12, fp=0, fn=0, tn=36`；证明冻结补查标注命中，不证明任意领域补查 |

Artifact 还记录工具选择 `tp=276, fp=24, fn=0`，Precision `0.92`、Recall `1.00`、F1 `0.9583`；实际执行 `267` 次，其中成功 `258` 次，执行成功率 `258/267 = 0.9663`，invalid tool rate `0.00`。这些指标解释了“根因分类满分”并不等于工具选择完全无冗余或所有工具都成功。

### 指标公式怎样对应误诊风险

Top-1 Accuracy 的分母是 48 个 Incident，分子是首选类别等于标注类别的 48 个；Top-3 Recall 的分母同样是 48，分子是标注类别进入前三的 48 个。它们都按 Incident 计数，不按 Evidence 条数计数。

对某个根因类别，Precision 是 `TP/(TP+FP)`，回答“系统判成这一类的 case 中有多少正确”；Recall 是 `TP/(TP+FN)`，回答“真实属于这一类的 case 中找回多少”；F1 是二者调和平均。Macro Precision/Recall/F1 先对 8 类分别计算再等权平均，所以 support=3 的 CPU 与 support=15 的 dependency 对最终 Macro 贡献相同。

`needs_human` 是另一个二分类任务。这里 3 个应转人工正例全部命中，没有把 45 个反例误判为转人工，因此 `tp=3, fp=0, fn=0, tn=45`。但正例只有 3 个，无法覆盖真实世界中权限不足、跨区域故障、观测矛盾和未知新类型等丰富原因。只写 F1=1.00 而不写 3/45，会掩盖这个小分母。

同理，Replan 的 12 个正例与 36 个反例用于验证该不该补查；它不评估补出的每个工具是否成本最优。工具选择 Precision 0.92 与 `fp=24` 提醒我们：最终 RCA 满分时仍存在冗余工具，分类指标与执行效率指标不能互相替代。

Top-1 检验首选，Top-3 检验候选覆盖，Macro 检验类别公平，必要证据召回和结论支持率防止“猜对但无依据”，`needs_human` 防止系统永远强答。`48/48` 仅说明确定性 Analyzer 在冻结 Redis、MySQL、K8s 和通用 fixture 上满足回归契约，不能证明未知根因、跨系统因果链、时间漂移或生产噪声下仍保持 1.00。

## 设计取舍：透明规则不是唯一答案

### 为什么不把全部结果直接交给 LLM

LLM 能处理复杂语言，却容易在来源、否定、冲突和停止条件上漂移，同一输入也未必给出同一证据矩阵。当前先用确定性规则形成 EvidenceAnalysis，使重试、补查、来源封顶和报告状态可以单元测试；可选模型只在受约束范围内提供 critic。这牺牲开放领域能力，换取可解释和回归稳定。

### 为什么不只用关键词

纯关键词无法可靠理解否定、时间和同源相关性。当前规则已经加入结构化空结果/normal 优先、ratio 与百分数单位、类别特异关键词、来源、类型、stance 和 explicit `rca_category`，但它仍不是语义解析器。升级方向可以是结构化适配器 verdict、通用否定范围识别、时间窗对齐、依赖图和经过标注数据训练的排序模型。

### 为什么保留 refuting Evidence

只保存支持证据会形成确认偏误，排名永远越查越高。反驳 ID 让报告展示“为什么还不能确定”，也让后续工具选择针对冲突。反证不是失败数据；它可能说明假设错误、现象已恢复、采样窗口不一致或存在级联故障。

### 为什么允许不完整报告

真实依赖可能持续失败。无限重试比输出不完整报告更糟：它消耗预算、延迟人工接管，还可能在上下文中堆积重复错误。当前单次重试后可生成 degraded/incomplete/needs_human，让已知、未知和失败工具一起交接。停止不等于成功。

### 生产边界必须说到哪里

当前已实现结构化 Evidence、静态候选、启发式排名、三种冲突、来源质量、缺失步骤、一次重试、报告充分性与回链门禁。自动化测试验证了这些契约，official Artifact 验证了冻结 fixture。尚未实现或没有证据的包括：通用因果发现、置信概率校准、自动时钟与采样对齐、同源 Evidence 相关性降权、任意领域信号发现、生产根因准确率和长期漂移监控。

## 面试时如何把“根因分析”讲准确

### 一句话定位

> 本章把异构工具回执转成保留事实、推断、来源、立场和缺口的 Evidence，再用可反驳假设驱动补查与降级收敛。

### 30 秒回答

> ToolCallRecord 负责执行审计并区分真实工具调用与 analysis fallback；Evidence 显式记录来源、质量、supporting/refuting、fact、inference 与 uncertainty，Hypothesis 用方向一致且可用的 Evidence ID 组织候选。failed、empty、stale 和 fallback 不进入 RCA 排名，普通 success 也必须是 supporting 才能满足主域与症状覆盖；最终只选择有直接支持的候选。这是可解释基线，不是通用因果推理。

### 3 分钟展开思路

先用“调用成功不等于根因正确”和“查询条件里有 timeout 不等于查到 timeout”说明裸结果与关键词方案失败。然后区分 Incident、ToolCallRecord、Evidence、Hypothesis，解释 fact/inference/uncertainty 与四种 stance，并说明结构化空结果、normal 状态和单位先于文本关键词。再写出 `0.22 + supporting×0.18 - refuting×0.15 - missing×0.06` 的有界公式，强调分数不是概率。随后说明类别必须命中领域信号、Analyzer 与报告两层充分性、fallback 0.50/0.72 封顶、三类冲突和一次重试。最后用 Redis 五条 Evidence 展示知识、现场和推断如何收敛，并用 48 条、8 类、3/45 needs-human 边界收尾。

### 可写入简历的准确表述

> 构建 provenance-aware Evidence 与根因假设模型，统一承接 Registry 校验、Policy、MCP 协议错误和 Adapter 异常的 failed 语义；通过 synthetic/mock 来源标记、支持/反驳回链、来源质量门禁、冲突检测和缺失证据补查驱动确定性 Replan，并以报告充分性与 Conclusion Alignment 防止失败结果或模拟数据冒充现场结论。

### 不可夸大的表述

- “实现任意领域的通用因果发现与自动正确 RCA。”
- “0.76 表示根因真实概率为 76%。”
- “规则已经正确处理所有否定语义与时间窗口。”
- “48/48 证明生产根因准确率 100%。”
- “fixture trusted source 等于生产可信数据。”
- “MCP 返回 TextContent 或 `isError=false` 就一定是有效现场 Evidence。”
- “报告生成就表示根因已确认并可自动修复。”

### 高频追问与回答要点

1. **ToolCallRecord 与 Evidence 有什么区别？** 前者证明调用与结果状态，后者解释这次观察的来源、事实、立场和不确定性；调用成功不自动产生 supporting Evidence。
2. **Evidence 与 Hypothesis 如何避免循环论证？** Evidence 的 fact 回到工具输出，inference 单独标注；Hypothesis 只引用 Evidence ID，不把自己的结论再生成成新事实。
3. **0.76 是概率吗？** 不是，是固定加减权重得到的排序分，未校准、未处理来源相关性。
4. **为什么 Runbook 不能确认根因？** 它是知识层，告诉系统通常如何检查；本次事故仍需 live 主域与症状 Evidence。
5. **“0 条日志，但 query 是 ERROR OR timeout”当前会怎样？** 结构化空结果优先判 refuting，query metadata 不参与异常观察；更复杂的自然语言否定仍不是通用保证。
6. **工具失败为何是 unknown 而不是 refuting？** 没拿到观测不能推出目标系统正常；失败只证明证据缺失。
7. **冲突如何处理？** 当前检测三种固定模式，降低置信并生成 evidence_sufficient=false 的待确认报告，不做通用时序仲裁。
8. **为什么 Analyzer 能报告，最终状态仍可能 incomplete？** Analyzer 判断调查是否可停止；报告门禁另查主域、症状、参考和回链，职责不同。
9. **如何防止 mock 证据冒充真实现场？** data_source 归一、source quality 分组和 0.50/0.72 封顶；报告仍要展示 uncertainty。
10. **为什么要保存 refuting Evidence？** 它防止确认偏误，解释降置信原因，并驱动时间窗或替代根因补查。
11. **Top-1 满分为什么不够？** 还要看 Top-3、Macro、必要 Evidence、结论回链、needs-human 与样本分母；48 条 fixture 不能外推生产。
12. **怎样升级成学习排序？** 需要真实结局标签、时间对齐、来源可靠度、同源去重、特征版本、概率校准和线上漂移监控。
13. **为什么 MCP content block 需要先解包？** 失败状态和业务字段可能藏在 TextContent JSON 或 artifact 中；不解包时外层调用成功会把内层 failed 当成普通对象。
14. **为什么 synthetic 成功仍不能按普通 MCP 置信度处理？** 协议互操作可信不等于数据来自现场；显式 mock 来源必须进入 fallback quality 和更低 confidence。
15. **为什么外层 success 仍可能成为 failed Evidence？** `None`、空字符串、空集合和空字典不包含可用观测，持久化前会统一规范化为 `failed/no_data`，避免“没数据”被解释成正常。
16. **Top-1 为什么可能没有 selected root cause？** 候选可以因 Incident 关键词进入排名，但最终选择必须至少保留一条方向正确、状态成功且质量可用的 supporting Evidence；否则只展示候选，不宣布根因。

## 用练习检验是否真正理解

1. 给定 `P95=3250ms`、`5xx=8.2%`、日志含 Redis timeout、Redis `9950/10000`，列出 fact、inference、uncertainty、stance 和 supporting IDs；再加入当前 `1200/10000`，说明冲突如何表达。
2. 按精确公式计算：上下文命中，2 条 supporting、1 条 refuting、2 个 missing 时的分数；解释为什么它不能解读为概率。
3. 为“消息队列积压”设计主域、症状、变更和历史 Evidence，说明当前 catalog 为什么不会自动得到可靠 MQ 根因，以及需要哪些 golden/negative cases。
4. 构造“Redis 当前正常、事故窗口日志异常、metrics 工具失败”的 Evidence 矩阵，分别标 supporting/refuting/unknown，并给出下一步。
5. 解释 Analyzer 充分性与报告充分性的差异：只有 Redis + metrics 时，调查、报告状态和置信上限分别可能怎样变化。
6. 检查一个缺 supporting IDs 的旧 hypothesis 被兼容路径回填的场景，说明为什么 Conclusion Alignment 通过仍不等于因果证明。
7. 为“双重否定、曾异常现正常、query 含异常词但 entries 为空”设计难负例，并让日志适配器输出结构化计数、窗口与 verdict。
8. 设计生产校准方案：如何收集最终根因标签、处理 Incident 漂移、避免同源日志重复计票，并验证 0.76 是否对应真实频率。

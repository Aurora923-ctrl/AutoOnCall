# AutoOnCall 项目量化指标与校招验收：怎样把“效果很好”变成可复核证据

## 一、架构定位：前十三篇解释系统，这一篇证明系统

专题 03–12 已分别解释知识入库、检索、生成、Agent、治理和运行态，专题 13 又建立了分层评测方法。本篇继续向前一步：把散落在不同评测 Artifact 中的结果，整理成一套可以复核、可以追问、也知道不能证明什么的校招验收证据。

它不是一张“成绩海报”，而是连接四类对象的课程章：

```text
项目主张 → 指标定义 → 运行证据 → 能力边界
```

本文的冻结数据主要来自 2026-07-12（Asia/Shanghai）的验收记录，但“同一天”不代表“同一次
run”：clean-commit official baseline、dirty RAGAS 稳定性实验、local-live 性能和 controlled-fault
分别有独立 Artifact。其中另行标注的 2026-07-13 RAG candidate 也不能覆盖 7 月 12 日 official
baseline。当前工作区和测试数量已经继续变化，因此这里的数字是**带来源状态的历史快照**，不是
“当前永远如此”的常量。

## 二、问题现场：为什么一串 100% 反而容易在面试中失分

假设简历只写：

> RAG Recall@3 100%，Agent 根因准确率 100%，安全拦截率 100%。

面试官只要继续问四个问题，这句话就可能失去可信度：

1. 在多少个样本上测的，正例和拒答各有多少？
2. Recall@3、Top-1、F1 分别怎样计算？
3. 结果来自固定 fixture、本地真实模型，还是生产流量？
4. 数据集、Prompt、代码 commit 或阈值变化后还能直接比较吗？

所以项目不追求“数字越大越好”，而追求“每个数字都能回答它从哪里来、能证明什么、不能证明什么”。

## 三、最朴素的版本：只挑最好看的数字

最容易实现的做法，是从日志里找到一个成功演示，把成功次数写成准确率。这会混淆至少五件事：

- 单元测试通过不等于检索质量好；
- 固定答案通过不等于真实模型稳定；
- 离线 RCA 命中不等于生产根因准确率；
- 两条受控故障恢复不等于生产 MTTR；
- scorecard 被标记为 official，不等于其中每个模块都是 production 证据。

AutoOnCall 因此使用分层评测、独立 Artifact 和 provenance，把“基线是否可晋级”与“证据来自什么环境”拆成两个维度。

## 四、最小心智模型：一项指标必须带六个限定词

阅读后续所有数字时，使用下面这张检查卡：

```text
Metric = 定义 + 样本 + 阈值 + 环境 + Provenance + Boundary
```

例如，“Recall@3 = 1.00”只有在同时说明 80 条回归集、64 条非拒答案例、离线确定性索引、数据集 hash、运行日期和非生产边界后，才是一条可复核结论。

## 五、不要混淆三个分类维度

### 5.1 Baseline 状态：这次结果能否作为仓库基线

| 状态 | 含义 |
|---|---|
| `candidate` | 允许来自 dirty worktree、模块不完整或环境变化，只能用于分析 |
| `official` | 干净 commit、模块完整、无 stale 的可追溯正式基线 |
| `stale` | 当前代码、数据集、知识资产、配置或 Prompt 已与 Artifact 不一致 |

`official` 是基线治理状态，不是运行证据等级。一个 official scorecard 仍可同时包含
`offline_fixture`、`local_live`，以及不参与 official 晋级门槛的可选缺失模块。

### 5.2 Evidence level：数据在哪种环境产生

| 等级 | 含义 | 可对外声称的范围 |
|---|---|---|
| `offline_fixture` | 固定离线标注集或确定性模拟数据 | 可证明限定数据集回归，不可称生产准确率 |
| `local_live` | 本地服务、真实模型或真实适配器运行 | 可证明本地真实链路，不代表生产容量 |
| `controlled_fault` | 本地可恢复故障注入实验 | 可证明已设计故障的诊断恢复链，不可称生产 MTTD/MTTR |
| `production` | 真实生产运行数据 | 当前为 `not_enough_data`，不能给出生产结论 |

### 5.3 Review source：是谁给出了质量判断

`simulated_review` 表示模拟 reviewer 导入，用来验证评审模板、导入和统计链路；它不是独立真人盲评。真实人工评审还需要 reviewer 身份、盲评协议、分歧处理和样本量。

## 六、LangChain、LangGraph 和 RAGAS 各自在哪一层

| 技术 | 在 AutoOnCall 中的职责 | 不负责什么 |
|---|---|---|
| LangChain Core | `Document`、消息、Prompt、Embedding 和 Tool 抽象 | 不保存 Incident 生命周期，也不决定生产授权 |
| LangChain Text Splitters | Markdown 标题切分与递归字符切分 | 不提供 OCR 或文档内容治理 |
| LangChain Milvus | 将 Embedding 与 Milvus vector store 接口连接起来 | 不实现本项目的 lexical 索引、融合和 Trust Gate |
| LangChain MCP Adapters | 发现并调用 MCP 工具对象 | 不会自动把任意 MCP 工具晋级为受信任 Agent 工具 |
| LangGraph | 编排一次 Planner 与 Executor/Replanner 循环，并保存进程内 checkpoint | 不替代业务 Store、审批持久化或数据库级恢复 |
| RAGAS | 对固定样本运行 ID-based 和可选 LLM Judge 回答质量指标 | 不直接证明检索在线稳定性、生产事实正确率或 Agent RCA |

这一区分很重要：LangChain 主要提供组件接口，LangGraph 管理有状态流程，RAGAS负责回答质量评测；真正的 Incident、Evidence、Policy、Report 和 provenance 仍由项目自己的模型与服务实现。

## 七、先学会计算，再看项目结果

| 指标 | 最小定义 | 主要回答的问题 |
|---|---|---|
| Recall@K | 对每个正例计算前 K 个结果覆盖了多少标注相关项，再取平均 | 该找回来的资料找回来了吗 |
| Precision@K | 前 K 个结果中相关项所占比例，再对样本取平均 | 返回结果里噪声有多少 |
| MRR | 第一个相关结果排名倒数的平均值 | 第一个有用结果出现得够不够早 |
| MAP@K | 对每个查询计算截断 K 的平均准确率，再跨查询平均 | 多个相关项的排序整体如何 |
| nDCG@K | 将相关性增益按排名折损，再除以理想排序 DCG | 高相关结果是否排在更靠前位置 |
| Top-1 Accuracy | 第一根因等于标注根因的 case 比例 | 首选根因是否正确 |
| Top-3 Recall | 标注根因进入前三候选的 case 比例 | 候选集是否包含正确根因 |
| Macro-F1 | 每个根因类别分别算 F1 后等权平均 | 小类别是否被总体准确率掩盖 |
| Tool Selection F1 | 计划选择工具集合与标注工具集合的 Precision/Recall 调和平均 | Agent 是否查了应该查的系统 |
| P95 latency | 排序后覆盖 95% 样本的延迟分位点 | 绝大多数请求在多长时间内完成 |
| Wilson CI | 二项比例的置信区间估计 | 小样本 100% 到底有多不确定 |

`20/20 = 1.00` 并不意味着真实成功率必然为 100%。例如 20 个成功样本的 95% Wilson 下界约为 `0.8389`，这正是知识资产表仍保留置信区间的原因。

### 9.4 回答质量与 RAGAS

AutoOnCall 没有把“RAGAS 分数”当成一个单值。评测脚本先选择 answer source，再选择 metric profile：

| 维度 | 选项 | 含义 |
|---|---|---|
| answer source | `product-offline` | 固定离线检索，答案仍经过产品引用与拒答契约 |
| answer source | `reference-fixture` | 固定检索和固定答案，用来验证指标管道 |
| answer source | `context-fixture` | 固定上下文交给真实 grounded LLM，用来隔离生成变化 |
| answer source | `runtime` | Milvus 运行时检索加真实生成，最接近本地产品路径 |
| metric profile | `id-smoke` | 不需要外部 Judge，运行 ID-based Context Precision/Recall 与确定性业务门禁 |
| metric profile | `full` | 调用 RAGAS Faithfulness、Response Relevancy、ID metrics 以及 AspectCritic |

`full` profile 中，Faithfulness 判断回答中的主张能否从检索上下文获得支持；Response Relevancy 判断回答是否围绕用户问题；ID-based Precision/Recall 直接比较检索上下文 ID 与参考 ID，不依赖 LLM Judge。OnCall Actionability 和 Answer Completeness 使用 RAGAS `AspectCritic` 承载项目自定义 rubric，前者检查证据查询与受控操作是否可执行，后者检查事实、引用、处置选项和安全边界是否完整。

当前 RAGAS 适配显式构造 `SingleTurnSample`，字段为 `user_input`、`retrieved_contexts`、`retrieved_context_ids`、`reference_context_ids`、`response`、`reference` 和 `rubrics`；metric class 只从安装版本的公开兼容导出解析。ID metric 若退回确定性实现，会记录 engine/status/reason，而不是继续让报告看起来像原生 RAGAS。所有聚合项都输出 available/expected 与 missing case IDs；应有指标缺失时平均值为 unavailable，不能跳过失败行后缩小分母。

当前配置阈值如下。阈值不全是 hard gate：在 `id-smoke` 中 ID Context Precision 是 watch
metric，Faithfulness 和 Response Relevancy 未运行；`full` profile 才会调用 Judge。Artifact 中的
`hard_gates`、`watch_metrics` 与 `not_run` 必须一起读，不能看到阈值就自行判定整个 run 失败。
这些阈值也只是工程回归线，不是统计显著性结论：

| 指标 | 默认门槛 | 低于门槛意味着什么 |
|---|---:|---|
| Faithfulness | `0.85` | 回答可能超出检索上下文支持范围 |
| Response Relevancy | `0.80` | 回答可能没有聚焦用户问题 |
| ID Context Precision | `0.75` | 检索上下文混入过多非参考 chunk |
| ID Context Recall | `0.75` | 参考上下文覆盖不足 |
| OnCall Actionability | `0.80` | 缺少具体检查或受控处置步骤 |
| Answer Completeness | `1.00` | full profile 的关键事实、引用或边界不完整 |
| Refusal Boundary | `1.00` | OOD 或证据不足场景没有按契约拒答 |

```text
  → /api/eval/* 与工作台只读展示
```

关键点是 Scorecard 不重新“发明”指标，它聚合模块已经保存的样本数、分子、分母、置信区间、失败 case 和 Artifact 路径。某个模块缺失、`not_run`、`incomplete`、`retrieval_only_passed`、`passed_without_milvus`、`observed_not_accepted` 或 stale 时，都不能补成零、沿用旧数字或当作完整 passed；评测 API 还会把 stale summary 标成 `available=false`。

## 十六、设计取舍：为什么不做一个统一总分

| 方案 | 优点 | 风险 | 当前选择 |
|---|---|---|---|
| 所有能力压成一个总分 | 展示简单 | 检索、生成、安全和延迟互相抵消，无法定位退化 | 不采用 |
| 只跑真实模型端到端 | 最接近用户体验 | 成本高、波动大、失败难定位 | 只用于 local-live 补充 |
| 只跑 deterministic fixture | 快速、稳定 | 不能代表真实 Embedding、模型和外部依赖 | 作为回归主干 |
| 分层模块加统一 provenance | 能定位退化，也能保留真实链路证据 | Artifact 和口径治理更复杂 | 当前采用 |

## 一次“成功诊断”究竟成功了什么

凌晨 02:07，`order-service` 返回 503，P95 超过 3 秒，日志出现 `Redis connection timeout`，Redis 的 `connected_clients` 接近 `maxclients`。AutoOnCall 接收故障、查到资料、调用工具并生成报告。此时说“自动诊断成功”看似自然，却至少可能指六件不同的事：代码分支存在、固定测试通过、离线样本跑通、本地真实依赖跑通、受控故障复现成功，或者真实生产 Incident 得到了验证。

这六种结论不能互换。接口返回 HTTP 200 只说明请求被处理；SSE 正常结束只说明流结束；报告中出现根因文字也不代表根因正确。若不先约定证据口径，后续章节中的 Recall、F1、P95、覆盖率和“通过率”都会变成没有边界的宣传数字。

本篇因此先建立全系列共同使用的规则：**声明的强度不得超过证据的强度**。下一篇会用这把尺子推导受控诊断闭环；第 02 篇再处理告警如何成为稳定的故障事件。

### 这不是免责声明，而是一项架构能力

初学者容易把“证据边界”理解成文档末尾的一段免责说明：系统先尽量表现得聪明，最后再补一句“结果仅供参考”。这种做法的问题是，边界只存在于文字里，没有进入系统状态、接口契约和评测产物。一旦前端只展示“诊断完成”，调用方就会自然地把它解释成“根因已确认”；一旦评测汇总只保留一个百分比，面试官也无法判断它来自固定 fixture、真实依赖还是生产事故。

真正的工程做法是让证据强度参与设计。工具调用要记录来源和时间窗口，报告要能从结论回链到 Evidence，评测结果要记录命令、数据集指纹和代码状态，运行接口要区分 completed、degraded、needs_human 等业务终态，健康检查也要把“进程活着”和“依赖可服务”拆开。这样，边界不再依赖作者是否谨慎，而是由结构化字段和状态转换共同约束。

这个原则也解释了 AutoOnCall 为什么没有把所有成功条件压成一个布尔值。对 Redis 案例来说，HTTP 请求成功、Prometheus 查询成功、日志命中 timeout、连接数接近上限、候选根因排名第一、报告生成成功，是六个不同层次的事件。它们可以同时成立，也可以只成立一部分。只有保留层次，系统才知道下一步应继续取证、降级输出还是交给人。

### 学习本系列时始终追问四句话

后续每看到一个“成功”“通过”或“已支持”，都应追问：它描述的对象是什么；分母和样本来自哪里；在哪种环境中运行；结论不能外推到哪里。比如“检索引用元数据率 100%”若只描述固定正例中返回的 chunk 都带 source/chunk 定位，只能证明检索结果可回查，不能证明模型实际引用、更不能证明引用语义支持答案；“本地真实 Redis 查询成功”只能证明适配器在这台机器、这份配置和这个时刻可用，不能证明生产网络、权限和容量均可靠。

这四个问题也是课程阅读顺序的骨架。先理解问题和失败，再看通用概念；先看概念如何变成结构化约束，再看 AutoOnCall 的实现；最后用样本、运行环境和 Artifact 检验结论。若把代码路径提前到原理之前，读者只会记住函数名，却无法解释为什么要这样设计。

## 把“做了”拆成三类问题

判断一项能力时，应依次问三件事。

第一，**是否已实现**。严格地说，它表示当前代码中存在相应的数据结构、分支或接口；它解决“设计是否落到代码”的问题，但不负责证明依赖可用、结果正确或系统稳定。例如代码里有 Redis 适配器，只能证明有调用路径，不能证明生产 Redis 已接通。

第二，**是否被自动化测试验证**。它表示在受控输入、替身或临时存储下，某个明确契约通过；它解决“改动是否破坏已知行为”的问题，但不负责覆盖未知流量、真实依赖抖动和多副本竞争。例如固定输入下能拦截危险工具，是测试事实，不是生产安全认证。

第三，**留下了什么运行证据**。这里的运行证据通常保存为 **Artifact**：带日期、环境、命令、代码版本和数据指纹、可以复查的产物。Artifact 解决“这次结果如何复现”的问题，但不自动提升证据等级；一个 dirty worktree 产生的离线 JSON 仍然只是候选记录。

运行证据使用 `evidence_level` 标注：

| 等级 | 严格定义 | 能证明什么 | 不能证明什么 | Redis 例子 |
|---|---|---|---|---|
| `offline_fixture` | 固定样本、替身或离线数据上运行 | 已知样本的可复现行为 | 实时依赖和真实故障表现 | 用固定的 `9940/10000` 样本生成报告 |
| `local_live` | 本地连接真实模型或真实依赖 | 本地集成链可运行 | 生产稳定性、容量和治理 | 本机真实 Milvus/Redis 查询成功 |
| `controlled_fault` | 人为构造并限定边界的故障环境 | 能处理指定故障 | 未知故障覆盖和生产泛化 | 在 sandbox 中触发连接耗尽 |
| `production` | 真实生产流量与治理环境 | 只能支持其来源明确的生产结论 | 不能外推到未观测业务或未来流量 | 带来源的真实 Incident Artifact |

Implementation 和 Automated Test 不是 `evidence_level`。反例是“类已实现，所以已有 `offline_fixture` 证据”；没有实际运行产物时，这个推论不成立。

### 已实现、测试通过和运行过为何必须分开

“已实现”回答的是静态存在性。它可以由代码审查确认：是否有字段、分支、状态机和错误路径。但静态存在性无法告诉我们分支是否真的可达，也无法证明外部依赖返回了作者预期之外的数据时仍能工作。一个 MySQL Store 类即使实现了全部方法，只要从未连接真实 MySQL 做中断恢复，就不能声称完成了真实恢复验证。

“自动化测试已验证”回答的是契约回归。测试通过意味着给定前置条件时，输出符合断言。它最适合验证权限矩阵、状态优先级、工具参数白名单、引用格式和幂等键等确定性行为。它不适合替代真实模型质量、复杂网络故障或生产用户行为，因为测试中的 monkeypatch、临时 SQLite 和固定时间会主动消除大量不确定性。

持久化修复尤其需要保持这一区分。SQLite/MySQL 行为一致性、事务回滚、条件更新、retention 保护和迁移幂等可以用自动化测试证明；当前本地 MySQL 容器的 health、七表和权限可以用只读查询证明。但两者合起来仍不能证明真实进程崩溃、数据库重启、备份恢复或多副本争用已经通过 controlled-fault 验证。

“运行证据”回答的是某次执行到底发生了什么。它必须能被别人复核，所以不仅要保存结果，还要保存 provenance，也就是结果的来历。至少包括日期、命令、代码版本、工作区是否 dirty、数据集 hash、环境说明和 Artifact 路径。缺少这些字段时，数字仍可作为探索记录，却不能成为稳定基线。

这三类证据不是简单的高低排序。某些安全规则更适合用确定性测试证明，某次真实依赖运行也可能因为样本过少而不能说明质量。合理做法是针对声明选择证据，而不是盲目追求“看起来更真实”的运行环境。

### candidate、official、stale 与 evidence level 是不同维度

项目把候选结果称为 `candidate`：它可以来自 dirty worktree，也可以缺少一部分复现字段，适合开发阶段比较。`official` 表示结果满足当时定义的冻结、完整 provenance 和基线晋级规则，便于团队把它作为后续比较对象。official 只说明“这是一份合格的基线记录”，不说明它来自生产；一个完全可复现的离线 fixture 仍然只是 `offline_fixture`。

`stale` 描述的是时效性。知识文件更新后，旧 chunk 即使曾经 official，也可能已经过期；运行 Artifact 对应的代码、数据集或环境发生变化后，也不能直接代表当前版本。时效性和证据环境是正交的：一个 production Artifact 也可能 stale，一个刚生成的 offline fixture 则可能 fresh。

因此，完整声明至少是一个多维坐标，而不是单标签。例如：“2026-07-12 的 official RAG 基线，`evidence_level=offline_fixture`，代码 clean，数据集 hash 已记录，目前未 stale。”省略其中任一维度，读者都可能把结论理解得过强。

### 怎样判断一条声明是否越界

可以使用一个简单的反向检查：先写出准备对外说的话，再问需要什么证据才能使它成立。如果想说“系统能拦截所有危险操作”，就需要明确“所有”的全集、危险操作定义、误报和漏报，还要覆盖绕过路径；几个单元测试显然不够。如果实际只有 43 条冻结安全 case 全部通过，准确表达应保留 `43/43`、case 来源和环境，而不是删掉限定词。

同理，“诊断延迟 P95 为某个数值”必须说明测量起止点、样本数、并发度、是否包含模型和依赖时间。P95 是排序统计量，不是最坏值；样本只有几十条时，尾部结论尤其有限。证据边界的价值就在于把这些容易被忽略的问题变成默认检查项。

## 为什么服务能启动仍不等于能对外承担责任

AutoOnCall 是 Python 3.11 FastAPI 应用，提供健康检查、聊天、文件索引、AIOps 诊断、告警、审批、Incident、评测、反馈和可选 A2A 门面，并挂载静态工作台。这说明接口骨架已经存在，却没有回答身份、暴露面、依赖和恢复能力是否达到生产要求。

### 鉴权关闭是开发便利，不是安全结论

项目实现的是静态 API Token + scope RBAC：`viewer` / `reader` 只读，`operator` 具有读、诊断、聊天写入、知识写入和评测权限，`approver` 只有读取与审批权限，`change_operator` 只有读取与变更办理权限，`admin` 拥有全部 scope。人工回执还要求执行 actor 不得等于该请求的审批 actor。

当 `API_AUTH_ENABLED=false` 时，普通本地演示请求仍可使用未认证 Principal；审批与 change API 则返回 `503`，避免匿名主体完成高风险闭环。启用鉴权却没有可用 token 时同样返回 `503`。这里已实现的是静态内部 token/scope 与应用层职责分离，不是 SSO、OIDC、企业目录、多租户隔离、短期凭证轮换或组织级多人会签。

监听 `0.0.0.0`、`::`、内网 IP 或非 loopback 主机名时，程序会统一视为非本地绑定，并检查 debug、鉴权是否关闭、鉴权是否没有可用 Token、CORS 是否为 `*`，以及 AIOps mock fallback 是否开启。只要发现风险，默认即拒绝启动；只有显式设置 `ALLOW_INSECURE_EXTERNAL_DEMO=true` 才会带警告继续。配置中仍保留的 `PRODUCTION_EXPOSURE_STRICT` 不是当前执行该决策的开关，不能据其取值推断启动会放行。当前门禁仍不检查 TLS、限流、密钥来源和备份，不能把一份部署建议反写成运行代码已经覆盖的事实。

### 认证、授权和审计是三件事

认证回答“调用者是谁”。AutoOnCall 接受 Bearer Token 或 `X-AutoOnCall-Token`，并用常量时间比较降低直接字符串比较带来的时序泄漏风险。授权回答“这个身份能做什么”，由 scope 决定 read、diagnose、approve、change 等能力。审计则回答“谁在什么时间做了什么”，审批 actor 在启用鉴权时取自 `AuthPrincipal.token_name`，而不是盲信请求正文里自报的用户名。

三者不能互相替代。只验证 token 而不分 scope，会让只读观察者拥有审批能力；只做授权而不保留 actor，事故后无法解释是谁批准；只在日志里打印用户名而不认证，攻击者可以伪造审计身份。当前静态 token 方案已经拆分诊断、审批和 change scope，并在单次变更上拒绝审批人自报执行结果；token 生命周期、撤销、组织身份、代理关系和多人会签仍需要外部身份系统补齐。

Token 本身可用，也不代表审计名称可以直接信任。JSON Token 注册表允许给 Token 配置 `name`，它会进入审批 actor 和日志上下文；若名称包含换行、控制字符或任意长文本，攻击者可能伪造下一行日志或污染审计检索。当前实现只接受受限字符集和长度内的名称，不合法时退回由 Token 哈希派生的稳定匿名名；明显占位值、少于 16 个字符或包含空白/控制字符的 Token 则直接忽略。这里的工程重点不是“16”这个数字本身，而是**示例配置不能在生产中自动变成有效凭证，审计标签也必须按不可信输入处理**。

### `401`、`403` 与 `503` 表达不同失败

当鉴权开启但请求没有 token，系统返回 `401`，表示需要提供身份；token 存在但无效同样是身份失败。token 有效却缺少目标 scope 时返回 `403`，表示身份已知但没有权限。更容易被忽略的是：鉴权开关已启用、服务端却没有配置任何 token 时返回 `503`。这不是用户输错凭证，而是服务自身处于不安全的错误配置状态，因此选择 fail closed。

这种状态语义有利于运维。若所有情况都返回 `401`，值班人员会把服务端配置缺失误判为客户端问题；若为了“可用性”在无 token 配置时自动放行，则安全开关反而制造了开放接口。课程中应记住：失败码不仅服务调用者，也服务故障定位。

### 当前暴露保护为何仍然只是第一道门

启动时只有在监听非本地地址时才执行这组暴露检查，风险项包括 debug 开启、鉴权关闭、鉴权开启却没有可用 Token、wildcard CORS 和 mock fallback。任一风险存在时默认拒绝启动；只有显式设置 `ALLOW_INSECURE_EXTERNAL_DEMO=true` 才会记录警告并继续。配置中保留的 `PRODUCTION_EXPOSURE_STRICT` 不参与当前决策。这能防止最常见的“把本地演示配置直接绑定到公网或内网网卡”，却不是完整生产安全基线。

它没有证明 TLS 已终止，没有检查密钥是否来自 Secret Manager，也没有统一验证限流、请求体大小、上传目录、网络策略和审计留存。更合理的生产方案是把应用内门禁与网关、身份提供商、网络隔离和发布策略组合起来。应用内检查负责尽早发现明显错误，平台治理负责处理跨服务和持续运营问题。

### 调试入口与静态资源也属于攻击面

FastAPI 的 Swagger UI、ReDoc 和 OpenAPI JSON 会集中暴露路由、请求模型、字段和错误结构，适合开发，却不应因为框架默认开启就自动进入生产暴露面。当前应用仅在 `DEBUG=true` 时注册 `/docs`、`/redoc` 和 `/openapi.json`；根路径也只在调试模式返回 docs 链接。关闭文档并不能代替鉴权，但能减少不必要的接口枚举入口。

静态工作台使用从 `app/main.py` 推导出的项目绝对路径挂载，而不是依赖进程当前工作目录。这不是代码风格问题：若容器、进程管理器或测试从不同目录启动，相对路径可能导致应用启动失败或错误挂载其他目录。Docker 镜像同时改为非 root 用户运行，并让启动命令和 HEALTHCHECK 读取 `HOST`、`PORT`；它降低容器逃逸后的权限和配置漂移风险，但仍不能证明镜像供应链、文件系统只读、Seccomp 或 Kubernetes SecurityContext 已完成。

### 健康不是一个布尔值

健康检查按问题拆开：

```text
/health/live        进程能否响应
/health/ready/rag   RAG 需要的 Milvus 是否可用
/health/ready/aiops AIOps 是否至少有一个真实探针成功的外部适配器
/health/ready       汇总 RAG 与 AIOps readiness
```

`/health` 保留兼容性，但采用 readiness 语义。这样可以避免“进程活着”被误写成“全部能力可接流量”。当前 RAG readiness 使用独立临时 Milvus client 做只读契约探测：`biz` collection 必须存在，schema、向量维度、vector index、load state、待建索引行及抽样 metadata 必须符合应用约定；同时要求 Embedding API key、模型名和维度配置完整。它不会创建 collection、建索引或写入数据，也没有发出一次真实 Embedding 请求、检索请求或模型生成请求，因此 `ready` 只能说明“RAG 检索与入库的最低依赖和配置可尝试”，不能证明供应商调用、知识质量或回答质量。AIOps 至少要求 Prometheus 或 Loki 有一个真实探针返回 `connected`，且不能存在已配置但探测失败的 Adapter。Redis、MySQL 或 Kubernetes 单独连通只说明某个领域信号可取，不能把缺少指标和日志主观测面的实例提升为 AIOps ready。只知道 URL 已配置、但没有执行真实探针的 Log Gateway 和 Ticket 只会形成 `unverified`，不能把配置存在包装成能力可用。

### liveness 只回答“要不要重启进程”

`/health/live` 不访问 Milvus，也不探测 Redis、MySQL 或 Prometheus。它只确认 FastAPI 事件循环仍能响应。这样做是为了避免依赖短暂抖动触发容器反复重启：重启应用通常无法修复远端数据库不可用，反而会丢失进程内 checkpoint，并制造更大的流量波动。

因此 liveness 应当稳定、快速、少依赖。它不能用来决定是否把业务流量发给实例。把所有外部依赖都塞进 liveness，是新手常见错误，因为它混淆了“进程坏了”和“能力暂时不可用”。

### readiness 应按能力拆分，而不是要求所有依赖都在线

AutoOnCall 同时提供 RAG 和 AIOps。RAG 的核心依赖是 Milvus；AIOps 则要求至少存在一个真实探针成功、且其他已配置探针没有失败的外部适配器。如果 Milvus 故障而 Redis/Prometheus 可用，AIOps 诊断能力可能仍可接流量，RAG 搜索和上传索引则不应接流量。反过来，Milvus 正常但所有 AIOps 适配器都未配置，或只有无法在线验证的配置项时，聊天检索可以工作，实时诊断不能伪装成 ready。

这种“能力 readiness”比单一全局状态更接近真实系统。全局 `/health/ready` 仍会汇总未就绪能力，方便简单部署；细分端点让网关、运维脚本和人工排障获得更准确的信号。

### `not_configured`、`configured`、`connected` 和 `failed` 的差别

未配置表示系统明确知道没有该数据源，不应该尝试生成实时事实；configured 表示存在配置，但某些适配器只做配置级检查，尚未证明远端调用成功；当所有可用信号都停在 configured 时，总体状态是 `unverified`；connected 表示实际探针成功；failed 表示配置存在但调用失败。外部总体状态不会把一组全部未配置或仅配置未验证的适配器提升为成功。即使某个领域 Adapter 已 connected，AIOps readiness 仍要求 Prometheus/Loki 至少一个主观测源可达；否则能力状态为 `partial`，不接收完整诊断流量。

这里仍有设计取舍：当前策略对任何已配置依赖失败都较保守，可能让部分仍可用的诊断场景被整体判为 degraded。更复杂的生产系统可以按故障域声明最低依赖集合，例如 Redis Incident 要求 Redis 与日志至少一个可用，而 Kubernetes Incident 需要集群 API。但这需要更复杂的能力注册和路由，本项目没有把它包装成已经实现的功能。

### 健康检查也必须避免泄密

适配器失败后，响应只暴露分类后的 `error_type` 和公共错误信息，而不是把原始异常、连接串、token 或数据库地址返回给调用者；即使连接成功，Redis/MySQL ping 返回的内部 endpoint 也不会透传。适配器构造失败同样被收敛为结构化失败，避免异常越过健康接口边界。内部日志也只记录错误类型，不直接拼接原始异常。这说明可观测性和安全之间需要平衡：公开健康接口只给出足以路由和排障的摘要，受控内部日志也不应默认成为秘密存储。若为了方便把完整异常直接返回或记录，健康端点和日志系统都会成为凭据与网络拓扑的泄漏面。

### readiness 不应把事件循环卡在同步探针上

Milvus readiness 是同步调用，而 Redis、MySQL 和 HTTP 探针是异步调用。如果在 FastAPI 协程中直接执行同步网络检查，单个依赖超时会阻塞事件循环，连其他健康请求也可能被拖慢。当前实现把 Milvus 探针放进工作线程，并让 Milvus、HTTP、Redis 和 MySQL 检查并发进行；这降低探针串行累积延迟，却不是无限并发许可证，超时仍必须为正数且由配置校验提前拒绝非法值。

生命周期也遵循同样原则：Milvus 继续懒连接，但 schema 管理连接使用独立 alias 和有界重试；`lifespan` 使用 `try/finally`，即使应用运行体抛异常，也会关闭 LangChain 持有的同步/异步 client、释放 ORM collection 并断开 schema alias。面试中可以把它概括为“启动失败关闭、运行异常也执行资源清理”，而不能说成所有外部连接都已经具备完整优雅停机与请求排空。

## RAG 的“用了什么后端”和“多花了多少成本”也必须有口径

检索质量改进很容易制造新的误解：看到本地词法命中，就把运行时说成“本地 RAG”；看到修复后答案合格，就只报最终答案而隐藏额外模型调用。当前实现把这两件事显式拆开。默认运行路径把 Milvus 标为权威向量后端，响应中的 `vector_backend` 与 `retrieval_backend`、以及 `observability.runtime` 用来说明这一身份；即使 Milvus 调用失败，失败 payload 仍保留 `milvus` 身份和结构化错误，而不是把失败悄悄改名为“本地检索成功”。

本地 JSON 词法索引仍参与混合检索和降级：当向量候选不足或向量链失败时，它可以提供候选；显式多来源问题还会先把来源偏好转成 **Milvus 的定向来源探测**，以补足指定文档的召回。它们是候选与恢复策略，不把本地索引提升为运行时权威向量库。阅读后文的检索指标时，应同时看后端身份、`retrieval_degraded`、vector/lexical error 类型、各阶段耗时和候选—去重—可信筛选后的数量；不能只凭最终有几条引用推断 Milvus 正常或两种后端都贡献了答案。

回答侧也不是“模型第一次说对就结束”。可信 chunk 先形成受限的生成证据与子目标覆盖矩阵；若最终答案遗漏必要来源、遗漏已绑定的子目标，或未通过逐条引用约束，当前实现会进行有界的再生成校正。仍无法形成合格引用时，才尝试从同一批允许证据构造抽取式兜底；没有可信来源时保持拒答。这里的“修复成功”只说明固定契约下的候选答案经过门禁，不等于模型完成了通用事实核验，更不等于现场根因被确认。

重试不是免费的。每次再生成会合并已观测的 `llm_generation_ms` 与 token usage，`total_ms` 也覆盖整个回答链；`generation_repair` 至少暴露是否发生过合并式修复及必要来源修复的原因/来源。价格快照尚未进入运行时契约时，`estimated_cost.status` 必须保持 `not_observed`，不能由 token 数倒推出人民币或美元成本。覆盖修复与抽取式兜底目前也不是完整的稳定对外计费/尝试次数接口：它们应被视为实现和定向自动化测试支持的行为，若要比较版本，仍需在 Artifact 中记录模型、后端、数据集、命令、原始观测与是否 dirty。

这套口径把第 04、05、13、15 篇中的“检索改进”约束为可复核的链条：先说明本次运行使用的后端和候选，再说明覆盖与引用门禁是否触发修复，最后才报告耗时、token 和通过率。它避免把局部测试中的修复路径包装成生产成本结论。

## 数据落在哪里，决定故障时能承诺什么

### 先区分业务状态、过程记录与大结果

长时诊断至少产生三类数据。业务状态回答 Incident 当前处于什么阶段，例如等待审批还是已经生成报告；Trace 记录 Planner、工具、Evidence 和状态转换的过程；大工具输出可能包含数万字日志或查询结果，只适合写入 Artifact 文件，状态中保存摘要、hash、大小和引用。

若把三类数据全部塞进一个 JSON，短期实现简单，长期会出现三个问题：状态对象越来越大，任何更新都要重写全部内容；敏感日志更容易通过普通查询接口泄漏；多副本下并发更新更容易覆盖。AutoOnCall 的 Store Protocol、Trace/Report/Snapshot 分离和大结果外置，是为这些具体失败服务，而不是为了抽象而抽象。

| 数据类别 | 当前实现 | 明确边界 |
|---|---|---|
| AIOps 业务状态 | 默认 SQLite，可切换 MySQL | MySQL 实现和测试不等于真实恢复演练 |
| RAG 向量索引 | Milvus；默认运行时向量后端身份写入 retrieval observability | 未声明 HA、备份与容量基线；ready 不等于真实 Embedding/LLM 调用成功 |
| RAG 词法索引 | 本地 JSON，线程锁 + 跨进程文件锁 + 原子替换；混合候选与 Milvus 故障降级 | 适合共享可靠锁文件的单机/小规模部署，不是分布式搜索服务，也不是默认运行时权威向量后端 |
| 工具大结果 | 本地 Artifact 文件 | 不是分布式对象存储，retention 治理有限 |
| 评测与质量记录 | 本地 JSON/JSONL Artifact | 必须携带日期、命令和环境，不能当生产指标 |

Prometheus、日志网关、Kubernetes、Redis、MySQL 和工单系统经适配器接入。未配置或失败时返回结构化失败，不应生成模拟成功证据。

适配器“连接上”也不等于语义正确。修复后的边界包括：HTTP 2xx 业务失败不算 success；Prometheus 使用 `query_range`、真实窗口与 step，并拒绝 NaN/Infinity；Redis 历史 hash/Stream 只在请求窗口内覆盖当前 INFO；MySQL 分离服务端连接与应用池；指标和日志主源失败后继续尝试后备源并保留 `fallback_errors`；显式未知 Redis/MySQL 实例返回 `not_found`。这些是实现和自动化契约，不替代新的 local-live 运行 Artifact。

仓库的四类运行材料也不能合并成“完整部署方案”：

| 材料 | 实际用途 | 不能据此声称 |
|---|---|---|
| `vector-database.yml` | 本地启动 Milvus、etcd、MinIO，支撑 RAG 向量链 | 已包含完整 AIOps 依赖 |
| `interview-stack.yml` / `full-stack-compose.yml` | 启动 Redis、MySQL、exporter、Prometheus、Loki、日志 emitter，支撑本地真实适配器演示 | FastAPI、模型供应商、密钥平台和网关已生产化 |
| `sandbox-compose.yml` | 为受控故障和适配器检查提供较小的 Redis/MySQL/Prometheus 环境 | fixture 就是生产沙箱 |
| `Dockerfile` 与 `production.md` | 应用镜像和上线前检查建议 | 已完成编排、扩缩容、TLS、备份和 HA/DR |

默认面试栈把 CMDB、发布历史和历史工单种在真实 MySQL 表中，而不是三个 HTTP mock；Kubernetes golden case 仍是离线 fixture。Milvus/RAG 需要单独启动和入库，也不属于默认 AIOps 面试栈。于是“本地真实适配器”“RAG 向量链”和“生产部署”必须分开陈述。

Compose 文件是可复核的部署定义，不是本次文档的运行 Artifact。除非后文给出带日期、命令、环境和结果的只读记录，不能由镜像标签、端口映射或 volume 配置声称当前容器已运行；即使有这样的本地记录，也只能说明当时配置相符，不能证明备份、迁移、磁盘故障恢复或生产容量。

### SQLite 与 MySQL 的选择不只是修改连接串

显式传入 storage path 时，Store Factory 固定选择 SQLite，这是为了让测试和本地工具保持确定性；默认单例服务才根据 `AIOPS_STORAGE_BACKEND` 选择 SQLite 或 MySQL。这个细节避免测试环境意外连接开发者 MySQL，也让依赖注入的临时路径语义稳定。

但实现同一 Protocol 不等于两种后端拥有完全相同的运行保证。SQLite 适合单机和低并发演示，事务与文件备份简单；多副本同时写入时会遇到锁竞争和共享文件问题。MySQL 更适合集中状态，但需要连接池、迁移、备份、权限、故障切换和真实恢复演练。当前代码和测试能证明接口契约与 SQL 路径存在，不能直接证明多副本生产耐久性。

### latest Snapshot 不是事件溯源

Snapshot 采用 upsert 保存会话的最新业务状态，适合页面查询和审批恢复 fallback。它不保存每个历史版本，也不保证从任意 LangGraph 节点精确恢复。Trace 虽然保留过程事件，也没有被设计成可重放所有副作用的事件日志。把二者组合称为“完整事件溯源”会掩盖工具幂等、事件版本和重放副作用等尚未解决的问题。

Replay 只是读模型：它从已有 Trace、审批、变更执行、报告和 IncidentState 派生时间线，让人理解当时发生了什么。它不会重新查询 Redis，也不会把今天的状态改写成事故时事实。生产级恢复还需要数据库 checkpoint、幂等工作队列、跨实例租约和故障注入证明。

### Artifact 外置解决体积问题，不自动解决治理

工具输出超过内联预算时写入本地 Artifact 目录，Trace 和报告保存摘要与引用。这样减少状态膨胀，也能在普通接口中隐藏原始载荷。但本地目录不是对象存储：多副本实例可能看不到彼此文件，磁盘损坏会丢失产物，retention、加密、WORM 和访问审计也不完整。

因此准确说法是“实现了大结果外置与引用”，而不是“实现了企业级 Artifact Store”。若走向生产，应把引用指向受控对象存储，增加内容 hash 校验、生命周期策略、租户隔离、下载鉴权和删除审计。

## 一条能力声明怎样沿证据链升级

### 第一步：把自然语言承诺改写成可测契约

假设目标是“危险 Redis 写操作不会被普通诊断请求直接执行”。先定义危险操作集合、普通请求身份、预期状态和允许的失败方式。契约可以写成：operator 发起的 `CONFIG SET` 被 pre-tool policy 阻断或转入审批，不产生真实写调用；approver 只能审批既有请求，不能绕过变更状态机。

这一步解决定义模糊。若“危险”没有枚举边界，任何通过率都缺少分母；若只检查最终报告里的文字，无法证明工具层没有执行副作用。

### 第二步：用自动化测试证明确定性路径

针对身份、scope、策略、审批并发和幂等键编写测试，覆盖允许、拒绝、缺配置和竞争条件。此时可以说“这些固定契约在当前 candidate 工作区通过”，不能说生产环境已经安全。测试还应保留负例，因为只测成功路径会把 fail-open 隐藏起来。

### 第三步：在离线和本地真实环境留下 Artifact

离线安全 case 可以冻结输入并计算 Precision、Recall、F1；本地真实依赖验证则确认适配器没有被 mock 替代。两类结果用途不同，必须分别标记。运行时记录 command、commit、dirty、dataset hash 和环境，才能比较后续版本。

### 第四步：通过受控故障验证跨组件行为

受控停止 Redis、制造连接耗尽或让日志网关返回错误，可以观察 Agent 是否产生结构化失败、是否停止补查、是否错误地生成 confirmed 根因。这里验证的是预先设计的故障，不是未知故障泛化。故障注入还必须有清理和环境恢复记录，否则下一次样本会被污染。

### 第五步：生产证据需要治理和长期样本

生产验证不仅是“连到生产跑一次”。它需要合法授权、脱敏、事故标签、人工复核、失败样本、时间窗口定义和持续监控。若要声称降低 MTTR，还要有稳定起止点和对照组，排除告警策略、团队变化等混杂因素。当前 AutoOnCall 没有这类数据，因此主动停在实现、测试、离线、本地真实和受控故障层级。

## 接口返回成功，业务可能仍未成功

### HTTP 成功只说明服务接受了请求

FastAPI 路由返回 2xx，通常只能证明输入通过校验并进入了处理路径。对普通查询接口，这可能已经足够；对长时 Agent 流程，它只是开始。诊断请求可能随后遇到模型不可用、工具未配置、证据不足或等待审批。如果监控只统计 HTTP 成功率，就会把大量业务降级隐藏在“接口可用”之下。

相反，HTTP 4xx 也不一定是系统故障。`401` 和 `403` 可能表示安全策略正确工作，错误输入返回 `422` 说明契约拒绝了不合法参数。运维指标应把调用错误、权限拒绝、内部错误和业务降级分开，否则团队会为了降低错误率而错误地放松校验。

### SSE 流结束不等于根因被确认

AIOps 诊断通过 SSE 连续发送规划、工具、Evidence、审批和报告事件。网络流正常收到结束事件，只说明服务端不再发送数据；它可能以 completed、degraded、needs_human 或 failed 结束。前端必须读取结构化终态，而不能把“连接关闭”渲染成绿色成功。

同样，浏览器中途断开也不等于工作流停止。服务端可能仍在运行并持久化 Snapshot，客户端需要通过 run-status 查询最新状态。当前系统不支持完整 Last-Event-ID 字节级重放，因此断线后的正确动作是查持久化读模型，不是声称恢复了原 SSE 的每个事件。

### 报告生成成功不等于报告结论充分

报告是结构化产物，它可以在证据不足时诚实地生成 incomplete 或 needs_human 版本。生成器成功完成，只能证明格式和存储路径工作；根因是否可信还要看 Evidence Sufficiency、Conclusion Alignment、冲突和未知项。把“有报告”当作“有根因”，会奖励模型编造完整故事。

因此，系统监控至少要同时观察传输层、工作流层和结论层：HTTP/SSE 是否可用，工作流以何种状态结束，报告是否通过证据门禁。这三个维度也是面试时解释“成功”的最好切入点。

## 安全默认值体现了怎样的取舍

### 延迟连接 Milvus，避免把可选能力变成启动单点

应用启动时不立即强制连接 Milvus，而是在检索或索引首次使用时按需建立并初始化业务连接；readiness 使用独立临时 client 做无副作用探测。这样，即使向量库暂时不可用，进程仍能启动并暴露健康信息，AIOps 的其他能力也有机会工作。代价是“进程已启动”更不能代表 RAG ready，所以必须有独立 readiness。

这个选择适合多能力应用。如果系统只有一个核心依赖，启动即连接可能更早暴露配置错误；AutoOnCall 同时包含工作台、Incident 查询、AIOps 和 RAG，因此采用懒连接并用能力健康检查补足可见性。它不是唯一正确答案，而是启动可诊断性与快速失败之间的权衡。

### A2A 默认关闭，避免无意增加北向攻击面

代码注册了 A2A discovery 和业务路由，但配置默认 `a2a_enabled=false`。这体现“实现存在”和“运行暴露”分离：仓库可以包含协作能力，部署者仍需显式 opt-in。否则升级版本可能在没有安全评审的情况下突然暴露 Agent Card、Task 或流式接口。

默认关闭不能代替鉴权。启用后仍要通过 read/diagnose scope、网关和网络策略控制。它只减少误暴露概率，不证明协议达到完整 conformance，也不证明第三方 Agent 互操作安全。

### mock fallback 与原始 payload 默认关闭

实时适配器未配置或失败时若自动返回 mock，演示会更顺畅，却会制造最危险的假证据：调用状态显示成功，内容却不是现场。当前配置默认关闭 AIOps mock fallback，让缺失依赖变成结构化失败。生产导向部署更应坚持这一点，只有明确标识的离线评测才允许使用 fixture。

原始外部 payload 默认不持久化，原因是日志、数据库和告警消息可能包含敏感字段。系统优先保留摘要、hash 和必要结构。这个默认值降低数据暴露，却也减少事后深度复查材料；若调试期间临时开启，必须配套访问控制、短 retention 和明确删除流程。

工具持久化边界已经从“主要清洗输入输出”收紧为统一处理 `ToolExecutionResult` 的输入、输出、公开错误和 metadata；本地 MCP monitor/CLS 日志也只保存参数或异常的长度与哈希指纹，不再记录原文。这样能防止 token 从 retry metadata、结构化错误或 MCP server 日志旁路，但未知凭据格式、自然语言 PII 和第三方服务日志仍需数据分类与独立审计。

### 默认值只能降低风险，不能承担治理责任

安全默认值解决的是“没有额外配置时尽量不要做危险的事”。它无法判断组织的合规要求、数据分类、网络环境和应急流程。部署者仍需把配置模板转成经过评审的发布策略，并通过自动检查证明实际环境没有偏离。课程中应把默认值视为 guardrail，而不是认证证书。

### CORS 空配置不应偷偷回退到全开放

CORS 配置过去最危险的便利逻辑是“解析不到 origin 就返回 `*`”：运维人员以为留空代表不开放跨域，程序却悄悄允许所有来源。当前解析会去除尾部斜杠、去重，并在空配置时返回空列表；浏览器 cookie credentials 也保持关闭，因为项目使用显式 API Token，而不是跨域 Cookie 会话。

这里要区分两个概念：CORS 是浏览器同源策略的响应头，不是服务端鉴权；关闭 credentials 也不等于 Token 不会泄露。前端仍需妥善保存 Token，服务端仍需鉴权、网络隔离和 TLS。这个案例的面试价值在于说明：**安全配置的空值语义必须明确，不能用“方便开发”的 fallback 改写运维意图**。

## Redis 案例：从一条告警到一份不过界的结论

### 现场刚出现时，我们只拥有观察

02:07，`order-service` 出现 503，P95 超过 3 秒，日志出现 `Redis connection timeout`，`connected_clients=9940`、`maxclients=10000`。若这些数值来自事故窗口真实适配器，它们是现场 Evidence；若来自冻结 case，它们只是 offline fixture 中的事实。两者文本相同，证据等级却完全不同。

此时可以形成 Incident，把服务、严重级别、时间窗口和标签稳定下来。不能直接宣布“Redis 连接耗尽是唯一根因”，因为还缺少指标与日志的时窗对齐、连接增长原因、线程池等待、网络错误和部署变更等信息。Runbook 只能告诉我们 maxclients 附近通常如何排查，属于知识，不是本次现场证明。

### 下一步为什么不是立刻修改 maxclients

提高 maxclients 可能暂时缓解拒绝连接，也可能把压力转移到内存、文件描述符或下游。诊断系统应先查询 Redis 当前与事故窗口状态、Prometheus 延迟和错误率、Loki 日志、CMDB 拓扑、部署历史与历史工单。每一步都应说明它填补哪个证据缺口，而不是为了展示工具数量。

若实时 Redis 已恢复正常而事故窗口 key 保存了 9940/10000，报告必须明确区分 `live_info` 和 `incident_evidence`。把当前健康状态说成“仍然饱和”是时间语义错误；把历史 key 当成当前实时值同样错误。

### 哪些情况下继续、降级或转人工

当指标、日志和 Redis 状态在同一时窗相互支持，且没有强反证时，可以把“连接资源耗尽”提升为高置信候选根因；仍应保留证据回链和未知项。若 Redis 工具未配置，但日志和 Runbook 指向同一方向，只能输出有限诊断并明确缺少主域证据。若实时状态与事故窗口材料冲突，应继续查询时间边界或降级，而不是通过多数投票消除冲突。

任何修改 maxclients、重启实例或终止连接的动作都进入风险策略与审批。审批通过也不等于生产执行完成；当前项目主要生成 ChangePlan、pre-check、dry-run 和人工结果记录，没有通用生产写适配器和自动回滚。因此最终报告可以提出建议，不能把建议、批准和执行混为一谈。

### 这条主线如何贯穿后续专题

第 01 篇推导受控诊断闭环；第 02 篇解决重复告警如何成为同一 Incident；第 03—05 篇解释知识如何入库、可信检索并生成带引用回答；第 06—09 篇让 Agent 规划、调用工具、分析 Evidence 并形成报告；第 10—11 篇处理身份、审批与执行边界；第 12—14 篇处理 Trace、Snapshot、Replay、评测和外部 Agent 协作。每层解决一个具体失败，也保留自己不负责的部分。

## 从本地演示走向生产，还缺哪些工程闭环

### 身份与暴露面

静态 token 应替换或接入 SSO/OIDC，身份映射到现有 scopes；审批与执行最好由不同主体承担，并支持短期凭证、撤销和密钥轮换。服务应位于内部网关后，由 TLS、WAF、限流、请求大小和网络策略保护。启动门禁可继续作为最后一道配置检查，但不能承担全部平台职责。

### 状态、恢复与多副本

需要验证 MySQL 迁移、连接池和备份恢复，引入 durable LangGraph checkpoint 或等价工作流状态，并保证工具调用幂等。多副本要有任务租约、单次审批竞争控制和 Artifact 共享存储。恢复测试不能只模拟函数异常，应包含进程退出、节点重启、网络分区和依赖超时。

### 数据与隐私

外部日志、工单和数据库结果可能包含凭据、用户信息和业务数据。生产化需要字段级脱敏、最小化保存、加密、访问审计、retention 和删除流程。当前默认不保存原始外部 payload 是安全起点，但摘要仍可能泄露敏感信息，必须按数据分类继续治理。

### 质量与运营

离线回归集需要持续吸收经审核的真实失败案例，防止只优化演示主线；阈值和规则变更要有 candidate/official 晋级流程。线上应监控拒答率、转人工率、工具失败率、证据覆盖和审批等待，而不是只看“完成率”。任何自动动作还需要变更窗口、观察指标、停止条件和回滚责任人。

这些缺口不是否定当前项目价值。相反，能准确说明已完成的闭环和剩余风险，才说明作者真正理解 Agent 系统从 Demo 到生产的距离。

做架构评审时，还可以要求每项亮点同时写出一个反例：检索命中但来源过期、工具成功但时间窗口错误、审批通过但没有执行回执、Replay 完整但无法恢复节点。正例说明系统会做什么，反例说明边界在哪里；两者一起出现，课程内容才不会退化成产品宣传。对校招面试而言，这种主动限定往往比堆叠框架名称更能证明工程判断力。

## 一套可反复使用的防夸大翻译表

后文把一次需要调查和处置的结构化故障称为 **Incident（故障事件）**：它解决告警消息如何成为稳定协作对象，不负责证明根因；本篇 Redis 503 可以形成一个 Incident，十条重复 firing 各自建十个 Incident 则是反例。**Trace（轨迹）**记录系统做过什么，不负责崩溃恢复；**Snapshot（快照）**保存某时刻最新状态，不是完整历史；**Replay（回放读模型）**聚合历史对象供复盘，不会重新执行工具；**Trust Gate（可信门禁）**按阈值决定检索候选能否进入回答，不是通用事实核验。这些对象在各专题首次成为主角时还会给出完整定义与项目映射。

| 容易说过头的表达 | 当前准确表达 |
|---|---|
| 通用自主根因分析平台 | 面向限定信号和工具集合的受控 Incident 诊断 Agent |
| 告警 exactly-once | 严格校验、指纹幂等、AlertEvent/IncidentState 单条事务投影；后台诊断仍为进程内去重与并发限制 |
| 事件溯源状态机 | 最新 IncidentState 投影、终态/审批/change 回退守卫与显式 alert reopen；没有完整迁移历史 |
| 模型化 reranker | 规则加权融合或 RRF |
| 通用 OOD 检测 | Trust Gate、阈值和无可信来源拒答 |
| 逐句事实核验 | 引用 ID 契约与生成前后引用检查 |
| 通用因果推理 | 规则、关键词和静态信号目录驱动的 RCA 基线 |
| 任意节点持久化恢复 | MemorySaver、latest Snapshot 与审批恢复 fallback |
| 事件重放 / time travel | 聚合多个存储对象的 Replay 派生读模型 |
| 真实沙箱和自动回滚 | 结构化 dry-run、人工结果记录与本地 fixture sandbox |
| 生产准确率 / 生产 MTTR | 指定数据集与指定环境下的离线或本地指标 |

回到 Redis 案例：`connected_clients=9940`、`maxclients=10000` 若来自 fixture，它是离线事实；Redis 官方文档是知识；“连接耗尽导致 503”是需要现场证据支持的推断。缺少本次事故窗口指标时应继续取证；工具未配置、证据冲突或风险动作出现时，应停止、降级或转人工。

**一句话定位**

**30 秒回答**

**3 分钟展开**

**可写入简历**

> 为 FastAPI 诊断服务补充静态 token/scope RBAC、liveness 与 RAG/AIOps 分能力 readiness；将非本地绑定下的无鉴权、无可用 Token、wildcard CORS 和 mock fallback 纳入 fail-closed 启动门禁，并关闭生产 Swagger/OpenAPI 暴露。

**不可夸大**

> 完成企业级生产安全体系和高可用部署，保证服务全年稳定运行。

### 高频追问

1. **测试通过为什么不是生产证据？** 测试只覆盖声明的输入和替身，不能代表真实依赖、并发、容量与治理环境。
2. **`local_live` 与 `controlled_fault` 有何区别？** 前者强调本地真实依赖连通，后者强调人为触发的特定故障；二者都不是生产流量。
3. **dirty worktree 有什么影响？** 结果仍可作为 candidate，但代码状态无法仅由 commit 完整复原，不能晋级 official baseline。
4. **为什么 liveness 与 readiness 分离？** 进程可响应并不代表 Milvus 或 AIOps 适配器可服务，调度决策需要更细的信号。
5. **Dockerfile 为什么不能证明生产化？** 镜像不包含身份、网关、编排、备份、扩缩容和灾备的运行证据。
6. **怎样把离线指标升级为生产证据？** 冻结定义和数据，记录版本、命令、Artifact，在真实治理环境收集足够且可审计的 Incident 样本，并保留失败案例。
7. **为什么“配置了 URL”仍不能 ready？** 配置存在只证明程序知道去哪里调用；DNS、网络、证书、权限和远端状态仍未验证，因此只能标记 `unverified`。
8. **关闭 Swagger/OpenAPI 是否等于接口安全？** 不是。它减少枚举面，但真正的边界仍是鉴权、授权、网络隔离、TLS 和审计。

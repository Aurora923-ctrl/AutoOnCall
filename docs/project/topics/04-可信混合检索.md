# 为什么“Top-K”还不够：把检索变成可拒绝、可追踪的证据入口

> 专题编号：04
> 审计日期：2026-07-22
> 上一篇：[运维知识如何可靠入库](03-运维知识如何可靠入库.md)已经把文档变成带稳定身份、标题与新鲜度信息的知识条目
> 下一篇：[可审计的检索增强回答](05-可审计的检索增强回答.md)将把本篇允许使用的证据变成有引用、可修复也可拒答的回答
> 事实范围：2026-07-22 当前工作区的实现和自动化测试，包含未提交改动；这不是生产查询分布、线上可用性或真实 Milvus 延迟的证据。

## 02:07 的问题不是“搜 Redis”，而是“凭什么现在能给出这句话”

凌晨 02:07，`order-service` 出现 503，P95 抬升；日志里有 `Redis connection timeout`，监控显示 `connected_clients` 接近 `maxclients`。值班员问：

> 先看哪些证据？历史上是否发生过类似连接压力？现在能否直接扩容或重启？

一个朴素的向量 `Top-K` 会认真地交出几个“最接近”的片段，却没有回答三个更重要的问题：这些片段是否真与问题有关，是否分别覆盖了“机制、历史、处置边界”，以及它们是否足以支撑一条面向当前 Incident 的结论。

这里要先把三种东西分开。知识库中的 Redis 官方快照或历史复盘描述的是**可复用知识**；本次 `connected_clients`、客户端连接池和最近发布是需要监控、日志、变更系统查询的**现场证据**；“连接容量导致 503”则是待验证的**推断**。可信检索只负责决定哪些静态知识可以进入后续回答，既不能把历史复盘说成当前现场，也不能替代后文的 Evidence、审批和变更流程。

因此，本篇把检索拆成一条有停止条件的链：

```text
问题规划 → 候选召回 → 定向来源探测 → 融合排序
       → 元数据/Trust Gate → 来源与标题覆盖 → 可生成证据
       → 非流式：回答覆盖检查、引用修复、抽取式兜底或拒答
       → 流式：citation prefix 检查、终态 finalize、替换或拒答
```

其中“排序靠前”和“允许说”是两个不同决定。前者是相对比较，后者是绝对准入；把它们混成一个分数，会让系统在没有可靠依据时仍然自信地回答。

## 先把自然语言问题拆成需要被证明的子目标

同一句“Redis 快满了怎么办”可能只是要解释机制，也可能要求历史对照，甚至暗含高风险处置。当前链路先从问题中识别领域和显式实体：例如 Redis 容量问题会保留 `connected_clients`、`maxclients`、`effective_capacity`、`blocked_clients`；`payment-service` 的慢查询问题会保留服务名、`pool_waiting`、`active_connections`、慢查询和 `EXPLAIN`。

再按问题措辞生成子目标。要求检查、指标或日志时有 evidence；要求判断、区分或根因时有 diagnosis；明确询问重启、扩容、回滚、删除等动作时才有 boundary；问历史、复盘、发布或 incident window 时才有 temporal boundary。显式要求“官方限制和事故复盘”还会提出 official、postmortem 两类来源角色。没有动作请求的慢查询排查，不会因为知识片段提到了审批就被硬塞成变更问题。

这一步不是让模型先编一份计划，而是确定性的 `QuestionPlan`：它限定实体、子目标和回答声明上限。它有两个直接收益：

- 检索阶段知道不能只找一篇“Redis 概述”，还要为要求的来源和标题留出候选空间；
- 生成阶段知道哪些问题必须被回答、哪些现场信息缺失时只能写“当前证据不足”。

这也解释了为什么“官方文档 + 历史复盘”不能合并成一个模糊的引用。官方材料用于说明限制或机制，复盘用于说明历史假设、排除过程或风险模式；两者都不能替代此刻 incident window 的指标和日志。

## 召回先求不漏，再谈谁排第一

向量检索擅长把“连接槽位耗尽”与“客户端数触顶”放在一起，却可能稀释 `maxclients`、错误码、命令等精确运维符号；词法检索擅长这些符号，却不理解同义表达和跨语言说法。因此代码仍保留向量候选与本地 BM25-like 词法候选的合并能力，并以稳定的 source/chunk identity 去重。

但“具备双路实现”不等于“默认线上每次都会两路查询”。这是当前最容易讲错的边界：

- 默认 `rag_retrieval_service` 路径标识运行时向量后端为 `milvus`，由 Milvus 负责权威向量检索；默认路径将 `allow_lexical_fallback` 设为 false。Milvus 无法取得时结果为 `failed`，不会悄悄改用本地词法索引。
- 显式注入 vector store/provider 的测试、评测或调用路径，默认允许词法补充或降级。因此可以验证“向量失败而词法仍越过门槛”以及反方向的降级语义，但不能把它包装成当前默认生产路径的高可用承诺。
- 词法索引仍承担本地索引、stale 查询、显式注入链路和离线验证的职责；它不是当前默认 Milvus 运行路径的同等权威副本，更不意味着双索引跨存储事务一致。

候选预算也大于最终 `top_k`。默认配置的 `top_k=3`、candidate multiplier `4` 意味着先在更宽的候选池内搜索、合并和门禁，最终才返回少量片段。这避免了“先截成三条，再发现其中两条不可信”时无法从后续候选补位的问题。

## Milvus 仍是权威后端时，怎样补齐指定来源

多来源问题的难点不是把本地词法结果硬塞给 Milvus 结果，而是让权威向量库知道要寻找哪一类资料。当前运行时在 Milvus 首次全局搜索后，会根据显式问题偏好确定 required source 和定向查询词，并使用 `_file_name` 的精确 metadata expression 对 Milvus 做来源范围内探测；然后以 source/chunk identity 保留距离更小的重复命中。

例如用户明确要求 Redis 官方限制和事故复盘，候选池会要求 `official_redis_clients.md` 与 `redis_postmortem.pdf`；带 `payment-service`、发布历史和回滚判断的问题会探测 `tickets.xlsx`。标题型问题也会扩展出面向特定 Runbook 的查询语句。函数名中有“lexical queries”的历史命名，但在这条默认路径里它提供的是**定向查询提示**，实际检索仍由 Milvus 完成。

这项设计的取舍是清楚的：它提升了多来源召回和标题覆盖，却增加了向量查询次数，因而不能把一次 `milvus_search_ms` 理解成单次 ANN 请求延迟。它也只对代码中明确识别的来源和问题模式生效，不是任意自然语言都能可靠推导出全量来源约束的通用规划器。

## 融合只负责排序，不能伪装成可信度

候选合并后，每个 chunk 会保留来源信号。Milvus 的原始 `score` 是 L2 distance，**越小越好**；词法原始分数会被换算为有界 lexical relevance，**越大越好**。因此不能直接把两者相加，更不能把最终 `rerank_score` 当作答案正确概率。

当前 weighted 策略把归一化向量相关度 `1 / (1 + distance)`、词法相关度和原始名次基准分别按 `0.55 / 0.35 / 0.10` 组合，再乘以面向来源类型、标题、显式实体等的 intent multiplier。RRF 是另一种可选策略：它只累加各路 rank 的倒数，不依赖原始分数尺度。两者都只是把候选排得更合理：

- weighted 能表达“精确命中”和“指定官方来源”这样的规则偏好，但权重需要随语料、模型和任务重新评测；
- RRF 对异构分数尺度更稳健，却丢失了“第一名领先多少”的强度信息；
- RRF 为正只说明某路给过名次，绝不说明该 chunk 可以进入回答。

对于问证据、判断、审批或回滚的查询，当前实现会优先考虑标题覆盖，避免相对 70% relevance floor 把唯一的“首轮证据”或“升级与审批”段落裁掉。要求多来源时也可以先保留不同来源；要求某一特定来源时，会先在该来源内挑选有语义内容的片段，避免以文档元数据占掉最后一个席位。这里的 heading coverage 是由查询标记、预设标题词组和顺序规则驱动的有限选择，不是标题语义模型；整套选择也不是 reranker，没有 Cross-Encoder 或学习排序模型。

## Trust Gate：先验明“可以用”，再裁最终 Top-K

融合完成后，每条候选必须通过独立的 Trust Gate。它不看 `rerank_score` 是否漂亮，而是检查以下条件：

1. `source_file`、source id、chunk id、正文和后端来源类型完整，且同一 identity 没有映射到冲突正文；
2. stale 注册表没有把来源标为已过期；stale 查询本身异常时按 fail-closed 处理为不可用；
3. metadata filter 只接受受限的 exact/in 标量规则，不能构造安全表达式就失败；后端不支持表达式时不撤销过滤重试；
4. 向量或 hybrid 命中要有可解析的 L2 distance 且不超过 `max_l2_distance`；仅有 rank、没有距离，不会被视为可信；
5. lexical 或 hybrid 命中要有 lexical score 且不低于 `min_lexical_trust_score`。

当前默认门槛为 L2 `≤ 2.0` 和 lexical `≥ 0.20`。它们是此工作区当前配置和测试契约，不是跨模型、跨向量维度、跨 chunk 粒度都成立的自然常数。修改 Embedding、距离度量、词法索引规模或知识资产后，应重新用含近域负例和域外负例的冻结集校准，而不是继续引用旧门槛或旧分数。

门禁在最终 `Top-K` 前运行。若高排候选因距离、元数据或 stale 失败被拒绝，后面已越过绝对门槛的候选仍可补上；若要求的来源经过门禁后缺席，`enforce_source_coverage` 会清空该回答的可信候选并记录缺失来源，而不是强行拿相邻资料凑齐。对不包含 OnCall 信号、与候选又没有词项交集的问题，当前还有一个有限的 out-of-scope 检查。它能阻止一部分“旅行问题也回答 Redis”，不是可泛化的 OOD 分类器。

## “成功”也可能是显式降级，“没有答案”不是“系统坏了”

检索状态至少要区分三种：

| 终态 | 含义 | 后续行为 |
|---|---|---|
| `success` | 存在通过 Trust Gate 且满足必要来源覆盖的 chunk | 交给下一篇的生成与引用链路 |
| `no_answer` | 后端可用，但没有可信 chunk，或必要来源无法覆盖 | 用受控拒答说明知识不足，不编造答案 |
| `failed` | 检索基础设施或请求校验失败 | 返回公开错误语义；默认 Milvus 运行路径不以本地词法静默兜底 |

在允许词法降级的显式注入链路中，向量失败且词法仍通过门槛，可返回 `lexical_degraded...`；词法失败而向量候选足够，则为 `vector_degraded...`。两个方向都会保留 `retrieval_degraded`、模式和公开错误消息。降级不是信任放宽：词法路径仍要过 lexical 门槛，向量路径仍要有合格 L2。两路都不可用才是失败；两路可用但都弱，才是 no-answer。

这种分层使“拒答”成为正常控制流，而不是异常吞掉。它也阻止了一个常见的错误演示：把本地测试的 lexical fallback 成功，讲成 Milvus 故障时生产聊天仍保证可回答。当前代码与测试支持的是可观察、可验证的路径边界，不是容量或可用性 SLA。

## 覆盖不是多塞几个 chunk，而是让每个子问题有可追溯证据

Trust Gate 通过的 chunk 还不自动构成可生成上下文。当前回答前置链路会以 QuestionPlan 和字符预算构造 FrozenGenerationEvidence：每个被选片段绑定 subgoal、source role、匹配实体和稳定 citation index；选中的 excerpt 固化一次，生成、引用检查和上下文格式化复用同一份证据，而不是各阶段各自截取一段文本。

必要来源角色或指定来源在预算内放不下时，冻结证据为空并拒绝生成；这是宁可少答也不让模型引用未进入上下文的材料。若只覆盖了“原因判别”而没有“回滚边界”，系统保留已覆盖的信息，同时把 boundary 标成缺口——缺口不是允许模型补常识的邀请，而是要求明确说“当前证据不足”。

`answer_coverage` 是面向回答的矩阵：它为 evidence、diagnosis、alert design、boundary、temporal boundary 分别记录是否有 chunk 绑定、绑定到哪个来源和原因。在**非流式 `query_with_retrieval`** 中，生成后会检查回答是否真的触及这些子目标：遗漏必要来源的引用会进入一次定向重试，遗漏回答子目标会进入一次覆盖修复重试；引用校验仍失败时还会执行独立的引用修复重试，并尝试以已有 frozen evidence 组织抽取式回答，最终仍不能形成 allowlist 内逐条引用才拒答。这些是顺序执行的多条独立修复分支，不是受一个统一 retry budget 管理的单次闭环，因此一次请求可能发生多次模型生成。“修复”只允许重新组织已批准的证据，不能调用工具补现场数据，也不能绕过 Trust Gate。

流式链路没有复用上述必要来源修复、子目标覆盖修复或抽取式兜底。它只对已产生的流式文本维护 citation prefix，并在终态执行 finalize；校验不通过时可以替换已有输出或拒答。因而“受控重试和抽取式兜底”只能作为非流式能力陈述，不能泛化为聊天服务所有生成模式都具备的统一质量合同。

有一个必须诚实保留的边界：`AnswerContract`、QuestionPlan、FrozenGenerationEvidence 和覆盖矩阵都已在当前生成 guard 中构建并被测试；但 claim-level `validate_answer_contract` 目前主要是独立契约/单元测试能力，未见非流式主回答循环将其违例作为最终拒答或再次重试的强制门。因而可以说“主链已接入问题规划、冻结证据、来源与子目标覆盖修复”，不能说“所有 claim-level 合同违例已在运行时强制阻断”。

## 可观测性：先识别后端，再解释耗时和结果

一次检索的 payload 会记录 `vector_backend`/`retrieval_backend`、模式、候选数、合并去重数、stale 与 metadata 过滤数、trusted/rejected 数、required/missing sources，以及 `answer_coverage`。默认运行时标签为 `milvus`；显式注入测试路径标为 `injected`，避免把 fake store 的结果误称线上 Milvus。

阶段耗时包含 `vector_search_ms`、`lexical_search_ms`、`fusion_rerank_ms` 和 `retrieval_total_ms`。当运行时后端是 Milvus 时，`milvus_search_ms` 复用合并后的 vector search 时间；`embedding_ms` 仍是 `not_observed`，collection version 也是 `not_observed`。原因并不神秘：当前 LangChain Milvus 调用将 query embedding 与相似度搜索包在一起，代码没有把它们拆成可独立测量的计时点。把这一个值写成“纯 Milvus ANN 延迟”或“Embedding 延迟”都不成立。

聊天 API 会把后端身份、检索查询耗时和命中的 source/chunk 摘要写入 trace metadata。它有助于排查“这次回答来自哪个运行路径”，但不是日志已经证明线上质量、成本或用户满意度。生成侧会合并重试观察；只有 provider 实际返回 token usage 时才标为 observed，成本没有价格快照时保持 `not_observed`。

## 用失败用例校验边界，而不是只展示漂亮排序

以下情形比“Redis 问题答对了”更能检验可信检索：

1. **域外问题**：候选总会有最近邻；没有 OnCall 范围和足够词项支持时应 no-answer，而不是从 Redis 文档编答案。
2. **无距离的向量命中**：排名存在不代表距离存在；当前门禁拒绝 unscored vector result。
3. **RRF 正分**：合法 rank 会产生正值；若 L2 与 lexical 都没过门，仍必须拒绝。
4. **指定来源缺席或 stale**：required source 不绕门禁；门禁后缺任一指定来源就不产生可信答案。
5. **metadata 或 stale 检查失败**：当前选择 fail-closed，牺牲部分可用性以避免错误放行；这不等于已经解决跨存储更新、首写失败或进程中断的一致性窗口。
6. **默认 Milvus 不可用**：应得到带后端身份的 `failed`，而不是误把本地词法结果宣传为运行时回退成功。
7. **回答引用看似正确但没有覆盖问题**：非流式 `query_with_retrieval` 会沿必要来源、子目标和引用校验的独立分支尝试修复，并可退到抽取式、仅证据化输出；流式只做 citation prefix 与终态 finalize，失败时替换或拒答。

这些行为由检索、问题规划、冻结证据、覆盖、生成 guard、聊天 API 和相关单元测试共同约束。自动化测试证明的是这些确定性分支和 payload 合同，不证明真实大语料的 Recall、在线 Embedding/Milvus P95、模型生成稳定性或生产故障处置效果。

## 面试时，把它讲成四道独立闸门

30 秒版本可以这样说：

> 我没有把向量 Top-K 当答案。系统先从问题提取实体、证据和动作边界，再在较宽候选池中做向量召回；默认运行时以 Milvus 为权威，对显式来源做受限 metadata 定向探测。融合只改善相对排序，真正准入要经过来源身份、stale、metadata、L2 或 lexical 的 Trust Gate。随后按来源和有限标题规则冻结生成证据；非流式链路对缺失的必要来源、回答子目标和引用分别尝试修复，并可做抽取式兜底，流式链路则只做 citation prefix 和终态 finalize。默认 Milvus 路径失败闭合，本地词法降级仅在显式注入的测试/评测路径允许。

追问时应主动说明四个限制：默认阈值不是通用校准值；RRF/weighted 分数不是概率；静态复盘不是当前 Incident 证据；测试覆盖分支不等于生产延迟和真实召回证明。下一篇要解决的正是：即使 chunk 可信，怎样让每条回答只引用冻结证据、覆盖问题，并在无法满足时拒绝。

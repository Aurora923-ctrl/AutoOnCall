# 为什么向量 Top-K 仍不可信：双路召回、融合与 Trust Gate

> 专题编号：04  
> 审计日期：2026-07-13  
> 上一篇：[运维知识如何可靠入库](03-运维知识如何可靠入库.md)产出了带来源、版本与 stale 状态的双索引  
> 下一篇将解决：trusted chunks 怎样变成带引用、能拒答且不越过工具边界的回答  
> 本篇事实范围：当前代码、自动化测试和带 provenance 的冻结评测；不代表生产查询分布或线上 Milvus 延迟

## 排名第一，只能证明它比候选池里的其他内容更像

凌晨 02:07，`order-service` 出现 503，P95 上升，日志持续出现 `Redis connection timeout`，监控显示 `connected_clients` 接近 `maxclients`。用户问：“Redis 接近 `maxclients` 时先查什么，扩容前有哪些审批边界？”

这里的 **Incident（故障事件）**通俗地说是正在调查的一次事故；严格定义是由服务、环境、严重级别、时间窗和症状组成的结构化上下文。它负责界定“我们在讨论哪次异常”，不负责证明根因。503、P95 与现场 Redis 指标是待核验事实；官方资料与内部 Runbook 是知识；“连接耗尽导致 503”是推断；客户端连接池、最近变更和完整事故窗口仍是未知。

如果直接做向量 `similarity_search(k=3)`，系统一定尽量返回三个“最近”的片段。知识库即使完全没有 PostgreSQL 主从切换内容，向量库也会选出与“数据库、主从、故障”较近的 Redis 或 MySQL 文档。Top-K 的含义是相对排序，不是“前 K 条都足以回答”。

同一 Redis 查询还暴露另一类问题：`maxclients`、`connected_clients`、`CONFIG GET`、503 等精确符号可能在语义向量中被通用“容量、连接、服务不可用”稀释；只用关键词又会漏掉“连接槽位耗尽”“客户端达到并发上限”等同义表达。Embedding 或 Milvus 故障时，单路向量链还会整体失效。

因此可信检索不能只问“谁排第一”，而要连续回答三个不同问题：

1. 候选池里有没有可能相关的片段？
2. 多路候选怎样排出更合理的相对顺序？
3. 排名靠前的片段是否越过允许进入回答的绝对底线？

这三个问题分别对应召回、融合排序和 Trust Gate。把它们压成一个“相关度分数”，是许多 RAG 系统看似简洁、实际无法解释强答与拒答的根源。

## 从三个失败版本推导三阶段检索

### 只做向量：懂同义表达，却会丢精确运维符号

向量检索擅长语义相似，能把“连接容量耗尽”和“客户端数量触顶”放在一起。但配置键、错误码、SQL、命令和产品名是运维问题中的高信息密度信号。通用 503 Runbook 的向量距离可能优于 Redis 专项文档，仅因为它复述了更多“服务不可用、超时、故障”语义。

更重要的是，向量 Top-K 没有天然的“都不相关”。除非增加绝对阈值或专门 OOD 模型，它总能从有限库中找最近邻。第一名可能只是“最不坏”。

### 只做关键词：精确命中，却不理解表达变化

关键词对 `maxclients` 很强，但用户可能只说“Redis 新连接全部超时，容量槽位是不是满了”。中文分词、英文标识、大小写、斜杠和冒号也会影响召回。只用词法会把完全相同的符号看得很重，却难理解同义、上下位和跨语言表达。

### 把两路分数直接相加：尺度不同，结果没有共同含义

Milvus L2 是距离，越小越好；词法分数通常越大越好；原始范围也不同。直接相加会把数值尺度误当证据。即使归一化后可以排序，融合分仍混合了相对名次、规则偏好和多路强度，不能直接解释为“可信概率”。

由此得到课程中的第一条架构结论：扩大召回、改善相对排序、决定是否准入，必须是三个可独立测试的阶段。

## 召回、融合和门禁各自负责什么

### Retrieval：先扩大“可能有答案”的候选集合

**召回（retrieval）**通俗地说是先把可能有用的材料捞上来；严格定义是根据查询、过滤条件和候选预算，从一个或多个索引取回候选集合。它负责提高相关项进入可见范围的机会，不负责证明第一名正确，也不负责内容来源可信。Recall@K 主要评价这一层。

AutoOnCall 从 Milvus 语义索引和本地词法 JSON 索引取候选。默认 `top_k=3`，但召回数量不是 3，而是：

```text
candidate_k = top_k × rag_hybrid_candidate_multiplier
```

默认 multiplier 为 `4`，所以默认候选预算是 12。先扩大候选，再融合裁剪，避免某一路前三名直接垄断最终上下文。

### Fusion/Rerank：只决定候选之间谁更适合当前查询

**融合排序（fusion/rerank）**通俗地说是综合多路信号重新排队；严格定义是将向量距离、词法强度、各路名次、基础位置和有限意图规则变成可比较的相对排序。它负责谁先进入 Top-K，不负责把 `rerank_score` 解释为概率，也不证明候选足够可信。MRR、MAP 和 nDCG 主要评价这一层。

### Trust Gate：排名靠前仍要通过绝对准入

**Trust Gate（可信门禁）**通俗地说是“第一名也要够强才准用于回答”；严格定义是基于候选原始向量距离或归一化词法信号的确定性接受/拒绝规则。它负责阻止 Top-K 必然返回结果造成的强答，不负责验证生成答案的每一句是否被文本蕴含，也不是训练出来的通用 OOD 分类器。

例如一个通用 503 文档融合后排名第一，但 L2 为 `2.4300`，高于 `2.0000`，词法为 `0.1200`，低于 `0.2000`，它仍必须进入 rejected。只有 trusted chunk 才能作为下一章的回答上下文。

## 双路召回并不是“同时调用两个搜索函数”这么简单

### 向量路：距离必须来自后端，不能用排名伪造

向量链通过 `langchain_milvus.Milvus` 使用查询 Embedding 搜索候选，优先调用 `similarity_search_with_score` 取得尽可能真实的 L2 distance。若后端只提供无分数的 `similarity_search`，候选仍可参与相对排序，但默认不能通过向量 Trust Gate；测试明确要求返回“检索后端未返回距离分数”。系统不会把“第一个返回”伪装成可信距离。

L2 越小越接近。排序时使用：

```text
vector_score = 1 / (1 + distance)
```

这个转换只为融合得到 0–1 范围，原始 distance 仍保留并用于门禁。无分数输入在排序辅助函数中有兼容值，不会因此获得准入资格。

Vector Store 与 Embedding 采用惰性初始化：构造 manager 不立即连接 Milvus，没有文档时 `add_documents([])` 也不触发连接。维度不匹配默认不会删除 collection，只有显式允许才重建，防止启动时误删数据；健康探测失败会关闭陈旧客户端。这些是基础设施安全边界，不代表 Milvus 已具备生产 HA。

### 词法路：紧凑 BM25-like，不是完整搜索引擎

词法索引把 ASCII token 与连续中文二元、三元片段放入集合，用 TF、IDF 和长度归一思想计算紧凑 BM25-like 分数，参数中 `b=0.75`。索引以 JSON 保存，写入使用临时文件加原子替换；同一来源重建成功会清除 stale 标记。

为什么只能叫 BM25-like？特征集合会丢失完整词频语义，也没有搜索引擎成熟的 analyzer、倒排压缩、分片、近实时刷新和复杂查询语言。它的价值是本地确定、可测试、能补精确符号与故障降级，不是 Elasticsearch 的等价替代。

词法索引原始分在融合时按 `/5` 缩放并截断到 `1.0`。若候选没有索引分，系统使用查询词与来源、标题、内容的重叠生成有界分数。门禁读取的是候选最终 `lexical_score`，默认阈值 `0.20`。

### 为什么注入测试 Store 时不总查全局词法索引

生产主路径使用全局 vector store 时会同时受益于本地 lexical。测试和离线 evaluator 常注入一个小型 store 来精确构造阈值案例；如果此时仍混入全局词法索引，测试结果会受开发机历史状态污染。因此注入 store 且已有向量结果时，默认不附加全局 lexical；只有向量无结果等条件才触发。更严格地说，显式注入的 vector store 抛错也不会自动使用全局 lexical degraded，避免测试依赖泄漏。

这不是生产 Hybrid 的能力缺失，而是依赖注入下的确定性边界。面试中应能区分“运行模式”与“测试隔离策略”。

## 合并候选时，身份比文本相似度更重要

向量路与词法路可能命中同一片段。如果不去重，一个 chunk 会占两个 Top-K 席位，让“多路同意”错误地变成“两条独立证据”。AutoOnCall 先按 `(doc_id, chunk_id)` 合并，缺稳定身份时才使用内容哈希 fallback。

合并后保留：

- `_vector_score` 与 `_vector_rank`；
- `_lexical_score` 与 `_lexical_rank`；
- `_retrieval_source = vector | lexical | hybrid`；
- 来源文件、路径、标题路径、chunk ID 和文档元数据。

同一片段被双路命中时标为 hybrid，仍只占一个席位。这里的多路一致性是一组排序信号，不是两份独立事实。

稳定 chunk ID 也支撑后续 citation。若入库元数据缺 ID，检索层可根据来源、标题和内容生成 `chunk-<hash>`；这保证本次响应结构稳定，但真正可长期复现仍依赖上游文档版本和 chunk hash。

## 过滤不是附加功能，而是可信来源选择的一部分

### metadata filter 只支持安全键和精确条件

用户或服务可提供 metadata filter。系统只接受匹配 `[A-Za-z_][A-Za-z0-9_]*` 的键，值支持单值精确相等或列表 `in`。非法键会被忽略并记录 warning，而不是让整次请求失败。Milvus 表达式会对字符串转义；非 Milvus/测试 store 还会执行本地 post-filter，保持语义一致。

后端若不接受 `expr` 参数，兼容路径会重试不带 expr 的搜索，再由本地过滤剔除不匹配项。这提高兼容性，但意味着候选召回阶段可能多取无用项，不能把它描述成所有后端都完成原生 predicate pushdown。

### stale 表示旧版本不再允许进入可信链

**stale（陈旧）**通俗地说是“旧索引已经不能代表当前文件”；严格定义是来源更新过程中双写失败后，在词法索引登记的不可用标记。它负责阻止旧 lexical 和旧 vector chunk 继续参与回答，不负责删除所有物理数据。

检索 lexical 时直接排除 stale source；向量候选返回后，也通过同一 stale registry 过滤。因此即使 Milvus 仍残留旧向量，只要来源被标 stale 就不能进入 trusted 或 rejected，它会直接从候选中消失。重建成功后清除标记，来源才重新可见。

当前边界是：查询 stale registry 若异常，代码记录 warning 并 fail-open 返回“非 stale”。这避免注册表故障拖垮所有检索，却可能让旧向量重新可见。生产环境若把版本一致性置于可用性之上，应考虑 fail-closed、把 stale 状态复制到向量 metadata，或使用可事务化的索引清单。

## 手算 weighted fusion，理解 0.747 不代表 74.7% 正确

默认 weighted 策略先计算向量、词法和基础候选位置三个信号：

```text
weighted = 0.55 × vector_score
         + 0.35 × lexical_score
         + 0.10 × base_rank_score

rerank_score = weighted × intent_multiplier
```

其中 `base_rank_score = 1 / 当前候选位置`。下面只是教学假设，不是历史评测数据：

| chunk | L2 | vector | lexical | base |
|---|---:|---:|---:|---:|
| Redis 容量 Runbook | 0.50 | 0.667 | 0.80 | 1.00 |
| 通用 503 Runbook | 0.30 | 0.769 | 0.10 | 0.50 |

```text
Redis = 0.55×0.667 + 0.35×0.80 + 0.10×1.00 ≈ 0.747
503   = 0.55×0.769 + 0.35×0.10 + 0.10×0.50 ≈ 0.508
```

通用 503 的向量距离更近，Redis 专项内容仍因精确词法和基础信号领先。这正是 Hybrid 的价值。但 `0.747` 只是当前公式下的相对排序值，权重和 intent 改变后尺度也会变化；不能写成“74.7% 的概率正确”。

weighted 的优势是能利用信号强度和领域规则，代价是权重、归一化和规则需维护。0.55/0.35/0.10 不是数学最优常数，而是需要冻结数据集持续验证的工程配置。

## RRF 为什么故意忽略原始分数强度

可选 **RRF（Reciprocal Rank Fusion，倒数排名融合）**只依赖多路名次：

```text
RRF = Σ 1 / (60 + rank)
```

当前 ranks 包括向量名次、词法名次和基础候选名次。它适合各路分数尺度难以校准的情况：只要同一 chunk 在多路都靠前，就得到更高分。缺点是看不到第一名比第二名究竟领先多少，也不使用 weighted 的 `intent_multiplier`。

RRF 排第一仍不能绕 Trust Gate。自动化测试专门构造了 `rrf_score > 0` 但原始向量和词法都不达标的候选，结果仍为 `no_answer`。这再次说明融合与准入是两层逻辑。

weighted 和 RRF 没有脱离数据的“唯一正确”。如果向量距离和词法分已经稳定可比，weighted 能利用强度；如果来源多、尺度差异大，RRF 更稳。选择应比较 Recall、MRR、nDCG、拒答和延迟，不应只挑一个演示案例。

## 意图偏置只能轻推排序，不能制造相关性

`intent_multiplier` 读取明确的文档类型、扩展名、来源词、标题词和少数领域词。例如用户明确问复盘时偏好 PDF，问审批/回滚时偏好相应标题，包含 `maxclients` 时偏好 Redis 来源并压低通用 503。历史工单查询可以偏好 CSV/表格记录，而不是泛化 Runbook。

规则只对 weighted 生效；RRF 中 multiplier 固定为 1。它是透明的业务 reranker，不是 Cross-Encoder 或 LLM reranker。规则倍数可能连乘，因此必须由 Trust Gate 阻止“规则推高一个原本很弱的候选”。

targeted lexical query 也只扩展明确子目标。普通“Redis timeout”不会凭空定向多个固定文件；用户显式要求 Redis 官方说明与事故复盘、Kubernetes Pods 与 Services 调试、Loki 摄取与 Prometheus 告警实践，或明确 CPU/内存/磁盘 Runbook 小节时，才生成保守来源查询。它是硬编码规则表，不是通用 query expansion 模型。

## required source 保留席位，也不能绕过 Trust Gate

对少量已知多来源问题，`required_sources` 会在排序阶段为每个指定来源保留一个席位。例如：

- `official_redis_clients.md` + `redis_postmortem.pdf`；
- Kubernetes Pods + Services 调试文档；
- Loki 摄取排障 + Prometheus 告警实践。

“保留席位”只防止一个来源把 Top-K 占满。候选随后仍要通过 L2/lexical 绝对阈值；不存在或太弱的指定来源不能凭规则进入 trusted。required source 模式还会跳过默认 70% 相对地板，防止较弱但已通过绝对门槛的第二来源被裁掉。

当前能力不支持任意“请结合三份 Runbook”的动态来源约束，也没有求解相关性与多样性的通用优化。规则没覆盖时，只能按普通相关性返回并诚实说明来源数量。

## Trust Gate 为什么看原始信号，而不是融合分

默认可信条件是二选一：

1. 存在向量信号，且原始 L2 distance `<= 2.0`；
2. 来源为 lexical 或 hybrid，且归一化 lexical score `>= 0.20`。

向量分数缺失、不可解析或大于阈值，不能通过向量门；纯 vector 候选也不能借查询词重叠通过 lexical 门，除非它真实来自 lexical/hybrid。这样避免把 reranker 为排序计算的辅助信号冒充召回来源证据。

通过项进入 `retrieval_results`，失败项进入 `rejected_results`，每条带明确 `retrieval_reason`。如果 trusted 为空，返回：

```text
status = no_answer
answer_policy = refuse_without_trusted_source
no_answer_rejected = true
```

这与系统异常不同。`no_answer` 表示链路工作正常，但没有候选达到知识准入线；`failed` 表示检索基础设施或代码无法完成检索。下一章应分别把前者转成拒答，把后者转成服务降级，而不是统一写“没搜到”。

weighted 且没有 required source 时，trusted 还会按最佳 `rerank_score` 的 70% 做相对裁剪，再限制 Top-K。绝对门防垃圾，相对地板防“勉强及格但远弱于第一名”的内容污染上下文。两者职责不能互换。

## 把一次检索写成状态机，才能看清每个失败属于哪一层

如果只看最终 `retrieval_results`，很容易把“没候选”“候选太弱”和“基础设施失败”混为一谈。更好的心智模型是把检索看成六段状态转换：

```text
query + filter
  → raw vector / lexical candidates
  → merged unique candidates
  → ranked candidates
  → trusted + rejected
  → relative pruning / source reservation
  → success | no_answer | failed
```

第一段规范查询和 filter。空值被清理，非法 metadata key 被忽略；这一步只建立查询边界。第二段获取 raw candidates，并分别记录向量、词法耗时和错误。此时 vector distance 与 lexical raw score 仍是不同尺度，不能比较。

第三段合并 identity。同一 chunk 的两路信号被放回一个对象；stale 与 metadata post-filter 在候选层剔除不允许来源。注意：被 stale 过滤的内容既不进入 trusted，也不进入 rejected，因为它不是“相关性不足”，而是“版本不允许参与本次检索”。若要审计 stale 丢弃数，当前 payload 还没有单独计数，这是可观测性可以补强的地方。

第四段排序。weighted/RRF、intent 和 required source 在这里工作。排序只改变先后，不改变候选原始 distance 和 lexical provenance。第五段 Trust Gate 对每个已排序候选作绝对判断，给出接受/拒绝原因。第六段才执行相对地板和 Top-K；required source 模式为多来源覆盖保留绝对过门项。

最终状态需要严格解释：

| 状态 | 必要条件 | 应怎样对用户说 |
|---|---|---|
| `success` | 至少一条 trusted | 可以基于这些知识回答，并附 citation；degraded 时同时说明语义链失效 |
| `no_answer` | 检索完成，但 trusted 为空 | 知识库没有达到准入线的来源，不应强答 |
| `failed` | 检索过程无法完成 | 服务暂不可用，不能伪装成“知识库没有答案” |

这个状态机还能帮助定位指标归属。raw candidates 没有相关项主要是召回问题；相关项进入候选却排不到前 K 主要是排序问题；相关项排名靠前却被阈值误拒是门禁校准问题；后端异常导致 failed 是可靠性问题。旅行问题被错误放行和 Redis 文档完全没被召回，不能用同一个“RAG 准确率”解释。

伪代码可以压缩为：

```python
vector, vector_error = search_vector(query, candidate_k, filter)
lexical = search_lexical_or_degrade(query, candidate_k, filter, vector_error)
candidates = merge_by_chunk_id(vector, lexical)
candidates = remove_stale_and_filter(candidates)
ranked = fuse(candidates, strategy, intent, required_sources)
trusted, rejected = trust_gate(ranked, l2=2.0, lexical=0.20)
trusted = relative_prune(trusted, floor=0.70, unless_required_sources=True)
return terminal_payload(trusted, rejected, vector_error)
```

这不是源码逐行翻译，而是用于审查职责是否串层：例如把 intent 放进 Trust Gate，或让 degraded 跳过阈值，都应立刻被视为架构回归。

## rejected 不是调试垃圾，而是拒答能够被解释的证据

许多 RAG 接口只返回最终 Top-K，所有落选项都被丢弃。这样做对简单搜索足够，却无法回答一个关键问题：系统为什么拒绝？是没有候选，还是有一个很像的候选因为距离过大？是词法命中了 `maxclients`，但来源已经 stale，还是后端根本没有给分数？

AutoOnCall 将未过 Trust Gate 的候选放入 `rejected_results`，并保存 source、chunk、原始 score、lexical/vector/rerank/RRF 信号和 `retrieval_reason`。例如“L2 distance 3.5000 大于阈值 1.0000”“检索后端未返回距离分数”“lexical score 0.1200 小于阈值 0.2000”。这让自动化测试可以精确断言拒绝原因，也让值班人员区分“知识库没有材料”和“材料存在但未达到门槛”。

不过 rejected 不是新的可信上下文。生成模型不能因为它也在 payload 中，就偷偷引用被拒绝文本。安全读模型可以把 rejected 的有限 preview 与原因展示给调试/审计界面，但回答构造必须只使用 `retrieval_results`。下一章的 citation 也只从 trusted 项生成。

同样需要区分“被拒绝”和“被过滤”。metadata 不匹配、source stale、身份去重淘汰的候选没有经过 Trust Gate，当前不会进入 rejected；它们代表不允许参与或与已有候选重复，而不是相关性不足。若生产审计需要完整漏斗，应增加 `filtered_stale_count`、`filtered_metadata_count`、`deduplicated_count`，而不是把所有丢弃都塞进 rejected 破坏语义。

保留 rejected 还有助于阈值校准。若大量正确标注候选都只差一点被拒，可以检查 Embedding、距离阈值或 lexical 归一化；若错误候选轻易过门，说明负例、近域 OOD 或来源约束不足。但调阈值必须在冻结验证集上进行，不能看见一个失败就为它写特例。

## 来源 provenance 为什么必须贯穿召回、引用和评测

一个片段是否可用，不只取决于内容。`source_file` 用于用户可读 citation，`source_path` 用于内部定位，`doc_id/chunk_id` 用于稳定身份，heading/page/sheet/row/table/primary key 用于多格式回查，文档版本和 hash 用于判断旧 Artifact 是否仍可比较。缺少这些字段时，回答即使语义正确，也难以复核“到底引用了哪一版哪一段”。

公开 API 不能直接暴露服务器绝对路径，因此 `public_source_path` 只留下文件名；内部对象仍可保存受控定位信息。安全不是把所有 provenance 都删除，而是按受众提供不同 read model：模型获得可信内容和 citation ID，前端获得安全来源与 preview，服务端日志/Artifact 保留必要诊断信息。

历史复盘、官方文档和内部 Runbook 的权威性也不同。当前 intent/required source 能对少量来源做业务偏置，但没有通用 source authority score。即使官方片段排名靠前，它也只证明机制知识；复盘只证明历史案例；内部 Runbook 只证明组织流程。真正把来源角色连接到本次事故，还要靠下一阶段的 Evidence 模型。

评测 provenance 同样不可省略。80 条结果必须绑定 dataset SHA-256、commit、dirty、运行命令和 rag.json；否则换过切分、索引或阈值后仍展示旧 Recall，就无法判断提升来自算法还是样本变化。查询结果的 citation provenance 与评测结果的 Artifact provenance，本质上解决的是同一个问题：任何结论都应能回到产生它的具体输入和版本。

这也解释了为什么来源字段不应在各层临时拼接。入库阶段应产生稳定 doc/chunk 身份和版本，检索阶段只补充排名信号与准入理由，生成阶段引用现有身份，评测阶段比较同一身份集合。若检索层根据文件名猜 chunk、生成层又重新编号，离线 ID Recall、线上 citation 和事后审计就会指向不同对象。生产化时还应为来源移动、文件重命名和内容相同但版本变化设计明确迁移策略，而不是依赖路径字符串碰巧不变。

当来源元数据缺失时，系统宁可显示“未知来源”并降低可审计性，也不应让模型根据正文猜出处。猜测出的引用看起来完整，却比明确缺失更危险，因为审阅者会把它误当成已经核验的知识链。

## 向量链故障时，降级成功也必须显式承认

默认全局链中，如果 vector store 初始化、查询 Embedding 或 Milvus search 抛错且 Hybrid 开启，系统记录异常类型，改用本地 lexical，并把 mode 标成 `lexical_degraded`、`lexical_degraded_rerank` 或 `lexical_degraded_rrf_rerank`。对外公开消息是“向量检索暂不可用，已降级使用本地词法索引”，内部 detail 不应暴露到 Runbook 工具输出。

降级后的 lexical 候选仍需达到 `0.20`，不是“为了可用性随便返回”。因此会出现三种结果：

| 情况 | 结果 | 业务含义 |
|---|---|---|
| vector 失败，lexical 强命中 | success + `retrieval_degraded=true` | 可用知识，但召回能力退化 |
| vector 失败，lexical 只有弱项 | no_answer + degraded | 正常拒答，并暴露向量故障 |
| vector 与 lexical 都异常 | failed | 检索不可用，不伪装 no-answer |

降级模式保留 `vector_error_type` 和内部 payload 中的 detail 供服务端审计；前端安全读模型和 `search_runbook` 工具会隐藏服务器路径与敏感错误 detail，只保留公开错误类型/消息。公开安全不应以丢失运维可观察性为代价，所以内部与外部视图必须分层。

## 可观测性必须承认哪些阶段没有被分别测到

检索 payload 记录：`vector_search_ms`、`lexical_search_ms`、`fusion_rerank_ms`、`retrieval_total_ms`，以及向量/词法候选数、trusted/rejected 数、retrieval mode、Embedding 模型和 rule-weighted reranker 标签。

但 LangChain Milvus 的一次 `similarity_search` 同时完成查询 Embedding 与 Milvus search，当前代码无法可靠拆出两段。因此：

```text
embedding_ms = not_observed
milvus_search_ms = not_observed
collection_version = not_observed
```

只有组合后的 `vector_search_ms` 可观测。把总耗时随意拆成两个数字会制造假精度。`candidate_count` 当前按 vector count + lexical count 统计，发生同 chunk 双路命中时可能高于合并后的唯一候选数，读图时要结合 trusted/rejected 和去重逻辑理解。

API/前端的 compact read model 只输出稳定 citation 字段：rank、doc/chunk/source、标题路径、文档类型/页码/表格定位、各类分数、preview 和原因；`public_source_path` 只暴露文件名，防止服务器绝对目录泄漏。

RAG 候选数量由 `candidate_k`、Top-K、身份去重、Trust Gate、70% 地板和生成阶段近重复去重共同控制。通用 `ContextBudgeter` 默认按字符裁剪，默认 `3000` 字符、raw alert `4000` 字符并使用统一截断标记；它主要服务 AIOps Prompt/Replanner。当前 RAG 生成没有模型 tokenizer 驱动的统一精确 token budget，所以不能从字符预算推断精确 token 或费用。

## Redis 主线：检索到的是知识，不是这次事故的现场证据

对“`maxclients` 接近上限如何排查与审批”这一查询，系统先规范 metadata filter，构造 Milvus expr；用 `candidate_k=top_k×4` 扩大两路候选；合并相同 doc/chunk；计算 weighted 或 RRF；施加有限 Redis intent 和来源席位；最后先过绝对 Trust Gate，再按需要做 70% 相对裁剪。

当前已经知道什么？来自官方 Redis 文档的 maxclients 机制、Runbook 的检查流程、复盘的历史窗口可以成为 trusted **知识**。哪些是事实？本次 Incident 的 P95、503、timeout、`connected_clients` 必须来自指标、日志和 Redis 工具，不由知识检索证明。哪些是推断？“连接耗尽导致 503”仍需现场跨源支持。还缺什么？客户端池状态、变更时间、事故窗口峰值和风险审批。

为什么执行下一步？trusted chunk 可以指导 Agent 先查 `connected_clients`、`blocked_clients`、服务错误率与连接池，而不是直接建议扩容。什么情况下停止？两路都弱就拒答；Milvus 失败但 lexical 强命中可降级回答知识问题；来源 stale 就过滤；要求的第二来源不存在或未过门就明确缺失。什么情况下转人工？检索链本身只决定知识可否使用，高风险配置是否执行必须交给后续风险策略与审批。

这条边界非常适合面试表达：RAG grounding 约束的是“回答依据什么知识”，AIOps Evidence 约束的是“本次事故发生了什么”，审批约束的是“谁可以采取什么动作”。三层不能相互替代。

## 五个负例，检验系统是不是真的可信

### 负例一：旅行问题召回 Redis 文档

向量库总会给最近邻，但 L2 和 lexical 都弱时应 no-answer。独立企业冻结集唯一失败就是通用旅行域外拒答；它提醒我们固定门禁不是通用 OOD detector，失败 case 不能从分母删除。

### 负例二：RRF 分数为正就放行

任何合法 rank 都会产生正 RRF。若用 `rrf_score > 0` 当可信，就等于取消门禁。正确行为是继续检查原始 L2 或 lexical。

### 负例三：指定官方来源就强制返回

required source 只预留排序席位，不应绕过绝对阈值。来源文档不存在、stale 或只弱命中时，必须缺席或拒答。

### 负例四：Milvus 失败后隐藏 degraded

lexical 成功只能说明精确词法路径仍可用，语义召回已经丢失。返回 success 时也要保留 `retrieval_degraded=true`、mode 和公开错误消息。

### 负例五：stale registry 查询失败就声称绝对安全

当前代码在 stale 检查异常时 fail-open。它偏向可用性，不是强一致保证。报告和面试不能写“双索引在任何故障下绝不返回旧版本”。

## 指标要分别回答召回、排序、拒答和延迟

2026-07-12 的 official `offline_fixture` 回归集共 80 条，其中 64 条非拒答、16 条拒答：

| 指标 | 结果 | 为什么需要 | 能证明/不能证明 |
|---|---:|---|---|
| Recall@1/@3/@5 | `0.6641 / 1.0000 / 1.0000` | 相关项是否进入前 K | Top-3 覆盖标注；Top-1 仍可改进 |
| Precision@1/@3/@5 | `0.7812 / 0.4166 / 0.2500` | 返回项中标注相关的比例 | K 增大引入更多非标注项 |
| MRR | `0.8776` | 首个相关项是否靠前 | 通常靠前，不是全部第一 |
| MAP@1/@3/@5 | `0.7812 / 0.8763 / 0.8763` | 多相关项的平均排序质量 | Top-3 后趋稳，不代表全库完美 |
| nDCG@1/@3/@5 | `0.7188 / 0.8889 / 0.8889` | 高相关项是否更靠前 | 与理想排序仍有差距 |
| Strict multi-source@3/@5 | `1.00 / 1.00` | 指定来源组合是否都出现 | 仅当前 16 条多来源标注 case |
| 引用覆盖率 | `1.00` | 正例是否能生成 citation | 不证明 citation 语义蕴含答案 |
| 拒答 P/R/F1 | `1.00 / 1.00 / 1.00` | 该答与不该答是否区分 | 仅 16 条固定拒答 case |
| 离线 P50/P95/P99 | `12.27 / 18.38 / 20.10ms` | deterministic pipeline 延迟分布 | 本地 surrogate，不是线上 Milvus+真实 Embedding |

Recall@3=1.0000 不能单独写成“检索完美”：Recall@1 只有 0.6641，Precision@3 为 0.4166，MRR 与 nDCG 也不是满分。低 Precision 还要结合标注口径理解：未标注片段不一定完全无用，但在冻结契约中不能算相关。

独立企业排障冻结集为 `11/12`；9 条企业排障正例 `9/9`，Recall@3 `1.00`、MRR `0.94`、nDCG@3 `0.96`。唯一失败是通用旅行域外拒答。这个失败证明门禁的覆盖边界，不能为了得到 12/12 删除。

### 先理解公式，才能知道该优化哪一层

Recall@K 的分母是标注相关项，回答“应该召回的内容有多少进入前 K”；Precision@K 的分母是实际返回 K 项，回答“返回内容中有多少被标注为相关”。因此扩大 candidate 或 K 往往提高 Recall、降低 Precision。对 OnCall，漏掉唯一关键 Runbook 可能比多带一条相关背景更危险，但噪声又会挤占上下文并诱导生成，所以两者必须共同看。

MRR 只关注每个查询第一个相关项的倒数排名：第一名贡献 1，第二名贡献 1/2，第三名贡献 1/3。它非常适合“值班员首先看到的资料是否对”，却不评价第二、第三个相关来源。MAP 对每个相关命中位置计算 Precision 再平均，更适合一个查询有多条正确片段。

nDCG 允许标注不同相关等级。相关性增益随排名对数折损，再除以理想排序 DCG；因此“都召回了，但高价值官方资料排在普通背景之后”会降低 nDCG。Strict multi-source 则不是普通相关性，它检查要求的来源组合是否同时进入 K；只有 16 条多来源标注时，`1.00` 的统计范围就是这 16 条，不能扩成所有查询。

拒答 Precision 回答“系统拒绝的查询中，有多少确实应拒绝”，Recall 回答“所有应拒绝查询中，有多少被挡住”。只把阈值调得很严会提高部分拒答表现，却可能误拒正例；只追 Recall@3 又可能让域外问题强答。拒答 F1 与正例检索指标一起看，正是为了防止通过极端阈值获得虚假成绩。

P50、P95、P99 是延迟分位数：一半请求不超过 P50，95% 不超过 P95，99% 不超过 P99。离线 `12.27 / 18.38 / 20.10ms` 来自确定性本地 surrogate，能检测算法或 I/O 回归，却没有真实网络 Embedding 和 Milvus 尾延迟。它甚至不能推出本地并发容量，更不能与生产 SLA 直接比较。

指标之间不存在一个可以替代全部的总分。优化词法规则可能提高 Recall，却因过度扩展降低 Precision；加强 required source 可能改善 multi-source，却牺牲纯相关性 nDCG；收紧 Trust Gate 提升拒答 Precision，可能降低正例 Recall。可靠优化应先归因失败属于召回、排序、门禁还是依赖，再选择对应指标。

### 冻结运行身份必须和分数一起引用

80-case 来源是 official benchmark Run ID `20260712T074629Z-81f1f98c-0d69abdc`；命令：

```text
scripts/eval/run_benchmark_baseline.py --output-root logs/benchmarks
```

数据集 `eval/rag_relevance_cases.yaml`，SHA-256 `7bd84558b5c76143c2ff49f564def5ccf62f13994e31168b91a2c6c2aff38de3`；Artifact [rag.json](../../../logs/benchmarks/20260712T074629Z-81f1f98c-0d69abdc/rag.json)；commit `81f1f98ccbc4588f8470ad82d3fb82ad849c431f`，`git_dirty=false`；`evidence_level=offline_fixture`。其中 vector-only 是确定性 term-vector surrogate，不是真实 Embedding/Milvus 延迟。

**Artifact（评测制品）**通俗地说是某次运行的机器可读成绩与环境记录；严格定义是绑定 Run ID、commit、dirty、dataset hash、配置和模块结果的冻结文件。它负责复现和审计，不保证当前工作区仍有相同表现。`official` 表示满足基线治理条件，`candidate` 表示可分析但尚未晋级，`stale` 表示代码/数据/配置已与旧 Artifact 不一致；这些状态也不等于 evidence level。

这些数字证明冻结数据集上的回归，不证明生产查询分布、大语料召回、线上阈值、真实 Milvus 延迟或模型化 reranker。完整公式、Scorecard、RAGAS 和样本口径见[第 15 篇](15-项目量化指标与校招验收.md)。

## 为什么当前方案不是唯一正确答案

| 方案 | 优势 | 代价 | 当前定位 |
|---|---|---|---|
| 向量单路 | 语义泛化、实现简单 | 精确符号与 OOD 边界弱 | 不足以独立承担可信检索 |
| 词法单路 | 精确、便宜、可降级 | 同义和跨语言能力弱 | 第二召回路与故障 fallback |
| weighted | 利用强度、领域偏置透明 | 权重/尺度/规则需维护 | 默认方案 |
| RRF | 不依赖分数尺度，稳定简单 | 丢失领先幅度和 intent | 可选融合策略 |
| Cross-Encoder | 更强语义交互 | 额外模型、延迟与降级复杂 | 当前未实现 |
| 学习排序 | 可从标注学习权重 | 需要规模化高质量标签 | 当前未实现 |
| OOD 分类器 | 可学习回答边界 | 需要域外样本和校准 | 当前未实现 |

如果引入 Cross-Encoder，放在召回后、Trust Gate 前，可以重排更多候选，却必须证明模型分数怎样参与绝对准入；放在 Gate 后，只重排 trusted，安全边界稳定但可能提前丢掉语义上可救回的候选。无论放哪，都要增加超时 fallback，并重新评测 Recall、nDCG、拒答和 P95。

生产化还需要：更大且持续更新的数据集，按来源/服务/语言分层的阈值校准，真正在线 Embedding+Milvus 的 P50/P95/P99，索引版本强一致策略，分布式词法索引或专业搜索引擎，query drift 与 stale Artifact 监控，以及真人对引用支持性的盲审。当前 80 条和本地 surrogate 是可复现基线，不是容量证明。

## 阈值怎样从“拍脑袋常量”升级为生产校准

默认 L2 `2.0`、lexical `0.20` 和相对地板 `0.70` 是当前冻结契约的一部分，不是跨模型、跨索引永远有效的自然常数。更换 Embedding 模型、向量维度、距离类型、chunk 粒度或语料分布，L2 的经验范围都可能变化；词法索引文档数量变化会改变 IDF，raw score `/5` 的分布也会漂移。

生产校准至少需要四类样本。第一类是明确可答正例，覆盖 Redis、MySQL、Kubernetes、日志、指标和多格式知识；第二类是近域负例，例如知识库有 Redis maxclients，却没有 Redis Cluster 脑裂处置；第三类是远域 OOD，如旅行问题；第四类是来源冲突和 stale 场景。只用明显正例和明显 OOD，会得到看似漂亮却无法处理近域误答的阈值。

对每条候选保存原始 L2、lexical、来源、是否相关和是否最终被引用，绘制正负样本分布，再分别评估门禁的接受率、误拒率和错误放行率。可以按 doc type、语言和 source 建分层阈值，但规则越细越容易过拟合，必须有独立验证集。阈值选择还要考虑业务损失：对生产变更建议，错误放行的代价通常高于保守拒答；对一般解释查询，可允许更高召回。

融合权重也应与门禁分开调。先固定 Trust Gate 比较 weighted/RRF 或 Cross-Encoder 的 MRR/nDCG，再固定排序比较 L2/lexical 阈值的拒答与正例 Recall。一次同时改权重、阈值、数据集和 Prompt，即使总分提高也无法知道原因。

每次校准应生成新的 dataset hash、配置快照、Run ID、commit、dirty 状态和 Artifact。若知识文件、切分、Embedding 或阈值变化，旧结果应标 stale；不能把旧 official 的 `1.0000` 搬到新索引旁边继续展示。灰度阶段还应监控 no-answer 比例、degraded 比例、来源分布、P95/P99、人工改判和引用点击/反馈，以识别线上 query drift。

即便加入学习型 OOD 或 reranker，确定性 fallback 仍有价值：模型不可用时可以退回现有 weighted/RRF 与原始门禁；新模型的输出也应记录版本和延迟。可信检索的核心不是某个算法更先进，而是每次能力升级都保留可解释拒答、可复现对照和诚实降级。

## 面试转化：把混合检索讲成三个独立决策

### 一句话定位

> AutoOnCall 用向量与词法双路扩大召回，用 weighted/RRF 改善相对排序，再用独立 Trust Gate 决定回答、降级或拒绝。

### 30 秒回答

> 默认 top_k=3，但先以 4 倍候选做向量和 BM25-like 词法召回，按 doc/chunk 去重后可用 0.55 向量、0.35 词法、0.10 基础排名的 weighted，或 RRF。intent 和 required source 只影响排序，最终仍要满足 L2≤2.0 或 lexical≥0.20。Milvus 失败可显式 lexical degraded，两路都弱就 no-answer。

### 3 分钟展开

先用 PostgreSQL 域外问题说明 Top-K 永远有最近邻，再用 `maxclients` 说明向量漏精确符号、词法漏同义表达。接着区分召回、融合、门禁，手算 0.747 与 0.508；比较 weighted 和 RRF，强调 RRF 正分不等于可信；解释 required source 只预留席位、stale 和 metadata filter；再走 vector failure 的 success-degraded/no-answer/failed 三路径；最后用 80 条、64/16、Recall@3 1.0000、MRR 0.8776、nDCG@3 0.8889、拒答 F1 1.00 与离线 P95 18.38ms 说明每个指标和 evidence boundary。

### 可写入简历的准确表述

> 实现 Milvus 语义召回与本地 BM25-like 词法召回的 Hybrid Retrieval，支持 weighted/RRF 融合、可解释 Trust Gate、显式来源约束、stale-source/metadata 过滤及向量故障下的 lexical 降级；在 80 条冻结离线用例上 Recall@3 为 1.0000、MRR 为 0.8776，并保留 16 条拒答样本的固定边界。

### 不可夸大的表述

- “使用完整 BM25 搜索引擎和 Cross-Encoder 重排。”
- “Trust Gate 是通用 OOD 分类器。”
- “required source 支持任意多来源约束。”
- “RRF/weighted 分数就是答案正确概率。”
- “离线 P95 18.38ms 就是线上 Milvus 延迟。”
- “stale 在任何故障下都绝不会漏过旧向量。”

### 高频追问与回答要点

1. **融合分高为什么仍拒绝？** 它只表达相对顺序，绝对准入必须看原始 L2 或真实 lexical 信号。
2. **weighted 与 RRF 如何选？** 前者利用强度和领域偏置，后者规避尺度差异；应在冻结集同时比较排序、拒答与延迟。
3. **为什么无分数向量不 trusted？** 排名不能替代绝对距离；否则后端退化会静默取消门禁。
4. **lexical degraded 会返回垃圾吗？** 不会自动放行，仍要过 lexical≥0.20；弱项 no-answer，并标 degraded。
5. **required source 能否绕门禁？** 不能，只预留席位；来源仍需存在、非 stale 且越过阈值。
6. **Recall@3=1 为何不完美？** Recall 不看 Top-1 与噪声；Recall@1、Precision、MRR、nDCG 均显示改进空间。
7. **为什么不直接用 rerank_score 门禁？** 权重、RRF 和 intent 会改变尺度，它没有稳定绝对语义。
8. **metadata filter 安全边界是什么？** 只接受安全键与 exact/in；非法键忽略，兼容后端可能依赖本地 post-filter。
9. **stale 如何影响向量结果？** stale registry 同时过滤 lexical 与返回后的 vector；但 registry 查询异常当前 fail-open。
10. **为什么 embedding_ms 是 not_observed？** LangChain Milvus 把 query embedding 与 search 包在一次调用里，只能诚实记录合并 vector_search_ms。
11. **Cross-Encoder 放在哪里？** Gate 前提高语义排序但要重新校准准入；Gate 后保持边界但可能提前丢候选，两者都需超时降级。
12. **检索 trusted 是否等于本次根因成立？** 否，trusted chunk 是知识；本次事故事实仍来自指标、日志和依赖工具。

## 检验真正理解的练习

1. 为三个候选设置 L2、raw lexical、base rank，手算归一化、weighted 和 RRF，再分别执行 Trust Gate 与 70% 地板。
2. 构造一个 RRF 排名第一但 L2>2.0、lexical<0.20 的候选，解释为什么结果必须 rejected。

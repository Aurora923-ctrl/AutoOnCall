# 运维知识如何可靠入库：从“有文件”到可信知识发布

> 审计日期：2026-07-22

> 上一篇：[告警如何变成故障事件](02-告警如何变成故障事件.md)已经给 02:07 的告警建立稳定 Incident 身份

## 02:07，值班者缺的不是一个文档目录

`order-service` 大量返回 503。

P95 延迟上升，日志出现 `Redis connection timeout`，`connected_clients` 靠近限制。

上一篇已经回答“这些告警是否属于同一事件”。

现在系统有了稳定 Incident，却还不知道怎样解释它。

值班者手里可能有四类资料：

- Redis 上游客户端与延迟说明；
- 内部容量 Wiki；
- 历史事故复盘 PDF；
- 带工单号、服务名和变更版本的 XLSX。

先把三类对象分开：

1. 503、P95、timeout 和当前连接数是本次 Incident 的现场事实；
2. 官方资料、Runbook、Wiki、复盘与工单是可供诊断参考的知识；
3. “连接耗尽导致 503”仍只是需要现场证据支持或反驳的假设。

入库只负责第二类。

它不能把历史经验升级成本次事实，也不能直接确认根因。

但若入库做坏了，后续检索和模型再强也无法补救：

- PDF 页码丢失，引用只能说“某份复盘提到过”；
- HTML 导航与页脚成为高频噪声；
- Excel 多行被揉成一段，无法定位 sheet、行号和工单主键；
- Markdown 的诊断、审批和回滚恰好在固定长度处断开；
- 同名文件更新后，新旧 chunk 同时出现；
- 向量索引写入新版本，词法索引仍保留旧版本；
- 文件已经保存，索引失败，接口却返回“成功”。

这些问题都发生在回答生成之前。

来源结构、版本身份和发布状态一旦丢失，模型只能猜。

所以可靠入库不是“把字符串塞进向量库”，而是把外部来源变成：

> 可解析、可定位、可切分、可版本化、可双路检索、失败可见且可重新发布的知识资产。

## 从文件到资产，需要连续回答七个问题

当前链路可以抽象为：

```text
受控来源
  → 格式感知 Loader
  → 清洗后的逻辑单元
  → 结构优先、长度受控的 chunk
  → source / chunk / content 三层身份
  → Embedding 与 Milvus schema 校验
  → 向量索引 + 本地词法索引发布
  → 质量记录、stale registry、补偿缺口、重建与迁移
```

七个问题分别是：

1. 这份文件是否允许读取？
2. 容器中的正文和来源坐标怎样恢复？
3. 怎样形成适合检索、又不破坏语义的块？
4. 同一个来源、同一个引用块和同一版内容如何区分？
5. Embedding 输出能否安全进入向量库？
6. 两套索引怎样避免静默混版？
7. 怎样证明一次发布在特定样本与环境中健康？

这条链有两个职责边界。

第一，入库不决定最终召回哪些 chunk。

向量召回、词法召回、融合、来源覆盖和 Trust Gate 是下一篇的主题。

第二，入库成功不等于知识内容正确。

一个结构完美但已过期的 Runbook，仍可能是危险知识。

## Loader 先恢复来源结构，不抢 Splitter 的工作

Loader 的职责是理解文件容器。

当前注册表支持：

- `txt`；
- `md`、`markdown`；
- `pdf`；
- `html`、`htm`；
- `csv`；
- `xlsx`。

每种 Loader 返回若干 `LoadedDocument(content, metadata)` 逻辑单元，以及一份清洗报告。

共享清洗会识别：

- 空单元；
- 有效字母或数字过少的低信息单元；
- 同一文件解析批次中的精确重复单元；
- 有上限的 warning。

这里的重复过滤不是全库语义去重。

它只避免同一个来源在一次解析中把完全相同的内容重复发布。

### Markdown 与纯文本：保留原文结构

纯文本按 UTF-8 读取，Markdown 会识别 BOM。

Loader 不提前抹平代码块、列表缩进和标题。

因为这些结构会在后续切分时成为语义边界。

若一开始统一成一行纯文本，Splitter 就失去了判断“诊断步骤”和“回滚条件”是否属于不同章节的依据。

### HTML：去掉页面外壳，保留标题路径

HTML Loader 会移除脚本、样式、导航、页眉页脚、表单控件、隐藏元素等噪声。

正文按标题路径形成 section，并保留 `heading_path`、`html_title` 及各层标题 metadata。

它还会避免 `li > p`、`pre > code` 这类嵌套语义块被重复提取。

因此 `redis_capacity_wiki.html` 的引用可以指向具体标题路径，而不是只给出文件名。

当前最多接收 10,000 个 HTML section，超出部分会产生 warning。

这是一条资源上限，不是页面完整性承诺。

### PDF：页级恢复，而不是假装支持 OCR

PDF Loader 逐页提取文字并保留 `page_number`。

某一页抽取异常时，该页记 warning，其他正常页仍可入库。

最多处理 2,000 页。

若 PDF 只有扫描图像，没有文字层，当前实现不会自动 OCR。

没有有效页时，结果应是 empty，并明确提示扫描件需先 OCR。

这比把空字符串写入索引更诚实。

### CSV 与 XLSX：一行就是一个可审计单元

表格 Loader 将数据行变成独立知识单元。

它保留：

- `sheet_name`；
- `row_number`；
- `table_name`；
- 从 `ticket_id`、`incident_id`、`service_name`、`error_code`、`id` 中识别的 `primary_key`。

空表头会生成稳定列名，重名表头会重命名并记 warning。

额外数据列不会因表头较短而静默丢失，但会受最大列数约束。

XLSX 以只读模式加载，保留公式文本，不执行公式。

CSV 兼容 UTF-8 BOM、GB18030 和常见分隔符。

当前边界包括：

- 单元格最多保留 500 字符；
- 最多 24 列；
- 每个表最多 10,000 行；
- 工作簿最多 32 个 sheet；
- XLSX 解压后最多 100 MiB。

这些限制既防内存失控，也把截断暴露为 warning。

### Redis 案例中的四种引用坐标

经过 Loader 后：

- 上游 Markdown 依靠标题与 chunk；
- Wiki HTML 依靠标题路径；
- 复盘 PDF 依靠页码；
- 工单 XLSX 依靠 sheet、行和主键。

这四种坐标最终都会随 chunk 进入 citation metadata。

如果某一步丢失坐标，05 篇的引用校验也无法凭空恢复。

## Splitter：先尊重结构，再限制长度

Loader 输出的是来源逻辑单元，不一定是最终检索粒度。

一页 PDF 可能很长，一个 Markdown 章节也可能包含多个独立步骤。

当前 Splitter 对 Markdown 采用两阶段策略：

1. 先按一级、二级标题切分；
2. 再用递归字符切分器做长度约束。

三级标题不作为第一阶段硬边界，避免过度碎片化。

标题保留在 chunk 正文中，帮助一个块脱离全文后仍能说明主题。

配置中的 `chunk_max_size` 默认为 800，递归切分器实际使用二倍长度，即 1,600 字符；overlap 默认为 100。

相邻小块只有同时满足下列条件才会合并：

- 至少一块短于 300 字符；
- 合并后不超过 1,600 字符；
- Markdown 的 h1、h2 上下文一致。

合并时还会检测相邻块重叠，避免把 overlap 再复制一遍。

非 Markdown 来源以 Loader 给出的页、HTML section 或表格行为边界，再做递归限长。

这个策略背后的原则是：

> 格式结构决定“尽量不要在哪里切”，长度预算决定“最多允许多大”。

它不是普适最优解。

标题写得差、超长代码块、复杂表格或语言模型辅助语义分段，都可能需要不同策略。

是否值得继续切，应由检索效果和质量评测判断，而不是只看字符数。

## 三层身份：来源、引用和内容不能混成一个字段

可靠更新的核心问题是：“新旧数据到底是不是同一个东西？”

当前实现区分三层身份。

### 第一层：规范来源 `_source_id`

`_source` 保留当前绝对或规范化路径。

`_source_id` 则尽量脱离部署根，识别同一逻辑来源。

它优先使用配置的允许根，并识别：

- `docs/knowledge-base/...`；
- `uploads/...`；
- 旧 `aiops-docs/...` 路径。

Windows 风格路径还会做适当的大小写归一。

这使得仓库从一个盘符迁移到另一个盘符时，不会仅因绝对路径变化就产生第二份逻辑来源。

### 第二层：可读引用 `_chunk_id`

初次切分通常得到 `文件名#0001` 这样的 ID。

它适合人阅读和引用。

重新入库时，服务先读取该来源的词法快照，再按 `_chunk_hash` 尽量复用未变 chunk 的旧 ID。

若已有来源中出现新增块，可生成带内容哈希片段的 ID，减少前部插入导致后续所有 citation 漂移。

但 `_chunk_id` 不是永久内容指纹。

它是引用锚点，稳定性是尽力保持，不是数学保证。

### 第三层：内容版本与向量主键

每个文档与 chunk 都有：

- `_document_hash`；
- `_document_version`；
- `_chunk_hash`；
- `_version_key`。

Milvus 主键由规范来源 ID、chunk ID 和 chunk hash 共同哈希生成。

因此：

- 同一 chunk 内容未变，即使整份文档其他部分变化，向量主键不应漂移；
- chunk 内容变化时，向量身份随之变化；
- 重试同一版本时，稳定主键允许 upsert 幂等收敛。

查询和删除按 `_source_id` 匹配，同时保留旧 `_source` 精确路径作为 legacy 回退。

## Embedding 不是黑盒：批次失败必须整体可见

切分完成后，文本还不能直接写入 Milvus。

当前 `DashScopeEmbeddings` 通过 OpenAI 兼容接口调用 `text-embedding-v4`，默认维度为 1,024。

文档按配置批量发送，默认 batch size 为 10，单批最多重试 2 次，超时默认 30 秒。

这里有一个重要原则：

> 一批输入必须得到数量一致、顺序可恢复、维度正确的整批输出；否则整批失败。

为什么不能接受部分向量？

假设输入是 `[chunk-A, chunk-B, chunk-C]`，服务只返回两个向量。若调用方继续按位置写入，就可能把 B 的向量绑定到 C 的正文。这种错误比“本批失败”更危险，因为它不会立即报错，却会长期污染召回。

当前校验包括：

- 返回数量必须等于输入数量；
- provider 返回 index 时，index 必须完整、唯一并覆盖 `0..n-1`；
- 每个向量必须是可迭代数值；
- 维度必须与配置一致；
- 不得包含 NaN 或 Infinity；
- 不得是零向量。

任何校验失败，整个 batch 抛错，由索引编排进入失败补偿。这里不能提前等同为“最终一定保留 stale”：stale 是否可靠留下，还取决于后续词法快照恢复与异常处理，当前协议存在下文所述缺口。

### 为什么客户端还要做 L2 归一化

当前 Milvus 使用 L2 距离。若不归一化，向量模长和方向会共同影响距离；文档向量与查询向量在客户端统一归一化为单位向量后，L2 更接近方向差异，阈值也更容易保持一致。

但归一化不是“让所有模型都等价”。更换 Embedding 模型、维度或归一化规则后，仍需重新构建索引并校准检索阈值。

### 为什么维度正确仍不足以证明 collection 可用

Milvus schema 不只是一个向量维度。

当前契约要求：

- `id` 是长度受限的 VARCHAR 主键；
- `auto_id=False`，因为主键由应用稳定生成；
- `vector` 是配置维度的 FLOAT_VECTOR；
- `content` 是长度受限的 VARCHAR；
- `metadata` 是 JSON；
- `enable_dynamic_field=False`；
- 索引类型为 IVF_FLAT；
- 距离类型为 L2。

readiness 还会抽样检查 metadata 是否包含必要身份字段，以及 `_vector_id` 是否与主键一致。

遗留 collection 即使维度相同，只要缺 metadata、启用 auto ID、字段长度或索引参数不一致，都可能破坏幂等写入、引用和清理。

因此 schema 检查解决的是“存储契约一致”，不是只解决“数组长度一致”。

## 为什么需要向量与词法两套索引

Redis 的“连接数耗尽”和“maxclients exhausted”语义相近，向量召回更有优势。

但 `INC-REDIS-009`、`rc1`、`rejected_connections`、命令和错误码需要精确词法保护。

本地词法索引保存 chunk、metadata、词项和 stale registry。

它使用 BM25-like 评分，提取 ASCII token 与中文二元、三元片段，也支持：

- 精确实体命中；
- metadata 过滤；
- 标题和文件名参与可搜索文本；
- 线程锁与跨进程文件锁；
- 临时文件加原子替换；
- 损坏 JSON 时 strict read fail-closed。

它不是 Milvus 的缓存，而是独立召回后端，因此入库必须把同一版本发布到两端。两端没有跨库 ACID 事务，这正是下一节的难点。

## 双索引发布：正常路径先标 stale，失败补偿仍有窗口

单文件索引使用同源跨进程锁，并在开始前快照该来源的词法状态。

正常发布顺序是：

1. 将 source 标记为 `indexing_in_progress` stale；
2. Loader、清洗、Splitter 产生新 chunks；
3. 按旧词法快照尽量复用 chunk ID；
4. Milvus 以稳定 ID upsert 新向量；
5. `flush` 确认新向量可见；
6. 替换词法来源，但暂不清 stale；
7. 删除该来源中不属于当前 vector ID 集合的旧向量；
8. 再次确认清理完成后清除 stale。

若向量写入部分失败，管理器会尝试按已知 ID 补偿删除。若词法提交或旧向量清理失败，编排层会恢复词法快照，并尝试删除本次新向量。这些都是 **best-effort 补偿**，不能写成事务回滚，也不能写成失败后一定保持 stale。

当前有三个具体缺口。第一，`delete_by_ids(..., raise_on_error=False)` 的删除结果没有被索引编排检查，补偿删除失败可以被吞掉；第二，向量 ID 对未变 chunk 是稳定的，按“本次 ID 集合”补偿删除时，可能同时删掉旧版仍需要的未变 chunk；第三，外层的 `rollback_succeeded` 主要表示词法快照是否恢复成功，词法恢复成功后通常不会再次 `mark_source_stale`，即使向量端已处在不确定状态。

empty 路径也有同类窗口。它先删向量、再删词法；若向量已删除而词法删除失败，外层可以恢复旧词法快照并把 `rollback_succeeded` 记为成功，source 却不一定重新标 stale。最终可能出现“旧词法已恢复、向量已缺失、registry 没有 stale 标记”的未标记不一致。

因此当前协议的准确结论是：正常路径按 stale 门禁收敛，已标 stale 的失败路径可被检索层隔离；但任意补偿失败并不保证重新标 stale，存在未标记不一致窗口。若要闭合，应把向量补偿结果纳入状态判断，并在任何跨索引结果不确定时无条件重新写入 stale；更强方案是版本化构建后再切换查询别名。

### stale registry 如何同时保护向量与词法候选

必须准确区分两个层次。词法索引自身在 search 时排除 **registry 中已经标记** 的 stale source，向量索引并没有与这个 JSON registry 共享数据库事务。上层结构化检索会在两路结果合并为 candidate 后，统一调用同一个 stale registry：

- 词法候选再次受检查；
- 向量候选也按 `source_path` 受检查；
- registry 读取异常时返回 stale，fail closed；
- 被过滤数量进入 `filtered_stale_count` 可观测字段。

因此当前事实是：

> 上层对 registry **已经标 stale** 的向量和词法候选统一 fail closed；registry 读取异常也 fail closed。但发布补偿并不保证每种不一致都留下 stale 标记，所以这道过滤不能覆盖未标记窗口，双索引发布本身也不是原子事务。

不能把前半句省略成“只有 lexical 会隔离”，也不能把它扩大成“所有发布失败都已隔离”。上层过滤实现了可靠的 **stale-marker consumer**，当前 Critical 缺口在 producer：失败编排未保证不一致时一定产出 stale marker。

### source stale 与 Artifact stale 不是一回事

`source stale` 的对象是一份知识来源。**当 registry 中存在标记时**，它表示该 source 可能因正在更新或更新失败而两路版本不一致，治理动作是暂时不让它进入检索；当前不能反推“registry 中没有标记就一定一致”，因为补偿协议存在漏标窗口。

`Artifact stale` 的对象是一份评测产物。它表示 Artifact 中冻结的代码、知识资产、配置、依赖、Prompt 或数据集指纹与当前候选不一致，治理动作是禁止把历史成绩继承给当前工作区。两者可以同时发生，也可以完全独立。

例如：

- 某 Runbook 更新失败且 registry 已标 source stale，运行时候选被隔离，但历史评测 Artifact 仍忠实描述它自己的旧 commit；
- 当前工作区所有 source 都 ready，但知识文件已改变，历史 Artifact 相对当前工作区仍 stale。

不要用 source stale 解释评测 provenance，也不要用 Artifact stale 决定运行时是否召回某个 chunk。

## 迁移与全量重建解决不同问题

`migrate_legacy_lexical_index.py` 只迁移本地词法 JSON 的身份字段。

它默认 dry-run。

显式 `--apply` 时才会写备份并原子替换，将旧 source path 和 stale key 规范化为稳定 source identity。

它不改变正文和词项，也不重新生成 Embedding。

`rebuild_rag_index.py --confirm-drop` 则是维护级全量发布。

它会：

1. 预加载所有受支持资产；
2. 生成期望身份集合；
3. 在影子 Milvus collection 中批量写入；
4. 验证 source、chunk、hash 与主键身份；
5. 将 active collection 重命名为备份；
6. 把影子 collection 切成 active；
7. 全量替换词法索引；
8. 再验证向量与词法身份一致。

后半段失败时会恢复词法快照，并在已切换时尝试恢复备份 collection。

这比逐文件更新更接近版本化发布，但仍是维护操作，不是无条件的零停机事务。

它需要真实 Milvus 与 Embedding 配置，并会进行 collection 切换，必须纳入变更审批和回滚计划。

## 文件 API：保存成功与知识 ready 是两个阶段

上传入口受知识写 scope 保护。

进入 Loader 前，接口会检查：

- 文件名不能为空；
- 不接受路径分隔符、控制字符、系统保留名和 Unicode 规范化碰撞；
- 扩展名必须受支持；
- 显式 MIME 不能与扩展名冲突；
- PDF 与 XLSX 必须通过容器签名检查；
- 文本不得含 NUL，TXT/Markdown 必须是 UTF-8；
- 流式上传不能超过配置大小。

同名上传锁覆盖“落盘 + 后台线程索引结束”的完整区间。

若覆盖上传的新版本未 ready，会恢复原文件，而不是只恢复索引。

请求取消时也会等待线程内索引结束，再释放同名锁，避免后一请求读到前一请求尚未完成的中间状态。

返回值因此不是一个布尔值：

| 结果 | 含义 |
| --- | --- |
| 400 | 文件名、格式、MIME、签名、编码或大小无效；正式文件不应被替换 |
| 403 | 运维目录超出允许根 |
| 200 | 索引返回合法 `success` 且 chunk 数为正 |
| 207 | 文件处理进入部分成功，但知识未 ready，调用方不能立即检索 |

单文件索引只接受 `success`、`empty`、`failed` 三类规范状态。

所谓 success 若 chunk 数为零，也会 fail closed。

目录索引会逐个继续处理，分别统计 success、empty 和 failed。

只要出现 empty 或 failed，目录总体不包装为完全成功。

但它不是跨文件事务：前一个文件成功，后一个失败，不会回滚前一个。

## 质量观测：先问分母，再看百分比

运行时质量 JSONL 记录文件级结果：

- source 文件名与公开路径；
- 操作类型；
- success / empty / failed；
- chunk 数与耗时；
- Loader 类型；
- raw、indexed、dropped、empty、duplicate、low-information 单元数；
- 有界 warning；
- 对外脱敏的错误。

它使用文件锁追加，损坏行会被跳过并告警。

质量记录是旁路观测。

记录失败不会把已经完成的索引回滚，也不能把索引失败改写成成功。

`eval_knowledge_quality.py` 则生成可复现的质量 Artifact。

它用生产 Loader 与 Splitter 检查：

- 文件是否发现、解析、切分并产生可索引块；
- chunk 长度、空块和定位 metadata；
- 精确与近似重复；
- freshness；
- 可选的临时 Milvus CRUD 与清理。

freshness 默认以运行 `--as-of` 到文件最近 Git 提交时间计算；没有 Git 历史才回退文件 mtime。

默认 stale 阈值为 365 天，近似重复阈值为 0.90，超长观察阈值为 1,600 字符。

当前实现没有完整覆盖：

hash 能证明内容身份，不能证明内容被批准。

## 设计取舍：为什么当前实现这样选

### 为什么不先统一转纯文本

纯文本会丢页码、标题路径、sheet、行号和主键。

可以设计统一中间格式，但中间格式必须携带这些坐标。

### 为什么 Markdown 不按每个标题切

更细标题提高定位，却可能制造大量短块。

当前选择 h1/h2 加递归限长，再通过指标观察，而不是假定标题越细越好。

### 为什么一批 Embedding 失败要整体失败

数量或顺序不一致会造成文本与向量错绑。显式失败和尝试补偿的代价小于静默接受非法输出；但批次失败后的跨索引补偿仍是 best effort，不应由此推导出一致性已经恢复。

### 为什么先 upsert 新向量再删旧向量

它减少先删后写导致完全无向量的窗口，代价是需要 stale marker、稳定 ID、已知 ID 补偿和旧向量清理。当前补偿结果没有完全进入状态判断，稳定 ID 还让“删除本次 ID”可能影响旧版未变 chunk，因此这是有已知一致性缺口的可恢复性设计，不是原子提交。

### 为什么设计意图选择 stale 时不可用

运维知识含生产命令、审批与回滚规则，设计意图是短期缺少一个来源也不要混用半新半旧版本。检索层确实会过滤 registry 已标 stale 的两路候选，但当前失败 producer 可能漏标；所以这仍是安全策略目标，而不是所有异常分支都已满足的实现保证。

### 为什么还保留本地词法索引

它低成本保护错误码、命令、版本号和工单号。代价是只适合共享可靠锁文件的单机或小规模部署，不适合独立文件系统的多副本大规模服务。

### 为什么全量重建用影子 collection

它把完整身份验证放到切换前，并保留回退集合。代价是额外存储、维护复杂度和切换期间的运维协调。

## 高频追问

1. **为什么 Loader 和 Splitter 必须分开？**
   Loader 恢复来源结构，Splitter 控制检索粒度；混在一起会丢定位，也难以区分解析问题与切分问题。

2. **扫描 PDF 会怎样？**
   没有文字层时产生 warning 并成为 empty；当前不自动 OCR。

3. **为什么 Embedding 数量不一致不能接受部分结果？**
   因为位置错位会把向量绑定到错误正文，静默污染比整批失败更危险。

4. **归一化解决了什么？**
   让 L2 更聚焦方向差异；它不免除换模型后的重建与阈值校准。

5. **维度相同为什么 schema 仍可能不兼容？**
   auto ID、metadata 字段、VARCHAR 长度、dynamic field、索引类型和样本 metadata 都可能破坏应用契约。

6. **`_chunk_id` 永久稳定吗？**
   不是；它是尽力稳定的可读引用。内容 hash 与向量主键承担更严格的版本身份。

7. **向量成功、词法失败怎么办？**
   当前会恢复词法快照并尝试删除本次已知向量 ID，但删除结果不检查，稳定 ID 还可能对应旧版未变块；词法恢复成功后也不保证重新 mark stale。因此只能说“尝试补偿”，不能说一致性已恢复或失败必然被隔离。

8. **stale 是否只保护 lexical？**
   不是。对 registry 已标 stale 的来源，词法 search 会过滤，上层还会统一过滤向量与词法；registry 读取异常也 fail closed。但 registry 没有标记时，上层看不见补偿造成的不一致。

9. **source stale 与 Artifact stale 有什么区别？**
   前者是运行时来源标记：只有 marker 存在时，检索层才隔离该来源；当前没有 marker 不能证明一致。后者是评测 provenance 状态，阻止历史成绩继承到指纹不同的候选。

10. **为什么上传会返回 207？**
    文件持久化与知识 ready 是两个阶段；后者未完成时不能返回 200。

11. **为什么空文件覆盖要删除旧索引？**
    否则用户已经删除的知识仍会被召回。当前 empty 顺序是先删向量再删词法；若后者失败而词法快照恢复，source 不一定重新标 stale，可能留下旧词法与缺失向量的未标记窗口，这是待修缺口。

12. **20/20 为什么不是 100% 解析能力？**
    它只描述 20 份冻结样本；95% Wilson 下界仍是 0.8389，且不含当前新增资产与未覆盖格式。

13. **当前 26 份资产能否引用 221 chunks？**
    不能。221 是 2026-07-19 的冻结分母，当前工作区未晋升匹配的新 official local_live baseline。

14. **九份 official 文档有 revision，为什么仍说 provenance 未闭环？**
    revision 说明上游锚点；当前文件 hash 还必须进入匹配的 official manifest，当前这一步尚未完成。

15. **本地词法索引适合多副本吗？**
    只适合共享可靠锁文件的小规模部署；独立文件系统、多地域和大规模索引应采用专用搜索后端与版本切换。

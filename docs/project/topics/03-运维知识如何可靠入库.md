# 运维知识如何可靠入库：从“有文件”到可追踪知识资产

> 专题编号：03  
> 审计日期：2026-07-15  
> 上一篇：[告警如何变成故障事件](02-告警如何变成故障事件.md)已经给诊断提供稳定 Incident  
> 本篇目标：建立离线知识构建链；检索排序与回答生成留给后两篇

## Runbook 放进目录，为什么还不算拥有了知识库

先从一场 Redis 故障开始。`order-service` 大量返回 503，P95 延迟上升，日志出现 `Redis connection timeout`，`connected_clients` 接近 `maxclients`。值班工程师手里有四份资料：Redis 官方限制说明 `official_redis_clients.md`、内部容量 Wiki 导出的 `redis_capacity_wiki.html`、历史复盘 `redis_postmortem.pdf`，以及历史工单 `tickets.xlsx`。

此刻必须先分清三类信息：503、P95 和 timeout 是本次 Incident 的现场事实；官方说明、Runbook 和历史工单是知识；“连接耗尽导致 503”仍是需要现场证据支持或反驳的推断。知识入库的职责，是让后续系统能准确找到并引用第二类信息，不负责把知识自动升级为本次事故的事实，更不负责直接确认根因。

新手最容易实现的是：遍历目录、读取字符串、每 800 字切一块、生成向量、写入数据库。它看起来已经具备 RAG 的全部关键词，却会在真实资料上连续失败：

- PDF 的页码被丢掉，回答者只能说“某份复盘提到过”，无法指出第几页。
- HTML 的导航栏、页脚和按钮成为高频文本，“Redis”查询总命中菜单而不是 Runbook 正文。
- Excel 多行工单被拼在一起，引用无法定位到 sheet、行号和业务主键。
- Markdown 的“诊断步骤”“审批条件”“回滚边界”恰好在固定长度处被切断。
- 同名文件更新后，新旧 chunk 同时存在；检索结果语言通顺，却混用了两个版本的操作规则。
- 向量索引写成功、关键词索引写失败，系统仍返回“入库成功”，后续查询得到彼此不一致的候选。
- 文件上传成功但索引失败时，接口只返回 200，调用方误以为资料已经可检索。

这些失败有一个共同点：问题发生在生成回答之前。来源结构、版本身份和写入状态一旦在入库阶段丢失，后面的检索器和大模型没有可靠方法把它们猜回来。因此，可靠入库不是“把文本塞进向量库”，而是把一个外部文件变成**可解析、可定位、可版本化、可检索、可观测且失败可见的知识资产**。

## 先建立一条最小可信链，再决定用什么框架

可靠入库可以抽象为六个连续问题：这个文件是否允许被系统读取？容器里的正文和结构如何恢复？怎样形成合适的检索粒度？如何给每个产物建立身份？怎样把同一批内容发布到多个索引？最后，怎样证明这次发布真的健康？

```text
受控来源
→ 格式感知解析
→ 清洗与逻辑单元
→ 结构感知切分
→ chunk 身份与版本
→ 向量索引 + 词法索引
→ 质量记录、失败补偿与 stale 门禁
```

这条链有两个重要边界。第一，入库只负责构建知识资产，不决定某次问题最终召回哪些 chunk；双路召回、融合排序和 Trust Gate 是下一篇的主题。第二，入库成功只说明资料可进入检索，不说明资料内容正确、最新或适用于本次事故。一个排版完美但已经过期的 Runbook，依然可能是危险知识。

### Loader 负责恢复来源，不负责最终粒度

**Loader（加载器）**通俗地说，是“把 PDF、网页、表格等容器里的正文和出处找回来”。严格定义是：针对一种来源格式，把文件解析成若干 `LoadedDocument(content, metadata)` 逻辑单元，并产生清洗报告。它解决格式差异、页码、标题路径、sheet、行号和主键定位；不负责最终检索 Top-K，也不负责判断某段内容是否能证明根因。

例如，PDF 的一页可以是一个逻辑单元，Excel 的一行可以是一个逻辑单元。反例是用统一的 `read_text()` 读取所有文件，或让 Loader 直接把所有格式切成固定 800 字。AutoOnCall 用来源专属 Loader 恢复结构，再把逻辑单元交给统一切分阶段。

### Splitter 负责检索卡片，不负责猜来源结构

**Splitter（切分器）**通俗地说，是“决定一张检索卡片应该放多少内容”。严格定义是：在保留 Loader metadata 的前提下，将过长逻辑单元递归拆分，并在受限条件下合并过短片段。它解决召回粒度、上下文预算和语义完整性的取舍；不负责解析 PDF，也不应猜测表格的业务主键。

若一个 PDF 页很长，Splitter 可以在这一页内部继续切，但每个子块仍应带同一个 `page_number`。若 Excel 有两行独立工单，Splitter 只能在某一行内部拆分，不应先把两行合并。来源边界由 Loader 决定，长度边界由 Splitter 决定，职责混在一起就很难解释错误来自哪里。

### Chunk 是知识发布单元，不是原始文档的替身

**Chunk（知识分片）**通俗地说，是最终参与召回的一张卡片。严格定义是：带正文、来源定位、版本、内容哈希和存储身份的最小索引单元。它解决“检索返回什么、答案引用哪里、更新替换谁”；不负责代替完整原文，也不保证单个分片包含全部上下文。

Redis Runbook 中“升级 `maxclients` 前需审批”是一张有价值的卡片，但它必须同时保留所属文件、标题路径和版本。只有一句“需审批”而没有上文条件的碎片，是低质量 chunk；包含诊断、变更、回滚三章的超大块，也会让召回与引用失去精度。

## 入口安全是知识可信链的第一环

“目录索引”本质上允许服务端读取文件；如果把任意路径直接交给它，知识库接口就可能退化成服务器文件读取接口。因此，安全不能只靠文件扩展名。

AutoOnCall 先把目标路径 `resolve` 成规范绝对路径，再检查它是否位于配置允许的根目录中。默认允许根包括上传目录和 `docs/knowledge-base`。目录入口当前只扫描目标目录的第一层普通文件，并不递归下钻；支持的扩展名来自 Loader Registry，而不是一份独立、容易漂移的白名单。路径越界返回 403，不存在或不是目录返回 400，对外只显示目录名和公共错误，不暴露服务器绝对路径、允许根列表或底层 Milvus 错误。

上传入口不会再“清理后接受”可疑文件名，因为两个不同原名可能被清理成同一个目标名。当前做法是拒绝路径分隔符、Windows 非法字符、控制字符、不可见格式字符、首尾空白和非 NFC Unicode 名称；同时拒绝空 basename、超过 160 字符或 UTF-8 超过 200 字节的名称，以及 `CON.txt.md` 这类带多重扩展名的 Windows 设备名。扩展名必须属于 `txt,md,markdown,pdf,html,htm,csv,xlsx`，默认最大文件大小是 10 MB。

扩展名只是第一层类型声明。接口还会检查调用方显式提交的 MIME 是否与扩展名相容；命令行客户端常见的空 MIME 和 `application/octet-stream` 作为通用类型保留兼容。文件写入临时路径后，PDF 必须有 `%PDF-` 签名，XLSX 必须是包含 `[Content_Types].xml` 与 `xl/workbook.xml` 的 ZIP 容器；文本类文件不能包含 NUL，TXT/Markdown 还必须能按 UTF-8 解码。强签名与扩展名冲突时，正式文件不会被覆盖。这里仍不是恶意文件扫描，也不证明文件内容安全。

更容易被忽略的是覆盖和并发过程。系统不会先清空正式文件再慢慢接收上传，而是以 1 MB 为读取块写入同目录 UUID 临时文件，累计大小超过上限、校验失败或协程取消时都清理临时文件，全部成功后才用 `replace` 替换正式文件；下次同名上传还会在持锁状态下清理遗留临时文件。`UploadFile` 无论成功或失败都会关闭。这样，上传一个过大的同名 `redis.md` 不会把原文件截断成半份。

同名文件的“落盘 + 索引”现在由目标路径对应的 `AsyncFileLock` 串行化，而不是只保护 `replace`。即使请求取消，工作线程中的索引调用仍会等待结束后再释放锁，防止后一请求覆盖文件时，前一索引任务读到后一版本。它能协调共享同一锁文件、且文件锁语义可靠的本机或共享文件系统进程；不是跨地域分布式事务，也不能替代对象存储版本号或发布队列。

上传后索引失败时，接口返回 `207 partial_success`：文件已经保存，但 `indexing_ready=false`，索引状态为 `failed`；空文件或扫描 PDF 未产生 chunk 时也返回 207。索引服务若返回未知状态、负数计数，或者声称 `success` 却 `chunk_count=0`，上传层会 fail closed 为部分成功，而不是接受一个格式错误的“成功”。上传配置、200 和 207 响应也已有 Pydantic/OpenAPI 契约。质量记录属于旁路可观测性，写入失败会记录服务端日志，但不会把已经成功的索引降级成失败。这一区分非常重要：

```text
upload succeeded ≠ parsing succeeded
parsing succeeded ≠ chunks produced
chunks produced ≠ both indexes published
both indexes published ≠ knowledge is current or correct
```

## 六类资料不能被压成一种字符串

### Markdown 与纯文本：标题结构和编码边界不同

TXT、`.md` 与 `.markdown` 当前按 `utf-8-sig` 读取，允许常见 UTF-8 BOM，又不会把 BOM 留在第一个 Markdown 标题前。正文清洗只移除 NUL 和首尾空白，不再逐行 `strip`，因此代码块缩进和列表层级不会在 Loader 阶段被破坏。Markdown 先作为一个逻辑文档载入，标题结构在后续切分中恢复；TXT 没有标题语义，直接进入递归切分。当前没有通用编码自动探测，所以非 UTF-8 的 TXT/Markdown 会失败，而不是忽略解码错误得到看似正常的乱码。

Markdown 只按一级、二级标题建立首轮边界，并保留标题正文。三级标题不作为首轮强制边界，这是为了避免运维文档被过度碎片化；它不是普适最佳实践。如果某个团队大量使用三级标题承载独立操作，切分策略就应重新评测。

### PDF：页码比“读出了文字”更重要

文本型 PDF 逐页提取，每个页面先成为逻辑单元，metadata 保存 `page_number`。后续即使一页被拆成多个 chunk，这些 chunk 仍能回到同一页。单页 `extract_text` 失败时，该页记 warning 并作为空页处理，其他正常页继续入库，不再让一页损坏拖垮整份 PDF。页数上限为 2000，超出部分忽略并进入质量 warning。扫描 PDF 没有文字层时，当前系统不会自动 OCR；若整份文件没有有效文字，会写入“扫描件需要 OCR”的 warning，并作为 empty 处理。

这说明“支持 PDF”必须说完整：已实现并验证的是文本型 PDF 的页级抽取与定位，不是扫描件 OCR、版面分析、表格还原或任意复杂 PDF。把扫描 PDF 的空结果记成成功，反而会让知识缺失悄悄发生。

### HTML：先去页面噪声，再恢复标题路径

本地 HTML 直接把原始字节交给解析器，让 HTML 声明与解析器共同处理编码；随后移除 `script`、`style`、`nav`、`header`、`footer`、表单控件、侧栏和菜单等噪声，再沿 H1–H3 与正文元素形成标题区块。`li > p`、`pre > code` 等嵌套语义块只取外层一次，避免同一正文重复索引。每个区块保留 `heading_path`、`html_title` 以及可用的 `h1/h2/h3`；最多保留 10000 个区块，超出部分写 warning。

在 Redis Wiki 中，“容量判断 > maxclients 耗尽”应成为可定位区块，而“首页导航”“提交搜索”“相关文章”不应进入索引。当前实现处理的是本地导出 HTML，不会登录 Confluence、Notion 或内部 Wiki 拉取内容，也没有定时同步和权限继承，因此不能称作在线 Wiki connector。

### CSV 与 XLSX：一行就是一条可审计知识

工单、服务目录和 SLO 表格的自然单元通常是行。CSV/XLSX Loader 把每行转换为独立文本，并携带 `sheet_name`、`row_number`、`table_name` 和 `primary_key`：

```text
表格: tickets.xlsx
sheet: incidents
row: 12
primary_key: incident_id=INC-1024
service_name: order-service
summary: Redis connection timeout
```

主键按 `ticket_id`、`incident_id`、`service_name`、`error_code`、`id` 的顺序寻找。CSV 会尝试 `utf-8-sig` 和 `gb18030`，并在逗号、分号、Tab、竖线中探测常见导出分隔符。空表头会生成 `column_n`，重复表头会稳定重命名，数据列多于表头时不会静默丢失，而是补生成列名并写 warning。只有表头没有数据的 CSV 或 sheet 也会留下结构 warning。

XLSX 在解析前先检查 ZIP 解压后总大小，默认上限 100 MB，防止小上传包异常膨胀；工作簿最多读取 32 个 sheet，每个 CSV/sheet 最多读取 10000 行。XLSX 使用只读、`data_only=False`、禁用外部链接的模式，所以公式单元格保留公式文本，不会因缺少缓存结果变成空值，也不会在服务端执行公式。

为避免异常表格拖垮索引，当前每行最多取 24 列，单元格最多保留 500 字符，超出部分被截断并写 warning。这是资源保护策略，不是无损解析。若审批条件藏在第 25 列，或者长单元格末尾恰好是回滚条件，现有结果就可能不完整，质量报告必须让这种截断可见。

## 清洗的目标不是让文本好看，而是让丢弃可解释

Loader 输出后进入公共清洗。系统移除 NUL 和首尾空白，丢弃空单元；普通文档按字母或数字少于 30 个判断低信息，表格行采用 10 个的独立阈值，避免只有标点、分隔符或定位前缀的内容误过门。去重仍使用 SHA-256，但表格按实际行内容去重，不把 `sheet/row` 定位前缀算入摘要，因此两行业务内容完全相同也能识别为重复，同时保留被接受行的定位 metadata。

每个文件形成 `DocumentCleaningReport`，记录 `raw_units`、`indexed_units`、`dropped_units`、空、重复、低信息数量以及 warnings。清洗和表格 warning 都有 200 条上限，超过后用一条 omitted 提示收敛，防止脏文件让报告无限膨胀。持续质量记录使用 JSONL，并通过跨线程、跨进程文件锁串行化追加与读取；路径和失败错误在进入公共记录前收敛为 basename 与公共消息，底层 PDF 路径或解析异常不会原样暴露。

于是，“`redis_postmortem.pdf` 没有 chunk”不再只有一个模糊的异常。可能是文件损坏导致解析失败，可能是扫描件没有文字层，可能是页面全部为空，可能是每页有效信息不足 30 字，也可能是切分阶段失败。这些原因对应不同处理：损坏文件应重新导出，扫描件需要 OCR，低信息阈值可能要调参，切分错误则要修实现。

清洗也有明确边界。当前哈希去重只发生在一次 Loader 运行内部，且只识别完全相同的逻辑单元；它不是跨文件去重，不识别语义重复，也不会自动判断 CSV 与 XLSX 中同一张工单是否应该只保留一份。全库近似重复只在质量评测中观察，是否删除仍需业务判断。

## 切多大没有标准答案，只有可验证的取舍

默认配置中的 `chunk_max_size=800` 容易让人误以为最终块上限就是 800。实际递归切分大小是它的两倍，即 1600 字符，overlap 为 100。Markdown 先按一级、二级标题切，再递归处理过长区块；相邻块只有至少一侧小于 300 字符、二者 `h1/h2` 相同，并且去掉重复 overlap 后不超过 1600 字符，才会合并。不同标题下的“诊断步骤”和“回滚条件”不会为了凑长度而合并。

PDF、HTML、CSV、XLSX 则先遵守 Loader 形成的页、标题区块或行边界，再在每个逻辑单元内部递归切分。系统不会为了获得更大的文本，把 PDF 相邻页面或两行工单先拼起来。这个区别解释了为什么“所有格式都用同一个 LangChain Splitter”并不准确：底层递归算法可以共用，进入算法前的语义边界不同。

块过大会混入多个意图，召回“扩容审批”时把诊断与复盘一起带入上下文；块过小则会把条件、动作和回滚拆开，模型只看到“执行修改”而看不到“仅在审批后”。overlap 能缓解边界丢失，却会增加重复与存储成本。当前合并逻辑会识别相邻块首尾的真实 overlap，只保留一份后再判断上限，避免把递归切分器已经复制的 100 字重复写进合并块。可靠做法不是争论 800 还是 1600，而是同时观察长度分布、空块、metadata 缺失、近似重复以及下游检索指标。

2026-07-12 official Artifact 中仍记录 Max 1858 和 `4/209` 超长块，那是修复前代码与冻结资产的历史事实，不能被当前工作区直接改写。当前 candidate 已把小块合并收紧到 1600 字符以内；要证明新资产分布和超长率已经变化，必须用新 commit、manifest 和 Run ID 重新生成 knowledge-quality Artifact，而不是把单元测试结果覆盖旧基线。

## 一个 chunk 为什么需要三套身份

同一个字段很难同时满足“人类可读、全局唯一、跨版本稳定、适合数据库主键”。AutoOnCall 因此把身份拆成三个层次。

### 可读定位：告诉人去哪里看

`_chunk_id` 形如 `redis_capacity_wiki.html#0001` 或 `tickets.xlsx#0002`。它适合出现在引用和调试界面，能让人理解这是该文件的第几个分片。它不保证全局唯一：不同目录可能有同名文件；在文件前部插入一个 chunk，也可能让后续序号变化。

因此，`_chunk_id` 解决的是可读定位，不是不可变内容寻址。面试时说“chunk ID 永久稳定”是不准确的。

### 文档与内容哈希：告诉系统是哪一版

系统把全部已加载内容做 SHA-256 得到 `_document_hash`，取前 12 位作为 `_document_version`；对当前块正文做 SHA-256 得到 `_chunk_hash`；再形成文件名与短版本组成的 `_version_key`。同名 Runbook 内容变化后，版本随之变化，审计者可以区分旧版与新版。

这些哈希也有边界：只要清洗结果、拼接顺序或切分正文变化，哈希就会变化；它们不表达“这一版已经人工批准”，也不判断内容是否正确。版本身份与治理状态是两回事。

### 向量主键：告诉存储 upsert 谁

Milvus 主键由 source、文档 hash/版本、可读 chunk ID、chunk hash 和正文共同计算 SHA-256，形式为 `vec-...`。相同输入重复入库会得到相同 ID，适合 upsert；内容或版本变化会得到新 ID。写入后该 ID还会回填为 `_vector_id`。

这套主键更接近存储幂等身份，但不适合直接展示给用户。可读 ID、版本哈希和向量主键各自解决一个问题，远比让一个字段承担全部职责可靠。

## Embedding 失败时，宁可整批失败也不要返回错位向量

文档 Embedding 使用 DashScope 的 OpenAI 兼容接口，模型配置默认指向 `text-embedding-v4`，请求维度固定为 1024。文档列表按配置分批，但每批最大 10 条；默认最多重试 2 次，指数退避等待上限 5 秒。服务还检查返回向量数是否与输入文本数相同。

为什么不能把成功的 9 条向量和失败的 1 条直接返回？因为下游通常按顺序把向量与 chunk 绑定，数量不一致可能把 A 文本的向量写到 B chunk 上。这种静默错位比显式失败更危险。因此任一批最终失败，整个 `embed_documents` 抛错，不返回残缺列表，索引编排转入失败补偿。

查询 Embedding 是单次调用，没有复用文档批处理的重试循环。它属于检索时运行边界，不应拿文档入库重试行为替它背书。实际客户端采用延迟初始化：没有真正调用 Embedding 时，不会仅因模块导入就创建外部客户端。

## 双索引为什么能提高召回，却让发布更难

向量索引擅长表达语义相近，例如“连接数耗尽”与“达到 maxclients”；词法索引擅长保护 `ERR max number of clients reached`、命令名、错误码和专有名词。AutoOnCall 同时维护 Milvus 语义索引和本地 JSON 的 BM25-like 词法索引。前者使用 collection `biz`、1024 维、2 个 shard、`IVF_FLAT`、`L2`、`nlist=128`，正文存储字段上限为 8000 字符；后者提取英文/数字 token 以及中文二元、三元片段。

已有 Milvus collection 若维度不是 1024，默认 fail closed，不会自动删库。只有显式设置 `MILVUS_RECREATE_ON_DIMENSION_MISMATCH=true` 才允许删除并重建。这项保护牺牲了开发便利，却避免配置变更在生产环境悄悄清空知识数据。

本地词法索引写入时使用进程内线程锁，先生成临时 JSON，再用 `os.replace` 替换正式文件。它降低了单进程写到一半留下截断 JSON 的概率，但不是跨进程锁，也不是分布式数据库事务。多副本同时写、超大索引性能、复杂过滤与高可用都超出当前实现边界。

### 成功更新的真实顺序

同一个来源更新时，当前顺序是：

```text
1. 新 chunk 生成向量并 upsert
2. 删除该 source 中不属于本批 vector IDs 的旧向量
3. 整体替换该 source 的 lexical chunks
4. lexical 写成功时清除该 source 的 stale 标记
```

先写新向量再删旧向量，可以减少“先删后写、写入失败后完全无知识”的窗口；但步骤 1 到 3 不在同一个事务中。若步骤 1 成功、步骤 2 失败，Milvus 可能短暂保留新旧向量；若向量已完成而步骤 3 失败，词法仍可能是旧版。可靠系统必须承认这个窗口，而不是把三个调用包装成“原子更新”。

### stale source 是查询隔离标记，不是物理回滚

这里出现第一个容易混淆的术语：**stale source（陈旧来源）**。通俗地说，它表示“这个来源的两套索引可能已经不一致，暂时不要信”。严格定义是：索引写入或旧数据清理失败后，记录在词法索引状态中的来源级失效标记；检索阶段据此同时排除该来源的词法候选和向量候选。它解决部分失败被静默使用的问题；不负责撤销已经写入的 Milvus 数据，也不保证 stale 标记本身一定写成功。

例如，新版 Redis Runbook 的向量已经写入，但 lexical replace 失败。系统尝试标记该 source stale；之后即使 Milvus 能搜到它，检索层也不应把它交给回答模型。反例是仅在词法搜索中过滤 stale，却继续使用同 source 的向量结果，那仍然会暴露混合版本。

stale 是 fail-closed 补偿：宁可短期少一个 Runbook，也不静默混用新旧审批规则。它不是事务回滚，仍存在两个极端窗口：进程可能在失败后、写 stale 前崩溃；stale JSON 本身也可能写失败。更强的方案是为向量与词法都建立不可变版本，两个版本准备完成后再原子切换查询别名。

### 空内容更新为什么也要执行删除补偿

如果同名 `redis.md` 以前有 10 个 chunk，现在被更新为空文件，不能简单返回“没有内容”并保留旧索引，否则已删除的知识会继续被召回。当前系统先删除该 source 的全部向量，再删除 lexical chunks，最终返回 `empty`，说明旧索引已经清理。

如果向量删除失败，系统不会继续删除 lexical 并制造两侧不一致，而是进入异常路径并标 stale。这里同样不是原子事务，但至少失败可见、查询可隔离。当前没有独立的 source 删除 API；空内容覆盖触发的清理不等于已经实现完善的知识生命周期、回收站或审核删除。

## 质量记录与评测 Artifact 解决的是两个不同问题

每次上传或目录索引会把文件级清洗结果追加到 JSONL 质量记录中，包括操作类型、状态、chunk 数、耗时、格式、原始/有效/丢弃单元、warning 和公共错误。质量查询可以按 `doc_type` 聚合并列出 empty、failed、有 warning 或发生丢弃的低质量文件。

这个读模型回答“最近哪些文件入库有问题”，不等于一次冻结的性能评测。它是持续运行记录，可能包含多次上传同一文件；若直接求和，分母是记录次数而非唯一资产数。相反，benchmark Artifact 会冻结资产集合、命令、代码版本和环境，适合比较某次候选是否达标。

还要区分两种 stale：source stale 是双索引发布失败后的查询隔离状态；评测 Artifact 的 `stale` 是当前代码、工作区或资产 manifest 已偏离当时运行指纹，历史结果不能代表当前候选。二者都表达“不应直接信任”，但对象、触发条件和处理方式完全不同。

## Redis Runbook 的完整入库推演

现在把四份 Redis 资料沿整条链走一遍，并持续回答“已经知道什么、还缺什么、为什么继续”。

### 第一步：文件进入受控边界

上传者具备知识写入权限，文件名、扩展名和大小通过校验。此时只知道文件字节已经安全落盘，不知道格式能否解析，也不知道索引是否完成。若同名文件覆盖，正式文件在完整接收后才被替换。

`redis_capacity_wiki.html` 由 HTML Loader 去掉菜单、页脚和脚本，把“容量判断”“升级与审批”“回滚边界”恢复成带标题路径的逻辑区块。`official_redis_clients.md` 保留一级、二级标题。`redis_postmortem.pdf` 按页保留页码。`tickets.xlsx` 按 sheet 和行生成逻辑单元，并尽量提取 `incident_id` 或 `ticket_id` 作为主键。

此时我们知道各段知识来自哪里；仍不知道它们是否足够长、是否重复、是否能形成健康 chunk。

### 第二步：清洗与切分让缺失原因可见

公共清洗丢弃空、低信息和同文件完全重复单元，报告每类数量。切分器在每个来源边界内控制长度，保留页、标题或行定位。每个 chunk 获得 `_chunk_id`、文档/内容哈希和版本键。

此时我们知道可以发布多少 chunk，也可以解释为什么某页或某行没有进入索引；仍不知道 Embedding 和两个存储是否成功。

### 第三步：双索引发布并清理旧版本

Embedding 分批生成 1024 维向量，Milvus 以稳定主键 upsert；随后清理该 source 的非本批旧向量，再整体替换词法 source。全部完成后，来源退出 stale 状态。任一步异常都使整次索引返回失败，并尝试标 source stale。

此时可以说“该版本知识已具备被检索的资格”，仍不能说“检索一定会选中正确 chunk”，也不能说“连接耗尽就是本次 Incident 根因”。下一篇会继续处理召回和 Trust Gate。

### 第四步：现场诊断仍要把知识与事实分开

后续 Redis 调查如果召回“连接使用率接近上限时检查泄漏与连接池”的 Runbook，它只是知识依据。只有事故窗口的 `connected_clients`、`blocked_clients`、timeout 日志和服务影响，才能成为现场 Evidence。Runbook 可以指导下一步，也可以提供审批与回滚边界，但不能替代实时观测。

MySQL 的历史慢查询表和 Kubernetes 的发布 Runbook 可以沿用相同链路，差别主要在文档结构与业务主键。它们用于说明方法可迁移，不占据本篇 Redis 主线，也没有因此获得生产验证。

## 量化验收：20/20 不是“支持任意文档 100%”

入库质量至少要同时回答文件、chunk、存储和来源四个问题：发现的资产是否都能解析与切分？chunk 是否为空或失去引用？双索引的主存储 CRUD 是否一致？这组数字来自什么冻结样本和环境？

2026-07-12 的 `local_live` 知识质量验收使用 20 份仓库资产：Markdown 14、PDF 2、HTML 2、CSV 1、XLSX 1。

| 指标 | 结果 | 为什么测 / 能证明什么 / 不能证明什么 |
|---|---:|---|
| 解析成功率 | `20/20 = 1.00` | 证明当前 20 份资产都通过生产 Loader；95% Wilson CI `[0.8389, 1.0000]`，不能外推为任意格式 100% |
| 切分成功率 | `20/20 = 1.00` | 证明当前样例能形成 chunk；覆盖文本型 PDF，不包含扫描 PDF OCR |
| Index-ready | `20/20 = 1.00` | 证明当前 metadata 与索引前置条件满足；不证明线上检索排序正确 |
| Chunk 数量 | `209` | 给出本次固定资产的产物规模；平均长度 `902.34` 字符 |
| Chunk P50/P95/Max | `762 / 1596 / 1858` | 观察粒度分布；不是越短越好，也不是运行时绝对上限声明 |
| 空 Chunk | `0/209` | 当前样本没有空块；95% Wilson upper `0.0180`，仍不能声称未来永不产生空块 |
| metadata 缺失 | `0/209` | 当前资产的基础引用字段齐备；不等于所有引用内容语义正确 |
| 超长 Chunk | `4/209 = 1.91%` | watch metric，要求复核；不能因总体验收通过而隐藏 |
| 完全/近似重复 | `0 / 8` | 近似重复率 `3.83%`，说明仍存在观察项；其中可能包含刻意的多格式样例 |
| Milvus CRUD 一致性 | `passed` | 证明本地临时 collection 的写、读、删和清理链运行；不证明生产容量、HA 或性能 |

比例指标必须保留分子和分母，因为 `20/20` 与 `20000/20000` 都显示 100%，不确定性却完全不同。Wilson 区间比“满分”更诚实地表达小样本边界。P50 表示一半 chunk 不超过 762 字符，P95 表示 95% chunk 不超过 1596 字符；Max 1858 解释了为什么超长块被单独设为观察项。

这些数字来自 **official benchmark** run `20260712T074629Z-81f1f98c-0d69abdc`。official 表示满足当时的冻结、可复现与 clean commit 规则，不表示 production evidence。命令为 `scripts/eval/run_benchmark_baseline.py --output-root logs/benchmarks`；Artifact 为 `logs/benchmarks/20260712T074629Z-81f1f98c-0d69abdc/knowledge_quality.json`；Git commit `81f1f98ccbc4588f8470ad82d3fb82ad849c431f`，`git_dirty=false`；知识资产 manifest SHA-256 `d4fda79b4530f150896344ab3f24001d9b4e973912a88ea9873159ee75e2bcab`。

该 Artifact 的 `local_live` 指真实仓库文件经生产 Loader/Splitter，并对临时 Milvus collection 做 CRUD；向量使用确定性本地实现。评测临时 collection 使用本地确定性 128 维向量与 FLAT 索引来验证存储一致性，这与应用运行时的 DashScope 1024 维 Embedding、`biz` collection 和 IVF_FLAT 配置不是同一件事。它能证明上述 20 份资产与本地真实 Milvus 存储链，不能证明真实 DashScope Embedding、生产 Milvus 性能、OCR、在线 Wiki 或持续同步。

完整 Scorecard、候选转 official 的规则和置信区间口径见[第 15 篇](15-项目量化指标与校招验收.md)。本篇保留入库机制直接相关的指标，是为了让读者在理解设计时就看到它被什么证据支持，而不是把全部数字移到总表。

## 结构质量通过以后，内容治理为什么仍不能省略

解析成功、引用完整和双索引一致，只能证明“系统忠实发布了文件里的内容”，不能证明文件本身值得值班工程师遵循。假设 `redis_capacity_wiki.html` 仍写着旧集群的 `maxclients` 调整方法，Loader 可以完美恢复标题，209 个 chunk 的质量指标也可能全部通过，但这个知识在新集群上依然可能危险。结构质量回答“有没有丢、能不能找回来源”，内容治理回答“谁批准、何时生效、适用于哪里、何时过期”，二者不能互相替代。

一个更完整的知识资产通常还需要责任人、适用服务与环境、审核状态、生效时间、过期时间、替代版本和敏感级别。对于变更类 Runbook，还应明确前置检查、审批角色、观察窗口和回滚条件。它们可以来自文件头部、外部目录或独立治理数据库，但必须进入可过滤、可审计的 metadata，而不能只藏在正文里等待大模型理解。

例如，Redis 文档中的“把 `maxclients` 调到 12000”只有在目标集群、内存与文件描述符前置条件、审批状态和版本都匹配时，才可能成为建议依据。若同一文件同时写着生产与测试命令，至少要用环境 metadata 区分；若新版已经替代旧版，应让旧版退出可见发布，而不是仅依赖相似度把新版排在前面。入库链保证“版本不静默混用”，但当前并没有完整实现内容审批、所有权和生效策略，不能把 hash 当成批准签名。

安全边界也不止路径遍历。运维文档可能包含 token、内网地址、客户数据或可直接执行的高风险命令。扩展名与大小检查无法识别这些内容。生产化通常还需要恶意文件扫描、秘密信息检测、文档级访问控制、加密、审计和删除证明。AutoOnCall 当前的知识写入 scope 保护接口，公共响应隐藏本地路径和底层错误，但没有证据表明它已经具备企业级内容防泄漏与租户隔离。

因此，判断一个来源能否进入回答，可以把门槛分成四层：文件层确认来源允许读取；结构层确认正文、定位和 chunk 健康；发布层确认双索引版本一致；治理层确认内容已批准、未过期且调用者有权访问。本篇实现重点覆盖前三层中的工程基础，第四层仍是生产升级边界。下一篇的 Trust Gate 会在检索候选上继续做质量与来源约束，却也不能凭空补齐不存在的审批元数据。

## 失败路径决定了这套链路的真实上限

### 正常路径之外的五种结果

一个文件的最终结果至少应区分：

| 结果 | 含义 | 调用方下一步 |
|---|---|---|
| `success` | 有 chunk，向量写入、旧向量清理和 lexical replace 完成 | 可进入检索，但仍受召回门禁约束 |
| `empty` | 没有可检索 chunk，旧索引已清理 | 检查空文件、低信息、扫描 PDF 或清洗阈值 |
| `failed` | 解析、切分、Embedding、存储或补偿链出现异常 | 查看服务端日志与公共质量记录，source 可能被标 stale |
| `partial_success` | 上传文件成功，但索引未 ready | 文件可修复或重试，不能宣称已入库 |
| `forbidden/invalid` | 路径越界或目录无效 | 修正权限、允许根或运维参数，不进入解析链 |

目录索引会继续处理其他文件，并分别统计 success、empty 与 failed；只要出现 empty 或 failed，目录总体不会被包装为完全成功。它适合运维批处理，不是跨文件事务：第一个文件成功、第二个文件失败时，不会回滚第一个文件。

上传入口还有一层容易被忽略的错误优先级。文件名、MIME、大小、签名或 UTF-8 校验失败时，正式文件未替换，接口返回 400；文件已经安全落盘，但解析、切分或索引未 ready 时，返回 207；只有产生正数 chunk 且索引服务返回合法 `success` 才是 200。质量 JSONL 写入失败是旁路观测故障，不会反向篡改已经完成的索引结果。这样可以把“输入无效”“文件已保存但知识未 ready”“知识 ready”“观测旁路失败”四种责任分开。

### 哪些问题当前仍未解决

- 没有 OCR、版面分析和复杂 PDF 表格恢复。
- 没有在线 Wiki connector、文件 watcher、增量抓取和来源权限同步。
- 没有独立 source 删除 API、审核发布、回收站和完整知识生命周期。
- Milvus 与本地 lexical 没有跨库事务；stale 只是查询隔离补偿。
- lexical JSON 只有进程内锁和原子文件替换，不适合多副本并发写和大规模索引。
- 同名上传锁依赖共享且可靠的文件锁语义，不是跨地域分布式锁，也没有上传幂等键或不可变对象版本。
- 没有全库语义去重；近似重复目前主要通过评测观察。
- 真实 DashScope、生产 Milvus 容量、吞吐、HA/DR 尚无本篇 production evidence。

这些边界不是“项目没做完”的简单清单，而是架构选择的结果。课程项目优先把来源、身份、失败和审计做实，再保留向版本化发布、分布式索引和在线同步升级的接口。若场景只有少量受控 Runbook，本地 lexical 与 stale 补偿是合理成本；若面对数百万文档和多租户并发，就应选择专用搜索引擎、发布队列、不可变版本与别名切换。

## 设计取舍：当前方案不是唯一答案

**为什么不先统一转纯文本？** 因为统一文本会丢页码、标题、行号和主键。可以先转统一中间格式，但中间格式必须保存这些来源结构，而不是只保留正文。

**为什么 Markdown 不按每个标题切？** 更细标题能提高定位，却可能造成大量短块。当前选择一级、二级标题加递归限长，再以实际质量指标校验。不同知识库可以选择 AST 切分、语义切分或模型辅助切分，但都要保留可复现参数与来源定位。

**为什么先写新向量再删旧向量？** 它减少更新失败后完全无向量的窗口，但会引入短暂重复和双写不一致。先删后写、事务日志、双版本构建加别名切换各有不同可用性与复杂度取舍。

**为什么 stale 选择不可用而不是旧版可用？** 运维知识包含审批、回滚和生产命令，混合版本的风险可能高于短期缺失。若业务更看重可用性，可以把“最后一个已批准版本”作为独立 immutable release，更新失败时继续服务旧 release，而不是服务半新半旧状态。

**为什么质量长度是 watch，而空块是 hard gate？** 长块是否有害要结合语义和检索验证；空块和引用缺失则几乎没有合理用途。门禁与观察项分开，避免为追求所有数字为零而破坏语义完整性。

### 高频追问与回答要点

1. **为什么 Loader 和 Splitter 必须分开？** Loader 恢复页、标题、行等来源边界，Splitter控制检索长度；混在一起会让来源定位和粒度策略互相污染，也难以归因失败。
2. **为什么 Markdown 不直接固定长度切？** 标题是语义边界。先按标题再限长，能减少诊断、审批和回滚被拆散；仍需用检索指标评估，而非认定标题切分总是最优。
3. **`_chunk_id` 是否永久稳定？** 不是。相同文件与切分结果下可复现，但前部插入块会改变序号；文档/内容 hash 与向量主键承担版本和存储身份。
4. **向量成功、lexical 失败怎么回滚？** 当前不做物理原子回滚，而是标 source stale，检索同时排除两路候选；更强方案是双版本准备完成后切换别名。
5. **为什么 Embedding 一批失败不能返回部分向量？** 返回数量与输入不一致可能造成向量和 chunk 错位。当前整批失败，让编排层统一进入补偿，比静默写错更安全。
6. **扫描 PDF 会怎样？** 无文字层时产生 OCR warning，并成为 empty；当前不自动 OCR，不能包装成解析成功。
7. **上传为什么可能返回 207？** 文件落盘与索引 ready 是两个阶段。文件已保存但解析或索引失败时，用 207 暴露 partial success，调用方不能立即检索。
8. **空文件覆盖为什么要删除旧索引？** 否则用户已经删除的知识仍会被召回；当前先删向量再删 lexical，删除失败则标 stale。
9. **为什么需要向量和词法两套索引？** 向量保护语义变体，词法保护错误码、命令和专名；代价是发布不再天然原子，需要补偿和查询门禁。
10. **`20/20` 为什么不等于任意文档 100%？** 分母只有 20 份冻结资产，95% Wilson 下界仍是 `0.8389`，且不含扫描 PDF、在线 Wiki 和生产负载。
11. **质量 JSONL 与 benchmark Artifact 有什么区别？** 前者是持续的文件级运行记录，后者冻结样本、代码和环境用于可复现验收，不能混成同一个统计分母。
12. **source stale 与 Artifact stale 有什么区别？** 前者隔离双索引不一致来源，后者说明历史评测指纹已不匹配当前候选；对象和治理动作不同。
13. **同名并发上传为什么要锁到索引完成？** 只锁 `replace` 时，前一请求的后台索引可能在后一请求覆盖后读取错误版本；当前同名路径串行化整个落盘与索引事务，请求取消也等工作线程结束再释放锁。
14. **MIME 校验为什么不能替代文件签名？** MIME 来自调用方，可能缺失或伪造；PDF/XLSX 强签名检查用于阻止明显容器错配，但仍不是病毒扫描或内容安全审查。
15. **XLSX 为什么读取公式而不是缓存值？** `data_only=False` 保留公式文本，避免没有缓存结果时静默为空；服务端不会执行公式，因此也不能把公式结果当已计算事实。

## 用练习检验是否真正理解

1. 给 `tickets.xlsx` 设计一条从上传、sheet/row/primary key、chunk ID、向量主键到最终 citation 的完整数据流，指出每一步丢失定位信息会造成什么后果。

3. 构造“空 PDF 覆盖已有复盘”的场景，分别讨论向量删除成功、向量删除失败时的最终状态。
4. 设计版本化双索引发布：两个新版本都 ready 后才切换查询别名；列出构建、校验、切换、回滚和垃圾回收状态。

8. 比较三种更新策略：先删后写、先写后删加 stale、双版本别名切换；从可用性、存储成本、实现复杂度和错误可恢复性评估取舍。

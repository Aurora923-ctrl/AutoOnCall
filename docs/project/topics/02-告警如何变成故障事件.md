## 十分钟内来了四条消息，究竟发生了几次故障

02:07，`order-service` 出现 503、P95 上升，日志反复出现 `Redis connection timeout`，`connected_clients` 接近 `maxclients`。这些是告警携带的现场现象；Redis 官方限制属于知识，“连接耗尽就是根因”仍只是推断，事故窗口与客户端池状态仍未知。Alertmanager 发来 `firing`：

```text
alertname=RedisConnectionPressure
service=order-service
environment=prod
fingerprint=redis-order-prod-001
```

02:08 分组刷新，同一 fingerprint 又发一次 firing；02:15 指标短暂恢复，发来 resolved；02:17 连接再次上升，又发 firing。

在人的直觉里，这四条消息描述的是同一条故障线索：首次触发、重复刷新、暂时恢复、再次触发。对一个只会“收到 JSON 就创建任务”的程序来说，它们却可能变成 `inc-1` 到 `inc-4`：两次重复模型调用、一个找不到原任务的恢复通知，以及一份丢掉此前调查上下文的新诊断。

先把事实边界说清楚。503、P95、日志超时和连接数是告警携带的现场现象，仍要核验来源和时间窗口；Redis 连接上限机制与 Runbook 属于知识；“连接耗尽就是唯一根因”是 **Hypothesis**，即可以被后续材料支持或反驳的根因假设。**Evidence** 是带来源、事故窗口和查询上下文的观测，例如同一窗口内 `connected_clients/maxclients` 的采样。告警能够启动调查，却不能单独完成因果证明。

真正的问题不是 JSON 反序列化，而是连续回答五件事：哪些消息属于同一业务故障，外部字段怎样变成稳定内部契约，重复/恢复/重开各自意味着什么，谁有权更新当前状态，以及摄入是否应该自动触发昂贵诊断。把这些问题塞进一张消息表，会让身份、诊断输入和工作台状态互相污染。

## 四个朴素方案为何会在真实告警流中失败

### 每条消息生成随机 ID

随机 ID 适合标识一次不可重复的尝试、Trace event 或 diagnosis session，却不适合标识一条会重复刷新的业务故障。Alertmanager 会重复发送仍在 firing 的分组，网络重试也可能复制请求。每条消息都创建 Incident，会使 resolved 无法定位原调查，值班工程师也不知道应该审批哪一个任务。

同一 Incident 可以有多轮诊断 session。前者表示业务故障域，后者表示一次具体执行尝试。把二者混为一个 ID，既会丢掉重开的新轮次，也会让重复消息制造新故障。

### 只用 alertname 去重

如果直接用 `RedisConnectionPressure` 作为主键，生产 `order-service` 与测试环境的同名告警会被误合并。若反过来把所有 labels 都加入主键，Kubernetes Pod UID、滚动发布 Pod 名等易变标签又会把一次服务级故障拆成几十个 Incident。

身份字段太少会误合并，太多会误拆分。fingerprint 不是监控系统赐予的天然真理，而是对“故障对象是什么”的业务选择。Redis 服务级故障、MySQL 实例级故障和 Kubernetes 服务级故障，理应采用不同的身份策略；当前统一 fallback 规则只是一个可解释起点。

### 把原始 JSON 直接交给 Agent

外部字段可能叫 `service`、`service_name`、`app`、`app.kubernetes.io/name` 或 `job`；环境可能是 `environment`、`env`、`stage` 或 `cluster`；严重度可能是 `critical`、`warning`、`P1`，甚至缺失。若 Planner、Executor、报告和页面各自解析原始 JSON，每个模块都会重复处理外部差异。

payload 还可能带 token、Authorization、DSN、超长 URL 和大段 annotation。原样写入 Prompt、日志、Trace 或数据库，会同时放大泄漏、成本和存储风险。外部消息是输入材料，不应自动成为内部长期契约。

### 一张记录同时保存消息、调查和页面状态

resolved 想更新监控状态，诊断流程正在写 `waiting_approval`，safe-change 又准备进入 `change_dry_run`。如果只采用最后写入者获胜，晚到 resolved 可能把等待审批改成“已恢复”，旧报告也可能把正在执行的变更拉回普通完成状态。

监控恢复与处置完成描述不同维度。Redis 指标暂时回落只说明告警条件消失，不证明连接泄漏已修复，也不允许跳过审批。由这些失败可以推导出一个更稳妥的结构：外部最新快照、Agent 调查输入和查询生命周期必须分层。

## 三个对象描述同一故障，却回答三个不同问题

### AlertEvent：外部告警的最新标准化快照

通俗地说，**AlertEvent** 是“监控最近一次怎么说”。严格地说，它是以 fingerprint 为身份的标准化外部告警最新快照，保存来源、Incident ID、`firing/resolved`、告警名、服务、P1–P4 严重度、环境、摘要、描述、labels、annotations、起止时间、生成器 URL、受控原始载荷及创建/更新时间。

它解决外部字段差异、列表查询和最新状态更新问题；不负责保存完整不可变消息历史，也不直接证明根因。Redis 的 02:08 firing 会覆盖同 fingerprint 的 AlertEvent；如果要回答“02:07 与 02:08 两条请求分别是什么”，当前 latest upsert 无法提供答案。

单独调用 `save_alert_event` 时，SQLite 使用 `ON CONFLICT(fingerprint) DO UPDATE`，MySQL 使用 `ON DUPLICATE KEY UPDATE`；正式 webhook 摄入则通过后文的 `persist_alert_ingestion`，把 latest 裁决、AlertEvent 与 IncidentState 放进同一事务。两种路径都只保留一个最新行，仍不能证明消息投递和后台诊断副作用只发生一次。AlertEvent 更接近物化快照，而不是 append-only event log。

### Incident：供诊断消费的稳定内部契约

通俗地说，**Incident** 是“一次故障调查的病历首页”。严格地说，它是包含 `incident_id`、标题、服务、P1–P4 严重度、症状、开始时间、环境、有大小边界的 `raw_alert` 以及调查状态和时间戳的结构化输入。它解决 Planner/Executor 不应理解 Alertmanager 全部变体的问题；不负责保存每一次状态迁移，也不等于某条 webhook。

手工诊断也能直接构造 Incident，不必伪造 webhook。当前仓库没有独立 Incident repository，长期查询依赖 IncidentState、Snapshot、Trace、Report 和 Approval 等事实。

这里的 **Trace** 是诊断节点、工具、风险、审批和错误等追加式审计事件，解决“过程发生了什么”，不等于原始日志仓库；**Snapshot** 是一次 diagnosis session 的最新业务状态，解决查询与有限恢复，不是任意节点数据库 checkpoint。二者会在后续专题展开，本章只需理解它们不是 Incident 本身。

### IncidentState：供查询的最新生命周期投影

通俗地说，**IncidentState** 是“大厅屏幕上现在进行到哪一步”。严格地说，它按 `incident_id` 保存最新生命周期投影，汇集状态、原因、服务、摘要、根因、Trace/Session/Report ID、审批和人工动作标记，以及有大小边界的 metadata。它解决工作台和 API 对当前状态的统一查询；不负责保存每次迁移的完整历史，因此不是事件溯源状态机。

例如告警已 resolved，但诊断正在 `waiting_approval`，IncidentState 应保留更深的工作流状态，只更新告警 metadata；反例是让晚到 resolved 把等待审批直接改成“已恢复”。

后续的 **Replay** 会把 Trace、审批、变更、报告和 IncidentState 组合成复盘视图。Replay 解决“如何解释已有事实”，不重新执行工具，也不会把 latest projection 变成完整历史。

| 要回答的问题 | 对象 | 主身份 | 当前是否保留完整历史 |
|---|---|---|---|
| 外部监控最近怎么说 | AlertEvent | fingerprint | 否，latest upsert |
| Agent 应调查什么 | Incident | incident_id | 无独立 repository |
| 页面当前显示什么 | IncidentState | incident_id | 否，latest projection |

分层的代价是同一业务事实可能出现在多处，需要明确写入所有权与读模型优先级；收益是 resolved 不会天然覆盖审批，Agent 也不必理解每种监控 payload。

## 身份设计：fingerprint 的目标不是绝不冲突，而是稳定表达故障域

最小逻辑如下：

```text
event = normalize(webhook_alert)
event.incident_id = stable_id(event.fingerprint)
incident = build_incident(event)
event, state, created, previous_status, stale_ignored, reopened
    = store.persist_alert_ingestion(event, incident)
```

这里最重要的变化不是把多个函数合成一个名字，而是把“读取旧 Alert、裁决新旧、构建 State、写 AlertEvent、写 IncidentState”收进 Store 的同一事务边界。SQLite 使用 `BEGIN IMMEDIATE` 串行化写事务；MySQL 先做幂等插入，再用 `SELECT ... FOR UPDATE` 锁定 AlertEvent 与 IncidentState 行。只有两份 latest projection 都成功，摄入才会返回成功；任一写入失败都会回滚该条告警。

### 上游有 fingerprint：优先信任，但限制长度

若上游提供非空 fingerprint，系统去掉首尾空白后使用。模型允许的 fingerprint 最长为 `128`；超过该长度时不是截取前缀，而是对完整值做 SHA-256，得到 `64` 个十六进制字符。这样避免超长外部值突破数据库约束，也比直接截断更少出现相同前缀碰撞。

空白字符串不算有效身份。自动测试覆盖空白 fingerprint 的 fallback，并让 `instance=redis-a` 与 `instance=redis-b` 产生两个不同的 `64` 字符 fingerprint。上游 ID 的质量仍由上游负责；本项目没有验证其全局唯一性。

### 上游无 fingerprint：有限稳定字段共同决定身份

若缺失，则对 `source`、`alertname`、规范化 service、environment，以及存在的 `namespace / pod / instance / job / severity / cluster` 进行键排序、拼接并计算 SHA-256。

字段太少会把不同故障误合并，太多会因易变标签把同一故障拆散。当前实现没有按 alertname 配置独立 identity policy，统一字段集合只是通用启发式。生产化时应为高价值告警定义故障域，并观测误合并率、误拆分率与上游 fingerprint 漂移。

Incident ID 稳定派生为：

```text
inc-alert-<fingerprint SHA-256 的前 12 位>
```

因此重复 firing、resolved、reopened 都落到同一个 Incident。前 `12` 位是工程折中，不是数学意义的零碰撞保证；规模增长后可增加位数或对完整业务键建立唯一约束。

### 稳定身份、幂等写和 exactly-once 是三个层次

稳定身份只保证同一 fingerprint 能定位同一业务对象；latest upsert 保证冲突后数据库最终是一行；它们都不保证消息只到达一次或后台诊断只运行一次。

**去重不等于丢弃消息**：已有 fingerprint 的当前或更新一代消息仍可刷新 AlertEvent 和 metadata。更容易误解的是 `deduplicated=true`：其严格含义是 fingerprint 之前存在，不是本次 payload 与上次逐字段相同。因此 firing→resolved 的 resolved 也为 deduplicated，新一代 firing 的 reopened 同样如此。状态变化必须同时查看 `previous_status`、`status_changed`、`reopened` 和 `stale_ignored`；最后一个字段表示请求被接收和识别，但因属于更旧生命周期而没有改写 latest projection。

随机 session ID 与稳定 Incident ID 也不冲突。同一 Incident 的首次诊断和重开诊断需要不同 session，以便 Trace/Snapshot 区分执行轮次；Incident ID 负责把它们聚合回同一故障。

## 标准化不是清洗细节，而是内部契约的防火墙

### 公共字段与单条字段怎样合并

Alertmanager webhook 可能同时给出 commonLabels/commonAnnotations 和单条 alert 的 labels/annotations。系统先读取公共字段，再用单条字段覆盖同名键。具体告警胜过分组公共值，能避免一条 alert 的明确 service 被组级默认值覆盖。

服务名按 `service → service_name → app → app_kubernetes_io_name → app.kubernetes.io/name → job` 推断，缺失时为 `unknown-service`。环境按 `environment → env → stage → cluster` 推断，缺失时为 `unknown`。fallback 让下游始终能构造模型，但 unknown 不是可靠业务事实；它应成为标签治理问题，而不是让 Agent自行猜测。

### 外部状态为何压缩成两种

`firing/active/triggered` 统一为 `firing`；`resolved/inactive/ok/closed` 统一为 `resolved`。webhook 摄入采用严格模式：未知或空状态直接作为无效生命周期字段拒绝，API 返回 HTTP 422，不再默认为 firing。列表查询仍只接受有限状态枚举。这样把“保守保留调查”从隐式猜测改成显式契约，避免上游新增 `pending` 等状态后被误触发诊断。

严重度映射也有确定规则：`p0/p1/critical/page/fatal` → P1，`p2/warning/warn/major` → P2，`p3/info/notice/minor` → P3，`p4/debug` → P4，未知值默认 P2。默认 P2 是路由契约，不是模型判断出的业务影响。

需要纠正一个容易写错的状态名：当前 firing 投影为 IncidentState `alert_firing`，resolved 投影为 `resolved`。`alert_resolved` 虽保留在可被告警更新的状态集合中，但当前 `incident_status_from_alert_status` 的 resolved 目标并不是它。

### 长度边界和脱敏为什么属于正确性

AlertEvent 的 summary、description 等文本最多 `4096` 字符，URL 最多 `2048`，fingerprint 模型上限 `128`。Incident 的 `raw_alert` 序列化后最多 `20_000` 字符，IncidentState metadata 同样最多 `20_000` 字符。上限既控制数据库与 Prompt 体积，也避免恶意或异常 payload 无限传播；它不负责完整取证原文。

`aiops_store_raw_external_payload` 默认 `false`。默认 raw payload 标记 `raw_truncated=true`，只保存 receiver、status、group/common labels、common annotations、external URL，以及当前 alert 的 status、fingerprint、labels、annotations、起止时间和 generator URL，不在每条记录中重复整个 alerts 数组。

labels、annotations 与 raw payload 都执行敏感 key/value 脱敏。API token、password、Authorization、DSN、access_key 等会被替换为 `[REDACTED]`；内联 token 和 Bearer 也会清洗。即使显式开启完整外部 payload，webhook 和 alert 仍递归脱敏。配置扩大的是留存范围，不是绕过安全边界。

脱敏不代表零风险。未知凭据格式、客户标识和业务隐私不一定命中 secret 规则，生产仍需数据分类、加密、访问控制、retention 与抽样审计。

### 时间字段既要合法，也参与有限的代际裁决

`startsAt/starts_at` 是必填字段，必须是带时区的 ISO-8601 时间；resolved 还必须提供 `endsAt/ends_at`，且结束时间不能早于开始时间。解析后统一转换为 UTC。Alertmanager 对仍在 firing 的告警可能发送 `0001-01-01T00:00:00Z` 作为零值结束时间，系统将它规范为 `None`。非法时间、无时区时间和缺失必填时间都会在持久化前返回 HTTP 422。

同一 fingerprint 下，`startsAt` 被视为告警 occurrence 的代际时间。incoming `startsAt` 小于 latest 时记为 `stale_ignored=true`，旧 resolved 不能关闭新 firing；同一 `startsAt` 下，resolved 后再到的 firing 也视为旧消息，不能隐式重开；重复 resolved 还会比较 `endsAt`，更早的结束时间不能覆盖更新结果。只有 `startsAt` 更大的 firing 才是新一代，并允许 reopened。

这是一套针对 Alertmanager 生命周期的有限乱序规则，不是通用事件版本协议。系统仍没有 source sequence、received_at、相同版本冲突日志或 append-only receipt；若上游复用错误的 `startsAt`，或者不同来源的时钟语义不一致，单靠时间仍无法完成普适裁决。

## 生命周期不是简单覆盖，而是多个写入者的所有权协议

同一 fingerprint 的严格返回语义如下：

| 上一快照 | 本次状态与时间 | created | deduplicated | status_changed | reopened | stale_ignored | 浅层 IncidentState 目标 |
|---|---|---:|---:|---:|---:|---:|---|
| 不存在 | firing，`startsAt=t1` | true | false | false | false | false | `alert_firing` |
| firing `t1` | firing `t1` | false | true | false | false | false | `alert_firing` |
| firing `t1` | resolved `t1` | false | true | true | false | false | `resolved` |
| resolved `t1` | firing `t2 > t1` | false | true | true | true | false | `alert_firing` |
| firing `t2` | resolved `t1 < t2` | false | true | false | false | true | 保持 `alert_firing` |
| resolved `t1` | firing `t1` | false | true | false | false | true | 保持 `resolved` |

批次级 `received` 是通过结构检查的 alert dict 数，`created` 是此前没有 fingerprint 的数量，`deduplicated` 是已有 fingerprint 的数量，`resolved` 只统计实际接受并写入的 resolved，`stale_ignored` 单独统计被代际规则拒绝的旧消息。重复但不 stale 的 resolved 仍会增加本批 resolved 计数，所以它不是恢复成功率。`reopened` 表示出现了更大的 `startsAt` firing，不要求上一状态一定为 resolved；这让上一代仍显示 firing、但新 occurrence 已开始的情况也能重新诊断。

### 告警只拥有浅层生命周期

可被告警更新的状态包括 `created`、`investigating`、`alert_firing`、`alert_resolved` 和 `resolved`。当现有状态进入 `diagnosing`、`waiting_approval`、审批结果或 safe-change 等更深阶段，新 firing/resolved 通常只更新 fingerprint、alert status、labels、annotations 与起止时间，并写 `preserved_incident_status`；原 status 与 status_reason 保持。

这样页面可以同时表达 `alert_status=resolved` 和 `status=waiting_approval`：监控条件消失，但处置流程仍需人决策。resolved 不是 Incident 完成的同义词。

### 自动诊断失败是一个受控例外

后台诊断失败时，IncidentState 进入 `failed`，metadata 标记 `alert_auto_diagnosis_status=failed`。builder 特别允许后续告警覆盖“仅由自动诊断失败造成的 failed”。例如随后 resolved 到达，status 可变为 resolved，但失败 metadata 保留。

该例外防止模型服务暂时不可用让工作台永久卡在 failed，却不表示诊断故障已经修复。复盘仍应展示失败 Trace 与安全摘要。

### Store merge 再执行一次迁移守卫

Alert builder 先判断告警是否能覆盖现有 State；Store 保存时又读取当前 State 并执行通用 merge。若现有状态属于 change lifecycle，而更新来自普通 report 且只是 report-only 状态，Store 保留 change status、reason、approval 和人工动作标记，同时合并新的非空信息与 metadata。

修复后的 merge 还增加了确定性迁移守卫：审批状态不能被晚到的 diagnosing/running 等运行态拉回；change lifecycle 不能被非 change 更新回退；`completed/resolved/closed/failed` 等硬终态拒绝隐式重开。只有更大 `startsAt` 的 Alertmanager firing，或带明确 auto-diagnosis failure metadata 的恢复事件，才允许离开终态。被拒绝的更新不会替换 title、service、summary、root cause、trace/session/report 等业务字段，并在 metadata 记录 `ignored_status_update` 与来源。

它仍不是完整事件溯源 FSM：没有枚举所有状态之间的合法边，没有版本化迁移历史，也没有统一 transition event 表。准确说法是“latest projection 上的高风险回退守卫与显式 reopen 例外”，而不是“所有状态迁移都由严格 FSM 证明合法”。

## 原子摄入解决了两张 latest 表的一致性，但没有解决所有副作用

### 单条告警的数据库保证

AlertEvent 以 fingerprint 唯一，IncidentState 以 incident_id 唯一。`persist_alert_ingestion` 在一个数据库事务中锁定或串行化该条告警，读取 latest AlertEvent 与 IncidentState，完成 stale/new-generation 裁决，构建并 merge State，再一起写入。已有 AlertEvent 保留 created_at、接受的新事件刷新 updated_at；IncidentState 同样保留创建时间并更新最近时间。

如果 IncidentState 保存失败，AlertEvent 写入也回滚，API 不会出现“告警返回成功，但 IncidentState 没有持久化”的半成功。SQLite 回归测试通过注入 State 写失败验证 AlertEvent 不落库；MySQL 使用显式 transaction、row lock 与 rollback 实现同一协议。

### 仍然没有得到的保证

数据库事务只覆盖一条 alert 的两份 latest projection，不覆盖后台诊断任务。多副本即使对 created/new-generation 得到一致数据库裁决，事务提交后到 `BackgroundTasks` 调度之间仍没有 durable outbox；进程可能在返回前后崩溃，任务也可能因多个副本、重试或路由方式产生重复或遗漏。进程内 set 与 Semaphore 仍不是集群 exactly-once。

IncidentState 的其他写入者也不是全部通过同一个全局 version/CAS 协议。merge guard 能拒绝已知的终态、审批和 change 回退，但无法替代跨所有服务的乐观锁、事件版本和完整状态迁移审计。

### 批量 webhook 先全量校验，再逐条原子提交

服务先把整个 `alerts` 数组全部做结构、生命周期和 Pydantic 边界校验，再开始任何持久化。第二条时间非法、缺 alertname、未知状态或非对象时，整批在写入前失败，因此不会再出现“前一条因后一条校验失败而已提交”。

校验通过后，每条 alert 仍是独立数据库事务，不是整个 webhook 的跨条目事务。若第一条已经提交，第二条发生数据库故障，仍可能得到“请求失败但前项已写”的存储级部分成功。生产设计仍需在整批事务、逐条 ack、幂等重试与死信之间明确选择；当前修复消除的是校验导致的部分提交，以及单条 AlertEvent/IncidentState 半提交。

## 自动诊断为什么默认关闭

### 摄入不是模型调用授权

`POST /api/alerts/alertmanager` 的 `auto_diagnose` query 默认 `false`。接收、标准化和保存告警不应自动等价为调用模型、Prometheus、日志网关、Redis、MySQL 或 Kubernetes。组织可能还需要告警抑制、维护窗口、路由与人工确认；默认关闭让可靠摄入与昂贵诊断解耦。

显式开启后，只对第一次 created firing 和“更大 `startsAt` 的新一代 firing”调度后台任务。重复同代 firing、stale firing、stale resolved 与所有已接受的 resolved 都不启动诊断。若首次收到的就是 resolved，记录可以 created，但不会因其 created 而诊断，因为 resolved 条件优先跳过。

### in-flight set 与 Semaphore(2) 解决不同问题

受线程锁保护的 Incident ID `set` 判断本进程是否已有同一 Incident 在途，减少本进程重复任务；`asyncio.Semaphore(2)` 把单进程同时执行的诊断限制为 `2`，提供简单背压。guard 在 finally 中清理标记，异常也不会让 Incident 永久占用 set。

它们都不是持久任务系统：多 worker 和多副本各有独立 set、Lock、Semaphore；服务崩溃会丢失排队/在途任务与标记；FastAPI BackgroundTasks 没有 ack、attempt、租约、心跳、延迟重试或死信；Semaphore 也不代表集群总并发为 2。

准确表述只能是“BackgroundTasks + 进程内去重 + 单进程并发上限 2”，不能说 distributed exactly-once 或可靠队列。

### 每一轮诊断拥有新的 session

后台任务从 AlertEvent 重建 Incident，每次生成 `alert-<incident_id>-<uuid4 hex>`。同一 Incident 首次 firing 与 reopened 的 session 不同；测试也验证连续两次直接执行得到两个唯一 session。

这让 Trace 和 Snapshot 区分调查轮次，Incident ID 则负责归并业务故障。若 session ID 直接复用稳定 Incident ID，重开可能覆盖上一轮最新 Snapshot，难以说明两次调查分别发生了什么。

### 失败可见，但不泄露内部异常

后台诊断异常不会回滚已经成功的 webhook 摄入。系统写 `workflow_error` Trace，使用已有 trace_id 或 `trace-<incident_id>`，记录 session、fingerprint 和 source；错误先经 public error 映射，再进入 IncidentState `failed` 与 metadata。

自动测试让内部异常包含 `planner unavailable`，并验证公开 state 不包含原文本，而显示“诊断服务暂时不可用”一类安全摘要。日志仍记录内部异常用于运维。这里失败的是异步诊断，不是告警摄入；不能把两者合并计算成一个成功率。

## API 与读模型决定值班人员最终看见什么

### 写入口和读入口权限不同

webhook 路由要求 `diagnose` scope；Alert 列表、Alert 详情、Incident 列表与详情要求 `read`。当前是静态 API token/scope，不是 Alertmanager mTLS、签名 webhook 或 OIDC workload identity。生产仍需来源验证、防重放、网络策略、TLS 和限流。

payload 没有 alerts，且顶层也没有 labels 时，service 返回 `received=0`，API 转成 HTTP 400。顶层带 labels 则兼容成一条告警。`alerts` 不是数组、数组中存在非对象、labels/annotations 不是对象、缺 alertname、状态未知、时间非法或模型边界不满足时返回 HTTP 422；这些字段错误不会被静默跳过，也不会创建 Incident。

### Alert 查询只能回答最新状态

`GET /api/alerts` 支持 status、service_name、limit。status 仅接受 `firing/resolved/active/triggered/inactive/ok/closed`，未知值在 API 参数校验阶段返回 422；limit 默认 `50`，范围 `1..200`，Store 也收敛到该范围。详情按 fingerprint 查询，不存在返回 404。

列表适合回答“最近哪些告警目前是什么状态”，不适合统计每次到达、抖动次数和乱序。若产品需要这些能力，应新增 append-only receipt/event，而不能从 latest AlertEvent 反推。

### Incident 列表是多事实源的并集

`GET /api/incidents` 读取 Report、Trace、Approval 和 IncidentState，以 incident_id 求并集，为每个 ID 构造 overview，按综合 updated_at 倒序；limit 默认 `20`，范围 `1..100`。因此即便 State 缺失，只要报告、Trace 或审批存在，该 Incident 仍可能出现在列表。

Overview 用 durable IncidentState 提供 lifecycle、status reason、session 等当前信息，同时聚合 Report、Trace 与 Approval。若 State 已是 `change_dry_run` 而旧 Report 仍为 `approval_approved`，读模型优先 durable lifecycle State。所有事实源都不存在才返回 404。

这是一种查询容错，不是跨 repository 强一致快照。几类事实可能在略有不同的时刻读取，读者应把 overview 理解为查询时聚合，而不是同一数据库事务时刻。

## 用五条不变量检查设计，而不是只检查接口返回 200

### 不变量一：同一业务身份必须导向同一 Incident

只要规范化后的 fingerprint 相同，Incident ID 就必须稳定；duplicate、resolved 和 reopened 不能因为到达时间不同而随机换 ID。这个不变量让报告、审批和变更有共同归属。它不反向保证 fingerprint 业务定义正确，身份冲突仍需治理。

### 不变量二：去重只能抑制副作用，不能抹掉新信息

重复 firing 不应重复启动诊断，但新的 labels、annotations、状态和更新时间仍要进入 latest AlertEvent。resolved 与 reopened 虽是已有 fingerprint，也必须写入。若实现把 deduplicated 简单处理为 return，恢复与重开会永远不可见。

### 不变量三：浅层信号不能覆盖深层业务状态

告警是监控系统对条件的判断，审批和 change 是处置流程的业务事实。resolved 可以更新 alert metadata，却不能自然取消 waiting_approval；普通旧报告也不能把 change lifecycle 拉回 completed。写入所有权必须比状态字符串的先后更重要。

### 不变量四：同步摄入必须原子，异步失败必须分栏

同步路径只有在 AlertEvent 与 IncidentState 同时提交后才返回成功；其中任一保存失败都回滚该条摄入。提交之后，后台模型失败是另一项事实：正确结果是保留 AlertEvent，新增失败 Trace 与 IncidentState，而不是回滚已经确认的告警或把 HTTP 响应伪装成诊断成功。安全错误摘要和内部日志也要分层，防止异常文本泄漏。

### 不变量五：证据声明不能超过实际运行环境

历史 SQLite fixture 中 `24/24` 测试通过，只说明 2026-07-13 固定输入的旧契约；当前工作区新增了严格字段、UTC 时间、stale/new-generation、事务回滚和迁移守卫回归。进程内 set 仍只说明本进程去重；MySQL transaction/row-lock 代码存在与自动化替身测试也不等于真实 HA。任何“高可用”“exactly-once”“生产吞吐”都必须由对应 `local_live`、`controlled_fault` 或 `production` Artifact 支持。

这五条不变量比具体类名更稳定。未来即使把 FastAPI BackgroundTasks 换成 Kafka，把 SQLite 换成 PostgreSQL，仍可用它们检查新架构是否保存了业务语义。

## 如果从零设计接口，先定义响应语义再写数据库

一个摄入响应至少要区分收到多少、首次创建多少、已有身份多少、resolved 多少、stale ignored 多少，并为每条返回 previous status、是否变化、是否 reopened、是否 stale、当前 Incident status 与 reason。否则调用方只能从 HTTP 200 猜测是否发生了副作用。

还应把“摄入确认”和“诊断调度确认”分开。当前响应描述摄入结果，BackgroundTasks 是否最终开始、重试或失败需要后续查询 IncidentState/Trace。生产队列可进一步返回 task ID、accepted/duplicate/already-running 与 durable enqueue 状态；但只有任务真正持久化后，才能承诺进程重启后继续。

对批量 payload，应预先选择原子语义。当前实现先整批校验，字段错误时零写入；校验通过后按 alert 逐条事务持久化。若要改成整批事务，任一存储失败必须全部回滚并返回可重试错误；若保留逐条处理，响应应列出每条 success/failure 与幂等键，失败项进入死信或允许单条重试。当前缺少的是存储阶段 partial response，不是校验阶段的零写入保证。

## 把 Redis 四条消息完整走一遍：每一步都问还缺什么

### 02:07 首次 firing：创建调查对象，不直接接受根因

公共 environment 与单条 labels 合并，`critical` 映射 P1，service 识别为 `order-service`。非空上游 fingerprint `redis-order-prod-001` 被采用，系统由它派生稳定 Incident ID。`startsAt` 校验并归一到 UTC 后，AlertEvent 与 IncidentState 在同一事务中首次写入，State 目标为 `alert_firing`，返回 `received=1`、`created=1`、`deduplicated=0`、`status_changed=false`、`reopened=false`、`stale_ignored=false`。

此时已经知道：监控系统声明生产 order-service 的 Redis 连接压力满足触发条件，告警开始时间可用于构造 Incident start_time。仍不知道：P95 与连接数是否同窗口变化，应用连接池是否泄漏，Redis 是否真的达到 maxclients，最近部署是否相关，日志超时是否来自同一依赖。

下一步为什么是诊断而不是 `CONFIG SET`？因为 alert annotation 是外部主张，Runbook 是一般知识，“连接上限导致 503”仍是假设。只有显式 `auto_diagnose=true`，系统才调度受控诊断；默认仅完成摄入和展示。

### 02:08 重复 firing：更新信息，不重复副作用

服务读取同 fingerprint 旧快照，保留 created_at、刷新 updated_at，保存本次最新 labels/annotations。响应为 `created=0`、`deduplicated=1`、`previous_status=firing`、`status_changed=false`、`reopened=false`。

去重没有丢弃这条消息：如果 annotation 增加了 runbook URL 或标签变化，latest AlertEvent 仍会更新。被抑制的是“因为重复 firing 再启动一次诊断”。若上一轮 diagnosis 是否仍在运行，需要查询 session、Trace 与 IncidentState，不能仅凭 AlertEvent 推断。

并发边界也在这里出现：Store 事务会把同一 fingerprint 的 latest 裁决与两表写入串行化，避免两个响应都基于“记录不存在”的旧读结果。但后台任务调度不在数据库事务与 durable outbox 中，多副本、进程崩溃和重试仍可能造成任务重复或遗漏。

### 02:15 resolved：告警恢复与流程完成必须分栏展示

归一后的 AlertEvent 变为 resolved，`previous_status=firing`、`status_changed=true`、`reopened=false`，本批 `resolved=1`。由于 fingerprint 已经存在，item 仍然 `deduplicated=true`；这再次说明 deduplicated 不是“相同状态”。

若 IncidentState 仍为 `alert_firing`、`investigating` 等浅层状态，builder 的目标是 `resolved`。若系统已进入 `waiting_approval`，它保留等待审批的 status/reason，只把 metadata 中 `alert_status` 更新为 resolved，并记录 `preserved_incident_status=waiting_approval`。

当前事实只是上游告警条件暂时消失。可能原因包括流量自然回落、连接被动释放、监控缺数或短暂网络恢复。没有 Evidence 证明连接泄漏已修复时，不能自动关闭审批或变更。合理动作是继续观察窗口、核验指标，或由值班人决定取消后续动作。

如果此前后台诊断仅因模型服务不可用而 failed，这条 resolved 可以把状态恢复为 resolved，但 `alert_auto_diagnosis_status=failed` 等 metadata 保留。页面应同时显示“告警恢复”和“诊断曾失败”，避免成功假象。

### 02:17 reopened：复用 Incident，创建新的诊断轮次

本次 firing 的 `startsAt` 大于上一代，因此 `reopened=true`，Incident ID 不变；若上一 latest 为 resolved，`status_changed=true`，若上一 latest 仍为 firing，状态字符串虽未变化也仍属于新一代 reopened。浅层 State 回到 `alert_firing`；已完成、resolved 等终态只允许通过这个显式代际条件重开，审批/change 回退仍受保护。显式 auto diagnose 可启动新 session，session ID 加新的 uuid，不覆盖上一轮执行身份。

此时已经知道同一 fingerprint 再次触发，故障具有抖动或复发特征。新一轮诊断应重新采集事故窗口 Evidence，并比较两轮结果；上一轮的指标只能作为历史材料，不能当成当前实时事实。如果本进程已有同 Incident in-flight，set 会抑制新任务；这可能节省重复工作，也可能错过“上一轮卡住而新一轮更重要”的场景，生产队列需要显式 coalescing/retry policy。

### 如果消息乱序，当前系统会怎样

设 02:17 新一代 firing 的 `startsAt=t2` 先处理，02:15 上一代 resolved 的 `startsAt=t1` 因网络延迟后处理。Store 看到 `t1 < t2`，返回 `stale_ignored=true`，保留 latest firing 与 IncidentState；旧 resolved 不计入本批 `resolved`，也不会触发状态变化。

同一代 resolved 后又到相同 `startsAt` 的 firing 也会被忽略，避免网络重试隐式重开；新 firing 只有 `startsAt` 更大才会 reopened。这个规则已经解决课程主线中的旧代覆盖问题，但正确的生产升级仍不能只依赖时间：应保存 source sequence、received_at 和相同版本冲突规则；旧消息可以进入 append-only history，却不能覆盖 latest projection。

## MySQL 与 Kubernetes 对照：模型可复用，身份字段不能照抄

MySQL 连接池告警可以复用 AlertEvent、Incident、IncidentState 三层模型，但 fingerprint 可能需要 cluster 或 instance。若只使用 service，主库与只读实例的故障可能误合并；若问题是应用侧连接池，加入瞬态 connection ID 又会误拆分。

Kubernetes CrashLoop 更能说明业务域。把 Pod UID 放入服务级 Incident 身份，每次 Pod 重建都会创建新故障，无法看到一次发布导致的连续崩溃；完全忽略 namespace/cluster，又可能合并两个环境。若调查目标确实是单 Pod，instance 反而应进入身份。

因此“字段越多越准确”并不成立。当前统一 fallback 是已实现事实；按 alert 类型配置 identity policy、冲突观测和迁移策略尚未实现。对照案例不能被写成当前已验证了 MySQL/Kubernetes 最佳告警聚合。

## 失败路径决定这套方案的生产边界

### 哪些情况必须立即停止

- payload 既无有效 `alerts` 列表，也不是带 labels 的单条告警：返回 `received=0`，API 转成 HTTP 400，不创建空 Incident。
- alerts 数组形状、必填 alertname、状态、时区或起止时间非法：整批在持久化前返回 422。
- 单条 AlertEvent 或 IncidentState 存储失败：该条事务整体回滚，不返回摄入成功；若此前其他条目已完成独立事务，仍需按存储级部分成功调查。
- fingerprint 详情查询不存在：返回 404，不根据相似 alertname 猜测对象。
- Alert 列表状态 filter 不在有限集合：返回 422，不把拼写错误默认为 firing。

### 哪些情况应降级为只保存、只展示

- `auto_diagnose=false`：完成摄入与投影，不调用模型和外部工具。
- service/environment 为 unknown：仍可保存，但应暴露标签质量问题，不能让 Planner臆造目标。
- 相同 Incident 本进程已有在途诊断：更新 AlertEvent，跳过重复后台任务。
- 深层 lifecycle 已存在：仅更新 alert metadata，保留审批或变更状态。
- 后台诊断失败：摄入不回滚，写安全 Trace/failed State，交给重试策略或人工。

### 哪些情况必须承认当前无法自动裁决

- **后台副作用竞态**：数据库 latest 裁决已事务化，但任务调度没有 durable outbox、跨副本唯一任务或 fencing。
- **通用乱序**：当前能按 `startsAt/endsAt` 处理 Alertmanager 代际旧消息，但没有 source sequence、received_at 与跨来源版本协议。
- **存储级批量部分成功**：整批先校验避免校验型部分提交，但多条 alert 仍逐条事务；后项数据库故障时没有逐条 ack/partial response。
- **身份冲突**：fallback 规则无法证明两个相同 fingerprint 一定属于同一业务故障。
- **进程崩溃**：BackgroundTasks 与 in-flight set 不耐久，不能自动恢复排队任务。
- **存储切换**：SQLite/MySQL 遵守同一 Store Protocol，但没有真实 MySQL HA 与故障接管证据。

## 从单机 latest projection 走向生产需要增加哪些约束

合理演进不是一次性替换所有模型，而是逐层补上当前缺失的语义。

第一层是 append-only AlertReceipt：每个接收请求保存 source、source message ID、received_at、source event time、payload hash、解析状态与安全原文指针。AlertEvent 变成由 receipt 投影出的 latest 状态。这样可以审计抖动、重算 projection，也能解释某次 HTTP 失败前哪些条目已经落盘。

第二层是版本化投影：IncidentState 增加 version，更新使用 expected version；冲突时重读并按状态所有权重新 merge。上游若提供 sequence，应与 source time 一同保存，明确不同 version、相同 version 和时钟倒退的处理规则。

第三层是可靠任务：消息队列按 Incident identity 分区，任务表使用唯一业务键；worker 持有 owner、attempt、lease expiry、heartbeat 与 fencing token；结果写入也带 attempt/fencing，防止旧 owner 在租约过期后覆盖新 owner。重试、退避、死信和人工重放都应可查询。

第四层是治理与观测：按 alertname 配置 identity policy，统计 fingerprint 冲突、reopen 频率、unknown 标签率、摄入延迟、队列积压、诊断启动率与失败率。只有保留分子、分母、时间窗口和环境，这些数才可以成为生产 SLO 证据。

当前方案选择 latest AlertEvent + latest IncidentState，是读取简单、易解释和易测试的折中，不是唯一正确答案。它适合单机演示和课程中的最小闭环；当需求进入多副本、高吞吐、严格审计与可重放时，上述约束才是能力升级，而不是把现有 upsert 换个“事件驱动”名称。

| 方案 | 得到什么 | 代价或缺口 | 当前状态 |
|---|---|---|---|
| 随机 Incident ID | 实现简单 | duplicate/reopen 无归属 | 不采用 |
| 全 labels fingerprint | 信息多 | 动态标签误拆分 | 不采用 |
| 有限稳定字段 fingerprint | 可解释、可复现 | 仍有冲突与业务域风险 | 已实现 |
| latest AlertEvent + State + 单条原子摄入 | 查询简单，两表不半提交 | 无完整历史，后台副作用不耐久 | 已实现 |
| append-only receipt + projection | 可审计、可重算 | 事件版本与存储复杂 | 尚未实现 |
| 进程内 set + Semaphore(2) | 低成本去重和背压 | 不跨进程、不耐久 | 已实现 |
| 队列 + 租约 + CAS | 多副本与接管语义 | 基础设施和运维成本高 | 尚未实现 |

## 生产观测不能只看 webhook HTTP 成功率

HTTP 200 说明本次返回的每条 alert 已完成 AlertEvent + IncidentState 同步持久化，不说明每条都是首次创建，更不说明后台诊断已经完成。一个可运营的告警入口至少应分别观测：有效 alert 数与 400/422 无效 payload 数、created/deduplicated/resolved/reopened/stale_ignored 的分子分母、unknown service/environment/severity 的比例、fingerprint 长度哈希次数、脱敏命中与 payload 拒绝、摄入延迟、事务回滚、后台任务 accepted/already-running/failed、队列等待和诊断完成。

这些指标还必须按 source、alertname、service、environment 和版本分组，并保留时间窗口。reopened 比例突然升高，可能是故障复发或上游反复生成新 `startsAt`；stale_ignored 升高，可能是网络乱序或时钟语义异常；deduplicated 比例突然下降，可能是上游 fingerprint 漂移；unknown service 升高，可能是标签变更。指标只指出异常方向，仍需 receipt 样本与 Trace 解释原因。

告警摄入 SLO 也应拆成多个阶段：请求接收、持久化确认、任务耐久入队、首次诊断进度、最终报告。当前实现只具备请求内同步持久化和非耐久 BackgroundTasks，因此不能定义“HTTP 成功即诊断已接管”的 SLO。只有引入 durable queue、任务状态和带 provenance 的压测/故障实验后，才能给出 P50/P95/P99、丢失率、重复率与恢复时间；本章没有这些历史数字，不能自行编造。

## 面试表达：把“去重”讲到状态语义

### 一句话定位

> AutoOnCall 用 fingerprint 把不稳定告警归并为稳定 Incident，并用最新快照与生命周期投影分别服务摄入、诊断和查询。

### 30 秒回答

> 我把外部告警拆为 AlertEvent、Incident 和 IncidentState：前者是按 fingerprint upsert 的标准化最新快照，中间是 Agent 的稳定输入，后者是工作台生命周期投影。webhook 先严格校验状态与带时区时间，再按 `startsAt` 区分同代重复、旧消息和新一代 reopen；Store 在单条事务中同时持久化 AlertEvent 与 IncidentState，迁移守卫阻止终态、审批和变更状态被晚到更新拉回。自动诊断仍只是进程内去重和 `Semaphore(2)`，不是 exactly-once。

### 3 分钟展开

> Redis 告警在十分钟内经历 firing、duplicate、resolved、reopened。每条随机 ID 会重复诊断，只用 alertname 又会误合并不同服务，所以系统优先上游 fingerprint，空白时用 source、alertname、service、environment 和有限 namespace/instance 等稳定字段做 SHA-256；超长 fingerprint 哈希为 64 字符，Incident ID 再取哈希前 12 位。外部字段先严格校验、UTC 归一、脱敏和限长，未知状态、缺 alertname、无时区时间与逆序起止时间返回 422。AlertEvent 保存监控 latest，Incident 是 Agent 输入，IncidentState 是当前业务投影；`startsAt` 更小或 resolved 后同代 firing 会 `stale_ignored`，只有更大 `startsAt` 的 firing 才 reopened。SQLite `BEGIN IMMEDIATE` 和 MySQL row lock 把 latest 裁决、AlertEvent 与 IncidentState 写入放进单条事务，State 写失败时 Alert 也回滚；merge guard 拒绝终态、审批和 change lifecycle 的隐式回退。自动诊断只对 created/reopened firing 启动，每轮 session 唯一，进程内并发上限 2。生产化仍缺 durable outbox/queue、跨副本任务租约、source sequence、append-only receipt 和完整事件溯源 FSM。

### 可写入简历的准确表述

> 设计 AlertEvent、Incident 与 IncidentState 分层模型，通过严格 webhook 校验、UTC 时间归一、fingerprint 幂等身份、单条事务化双投影写入、stale/new-generation 裁决和终态迁移守卫，将重复、恢复、乱序与重开告警转为一致的可诊断故障输入；为可选自动诊断保留单进程 in-flight 去重、并发上限与失败可见性。

### 不可夸大的表述

- 不写“基于事件溯源保存完整告警历史”，当前是 latest upsert。
- 不写“严格 FSM 保证所有迁移合法”，当前是 latest projection 上的终态/审批/change 回退守卫和显式 reopen 例外。
- 不写“分布式 exactly-once”，当前没有跨副本任务互斥。
- 不写“解决所有乱序”，当前只按 Alertmanager `startsAt/endsAt` 做有限代际裁决，没有 source sequence 与跨来源版本协议。
- 不写“真实 Alertmanager 大流量与 MySQL HA 已验证”。
- 不把 `24 passed` 写成生产告警成功率。

### 高频追问

1. **为何 Incident 不复用 AlertEvent？** Incident 是内部诊断契约并支持手工构造；AlertEvent 保留外部来源和告警生命周期。
2. **fingerprint 冲突怎么办？** 优先上游值，缺失时只选有限稳定字段；仍需标签治理和冲突观测。
3. **去重为何仍要 upsert？** resolved/reopened 和新 metadata 必须更新，真正被抑制的是重复诊断，不是信息更新。
4. **为什么不是严格 FSM？** 当前集中拒绝已知的终态、审批和 change 回退，也约束显式 reopen；但没有完整合法边表、transition event 和全部历史。
5. **多副本如何改造？** 建唯一任务与租约或队列消费，明确 owner、lease expiry、heartbeat、attempt 和幂等结果写入。
6. **怎样处理乱序？** 当前按 `startsAt` 区分代际、按同代状态与 `endsAt` 拒绝旧消息；生产仍应补 source sequence、received_at 和相同版本冲突规则。
7. **resolved 是否代表 Incident 已完成？** 不一定；它只是监控状态恢复，诊断、审批或变更可能仍在进行。
8. **事务化 upsert 为什么仍不是 exactly-once？** 它保证单条 AlertEvent 与 IncidentState 一致，并串行化 latest 裁决；消息投递、HTTP 重试和后台诊断副作用仍没有 durable outbox 与跨副本唯一执行。
9. **Semaphore(2) 能保证什么？** 只限制单进程同时运行两个诊断，不控制集群总并发，也不提供崩溃恢复。
10. **为什么 reopened 复用 Incident 却新建 session？** Incident 表示故障域，session 表示一次尝试；分开才能比较轮次又保留归属。
11. **批量 webhook 中途失败怎么办？** 全批先校验，字段错误时零写入；持久化仍逐条事务，后项数据库故障可能保留前项，生产需要整批事务或逐条 ack/死信。
12. **为什么 failure State 能被后续 alert 覆盖？** 只对带 auto-diagnosis failure metadata 的 failed 开例外，避免临时模型故障永久锁死；失败历史仍在 metadata/Trace。

本章最终建立的并不是“告警一来，Agent 就自动修复”，而是一个更克制也更重要的起点：先把不稳定、重复、可能乱序且可能含敏感信息的外部消息，转换成身份稳定、字段受控、状态不越权的调查对象。只有 Incident 的业务归属可信，后续检索、工具取证、Hypothesis 排序、审批和变更才知道自己处理的是哪一次故障；只有同时保留 AlertEvent 与 IncidentState，值班人员才能区分“监控刚刚怎么说”和“处置流程实际上走到哪里”。

这也是面试中判断设计是否真实的标准：不仅讲正常路径怎样创建记录，还能准确说明重复、乱序、并发、失败、权限与证据等级分别在哪里停止，哪些能力必须留给后续生产化工程。

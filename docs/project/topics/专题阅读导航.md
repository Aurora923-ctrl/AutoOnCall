## 一、这套文档应该怎样读

这不是一组互不相关的代码说明，而是一门围绕同一个 AutoOnCall 项目展开的课程。

全系列从凌晨 Redis 故障开始，逐步加入 Incident、知识入库、可信检索、诊断循环、工具治理、Evidence、报告、审批、变更边界、运行态和评测。每一篇只新增一个主要能力，并回答三个问题：

1. 上一阶段为什么还不够？
2. 当前能力的最小机制和真实实现是什么？
3. 面试时怎样准确表达亮点，同时主动说明生产边界？

第一次阅读建议严格按 00–13 顺序；14 是北向 Agent 协作选修，15 将前面所有能力收束成量化验收和校招表达。若目标是准备面试，15 应作为必读终章。

### 1.1 这套课程不会怎样写

为了避免重新退化成“按目录介绍类和函数”，正文遵循三条约束：

1. **先讲问题，再讲对象。** 读者会先看到人工值班遇到了什么困难、朴素方案为什么失败，然后才看到 `Incident`、`Evidence`、`Snapshot` 等项目对象。
2. **先讲通用原理，再落到 AutoOnCall。** 每个概念都要能迁移到别的 RAG、Agent 或 AIOps 项目，而不是只能背出当前仓库的类名。
3. **代码负责证明，不负责主导叙事。** 正文只保留能解释机制的伪代码、公式和关键数据结构；完整文件路径、测试入口和能力边界统一放在附录。

因此，第一次阅读时不需要同时打开 IDE。先用故事、反例和心智模型理解本章，再在“真实代码调用链”和“实现证据附录”中完成第二遍源码追踪。

### 1.2 一篇专题的推荐学习方法

每篇专题都可以分三遍学习：

- **第一遍：只回答为什么。** 重点读架构定位、问题现场、朴素方案和失败原因，能够说出“没有这一层，系统会在哪里出错”。
- **第二遍：回答怎么做。** 重点读核心概念、逐层约束、贯穿案例和真实调用链，自己画出输入、状态变化和输出。
- **第三遍：回答凭什么。** 重点读负例、量化验收、证据附录和第 15 篇总表，区分实现、测试、本地运行和生产证据。

如果第三遍只能背出一个百分比，却说不出分子、分母、数据集、运行环境和证据等级，就还没有真正掌握这项指标。

### 1.3 为什么指标既分散在各章，又集中保留在第 15 篇

各专题中的指标承担“解释设计”的职责。例如 Recall@3 应该在混合检索章节出现，因为读者需要立刻理解 Top-K、召回和排序之间的关系；Tool Selection F1 应该在 Agent 评测章节出现，因为它检验的是 Planner 是否选对动作。

[第 15 篇](15-项目量化指标与校招验收.md)则承担“完整汇总和面试取证”的职责，集中保留全部样本、指标、运行身份、证据等级、简历表述与禁用表述。两者不是重复：前者帮助学习，后者帮助复核。重构过程中，第 15 篇保持原文不动，各章已有量化数据也不会删除或改写成模糊结论。

> **目录版本口径**：目录计划第 2 节的早期汇总仍写“00–13 核心 + 14 选修”，但同一计划后部已经增加专题 15 的实施卡片和批次任务，当前 `topics/` 也确实存在 00–15 全部正文。本导航按“实际文件 + 后部实施卡片”登记 15；这不是把 15 悄悄算入原 13 篇主线，而是将它明确列为量化验收终章。后续若修订目录计划，应同步更新前部数量汇总。

## 三、全系列同时推进的三条叙事主线

### 1. 故障故事线：同一个 Redis Incident 逐步获得处理能力

```text
收到 Redis 告警
→ 识别重复、恢复和重新触发
→ 查找官方文档、内部 Runbook 与历史事故
→ 组合向量和关键词召回
→ 生成带引用或明确拒答的知识回答
→ 形成诊断计划并调用实时工具
→ 将工具结果变成 Evidence 与根因假设
→ 生成可回链报告
→ 风险动作进入审批和安全变更状态
→ 用 Trace、Snapshot 和读模型重建过程
→ 用分层评测证明改动没有破坏闭环
```

MySQL 慢查询/连接耗尽用来对照不同信号和发布关联；K8s CrashLoop/OOM 用来展示适配器、离线案例和证据等级边界。它们不是三套独立系统，而是检验同一套结构能否在限定故障域中复用。

### 2. 架构生长线：每篇只增加一块缺失能力

```text
事实口径
→ 业务闭环
→ Incident 身份
→ 知识入库
→ 可信检索
→ 可审计回答
→ Plan-Execute-Replan
→ 受控工具
→ Evidence/Hypothesis
→ DiagnosisReport
→ Policy/Approval
→ Safe Change 状态
→ 运行态重建
→ 分层评测
→ A2A-compatible facade（选修）
→ 量化 Scorecard 与校招验收
```

读者应始终能回答：“如果去掉当前这一层，上一阶段具体会在哪里失败？”

### 3. 面试表达线：从概念到可追问证据

每篇都固定训练以下能力：

- 30 秒解释本章解决的问题。
- 3 分钟讲清朴素版本、失败方式和当前设计。
- 追踪一条真实代码调用链。
- 解释一个设计取舍。
- 描述一个失败或降级案例。
- 说出一项尚未实现或尚无生产证据的能力。
- 把亮点缩成一条可以被继续追问的简历表述。

当你能把 01、06、09、10、12、13、15 的 30 秒版本自然串起来，就已经具备完整项目介绍、量化追问和证据边界的骨架。

## 四、按面试岗位选择速读路线

### RAG / 知识库工程

推荐顺序：

```text
00 → 01 → 03 → 04 → 05 → 13 → 15
```

重点准备：多格式来源定位、结构化切分、Embedding 维度/归一化、版本化 chunk、Milvus Schema/稳定 ID、双索引补偿、metadata filter fail-closed、weighted/RRF、Trust Gate 先于最终 Top-K、双向降级、预算化 frozen evidence、严格 source/chunk pair 引用、模型 timeout/retry/cancel、单 session 并发隔离、问题日志最小化、验证后流式发送，以及无数据泄漏、完整分母、RAGAS 版本兼容和 provenance-aware Scorecard。

不要夸大：没有 OCR、在线 Wiki connector、事务双索引、Cross-Encoder、通用 OOD、逐 claim 语义事实核验或持久化 grounded memory。

### Agent / LLM 应用工程

推荐顺序：

```text
00 → 01 → 05 → 06 → 07 → 08 → 09 → 12 → 13 → 15
```

重点准备：一次初始 Planner、Executor/Replanner 循环、结构化 Plan、静态工具治理、两层重试、Evidence 驱动 Replan、报告回链、运行态重建和 Agent 分层评测。

不要夸大：不是开放式无限 ReAct，不是每轮重新 Planner，不是任意 MCP 插件市场，也不是通用自主 RCA。

### AIOps / SRE / 可观测性

推荐顺序：

```text
00 → 01 → 02 → 06 → 07 → 08 → 09 → 10 → 11 → 12 → 13 → 15
```

重点准备：告警身份与状态投影、跨源取证、根因假设、充分性门禁、审批、变更状态、Trace、Snapshot、Replay 和 controlled fault。

不要夸大：告警不是 exactly-once，生命周期不是事件溯源 FSM，safe change 不执行真实生产写入，Replay 不重新执行工具。

### Python 后端 / 平台工程

推荐顺序：

```text
00 → 01 → 02 → 07 → 10 → 11 → 12 → 13 → 15
```

重点准备：FastAPI 契约、静态 token/scope、SQLite/MySQL Store 协议、幂等更新、结构化错误、Artifact、SSE、读模型、评测 provenance 和部署边界。

不要夸大：当前没有企业 SSO/OIDC、多租户、分布式 Artifact Store、数据库 LangGraph checkpointer 或完整 HA/DR 证据。

### 安全治理 / Agent 平台

推荐顺序：

```text
00 → 06 → 07 → 10 → 11 → 12 → 13 → 15
```

重点准备：确定性安全优先、工具元数据、Executor/Registry 双重门禁、静态 RBAC、审批幂等边界、三种变更模式和安全评测。

不要夸大：规则门禁不是通用 Prompt Injection 防御，审批不是生产执行许可，fixture sandbox 不是真实基础设施沙箱。

### Agent 协议 / 互操作

推荐顺序：

```text
00 → 01 → 05 → 06 → 10 → 12 → 14 → 15
```

重点准备：业务 Skill、Task、Artifact、SSE 映射，以及 facade 对既有领域服务和读模型的复用。

不要夸大：当前是 A2A-compatible facade，不是完整 conformance、官方 SDK 或跨实现互操作认证。

## 五、按可用时间选择速读路线

### 约 30 分钟：先能准确介绍项目

阅读：

```text
00 → 01 → 15 的核心指标总览与禁用表述
```

目标：讲清项目为什么存在、总体闭环是什么，以及为什么本地测试不能包装成生产结果。

### 约 60 分钟：准备一轮项目概览面试

阅读：

```text
00 → 01 → 06 → 07 → 08 → 09 → 10 → 12 → 13 → 15
```

优先阅读每篇的架构定位、完整案例、设计取舍、30 秒版本和能力边界附录。目标是串起 Agent 主链与治理边界。

### 约 2 小时：按目标岗位形成可追问答案

先完成 60 分钟路线，再加入对应岗位路线中的专题，并实际完成每篇至少一个只读追踪题。目标是能够从口头概念落到真实文件、模型和测试。

### 完整学习：准备系统设计和深挖面试

按 00–13 顺序阅读全文，需要协议互操作时读 14，最后用 15 完成指标手算和 Scorecard 验收。完整学习的标准不是“看完”，而是能够：

1. 不看文档画出主链。
2. 为 Redis 正常链和至少一个失败链逐步解释状态变化。
3. 指出每个亮点的实现、自动化测试和缺失生产证据。
4. 面对“为什么不用更自动的方案”给出设计取舍。

## 八、生产边界提醒：读完整套文档也不能越过的红线

### 1. 告警和运行状态

- AlertEvent 是按 fingerprint upsert 的最新快照，不是 append-only 事件库。
- 自动诊断去重和并发限制是进程内机制，不是分布式 exactly-once。
- IncidentState 是最新生命周期投影和优先级合并，不是严格事件溯源 FSM。

### 2. RAG

- PDF loader 不提供 OCR，HTML loader 不连接在线 Wiki。
- Milvus 与本地 lexical 双写不是事务；当前用发布前 stale、flush、已知 vector ID 补偿和最终清门禁尽力 fail closed。
- 当前 rerank 是规则加权或 RRF，不是 Cross-Encoder/LLM reranker。
- metadata filter 非法或后端不支持 `expr` 时直接失败，不会撤销过滤继续全库召回。
- 向量或词法单路失败可以显式降级，但不是多级熔断、生产容量或可用性证明。
- Trust Gate 不是通用 OOD 分类器。
- budgeted evidence、全量 citation 完整性与精确引用身份检查都不是逐 claim 语义蕴含。
- 聊天流会先 spool、重试并完成引用门禁再发送，因此不是低 TTFT 的逐 token 直播；session lock 也只在单进程内有效。

### 3. Agent 与 RCA

- 图是一次初始 Planner，之后在 Executor 与 Replanner 之间循环，不是每轮回到 Planner。
- Tool Registry 是静态标准工具集合，不是任意 MCP 插件市场。
- Tool schema 主要是描述性契约，输出仍可能是 `Any`，不是全局 JSON Schema 强校验。
- Evidence Analyzer 是规则、关键词和静态信号目录驱动的基线，不是通用因果推理。
- 启发式置信度不是根因真实概率。

### 4. 报告、安全与变更

- Evidence sufficiency 是覆盖门，不是数学充分性证明。
- Conclusion Alignment 是 evidence/citation 回链，不是事实蕴含证明。
- Evidence Graph 是报告内 portable 读模型，不是图数据库推理。
- 静态 token/scope 不是 SSO/OIDC 或完整企业身份平台。
- 审批通过不表示 Agent 已执行生产变更。
- dry-run 是结构和策略检查，sandbox 使用本地 fixture；当前没有生产写适配器和自动 rollback。

### 5. 恢复、回放、评测与互操作

- LangGraph 使用内存 MemorySaver；latest Snapshot 只支持有限业务恢复，不是任意节点精确恢复。
- Replay 聚合已保存记录用于展示，不重新执行工具，也不是 time travel。
- official baseline 仍不等于 production，离线 fixture 也不等于实时能力。
- case fixture、查询特判、未真实生成、缺失指标缩分母、跳过 Milvus、只跑检索、失败请求或混合 provenance 都不能冒充完整评测通过。
- stale 评测 Artifact 可以保留供审计，但读模型会标记 `available=false`，不能作为当前发布证据。
- 用户反馈进入待审核 bad case/backlog，不会自动训练模型。
- A2A 接口是默认关闭的业务 facade，不是完整协议 conformance。
- 浏览器断流后查询 latest run status，不是标准 `Last-Event-ID` 历史事件重放。
- 数据库 retention 不会自动清理本地工具 Artifact、上传文件、评测目录或 Milvus 数据。
- SQLite→MySQL 脚本是逐表迁移工具，不是在线双写和零停机切换方案。
- 仓库内 Monitor/CLS MCP Server 使用本地样例或生成数据，MCP 互操作成功不等于生产观测可信。

遇到不确定的描述时，回到 [全系列事实口径与生产边界](00-全系列事实口径与生产边界.md) 检查证据等级，再到 [检索增强与智能体分层评测](13-检索增强与智能体分层评测.md) 检查结果是否具有日期、命令、数据集、case hash、commit、工作区状态和 Artifact 来源。
